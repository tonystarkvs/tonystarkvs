Overview — What / Why / When / Where / How
⚡
What
Parallel execution = running multiple REST Assured test methods or classes simultaneously in separate threads, slashing total test suite runtime.
💡
Why
API tests are I/O-bound (HTTP calls, network wait). Running 100 tests sequentially may take 5 minutes; in parallel with 10 threads, same suite completes in ~30 seconds.
⏰
When
Use in CI/CD pipelines when test suite grows beyond 50 tests and sequential run time exceeds 2–3 minutes. Enable after test independence is confirmed.
📍
Where
Configured in testng.xml with parallel and thread-count attributes. REST Assured itself is stateless per request — safe for parallel use.
⚙️
How
Set parallel="methods" in testng.xml → avoid shared mutable state → use ThreadLocal for per-thread data → configure RestAssuredConfig per request.
Keyword Glossary
parallel="methods"
TestNG attribute: each @Test method runs in its own thread simultaneously.
parallel="classes"
TestNG attribute: each test class runs in its own thread; methods within a class run sequentially.
thread-count
Number of threads in the TestNG thread pool (e.g. 5 means 5 tests run at once).
ThreadLocal<T>
Java class providing per-thread variable storage — each thread gets its own isolated copy. Essential for thread-safe token/config sharing.
Race condition
Bug where two threads read/write shared state simultaneously causing unpredictable results — the #1 parallel testing problem.
RequestSpecification
REST Assured object bundling baseURI, headers, auth — create a new instance per test thread, never share across threads.
RestAssured.baseURI
Static field on RestAssured class — NOT thread-safe. One thread changing it affects all threads. Use request-level config instead.
synchronized
Java keyword making a block/method accessible by only one thread at a time — used for shared resource protection like token stores.
AtomicReference
Thread-safe wrapper for an object reference — compareAndSet() allows lock-free atomic updates to shared variables.
data-provider parallel
TestNG feature: @DataProvider(parallel=true) runs the same test method with different data in parallel threads.
Architecture — Parallel Execution Flow
TestNG parallel execution model with REST Assured
testng.xml config: <suite parallel="methods" thread-count="5"> │ ▼ TestNG Thread Pool (5 worker threads) ┌──────────┬──────────┬──────────┬──────────┬──────────┐ │ Thread-1 │ Thread-2 │ Thread-3 │ Thread-4 │ Thread-5 │ └──────────┴──────────┴──────────┴──────────┴──────────┘ testGetUser() testCreatePost() testDeleteUser() testUpdateUser() testListPosts() │ │ │ │ │ ▼ ▼ ▼ ▼ ▼ Each thread creates its OWN RestAssured request — no shared state ⛔ DANGER — shared static state (causes race conditions): RestAssured.baseURI = "..." ← ONE static field, ALL threads read/write it static String token = "..." ← mutable shared token — thread 2 overwrites it ✅ SAFE — per-request config (no sharing): given().baseUri("...").header("Auth","Bearer "+getToken()) ThreadLocal<String> token → each thread has its own isolated copy KEY RULE: API tests are STATELESS per HTTP call. Make your test code equally stateless — no shared mutable variables.
ThreadLocal pattern — safe per-thread token storage
Without ThreadLocal (UNSAFE): static String token; // Thread-1 sets "abc", Thread-2 sets "xyz" // Thread-1 reads "xyz" instead of "abc" → auth fails! With ThreadLocal (SAFE): ThreadLocal<String> TOKEN = new ThreadLocal<>(); Thread-1: TOKEN.set("abc") → Thread-1: TOKEN.get() = "abc" ✓ Thread-2: TOKEN.set("xyz") → Thread-2: TOKEN.get() = "xyz" ✓ // Each thread reads only its own value — completely isolated ThreadLocal lifecycle in @BeforeMethod / @AfterMethod: @BeforeMethod → TOKEN.set(fetchNewToken()) @Test → given().header("Auth", "Bearer " + TOKEN.get()) @AfterMethod → TOKEN.remove() ← IMPORTANT: prevents memory leak
Interview One-Liners
parallel="methods" in testng.xml runs each @Test method in its own thread — the most granular parallelism, best for I/O-bound REST API tests.
RestAssured.baseURI is a static field — NOT thread-safe — setting it in one thread affects all parallel threads; always use given().baseUri() per request instead.
ThreadLocal<T> gives each thread its own isolated copy of a variable — used to store auth tokens, request specs, or user credentials safely across parallel tests.
Race condition = when two threads read and modify the same shared variable simultaneously causing unpredictable behavior — eliminated by making each test fully self-contained.
@DataProvider(parallel=true) is TestNG's way to run the same test with different data sets in parallel — ideal for testing multiple user roles or environments simultaneously.
Thread pool size (thread-count) should be set based on available CPU cores × 2 for I/O-bound tests — too many threads causes overhead; typical range is 5–15 for REST API suites.

Best Practices
✔ Never use RestAssured.baseURI, RestAssured.port, or any static RestAssured field in parallel tests — they are shared across all threads. Use given().baseUri() per request.
✔ Always call ThreadLocal.remove() in @AfterMethod — thread pool threads are reused, so old ThreadLocal values persist indefinitely causing memory leaks and stale data bugs.
✔ Prefer parallel="methods" for API tests — HTTP calls are I/O-bound, so method-level parallelism maximises CPU utilisation without needing extra test class structure.
✔ Use @DataProvider(parallel=true) for data-driven API tests — it's the cleanest way to run the same endpoint across multiple users, IDs, or environments in parallel.
✔ Set thread-count to 5–10 for REST API suites — higher counts give diminishing returns beyond the server's connection pool limit and may cause 429 Too Many Requests.
Interview Q&A
Q1: Why is RestAssured.baseURI unsafe for parallel tests? +
RestAssured.baseURI is a public static field — a single shared value across the JVM. Thread-1 might set it to "https://staging.api.com" while Thread-2 sets it to "https://prod.api.com". Thread-1 then reads "https://prod.api.com" — the wrong value. Solution: use given().baseUri() inline per request, or store a per-thread RequestSpecification in a ThreadLocal.
Q2: What is a race condition in parallel testing and how do you avoid it? +
A race condition occurs when two threads read and write the same variable simultaneously with unpredictable interleaving — e.g. both threads read token="old", both try to refresh, one overwrites the other. Avoid by: (1) ThreadLocal for per-thread values, (2) AtomicReference for shared values with compare-and-set, (3) synchronized blocks for critical sections, (4) making test methods fully self-contained with no shared mutable state.
Q3: What is the difference between parallel="methods" and parallel="classes"? +
parallel="methods" runs each @Test method in a separate thread — maximum parallelism, all methods of all classes run concurrently. parallel="classes" runs each test class in its own thread, but methods within a class run sequentially. Use "methods" for independent API tests. Use "classes" when tests within a class have ordering dependencies (dependsOnMethods) but tests in different classes are independent.
Q4: How does @DataProvider(parallel=true) work? +
@DataProvider(parallel=true) tells TestNG to run each data row in a separate thread from the thread pool. The test method is invoked once per data row — all invocations run concurrently. The thread-count attribute in testng.xml controls the thread pool size. Data rows must be fully independent — avoid shared state between data iterations. Perfect for testing multiple user IDs, environments, or API endpoints simultaneously.
Q5: When should you NOT run REST Assured tests in parallel? +
Avoid parallel execution when: (1) tests have sequential dependencies (CRUD chains where step 2 depends on step 1 data), (2) tests share a unique resource (single test user account), (3) the API has rate limiting that rejects concurrent requests, (4) tests clean up shared data (one test deletes data another test needs), or (5) the API server is a single-threaded local dev server that can't handle concurrent connections.
Overview — What / Why / When / Where / How
🔑
What
OAuth2 token refresh = automatically obtaining a new access token before the old one expires — keeping a long-running test suite authenticated throughout execution.
💡
Why
Access tokens expire (typically 1 hour). A 500-test suite may run for 2+ hours. Without refresh, tests start failing with 401 Unauthorized midway — silently breaking coverage.
⏰
When
Needed for any test suite longer than the token's expiry window — usually 3600 seconds. Check token expiry in @BeforeMethod and refresh proactively before expiry.
📍
Where
Implement in a TokenManager utility class. Called from @BeforeSuite, @BeforeClass, or @BeforeMethod depending on token lifetime vs test count.
⚙️
How
Store token + expiry timestamp. Before each request, check if currentTime + buffer > expiry. If so, call the login/refresh endpoint → store new token → continue.
Keyword Glossary
OAuth2
Authorization framework — client gets an access token from an auth server and passes it in every API request header.
Access Token
Short-lived credential (JWT or opaque) sent in Authorization: Bearer header — typically expires in 1 hour.
Refresh Token
Long-lived credential (days/weeks) used exclusively to obtain a new access token without re-entering credentials.
JWT (JSON Web Token)
Self-contained token with header.payload.signature — payload contains exp claim (expiry as Unix timestamp).
exp claim
JWT payload field holding expiry as Unix epoch seconds — parse it to know exactly when the token expires.
grant_type
OAuth2 request field specifying the flow — password for direct credentials, refresh_token for token refresh.
client_credentials
OAuth2 grant type for service-to-service auth — no user involved, app authenticates with client_id + client_secret.
@BeforeSuite
TestNG annotation — runs once before the entire test suite. Use for initial token fetch shared across all tests.
buffer window
Refresh the token X seconds BEFORE actual expiry — prevents edge case where token expires mid-request.
synchronized
Java keyword ensuring only one thread executes the token refresh block — prevents multiple parallel threads from refreshing simultaneously.
OAuth2 Token Refresh Architecture
OAuth2 flow — initial auth + automatic refresh in test suite
INITIAL AUTH (@BeforeSuite): Test Framework → POST /oauth/token → Auth Server body: { grant_type:password, username, password, client_id, client_secret } ↓ Auth Server → Response: { access_token, refresh_token, expires_in:3600 } Store: TOKEN = access_token, EXPIRY = now() + 3600s, REFRESH = refresh_token TEST EXECUTION FLOW (@BeforeMethod check): ┌─────────────────────────────────────────────────────────────────┐ │ BEFORE EACH TEST │ │ │ │ currentTime + 60s < EXPIRY? │ │ YES → use TOKEN as-is → proceed to test │ │ NO → REFRESH NEEDED │ │ ├── POST /oauth/token { grant_type:refresh_token, │ │ │ refresh_token:REFRESH } │ │ ├── Extract new access_token + expires_in │ │ ├── Update TOKEN, update EXPIRY │ │ └── proceed to test with new TOKEN │ └─────────────────────────────────────────────────────────────────┘ PARALLEL SAFETY: synchronized block → only ONE thread refreshes at a time Double-check after lock → prevent duplicate refreshes AtomicReference → lock-free token read in hot path
JWT expiry parsing — decode exp claim without a library
JWT structure: header.payload.signature eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJ1c2VyMSIsImV4cCI6MTcwNTAwMDAwMH0.sig Decode exp from payload (base64url): String[] parts = token.split("\\."); String payload = new String(Base64.getUrlDecoder().decode(parts[1])); // payload = { "sub":"user1", "exp":1705000000 } long exp = JsonPath.from(payload).getLong("exp"); // exp is Unix epoch seconds → multiply by 1000 for milliseconds long expiryMs = exp * 1000L; // compare against System.currentTimeMillis()
Interview One-Liners
access_token expires typically in 3600 seconds — a long test suite must proactively refresh it before expiry to avoid 401 Unauthorized failures mid-suite.
Refresh token is a long-lived credential used only to obtain a new access token — sent with grant_type=refresh_token to the auth server's token endpoint.
Buffer window = refresh the token 60 seconds BEFORE actual expiry — prevents edge cases where a token expires between the check and the actual API call.
@BeforeSuite fetches the initial token once for the entire suite; @BeforeMethod checks expiry and refreshes if needed — together they handle any suite duration.
synchronized(lock) in the refresh method ensures only one thread calls the auth endpoint at a time — without it, 10 parallel threads would each fire a separate refresh simultaneously.
JWT exp claim = Unix timestamp in the token payload — decode the base64url payload, extract exp, compare with current time to determine if refresh is needed.

import static io.restassured.RestAssured.given; import java.util.Base64; public class TokenManager { // Singleton instance — one TokenManager for the entire test suite private static TokenManager instance; private static final Object INIT_LOCK = new Object(); private volatile String accessToken; // volatile = ensures all threads see the latest written value immediately private volatile String refreshToken; private volatile long expiryMs; // expiryMs = System.currentTimeMillis() + expires_in * 1000 private static final long BUFFER_MS = 60_000L; // BUFFER_MS = refresh 60 seconds before actual expiry // Prevents edge case: token valid at check but expired during API call // Auth server configuration private static final String TOKEN_URL = "https://auth.example.com/oauth/token"; private static final String PASSWORD = System.getenv("TEST_PASSWORD"); private TokenManager() {} public static TokenManager getInstance() { if (instance == null) { synchronized (INIT_LOCK) { if (instance == null) { instance = new TokenManager(); instance.fetchInitialToken(); // Double-checked locking: check null twice around synchronized // Prevents multiple threads from each creating a new instance } } } return instance; } public synchronized String getToken() { // synchronized = only one thread executes this at a time // Prevents two parallel threads both refreshing simultaneously if (isExpiringSoon()) { refresh(); } return accessToken; } private boolean isExpiringSoon() { return System.currentTimeMillis() + BUFFER_MS >= expiryMs; // true if current time + 60s buffer reaches or passes expiry } private void fetchInitialToken() { String response = given() .contentType("application/x-www-form-urlencoded") .formParam("grant_type", "password") .formParam("username", USERNAME) .formParam("password", PASSWORD) .formParam("client_id", CLIENT_ID) .formParam("client_secret", CLIENT_SECRET) .when() .post(TOKEN_URL) .then() .statusCode(200) .extract().asString(); parseAndStore(response); System.out.println("[TokenManager] Token refreshed at: " + System.currentTimeMillis()); } private void parseAndStore(String json) { JsonPath jp = JsonPath.from(json); accessToken = jp.getString("access_token"); refreshToken = jp.getString("refresh_token"); int expiresIn = jp.getInt("expires_in"); // expires_in = seconds until access_token expires (typically 3600) expiryMs = System.currentTimeMillis() + (expiresIn * 1000L); // Convert to milliseconds and add to current time for absolute expiry } }

Best Practices
✔ Always read credentials from environment variables (System.getenv()) — never hardcode client_id, client_secret, or passwords in test code or version control.
✔ Use a 60-second buffer window — refresh the token before it actually expires. A token valid at check time may expire during the API call's network round-trip.
✔ Use synchronized around the refresh block with double-checked locking — prevents N parallel threads each firing a separate refresh request simultaneously.
✔ Update the refresh_token every time you receive a new one — some OAuth2 servers rotate refresh tokens on each use; storing the old one will cause future refresh calls to fail.
✔ Declare accessToken and refreshToken as volatile when shared across threads — ensures all threads immediately see any value written by another thread.
Interview Q&A
Q1: How do you handle OAuth2 token expiry in a long test suite? +
Create a TokenManager singleton with fetchInitialToken() called in @BeforeSuite and a getToken() method that checks isExpiringSoon() before every return. Store the token expiry as System.currentTimeMillis() + expires_in*1000. Add a 60-second buffer. Use synchronized to prevent parallel threads from each refreshing simultaneously. Call getToken() in @BeforeMethod or directly in each test's given() block.
Q2: What is the difference between password grant and client_credentials grant? +
Password grant (grant_type=password) authenticates a real user — requires username + password alongside client credentials. Returns both access_token and refresh_token. Client credentials grant (grant_type=client_credentials) authenticates the application itself — no user involved, just client_id + client_secret. Returns access_token but typically no refresh_token (re-authenticate to get a new token). Use client_credentials for microservice-to-microservice API testing.
Q3: Why use volatile for shared token fields? +
Without volatile, Java's memory model allows each thread to cache a variable's value in its local CPU cache. Thread-1 may update accessToken but Thread-2 still reads the stale cached value. volatile forces all reads/writes to go directly to main memory — guaranteeing all threads see the latest written value immediately. It doesn't make compound operations atomic (for that you need synchronized or AtomicReference), but it ensures visibility for simple assignment and read operations.
Q4: How do you parse the JWT exp claim without adding an external JWT library? +
Split the JWT on "\\." to get [header, payload, signature]. Take parts[1] (payload). Add base64url padding if needed (4-length % 4 equals signs). Decode with Base64.getUrlDecoder().decode() — must use URL decoder not standard (JWT uses - and _ not + and /). Convert bytes to String. Use JsonPath.from(payload).getLong("exp") to extract the exp value. Multiply by 1000L to convert Unix seconds to Java milliseconds for comparison with System.currentTimeMillis().
Q5: What is double-checked locking and why is it needed for token refresh? +
Double-checked locking = check the condition BEFORE acquiring the lock (fast path — no synchronization overhead for most calls), then check AGAIN INSIDE the synchronized block. Pattern: if(expired){synchronized(lock){if(expired){refresh();}}}. The second check is critical: Thread-1 and Thread-2 both pass the outer check simultaneously, Thread-1 acquires the lock and refreshes, Thread-2 then acquires the lock — without the inner check, Thread-2 would refresh again unnecessarily, causing double network calls to the auth server.
Overview — What / Why / When / Where / How
📋
What
JSON Schema validation = asserting that a response's JSON structure matches a predefined schema contract — field names, data types, required fields, and value constraints.
💡
Why
Body assertions check individual values. Schema validation checks the complete contract — catching type changes (int → string), missing required fields, and structural regressions.
⏰
When
Add schema validation to every critical endpoint after the first passing test. Run in CI to catch API contract breaking changes before they reach production.
📍
Where
In REST Assured's then() chain: .body(matchesJsonSchemaInClasspath("schema.json")). Schema files live in src/test/resources/schemas/.
⚙️
How
Add rest-assured:json-schema-validator dependency. Write JSON Schema file. Use matchesJsonSchemaInClasspath() matcher in the then() assertion chain.
Keyword Glossary
JSON Schema
A JSON document describing the expected structure of another JSON document — defines types, required fields, formats, and constraints.
$schema
Top-level JSON Schema field declaring which draft version is used — e.g. "https://json-schema.org/draft/2020-12/schema".
properties
JSON Schema keyword: defines the expected fields of an object with their type and constraints.
required
JSON Schema keyword: array of field names that must be present in the JSON response — validation fails if any are missing.
type
JSON Schema constraint: valid values are string, integer, number, boolean, array, object, null.
format
JSON Schema field: validates string format — email, uri, date-time, uuid. Note: format validation is not enforced by default in all validators.
$ref
JSON Schema: reference to a reusable schema definition — e.g. {"$ref":"#/$defs/Address"} reuses the Address schema definition.
matchesJsonSchemaInClasspath
REST Assured static import matcher — validates response body against a JSON Schema file on the classpath.
json-schema-validator
REST Assured add-on module: io.rest-assured:json-schema-validator — provides schema validation matchers.
additionalProperties
JSON Schema keyword: false = reject any JSON fields not declared in properties. true = allow any extra fields.

Interview One-Liners
JSON Schema validation validates the complete response contract — field names, data types, required fields, and value constraints — in one assertion, replacing dozens of individual body() checks.
matchesJsonSchemaInClasspath("schema.json") = REST Assured matcher that loads a schema file from src/test/resources and validates the full response JSON against it.
"required":["id","email"] = the most critical JSON Schema keyword — it explicitly fails the test if any listed field is absent from the response, catching API regressions immediately.
additionalProperties: false = rejects any JSON response fields not declared in the schema — enforces that the API contract is exact and doesn't silently add undocumented fields.
$ref and $defs = JSON Schema's composition mechanism — define a reusable sub-schema once in $defs and reference it multiple times across the schema file to avoid repetition.
Schema-first API contract testing = generate schemas from OpenAPI/Swagger spec → use in tests — any API breaking change immediately fails the schema assertion before it reaches consumers.

Best Practices
✔ Always specify "required" explicitly in your schema — a schema without required fields only validates types but silently passes when fields go missing entirely.
✔ Store schema files in src/test/resources/schemas/ — one file per endpoint type. Name them after the resource: user-schema.json, post-schema.json, order-schema.json.
✔ Use $ref and $defs to reuse sub-schemas — if Address appears in User and Order schemas, define it once and reference it to keep schemas DRY and consistent.
✔ Run schema validation in CI — commit schemas to version control and treat schema violations the same as test failures. Schema diffs on PR review catch API breaking changes before merge.
✔ Combine schema validation with value assertions in the same test — schema validates structure, body() validates content. Together they provide complete API contract coverage.
Interview Q&A
Q1: What does JSON Schema validation catch that regular body() assertions don't? +
body() assertions check specific individual values. JSON Schema validation catches: (1) field type changes — id changed from integer to string, (2) required fields going missing — email removed from response, (3) structural changes — address changed from object to null, (4) value constraints — negative id, empty name string. One schema assertion replaces 20+ body() assertions while also catching structural regressions that no body() assertion would notice.
Q2: What is the required pom.xml dependency for REST Assured schema validation? +
Add io.rest-assured:json-schema-validator (same version as rest-assured, e.g. 5.3.2) with scope=test. This module bundles the Everit/FGE JSON schema validator and provides the matchesJsonSchemaInClasspath() matcher. The static import is: import static io.restassured.module.jsv.JsonSchemaValidator.matchesJsonSchemaInClasspath;
Q3: What is the difference between matchesJsonSchemaInClasspath and matchesJsonSchema? +
matchesJsonSchemaInClasspath("path") loads the schema file from the Java classpath — the file must be in src/test/resources. matchesJsonSchema(String) takes a schema as a Java String — useful for inline, dynamically generated, or environment-specific schemas. matchesJsonSchema(File) and matchesJsonSchema(InputStream) load from a file object or input stream respectively. All produce identical validation behavior; the difference is only the schema source.
Q4: What does additionalProperties: false do in a JSON Schema? +
additionalProperties:false rejects any field in the JSON response that is NOT declared in the schema's properties section. This enforces an exact API contract — any field the API adds that isn't in your schema causes the test to fail. Without it (default: true), the schema is lenient about extra fields. Use false in strict contract testing where you want to detect any undocumented additions. Use true (or omit) when you only care about required fields and want to allow the API to add new fields freely.
Q5: How do you generate JSON Schemas from an OpenAPI/Swagger specification? +
Tools: (1) openapi-generator-cli generates JSON schemas from OpenAPI component schemas. (2) Swagger UI and Swagger Parser can extract component schemas. (3) Manually copy the schema object from the OpenAPI spec — each component's "schema" section IS a JSON Schema. (4) Use json-schema-generator Maven plugin to generate schemas from POJO classes. Best practice: treat the OpenAPI spec as the single source of truth; generate both API implementation and test schemas from it to guarantee they stay in sync.
