Overview — JSONObject & JSONArray Request Creation
📦
What
org.json's JSONObject and JSONArray build request body JSON explicitly via put() chains — chainable, readable, and preserving insertion order.
💡
Why
More readable than raw String concatenation. Explicit JSON construction with type-safe put() — no Jackson dependency needed. String.toString() gives final body.
⏰
When
Use for complex nested PUT/POST bodies where you want explicit control over every field with no POJO class overhead. Good for one-off test payloads.
📍
Where
given().body(jsonObject.toString()) — JSONObject.toString() gives the JSON string. Must call .toString() explicitly.
⚙️
How
Build bottom-up: deepest JSONObject first → nest via put(key, JSONObject). Arrays via JSONArray with put(element). Final: body(root.toString()).

Keyword Glossary
JSONObject
org.json class building JSON objects via put(key, value). Chainable. toString() = final JSON string.
JSONArray
org.json class building JSON arrays via put(value). Holds strings, numbers, or JSONObjects.
put(key, value)
JSONObject method — adds a field. Value can be String, int, double, boolean, JSONObject, JSONArray.
put(value)
JSONArray method — appends an element. Chainable: new JSONArray().put("a").put("b").
toString()
Converts JSONObject/JSONArray to JSON String — MANDATORY when passing to body(). toString(2) = pretty-printed.
chaining
put() returns 'this' in both JSONObject and JSONArray — allows fluent builder style: jo.put("a","x").put("b","y").
bottom-up build
Create deepest nested objects first, then embed them into parent objects. Prevents NPE during construction.
pathParam()
REST Assured method replacing a {placeholder} in the URL path: pathParam("empId","EMP-1001").
queryParam()
REST Assured method appending ?key=value to the URL: queryParam("notify","true").
header()
REST Assured method adding an HTTP header to the request: header("X-Tenant-ID","ORG-777").
Interview One-Liners
JSONObject.put(key, JSONObject) = embeds one JSONObject inside another — used to create nested objects. put() returns 'this' enabling method chaining.
JSONArray.put(JSONObject) = appends an object element to a JSON array — used to build arrays of objects like projects[] in the employee body.
body(jo.toString()) = MANDATORY: JSONObject doesn't auto-serialize — you must explicitly call toString() before passing to REST Assured body().
Bottom-up construction = always build the deepest nested object first (geo), then embed it into its parent (location), then into the grandparent (department).
toString(2) = pretty-print with 2-space indent — use System.out.println(jo.toString(2)) to verify the exact JSON structure before sending.
pathParam("empId","EMP-1001") = replaces {empId} in the URL. Without this call, REST Assured would send a literal "{empId}" in the URL causing a 404.

Best Practices
✔ Always call body(jo.toString()) — never pass a JSONObject directly to body() without toString(). REST Assured won't serialize it automatically like Jackson does for POJOs.
✔ Build complex JSONObjects bottom-up — create the deepest nested object first (geo), embed it upward (location → department). This prevents NullPointerException during construction.
✔ Use toString(2) during development to print a formatted, readable version of the request body — verify the JSON structure before running assertions.
✔ Prefer JSONArray chaining: new JSONArray().put("A").put("B").put("C") over separate add() calls — keeps array construction concise and readable.
✔ Always verify the final URL using log().uri() when working with path params and query params — confirms {empId} was replaced and all query params were appended correctly.
Interview Q&A
Q1: Why must you call toString() when using JSONObject with body()? +
REST Assured's automatic serialization (Jackson-based) works for Map and POJO types. JSONObject from org.json is not a recognized type — REST Assured doesn't know how to serialize it. You must call jo.toString() to produce a raw JSON string, which REST Assured then sends as the body text. Without toString(), REST Assured would call JSONObject.toString() internally via Object.toString() which might not produce valid JSON depending on the implementation.
Q2: What is the bottom-up build pattern for nested JSONObjects? +
Always create the deepest nested object first, then embed it into its parent. Example: create GeoObject first, then pass it into LocationObject.put("geo", geoObject), then pass LocationObject into DepartmentObject.put("location", locationObject). If you try to build top-down, you'd be referencing objects that don't exist yet. Bottom-up ensures each object is complete before being embedded.
Q3: What is the difference between JSONArray.put(String) and JSONArray.put(JSONObject)? +
JSONArray.put(String) appends a string element — produces ["SDET","LEAD"]. JSONArray.put(JSONObject) appends an entire JSON object as an array element — produces [{"projectId":"PRJ-101","name":"HRMS Automation",...}]. Both return 'this' for chaining. Use put(String) for simple string arrays; use put(JSONObject) for arrays of complex objects like projects[] or items[].
Q4: How do pathParam and queryParam work together in REST Assured? +
pathParam("empId","EMP-1001") replaces the {empId} placeholder in the URL template "/v2/employees/{empId}" → produces "/v2/employees/EMP-1001". queryParam("version","2") appends ?version=2 to the URL. Multiple queryParams are joined: ?version=2¬ify=true&audit=strict. Both are set on the given() block. The final URL becomes: https://api.hrms.com/v2/employees/EMP-1001?version=2¬ify=true&audit=strict.
Q5: JSONObject vs HashMap for request body — when to choose which? +
JSONObject: use when you want explicit, readable JSON construction with preserved key order. Requires toString(). Extra org.json dependency. JSONObject.put() returns 'this' — fully chainable. HashMap: use when you want REST Assured + Jackson to auto-serialize — no toString() needed, just pass the Map directly to body(). Jackson serializes it automatically when Content-Type is application/json. For complex nested structures, Jackson-based Map is often preferred since it doesn't need toString().
Overview — Map<String,Object> Request Creation
🗂️
What
Java's built-in HashMap<String,Object> to build request body — REST Assured + Jackson auto-serialize it to JSON when Content-Type is application/json. No toString() needed.
💡
Why
No extra dependency — just Java standard library. Jackson auto-serializes Maps. Nested objects = nested Maps. Arrays = Lists. Most common pattern in production test frameworks.
⏰
When
Use for dynamic test data, data-driven tests, or when you want full Jackson auto-serialization without creating POJO classes. Industry-standard approach.
📍
Where
given().contentType("application/json").body(map) — REST Assured detects jackson-databind on classpath and serializes the Map automatically.
⚙️
How
Nested JSON object = Map as value. JSON array of strings = List<String>. JSON array of objects = List<Map>. Jackson handles the entire serialization tree.
Keyword Glossary
HashMap<String,Object>
Java Map implementation — keys are String (JSON field names), values are any Object (String, int, boolean, Map, List).
auto-serialization
REST Assured + Jackson convert Map → JSON automatically when Content-Type is application/json. No toString() needed.
LinkedHashMap
Use instead of HashMap to preserve insertion key order in the serialized JSON — important when API validates field order.
put(key, Map)
Map.put() with a nested Map as value → produces a nested JSON object when Jackson serializes it.
List<String>
Jackson serializes List<String> as a JSON string array: ["SDET","LEAD","REVIEWER"].
List<Map>
Jackson serializes List<Map<String,Object>> as a JSON array of objects: [{...},{...}].
Arrays.asList()
Quick immutable List creation for fixed array fields. Use new ArrayList<>() for dynamic addition.
contentType()
MANDATORY trigger for Jackson auto-serialization. contentType("application/json") tells REST Assured to use Jackson on the body Map.
log().body()
REST Assured method printing the serialized request body — use in given() block to verify JSON before sending.
RequestSpecBuilder
Fluent builder creating a reusable RequestSpecification — bundles baseUri, headers, auth, and content type to avoid repetition.
Interview One-Liners
body(map) with contentType("application/json") = REST Assured detects Jackson on classpath and auto-serializes the Map to JSON — no toString() call needed unlike JSONObject.
Nested Map as value: map.put("department", deptMap) = Jackson serializes the inner Map as a nested JSON object — the same pattern works for any depth of nesting.
List<String> as value: map.put("roles", Arrays.asList("SDET","LEAD")) = Jackson serializes the List as a JSON string array: "roles":["SDET","LEAD"].
List<Map<String,Object>> as value: map.put("projects", listOfMaps) = Jackson serializes it as a JSON array of objects — each Map becomes one object in the array.
LinkedHashMap vs HashMap: LinkedHashMap preserves insertion order in the JSON output — use it when API validation or logging depends on field ordering.
log().body() in the given() block prints the exact serialized JSON that will be sent — essential for debugging complex nested request bodies before assertion failures.

Best Practices
✔ Always set contentType("application/json") when using Map as body — without it, REST Assured will not trigger Jackson serialization and may send form-encoded data instead.
✔ Use LinkedHashMap instead of HashMap when field ordering matters — Jackson preserves LinkedHashMap's insertion order in the serialized JSON output.
✔ Extract repeated request setup into a RequestSpecBuilder — keeps test methods focused on data/assertions, not on headers and params repetition.
✔ Use log().body() in the given() block during development — prints the exact JSON Jackson will send, making it easy to spot missing fields or incorrect nesting.
✔ Use Arrays.asList() for fixed string arrays and new ArrayList<>() when you need to add items dynamically — Arrays.asList() returns a fixed-size list that throws on add().
Interview Q&A
Q1: Why is contentType("application/json") mandatory when using Map as request body? +
Without contentType("application/json"), REST Assured doesn't know the desired format for the body. It may send the Map as URL-encoded form parameters (key1=val1&key2=val2) instead of JSON. The content type header signals REST Assured to use Jackson's ObjectMapper for serialization, which converts the Map tree into a proper JSON string for the request body.
Q2: How does Jackson handle nested Maps vs nested JSONObjects? +
For Map: Jackson automatically traverses the Map tree. When it encounters a Map value, it recursively serializes it as a nested JSON object. When it sees a List, it serializes it as a JSON array. No toString() needed — Jackson handles the entire tree. For JSONObject: Jackson doesn't recognize org.json's JSONObject type as a special type — you must call toString() to get the JSON string first, then pass that string to body().
Q3: What is the difference between Arrays.asList() and new ArrayList<>()? +
Arrays.asList() returns a fixed-size List backed by the array — you cannot add or remove elements (throws UnsupportedOperationException). Use it for static, known-at-compile-time arrays. new ArrayList<>() creates a mutable List — you can add/remove elements dynamically. Use it when building arrays in loops or when the size is unknown. Both serialize identically to JSON arrays when Jackson processes them.
Q4: How do you represent a JSON array of objects using Map approach? +
Create each object as Map<String,Object>. Add all objects to a List<Map<String,Object>>. Put that list as a Map value: body.put("projects", listOfMaps). Jackson serializes List<Map> as a JSON array of objects: "projects":[{"projectId":"PRJ-101",...},{"projectId":"PRJ-202",...}]. This is equivalent to JSONArray of JSONObjects in the org.json approach.
Q5: What does RequestSpecBuilder provide over repeating headers in every test? +
RequestSpecBuilder creates a reusable RequestSpecification that bundles baseUri, headers, query params, and path params. You call given(spec) instead of repeating .header(), .queryParam() etc. in every test method. Benefits: (1) DRY — change the token in one place, (2) consistency — every test uses identical headers, (3) readability — test methods focus on body data and assertions, not boilerplate setup, (4) thread safety when spec is created per-thread in parallel tests.
Overview — Gson + POJO Request Creation
☕
What
Design Java POJO classes mirroring the PUT request JSON structure, populate them, then use gson.toJson(pojo) to serialize to JSON string for the request body.
💡
Why
Type-safe request construction — IDE autocomplete, compile-time field checks, refactoring support. Gson doesn't need getters — simpler POJOs than Jackson-based approach.
⏰
When
Use for recurring complex request schemas used across multiple test classes. Define POJOs once, reuse everywhere. Best for enterprise frameworks with a defined API contract.
📍
Where
given().body(gson.toJson(employeePojo)) — toJson() converts the POJO graph to JSON string. Alternatively configure REST Assured to use Gson auto-detection.
⚙️
How
Define nested POJO classes (bottom-up). Populate fields directly (no setters needed). Gson.toJson(root) traverses the entire object graph and produces the complete JSON string.
Keyword Glossary
Gson
Google's JSON library — toJson(pojo) serializes any Java object graph to JSON string. No getters required.
toJson(obj)
Gson serialization method — Java POJO → JSON string. Traverses nested POJO fields recursively.
@SerializedName
Maps a Java field name to a different JSON key. Used when API uses snake_case but Java uses camelCase.
field access
Gson reads/writes POJO fields directly via reflection — no getters or setters needed in POJO classes.
nested POJO field
A POJO field typed as another POJO — Gson serializes it as a nested JSON object recursively.
List<String> field
A POJO field typed as List<String> — Gson serializes it as a JSON string array.
List<POJO> field
A POJO field typed as List<ProjectModel> — Gson serializes it as a JSON array of objects.
GsonBuilder
Factory for custom Gson — serializeNulls(), setPrettyPrinting(), setFieldNamingPolicy() etc.
serializeNulls()
GsonBuilder option — include null POJO fields in JSON output. Default: null fields excluded.
bottom-up POJO design
Define innermost POJO class first (GeoModel), then the class that contains it (LocationModel), etc.
Interview One-Liners
Gson.toJson(pojo) = serializes the entire POJO object graph to a JSON string — Gson traverses all nested POJO fields recursively without any configuration.
Gson POJOs need NO getters = unlike Jackson which uses getter methods for serialization, Gson accesses fields directly via reflection — simpler class definitions.
body(gson.toJson(emp)) = pass the JSON string to REST Assured body() — like JSONObject, Gson requires explicit toJson() call since REST Assured doesn't auto-detect Gson by default.
@SerializedName("dept_id") = maps JSON key "dept_id" to Java field "deptId" — Gson's equivalent of Jackson's @JsonProperty. Used when API uses snake_case naming.
GsonBuilder.serializeNulls() = include null fields in serialized JSON — important when API schema requires all fields present even if null (e.g. PATCH vs full PUT).
POJO design rule = create one POJO class per JSON object level, bottom-up: GeoModel first, then LocationModel (contains GeoModel), then DepartmentModel (contains LocationModel).

Best Practices
✔ Define POJO classes bottom-up — create the innermost class first (GeoModel), then the class that contains it (LocationModel). This makes the dependency chain clear and prevents forward-reference issues.
✔ Use a dedicated builder method (like buildEmployee()) that returns a fully populated POJO — keeps test methods clean and enables easy data-driven testing by passing different parameters.
✔ Use new GsonBuilder().setPrettyPrinting().create().toJson(pojo) only for debugging/logging — send the compact new Gson().toJson(pojo) in the actual request for smaller payload size.
✔ Apply @SerializedName when the API uses snake_case JSON keys but Java convention requires camelCase field names — keeps both the POJO and the JSON contract correct.
✔ Use emp.empId directly in pathParam("empId", emp.empId) — cross-references the POJO data to ensure the URL path parameter matches the request body ID field exactly.
Interview Q&A
Q1: Compare JSONObject, Map, and Gson+POJO for PUT request body creation. +
JSONObject: explicit construction via put() chains, preserves key order, requires toString() before body(). Extra org.json dependency. Good for one-off requests. Map: most common, no extra dependency, Jackson auto-serializes (no toString()), nested via nested Maps, arrays via Lists. Industry-standard. Gson+POJO: most type-safe, compile-time checking, IDE autocomplete, Gson reads fields directly without getters, requires toJson() like JSONObject. Best for recurring request schemas shared across multiple tests. Choose Map for simple flexible cases; Gson+POJO for complex recurring structures.
Q2: Why does Gson not require getters for serialization unlike Jackson? +
Gson uses Java reflection to directly access POJO fields regardless of their access modifier (public, private, protected). It doesn't rely on JavaBeans conventions (getter/setter patterns) that Jackson uses by default. Jackson needs getters because it follows the JavaBeans standard — it discovers properties by looking for public getXxx() methods. Gson bypasses this by directly reading field values via Field.setAccessible(true) and Field.get(object).
Q3: What does GsonBuilder.serializeNulls() do to the request body? +
By default, Gson excludes POJO fields that have null values from the JSON output — they simply don't appear. GsonBuilder.serializeNulls() overrides this and includes null fields as "fieldName": null. Use it when the API schema requires all fields to be present even if null (e.g. a full PUT that must explicitly null out optional fields). Without serializeNulls(), null fields are silently omitted — which might leave the server with old values if it only updates provided fields.
Q4: How does Gson serialize a List<POJO> field in the request? +
Gson automatically serializes a List<ProjectModel> field as a JSON array of objects. For each ProjectModel instance in the list, Gson creates a JSON object with the fields of that POJO. No TypeToken is needed for serialization — TypeToken is only required for DESERIALIZATION when Gson needs to know the element type to create instances. For serialization, Gson inspects the actual runtime type of each List element and serializes it accordingly.
Q5: When should you choose Gson+POJO over Map for request creation in a production framework? +
Choose Gson+POJO when: (1) the same complex request schema is reused across 10+ test methods or test classes — define the POJO once, reuse everywhere, (2) compile-time validation is important — typos in field names cause compilation errors not runtime failures, (3) IDE support is critical — autocomplete on pojo.department.location shows all available fields, (4) refactoring — renaming a JSON field requires only one POJO change, not hunting through all Map.put() calls. Use Map for: quick prototyping, dynamic/unknown schemas, or one-off test requests where creating POJO classes is overkill.
