🌐 Overview — What / Why / When / Where / How
❓
WHAT
REST is an architectural style for designing networked apps using HTTP. API is a contract between client & server. Web Services expose business logic over a network.
💡
WHY
Enables interoperability — any language, any platform can talk to each other using standard HTTP. REST Assured is Java's most popular library to test REST APIs automatically.
⏰
WHEN
Use REST Assured when your team ships REST APIs and you need automated regression, integration, or contract testing as part of CI/CD.
📍
WHERE
SDETs use it in Maven/Gradle Java projects. Runs in TestNG/JUnit test suites, CI pipelines (Jenkins, GitHub Actions), API regression suites.
⚙️
HOW
Add rest-assured dependency → write Given/When/Then test → run via Maven → assert status, headers, body. BDD-style readable test code.
🎤 Interview One-Liners
REST = Representational State Transfer — a stateless, client-server architecture style that uses HTTP and treats every resource as a URL-addressable entity.
API = Application Programming Interface — a defined contract (endpoints, request/response formats) that allows two systems to talk to each other.
Web Service = a REST API that is deployed over HTTP/HTTPS on a server and is accessible via the internet or intranet.
JSON = JavaScript Object Notation — a lightweight, human-readable key-value text format used as the body of most REST API requests and responses.
REST Assured = a Java DSL library that lets you write BDD-style (Given/When/Then) automated tests for REST APIs with minimal boilerplate code.
Stateless = each HTTP request must carry all info the server needs; the server stores zero client-session data between calls.

🎯 SDET Interview Q&A
Q1: What is the difference between REST and SOAP? +
REST uses HTTP methods (GET/POST/PUT/DELETE) and JSON; it is stateless, lightweight, and uses URL-based resources. SOAP is a protocol that uses XML envelopes, mandatory WSDL contracts, and supports WS-Security. REST is preferred for modern APIs due to simplicity and performance.
Q2: Why do we use REST Assured instead of Postman for automation? +
Postman is a manual testing tool (though it has Newman for scripts). REST Assured integrates natively with Java, TestNG/JUnit, Maven, and CI/CD pipelines — enabling version-controlled, reusable, data-driven automated API regression suites that run headlessly in build pipelines.
Q3: What is GPath in REST Assured? +
GPath is a Groovy-based path expression language (similar to XPath for XML) used inside REST Assured's body() assertions. Example: body("users[0].address.city", equalTo("Delhi")) — the dot notation navigates nested JSON objects and [0] indexes into arrays.
Q4: What is the difference between baseURI and basePath? +
baseURI is the server domain (e.g. https://api.example.com). basePath is the common URL prefix appended to all requests (e.g. /api/v2). Together: baseURI + basePath + endpoint = full URL. Setting them globally avoids repetition across every test.
Q5: What is statelessness in REST? +
Statelessness means the server does not store any session/client context between requests. Every HTTP call must contain all information needed (auth tokens, parameters). This makes REST services scalable and independent, allowing load balancers to route requests to any server instance.
🌐 Overview
❓
WHAT
HTTP (HyperText Transfer Protocol) is the communication protocol used by REST APIs. Every API call is an HTTP request that gets an HTTP response.
💡
WHY
Understanding HTTP methods, status codes, headers, request & response anatomy is essential for writing meaningful API assertions and debugging failures.
⏰
WHEN
Every time you write a REST Assured test — you select an HTTP method, assert a status code, and validate response headers/body.
📍
WHERE
In REST Assured: get(), post(), put(), delete() = HTTP methods. statusCode() = HTTP response code assertion. header() = HTTP header assertion.
⚙️
HOW
Client sends Request (method+URL+headers+body) → Server processes → Server returns Response (status code + headers + body).
🎤 Interview One-Liners
GET = retrieves data from server, no body in request, idempotent (same result every call), safe (does NOT modify data).
POST = creates a new resource on server, sends body with data, NOT idempotent (each call creates a new record).
PUT = updates/replaces an existing resource completely, idempotent (calling twice gives same result).
DELETE = removes a resource from server, idempotent — deleting a deleted resource still returns 200 or 204.
HTTP 200 = OK (success). 201 = Created. 400 = Bad Request. 401 = Unauthorized. 404 = Not Found. 500 = Internal Server Error.
PATCH = partial update of a resource — only send the fields you want to change (unlike PUT which replaces the whole resource).

🎯 SDET Interview Q&A
Q1: Difference between PUT and PATCH? +
PUT replaces the ENTIRE resource — you must send all fields, even unchanged ones. PATCH does a PARTIAL update — you only send the fields that changed. Example: updating a user's email → use PATCH so other fields aren't overwritten.
Q2: What is idempotency and why does it matter for testing? +
Idempotent means calling the same request N times produces the same server state as calling it once. GET, PUT, DELETE are idempotent; POST is NOT. This matters in tests because retrying a PUT/DELETE is safe, but retrying a POST creates duplicate data.
Q3: What is the difference between queryParam and pathParam? +
pathParam replaces a variable segment in the URL path — e.g. /users/{id} → /users/5. queryParam appends ?key=value to filter or search — e.g. /posts?userId=1. Path params identify a resource; query params filter/paginate results.
Q4: When should you use log().all() vs log().ifValidationFails()? +
Use log().all() during development/debugging to see full request-response. Use log().ifValidationFails() in CI pipelines — it only outputs logs when a test fails, keeping build output clean for passing tests.
Q5: What does the Accept header do? +
The Accept request header tells the server what response format the client can handle. Accept: application/json → client wants JSON. Accept: application/xml → client wants XML. The server responds with Content-Type matching the requested format (or 406 Not Acceptable if unsupported).
🌐 Overview
❓
WHAT
Body assertions validate the JSON response content using Hamcrest matchers and GPath expressions. Soft assertions let ALL checks run even if one fails.
💡
WHY
Hard assertions (default) stop on first failure, hiding other issues. Soft assertions (SoftAssert) collect all failures and report them together — critical for comprehensive API validation.
⏰
WHEN
Use multiple assertions in one test when validating complete API contracts. Use soft assertions when you want to validate 10+ fields without losing failure info on subsequent fields.
📍
WHERE
In then().body() chains — using Hamcrest matchers. Soft assertions use TestNG's SoftAssert class outside the REST Assured chain.
⚙️
HOW
Chain multiple .body() calls in then(). For soft: create SoftAssert sa, call sa.assertEquals(), end with sa.assertAll().
🎤 Interview One-Liners
Hard Assertion = stops test execution immediately on first failure — you lose visibility into all other potential failures in the same test.
Soft Assertion = collects ALL assertion failures throughout the test, then reports them all together when assertAll() is called at the end.
GPath = Groovy Path — expression language for navigating JSON/XML structures using dot notation and array indexing inside REST Assured body().
Hamcrest = matcher library — provides human-readable assertion expressions like equalTo(), hasItem(), greaterThan(), containsString().
body() = method to assert JSON/XML response body content using a GPath expression and a Hamcrest matcher.
Multiple assertions = chain multiple .body() calls in one then() block to validate multiple fields in a single HTTP call — saves network round-trips.

🔍 GPath Expressions Explained
JSON RESPONSE (for GPath examples): { "id": 1, "name": "Chitti", "address": { "city": "Delhi", "zip": "110001" }, "roles": ["admin", "user", "viewer"] } GPath Examples: "name" → "Chitti" (top-level string) "id" → 1 (top-level number) "address.city" → "Delhi" (nested object, dot notation) "address.zip" → "110001" (nested object) "roles[0]" → "admin" (array element by index) "roles" → ["admin","user","viewer"] (whole array) "roles.size()" → 3 (size() = array length) "roles.contains('admin')" → true
⚡ Hard vs Soft Assertion Flow
HARD ASSERTION (default REST Assured behavior): Test Start → Assert statusCode == 200 ✓ PASS → continue → Assert name == "Chitti" ✗ FAIL → STOP! Exception thrown → Assert email (NEVER RUNS — test already stopped!) SOFT ASSERTION (TestNG SoftAssert): Test Start → SoftAssert sa = new SoftAssert() → sa.assertEquals(statusCode, 200) ✓ PASS → continue → sa.assertEquals(name, "Chitti") ✗ FAIL → RECORDED, continue → sa.assertNotNull(email) ✓ PASS → continue → sa.assertAll() → Reports ALL failures at once

🎯 SDET Interview Q&A
Q1: What happens if you forget assertAll() in SoftAssert? +
The test PASSES even if assertions failed! SoftAssert records failures internally but only reports them when assertAll() is called. Without it, a broken API appears healthy. Always put assertAll() as the last line of tests using SoftAssert.
Q2: What is the difference between hasItem() and equalTo() on arrays? +
equalTo() checks the ENTIRE array equals a given value. hasItem() checks that at least ONE element in the array matches. For example: body("ids", hasItem(5)) passes if any element is 5; body("ids", equalTo(5)) would try to match the whole array to 5 and always fail.
Q3: What is GPath and how is it different from JSONPath? +
GPath is a Groovy-based path language natively integrated into REST Assured body() assertions. JSONPath (used with jsonPath().get()) is a separate standard query language. GPath uses dot notation and is evaluated directly in then().body(). JSONPath is used explicitly when you call response.jsonPath().getString("path").
Q4: How do you assert response time in REST Assured? +
Use then().time(lessThan(2000L)) — asserts response time is less than 2000 milliseconds. The L suffix makes it a long literal. You can also use response.getTime() to get the value and assert it programmatically with SoftAssert.
Q5: Explain the $ symbol in GPath +
$ represents the ROOT of the JSON document. When the root is an array (not an object), use "$" to reference the array itself. Example: body("$", hasSize(10)) asserts the root array has exactly 10 elements. body("$", not(empty())) asserts the root array is not empty.

Q1: How do you extract a value from POST response and use it in the next test? +
Use extract().path("fieldName") at the end of the then() chain. Store it in a static variable. Example: int id = given()...then().extract().path("id"); Then use it in the next test: get("/resource/" + id). Also works with extract().response() to get the full Response object.
Q2: What is the difference between auth().oauth2() and adding a Bearer header manually? +
auth().oauth2(token) is REST Assured's built-in method that automatically prepends "Bearer " to your token. Manually adding the header gives you full control: header("Authorization", "Bearer " + token). They produce identical HTTP requests — oauth2() is just more readable and less error-prone.
Q3: What is RequestSpecification and why use it? +
RequestSpecification is an interface that bundles reusable request configuration (baseURI, headers, auth, contentType). Built with RequestSpecBuilder. Passed to given(spec). DRY principle — write auth/baseURI config once and reuse across 50+ tests instead of repeating in every test method.
Q4: How do you handle token expiry in API test automation? +
Call the login endpoint in @BeforeClass (or @BeforeSuite) to get a fresh token at the start of each test run. Store the token in a static variable. If tokens expire mid-run, implement a token refresh mechanism using @BeforeMethod to check expiry and re-authenticate when needed.
Q5: How do you test that a 401 is returned for missing auth? +
Make the call WITHOUT any Authorization header and assert statusCode(401). Example: given().baseUri(BASE).when().get("/secure/data").then().statusCode(401). This is a negative test validating that your API properly enforces authentication — critical security test.
🌐 Overview
❓
WHAT
JSONPath extracts specific values from JSON responses. REST Assured offers path(), jsonPath(), extract(), and XmlPath for different extraction scenarios.
💡
WHY
You need to extract IDs, tokens, and field values from responses to use in subsequent API calls, print debug info, or store in reports.
⏰
WHEN
Whenever you need data OUT of the response to USE elsewhere — not just assert it. E.g. extract created resource ID for next API call.
📍
WHERE
In then().extract() for extracting from fluent chain. response.jsonPath() or response.path() for extracting from stored Response object.
⚙️
HOW
4 ways: (1) response.path("field"), (2) response.jsonPath().getString("field"), (3) then().extract().path("field"), (4) deserialize to POJO with as(MyClass.class).
🎤 Interview One-Liners
path() = shortcut method on Response to directly extract a single value by its GPath/JSONPath expression without creating a JsonPath object.
jsonPath() = returns a JsonPath object from the response body — gives access to getString, getInt, getList, getMap, etc.
extract() = exits the REST Assured fluent assertion chain and returns an ExtractableResponse so you can grab values from the response.
XmlPath = REST Assured's built-in XML navigation object — use when the API returns XML instead of JSON, with the same dot-notation syntax.
as(Class) = deserializes (converts) the JSON response body directly into a Java object (POJO) — requires Jackson or Gson on classpath.
JsonPath vs GPath = GPath is used INSIDE body() assertions; JsonPath is used OUTSIDE after calling response.jsonPath() to extract values programmatically.
💡 Core Concepts
🗺️ 4 Extraction Methods Compared
METHOD 1: response.path("field") String name = response.path("name"); → Quickest for single value. Returns Object (needs cast sometimes) METHOD 2: response.jsonPath().getString("field") String city = response.jsonPath().getString("address.city"); → Explicit type: getString, getInt, getList, getMap METHOD 3: then().extract().path("field") int id = given()...then().extract().path("id"); → In fluent chain. Cannot assert AND extract simultaneously METHOD 4: then().extract().response().as(User.class) User user = response.as(User.class); → Full deserialization to POJO — for complex objects
📋 JsonPath Navigation Reference
JSON: { "user": { "id": 5, "name": "Chitti", "roles": ["admin","user"] } } Expression Returns user.id → 5 user.name → "Chitti" user.roles → ["admin","user"] user.roles[0] → "admin" user.roles.size() → 2 user.roles.get(1) → "user" LIST JSON: [ {"id":1,"name":"A"}, {"id":2,"name":"B"} ] [0].name → "A" [1].id → 2 name → ["A","B"] (extracts from all) findAll{it.id>0}.name → ["A","B"] (Groovy closure filter)

🎯 SDET Interview Q&A
Q1: What is the difference between path() and jsonPath().getString()? +
response.path("name") is a shortcut that returns Object — Java infers the type. response.jsonPath().getString("name") explicitly returns String and is type-safe. For numbers use getInt(), for collections use getList(). Prefer the JsonPath typed methods in production code to avoid ClassCastException.
Q2: Can you assert AND extract in the same then() chain? +
No — calling extract() ends the assertion chain. The workaround is: (1) Store the response first: Response res = given()...when().get(...); then assert with res.then().statusCode(200); and extract separately with res.path("id"). Or use SoftAssert with the stored Response object.
Q3: How does as(User.class) work internally? +
REST Assured uses Jackson (if present on classpath) to deserialize the JSON string into a Java object. Jackson maps JSON field names to Java field names by exact name match. You can use @JsonProperty("json_key") annotation if field names differ. Jackson must be on the classpath (jackson-databind dependency in pom.xml).
Q4: When would you use XmlPath over JsonPath? +
Use XmlPath when the API's Content-Type is application/xml or text/xml. Some legacy SOAP/REST APIs return XML. REST Assured automatically detects the content type — if XML, body() assertions use XmlPath. You can also call xmlPath() explicitly on the Response. The dot-notation syntax is identical to JsonPath.
Q5: What is a Groovy closure and how is it used in REST Assured? +
A Groovy closure is an anonymous function block written as {param -> expression}. In REST Assured's GPath/JsonPath: getList("findAll{u -> u.age > 18}.name") filters elements where age > 18 and extracts name. The 'it' shorthand works without param declaration: findAll{it.age > 18}. This enables powerful array filtering in a single expression.
🌐 Overview — Serialization & Deserialization
❓
WHAT
Serialization = Java object → JSON string (for request body). Deserialization = JSON string → Java object (from response body).
💡
WHY
Avoids manual string concatenation for complex request bodies. POJO-based tests are type-safe, readable, and maintainable.
⏰
WHEN
Use for any POST/PUT with complex nested JSON bodies. Use deserialization when you need to work with the response as a typed Java object.
📍
WHERE
In request: .body(javaObject). In response: .as(MyClass.class). REST Assured delegates to Jackson/Gson automatically.
⚙️
HOW
4 ways: HashMap, JSONObject (org.json), POJO + Jackson, or raw JSON string. POJO + Jackson is the industry standard.
🎤 Interview One-Liners
Serialization = converting a Java object into a JSON string to send as HTTP request body — Jackson's ObjectMapper.writeValueAsString().
Deserialization = converting JSON response string back into a Java object — Jackson's ObjectMapper.readValue() or REST Assured's as().
HashMap approach = build request body as Map<String,Object> → REST Assured serializes it to JSON automatically when contentType is JSON.
JSONObject approach = uses org.json library's JSONObject class to construct request body programmatically with put() methods.
POJO approach = most readable — create a Java class mirroring JSON structure, set field values, pass object to body() — Jackson auto-serializes.
@JsonProperty = Jackson annotation to map a JSON key name to a different Java field name — e.g. JSON "first_name" → Java field firstName.

🎯 SDET Interview Q&A
Q1: How does REST Assured know to use Jackson for serialization? +
REST Assured auto-detects Jackson on the classpath via classpath scanning. When you add jackson-databind to pom.xml, REST Assured's ObjectMapperDeserializationContext picks it up. You can also configure explicitly: RestAssured.config = RestAssuredConfig.config().objectMapperConfig(new ObjectMapperConfig(JACKSON)).
Q2: When would you use HashMap vs POJO for request body? +
HashMap: for simple, one-off requests or when the JSON structure is dynamic/unknown. POJO: for recurring request structures, especially in data-driven tests — provides type safety, IDE refactoring support, and readable code. For complex nested JSON (3+ levels), always prefer POJO for maintainability.
Q3: What is @JsonIgnoreProperties(ignoreUnknown = true)? +
A Jackson annotation placed on POJO class. Without it, if the JSON response contains fields NOT in your POJO, Jackson throws UnrecognizedPropertyException. With ignoreUnknown=true, extra fields are silently ignored. Essential for defensive deserialization — APIs often add new fields that break your tests.
Q4: What is the difference between JSONObject (org.json) and Jackson? +
org.json's JSONObject is a simple JSON builder — good for constructing JSON string manually. Jackson is a full object mapping framework — handles POJO serialization/deserialization, custom types, annotations, date formats, and complex transformations. Jackson is industry standard; org.json is simpler but less powerful.
Q5: What happens if a POJO field name doesn't match the JSON key? +
The field stays null (for deserialization) or is omitted (for serialization). Use @JsonProperty("json_key_name") on the field to explicitly specify the mapping. Example: @JsonProperty("first_name") private String firstName; maps JSON "first_name" → Java field firstName.

🏆 Top 20 SDET Interview Questions — REST Assured
Q1: Explain the Given/When/Then pattern in REST Assured +
Given = preconditions (baseURI, headers, auth, body). When = the action (which HTTP method to call). Then = assertions on the response. This is BDD (Behavior Driven Development) style — readable by non-developers. The fluent API chains all three: given().header().when().get().then().statusCode(200).
Q2: How do you run REST Assured tests in parallel? +
Configure TestNG XML with parallel="methods" and thread-count. Ensure no shared mutable static state between tests. Use ThreadLocal<> for per-thread objects. REST Assured's RequestSpecification is not thread-safe — create new instances per test, don't share a single spec across threads.
Q3: How do you handle OAuth2 token refresh in a long test suite? +
In @BeforeSuite, call login endpoint and store token + expiry time in a static field. Before each test (@BeforeMethod), check if token is expired; if so, call login again. Use synchronized keyword if running parallel. Better: use a TokenManager utility class that lazily refreshes the token when expired.
Q4: What is ResponseSpecification in REST Assured? +
ResponseSpecification is the counterpart of RequestSpecification — it bundles common response assertions. Built with ResponseSpecBuilder: new ResponseSpecBuilder().expectStatusCode(200).expectContentType("application/json").build(). Used in then(spec) to apply common assertions to multiple tests without repeating them.
Q5: How do you validate JSON schema in REST Assured? +
Use rest-assured's JSON Schema validation module: add io.rest-assured:json-schema-validator dependency. Then: then().body(matchesJsonSchemaInClasspath("user-schema.json")). The schema file defines expected types, required fields, and value constraints. This validates the API contract structurally, not just individual values.
