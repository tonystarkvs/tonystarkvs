🌐 Overview — What / Why / When / Where / How
❓
WHAT
Serialization = converting a Java object/map into a JSON string to send as the HTTP request body in REST Assured.
💡
WHY
Avoids error-prone string concatenation for JSON. Type-safe, readable, and maintainable for any level of nesting complexity.
⏰
WHEN
Every POST/PUT/PATCH request that sends a JSON body — from a simple flat object to deeply nested structures with arrays.
📍
WHERE
Inside REST Assured's given().body(). Serialization happens before the HTTP call; deserialization happens after response arrives.
⚙️
HOW
5 approaches: raw String, HashMap, JSONObject (org.json), POJO+Jackson, POJO+Gson. Each covered with simple→complex examples.

🎤 Interview One-Liners
Serialization = the process of converting a Java object (Map/POJO) into a JSON string so it can be sent as an HTTP request body.
Deserialization = the reverse — parsing a JSON response string back into a typed Java object using Jackson or Gson.
HashMap approach = using Map<String,Object> as request body — REST Assured + Jackson auto-converts it to JSON when Content-Type is application/json.
Jackson ObjectMapper = the core class in the Jackson library that handles both serialization (writeValueAsString) and deserialization (readValue).
Content-Type header = must be set to application/json in REST Assured for auto-serialization to trigger — without it, the Map is sent as a form param string.
Nested JSON = a JSON object that contains another JSON object or array as a value — represented in Java as a Map inside a Map, or a POJO with a nested POJO field.

✅ Best Practices
✔ Always set contentType("application/json") — without it REST Assured won't trigger Jackson auto-serialization for Maps/POJOs.
✔ Use Map<String,Object> (not raw Map) — the type parameter prevents ClassCastException at runtime.
✔ Never use raw String JSON for complex bodies — switch to HashMap or POJO for anything beyond 3 fields.
✔ Use log().body() during development to print the exact JSON being sent — verify it matches your intent.
✔ Prefer Arrays.asList() for fixed lists and new ArrayList<>() for mutable lists that need add/remove.
🎯 SDET Interview Q&A
Q1: Why does contentType("application/json") matter for HashMap serialization? +
Without it, REST Assured doesn't know you want JSON — it may send the Map as URL-encoded form parameters (key=value&key2=value2) instead. Setting contentType("application/json") signals REST Assured to use Jackson's ObjectMapper to serialize the Map into a proper JSON string for the request body.
Q2: How do you represent a JSON array of objects using HashMap? +
Create each object as a Map<String,Object>. Add all objects to a List<Map<String,Object>>. Then put the list as a value in the parent map: parentMap.put("items", listOfMaps). Jackson serializes List<Map> as a JSON array of objects: "items":[{...},{...}].
Q3: What is the difference between Arrays.asList() and new ArrayList()? +
Arrays.asList() returns a fixed-size List backed by the array — you cannot add/remove elements (throws UnsupportedOperationException). Use it only when the list is complete and won't change. new ArrayList<>() is mutable — you can add, remove, sort. For JSON array building in tests, use ArrayList when items are added dynamically, Arrays.asList() for hardcoded lists.
Q4: What serialization library does REST Assured use by default? +
REST Assured auto-detects the library on the classpath in priority order: Jackson (jackson-databind) first, then Gson, then JAXB (for XML). If Jackson is on the classpath, all Map/POJO body serialization uses Jackson's ObjectMapper automatically. You can configure it explicitly with RestAssured.config().objectMapperConfig().
Q5: How do you verify the exact JSON being sent in a REST Assured request? +
Add .log().body() in the given() phase — it prints the serialized request body to console before sending. For full request including headers: .log().all(). For production CI: .log().ifValidationFails() prints only when assertions fail, keeping logs clean.
🌐 Overview — JSONObject (org.json) & Gson Serialization
❓
WHAT
org.json's JSONObject/JSONArray = explicit JSON builders. Gson = Google's library that serializes Java POJOs to JSON using GsonBuilder and custom serializers.
💡
WHY
JSONObject gives explicit JSON construction with a clean put() API. Gson is lightweight and handles nested POJOs automatically without many annotations.
⏰
WHEN
Use JSONObject when you prefer explicit JSON construction. Use Gson when your team uses it as the company standard or when Jackson is not available.
📍
WHERE
JSONObject in given().body(obj.toString()). Gson: new Gson().toJson(pojo) → then pass that String to body(). Or configure REST Assured to use Gson globally.
⚙️
HOW
org.json: new JSONObject().put(k,v).put(k,v). Gson: new Gson().toJson(object). GsonBuilder for custom formatting and null handling.
🎤 Interview One-Liners
JSONObject = class from org.json library that builds JSON objects programmatically using put(key,value) — call toString() to get the JSON string.
JSONArray = class from org.json for building JSON arrays — call put(value) to add elements, and toString() for the JSON string.
Gson = Google's open-source Java library for JSON serialization/deserialization — simpler than Jackson with less annotation overhead.
toJson() = Gson method that converts a Java object (POJO/Map) into a JSON string — equivalent to Jackson's writeValueAsString().
fromJson() = Gson method that converts a JSON string into a Java object — equivalent to Jackson's readValue().
GsonBuilder = Gson's builder class for configuring serialization options: setPrettyPrinting(), serializeNulls(), setDateFormat(), excludeFieldsWithoutExposeAnnotation().
💡 Concept — JSONObject vs JSONArray vs Gson
🔧 org.json — Construction Pattern
JSONObject = { } | JSONArray = [ ] ─────────────────────────────────────────────────── JSONObject obj = new JSONObject(); obj.put("name", "Chitti"); ← adds String field obj.put("age", 25); ← adds int field obj.put("active", true); ← adds boolean field JSONArray arr = new JSONArray(); arr.put("admin"); ← adds string element arr.put("viewer"); obj.put("roles", arr); ← embed array in object JSONObject nested = new JSONObject(); nested.put("city", "Delhi"); obj.put("address", nested); ← embed object in object obj.toString() → {"name":"Chitti","age":25,"active":true, "roles":["admin","viewer"], "address":{"city":"Delhi"}}
☕ Gson — POJO to JSON
Gson gson = new Gson(); ─────────────────────────────────────────────────── User user = new User("Chitti", "c@test.com"); String json = gson.toJson(user); → {"name":"Chitti","email":"c@test.com"} User back = gson.fromJson(json, User.class); → Reconstructed User POJO from JSON string GsonBuilder for custom options: Gson prettyGson = new GsonBuilder() .setPrettyPrinting() ← formatted output (4-space indent) .serializeNulls() ← include null fields in JSON .setDateFormat("yyyy-MM-dd") ← format Date fields .create(); Gson vs Jackson key differences: Gson: no annotations needed for basic mapping Jackson: @JsonProperty, @JsonIgnore, @JsonInclude etc.

✅ Best Practices
✔ Always call body(jsonObj.toString()) explicitly for JSONObject — don't pass JSONObject directly, REST Assured behavior varies by version.
✔ Create one Gson instance per test class (or globally) — Gson is thread-safe after construction, so reuse is safe and efficient.
✔ Use @SerializedName("json_key") when Java field names use camelCase but JSON keys use snake_case (e.g., productName → "product_name").
✔ Use GsonBuilder().serializeNulls() when the API requires null fields to be present — by default Gson omits all null fields.
✔ Prefer Jackson over Gson for REST Assured projects — REST Assured auto-detects Jackson; Gson requires manual toJson() calls.
🎯 SDET Interview Q&A
Q1: What is the difference between JSONObject (org.json) and HashMap? +
Both build JSON objects but differently. HashMap is Java's built-in key-value store — REST Assured+Jackson auto-converts it. JSONObject is org.json's explicit JSON builder — you call toString() to get the JSON string. JSONObject preserves insertion order; HashMap does not (LinkedHashMap does). JSONObject provides getInt(), getString() helpers; HashMap does not.
Q2: How do you configure REST Assured to use Gson instead of Jackson? +
Configure globally: RestAssured.config = RestAssuredConfig.config().objectMapperConfig(new ObjectMapperConfig(GSON)). Or use GsonObjectMapperFactory. But the simplest approach is to manually call gson.toJson(pojo) and pass the String to body() — this always works regardless of REST Assured's serializer config.
Q3: How does @SerializedName work in Gson? +
@SerializedName("json_key") is placed on a POJO field and tells Gson to use that name when serializing to JSON and deserializing from JSON. Example: @SerializedName("first_name") String firstName — Gson serializes as "first_name":"value" and deserializes "first_name" from JSON into firstName field. Equivalent to Jackson's @JsonProperty.
Q4: What does serializeNulls() do in GsonBuilder? +
By default, Gson omits fields with null values from the JSON output — if address is null, no "address" key appears. GsonBuilder().serializeNulls() overrides this and includes null fields as "address": null. Use when the API's schema validation requires all fields to be present even if their value is null.
Q5: When would you choose Gson over Jackson? +
Gson when: (1) existing project already uses Gson, (2) you need simpler code without many annotations, (3) Android projects (Gson is lighter). Jackson when: (1) new REST Assured project, (2) need rich annotations like @JsonIgnore, @JsonInclude, @JsonCreator, (3) need advanced features like polymorphic deserialization, (4) better REST Assured integration (auto-detected).

✅ Best Practices
✔ Always add @JsonIgnoreProperties(ignoreUnknown = true) to every POJO — APIs evolve and add fields; this prevents test failures from unknown fields.
✔ Use @JsonInclude(NON_NULL) to keep serialized JSON clean — avoids sending null fields that the API might reject as invalid.
✔ Always provide a no-arg constructor in POJO — Jackson needs it for deserialization (even if you have a parameterized constructor).
✔ Use TypeRef<List<MyClass>> (not just List.class) when deserializing JSON arrays — plain List.class loses generic type due to Java type erasure.
✔ Use ObjectMapper.writerWithDefaultPrettyPrinter() during debugging to get formatted JSON and confirm the exact payload being sent.
🎯 SDET Interview Q&A
Q1: Why does Jackson need a no-arg constructor in POJO? +
Jackson's default deserialization (readValue) uses reflection to create an instance with no-arg constructor, then uses setters/field access to populate fields from JSON. Without a no-arg constructor, Jackson throws InvalidDefinitionException. You can work around it with @JsonCreator on a parameterized constructor, but then you must annotate every parameter with @JsonProperty.
Q2: What is Java type erasure and how does TypeRef solve it? +
Type erasure means generic types (List<User>) lose their type parameter at runtime — the JVM only sees raw List. So List<User>.class doesn't exist. TypeRef (or Jackson's TypeReference) creates an anonymous subclass that preserves the generic type via reflection on the class hierarchy. REST Assured uses this to tell Jackson to deserialize into List<User> specifically.
Q3: How do @JsonIgnore and @JsonIgnoreProperties differ? +
@JsonIgnore is field-level — marks a single POJO field to be skipped in both serialization and deserialization. @JsonIgnoreProperties is class-level — when set with ignoreUnknown=true, it tells Jackson to silently skip JSON fields that have no matching POJO field (defensive deserialization). These solve different problems: @JsonIgnore controls what you send; @JsonIgnoreProperties controls what you accept.
Q4: What is the difference between ObjectNode and JSONObject? +
Both build JSON objects dynamically, but: ObjectNode is from Jackson (com.fasterxml.jackson), ObjectNode is part of the Jackson tree model. JSONObject is from org.json library (separate dependency). ObjectNode requires ObjectMapper to create: om.createObjectNode(). ObjectNode integrates seamlessly with Jackson's serialization pipeline. If you're already using Jackson, prefer ObjectNode; it avoids mixing libraries.
Q5: How does @JsonAlias differ from @JsonProperty for deserialization? +
@JsonProperty("key") sets a single mapping — used for BOTH serialization (output) and deserialization (input). @JsonAlias({"key1","key2"}) adds ALTERNATIVE names for DESERIALIZATION only — the field is still serialized with its primary name. Use @JsonAlias when an API is migrating field names and temporarily sends either "name" or "full_name" for the same field.

✅ Best Practices
✔ Always build complex JSON bottom-up — deepest nested object first, then wrap outward to root — prevents NullPointerException during construction.
✔ For 3+ levels of nesting, use POJO+Jackson — HashMap code becomes unreadable and error-prone at this depth.
✔ Use ObjectNode/ArrayNode when the JSON structure is dynamic (determined at runtime) — avoids creating rigid POJO classes for variable schemas.
✔ Verify deep response paths with both .body("a.b.c", equalTo(...)) assertions AND jsonPath().getString("a.b.c") extraction — assert AND extract separately when needed.
✔ For arrays of objects in response: use items[0].field for specific index access, items.field to collect all values, items.size() for array length.
🎯 SDET Interview Q&A
Q1: How do you navigate a 4-level deep JSON path in REST Assured assertions? +
Use dot notation in body() path: body("customer.address.geo.lat", equalTo(28.6)). Each dot navigates one level deeper. For arrays: body("items[0].tags[0]", equalTo("java")) — [0] accesses array element by index. REST Assured evaluates this as a GPath expression using Groovy's navigation engine.
Q2: What is the difference between ObjectNode and POJO for building JSON? +
POJO is better for fixed/known structures — type-safe, IDE support, reusable across tests. ObjectNode (Jackson tree model) is better for dynamic structures where keys/values change at runtime, or when you need to modify the tree after building it. Both are part of Jackson; both integrate seamlessly with REST Assured body(). ObjectNode avoids creating extra Java classes for one-off structures.
Q3: How do you assert a specific element inside a nested array using REST Assured? +
Use array indexing in GPath: body("items[0].productId", equalTo(101)) accesses the first element's productId. For "any element matches": body("items.productId", hasItem(101)). For "all elements": body("items.productId", everyItem(greaterThan(0))). For "filtered": body("items.findAll{it.qty>1}.productId", hasItem(101)).
