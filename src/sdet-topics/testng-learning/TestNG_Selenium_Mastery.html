S1 - TestNG Basics, TestNG vs JUnit, Features, Architecture & Reports

TestNG (Test-Next-Generation) is a Java testing framework built for unit, integration, and end-to-end Selenium tests. Compared to JUnit 4 it ships with grouping, dependencies, parameterization, parallel runs, and rich HTML reports out of the box — making it the default choice for Selenium automation.

What / Why / When / Where / How

WhatOpen-source Java testing framework inspired by JUnit and NUnit.
WhyBuilt-in parallelism, data providers, dependencies, groups, rich reports.
WhenWhenever you write Java tests — UI (Selenium), API, or unit.
WhereMaven/Gradle project; tests under src/test/java; suite controlled by testng.xml.
HowAdd testng dependency → annotate @Test → run via Maven or testng.xml.

Keyword Glossary

TestNG
Annotation-driven Java testing framework.
JUnit
Older Java unit-test framework; JUnit 5 is closer but TestNG leads on suite features.
@Test
Marks a method as a test case.
Suite
Top-level container in testng.xml.
testng.xml
XML file controlling which tests run, in what order, in parallel.
ITestResult
Listener-side object exposing test name/status/throwable.
surefire-reports
Maven folder where TestNG drops HTML reports.
Reporter
TestNG class for custom log lines in HTML report.
@BeforeSuite
Runs once before any test in the suite.
@AfterSuite
Runs once after all tests.

Interview One-Liners

TestNG is annotation-driven; JUnit 4 used inheritance.
Suite > Test > Class > Method — that's the hierarchy.
testng.xml is your single source of truth for which tests run.
TestNG ships native parallel execution; JUnit needs external runners.
Use @BeforeSuite for driver setup so cost is paid once per run.
Reports land in target/surefire-reports/.

Interview Prep Q&A

Q1. TestNG vs JUnit 4?
A: TestNG offers built-in groups, dependencies, native parallelism, DataProvider, and richer suite control.
Q2. Why testng.xml?
A: It's the suite descriptor — defines which tests run, in what order, with what params.
Q3. Where does TestNG drop reports?
A: target/surefire-reports/ → emailable-report.html and index.html.
Q4. Order of TestNG annotations?
A: BeforeSuite > BeforeTest > BeforeClass > BeforeMethod > @Test > AfterMethod > AfterClass > AfterTest > AfterSuite.

A: Yes — @Test(enabled=false), or exclude it in testng.xml.

S2 - Annotations & Execution Flow with Multiple Test Classes

TestNG annotations control when a method runs in the test lifecycle. The same annotations behave differently at suite, test, class, and method scopes. Mastering execution flow across multiple test classes is critical when you build a real Selenium framework with shared driver setup.

What / Why / When / Where / How

WhatJava annotations marking hooks (@BeforeX/@AfterX) and tests (@Test).
WhySetup once, teardown once — keep tests DRY and fast.
WhenEvery framework needs at least @BeforeMethod (driver) and @AfterMethod (quit).
WhereTest classes; also a BaseTest parent class that real tests extend.
HowAnnotate methods; TestNG runs them in a guaranteed order.

Keyword Glossary

@BeforeSuite
Runs once before any test in the suite.
@AfterSuite
Runs once after all tests in the suite.
@BeforeTest
Runs before all tests inside a <test> tag.
@BeforeClass
Runs once before the first @Test of that class.
@BeforeMethod
Runs before each @Test method.
@AfterMethod
Runs after each @Test method.
@AfterClass
Runs after the last @Test of that class.
@AfterTest
Runs after all tests inside a <test> tag.
BaseTest
Convention: parent class holding hooks; children extend it.
alwaysRun
Attribute that runs hook even if dependsOnMethods skipped.

Interview One-Liners

Annotations + inheritance = clean Selenium framework architecture.
Hooks run outermost-in on entry, reverse on exit.
Use @BeforeSuite for grid setup; @AfterSuite for final cleanup.
A BaseTest with @BeforeMethod driver setup is the standard pattern.
alwaysRun=true keeps @AfterMethod running even if dependsOn failed.
Each @Test gets a fresh driver if setup is in @BeforeMethod.

Best Practices Checklist

  • ✅ Always create a BaseTest with @BeforeMethod driver setup.
  • ✅ Use alwaysRun=true on teardown so cleanup runs even after failures.
  • ✅ Don't put assertions in hooks — they hide test status.
  • ✅ Order classes inside <test> deliberately when state leaks.
  • ✅ Reserve @BeforeSuite for expensive global setup only.
  • ✅ Avoid static drivers — use ThreadLocal for parallel safety.
  • ✅ Log hook entries with Reporter.log to debug timing issues.
  • ✅ Name hooks by scope: suiteUp, classUp, methodUp, methodDown.
  • ✅ Don't share state between methods via instance vars unless intentional.
  • ✅ Use description= attribute — it shows up in the HTML report.

S3 - Priorities, Test Execution Control, Include/Exclude & testng-failed.xml

By default TestNG sorts @Test methods alphabetically. Use priority= to force a deterministic order. Use include/exclude tags in testng.xml to slice the suite. After every run TestNG drops a testng-failed.xml so you can rerun only the failed tests — a huge time saver in CI.

What / Why / When / Where / How

WhatMechanisms to control which tests run, in what order, and how to rerun failures.
WhyForce login before checkout; pick smoke vs regression; rerun only failed tests.
WhenWhenever default alphabetical order would break your scenario.
Wherepriority= on @Test; <include>/<exclude> in testng.xml; auto-generated testng-failed.xml.
HowLower priority runs first; default is 0; tie-break is alphabetical.

Keyword Glossary

priority
Integer attribute on @Test; lower runs first; default 0.
alphabetical
Default order when priorities tie or are absent.
<include>
testng.xml tag listing method names that MUST run.
<exclude>
testng.xml tag listing method names that MUST NOT run.
testng-failed.xml
Auto-generated suite holding only failed/skipped tests.
preserve-order
Suite attribute that runs classes in listed order.
enabled
@Test attribute — false skips this method entirely.
timeOut
@Test attribute capping max run time in milliseconds.
invocationCount
Runs a @Test N times — useful for flake hunting.
successPercentage
Allow N% of invocations to fail and still pass overall.

Interview One-Liners

Priority is just a tie-breaker; alphabetical kicks in when priorities are equal.
testng-failed.xml is your one-click rerun — point your CI at it.
Include/exclude lets you slice tests without touching code.
invocationCount + threadPoolSize = quick mini load test.
Never rely on default alphabetical order — set priorities explicitly when sequence matters.
timeOut catches a hung test before it eats your whole pipeline.

S4 - Dependencies — dependsOnMethods, dependsOnGroups, alwaysRun

Real flows have ordering: login must succeed before checkout. TestNG's dependsOnMethods and dependsOnGroups attributes encode that prerequisite chain. When a dependency fails, downstream tests are SKIPPED (not failed) — keeping reports clean and honest.

What / Why / When / Where / How

WhatAttributes declaring prerequisite tests/groups before a @Test can run.
WhyEncode real-world flow (login → search → checkout) and auto-skip downstream on failure.
WhenWhenever a test depends on state set up by another.
WheredependsOnMethods / dependsOnGroups on @Test.
HowList names; TestNG enforces order and skips dependents if prereq fails.

Keyword Glossary

dependsOnMethods
Comma-separated method names that must pass before this @Test runs.
dependsOnGroups
Group names whose tests must pass before this @Test runs.
SKIP
Outcome when a dependency fails — neither PASS nor FAIL.
alwaysRun
Forces a method to run even if its dependencies failed.
Dependency Tree
Graph TestNG builds to determine run order from your declarations.
@Test ordering
priority < dependsOnMethods (dependencies override priority).
Soft Dependency
Test runs even if dep failed — using alwaysRun=true.
Hard Dependency
Default behaviour — dep fails → dependent skipped.
Group Dependency
Whole group must pass; perfect for layered flows.
Cycle
Cyclic dependencies cause TestNG to throw a config error.

Interview One-Liners

dependsOnMethods turns a chain of tests into a guaranteed sequence.
If the prereq fails, dependents are SKIPPED — not failed.
Group dependencies are more flexible than method names.
alwaysRun=true overrides the default skip-on-prereq-fail behaviour.
Avoid deep dependency chains — they make reports hard to read.
Dependencies override priority — order is decided by the dep graph first.

S6 - Assertions — Hard & Soft Assertions

Hard assertion (Assert.*) stops the test on first failure. Soft assertion (SoftAssert) collects all failures and reports them at the end via assertAll(). Use hard for blockers (login worked?) and soft for collecting many UI checks.

What / Why / When / Where / How

WhatStatements that verify expected vs actual; fail the test if mismatched.
WhyWithout assertions a test only proves the code didn't throw — not that it's correct.
WhenAfter every meaningful action in a test.
WhereInside @Test methods, helper classes, custom assertions.
Howorg.testng.Assert.* for hard; org.testng.asserts.SoftAssert for soft (with assertAll).

Keyword Glossary

Assert.assertEquals
Strict equality; throws AssertionError on mismatch.
Assert.assertTrue
Asserts boolean expression is true.
Assert.assertFalse
Asserts boolean expression is false.
Assert.assertNotNull
Asserts value is not null.
Assert.fail
Forces test to fail with a message.
SoftAssert
TestNG class that accumulates failures.
assertAll
Call at end of soft block to throw collected failures.
Custom Message
Optional last arg explaining the failure context.
AssertionError
Exception thrown by failed hard assertions.
Reporter
TestNG class for adding HTML report log lines.

Interview One-Liners

Hard assertion stops on first failure; soft collects all and throws at end.
Always include a meaningful message — future-you will thank you.
Never use try/catch around assertions to swallow them.
Use assertEquals(actual, expected) — order matters in messages.
SoftAssert without assertAll = silent test (looks passed).
Mix wisely: hard for prereqs, soft for batched UI checks.
