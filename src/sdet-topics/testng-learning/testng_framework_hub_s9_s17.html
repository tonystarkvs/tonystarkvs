Overview — Framework Structure

A TestNG+Selenium framework is a structured Maven project where every concern lives in its own layer — page objects, tests, configs, utilities, and reports — all wired together by testng.xml and POM.xml.

What
Layered Maven project: Pages / Tests / Config / Utils / Reports
Why
Maintainable, reusable, scalable — one test class per feature, not one massive file
How
POM.xml manages dependencies; testng.xml drives suite; Maven Surefire runs it all

Interview Q&A
Q1: What is the Page Object Model and why do we use it? ▼
POM is a design pattern where each web page has a corresponding Java class with locators and methods. It separates UI structure from test logic — when a locator changes, you update only one class, not every test that uses that element.
Q2: Why is DriverFactory a separate class from BaseTest? ▼
Separation of concerns. BaseTest handles the test lifecycle (setup/teardown annotations). DriverFactory handles browser instantiation logic. This means you can change browser creation (add headless, add options, add remote) without touching BaseTest.
Q3: How do you read environment-specific config in your framework? ▼
ConfigReader loads the right .properties file based on an environment parameter passed via testng.xml or Maven -Denv=uat. It uses Properties.load() with FileInputStream, and a singleton pattern so the file is read only once per run.
Q4: Where do you keep test data in your framework? ▼
Static config (URLs, timeouts, credentials) goes in .properties files. Dynamic data sets go in Excel files under src/test/resources/testdata/ and are read by a DataProvider method using Apache POI. Inline test data goes directly in the DataProvider's Object[][].
Q5: How does TestNG integrate with Maven in your framework? ▼
The maven-surefire-plugin in pom.xml points to the testng.xml file via suiteXmlFiles. Running mvn clean test triggers Surefire, which executes TestNG, which reads the XML, creates threads, and runs tests. The -P profile flag selects different pom.xml profiles for different environments.
Q6: What is the difference between src/main/java and src/test/java in your framework? ▼
src/main/java holds the reusable framework components — BaseTest, page objects, utilities, listeners — that are not themselves test cases. src/test/java holds the actual test classes (@Test methods). This separation means the framework code can be packaged as a library and shared across projects if needed.

Code — DriverFactory.java
DriverFactory.java — creates and stores WebDriver per threadCore
package com.framework.utils;

import org.openqa.selenium.WebDriver;
import org.openqa.selenium.chrome.ChromeDriver;
import org.openqa.selenium.chrome.ChromeOptions;
import org.openqa.selenium.firefox.FirefoxDriver;
import org.openqa.selenium.firefox.FirefoxOptions;
import org.openqa.selenium.edge.EdgeDriver;
import org.openqa.selenium.safari.SafariDriver;

public class DriverFactory {

    // ThreadLocal = each thread gets its OWN WebDriver instance
    private static ThreadLocal<WebDriver> driverThread =
        new ThreadLocal<>();

    // Factory method — call this to create the right browser
    public static void initDriver(String browser, boolean headless) {
        WebDriver driver;

        switch (browser.toLowerCase().trim()) {
            case "chrome":
                ChromeOptions co = new ChromeOptions();
                if (headless) co.addArguments("--headless=new");
                co.addArguments("--no-sandbox", "--disable-dev-shm-usage");
                driver = new ChromeDriver(co);  // WebDriverManager handles driver
                break;

            case "firefox":
                FirefoxOptions fo = new FirefoxOptions();
                if (headless) fo.addArguments("-headless");
                driver = new FirefoxDriver(fo);
                break;

            case "edge":
                driver = new EdgeDriver();
                break;

            case "safari":
                driver = new SafariDriver();  // Mac only, no headless
                break;

            default:
                throw new IllegalArgumentException(
                    "Browser not supported: " + browser);
        }

        driver.manage().window().maximize();
        driverThread.set(driver);  // store in THIS thread's slot
    }

    // Getter — used by page objects and test classes
    public static WebDriver getDriver() {
        return driverThread.get();
    }

    // Cleanup — MUST be called in @AfterMethod
    public static void quitDriver() {
        if (driverThread.get() != null) {
            driverThread.get().quit();
            driverThread.remove(); // prevents memory leak in thread pools
        }
    }
}
BaseTest.java — test lifecycle with @Before/@AfterLifecycle
package com.framework.base;

import com.framework.utils.DriverFactory;
import com.framework.utils.ConfigReader;
import org.testng.annotations.*;

public class BaseTest {

    // Reads env param from testng.xml <parameter name="env" value="uat"/>
    @Parameters({"browser", "headless", "env"})
    @BeforeMethod(alwaysRun = true)
    public void setUp(
            @Optional("chrome")  String browser,
            @Optional("false")  String headless,
            @Optional("dev")    String env) {

        ConfigReader.init(env);   // load dev/uat/prod.properties
        DriverFactory.initDriver(browser, Boolean.parseBoolean(headless));
        DriverFactory.getDriver().get(ConfigReader.get("base.url"));
    }

    // Convenience getter for test classes: getDriver().findElement(...);
    protected org.openqa.selenium.WebDriver getDriver() {
        return DriverFactory.getDriver();
    }

    @AfterMethod(alwaysRun = true)  // alwaysRun ensures cleanup on failure
    public void tearDown() {
        DriverFactory.quitDriver();
    }
}
BasePage.java — reusable Selenium helpers for all page objectsPages
package com.framework.base;

import org.openqa.selenium.*;
import org.openqa.selenium.support.ui.*;
import java.time.Duration;

public class BasePage {

    protected WebDriver driver;
    protected WebDriverWait wait;

    // Constructor injection — page objects call super(driver)
    public BasePage(WebDriver driver) {
        this.driver = driver;
        this.wait = new WebDriverWait(driver, Duration.ofSeconds(15));
    }

    // Wait until element is clickable, then click
    protected void click(By locator) {
        wait.until(ExpectedConditions.elementToBeClickable(locator)).click();
    }

    // Clear and type text into an input field
    protected void type(By locator, String text) {
        WebElement el = wait.until(
            ExpectedConditions.visibilityOfElementLocated(locator));
        el.clear();
        el.sendKeys(text);
    }

    // Get visible text of an element
    protected String getText(By locator) {
        return wait.until(
            ExpectedConditions.visibilityOfElementLocated(locator)).getText();
    }

    // Check if element is displayed (without throwing exception)
    protected boolean isDisplayed(By locator) {
        try {
            return driver.findElement(locator).isDisplayed();
        } catch (NoSuchElementException e) {
            return false;
        }
    }
}
LoginPage.java — example Page Object extending BasePagePOM
package com.framework.pages;

import com.framework.base.BasePage;
import org.openqa.selenium.By;
import org.openqa.selenium.WebDriver;

public class LoginPage extends BasePage {

    // All locators are private final — they never change after construction
    private final By usernameField = By.id("username");
    private final By passwordField = By.id("password");
    private final By loginButton   = By.cssSelector("button[type='submit']");
    private final By errorMessage  = By.cssSelector(".alert-danger");

    public LoginPage(WebDriver driver) {
        super(driver);  // passes driver to BasePage constructor
    }

    // Action methods — return Page Object for fluent chaining
    public LoginPage enterUsername(String user)  { type(usernameField, user); return this; }
    public LoginPage enterPassword(String pass)  { type(passwordField, pass); return this; }
    public HomePage  clickLogin()               { click(loginButton); return new HomePage(driver); }
    public String   getErrorText()              { return getText(errorMessage); }
}
LoginTest.java — test class using BaseTest + LoginPageTest
package com.tests;

import com.framework.base.BaseTest;
import com.framework.pages.LoginPage;
import com.framework.utils.DriverFactory;
import org.testng.Assert;
import org.testng.annotations.*;

public class LoginTest extends BaseTest {  // inherits setUp/tearDown

    @Test(groups = "smoke", description = "Valid user can log in")
    public void validLoginTest() {
        LoginPage login = new LoginPage(getDriver());
        HomePage home = login
            .enterUsername("admin")
            .enterPassword("admin123")
            .clickLogin();               // fluent chaining
        Assert.assertTrue(home.isWelcomeDisplayed(), "Welcome not shown");
    }

    @Test(groups = "regression")
    public void invalidLoginTest() {
        LoginPage login = new LoginPage(getDriver());
        login.enterUsername("wrong")
             .enterPassword("wrong");
        login.clickLoginExpectingError();
        Assert.assertEquals(login.getErrorText(), "Invalid credentials");
    }
}
Best Practices
✓Use constructor injection for WebDriver in page objects — never use DriverFactory.getDriver() inside a page class.
✓Return this or the next page object from action methods — enables fluent API chaining.
✓Always use WebDriverWait in BasePage — ban Thread.sleep from the entire framework with a code review rule.
✓Mark @BeforeMethod and @AfterMethod with alwaysRun=true so teardown never gets skipped.
✓Declare all locators as private final By at the top of each page class — never scatter locators in methods.
Interview Q&A
Q1: Why do page objects receive WebDriver in the constructor rather than getting it from DriverFactory? ▼
Constructor injection makes page objects decoupled from DriverFactory — you can unit test them, mock the driver, or reuse the same class with different drivers. Getting from DriverFactory directly creates a hidden dependency and breaks the Single Responsibility Principle.
Q2: What is the purpose of BasePage? ▼
BasePage centralises all common Selenium operations — click, type, getText, waitForElement — so every page object inherits them without duplicating code. When wait strategy needs to change from 15s to 30s, you change it in one place.
Q3: How does ThreadLocal protect WebDriver in parallel runs? ▼
ThreadLocal creates a separate storage slot per thread. Thread-1 stores its ChromeDriver in slot-1, Thread-2 stores its FirefoxDriver in slot-2. getDriver() always returns the calling thread's own driver — no cross-thread contamination.
Q4: What happens if you forget driverThread.remove() in teardown? ▼
In parallel execution with a thread pool, old threads get recycled. If the previous test's driver was not removed, the next test that reuses that thread will get a stale, already-quit WebDriver — causing NoSuchSessionException. Always call remove().
Q5: What is the @Optional annotation used for in BaseTest? ▼
@Optional provides a default parameter value when the testng.xml <parameter> tag is missing. For example, @Optional("chrome") means the test defaults to Chrome even when run without a testng.xml specifying a browser — useful for local IDE runs.
Q6: Should test classes extend BaseTest or contain it as a field? ▼
Both work but extending (inheritance) is the standard convention in TestNG frameworks — it makes @Before/@After methods from BaseTest automatically apply to child classes. Use composition when you want to mix multiple bases or when the framework is shared across teams as a library.

✓Always define versions in <properties>, never inline — makes upgrades a single-line change.
✓Use Maven Surefire 3.x with TestNG 7.x — old Surefire 2.x has XML parsing bugs with TestNG's new DTD.

Interview Q&A
Q1: What is the role of maven-surefire-plugin in a TestNG framework? ▼
Surefire is the bridge between Maven and TestNG. It picks up the testng.xml file from suiteXmlFiles, sets system properties, configures thread counts, and launches TestNG. Without it, mvn test doesn't know how to execute TestNG suites.
Q2: How do Maven profiles enable multi-environment testing? ▼
Each profile overrides the <env> property and <suiteFile> path. Running mvn test -Puat activates the uat profile, which sets env=uat and suiteFile=testng-uat.xml. ConfigReader picks up uat.properties and tests run against the UAT environment.
Q3: What does WebDriverManager do and why is it better than manual driver setup? ▼
WebDriverManager automatically downloads the correct browser driver binary matching the installed browser version and places it in the path. Without it, you'd manually download chromedriver, check it into git, and update it every time Chrome updates — a maintenance nightmare.
Q4: Why do we need the aspectjweaver javaagent for Allure? ▼
Allure uses AspectJ to intercept method calls annotated with @Step and @Attachment at bytecode level. The javaagent must be loaded before the JVM starts executing test code. Surefire's argLine is the correct place to set it — without it, Allure generates empty reports.
Q5: How do you pass -Dbrowser=firefox to tests via Maven? ▼
The Surefire plugin's <systemPropertyVariables> block maps Maven property ${browser} to a JVM system property. When you run mvn test -Dbrowser=firefox, Maven sets the ${browser} property, Surefire injects it as a system property, and TestNG reads it via @Parameters or System.getProperty("browser").
Q6: What is the default profile and how does it activate? ▼
The dev profile has <activeByDefault>true</activeByDefault> so it activates when no -P flag is given. Running plain mvn test uses the dev profile, which loads testng.xml and dev.properties. Only one <activeByDefault> profile should exist per pom.xml.

ConfigReader.java — singleton properties loaderUtility
package com.framework.utils;

import java.io.*;
import java.util.Properties;

public class ConfigReader {

    private static Properties props;

    // Called once in BaseTest.setUp() with the env name
    public static synchronized void init(String env) {
        if (props != null) return;  // singleton: load only once

        props = new Properties();
        String fileName = env.toLowerCase() + ".properties";  // e.g. uat.properties

        try (InputStream is =
                ConfigReader.class.getClassLoader()
                    .getResourceAsStream(fileName)) {
            if (is == null)
                throw new RuntimeException(
                    "Config file not found: " + fileName);
            props.load(is);
        } catch (IOException e) {
            throw new RuntimeException("Failed loading config: " + fileName, e);
        }
    }

    // Get a required property — throws if missing
    public static String get(String key) {
        String val = props.getProperty(key);
        if (val == null)
            throw new RuntimeException("Missing config key: " + key);
        return val.trim();
    }

    // Get property with a fallback default value
    public static String get(String key, String defaultVal) {
        return props.getProperty(key, defaultVal).trim();
    }

    // Convenience int getter for timeouts
    public static int getInt(String key) {
        return Integer.parseInt(get(key));
    }
}

Interview Q&A
Q1: Why use a singleton for ConfigReader? ▼
A singleton loads the properties file only once per JVM run, avoiding repeated file I/O. With parallel tests, multiple threads might call init() simultaneously — the synchronized keyword ensures only one thread loads the file while others wait.
Q2: How do you handle passwords securely in a test framework? ▼
Store them as CI environment variables (Jenkins Credentials Binding, GitHub Secrets, Azure Key Vault). In the .properties file, use a placeholder like app.password=${APP_PASSWORD}. At runtime, the CI system injects the real value. Never commit real credentials to source control.
Q3: What is the difference between getResourceAsStream and FileInputStream for loading .properties? ▼
getResourceAsStream uses the ClassLoader to find files on the classpath — works regardless of the working directory, in JARs, and on CI servers. FileInputStream uses an absolute or relative filesystem path — fragile on CI where working directory differs from local.
Q4: Can you override a config value at runtime without editing the properties file? ▼
Yes. In ConfigReader.get(), check System.getProperty(key) first, then fall back to the .properties value: String val = System.getProperty(key, props.getProperty(key)). This lets -Dbase.url=https://override.com on the Maven command line override any config file value.
Q5: Where should the env parameter be set — testng.xml or Maven command line? ▼
Both, with Maven taking priority. The testng.xml <parameter> provides the default. The Maven -Denv=prod overrides it via system properties. This means QAs can run from testng.xml in the IDE without flags, while CI uses the Maven command for explicit environment selection.
Q6: How do you load a different properties file mid-test if needed? ▼
The singleton pattern prevents mid-run reloading by design — the init() returns early if props is not null. For multi-environment scenarios within a single run, use a non-singleton approach: ConfigReader.load("uat") returns a Properties object without modifying the singleton state. Keep these as separate static helpers.

import io.github.bonigarcia.wdm.WebDriverManager;

public static void initDriver(String browser, boolean headless) {
    WebDriver driver;
    switch (browser.toLowerCase().trim()) {

        case "chrome":
            WebDriverManager.chromedriver().setup(); // auto-download
            ChromeOptions co = new ChromeOptions();
            if (headless) co.addArguments("--headless=new",
                                           "--window-size=1920,1080");
            co.addArguments("--no-sandbox", "--disable-dev-shm-usage",
                            "--disable-gpu");
            driver = new ChromeDriver(co);
            break;

        case "firefox":
            WebDriverManager.firefoxdriver().setup();
            FirefoxOptions fo = new FirefoxOptions();
            if (headless) fo.addArguments("-headless");
            driver = new FirefoxDriver(fo);
            break;

        case "edge":
            WebDriverManager.edgedriver().setup();
            EdgeOptions eo = new EdgeOptions();
            if (headless) eo.addArguments("--headless=new");
            driver = new EdgeDriver(eo);
            break;

        case "safari":
            // Safari: Mac only, no headless, SafariDriver built into macOS
            driver = new SafariDriver();
            break;

        case "remote_chrome":  // Selenium Grid / BrowserStack
            DesiredCapabilities caps = new DesiredCapabilities();
            caps.setBrowserName("chrome");
            driver = new RemoteWebDriver(
                new URL("http://hub:4444/wd/hub"), caps);
            break;

        default:
            throw new IllegalArgumentException("Unknown browser: " + browser);
    }
    driver.manage().window().maximize();
    driver.manage().timeouts()
          .implicitlyWait(Duration.ofSeconds(10));
    driverThread.set(driver);
}

Interview Q&A
Q1: How does WebDriverManager know which chromedriver version to download? ▼
WebDriverManager checks the installed Chrome version (via registry on Windows, command line on Linux/Mac), then queries its online metadata store to find the matching chromedriver version. It caches the binary in ~/.cache/selenium to avoid re-downloading on every run.
Q2: How do you run the same test on all three browsers from one Maven command? ▼
Use testng-crossbrowser.xml with three <test> blocks each having a different browser parameter. Run: mvn test -DsuiteFile=src/test/resources/testng-crossbrowser.xml. For parallel cross-browser, add parallel="tests" to the <suite> tag.
Q3: What is headless mode and when should you use it? ▼
Headless mode runs the browser without a visible UI — no window opens. Use it in CI pipelines where there's no display server (Linux servers), and for faster execution since rendering the UI takes CPU. Avoid it for debugging — headless can mask visual bugs that only appear in a real browser window.
Q4: How do you run tests on BrowserStack using the same framework? ▼
Add a "remote" case in DriverFactory that creates RemoteWebDriver with BrowserStack hub URL and your access key. Pass bstack.browser, bstack.os, and bstack.username as parameters from testng.xml. No test class changes needed — only DriverFactory and the XML configuration change.
Q5: Safari doesn't support headless — how do you handle that in your framework? ▼
In DriverFactory's safari case, ignore the headless parameter and log a warning: "Safari headless not supported — running headed." In CI, skip Safari tests entirely using a separate XML file that excludes the Safari <test> block. Alternatively, run Safari tests only when the agent is macOS.
Q6: What Chrome options do you add for CI stability? ▼
--no-sandbox (required inside Docker containers), --disable-dev-shm-usage (prevents /dev/shm crash in Docker), --disable-gpu (avoids GPU init errors on headless servers), --window-size=1920,1080 (prevents element-not-visible issues), and --disable-extensions (faster startup).

public class SimpleBaseTest {

    // Simple field — no ThreadLocal needed for sequential single-thread run
    protected WebDriver driver;

    @Parameters({"browser", "env"})
    @BeforeMethod(alwaysRun = true)
    public void setUp(
            @Optional("chrome") String browser,
            @Optional("dev")    String env) {
        ConfigReader.init(env);
        // SimpleBaseTest: just use switch directly, no ThreadLocal
        switch(browser.toLowerCase()) {
            case "firefox":
                WebDriverManager.firefoxdriver().setup();
                driver = new FirefoxDriver();
                break;
            default:
                WebDriverManager.chromedriver().setup();
                driver = new ChromeDriver();
        }
        driver.manage().window().maximize();
        driver.get(ConfigReader.get("base.url"));
    }

    @AfterMethod(alwaysRun = true)
    public void tearDown() {
        if(driver != null) { driver.quit(); driver = null; }
    }
}

Interview Q&A
Q1: When should you choose sequential over parallel? ▼
Use sequential when tests have data dependencies (test A creates data that test B reads), when the test suite is small (under 30 tests), when debugging — since parallel logs are interleaved and harder to read — and for smoke tests in a CI gate where you want clear, ordered output.
Q2: What is the default execution order of @Test methods in a sequential run? ▼
Without priority or dependsOnMethods, TestNG sorts methods alphabetically within a class. Methods with priority=0 (default) run first alphabetically, then priority=1 alphabetically, and so on. The order of <class> elements in testng.xml controls class-level order.
Q3: Do you still need ThreadLocal in a sequential framework? ▼
Technically no — with one thread, a simple protected WebDriver field works fine. However, it is a best practice to always use ThreadLocal even in sequential frameworks because: (1) you may add parallel later and won't need to refactor, (2) it makes the framework safe by default.
Q4: How do you prevent one test failure from skipping all subsequent tests in a sequential suite? ▼
Avoid dependsOnMethods between unrelated tests. By default, a test failure does NOT skip other tests unless they explicitly declare dependsOnMethods pointing to the failed test. Configure the retry analyzer so flaky tests are retried rather than causing downstream skips.
Q5: What Maven command runs the smoke suite sequentially on Firefox against UAT? ▼
mvn clean test -Puat -Dbrowser=firefox -DsuiteFile=src/test/resources/smoke.xml. The -Puat activates the UAT profile (loads uat.properties), -Dbrowser=firefox overrides the browser parameter, and -DsuiteFile points to smoke.xml.
Q6: How do you run only a specific test class sequentially without modifying testng.xml? ▼
Use Maven Surefire's test parameter: mvn test -Dtest=LoginTest. This bypasses testng.xml entirely and tells Surefire to run only the LoginTest class. For specific methods: -Dtest=LoginTest#validLoginTest. Note: @Parameters from XML won't be injected — use @Optional with defaults.

ScreenshotListener — thread-safe failure screenshotListener
public class ScreenshotListener implements ITestListener {

    @Override
    public void onTestFailure(ITestResult result) {
        // DriverFactory.getDriver() uses ThreadLocal, so this is thread-safe
        WebDriver driver = DriverFactory.getDriver();
        if (driver == null) return;  // guard: driver may be null

        try {
            TakesScreenshot ts = (TakesScreenshot) driver;
            File src = ts.getScreenshotAs(OutputType.FILE);
            // Thread-safe file naming: include thread ID
            String name = result.getName() + "_T" +
                          Thread.currentThread().getId() + ".png";
            File dest = new File("screenshots/" + name);
            FileUtils.copyFile(src, dest);
            System.out.println("Screenshot: " + dest.getAbsolutePath());
        } catch (Exception e) {
            System.err.println("Screenshot failed: " + e.getMessage());
        }
    }
}

Interview Q&A
Q1: What is the main risk of parallel Selenium tests and how do you prevent it? ▼
Thread interference — two threads acting on the same WebDriver instance. Prevented by ThreadLocal<WebDriver> in DriverFactory. Each thread calls set() with its own driver and get() always returns only that thread's driver — the JVM guarantees isolation.
Q2: What is the difference between parallel="methods" and parallel="tests"? ▼
parallel=methods runs each @Test method on its own thread — maximum parallelism but requires every method to be independent. parallel=tests runs each <test> XML block on its own thread — methods within one block run sequentially, making it safer for test classes that share state.
Q3: How do you achieve cross-browser parallel testing? ▼
Create multiple <test> blocks in testng.xml, each with a different browser parameter. Set parallel="tests" and thread-count matching the number of browsers. Thread-1 runs Chrome tests, Thread-2 runs Firefox tests simultaneously. ThreadLocal ensures each gets its own browser instance.
Q4: How do you debug failures in parallel runs where logs are interleaved? ▼
Prefix every log line with Thread.currentThread().getId() + " | " + testName. Use a logging framework (Logback with MDC) to tag each thread's context. Screenshots named with thread ID help correlate logs to visual evidence. Temporarily switch to sequential mode to reproduce and fix the issue.
Q5: What happens when thread-count is higher than the number of <test> blocks? ▼
The excess threads are simply unused — TestNG won't create more threads than there are parallel work units. If you have 3 <test> blocks and thread-count=10, exactly 3 threads run. Setting thread-count lower than the <test> count creates a queue — some blocks wait for a free thread.
Q6: Can @BeforeClass and @AfterClass methods cause issues in parallel runs? ▼
Yes, if the same class is used in multiple <test> blocks that run in parallel, @BeforeClass runs once per class per <test> block — so it runs in parallel on different threads. Each thread gets its own instance, so instance fields are safe. Static fields shared across instances are NOT safe and will cause race conditions.

Interview Q&A
Q1: What is the difference between -DskipTests and -Dmaven.test.skip=true? ▼
-DskipTests compiles test sources but does not execute them — useful to check compilation without running. -Dmaven.test.skip=true skips both compilation and execution of tests. The latter is faster but means test class compilation errors won't be caught.
Q2: Can you activate multiple Maven profiles simultaneously? ▼
Yes, use multiple -P flags or comma-separate them: -Puat,parallel. Maven merges the properties from all activated profiles. If two profiles set the same property, the last one wins. This is how we run UAT environment with the parallel suite: -Puat -Pparallel.
Q3: What does mvn clean do and why is it important before test? ▼
mvn clean deletes the target/ directory, removing all compiled classes, old test results, and reports. This ensures the next run starts fresh. Without clean, if a test file was deleted from src but its compiled class still exists in target, Maven may still execute the old test.
Q4: How does Maven know the test failed and how does Jenkins read that? ▼
Surefire generates XML reports in target/surefire-reports/. If any test fails, Maven exits with code 1. Jenkins reads the exit code — non-zero = build failure. Jenkins also has a JUnit-style XML parser that reads surefire reports to display test counts and individual failure details in the build UI.
Q5: How do you run a specific test method from command line? ▼
mvn test -Dtest=LoginTest#validLoginTest. The format is ClassName#methodName. For multiple methods in the same class: -Dtest="LoginTest#validLogin+invalidLogin". Note: @Parameters from testng.xml aren't injected when using -Dtest — ensure @Optional defaults cover all parameters.
Q6: What Maven command would you give to a developer who wants to quickly verify the login test works on their machine? ▼
mvn test -Dtest=LoginTest -Dbrowser=chrome. This skips the full suite, runs only LoginTest with Chrome, and uses default @Optional values from BaseTest so no XML file is needed. If they need specific env: add -Denv=dev. No other flags needed for a quick local check.

            // Attach to Allure report as inline image
            Allure.addAttachment(
                "Failure Screenshot: " + result.getName(),
                "image/png",
                new ByteArrayInputStream(screenshot),
                ".png"
            );
        } catch (Exception e) {
            System.err.println("Screenshot capture failed: " + e.getMessage());
        }
    }
}
Jenkinsfile — Declarative Pipeline for smoke + regressionJenkins
// Jenkinsfile (placed in project root)
pipeline {
    agent any

    tools {
        maven 'Maven3'    // Maven installation named in Jenkins tools
        jdk   'JDK17'     // JDK installation named in Jenkins tools
    }

    parameters {
        choice(name: 'ENV',     choices: ['dev','uat','prod'], description: 'Environment')
        choice(name: 'BROWSER', choices: ['chrome','firefox','edge'], description: 'Browser')
        booleanParam(name: 'HEADLESS', defaultValue: true, description: 'Headless?')
    }

    stages {
        stage('Checkout') {
            steps { checkout scm }
        }

        stage('Smoke Tests') {
            steps {
                sh """mvn clean test \
                  -DsuiteFile=src/test/resources/smoke.xml \
                  -Dbrowser=${params.BROWSER} \
                  -Denv=${params.ENV} \
                  -Dheadless=${params.HEADLESS}"""
            }
        }

        stage('Regression Tests') {
            when { expression { params.ENV != 'prod' } }  // skip on prod
            steps {
                sh """mvn clean test \
                  -P${params.ENV} \
                  -Dbrowser=${params.BROWSER} \
                  -Dheadless=${params.HEADLESS}"""
            }
        }

        stage('Allure Report') {
            steps {
                allure includeProperties: false,
                       jdk: '',
                       results: [[path: 'allure-results']]
            }
        }
    }

    post {
        always {
            junit '**/target/surefire-reports/*.xml'
            archiveArtifacts artifacts: 'screenshots/**', allowEmptyArchive: true
        }
        failure {
            emailext to: 'qa-team@company.com',
                     subject: "FAILED: ${env.JOB_NAME} #${env.BUILD_NUMBER}",
                     body: "Check ${env.BUILD_URL}allure"
        }
    }
}

        uses: actions/upload-artifact@v4
        if: failure()
        with:
          name: screenshots-${{ matrix.browser }}

Extent Reports integration (alternative to Allure)Extent
// pom.xml dependency:
// <dependency><groupId>com.aventstack</groupId>
// <artifactId>extentreports</artifactId><version>5.1.1</version></dependency>

public class ExtentReportListener implements ISuiteListener, ITestListener {

    private static ExtentReports extent;
    private static ThreadLocal<ExtentTest> test = new ThreadLocal<>();

    @Override
    public void onStart(ISuite suite) {
        ExtentSparkReporter spark = new ExtentSparkReporter("extent-report.html");
        spark.config().setTheme(Theme.DARK);
        extent = new ExtentReports();
        extent.attachReporter(spark);
    }

    @Override
    public void onTestStart(ITestResult result) {
        ExtentTest t = extent.createTest(result.getMethod().getMethodName());
        test.set(t);  // ThreadLocal — each test gets its own ExtentTest
    }

    @Override
    public void onTestSuccess(ITestResult r) { test.get().pass("PASSED"); }

    @Override
    public void onTestFailure(ITestResult r) {
        test.get().fail(r.getThrowable());  // logs exception stack trace
    }

    @Override
    public void onFinish(ISuite suite) {
        extent.flush();  // MUST call to write report to disk
    }
}
Best Practices
✓Use Allure.addAttachment() with byte array in the listener — never save to disk then attach, it's slower and can miss cleanup.
✓Archive allure-results/ across Jenkins builds for trend charts — without history, Allure shows no trends.
✓In GitHub Actions, use if: always() on report upload steps — they must run even when tests fail.
✓Use GitHub Actions matrix strategy for cross-browser — it runs Chrome and Firefox in parallel automatically.
✓Gate production deployments on smoke test success — add the test stage before the deploy stage in your pipeline.
Interview Q&A
Q1: How does Allure generate reports from TestNG? ▼
Allure-TestNG is a listener that intercepts TestNG events and writes JSON files to allure-results/ during the test run. After the run, mvn allure:report reads those JSON files and generates a rich HTML report. The aspectj javaagent enables @Step annotation interception at the bytecode level.
Q2: What is the difference between Allure and ExtentReports? ▼
Allure generates JSON intermediaries and needs a separate generation step — better for CI with historical trends, categories, and flakiness tracking. ExtentReports writes directly to HTML in a single pass — simpler setup, better for teams without CI report plugins. Allure is the industry preference for SDET roles.
Q3: How do you integrate TestNG results with Jenkins? ▼
Surefire generates JUnit-compatible XML reports in target/surefire-reports/. Use the junit post-build step in Jenkinsfile: junit '**/target/surefire-reports/*.xml'. Jenkins parses these to show pass/fail counts, test trends, and mark builds as unstable (some failures) or failed (all failures).
Q4: How do you ensure screenshots don't get overwritten in parallel runs? ▼
Include the Thread ID and timestamp in the filename: result.getName() + "_T" + Thread.currentThread().getId() + "_" + System.currentTimeMillis() + ".png". Better yet, attach directly to the Allure report as a byte array — no filesystem naming collision possible.
Q5: How do you send email notifications for failed test runs from Jenkins? ▼
Use the emailext plugin in the Jenkinsfile post { failure { } } block. Configure the SMTP server in Jenkins global settings. The email includes the build URL, so recipients can click through to the Allure report. You can also attach the surefire summary as a log attachment.
Q6: What is the GitHub Actions matrix strategy for cross-browser testing? ▼
The strategy.matrix block defines axes (browser: [chrome, firefox], env: [uat]). GitHub Actions creates one job per matrix combination — so 2 browsers × 1 env = 2 parallel jobs, each running mvn test with its own browser and env value. Each job gets its own fresh runner VM — true isolation, no ThreadLocal needed at the CI level.
