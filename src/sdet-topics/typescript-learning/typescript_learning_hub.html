TypeScript — JavaScript with superpowers

TypeScript is a statically typed superset of JavaScript developed by Microsoft. Every valid JavaScript is valid TypeScript, but TypeScript adds a type system on top, catching bugs at compile time — before the code runs.

What
JS + static type annotations
Why
Catch errors early, better IDE support, readable code
When
Any JS project, especially large or team-based
Where
Frontend (React), Backend (Node), Testing (Playwright, Jest)
How
Write .ts files → tsc compiles to .js → run in browser/Node
Interview one-liners (say exactly these)
Line 1"TypeScript is a statically typed superset of JavaScript that compiles down to plain JavaScript."
Line 2"The main advantage of TypeScript is catching type errors at compile time rather than at runtime."
Line 3"TypeScript's type system is structural, not nominal — if the shape matches, the type matches."
Line 4"In automation testing, TypeScript gives us type-safe selectors, typed API responses, and self-documenting test code."
Line 5"TypeScript is a development-time tool; after compilation, only JavaScript runs in the browser or Node.js."
Core concepts — TypeScript vs JavaScript architecture

Primitive types

string, number, boolean, null, undefined, symbol, bigint — same as JS primitives, but now declared explicitly.

Special TS types

any (opt out of typing), unknown (safe alternative to any), never (impossible value), void (no return value), tuple (fixed-length array).

Type inference

TypeScript is smart — if you write let age = 25, it infers age: number. You don't always need to write types explicitly.

Interview prep — SDET Q&A
Q1. What is TypeScript and how does it differ from JavaScript?▼
TypeScript is a statically typed superset of JavaScript developed by Microsoft. The key difference is that TypeScript performs type checking at compile time, catching errors before the code runs, while JavaScript only reveals type errors at runtime. TypeScript compiles to plain JavaScript, so it runs everywhere JS runs. In SDET work, this means our test scripts are safer and easier to maintain.
Q2. What is the difference between any and unknown?▼
any completely disables type checking — you can do anything with it. unknown also accepts any value, but forces you to narrow the type before you use it (e.g., using typeof). As an SDET, I prefer unknown for handling API responses where the shape isn't confirmed, then use type guards to safely access fields.
Q3. What is a Union type and where would you use it in test automation?▼
A Union type (A | B) means a value can be one of several types. In test automation I use it for test status: type Status = "pass" | "fail" | "skip" — this prevents invalid status values being assigned, making test result tracking type-safe and self-documenting.
Q4. What is a Tuple and how is it different from an Array?▼
An Array holds many elements of the same type. A Tuple is a fixed-length array where each position has its own specific type. For example: [string, number, boolean] maps to [testName, responseTime, passed]. TypeScript enforces both the count and the type at each position — arrays do not.
Q5. Why would you use string enums instead of numeric enums in automation?▼
String enums produce human-readable values in logs and reports. If a test fails and logs show Environment.QA = "QA", that's immediately understandable. Numeric enums show 1 which requires a lookup. In CI pipelines and test reports, string enum values are self-explanatory without needing the source code.

Example 2 · Optional and default parametersTypeScript
// Optional param: add ? after param name

navigateTo("https://app.com", 3000); // ✅ explicit waitTime

// Default param: assign value directly

}

retry(myTest, 5); // ✅ attempts = 5

✓Always annotate return types explicitly — don't rely on inference for public APIs
✓Optional params (?) should always come AFTER required params
✓Prefer ?? (nullish coalescing) over || for defaults — || treats 0 and "" as falsy

⚠Avoid using Function type — use explicit function types like (x: string) => void
Interview prep
Q1. How do you declare an optional parameter in TypeScript?▼
By appending ? after the parameter name: function go(url: string, wait?: number). Optional params must always appear after all required params. Inside the function, the optional param's type becomes number | undefined, so you must handle the undefined case before using it.
Q2. What is the difference between void and never?▼
void means the function returns, but the return value is undefined — used for functions that perform side effects. never means the function NEVER returns at all — either it always throws an error or runs an infinite loop. In test automation, helper steps often return void; an error-throwing utility returns never.
Q3. How do function overloads work in TypeScript?▼
You declare multiple overload signatures (no body), then write one implementation signature that handles all cases. TypeScript uses the overload signatures for type checking callers. The implementation is invisible to callers. This is useful when a function behaves differently based on input type.
Q4. What is a rest parameter and when would you use it?▼
Rest parameters (...args: string[]) collect any number of trailing arguments into a typed array. In test automation I use this for helpers like logSteps(suite, ...steps) where the number of steps isn't fixed. It's cleaner than passing an array explicitly.
Q5. What does it mean when a function returns Promise<void>?▼
It means the function is asynchronous (returns a Promise) but when the Promise resolves it produces no meaningful value. In Playwright and Jest tests, most test step functions are async and return Promise<void> — you await them but don't capture a return value.

How
interface → class implements → extends for inheritance

  }
}

}

Interview prep
Q1. What is the difference between interface and type in TypeScript?▼
Both describe object shapes. Interfaces support declaration merging and can be extended using extends. Types can represent unions, intersections, and mapped types — things interfaces can't do. For object shapes in POM, I use interfaces. For union types like Status = "pass" | "fail", I use type aliases.
Q2. What are access modifiers and do they exist in compiled JavaScript?▼
Access modifiers (public, private, protected, readonly) are compile-time constructs in TypeScript. They enforce encapsulation during development but are completely erased from the compiled JavaScript — the output JS has no concept of private. For true runtime privacy, JavaScript uses # (private class fields).
Q3. How does TypeScript support the Page Object Model pattern?▼
TypeScript makes POM much stronger. We define an interface or abstract class for BasePage with common contracts. Each page class extends BasePage and implements its abstract methods. Locators are private, actions are public. Constructor shorthand reduces boilerplate. Type-safe methods prevent calling actions with wrong arguments.
Q4. What is an abstract class? When would you use it over an interface?▼
Abstract classes can have both implemented methods (shared logic) and abstract methods (that subclasses must implement). Interfaces are pure contracts with no implementation. I use abstract class for BasePage because I want to share real code like waitForLoad() while still requiring each page to implement isLoaded().
Q5. What is constructor shorthand in TypeScript?▼
Instead of declaring a property and then assigning it in the constructor body, TypeScript lets you add the access modifier directly to the constructor parameter: constructor(private url: string). TypeScript automatically declares the property AND assigns it. It significantly reduces boilerplate in page object constructors.

Generics — Write Once, Use for Any Type

Generics are TypeScript's way of writing reusable, type-safe code that works with different types. Instead of duplicating functions for each type or losing safety with any, generics use a type parameter (like <T>) that gets filled in when called.

What
Placeholder type <T> resolved at call site
Why
DRY + type-safe — no duplicating code, no losing types
When
API wrappers, test data factories, utility helpers
Where
Generic API clients, test fixtures, collection helpers
How
function name<T>(param: T): T { … }
Interview one-liners
Line 1"Generics let you write a single function or class that works with multiple types without losing type safety."
Line 2"<T> is a type variable — a placeholder that gets replaced with the actual type when the function is called."
Line 3"Generic constraints (<T extends SomeType>) ensure that the type variable has specific properties available."
Line 4"In SDET work, I use generics for a typed API client wrapper so every endpoint returns its specific response type."
Line 5"TypeScript's built-in utility types like Partial<T> and Pick<T, K> are themselves generic types."
Code examples
Example 1 · The problem generics solveJS vs TS
// Without generics — duplicate code OR unsafe any
function firstString(arr: string[]): string  { return arr[0]; }
function firstNumber(arr: number[]): number  { return arr[0]; }
// same logic, different types = code duplication

// With generics — ONE function, any type, fully safe
function first<T>(arr: T[]): T {
  return arr[0]; // T is resolved at call time
}

const s = first(["login", "checkout"]); // T = string → returns string
const n = first([10, 20, 30]);           // T = number → returns number
// s and n are FULLY typed — autocomplete works, no casting
Example 2 · Generic constraintsTypeScript
// T extends object — T must have a specific shape
function getField<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const user = { name: "Alice", age: 30, role: "admin" };
const name = getField(user, "name");  // ✅ returns string
const age  = getField(user, "age");   // ✅ returns number
const bad  = getField(user, "email"); // ❌ 'email' not in User

// Constraint: T must have a .length property
function logLength<T extends { length: number }>(val: T): void {
  console.log(val.length); // TS knows T has .length
}
logLength("hello");    // ✅ string has .length
logLength([1,2,3]);    // ✅ array has .length
logLength(42);         // ❌ number has no .length
Example 3 · Generic API client (SDET gold)TypeScript
// Generic typed HTTP response
interface ApiResponse<T> {
  data: T;
  status: number;
  message: string;
}

// Specific response types
interface User    { id: number; name: string; email: string }
interface Product { sku: string; price: number }

// Generic fetch wrapper
async function apiGet<T>(url: string): Promise<ApiResponse<T>> {
  const res = await fetch(url);
  return res.json();
}

// Usage — fully typed, no casting needed
const userResp    = await apiGet<User>("/api/users/1");
console.log(userResp.data.email); // ✅ TS knows .email is string

const productResp = await apiGet<Product>("/api/products/1");
console.log(productResp.data.price); // ✅ TS knows .price is number

Best practices
✓Name type params meaningfully: T for general, K for key, V for value, TResponse for API responses
✓Use extends constraints to ensure the type variable has the properties you need
✓Build a generic ApiClient<T> class for your test framework — one wrapper for all endpoints
⚠Don't use any where a generic would work — generics preserve type information, any destroys it
⚠Avoid deeply nested generics (>2 levels) — split into smaller typed interfaces for readability
Interview prep
Q1. What are Generics and why do we need them?▼
Generics let you write reusable code that works with multiple types while preserving type safety. Without generics, you'd either duplicate logic for each type or use any and lose all type information. For example, a generic first<T>(arr: T[]): T works for arrays of any type and the return type is always the element type, not any.
Q2. What is a generic constraint and when would you use it?▼
A generic constraint (T extends SomeType) limits which types can be passed as the type variable. You use it when your function needs to access a specific property or method on the type variable. For example, T extends { length: number } ensures that whatever type T is, it has a length property you can access safely.
Q3. How do you use generics for an API client in test automation?▼
I define a generic response interface ApiResponse<T> and a generic fetch function apiGet<T>(url: string): Promise<ApiResponse<T>>. Each endpoint call specifies the expected type: apiGet<User>("/users/1"). This gives full autocomplete and type checking on the response data without writing separate functions per endpoint.
Q4. What is Partial<T> and when would you use it?▼
Partial<T> makes all fields of T optional. I use it for update/patch functions where you only need to provide the fields that changed. For test config overrides: function overrideConfig(patch: Partial<TestConfig>) means callers only pass what they want to change, not the entire config object.
Q5. What is the difference between Pick and Omit?▼
Pick<T, K> selects only the specified keys from T, creating a new type with just those fields. Omit<T, K> does the opposite — it includes all fields except the specified keys. Use Pick when you want a small subset; use Omit when you want most fields but need to exclude a few.

Advanced types let you build highly expressive, safe type logic: narrow union types at runtime, create new types from existing ones, and leverage conditional types to make your framework self-describing.

What
Type guards, narrowing, mapped, conditional types
Why
Handle real-world ambiguity safely without any
When
API response parsing, error handling, dynamic configs
Where
API clients, assertion libraries, typed test fixtures
How
typeof / instanceof / custom predicates → narrow type
Interview one-liners
Line 1"Type narrowing is the process of refining a broad type to a more specific one inside a conditional block."
Line 2"A user-defined type guard is a function that returns value is Type — it tells TypeScript what type a value is."
Line 3"Discriminated unions use a shared literal field (like kind) to let TypeScript narrow union members safely."
Line 4"Mapped types transform each property of an existing type — for example, making all fields optional or readonly."
Line 5"Conditional types (T extends U ? A : B) are TypeScript's ternary operator at the type level."
Code examples
Example 1 · Type narrowing with typeof and instanceofTypeScript
function formatId(id: string | number): string {
  if (typeof id === "string") {
    // TS narrows: id is string here
    return id.toUpperCase();   // ✅ string methods available
  }
  // TS narrows: id is number here
  return id.toFixed(0);       // ✅ number methods available
}

// instanceof narrows to class type
function handleError(err: unknown): string {
  if (err instanceof Error) {
    return err.message;        // TS knows .message exists
  }
  if (typeof err === "string") {
    return err;
  }
  return "Unknown error";
}
Example 2 · User-defined type guardsTypeScript
interface ApiSuccess { status: "success"; data: object }
interface ApiError   { status: "error";   message: string }
type ApiResult = ApiSuccess | ApiError;

// Type guard — return type "value is Type" is the magic
function isSuccess(r: ApiResult): r is ApiSuccess {
  return r.status === "success";
}

async function callApi(): Promise<void> {
  const result: ApiResult = await fetchSomething();

  if (isSuccess(result)) {

Interview prep
Q1. What is type narrowing in TypeScript?▼
Type narrowing is when TypeScript refines a broad type (like string | number) to a more specific type inside a conditional block. Using typeof, instanceof, or custom type guard functions tells the compiler what the actual type is in each branch, giving you access to the correct methods and properties.
Q2. What is a discriminated union and why is it useful in test automation?▼
A discriminated union is a union where each member has a common literal field (the discriminant, like kind or type) that uniquely identifies it. In test automation I use it for test events: "start" | "pass" | "fail". TypeScript's switch narrows to the exact member in each case, so you only access properties that actually exist on that variant.
Q3. How would you safely handle an unknown error type in a catch block?▼
In TypeScript's strict mode, caught errors are typed as unknown. I use instanceof Error to narrow: if (err instanceof Error) { return err.message; }. If you need to handle custom error classes, create a type guard function that checks the shape. Never cast directly to Error without checking — the thrown value might not be an Error instance.
Q4. What is a mapped type and give a practical example?▼
A mapped type creates a new type by iterating over the keys of an existing type: { [K in keyof T]: ... }. A practical example is making all config fields optional: type Optional<T> = { [K in keyof T]?: T[K] }. Built-in utility types like Partial, Readonly, and Required are all implemented as mapped types.
Q5. What is ReturnType<T> and how does it help in automation?▼
ReturnType<typeof fn> extracts the return type of a function at the type level. In automation, if I have a data factory function that builds test objects, I can use ReturnType<typeof buildUser> to get the User type automatically without defining a separate interface. This keeps the type always in sync with the factory.

Interview prep
Q1. What is the most important tsconfig setting and why?▼
"strict": true is the single most impactful setting. It enables strictNullChecks, noImplicitAny, strictFunctionTypes, and several others. Without it, TypeScript is significantly more permissive and misses many real bugs. I always set it to true at project start.
Q2. What is a barrel file and why is it useful?▼
A barrel file is an index.ts that re-exports from multiple modules in a folder. Instead of import { LoginPage } from "../../pages/LoginPage", you get import { LoginPage } from "@pages". It creates a clean public API for each folder and reduces brittle relative paths throughout the codebase.
Q3. What is the difference between import and import type?▼
import type imports only the type information and is completely erased at compile time — no JavaScript is emitted for it. Regular import can include runtime values. Using import type for type-only imports improves build performance, prevents accidental runtime usage, and avoids circular dependency issues.
Q4. What is a .d.ts file?▼
A declaration file (.d.ts) contains only type information — no implementation. TypeScript uses these to understand the types of JavaScript libraries that weren't written in TypeScript. Most popular npm packages have @types/package-name packages containing declaration files. You can also write your own to type custom JS utilities.
Q5. How do path aliases improve an automation framework?▼
Path aliases replace long relative imports (../../../src/pages/Login) with short, readable aliases (@pages/Login). They're configured in tsconfig under paths. The benefits are: cleaner imports, refactoring safety (move a folder without updating 50 import paths), consistent imports across the team, and better readability in code reviews.

Example 2 · Typed HTTP client classTypeScript
class HttpClient {
  constructor(
    private readonly baseUrl: string,
    private readonly token: string
  ) {}

  private headers(): Record<string, string> {
    return { "Authorization": `Bearer ${this.token}`,
              "Content-Type":  "application/json" };
  }

  async get<T>(path: string): Promise<T> {
    const res = await fetch(`${this.baseUrl}${path}`, {
      headers: this.headers()
    });
    if (!res.ok) throw new Error(`GET ${path} → ${res.status}`);
    return res.json() as T;
  }

  async post<TBody, TResponse>(path: string, body: TBody): Promise<TResponse> {
    const res = await fetch(`${this.baseUrl}${path}`, {
      method:  "POST",
      headers: this.headers(),
      body:    JSON.stringify(body)
    });
    return res.json() as TResponse;
  }
}

// Usage — fully typed
const client = new HttpClient("https://api.app.com", token);
const user = await client.get<User>("/users/1");    // returns User
const order = await client.get<Order>("/orders/5"); // returns Order

class ApiError extends Error {

    public readonly statusCode: number,
    public readonly endpoint: string
  ) {

    this.name = "ApiError";
  }

async function safeGet<T>(url: string): Promise<T> {
  try {
    const res = await fetch(url);
    if (!res.ok) {
      throw new ApiError(`Failed: ${res.statusText}`, res.status, url);
    }
    return res.json() as T;
  } catch (err: unknown) {
    if (err instanceof ApiError) {
      console.error(`API Error ${err.statusCode} @ ${err.endpoint}`);
      throw err; // re-throw typed error
    }
    throw new Error("Network failure");
  }
}

Interview prep
Q1. How does TypeScript handle async/await return types?▼
Any function marked async automatically returns a Promise. TypeScript infers the type: async function getUser(): Promise<User> — the function returns Promise<User> not User. At the call site, await getUser() unwraps to User. TypeScript tracks this correctly through the async/await chain without needing explicit annotations at every step.
Q2. When would you use Promise.all vs Promise.allSettled?▼
Promise.all fails fast — if any promise rejects, the whole call rejects. Use it when all calls must succeed (e.g. setting up all test data). Promise.allSettled waits for all promises regardless — each result has a status: "fulfilled" | "rejected" discriminator. Use it when you need all results even if some fail — e.g., cleaning up multiple test resources.
Q3. How would you architect a typed API client for an automation framework?▼
I build a generic HttpClient class with typed get<T> and post<TBody, TResponse> methods. The class handles auth headers centrally. Each endpoint becomes a method on a service class that calls the client: userService.getUser(id): Promise<User>. All interfaces live in src/types/. This gives every test strongly typed API interactions with zero duplication.
Q4. How do you handle errors in async TypeScript code?▼
I use try/catch with catch (err: unknown) — in strict mode errors are unknown. I narrow with instanceof: check for my custom error classes first, then for the base Error class. I create custom error classes (ApiError, NetworkError) that carry typed context. I re-throw after logging so the test runner reports the failure correctly.
