Overview

TestNG (Test Next Generation) is a testing framework for Java, inspired by JUnit and NUnit, designed to cover all categories of tests — unit, functional, end-to-end, integration.

What
Java test framework with annotations, grouping, parameterization & parallel execution
Why
More powerful than JUnit — supports data-driven tests, test dependencies, parallel runs
When
Whenever you need structured, configurable test suites in Java/Selenium projects

"I prefer TestNG over JUnit because it gives me more control over test configuration through testng.xml."
"TestNG generates detailed HTML and XML reports automatically after every test run without any extra setup."
"In TestNG, all test configuration is done through annotations — no need for inheritance or rigid class hierarchies."
"TestNG integrates seamlessly with Maven, Gradle, and CI tools like Jenkins, making it production-ready."

✓Add a description attribute to every @Test — it appears in reports and makes failures self-explanatory.

✓Use Assert.fail() with a clear message rather than letting tests silently pass on wrong state.
Interview Q&A
Q1: What is TestNG and how is it different from JUnit? ▼
TestNG is a Java testing framework inspired by JUnit, but more powerful. Unlike JUnit, TestNG supports grouping of tests, parallel execution, data providers, test dependencies, and flexible configuration via testng.xml — all without needing extra extensions.
Q2: What are hard assertions vs soft assertions? ▼
Hard assertions (Assert.*) immediately stop the test on failure. Soft assertions (SoftAssert.*) collect all failures and report them together when assertAll() is called — useful when validating multiple fields on one page.
Q3: How does TestNG generate reports? ▼
TestNG automatically generates two reports after a test run: an HTML report (index.html) and an XML results file (testng-results.xml) in the test-output folder — no extra configuration needed.
Q4: Can TestNG work with Maven? How? ▼
Yes. Add TestNG as a dependency in pom.xml with scope 'test'. Use the maven-surefire-plugin to point to your testng.xml file so Maven executes it during the 'test' phase.
Q5: What happens when an expectedExceptions test throws a different exception? ▼
TestNG marks the test as FAILED. The expectedExceptions attribute only passes if the exact specified exception class is thrown. Any other exception or no exception at all results in a test failure.
Overview — TestNG Annotations

Annotations are special markers placed above methods/classes that tell TestNG what to do and when to do it. They control the entire test lifecycle — setup, teardown, and test execution order.

What
Java metadata markers (@Before*, @After*, @Test) controlling execution lifecycle
Why
Remove boilerplate; declarative setup/teardown at Suite, Test, Class, and Method scope
Scope order
Suite → Test → Class → Method (before) | Method → Class → Test → Suite (after)
Spoken Lines in Interview
"TestNG has four levels of setup/teardown: Suite, Test, Class, and Method — each mapping to @BeforeSuite, @BeforeTest, @BeforeClass, and @BeforeMethod."
"@BeforeClass runs once before all test methods in a class, while @BeforeMethod runs before each individual test method."
"I use @BeforeSuite to launch the browser once and @AfterSuite to quit it, so the browser doesn't restart for every test."
"The @Test annotation is the core marker — without it, TestNG ignores the method entirely."
"@AfterMethod is perfect for taking screenshots on failure because it runs after every test regardless of pass/fail."
Annotation Lifecycle Flow
Code Examples
Example 1 — Complete lifecycle in one classLifecycle
import org.testng.annotations.*;

public class LifecycleDemo {

    @BeforeSuite   // runs ONCE before entire suite starts
    public void suiteSetup() { System.out.println("Suite Start"); }

    @BeforeTest    // runs before each <test> tag in testng.xml
    public void testSetup() { System.out.println("Test Start"); }

    @BeforeClass   // runs once before first method in this class
    public void classSetup() { System.out.println("Class Start"); }

    @BeforeMethod  // runs before EACH @Test method
    public void methodSetup() { System.out.println("Method Start"); }

    @Test
    public void test1() { System.out.println("Test 1 running"); }

    @Test
    public void test2() { System.out.println("Test 2 running"); }

    @AfterMethod   // runs after EACH @Test method
    public void methodTeardown() { System.out.println("Method End"); }

    @AfterClass   // runs once after last method in class
    public void classTeardown() { System.out.println("Class End"); }

    @AfterSuite   // runs ONCE after entire suite ends
    public void suiteTeardown() { System.out.println("Suite End"); }
}

Example 3 — Selenium + TestNG BeforeMethod/AfterMethod patternReal World
import org.openqa.selenium.WebDriver;
import org.openqa.selenium.chrome.ChromeDriver;
import org.testng.annotations.*;

public class SeleniumBase {

    protected WebDriver driver;   // protected — child classes can use it

    @BeforeMethod
    public void openBrowser() {
        driver = new ChromeDriver();  // fresh browser for every test
        driver.manage().window().maximize();
    }

    @AfterMethod
    public void closeBrowser() {
        if (driver != null) driver.quit(); // close browser after each test
    }
}
Example 4 — @BeforeClass vs @BeforeMethod: when to use whichDifference
public class LoginTests {

    @BeforeClass   // DB connection — expensive, do ONCE for whole class
    public void connectDB() {
        System.out.println("DB connected");
    }

    @BeforeMethod  // navigate to login page before EACH test
    public void goToLogin() {
        System.out.println("Opening login page");
    }

    @Test
    public void validLogin()   { System.out.println("Valid login"); }
    @Test
    public void invalidLogin() { System.out.println("Invalid login"); }
    // goToLogin() will run before BOTH validLogin and invalidLogin
    // connectDB() runs only ONCE before the first test starts
}
Example 5 — alwaysRun = true in @AfterMethodAlways Run
@AfterMethod(alwaysRun = true)  // runs even if test FAILS or SKIPS
public void takeScreenshotOnFailure(ITestResult result) {
    if (result.getStatus() == ITestResult.FAILURE) {
        // capture screenshot logic here
        System.out.println("Screenshot taken for: " + result.getName());
    }
}
// Without alwaysRun=true, @AfterMethod is skipped when @BeforeMethod fails
Best Practices
✓Use @BeforeSuite for one-time global setup (browser launch, DB connect) and @AfterSuite for cleanup.
✓Always mark @AfterMethod with alwaysRun=true so teardown runs even when a test fails.
✓Never put assertions inside @BeforeMethod — if it throws, @AfterMethod won't run without alwaysRun.
✓Keep setup/teardown methods concise — they are not test methods, so don't add test logic in them.
✓Use enabled=false to temporarily skip a test instead of commenting it out — it still shows in reports as skipped.
Interview Q&A
Q1: What is the difference between @BeforeClass and @BeforeMethod? ▼
@BeforeClass runs once before the first test method in a class — ideal for expensive setup like DB connections. @BeforeMethod runs before every single @Test method — ideal for navigating to a page or resetting state.
Q2: What happens if @BeforeMethod fails? ▼
The @Test method is marked as SKIPPED (not failed). The @AfterMethod runs only if alwaysRun=true is set. Without it, @AfterMethod is also skipped, which can leave browsers or connections open.
Q3: What is the purpose of alwaysRun=true? ▼
It forces the annotated method (usually @AfterMethod or @AfterSuite) to execute even if a dependent setup method failed or a test was skipped. This is critical for cleanup methods like closing browsers or taking screenshots.
Q4: Can you have multiple @BeforeMethod methods in one class? ▼
Yes. If multiple @BeforeMethod methods exist in a class, TestNG runs all of them before each test, but the execution order between them is not guaranteed unless you use dependsOnMethods or priority.
Q5: What is the execution order when a child class extends a base class with @Before annotations? ▼
Parent's @BeforeClass runs first, then child's @BeforeClass. For @BeforeMethod, parent's runs before child's. For @After methods, the order reverses — child's @AfterMethod runs first, then parent's.
Overview — testng.xml Suite Configuration

testng.xml is the master configuration file that tells TestNG which tests to run, in what order, and with which settings. It is the backbone of every professional TestNG setup.

What
XML file defining suites, tests, classes, and methods to execute
Why
Run specific subsets of tests without changing code; configure parallel execution, groups, parameters
Where
Root of project (src/test/resources or project root); referenced by Maven Surefire plugin
Spoken Lines in Interview
"testng.xml is the heart of a TestNG project — it controls which classes, methods, and groups are executed."
"I can have multiple <test> tags inside one suite to logically separate regression, smoke, and sanity runs."
"Using the <include> and <exclude> tags in testng.xml, I can run specific test methods without touching the code."
"The verbose attribute in testng.xml controls how much detail TestNG prints to the console — I usually set it to 2."
"testng.xml supports passing runtime parameters via <parameter> tags, which my tests read using @Parameters."

Best Practices
✓Always use a meaningful name attribute on <suite> and <test> — it appears in reports and Jenkins dashboards.
✓Use <packages> scanning for large projects so you don't manually list every class in XML.
✓Set verbose="2" locally for debug output; use verbose="0" in CI to keep logs clean.
✓Create separate testng.xml files for different environments (smoke.xml, regression.xml, api.xml) rather than one giant file.
✓Pass environment-specific values (URL, browser, credentials) as <parameter> tags rather than hardcoding in Java.
Interview Q&A
Q1: What is the difference between <suite> and <test> in testng.xml? ▼
A <suite> is the outermost container — it represents the entire test run (e.g. "Full Regression"). A <test> is a logical group inside the suite (e.g. "Smoke", "Regression"). You can have multiple <test> tags in one suite, each with different classes and configurations.
Q2: How do you skip a specific test method using testng.xml? ▼
Use the <exclude name="methodName"/> tag inside the <methods> block of that class. This excludes only that method while keeping all others in the same class active.
Q3: Where should testng.xml be placed in a Maven project? ▼
Typically in src/test/resources or the project root. In pom.xml, the maven-surefire-plugin references it via <suiteXmlFiles><suiteXmlFile>testng.xml</suiteXmlFile></suiteXmlFiles>.
Q4: How do <parameter> tags help in cross-browser testing? ▼
You define <parameter name="browser" value="firefox"/> in testng.xml and read it in Java with @Parameters("browser"). By creating two <test> blocks with different browser values, you can run the same tests on multiple browsers from one XML file.
Q5: What is the verbose attribute in testng.xml? ▼
verbose controls console output verbosity. Level 0 = silent (only failures), Level 1 = test names, Level 2 = test names + method names (most common for debugging), Level 10 = maximum detail including internal TestNG events.
Overview — Groups & Test Priority

Groups let you tag tests with logical labels (smoke, regression, api) and run only matching tests. Priority controls the execution order of test methods within a class.

Groups
Logical tags on @Test methods; run subsets via testng.xml <groups>
Priority
Integer value on @Test; lower number = runs first. Default is 0.
Use case
Run only "smoke" group in CI; run "regression" nightly; order login → search → checkout
Spoken Lines in Interview
"I use TestNG groups to tag tests as smoke, sanity, or regression — then run only the required group in CI pipelines."
"Priority in TestNG is an integer — tests with lower priority numbers run first; tests with equal priority run in alphabetical order."
"A test method can belong to multiple groups simultaneously — for example, a login test can be both 'smoke' and 'regression'."
"Without priority set, TestNG does NOT guarantee test execution order — it runs them alphabetically within a class."
"I can include a group in one testng.xml and exclude it in another — giving me full flexibility across environments."
Code Examples
Example 1 — Assigning groups to @Test methodsGroups
public class ShoppingTests {

    @Test(groups = "smoke")         // belongs to 'smoke' group
    public void verifyHomePage() { }

    @Test(groups = {"smoke", "regression"})  // belongs to BOTH groups
    public void verifyLogin() { }

    @Test(groups = "regression")    // only in regression group
    public void verifyCheckout() { }

    @Test(groups = "api")           // only in api group
    public void verifyOrderAPI() { }
}
Example 2 — Running groups from testng.xmlXML Config
<!-- Run ONLY smoke tests -->
<suite name="Smoke Run">
  <test name="Smoke Tests">
    <groups>
      <run>
        <include name="smoke"/>      <!-- only smoke group runs -->
        <exclude name="api"/>        <!-- explicitly exclude api -->
      </run>
    </groups>
    <classes>
      <class name="com.tests.ShoppingTests"/>
    </classes>
  </test>
</suite>
Example 3 — priority attribute to control orderPriority
public class OrderFlowTest {

    @Test(priority = 1)   // runs FIRST (lowest number = first)
    public void login() {
        System.out.println("Step 1: Login");
    }

    @Test(priority = 2)   // runs SECOND
    public void addToCart() {
        System.out.println("Step 2: Add to Cart");
    }

    @Test(priority = 3)   // runs THIRD
    public void checkout() {
        System.out.println("Step 3: Checkout");
    }

    @Test                  // no priority = default 0, runs BEFORE priority=1!
    public void openSite() {
        System.out.println("Step 0: Open Site");

Best Practices
✓Standardize group names across your team — use "smoke", "regression", "sanity", "api" consistently.
✓Use priority only when test order truly matters (like an e2e flow). Avoid overusing it — it creates coupling.
✓Remember default priority is 0 — methods without priority run before those with priority=1,2,3...
✓Prefer dependsOnMethods (Session 8) over priority for true sequential e2e flows — it's more explicit.
✓Create dedicated testng.xml files per group — smoke.xml, regression.xml — so CI jobs can trigger each independently.
Interview Q&A
Q1: What is the default priority of a @Test method? ▼
The default priority is 0. This means any @Test method without an explicit priority runs before those with priority=1,2,3 etc. Methods with the same priority run in alphabetical order of method name.
Q2: Can a test method belong to multiple groups? ▼
Yes. Use groups = {"smoke", "regression"} to assign multiple groups. This allows the same test to be included in both a quick smoke run and a full regression suite.
Q3: What is the difference between groups and priority? ▼
Groups are labels for filtering which tests to run. Priority is a number that controls the execution order within a class. They solve different problems — groups are about selection, priority is about sequencing.
Q4: How do you run only smoke tests from a Maven command? ▼
Point the maven-surefire-plugin to a smoke.xml file that has <include name="smoke"/> in its groups section, then run mvn test. Alternatively, use -Dgroups=smoke with the surefire plugin configured to accept it.
Q5: What is a "group of groups" in TestNG? ▼
It is a meta-group defined in testng.xml using <define name="all-ui"> that aggregates multiple existing groups. Instead of including smoke and regression separately in every XML, you include the combined all-ui group.
Overview — @Parameters

@Parameters allows you to pass external values from testng.xml directly into test methods. This is the simple parameterization approach — one set of values per run, ideal for environment configs.

What
Pass static values from testng.xml <parameter> tags into Java methods
Why
Avoid hardcoding browser, URL, credentials in code; switch environments without recompiling
Limit
Only one value per parameter per run — use @DataProvider for multiple rows
Spoken Lines in Interview
"@Parameters reads values defined in testng.xml and injects them as method arguments — no hardcoding needed."
"I use @Parameters for browser name, base URL, and environment — changing environment means only updating testng.xml."
"@Parameters works on @Test, @BeforeMethod, @BeforeClass — any TestNG-annotated method can receive parameters."
"The @Optional annotation lets me define a default value so the test doesn't fail if a parameter is missing from XML."
"@Parameters is for static, single-value config; @DataProvider is for multiple data sets — they serve different purposes."
Code Examples
Example 1 — Basic @Parameters usageBasics
// testng.xml defines: <parameter name="username" value="admin"/>

import org.testng.annotations.Parameters;
import org.testng.annotations.Test;

public class LoginTest {

    @Parameters({"username", "password"}) // names match XML param names
    @Test
    public void loginTest(String username, String password) {
        // username = "admin", password = "pass123" from XML
        System.out.println("Logging in as: " + username);
        Assert.assertNotNull(username);
    }
}
Example 2 — @Parameters on @BeforeMethod for browser setupCross-browser
// testng.xml: <parameter name="browser" value="firefox"/>

public class BrowserTest {

    WebDriver driver;

    @Parameters("browser")     // inject browser name into setup
    @BeforeMethod
    public void setup(String browser) {
        if (browser.equals("chrome")) {
            driver = new ChromeDriver();
        } else if (browser.equals("firefox")) {
            driver = new FirefoxDriver();
        }
        driver.get("https://example.com");
    }
}

import org.testng.annotations.Optional;

    // If testng.xml has NO 'env' parameter, uses default "staging"
    System.out.println("Running on: " + env);
}

Best Practices
✓Always use @Optional with sensible defaults so tests don't break when running locally without a full testng.xml.
✓Pass baseUrl, browser, and env as parameters — never hardcode these values in Java code.
✓Use test-level <parameter> to override suite-level values for specific test blocks (e.g. run one block on Firefox).
✓Don't use @Parameters for test data (multiple rows) — use @DataProvider for that (Session 6).
✓Keep parameter names consistent across all testng.xml files in your project — use a naming convention like app.browser, app.env.
Interview Q&A
Q1: What is the difference between @Parameters and @DataProvider? ▼
@Parameters reads a single value from testng.xml — best for environment config. @DataProvider feeds multiple rows of test data into the same test method, running it once per row — best for data-driven testing.
Q2: What happens if a @Parameters parameter is not defined in testng.xml? ▼
TestNG throws a TestNGException and the test fails. To prevent this, annotate the parameter with @Optional("defaultValue") so TestNG uses the default when the XML parameter is missing.
Q3: Can @Parameters be used on @BeforeClass methods? ▼
Yes. @Parameters works on any TestNG-annotated method — @Test, @BeforeMethod, @BeforeClass, @BeforeSuite, etc. This is commonly used to initialize WebDriver once in @BeforeClass with a browser parameter.
Q4: How do you pass multiple parameters to a single method? ▼
List all parameter names in the @Parameters annotation array: @Parameters({"browser", "url", "username"}). The Java method must have matching parameters in the same order as the annotation array.
Q5: Where does test-level parameter override suite-level parameter? ▼
When the same parameter name is defined at both <suite> and <test> level, the <test> level value takes precedence for classes inside that <test> block. Classes outside that block use the suite-level value.
Overview — @DataProvider (Data-Driven Testing)

@DataProvider is TestNG's built-in data-driven mechanism. It feeds multiple rows of data into a single test method — the method executes once per data row. Perfect for login combinations, form validations, and API payloads.

What
Method returning Object[][] that supplies multiple data sets to @Test
Why
Run one test with N data sets without duplicating test code — true data-driven testing
Return type
Object[][] or Iterator<Object[]> — each inner array = one method invocation
Spoken Lines in Interview
"@DataProvider returns an Object[][] where each row is one invocation of the test — enabling true data-driven testing."
"I use @DataProvider to test login with valid, invalid, and empty credentials — one test method, three executions."
"DataProvider can be placed in a separate utility class and shared across multiple test classes using dataProviderClass attribute."
"Setting parallel=true on @DataProvider runs all data rows concurrently — great for speeding up large data sets."
"Unlike @Parameters which is one value per run, @DataProvider runs the test N times — once per data row."
Code Examples
Example 1 — Basic @DataProvider with Object[][]Basics
import org.testng.annotations.DataProvider;
import org.testng.annotations.Test;

public class LoginDataTest {

    // @DataProvider returns Object[][] — each inner array = one test run
    @DataProvider(name = "loginData")  // name is referenced in @Test
    public Object[][] getLoginData() {
        return new Object[][] {
            { "admin",   "admin123",  true  },  // row 1: valid user
            { "user1",   "wrongpass", false },  // row 2: wrong password
            { "",        "",           false },  // row 3: empty fields
        };
    }

    // This test runs 3 times — once per row above
    @Test(dataProvider = "loginData")  // must match @DataProvider name

Interview Q&A
Q1: What does Object[][] mean in a DataProvider? ▼
Object[][] is a 2D array. The outer array represents test invocations (rows). The inner array represents the method arguments for that invocation (columns). So Object[3][2] means the test runs 3 times, each time receiving 2 arguments.
Q2: How do you use a DataProvider from a different class? ▼
Use dataProviderClass attribute: @Test(dataProvider="myData", dataProviderClass=DataUtil.class). The DataProvider method in the external class must be declared static for this to work.
Q3: What is the difference between @Parameters and @DataProvider? ▼
@Parameters reads a single static value from testng.xml at runtime — it does not cause the test to repeat. @DataProvider causes the test to execute once per data row — it is for multi-row data-driven execution.
Q4: Can a DataProvider run in parallel? How? ▼
Yes. Set parallel=true in @DataProvider. TestNG creates a thread per data row and runs them concurrently. You must ensure each test invocation is thread-safe — particularly that each thread has its own WebDriver instance via ThreadLocal.
Q5: How would you read test data from an Excel file for a DataProvider? ▼
Use Apache POI library to open the .xlsx file, iterate rows (skipping the header), read each cell value, populate an Object[][] and return it. This is a very common interview coding question — know the WorkbookFactory.create() pattern.

Example 3 — IRetryAnalyzer: retry failed testsRetry
import org.testng.IRetryAnalyzer;
import org.testng.ITestResult;

public class RetryAnalyzer implements IRetryAnalyzer {

    private int retryCount = 0;
    private static final int MAX_RETRY = 2;  // retry max 2 times

    @Override
    public boolean retry(ITestResult result) {
        if (retryCount < MAX_RETRY) {
            retryCount++;
            System.out.println("Retrying: " + result.getName()
                               + " attempt: " + retryCount);
            return true;   // true = retry the test
        }
        return false;  // false = final failure, stop retrying
    }
}

// Attach to @Test:
@Test(retryAnalyzer = RetryAnalyzer.class)
public void flakeyTest() { /* will retry up to 2 times if it fails */ }

    @Override

Best Practices
✓Register global listeners (screenshot, retry) in testng.xml, not via @Listeners — so they apply to all tests automatically.
✓Use IAnnotationTransformer to inject RetryAnalyzer globally — avoids adding retryAnalyzer= to every single @Test.
✓Keep retry count low (max 2) — if a test needs >3 retries, it signals a real bug, not flakiness.

Interview Q&A
Q1: What interface do you implement to take a screenshot on test failure? ▼
Implement ITestListener and override onTestFailure(ITestResult result). Inside the method, cast your WebDriver to TakesScreenshot and save the screenshot file. Register the listener in testng.xml for global coverage.
Q2: What is IRetryAnalyzer and when should you use it? ▼
IRetryAnalyzer is an interface that lets TestNG re-run a failed test up to a configured number of times. Use it for flaky tests caused by network issues or timing problems — not for tests that fail due to real bugs.
Q3: What is the difference between @Listeners and testng.xml <listeners>? ▼
@Listeners(MyListener.class) applies the listener only to the annotated test class. <listeners><listener class-name="..."/></listeners> in testng.xml applies it globally to every test in the suite — the preferred approach for screenshot and retry listeners.
Q4: What is IAnnotationTransformer? ▼
IAnnotationTransformer lets you modify TestNG annotations programmatically at runtime before tests run. The most common use is calling annotation.setRetryAnalyzer() in the transform method to globally inject retry logic without touching individual @Test annotations.
Q5: How does the retry count get reset between different test methods? ▼
Each @Test method gets its own IRetryAnalyzer instance, so retryCount naturally resets. However, if you declare retryCount as static, it shares state across tests — a common bug. Always use instance variables (not static) for retryCount.

Example 5 — ThreadLocal WebDriver for thread-safe parallel testsThreadLocal
public class ThreadSafeBase {

    // ThreadLocal ensures each thread gets its OWN WebDriver instance
    private static ThreadLocal<WebDriver> driverThread =
        new ThreadLocal<>();

    @Parameters("browser")
    @BeforeMethod
    public void setup(String browser) {
        WebDriver driver = browser.equals("chrome")
            ? new ChromeDriver()
            : new FirefoxDriver();
        driverThread.set(driver);   // store in THIS thread's slot
    }

    // Static getter — each thread retrieves ITS OWN driver
    public static WebDriver getDriver() {
        return driverThread.get();
    }

    @AfterMethod(alwaysRun = true)
    public void teardown() {
        if (driverThread.get() != null) {
            driverThread.get().quit();
            driverThread.remove(); // MUST remove to prevent memory leaks
        }
    }
}
Best Practices
✓Always call driverThread.remove() in @AfterMethod to prevent memory leaks in thread pools.
✓Use dependsOnGroups over dependsOnMethods in large suites — method names change; group names are more stable.
✓Start with parallel="tests" before parallel="methods" — it's easier to reason about and less prone to race conditions.
✓Never use shared static state (non-ThreadLocal) in parallel tests — it causes random, hard-to-debug failures.
✓Combine parallel="tests" with per-test <parameter name="browser"> for a clean cross-browser parallel setup.
Interview Q&A
Q1: What happens when a dependent test fails in TestNG? ▼
When a test specified in dependsOnMethods fails, all tests that depend on it are automatically marked as SKIPPED (not failed). They will not execute at all. This prevents cascading failures in an e2e flow.
Q2: What are the four parallel modes in TestNG? ▼
methods — each @Test method runs in its own thread. classes — all methods in a class share one thread, but classes run in parallel. tests — each <test> tag in testng.xml gets its own thread. instances — each class instance gets its own thread.
Q3: Why is ThreadLocal needed for parallel WebDriver? ▼
Without ThreadLocal, all parallel threads share the same WebDriver object. One thread might call driver.get() while another calls driver.findElement(), causing NullPointerException or wrong-page errors. ThreadLocal gives each thread its own private WebDriver copy.
Q4: What is the difference between dependsOnMethods and dependsOnGroups? ▼
dependsOnMethods names a specific method — fragile because renaming the method breaks the dependency. dependsOnGroups names a group tag — more maintainable because many methods can be in that group and any of them can satisfy the dependency.
Q5: How do you run the same test class on Chrome and Firefox in parallel? ▼
Create two <test> blocks in testng.xml — one with <parameter name="browser" value="chrome"/> and one with value="firefox". Both include the same class. Set parallel="tests" and thread-count="2". Use ThreadLocal<WebDriver> and @Parameters("browser") in the base class. Each <test> block runs its own thread with its own browser.
