S1 - CI/CD Fundamentals & GitHub Actions Basics for Playwright-TS

Playwright is Microsoft's modern E2E testing framework with native parallelism, auto-waiting, and built-in reporters. Pairing it with GitHub Actions gives you a zero-config CI/CD platform that runs Playwright tests across Chromium, Firefox, and WebKit on every PR.

What / Why / When / Where / How

WhatAutomated build/test pipeline for Playwright-TS that runs on every git event.
WhyCatch UI regressions early, run cross-browser tests on every PR, no manual ops.
WhenEvery PR (smoke), every merge (regression), every release (full suite).
WhereGitHub-hosted runners (ubuntu/windows/macos) or self-hosted machines.
HowCreate .github/workflows/*.yml that installs Node + Playwright + browsers, then runs npx playwright test.

GitHub Actions
GitHub's native CI/CD platform, YAML-driven, free tier 2000 min/month.
Workflow
Top-level YAML file in .github/workflows/ describing automation.
Runner
Ephemeral VM (Ubuntu/Windows/macOS) that executes workflow jobs.
Playwright
Microsoft's E2E test framework with native parallelism and 3 browsers.
@playwright/test
Test runner package with describe/test/expect API.
Project
playwright.config.ts unit - typically one per browser.
Trace
Recording of test execution - opens in trace viewer for debugging.
HTML Reporter
Built-in Playwright HTML report at playwright-report/.
Action
Reusable unit (e.g., actions/checkout@v4 or actions/setup-node@v4).
Browser Cache
~/.cache/ms-playwright keyed by Playwright version.

Interview One-Liners

Playwright + GitHub Actions = cross-browser E2E in CI without external services.
Workflows live in .github/workflows/ - YAML-driven, version-controlled.
Use actions/setup-node@v4 for Node + automatic npm cache.
Always cache ~/.cache/ms-playwright - browser downloads are 300 MB+.
Run npx playwright install --with-deps to install browsers + system deps.
Built-in HTML reporter gives screenshots, traces, and video for free.

# First-day issues with Playwright in CI

1. Browsers missing
   - Run: npx playwright install --with-deps
   - --with-deps installs Linux system libs

2. Tests timeout in CI but pass locally
   - Increase timeout in playwright.config.ts
   - Use waitForLoadState('networkidle')

Best Practices Checklist

  • ✅ Use actions/setup-node@v4 with cache: 'npm' - skip slow npm install.
  • ✅ Always run npx playwright install --with-deps - missing libs cause silent fails.
  • ✅ Cache ~/.cache/ms-playwright keyed by Playwright version.
  • ✅ Use retries: 2 in CI to absorb transient flakes.
  • ✅ Set workers: 2 in CI - matches free runner CPU count.
  • ✅ Upload playwright-report artifact with if: always() - capture failures.
  • ✅ Use trace: 'on-first-retry' for free debug info on flaky tests.
  • ✅ Pin action versions (@v4) - never @main.
  • ✅ Use 'github' reporter alongside 'html' for PR annotations.
  • ✅ Treat playwright.config.ts as production code - review in PRs.

S2 - Workflow Syntax & YAML Structure

Every workflow is a YAML file with top-level keys: name, on, env, jobs. YAML is indent-sensitive (2 spaces) and case-sensitive. Master the syntax so you can debug Playwright workflows by reading them.

What / Why / When / Where / How

WhatYAML grammar GitHub Actions uses to declare automation.
WhyWrong indentation = silent failure with cryptic errors.
WhenEvery time you write or edit a workflow file.
WhereAll .github/workflows/*.yml files.
How2-space indent, key: value pairs, lists with '-'.

Keyword Glossary

YAML
Yet Another Markup Language - human-readable data format.
name
Display name shown in the Actions UI.
on
List of trigger events (push, pull_request, schedule).
env
Environment variables visible to jobs/steps.
jobs
Map of job-id to job config.
runs-on
Runner type for a job.
steps
Sequential list of commands or actions.
uses
Reference to a reusable action.
with
Inputs passed to an action via 'uses'.
needs
Declares job dependency.

Interview One-Liners

YAML is indent-sensitive - 2 spaces, never tabs.
name is cosmetic; jobs.<id> is the real identifier.
on accepts string, list, or map - all three valid.
env can be set at workflow, job, or step level.
Use ${{ }} expression syntax to reference context.
Quote strings containing : or { to avoid mis-parse.

S3 - Triggers, Events & Schedules for Playwright

Workflows run only when triggered. Playwright workflows typically use pull_request for PR gates, push for main validation, schedule for nightly regression, and workflow_dispatch for manual runs (pick browser/env).

What / Why / When / Where / How

WhatEvents that start a Playwright workflow run.
WhyDifferent test suites for different events - smoke on PR, full regression nightly.
WhenDefine triggers in 'on:' at the top of every workflow.
WhereTop-level 'on:' key in workflow YAML.
HowSingle event, list, or map with filters (branches/paths/types).

Keyword Glossary

push
Triggered on every git push.
pull_request
Triggered on PR open/sync/reopen/ready_for_review.
schedule
Cron-style scheduled triggers (UTC).
workflow_dispatch
Manual trigger via UI or API.
workflow_call
Make workflow reusable by other workflows.
repository_dispatch
Trigger via REST API (external systems).
paths
Filter that runs only when matching files change.
branches
Filter that runs only on listed branches.
types
Filter on event subtype (opened, synchronize).
concurrency
Cancel in-progress runs on same group.

Interview One-Liners

Use push for main-branch validation, pull_request for PR gates.
schedule uses cron syntax in UTC - always document timezone.
workflow_dispatch lets QA trigger ad-hoc runs with browser/env choice.
Filter by paths to skip workflow when only docs change.
Use concurrency to cancel superseded PR runs and save minutes.
workflow_call turns a workflow into a reusable function.

export class LoginPage {

  readonly username: Locator;

  readonly error: Locator;

    await this.password.fill(pass);
    await this.loginButton.click();
  }

  async expectErrorContains(text: string) {
    await expect(this.error).toContainText(text);
  }
