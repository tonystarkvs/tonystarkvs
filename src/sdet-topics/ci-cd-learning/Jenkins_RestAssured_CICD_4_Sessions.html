FOUNDATION — Jenkins + CI/CD + Rest Assured Basics

Understand Jenkins, CI/CD, and how Rest Assured + TestNG + Maven API tests run inside a pipeline. (Beginner • Session 1 of 4)

Overview — What / Why / When / Where / How

What Jenkins is an automation server. Rest Assured is a Java library that lets us call REST APIs using a clean given-when-then style.
Why API tests are fast, stable, and catch backend issues early — perfect for CI/CD where speed matters.
When Run them on every code push, before UI tests, and as part of nightly regression.
Where On a Jenkins server/agent with Java, Maven, Git, and network access to API endpoints.
How Developer pushes code ➜ Jenkins triggers job ➜ Maven compiles ➜ TestNG runs Rest Assured tests ➜ reports archived.

Keyword Glossary

CI
Continuous Integration with frequent merges and automated checks.
CD
Continuous Delivery or Deployment of builds to environments.
Rest Assured
Java library for clean API testing using given/when/then.
TestNG
Java test framework used to run, group, and parallelize tests.
Maven
Java build tool that manages dependencies and runs Surefire.
baseURI
Common base URL like https://qa.api.example.com used by all tests.
Endpoint
Path like /users or /orders appended to baseURI.
Status Code
HTTP response code such as 200, 201, 400, 500.
JSON Path
Expression to read values from a JSON response body.
Smoke API Test
Quick critical check like GET /health or GET /users.

Interview One-Liners

Rest Assured lets us write API tests in plain English style: given, when, then.
API tests are the fastest feedback layer in any CI pipeline.
Selenium tests UI; Rest Assured tests the backend behind that UI.
I always assert status code and at least one body field, never just status.
Failed API tests usually point to a real backend bug, not flakiness.

package com.learn.base;

import io.restassured.RestAssured;
import io.restassured.filter.log.RequestLoggingFilter;
import io.restassured.filter.log.ResponseLoggingFilter;
import org.testng.annotations.BeforeClass;

public class BaseApiTest {

    @BeforeClass
    public void apiSetUp() {
        // Base URL for all tests — overridden by Jenkins env BASE_URI
        RestAssured.baseURI = System.getenv().getOrDefault(
                "BASE_URI", "https://qa.api.example.com");

        // Log request and response
        RestAssured.filters(new RequestLoggingFilter(), new ResponseLoggingFilter());
    }
}
● src/test/java/com/learn/api/UserApiTest.java 
package com.learn.api;

import com.learn.base.BaseApiTest;
import io.restassured.response.Response;
import org.testng.Assert;
import org.testng.annotations.Test;

import static io.restassured.RestAssured.given;

public class UserApiTest extends BaseApiTest {

    @Test(description = "Smoke - GET /users returns 200")
    public void getUsersSmoke() {
        Response res = given()
                .header("Accept", "application/json")
          .when()
                .get("/users")
          .then()
                .statusCode(200)
                .extract().response();

        // Assert at least one user exists in the response
        Assert.assertTrue(res.jsonPath().getList("data").size() > 0);
    }
}

Best Practices

  • ✓Start with a small stable smoke API pack before adding regression.
  • ✓Use Maven Surefire to run TestNG — keep it standard.
  • ✓Use RestAssured.baseURI once in BaseApiTest instead of repeating URLs.
  • ✓Always assert status code AND at least one important body field.
  • ✓Log request/response for fast remote debugging.
  • ✓Externalize URLs, tokens, and users into config files or Jenkins credentials.
  • ✓Keep the smoke API pack under a minute for quick CI feedback.
  • ✓Tag tests with TestNG groups like smoke/regression for clean filtering.
  • ✓Never commit real tokens or secrets to Git.
  • ✓Document ownership of API failures: app team, integration team, or data team.

Interview Prep — Q&A

Q1. What is Jenkins in simple words?
A: Jenkins is a tool that automatically runs build and test steps without manual effort.
Q2. Why use Rest Assured with Jenkins?
A: Because Jenkins can run our API tests automatically and share results across the team.
Q3. UI tests vs API tests?
A: UI tests validate the browser flow; API tests validate the backend directly — API tests are faster and more stable.
Q4. What is a TestNG suite?
A: An XML file that decides which tests run, with which parameters and groups.
Q5. Smoke vs regression in one line?
A: Smoke is quick critical checks; regression is broad coverage that runs less often.

PIPELINE DESIGN — Jenkinsfile, Parameters, TestNG Suites

Build a real-world, parameterized Jenkins pipeline file using environments, suites, and BASE_URI parameters. (Core • Session 2 of 4)

Overview — What / Why / When / Where / How

What A Jenkinsfile defines exactly what Jenkins runs — checkout, build, suite, archive — all as code.
Why It removes manual UI drift and keeps the pipeline reviewable in Git like normal source code.
When When many people share the pipeline and the same job must run across environments.
Where In the project repo root so the pipeline travels with the API framework.
How Add stages, parameters, environment variables, and post actions in a declarative pipeline.

A Jenkinsfile makes the API pipeline transparent and reviewable.

CODE EXAMPLES — S2 · Pipeline Design
● Jenkinsfile 
pipeline {
  agent any
  tools { jdk 'JDK17'; maven 'Maven3' }

  parameters {
    choice(name: 'ENV', choices: ['qa', 'uat'], description: 'Target environment')
    choice(name: 'SUITE', choices: ['testng-smoke.xml', 'testng-regression.xml'], description: 'TestNG suite file')
  }

  environment {
    QA_URI  = 'https://qa.api.example.com'
    UAT_URI = 'https://uat.api.example.com'
  }

  stages {
    stage('Checkout') { steps { checkout scm } }

    stage('Run Selected Suite') {
      steps {
        script {
          def uri = params.ENV == 'qa' ? env.QA_URI : env.UAT_URI
          sh "BASE_URI=${uri} mvn -B clean test -DsuiteXmlFile=${params.SUITE}"
        }
      }
    }
  }

  post {
    always {
      archiveArtifacts artifacts: 'target/surefire-reports/**, reports/**', allowEmptyArchive: true
    }
  }
}

package com.learn.utils;

import java.io.FileInputStream;
import java.util.Properties;

public class ConfigReader {
    private static final Properties props = new Properties();

    static {
        try (FileInputStream fis = new FileInputStream("src/test/resources/config.properties")) {
            props.load(fis);
        } catch (Exception e) {
            throw new RuntimeException("Could not load config", e);
        }
    }

    public static String get(String key) {
        // Jenkins env overrides config file
        String envKey = key.toUpperCase().replace('.', '_');
        return System.getenv().getOrDefault(envKey, props.getProperty(key));
    }
}

Best Practices

  • ✓Keep Jenkinsfile in repo root and review changes through pull requests.
  • ✓Expose ENV, SUITE, and BASE_URI as parameters instead of duplicating jobs.
  • ✓Keep environment URL mapping in one config file.
  • ✓Use TestNG groups like smoke, sanity, regression for filtering.
  • ✓Set sensible timeouts to stop stuck stages.
  • ✓Publish reports in post always so failures still expose artifacts.
  • ✓Add comments in Jenkinsfile for fresher onboarding.
  • ✓Do not mix deploy logic with long regression in one stage.
  • ✓Name stages by business outcome, not step1 or task2.
  • ✓Prefer pipeline code over manual UI tweaks for auditability.

DATA + EXECUTION STRATEGY — Test Data, DataProvider, Auth, Parallel API Runs

Manage JSON/Excel test data, configure TestNG DataProviders, fetch runtime OAuth2 tokens, and execute parallel suites. (Practical • Session 3 of 4)

Overview — What / Why / When / Where / How

What A real CI pipeline needs predictable test data, secure authentication, and fast execution for API tests.
Why Even a perfect API test fails when data is stale, tokens expire, or runs are too slow for the team.
When Apply once your smoke API pack is stable and you want broader, real-time runs.
Where Across QA/UAT, multiple regions, OAuth-protected APIs, and parallel TestNG suites.
How Externalize data, fetch tokens at runtime, use TestNG DataProvider, and configure parallel suites.

Interview One-Liners

Stable API automation is as much about data and tokens as it is about endpoints.
Parallel execution saves time only when tests are isolated.
I refresh tokens once per suite, not for every test.
I avoid reusing mutable accounts across parallel data rows.
A flaky API pipeline is often a token or data issue, not a real bug.

● src/test/java/com/learn/utils/AuthHelper.java 
package com.learn.utils;

import io.restassured.http.ContentType;
import static io.restassured.RestAssured.given;

public class AuthHelper {
    private static String cachedToken;

    public static synchronized String getToken() {
        if (cachedToken != null) return cachedToken;

        // OAuth2 client credentials grant — fetched once per suite
        cachedToken = given()
                .contentType(ContentType.URLENC)
                .formParam("grant_type", "client_credentials")
                .formParam("client_id", System.getenv("CLIENT_ID"))
                .formParam("client_secret", System.getenv("CLIENT_SECRET"))
          .when()
                .post("/oauth/token")
          .then()
                .statusCode(200)
                .extract().path("access_token");

        return cachedToken;
    }
}

import com.learn.base.BaseApiTest;

Best Practices

  • ✓Externalize API test data into JSON, Excel, or properties.
  • ✓Use unique users/records for any flow that modifies state.
  • ✓Refresh tokens with a thread-safe helper to avoid duplicate calls.
  • ✓Start parallel=methods with small thread-count and scale up carefully.
  • ✓Use Hamcrest matchers for body assertions — more readable than custom code.
  • ✓Never log full Bearer tokens in console output.
  • ✓Keep test data names clear like qa-buyer-01 or uat-admin-readonly.
  • ✓Mark idempotent tests as safe-for-parallel, others as serial.
  • ✓Investigate flaky tests; do not just retry forever.
  • ✓Prefer deterministic seed data over random uncontrolled values.

Interview Prep — Q&A

Q1. Why is test data important in API CI?
A: Pipelines are only useful with predictable, valid input every run.
Q2. How do you handle auth tokens?
A: Fetch once per suite via OAuth2, cache it, and inject as Bearer header in every request.
Q3. When to use parallel API execution?
A: When tests are independent, env is stable, and feedback must be fast.
Q4. Why externalize test data?
A: So we can update data without touching test code and reuse it across environments.
Q5. What is idempotent in API testing?
A: An action that returns the same result if repeated — safer for parallel and retry runs.

What Real-world CI patterns: reports, logging, scheduled runs, secure credentials, and cleanup.
Why A working pipeline is not enough; it must be secure, debuggable, and trusted by the team.

  • ✓Archive artifacts on every run, not just successful ones.
  • ✓Attach request and response details on failure in your report.

  • ✓Use cron sensibly — nightly regression at off-hours.

  • ✓Investigate flaky tests; do not hide them with retries.
  • ✓Notifications must include build, env, suite, and report link.
  • ✓Prepare simple interview answers — trigger, stages, env, artifacts, results.
