GitHub Actions — What & Why

CI/CD foundations, core vocabulary, your first workflow

What / Why / When / Where / How

WhatGitHub Actions is an automation platform built into GitHub that runs code automatically when events happen.
WhyAutomate building, testing, and shipping code (CI/CD) to catch bugs early.
WhenOn code push, pull request creation/sync, manual dispatch, or schedule.
WhereRuns on cloud virtual machines (Ubuntu, Windows, macOS) provided by GitHub.
HowDefine a YAML file under .github/workflows/ with triggers, jobs, and steps.

Keyword Glossary

Workflow
YAML file in .github/workflows/ that defines your automation
Trigger
Event that starts the workflow (push, schedule, manual)
Job
Group of steps running on the same machine
Step
Single task inside a job (run a command, use an action)
Runner
The virtual machine (ubuntu-latest) that executes your job
Action
Reusable plugin from GitHub Marketplace (e.g. actions/checkout)
YAML
Human-readable config format using indentation (spaces, NOT tabs)

Interview One-Liners

GitHub Actions is an automation engine inside GitHub that runs tests and deployments automatically.
A workflow is a YAML file telling GitHub what to do and when to do it.
CI means every code push is automatically built and tested before merging.
A runner is a cloud VM GitHub provides to execute your jobs.
Playwright lets us automate real browser interactions across Chrome, Firefox, and Edge.
TypeScript is JavaScript with types — it catches mistakes before your code runs.
A step is the smallest unit of work inside a GitHub Actions job.

Best Practices Checklist

  • Pin action versions with @v4, never @latest — prevents surprise breaking changes
  • Store all credentials in GitHub Secrets — never hardcode passwords in YAML
  • Use npm ci instead of npm install in CI — reproducible, faster
  • Always run npx playwright install --with-deps to get OS-level browser libraries
  • Name every step clearly — you'll thank yourself debugging failures at 2 AM
  • Use ubuntu-latest as your default runner — cheapest, fastest, most compatible
  • Set branch filters on push triggers to avoid running on every branch

Interview Prep Q&A

Q1. Q1. What is a GitHub Actions workflow?
A YAML file in .github/workflows/ that defines automation rules — what triggers it (on:), where it runs (runs-on:), and what steps to execute. It is the blueprint for your CI/CD pipeline.
Q2. Q2. Difference between 'uses' and 'run' in a step?
'uses' calls a pre-built action from GitHub Marketplace (like actions/checkout@v4) — it's importing a reusable plugin. 'run' executes raw shell commands directly on the runner VM, like typing in a terminal.
Q3. Q3. Why 'npm ci' instead of 'npm install' in CI?
npm ci does a clean install strictly from package-lock.json ensuring identical dependencies every run. npm install may update packages and is slower. Consistency and reproducibility are critical in CI environments.
Q4. Q4. What does actions/checkout@v4 do and why is it needed?
It downloads (clones) your repository code onto the runner VM. Without it the runner has an empty workspace and cannot find your test files. It must be the very first step in almost every job.
Q5. Q5. How do you securely use passwords or API keys in a workflow?
Store them in GitHub > Repository > Settings > Secrets and Variables > Actions. Reference them as ${{ secrets.YOUR_SECRET_NAME }} in YAML. GitHub automatically masks secret values in all log output so they are never exposed.

Best Practices Checklist

  • Always use UTC timezone for cron — GitHub Actions runs UTC regardless of your location
  • Schedule heavy regression at night — smoke should be on every push
  • Include workflow_dispatch alongside schedule so you can trigger manually anytime
  • Use --grep tags to route Smoke/Sanity/Regression to the right trigger type
  • Filter push triggers by paths (tests/**) to skip documentation-only changes
  • Use github.event_name conditional logic to run different packs per trigger
  • Keep daily smoke fast (under 10 min) — slow nightly runs are okay

Interview Prep Q&A

Q1. Q1. What is workflow_dispatch and why do SDETs use it?
workflow_dispatch adds a manual 'Run workflow' button in the GitHub Actions tab. SDETs use it to run specific test packs on demand without pushing code — useful before a release, debugging flaky tests, or running against a specific environment.
Q2. Q2. Explain cron syntax '0 2 * * 1-5'
Runs at 2:00 AM UTC every weekday (Monday through Friday). Field breakdown: 0=minute, 2=2 AM, *=any day-of-month, *=any month, 1-5=Mon to Fri. The dash creates a range.
Q3. Q3. How do you run different test packs per trigger type?
Use github.event_name in step-level if: conditions. 'if: github.event_name == schedule' runs regression nightly. 'if: github.event_name == push' runs smoke on every commit. Combined with --grep tags, each trigger runs the appropriate suite.
Q4. Q4. Difference between schedule and push triggers?
Push runs immediately when code is pushed — event-driven, linked to a specific commit. Schedule runs on a time basis (cron) even without code changes — ideal for health checks and nightly regression against a stable environment.
Q5. Q5. How do workflow_dispatch inputs work?
Inputs define a form shown before manual runs. Types: choice (dropdown), string (text), boolean (checkbox), number. Values are accessed at runtime via ${{ github.event.inputs.INPUT_NAME }}. The 'required' field prevents running with empty values.

Interview Prep Q&A

Q1. Q1. What is playwright.config.ts and its most important fields?
The central TypeScript config file for Playwright. Key fields: testDir (where tests live), projects (which browsers), use (shared options like baseURL, screenshot), retries (auto-retry), workers (parallelism), reporter (output format). defineConfig wrapper provides full IntelliSense autocomplete.
Q2. Q2. Difference between timeout and expect.timeout?
timeout is the maximum time an entire test can run before being killed (e.g. 30 seconds). expect.timeout is how long a single assertion waits for a condition (e.g. waiting for an element). Both need separate tuning — test timeout handles slow pages, expect timeout handles slow rendering.
Q3. Q3. How do retries work in CI?
When retries: 2 is set and a test fails, Playwright reruns it up to 2 more times. If it passes on retry, it is marked 'flaky' — not failed. trace: 'on-first-retry' records a trace file on the first retry for detailed debugging.
Q4. Q4. What is the projects array?
Each entry defines a browser to run your tests against. You name it (name: 'chromium') and specify settings using devices[] presets from @playwright/test. Running npx playwright test executes all tests once per project — 3 projects means 3 browser runs.
Q5. Q5. How do you dynamically set baseURL for different environments?
Read process.env.TEST_ENV and map it to the correct URL using a lookup object. Set TEST_ENV in the GitHub Actions workflow env: block. The same test files then run against dev, qa, uat, or staging without any code changes.

Interview Prep Q&A

Q1. Q1. Why is cross-browser testing important for SDETs?
Different browsers use different rendering engines (Blink for Chrome/Edge, Gecko for Firefox, WebKit for Safari). CSS rendering, JS APIs, and form behavior can differ. A bug only in Firefox could affect millions of users — cross-browser testing catches these before release.
Q2. Q2. How does Playwright handle cross-browser without installing many browsers?
Playwright's projects array defines each browser. npx playwright install downloads custom builds of Chromium, Firefox, and WebKit — no separate browser installations needed. Each project runs the full test suite independently.
Q3. Q3. How do you run tests against a specific environment in CI?
Set the BASE_URL environment variable in the workflow (env: BASE_URL: https://uat.myapp.com) or use a mapping script reading TEST_ENV to set BASE_URL. playwright.config.ts reads process.env.BASE_URL so all tests point to the correct server.
Q4. Q4. Difference between dev, qa, uat, and staging environments?
Dev is for developers — unstable, run smoke only. QA is stable for testers — run sanity and regression. UAT is client-facing for acceptance sign-off. Staging mirrors production exactly — final gate before release, full regression.
Q5. Q5. How do you handle a test that should only run in specific environments?
Use test.skip(condition, 'reason') at the start of a test with process.env.TEST_ENV === 'dev' as the condition. Alternatively use if(env === 'staging') blocks to conditionally declare tests. Both show skipped tests in the report so they remain visible.

What / Why / When / Where / How

WhatSplitting tests into packs based on execution speed, critical features, and coverage.
WhyBalance fast feedback on code changes with thorough coverage of edge cases.
WhenEvery push (Smoke), post-deployment (Sanity), and nightly (Regression).

Each test pack is a separate workflow job — a failing smoke gate blocks sanity from starting.
Test tagging with @smoke/@sanity/@regression is the industry-standard Playwright organization.
A smoke test that takes 30 minutes is NOT a smoke test — keep critical checks fast.

Q1. Q1. Difference between Smoke, Sanity, and Regression testing?

Use 'needs: sanity' in the regression job. GitHub Actions ensures regression only starts when sanity completes successfully. If sanity fails, regression is automatically skipped — saving compute time and cost.

POM is a design pattern where each page gets a TypeScript class containing all locators and interaction methods. Tests use these classes instead of raw selectors. If a selector changes, you fix it in one class rather than every test file.

WhatGitHub Actions strategy that auto-generates parallel jobs for combinations of variables.
WhyTest multiple browsers and environments in parallel without repeating YAML code.

Q1. Q1. What is strategy.matrix and why use it?

Parallel vs Sequential Execution

jobs.needs, parallel jobs, and when to use each

What / Why / When / Where / How

WhatRunning jobs at the same time (parallel) or in order (sequential using needs).
WhyGate slow tests behind fast tests (smoke gate) and run independent tests quickly.
WhenGating regression/sanity runs behind smoke success, and running cross-browser tests concurrently.
WhereConfigured using the needs: keyword and matrix definitions in the workflow file.
HowAdd needs: smoke to sanity jobs, and let browser matrix run in parallel.

Keyword Glossary

needs:. Total time = sum of all jobs. Use when job B depends on job A, or when you want fast-fail gating. needs:
Makes a job wait for another to complete successfully
needs: [job1, job2]
Wait for MULTIPLE jobs (all must succeed)
if: always()
Run regardless of previous job result
if: success()
Run only if all needed jobs passed (default)
if: failure()
Run only if any needed job failed

Interview One-Liners

Parallel jobs run simultaneously — a 3-browser test suite takes the time of ONE browser run.
The 'needs' keyword creates sequential dependency — job B starts only when job A succeeds.
if: always() makes a job run regardless of upstream failures — essential for report collection.
workers: 4 in playwright.config.ts runs 4 tests in parallel WITHIN a single job.
Parallel jobs are independent VMs — they cannot share files directly (use artifacts instead).
Sequential gating with 'needs: smoke' saves time on regression if smoke already failed.
Total parallel time = max(durations). Total sequential time = sum(durations).

  • Always use if: always() on report collection — you need reports most when tests fail
  • Set workers: 2 in CI — more workers means more memory and potential flakiness
  • Don't share state between parallel jobs — use artifacts for files, outputs for small values
  • Log parallel job start/end times to profile where your pipeline is slowest
  • Use matrix + fail-fast: false instead of manually writing parallel browser jobs

Interview Prep Q&A

Q1. Q1. Difference between parallel jobs and parallel workers?
Parallel jobs are separate runner VMs running simultaneously but cannot share memory. Workers (set in playwright.config.ts) are threads within a single job on one VM running multiple tests in parallel browser contexts. Jobs-level parallelism scales across cloud VMs; workers-level parallelism scales across CPU cores.
Q2. Q2. When should you use sequential instead of parallel?
When job B genuinely depends on job A's success. Main patterns: gating (run regression only if smoke passes), dependency (deploy only after build), and resource protection (avoid overloading a test server). If jobs are independent, always prefer parallel for speed.
Q3. Q3. What does if: always() do and why is it critical for reports?
'if: always()' makes a job or step run regardless of whether upstream jobs passed or failed. Without it, a report-collection job would be skipped when tests fail — exactly when you need the report most.
Q4. Q4. How do you pass data between parallel jobs?
Use artifacts for files (upload in one job, download in another). Use job outputs for small values: set 'outputs:' on the source job, reference ${{ needs.JOB_NAME.outputs.OUTPUT_NAME }} in downstream jobs. Jobs cannot share memory — they run on separate VMs.
Q5. Q5. What is fullyParallel in playwright.config.ts?
fullyParallel: true makes Playwright run every test across every file in parallel simultaneously (limited by workers count). fullyParallel: false (default) runs test files sequentially. Use fullyParallel: true for maximum speed in CI when tests are truly isolated.

Interview Prep Q&A

Q1. Q1. How do you explain this pipeline in an SDET interview?
We have a 5-stage pipeline: (1) Trigger layer supporting manual dispatch, scheduled cron, and push/PR auto-triggers. (2) Sequential smoke gate validating the app is alive. (3) Parallel sanity matrix running all browsers simultaneously after smoke passes. (4) Environment-protected regression requiring human approval before staging. (5) Report collection and notification layer that always runs to capture results and alert the team.
Q2. Q2. Why sequential smoke gate instead of everything parallel?
Fail-fast principle: if the app is down, smoke fails in 3 minutes and saves 30 minutes of sanity+regression from running. In parallel, all jobs waste compute time on a broken environment. The gate ensures you get smoke results first, giving the fastest possible feedback on critical path functionality.
Q3. Q3. How do you add a new environment (e.g. performance) to this pipeline?
Add 'performance' to the workflow_dispatch inputs options list. Add a URL mapping in the environment config. Add a GitHub Environment named 'performance' in Repo Settings if approval gates are needed. The matrix and smoke gate jobs automatically support new environments through the TEST_ENV variable — no other YAML changes required.
Q4. Q4. What happens when a nightly schedule triggers this workflow?
github.event_name = 'schedule'. Default ENV (qa) and PACK (smoke) apply. Smoke gate runs → passes → sanity matrix runs across all browsers → regression job runs (the if: condition passes for schedule) → report collection always runs → if anything failed, Slack notification fires.
Q5. Q5. How do you debug a test that passes locally but fails only in CI?
Download the trace artifact from the failed GitHub Actions run. Open with 'npx playwright show-trace trace.zip' — shows exact screenshots, network calls, console logs at each step. Common CI causes: different base URL (check BASE_URL), headless differences, timing issues (CI is slower — increase timeouts), missing test data (database not seeded). Adding DEBUG=pw:api to the run step also shows detailed Playwright logs.

Interview Prep Q&A

Q1. Q1. What makes a test flaky and how do you prevent it?
A flaky test sometimes passes and sometimes fails without code changes. Causes: hardcoded sleeps (use event-based waits), shared state between tests (use isolated data), timing dependencies (increase expect timeouts), environment inconsistency. Prevention: strict test isolation, data-testid selectors, proper async/await usage, and Playwright's built-in auto-waiting.
Q2. Q2. How do you handle authentication in a large Playwright test suite?
Use Playwright's storageState. Create a setup project that logs in once and saves cookies/localStorage to .auth/user.json. All test projects set 'use: { storageState: .auth/user.json }' to start every test already authenticated. Saves 2-5 seconds per test — for 100 tests that's 5-8 minutes of pipeline time saved.
Q3. Q3. How do you test error states without breaking the real API?
Use page.route() to intercept requests and return custom responses — simulate 500 errors, empty responses, slow responses, and network failures. Gives full control over edge cases hard to reproduce with real data, and tests run faster because they don't make real network calls.
Q4. Q4. What would you improve first in a slow unreliable CI pipeline?
First: identify flaky tests by looking at high retry rates in the report — fix root causes. Second: measure which jobs take longest and optimise the slowest. Third: add stored auth state to skip repeated logins. Fourth: review the matrix — are all combinations justified? Fifth: check workers setting for parallelism. Sixth: cache node_modules with actions/setup-node cache.
Q5. Q5. How would you introduce Playwright CI to a team with no existing automation?
Start with a proof of concept: 3-5 smoke tests for critical user journeys. Set up GitHub Actions with a simple push trigger. Show the team the HTML report after a run. Gradually add sanity tests as confidence grows. Add browser matrix after smoke+sanity are stable. Introduce schedule triggers and environment gates last. Key: start small, show value early, expand incrementally.
