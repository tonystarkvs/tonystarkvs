S1 - CI/CD Fundamentals & GitHub Actions Basics

Continuous Integration (CI) merges code frequently and validates each change with automated builds + tests. Continuous Delivery (CD) ships every green build. GitHub Actions is GitHub's native, YAML-driven CI/CD platform - no external server required.

What / Why / When / Where / How

WhatAutomated build/test/deploy pipeline triggered by git events.
WhyCatch bugs early, reduce manual ops, deploy with confidence.
WhenEvery PR (smoke), every merge (regression), every release (full).
WhereGitHub-hosted runners or self-hosted machines.
HowCreate .github/workflows/*.yml - GitHub Actions reads and runs it.

Keyword Glossary

CI
Continuous Integration - merge + validate frequently.
CD
Continuous Delivery/Deployment - auto-ship passing builds.
GitHub Actions
GitHub's native CI/CD platform, YAML-driven.
Workflow
Top-level YAML file describing automation.
Runner
Machine that executes workflow jobs.
Job
Group of steps run on the same runner.
Step
Single command or action inside a job.
Action
Reusable unit (e.g., actions/checkout@v4).
Pipeline
End-to-end automation from commit to deploy.
Smoke Test
Fast critical-path check on every PR.

Interview One-Liners

CI/CD is git-driven automation - push code, get verdict.
GitHub Actions lives in .github/workflows/ inside your repo.
Workflows are YAML files - human-readable, version-controlled.
Runners are ephemeral VMs that spin up per job.
Actions are reusable building blocks from marketplace.
Treat workflow files as production code - review and version.

Best Practices Checklist

  • ✅ Keep one workflow per purpose (smoke.yml, regression.yml).
  • ✅ Pin action versions (actions/checkout@v4 - never @main).
  • ✅ Use 'name:' on every job/step for readable logs.
  • ✅ Cache Maven dependencies for faster builds.
  • ✅ Store secrets in repo settings, never in YAML.
  • ✅ Use ubuntu-latest unless you need Windows/macOS specifics.
  • ✅ Add workflow_dispatch so you can manually trigger.
  • ✅ Set 'fail-fast: false' on matrix jobs.
  • ✅ Use branch protection to make CI a merge gate.
  • ✅ Review workflow files in PR like production code.

S2 - Workflow Syntax & YAML Structure

Every workflow is a YAML file with top-level keys: name, on, env, jobs. YAML is indentation-sensitive (2 spaces) and case-sensitive. Master the syntax so you can debug by reading.

What / Why / When / Where / How

WhatYAML grammar GitHub Actions uses to declare automation.
WhyWrong indentation = silent failure with cryptic errors.
WhenEvery time you write or edit a workflow file.
WhereAll .github/workflows/*.yml files.
How2-space indent, key: value pairs, lists with '-'.

Keyword Glossary

YAML
Yet Another Markup Language - human-readable data format.
name
Display name shown in the Actions UI.
on
List of trigger events (push, pull_request, schedule).
env
Environment variables visible to jobs/steps.
jobs
Map of job-id to job config.
runs-on
Runner type for a job.
steps
Sequential list of commands or actions.
uses
Reference to a reusable action.
with
Inputs passed to an action via 'uses'.
needs
Declares job dependency.

Interview One-Liners

YAML is indent-sensitive - 2 spaces, never tabs.
name is cosmetic; jobs.<id> is the real identifier.
on accepts string, list, or map - all three are valid.
env can be set at workflow, job, or step level.
Use ${{ }} expression syntax to reference context.
Quote strings containing : or { to avoid mis-parse.

Best Practices Checklist

  • ✅ Use 2-space indentation (never tabs).
  • ✅ Quote strings with special chars (:, {, *) to be safe.
  • ✅ Use 'name:' on every job and step - cleaner logs.
  • ✅ Set 'timeout-minutes:' on jobs - prevent hung runners.
  • ✅ Use 'env:' at lowest scope needed.
  • ✅ Validate with actionlint before pushing.
  • ✅ Use 'needs:' to express dependencies clearly.
  • ✅ Pass data between jobs with outputs, not artifacts when possible.
  • ✅ Use 'if:' conditions to skip irrelevant jobs/steps.
  • ✅ Keep workflows under 200 lines - split into reusable workflows.

S3 - Triggers, Events & Schedules

Workflows run only when triggered. GitHub Actions supports 30+ events: push, pull_request, schedule (cron), workflow_dispatch (manual), workflow_call (reusable), repository_dispatch (API). Picking the right trigger is half the design.

What / Why / When / Where / How

WhatEvents that start a workflow run.
WhyDifferent test suites for different events - smoke on PR, full on schedule.
WhenDefine triggers in 'on:' at the top of every workflow.
WhereTop-level 'on:' key in workflow YAML.
HowSingle event, list, or map with filters.

Keyword Glossary

push
Triggered on every git push.
pull_request
Triggered on PR open/sync/reopen.
schedule
Cron-style scheduled triggers.
workflow_dispatch
Manual trigger via UI or API.
workflow_call
Make workflow reusable by other workflows.
repository_dispatch
Trigger via REST API.
paths
Filter that runs only when matching files change.
branches
Filter that runs only on listed branches.
types
Filter on event subtype (opened, synchronize).
concurrency
Cancel in-progress runs on same group.

Interview One-Liners

Use push for main-branch validation, pull_request for PR gates.
schedule uses cron syntax in UTC.
workflow_dispatch lets QA trigger ad-hoc runs.
Filter by paths to skip workflow when only docs change.
Use concurrency to cancel superseded PR runs.
workflow_call turns a workflow into a reusable function.

S4 - Jobs, Steps & Runners - The Execution Model

A job is a group of steps run on the same runner. Jobs run in parallel by default unless connected with 'needs'. Steps run sequentially. Runners are GitHub-hosted VMs or self-hosted machines.

What / Why / When / Where / How

WhatJob = work unit on one runner; Step = single command/action.
WhyParallel jobs cut suite time; sequential steps preserve order.
WhenSplit by purpose: build, test, report, deploy.
Wherejobs.<id> in workflow; steps under jobs.<id>.steps.
HowUse needs: to chain; matrix: to fan-out; if: to gate.

Keyword Glossary

Job
Group of steps on a single runner instance.
Step
One command (run:) or action (uses:).
Runner
VM that executes jobs (ephemeral by default).
GitHub-hosted
Free runners managed by GitHub.
Self-hosted
Your own machines registered to the repo.
Matrix
Fan-out a job across param combinations.
Artifact
File saved from a job and shared/downloaded.
timeout-minutes
Cap on job duration (default 360).
continue-on-error
Don't fail job if a step errors.
if
Conditional execution at job or step level.

Interview One-Liners

Jobs are parallel by default - chain them with 'needs'.
Steps inside a job are always sequential.
GitHub-hosted runners are stateless - nothing persists.
Use self-hosted runners for restricted networks.
matrix is a loop over param combos - perfect for cross-browser.
Artifacts move files between jobs and out for download.

        co.addArguments("--headless=new");        // Selenium 4 + Chrome 109+
        co.addArguments("--window-size=1920,1080");
        co.addArguments("--no-sandbox");
        co.addArguments("--disable-dev-shm-usage");
        driver = new ChromeDriver(co);
        wait = new WebDriverWait(driver, Duration.ofSeconds(10));

    }

    @AfterMethod

Best Practices Checklist

  • ✅ Use GH Environments with required reviewers for prod deploys.
  • ✅ Wire failure notifications to Slack/Teams - close feedback loop.
  • ✅ Use OIDC instead of static cloud credentials - shorter-lived, safer.
  • ✅ Run Selenium Grid via service containers for parallel browsers.
  • ✅ Host Allure dashboard on GitHub Pages - free, no infra.
  • ✅ Make CI a hard merge gate via required status checks.
  • ✅ Use concurrency to cancel superseded runs and save minutes.

  • ✅ Document runbook for failed deploys in repo README.
  • ✅ Review pipeline costs monthly - shard or self-host if budget tight.

Interview Prep Q&A

Q1. What is a GitHub Environment?
A: Named deploy target with rules like required reviewers, wait timer, branch restrictions, and env-scoped secrets.
Q2. What is OIDC for GH Actions?
A: OpenID Connect - exchanges a GH-issued token for cloud credentials, removing long-lived secrets.
Q3. How to run Selenium Grid in Actions?
A: Use 'services:' to spin up selenium/hub + node-chrome + node-firefox containers.
Q4. How to make CI required for merge?
A: Enable branch protection on main with 'Require status checks to pass' and pick your workflow jobs.
Q5. How to notify Slack on failure?
A: Use slackapi/slack-github-action with if: failure() and SLACK_WEBHOOK_URL secret.
