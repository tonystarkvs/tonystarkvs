INTRODUCTION TO BDD — Philosophy, TDD vs BDD vs ATDD, Three Amigos

Understand what BDD is, how it differs from TDD/ATDD, and how the Three Amigos work together

What / Why / When / Where / How

WhatBDD (Behavior-Driven Development) is a way of writing software by first describing how it should behave in plain English that business and tech teams both understand.
WhyIt removes the gap between business requirements and test automation, so everyone agrees on what 'done' really means.
WhenUse it for any feature where business rules matter — login, checkout, payments, search, anything user-facing.
WhereAcross the whole team: PO writes acceptance criteria, dev writes code, QA writes automation — all from the same feature file.
HowDiscuss examples together -> Document them as Gherkin -> Develop code to make them pass -> Demonstrate to PO -> Done.

Keyword Glossary

BDD
Behavior-Driven Development — writing software based on examples of behavior.
TDD
Test-Driven Development — write a unit test first, then code to pass it.
ATDD
Acceptance Test-Driven Development — write acceptance tests first with the customer.
Three Amigos
Product Owner, Developer, and QA meeting to agree on examples before coding.
Behavior
What the system should do from the user's point of view.
Specification
Concrete example of behavior, written in a way everyone understands.
Ubiquitous Language
Shared words used the same way by business and tech (e.g. 'cart', 'order').
Acceptance Criteria
Conditions the feature must satisfy to be accepted by the business.
Outside-In
Start from user behavior, then go inward to code design.
Living Documentation
Up-to-date docs that come from passing feature files, not Word documents.

Interview One-Liners

BDD is TDD with a business-friendly language wrapped around it.
The Three Amigos meeting is where most BDD value is created, not in tools.
BDD answers 'what should it do', TDD answers 'is my code working'.
Feature files are not test scripts; they are shared specifications.
Living documentation is what makes BDD pay off long term.

Best Practices Checklist

  • Always start with a Three Amigos discussion before writing any feature file.
  • Use plain business language — avoid CSS selectors and SQL inside Gherkin.
  • Keep one behavior per scenario, not a long workflow.
  • Treat feature files as a contract with the business, not as test scripts.
  • Make examples concrete with real values, not vague text like 'some user'.
  • Use ubiquitous language consistently across code, tests, and conversations.
  • Pair on writing feature files so they are owned by the team, not one person.
  • Review feature files in pull requests like real code.
  • Keep feature files version-controlled in the same repo as the code.
  • Demonstrate working scenarios to the product owner in every sprint.

Interview Prep Q&A

Q1. What is BDD in one line?
BDD is a way to build software based on examples of behavior that everyone understands.
Q2. Difference between TDD and BDD?
TDD starts from a failing unit test; BDD starts from a business example written in plain English.
Q3. Who attends a Three Amigos meeting?
Product Owner, Developer, and QA — three different views on the same feature.
Q4. Why is BDD called outside-in?
Because we start from user behavior outside the system, then move inward to design and code.
Q5. What is living documentation?
Documentation generated from passing feature files, so it stays in sync with the real system.

GHERKIN SYNTAX — Feature, Rule, Given, When, Then, And, But

Learn the building blocks of Gherkin and how to write clean, readable scenarios

What / Why / When / Where / How

WhatGherkin is the plain-English language used in Cucumber feature files to describe behavior.
WhyIt lets non-developers read and even write tests because the syntax looks like normal language.
WhenEvery time you write a Cucumber test — Gherkin is the only language Cucumber understands in feature files.
WhereInside .feature files placed under src/test/resources in a Maven project.
HowStart with Feature, optionally Rule and Background, then write one Scenario per behavior using Given/When/Then/And/But.

Keyword Glossary

Feature
Top-level heading describing the capability under test.
Rule
Groups scenarios that belong to the same business rule (Gherkin 6+).
Background
Steps that run before every scenario in the feature.
Scenario
A single concrete example of behavior.
Given
Sets the initial context — preconditions of the test.
When
The action or event the user performs.
Then
The expected outcome the system should produce.
And
Continues the previous keyword to keep reading natural.
But
Negative form of And, used for exceptions or 'should not' steps.
Tag
Label starting with @ used to group and filter scenarios.

Interview One-Liners

Gherkin keywords always start the line — they are not just labels.
Given sets context, When triggers action, Then asserts outcome.
And/But are just readability helpers; Cucumber treats them like the keyword above.
If a scenario has more than one When, it is usually doing two things — split it.
Tags like @smoke or @regression are how we choose what to run in CI.

Best Practices Checklist

  • Use Feature as a business capability, not a screen name.
  • Keep each scenario focused on ONE behavior.
  • Use real, concrete data — '500 INR' is better than 'some amount'.
  • Avoid CSS selectors, XPath, and JSON inside Gherkin steps.
  • Prefer business verbs like 'logs in' over technical ones like 'clicks button id loginBtn'.
  • Use tags to mark @smoke, @regression, @wip clearly.
  • Group related scenarios under a Rule when needed.
  • Limit Background to truly common setup steps.
  • Review .feature files in PRs with both QA and PO.
  • Keep feature files short — split when they become unreadable.

Interview Prep Q&A

Q1. Why is Gherkin called business-readable?
Because keywords like Given/When/Then mirror normal English sentences.
Q2. Difference between And and But?
Both continue the previous step; But just reads more naturally for negative statements.
Q3. What is the Rule keyword used for?
To group scenarios that belong to the same business rule (introduced in Gherkin 6).
Q4. When should I use a Background?
When the same steps repeat in every scenario and only describe setup, not behavior.
Q5. Are tags case-sensitive?
Yes — @Smoke and @smoke are different, so pick one convention and stick to it.

ADVANCED GHERKIN — Background, Scenario Outline, Data-Driven

Reuse setup with Background and run the same scenario over many rows with Scenario Outline + Examples

What / Why / When / Where / How

WhatBackground lets us reuse steps; Scenario Outline lets us run the same scenario with many sets of data.
WhyThey cut duplication and let one scenario validate many real-world inputs without copy-paste.
WhenUse Background for common setup; use Scenario Outline whenever you would otherwise repeat the same scenario with different data.
WhereInside the same .feature file, before Scenarios (Background) or as a separate Scenario Outline + Examples block.
HowWrite Background steps once; use placeholders like <username> in Scenario Outline and feed them from an Examples table.

Keyword Glossary

Background
Steps that run before each Scenario in the feature.
Scenario Outline
A scenario template that runs once per row of Examples.
Examples
Table of values that feed into a Scenario Outline.
Placeholder
Word in angle brackets like <username> replaced from Examples.
Data-Driven
Running the same scenario with different input data.
Pipe Table
Table built with | characters used in Examples or data tables.
Header Row
First row of an Examples table — names the columns.
Test Case Explosion
When too many Example rows make scenarios hard to maintain.
DRY
Don't Repeat Yourself — Background and Outline help here.
Edge Case
Unusual data row added in Examples to test boundaries.

Interview One-Liners

Background is for setup, not for behavior.
If Background grows past 3-4 steps, your feature is probably doing too much.
Scenario Outline is BDD's data-driven testing — one scenario, many data rows.
Examples header names should match placeholders in <angle brackets>.
Don't add 50 rows to Examples — pick the most meaningful ones.

● when-to-use-background.txt 
Use Background when:
- Every scenario starts with the same step(s)
- The steps are setup, not behavior
- Keeping them out of each scenario improves readability

Do NOT use Background when:
- Only some scenarios need that setup
- The setup is actually part of the behavior under test
- It would grow longer than 4 steps
● data-driven-strategy.md 
# Data-Driven Strategy in BDD

1. Identify what truly varies across cases.
2. Pick a happy path + 1 to 3 edge cases.
3. Put them in an Examples table.
4. Avoid putting full UI flows in Examples — keep it data only.
5. Add a new Examples block for clearly different scenarios (e.g. valid vs invalid).

Rule of thumb:
  10 meaningful rows > 100 random rows.

Best Practices Checklist

  • Use Background only for true setup steps shared by every scenario.
  • Keep Background short — under 4 steps when possible.
  • Use Scenario Outline when scenarios differ only by data.
  • Pick representative Examples — happy path + 1 or 2 edge cases.
  • Use clear column names like <username>, <expectedMessage>.
  • Avoid mixing UI selectors in Examples values.
  • Use multiple Examples blocks to separate valid vs invalid data.
  • Keep the table readable — align pipes neatly.
  • Avoid huge tables — split features instead.
  • Review Outline tables in PRs like real code.

Interview Prep Q&A

Q1. What is Background used for?
Common steps that should run before every scenario in the feature.
Q2. Difference between Scenario and Scenario Outline?
Scenario runs once; Scenario Outline runs once per row of its Examples table.
Q3. Why use multiple Examples blocks?
To clearly separate cases, like valid users vs invalid users.
Q4. Is data-driven only Scenario Outline?
No — Data Tables in steps are also data-driven, just inline.
Q5. Can Background contain When/Then?
Technically yes, but it should only contain setup (Given) steps in real projects.

CUCUMBER SETUP — Maven, Step Definitions, Test Runner

Wire feature files to Java step definitions and configure the Test Runner with @CucumberOptions

What / Why / When / Where / How

WhatCucumber needs three things: feature files, step definitions in Java, and a Test Runner to glue them.
WhyWithout the Runner, Cucumber does not know where the features and step code live.
WhenSet this up once at the start of the project; reuse it for every new feature.
WhereMaven structure: features under src/test/resources, step + runner classes under src/test/java.
HowAdd Cucumber + Selenium + TestNG to pom.xml, write step definitions with @Given/@When/@Then, and configure @CucumberOptions on the Runner.

Keyword Glossary

pom.xml
Maven project file that declares dependencies and plugins.
Step Definition
Java method mapped to a Gherkin step using annotations.
@Given
Cucumber annotation for a Given step.
@When
Cucumber annotation for a When step.
@Then
Cucumber annotation for a Then step.
Glue
Package path Cucumber scans for step definitions.
Cucumber Expression
Modern way to capture step parameters like {string}, {int}, {word}.
Regex
Old way to capture parameters using regular expressions.
Test Runner
Class annotated with @CucumberOptions used to run scenarios.
Plugin
Reporter format like html, json, pretty.

Interview One-Liners

Step definitions are just normal Java methods with annotations.
Cucumber Expressions are easier and safer than raw regex.
Glue tells Cucumber where to find your step code.
One step definition can match many feature steps using parameters.
The Runner class is the entry point Maven Surefire executes.

import io.cucumber.java.en.Given;
import io.cucumber.java.en.When;
import io.cucumber.java.en.Then;

public class LoginSteps {

    @Given("the user is on the login page")
    public void openLoginPage() {
        // Navigate to the login URL
    }

    // {string} captures any quoted text, no regex needed
    @When("the user logs in with {string} and {string}")
    public void loginWith(String username, String password) {
        // Type credentials and click login
    }

    @Then("the dashboard page should be displayed")
    public void dashboardShown() {
        // Assert dashboard is visible
    }
}

Best Practices Checklist

  • Keep features in src/test/resources/features for clarity.
  • Keep step definitions in a stepdefs package under src/test/java.
  • Prefer Cucumber Expressions over raw regex for readability.
  • Use one step definition class per business area, not one per feature.
  • Avoid putting logic in step definitions — delegate to Page Objects or services.
  • Use @CucumberOptions plugin html and json for reports.
  • Add tags = "@smoke" to your CI Runner to run only smoke.
  • Use dryRun = true while drafting features to find missing steps.
  • Use monochrome = true for cleaner console output.
  • Keep one TestRunner per logical pack (smoke, regression).

Interview Prep Q&A

Q1. What is glue in Cucumber?
The package path Cucumber scans to find step definitions.
Q2. Cucumber Expression vs Regex?
Cucumber Expression is simpler (uses {string}, {int}); regex is more powerful but harder to read.
Q3. Why do we need a TestRunner class?
Because that is the entry point Maven Surefire executes; @CucumberOptions tells it what to run.
Q4. Can one step match multiple lines in features?
Yes — using parameters, one step definition can match many Gherkin lines.
Q5. Where do feature files belong in a Maven project?
Under src/test/resources, usually inside a features folder.

ADVANCED FEATURES — Tags, Hooks, Data Tables, TestNG Parallel

Group tests with tags, add setup/teardown via hooks, pass tables of data, and run scenarios in parallel

What / Why / When / Where / How

WhatTags, hooks, data tables, and parallel execution are how a real Cucumber + TestNG framework grows up.
WhyThey give us control over what runs, when, with what data, and how fast.
WhenAdd these once your feature files and step defs are stable and the framework is used by more than one tester.
WhereTags inside .feature files, hooks in Java step packages, data tables in features + steps, parallel config in junit-platform.properties or TestNG.
HowTag scenarios, write @Before/@After hooks, declare DataTable parameters in steps, and enable parallel via TestNG runner.

Keyword Glossary

Tag
Label like @smoke on Feature/Scenario for filtering.
Tag Expression
Boolean like @smoke and not @wip used in Runner or CLI.
Hook
Method that runs before/after a scenario or step.
@Before
Runs before each scenario.
@After
Runs after each scenario.
Tagged Hook
Hook that runs only for matching tagged scenarios.
Data Table
Tabular data passed inside a step.
List<Map>
Common Java mapping for a Cucumber Data Table.
Parallel Execution
Running scenarios at the same time using threads.
Thread Count
Number of parallel scenarios allowed at once.

Interview One-Liners

Tags are the cleanest way to choose what runs in CI.
Hooks are perfect for browser setup and screenshot on failure.
Tagged hooks let me reset DB only for @db scenarios.
Data Tables shine when a step needs many fields, not just one or two.
Parallel works only if scenarios are isolated and data-safe.

Best Practices Checklist

  • Pick a small, consistent tag vocabulary: @smoke, @regression, @wip, @db.
  • Use @Before and @After for common setup/teardown only.
  • Use tagged hooks for special cases like DB reset or auth.
  • Take screenshots in @After when scenario.isFailed().
  • Prefer List<Map<String,String>> for headered Data Tables.
  • Avoid heavy logic inside hooks — keep them thin.
  • Don't run @wip in CI — only locally.
  • Enable parallel only after data isolation is confirmed.
  • Use junit-platform.properties to control parallelism cleanly.
  • Document tag meaning in a TAGS.md so new joiners know the convention.

Interview Prep Q&A

Q1. How do tags help in CI?
They let CI run only what is needed, like @smoke on every PR and @regression at night.
Q2. Why use @Before instead of TestNG @BeforeMethod?
Because Cucumber hooks are scenario-aware and integrate with tags and reports.
Q3. How is a Data Table different from a Scenario Outline?
Data Table passes a table into a step; Scenario Outline runs a whole scenario per row.
Q4. Can I use multiple tags on one scenario?
Yes — separate them with spaces: @smoke @login.
Q5. How is parallel handled with cucumber-testng?
TestNG runs scenarios in parallel based on configured thread-count.

Generate readable reports from passing scenarios and plug Cucumber into a clean Page Object Model

What / Why / When / Where / How

WhatReporting turns Cucumber runs into living documentation, and POM keeps the UI code clean and reusable.
WhyReports give the team confidence and POM keeps step definitions short and stable when the UI changes.
WhenAdd reporting from day one; introduce POM once you have more than one or two pages.
WhereReports go under target/; POM classes go in src/main/java/com/learn/pages.
HowAdd html/json plugins to @CucumberOptions, add the ExtentReports Spark adapter, and refactor step defs to call Page Object methods.

Pattern where each page is a Java class with locators and actions.

    }

package com.learn.stepdefs;

import com.learn.pages.HomePage;
import com.learn.pages.LoginPage;
import io.cucumber.java.en.Given;
import io.cucumber.java.en.When;
import io.cucumber.java.en.Then;
import org.openqa.selenium.WebDriver;
import org.openqa.selenium.chrome.ChromeDriver;
import org.testng.Assert;

public class LoginSteps {
    private WebDriver driver;
    private LoginPage loginPage;
    private HomePage homePage;

    @Given("the user is on the login page")
    public void openLogin() {
        // Selenium Manager auto-handles driver in Selenium 4
        driver = new ChromeDriver();
        loginPage = new LoginPage(driver);
        homePage  = new HomePage(driver);
        loginPage.open();
    }

    @When("the user logs in with {string} and {string}")
    public void login(String user, String pwd) {
        loginPage.loginAs(user, pwd);
    }

    @Then("the dashboard page should be displayed")
    public void assertDashboard() {
        Assert.assertTrue(homePage.isDashboardVisible());
        driver.quit();
    }
}

Best Practices Checklist

  • Always configure at least html and json Cucumber plugins.
  • Add Extent Spark reporter for stakeholder-friendly reports.
  • Treat every run's HTML report as living documentation.
  • Move every UI action into a Page Object class.
  • Name page methods using business verbs like loginAs, addToCart.
  • Avoid Thread.sleep — use explicit waits in Page Objects.
  • Keep step definitions under ~10 lines each.
  • Centralize WebDriver creation in a single DriverFactory.
  • Use Selenium Manager (Selenium 4) instead of WebDriverManager when possible.
  • Archive HTML + JSON reports as build artifacts.

Interview Prep Q&A

Q1. What is living documentation?
Documentation that comes from your passing feature files, so it never goes stale.
Q2. Why integrate POM with Cucumber?
Because step defs should describe behavior, not click buttons — Page Objects handle the clicking.
Q3. Difference between html and json Cucumber reports?
html is for humans, json is for tools like Extent or dashboards.
Q4. What does Spark Reporter give us?
A modern, themed HTML report from ExtentReports — easy to share with stakeholders.
Q5. How does POM make tests stable?
If a locator changes, only one Page Object class needs an update, not every step.
