Session 1: Foundations & Key Concepts
Beginner What is BDD, TDD, ATDD & the full stack
Key Terms Explained
BDD
Behavior Driven Development — tests written in plain English (Gherkin) so devs, QA & business all understand. Focus: WHAT should happen.
TDD
Test Driven Development — write unit tests before code. Focus: HOW it works internally.
ATDD
Acceptance Test Driven Development — tests defined by the whole team before dev starts. BDD is a style of ATDD.
Cucumber
BDD framework that reads .feature files (Gherkin) and maps steps to Java/Python/JS code.
Selenium 4
Browser automation library. Drives Chrome, Firefox, Edge. New in v4: W3C protocol, CDP, relative locators, improved grid.
Gherkin
Language for writing feature files. Keywords: Feature, Scenario, Given, When, Then, And, But, Background.
Step Definition
Java method annotated with @Given/@When/@Then that links Gherkin step to actual automation code.
POM
Page Object Model — design pattern. One class per web page. Separates UI locators from test logic.
Full Framework Stack
// Typical BDD + Selenium 4 Maven project layers Feature File (.feature) ← Business readable tests ↓ Step Definitions (.java) ← Glue between Gherkin & code ↓ Page Objects (.java) ← UI element locators + actions ↓ WebDriver (Selenium 4) ← Browser automation ↓ Runner File (JUnit/TestNG) ← Test execution configuration ↓ Hooks.java ← Setup / teardown logic ↓ Reports (Extent/Allure) ← Test results

Step Definition (Java)
// src/test/java/stepdefs/LoginSteps.java import io.cucumber.java.en.*; import pages.LoginPage; public class LoginSteps { private LoginPage loginPage; @Given("the user is on the login page") public void navigateToLogin() { loginPage = new LoginPage(DriverManager.getDriver()); loginPage.open(); } @When("the user enters valid credentials") public void enterValidCredentials() { loginPage.enterEmail("user@test.com"); loginPage.enterPassword("Test@1234"); loginPage.clickLogin(); } @Then("the user should see the dashboard") public void verifyDashboard() { Assert.assertTrue(loginPage.isDashboardVisible()); } }

Session 3: Selenium 4 Essentials
Intermediate W3C, DriverManager, key new features
What's New in Selenium 4
W3C WebDriver
All browser communication is now W3C standard. No more JsonWireProtocol. More stable & consistent.
Chrome DevTools (CDP)
Direct access to browser internals — mock networks, intercept requests, capture console logs.
Relative Locators
Find elements relative to others: above(), below(), toLeftOf(), toRightOf(), near(). Useful for dynamic UIs.
New Window/Tab API
driver.switchTo().newWindow(WindowType.TAB) — open tabs programmatically.
Selenium Grid 4
Rebuilt from scratch. Supports Docker, Kubernetes. Better parallel test distribution.
ThreadLocal DriverManager (Thread-Safe)
// utils/DriverManager.java — CRITICAL for parallel tests public class DriverManager { // ThreadLocal = separate driver per thread (parallel safe!) private static ThreadLocal<WebDriver> driver = new ThreadLocal<>(); public static void initDriver(String browser) { WebDriver d; switch(browser.toLowerCase()) { case "chrome": WebDriverManager.chromedriver().setup(); ChromeOptions opts = new ChromeOptions(); opts.addArguments("--headless=new"); // headless v2 d = new ChromeDriver(opts); break; case "firefox": WebDriverManager.firefoxdriver().setup(); d = new FirefoxDriver(); break; default: throw new IllegalArgumentException("Unknown browser"); } d.manage().window().maximize(); d.manage().timeouts().implicitlyWait(Duration.ofSeconds(5)); driver.set(d); } public static WebDriver getDriver() { return driver.get(); } public static void quitDriver() { if (driver.get() != null) { driver.get().quit(); driver.remove(); // prevent memory leak! } } }
Common Mistake: Not Using ThreadLocal
// BAD — single static driver: parallel tests will CRASH! public static WebDriver driver; // ❌ Race condition! // GOOD — ThreadLocal: each thread gets its own driver private static ThreadLocal<WebDriver> driver = new ThreadLocal<>(); // ✅
Session 4: BDD Hooks — All 3 Phases × 3 Frameworks
Intermediate Before/After hooks in Cucumber, TestNG, JUnit 5
What Are Hooks?

Hooks are special methods that run automatically at specific lifecycle points — before/after scenarios, features, or steps. They handle setup & teardown without putting it inside every test.

Before SuiteRun once at start
Before/After ScenarioEach test setup
After SuiteRun once at end
Framework 1: Cucumber Hooks
// hooks/Hooks.java (Cucumber) import io.cucumber.java.*; public class Hooks { // Phase 1: Before each Scenario @Before public void setUp(Scenario scenario) { System.out.println("Starting: " + scenario.getName()); DriverManager.initDriver("chrome"); } // Phase 2: After each Scenario (with screenshot on fail) @After public void tearDown(Scenario scenario) { if (scenario.isFailed()) { byte[] screenshot = ((TakesScreenshot) DriverManager.getDriver()) .getScreenshotAs(OutputType.BYTES); scenario.attach(screenshot, "image/png", "Screenshot"); } DriverManager.quitDriver(); } // Phase 3: Tagged hook — only runs for @smoke @Before("@smoke") public void beforeSmoke() { System.out.println("Smoke test prep..."); } // Before/AfterStep — step level (Cucumber 6+) @BeforeStep public void beforeStep() { /* log step start */ } @AfterStep public void afterStep(Scenario s) { /* capture step result */ } }
Framework 2: TestNG Hooks (3 Phases)
// BaseTest.java (TestNG) public class BaseTest { // PHASE 1 — Suite level (runs once for entire suite) @BeforeSuite public void globalSetup() { ExtentReports.init(); // start report System.out.println("Suite started"); } // PHASE 2 — Method level (runs before each @Test) @BeforeMethod public void testSetup(Method method) { DriverManager.initDriver(System.getProperty("browser","chrome")); } @AfterMethod public void testTearDown(ITestResult result) { if (result.getStatus() == ITestResult.FAILURE) { captureScreenshot(result.getName()); } DriverManager.quitDriver(); } // PHASE 3 — Suite level teardown @AfterSuite public void globalTeardown() { ExtentReports.flush(); // finalize report } }
Framework 3: JUnit 5 Hooks (3 Phases)
// BaseTest.java (JUnit 5) import org.junit.jupiter.api.*; public class BaseTest { // PHASE 1 — Class level (once before all tests in class) @BeforeAll static void suiteSetup() { System.out.println("All tests starting..."); } // PHASE 2 — Method level (before EACH test) @BeforeEach void testSetup(TestInfo info) { System.out.println("Test: " + info.getDisplayName()); DriverManager.initDriver("chrome"); } @AfterEach void testTearDown(TestInfo info) { DriverManager.quitDriver(); } // PHASE 3 — Class level (once after all tests) @AfterAll static void suiteTearDown() { System.out.println("Suite complete."); } }

Session 5: Parallel Execution Configuration
Advanced TestNG XML, JUnit 5 config, Maven Surefire
Parallel Execution — Core Concept
Why Parallel?
Run multiple tests simultaneously → reduce total execution time from hours to minutes.
Thread Count
Number of tests running at same time. Recommended: CPU cores × 2 (e.g., 4 cores = 8 threads).
Thread Safety
Each thread must have its own WebDriver instance (use ThreadLocal). Shared state = flaky tests.
TestNG — testng.xml Parallel Config
<!-- testng.xml --> <!DOCTYPE suite SYSTEM "https://testng.org/testng-1.0.dtd"> <!-- parallel="tests" = each <test> block runs in own thread --> <suite name="BDDSuite" parallel="tests" thread-count="4" verbose="2"> <!-- Thread 1: Chrome --> <test name="ChromeTests"> <parameter name="browser" value="chrome"/> <classes> <class name="runners.TestRunner"/> </classes> </test> <!-- Thread 2: Firefox --> <test name="FirefoxTests"> <parameter name="browser" value="firefox"/> <classes> <class name="runners.TestRunner"/> </classes> </test> </suite> <!-- parallel options: "tests" | "classes" | "methods" -->
JUnit 5 — junit-platform.properties
# src/test/resources/junit-platform.properties # Enable parallel execution junit.jupiter.execution.parallel.enabled = true # Run test CLASSES in parallel (not methods — safer for Selenium) junit.jupiter.execution.parallel.mode.default = concurrent junit.jupiter.execution.parallel.mode.classes.default = concurrent # Thread pool: FIXED = exact count, DYNAMIC = auto based on CPUs junit.jupiter.execution.parallel.config.strategy = fixed junit.jupiter.execution.parallel.config.fixed.parallelism = 4
Maven Surefire Plugin Config (Required!)
<!-- pom.xml build section --> <plugin> <groupId>org.apache.maven.plugins</groupId> <artifactId>maven-surefire-plugin</artifactId> <version>3.1.2</version> <configuration> <!-- For TestNG --> <suiteXmlFiles> <suiteXmlFile>testng.xml</suiteXmlFile> </suiteXmlFiles> <!-- For JUnit 5 parallel (uncomment) --> <!-- <forkCount>4</forkCount> --> <!-- <reuseForks>true</reuseForks> --> <!-- Pass browser param to TestNG --> <systemPropertyVariables> <browser>chrome</browser> </systemPropertyVariables> </configuration> </plugin>

Common Runner Mistakes
❌
Missing hooks in glue: glue = {"stepdefs"} — forgetting to add "hooks" means @Before/@After never runs.
❌
Wrong feature path: Features path must match your actual directory. Use relative path from project root.
❌
dryRun=true in CI: Accidentally leaving dryRun=true means zero tests actually execute.
✅
Rerun failed tests: Add plugin "rerun:target/rerun.txt" + create a RerunRunner pointing to that file.
Session 7: Troubleshooting & Best Practices
Intermediate Flaky tests, timeouts, race conditions, driver issues
Flaky Test — Causes & Fixes
ImplicitWait + ExplicitWait together: Mixing causes unpredictable wait behavior. Use ONLY ExplicitWait (WebDriverWait) in production code.
Fix: WebDriverWait wait = new WebDriverWait(driver, Duration.ofSeconds(10)); then wait.until(ExpectedConditions.visibilityOfElementLocated(By.id("btn")));
Thread.sleep() abuse: Hard sleeps make tests slow AND still fail if the page loads slower than your sleep.
Fix: Replace all Thread.sleep() with explicit waits for the actual condition (element visible, clickable, URL changed).
StaleElementReferenceException: Element found but page refreshed before interaction.
Fix: Re-find element before each action, or use Page Factory with @FindBy lazy loading.
Timeout Strategy
// utils/WaitUtils.java — Centralize all wait logic public class WaitUtils { private static final int DEFAULT_TIMEOUT = 10; public static WebElement waitForVisible(By locator) { return new WebDriverWait( DriverManager.getDriver(), Duration.ofSeconds(DEFAULT_TIMEOUT)) .until(ExpectedConditions .visibilityOfElementLocated(locator)); } public static void waitForPageLoad() { new WebDriverWait( DriverManager.getDriver(), Duration.ofSeconds(30)) .until(d -> ((JavascriptExecutor) d) .executeScript("return document.readyState") .equals("complete")); } }
Parallel Test Issues & Fixes
Shared static fields in Step Defs: Static variables are shared across threads → race condition. Use instance variables or ThreadLocal.
Test data conflicts: Two parallel tests using the same username/email → one fails login. Use unique test data per thread (e.g., suffix with thread ID).
Screenshots mixed up: Using static filenames → parallel tests overwrite each other's screenshots.
Fix: Name screenshots with timestamp + thread ID: scenario.getName() + "_" + Thread.currentThread().getId()
Driver not quit on failure: Crashed tests leave browser processes open → memory leak → system slowdown.
Fix: Always quit driver in @After hook. Use try-finally pattern in DriverManager.quitDriver().

Session 8: Interview Preparation
Interview Common questions with crisp, confident answers
BDD & Cucumber Questions
Q: What is BDD and how is it different from TDD?
BDD focuses on behavior from user's perspective using plain English (Gherkin), involves the whole team (dev, QA, BA). TDD is developer-focused, uses unit tests written in code before implementation.
Q: What is the purpose of Background in Cucumber?
Background runs the same steps before every Scenario in a Feature file — avoids repeating setup steps like "Given I am on the login page" in every scenario.
Q: Difference between Scenario and Scenario Outline?
Scenario runs once with hardcoded values. Scenario Outline runs multiple times with different data from an Examples table — data-driven testing in BDD.
Q: What is a Step Definition?
A Java method annotated with @Given/@When/@Then that matches a Gherkin step using a regex or expression. It's the bridge between human-readable feature files and executable code.
Q: What is the glue option in CucumberOptions?
Package path(s) where Cucumber looks for Step Definitions and Hooks. If missing, steps show as undefined.
Hooks Questions
Q: What are Hooks in Cucumber? Name them.
Hooks are methods that run at lifecycle points: @Before (before each scenario), @After (after each scenario), @BeforeStep, @AfterStep, and tagged variants like @Before("@smoke").
Q: How do you take a screenshot on failure in Cucumber?
In @After hook: check scenario.isFailed(), take screenshot using TakesScreenshot, attach as byte[] via scenario.attach(bytes, "image/png", "name").
Q: Difference between @Before in Cucumber vs @BeforeMethod in TestNG?
@Before is scenario-level in Cucumber (for BDD). @BeforeMethod is test-method-level in TestNG. Both serve same purpose but @BeforeMethod has more granular control (groups, dependsOnMethods).
Parallel Execution Questions
Q: How do you run Cucumber tests in parallel?
Three ways: (1) TestNG: use @DataProvider(parallel=true) in AbstractTestNGCucumberTests runner + testng.xml with parallel="tests". (2) JUnit 5: enable in junit-platform.properties. (3) Maven Surefire: forkCount for JVM-level parallelism.
Q: Why do we use ThreadLocal for WebDriver in parallel tests?
ThreadLocal ensures each thread has its own isolated WebDriver instance. Without it, parallel threads share the same driver, causing tests to interfere — clicking wrong pages, incorrect assertions, crashes.
Q: What is the difference between parallel=tests and parallel=methods in TestNG?
parallel="tests" runs each <test> XML block in its own thread — good for cross-browser. parallel="methods" runs each @Test method in parallel — more granular but needs more careful thread safety.
Q: How do you handle test data conflicts in parallel execution?
Use unique test data per thread: generate emails/usernames with Thread.currentThread().getId() suffix, use separate test databases/schemas per thread, or use a data factory that locks records.
Selenium 4 Questions
Q: What's new in Selenium 4 vs Selenium 3?
W3C protocol (replaced JsonWireProtocol), Chrome DevTools Protocol (CDP) integration, relative locators (above/below/near), new window/tab API, rebuilt Selenium Grid 4 with Docker support, improved documentation.
Q: What are Relative Locators in Selenium 4?
Locate elements relative to other elements: above(), below(), toLeftOf(), toRightOf(), near(). Useful when elements lack unique IDs but are spatially positioned near known elements.
Q: Difference between implicit and explicit wait?
Implicit wait: global, applies to all findElement calls, waits up to N seconds. Explicit wait: targeted, waits for a specific condition on a specific element. Best practice: use only explicit wait to avoid conflicts.
Power Interview Tips
Show real project experience: Say "In my project, I used ThreadLocal DriverManager to handle parallel execution across 4 Chrome threads..."
Know the WHY: Don't just say what you did — explain why. "We used @DataProvider(parallel=true) BECAUSE it's the correct way to parallelize at the Cucumber scenario level with TestNG."
Mention trade-offs: "JUnit 5 parallel is newer but Cucumber's TestNG integration is more mature for scenario-level parallel control."
Talk about failures: "I once debugged a race condition where two parallel tests shared a static list — fixed by making it a ThreadLocal instance."
