FOUNDATION — BDD with Cucumber + Playwright-TS

Understand BDD philosophy and how the playwright-bdd library connects Gherkin features to the Playwright Test runner

What / Why / When / Where / How

WhatBDD describes what the app should do in plain English; Cucumber-style Gherkin is then executed as Playwright-TS tests using the playwright-bdd library.
WhyIt connects business language to fast, modern, cross-browser automation — Playwright is faster and more reliable than Selenium for most UI flows.
WhenUse it for any user-facing flow where the business cares about behavior — login, search, checkout, dashboards, multi-step wizards.
WhereIn a TypeScript Node.js project: feature files in features/, step definitions in steps/, generated tests in .features-gen/.
HowWrite Gherkin -> playwright-bdd generates Playwright spec files at build time -> Playwright Test runs them and produces reports.

Keyword Glossary

BDD
Behavior-Driven Development — describing software using examples of behavior.
Cucumber
Original tool that reads Gherkin and runs step definitions; here we use a Playwright-native equivalent.
playwright-bdd
Library that turns Gherkin feature files into Playwright tests at generation time.
Gherkin
Plain-English language used inside .feature files.
Playwright
Modern browser automation library for Chromium, Firefox, and WebKit.
TypeScript
Typed JavaScript used to write Playwright tests and step definitions.
Feature File
File ending in .feature containing a Feature and Scenarios.
Step Definition
TypeScript function mapped to a Gherkin step via createBdd().
.features-gen
Folder where playwright-bdd writes generated Playwright spec files.
Living Documentation
Up-to-date docs that come from passing feature files.

Interview One-Liners

Cucumber gives the language; Playwright does the clicking — fast and reliable.
playwright-bdd is the modern way to run Gherkin with Playwright Test.
Feature files describe behavior; TypeScript step defs perform Playwright actions.
Playwright handles browsers, waits, and tracing better than older Selenium stacks.
Living documentation is the long-term payoff of using BDD.

Best Practices Checklist

  • Start with one stable feature file and one step definitions module.
  • Use playwright-bdd codegen helper to scaffold steps faster.
  • Use Playwright role/text/label locators instead of brittle CSS or XPath.
  • Use expect from @playwright/test for assertions, not chai or jest.
  • Use storageState for logged-in sessions instead of logging in every test.
  • Externalize URLs into playwright.config.ts use.baseURL or env files.
  • Enable trace: 'on-first-retry' to debug failures quickly.
  • Add .features-gen to .gitignore — it is build output, not source.
  • Group features by domain (auth/, checkout/, search/), not by tech.
  • Document the team's tag vocabulary (e.g. @smoke, @regression) in TAGS.md.

Because Gherkin gives business-readable specs and Playwright runs them fast and reliably across browsers.

GHERKIN + STEP DEFINITIONS FOR PLAYWRIGHT-TS

Map Gherkin scenarios to async TypeScript step definitions using playwright-bdd and Playwright locators

What / Why / When / Where / How

WhatStep definitions are async TypeScript functions that turn each Gherkin step into a Playwright action or assertion.
WhyWithout step definitions, Cucumber cannot run any browser action — the Gherkin is just text.
WhenWrite them right after a feature file is reviewed and agreed by the Three Amigos.
WhereIn a steps/ folder using TypeScript modules; one file per business area is a good rule.
HowCall createBdd() to get Given/When/Then bindings, pass them async arrow functions that use page and expect.

Keyword Glossary

createBdd()
playwright-bdd helper that returns typed Given/When/Then functions.
Given
Step definition for context/preconditions.
When
Step definition for the user action.
Then
Step definition for the expected outcome.
Cucumber Expression
Parameter syntax like {string}, {int}, {word}.
page
Playwright Page fixture passed automatically into steps.
expect
Playwright assertion library from @playwright/test.
getByRole
Playwright locator using ARIA role + accessible name.
getByLabel
Playwright locator using form label text.
Scenario Outline
Gherkin template that runs once per Examples row.

Interview One-Liners

Step definitions should read like one short sentence in TypeScript.
Cucumber Expressions are easier and safer than raw regex.
Playwright auto-waits, so I almost never need explicit waits.
I prefer getByRole and getByText over CSS selectors.
Keep assertions in @Then steps, not in @Given or @When.

● steps/login.steps.ts 
import { createBdd } from 'playwright-bdd';
import { expect } from '@playwright/test';

// createBdd() gives us typed Given/When/Then bindings
const { Given, When, Then } = createBdd();

Given('the user is on the login page', async ({ page }) => {
  await page.goto('/login');
});

// {string} captures any quoted text — no regex needed
When('the user logs in with {string} and {string}', async ({ page }, user: string, password: string) => {
  await page.getByLabel('Username').fill(user);
  await page.getByLabel('Password').fill(password);
  await page.getByRole('button', { name: /login/i }).click();
});

Then('the dashboard page should be displayed', async ({ page }) => {
  // Playwright auto-waits for visibility — no manual wait needed
  await expect(page.getByRole('heading', { name: /dashboard/i })).toBeVisible();
});
● steps/search.steps.ts 
import { createBdd } from 'playwright-bdd';
import { expect } from '@playwright/test';

const { Given, When, Then } = createBdd();

Given('the user is on the home page', async ({ page }) => {
  await page.goto('/');
});

// Two {string} parameters from Scenario Outline columns
When('the user searches for {string} in {string}', async ({ page }, term: string, category: string) => {
  await page.getByLabel('Category').selectOption(category);
  await page.getByPlaceholder('Search').fill(term);
  await page.getByRole('button', { name: 'Search' }).click();
});

Then('at least {int} results should be shown', async ({ page }, minCount: number) => {
  // expect().toHaveCount retries until the assertion passes or times out
  await expect(page.locator('.result-item')).toHaveCount(minCount);
});
● steps/assertions.steps.ts 
import { createBdd } from 'playwright-bdd';
import { expect } from '@playwright/test';

const { Then } = createBdd();

Then('matching results should be shown', async ({ page }) => {
  // Web-first assertion: auto-retries up to the configured timeout
  await expect(page.locator('.result-item').first()).toBeVisible();
});

Then('out-of-stock items should not be visible', async ({ page }) => {
  await expect(page.locator('.result-item.out-of-stock')).toHaveCount(0);
});

Then('the URL should contain {string}', async ({ page }, slug: string) => {
  await expect(page).toHaveURL(new RegExp(slug));
});
● cucumber-expressions-cheatsheet.md 
# Cucumber Expressions Cheatsheet (playwright-bdd)

| Expression | Captures           | TypeScript Type |
|------------|--------------------|-----------------|
| {string}   | "quoted text"      | string          |
| {int}      | 42                 | number          |
| {float}    | 3.14               | number          |
| {word}     | single word        | string          |

## Rules of thumb
- Use {string} for any quoted business value.
- Use {int} for counts and amounts.
- Use {word} when no spaces or quotes are expected.
- Use Custom Parameter Types for domain words like {currency}.

## Tip
If your step starts with `^` and ends with `$`, you are using regex.
Otherwise you are using a Cucumber Expression — keep them readable.

Best Practices Checklist

  • Use one step definitions file per business area.
  • Prefer Cucumber Expressions over raw regex.
  • Use Playwright auto-waiting — avoid manual sleeps and timeouts.
  • Use getByRole/getByLabel/getByText before CSS selectors.
  • Keep step functions short — push UI logic into Page Objects.
  • Use expect(locator) assertions; they retry automatically.
  • Avoid hardcoded URLs — read them from baseURL or env vars.
  • Use TypeScript types so step parameters are checked at build time.
  • Run `bddgen` before `playwright test` (or use the combined npm script).
  • Keep .feature and .steps file names aligned for readability.

Interview Prep Q&A

Q1. Cucumber Expression vs Regex?
Cucumber Expression is simpler (uses {string}, {int}); regex is more powerful but harder to read.
Q2. Why no waits in most Playwright steps?
Because Playwright auto-waits for elements to be visible and actionable.
Q3. Where should assertions live?
Inside @Then steps so they describe the expected outcome.
Q4. How does the page fixture reach my step function?
playwright-bdd injects it automatically via Playwright Test fixtures.
Q5. Can one step definition match many feature lines?
Yes — parameters let one function handle many similar Gherkin steps.

ADVANCED FEATURES — Background, Hooks, Data Tables, Page Object Model

Reuse setup, capture screenshots on failure, pass tabular data, and integrate playwright-bdd with POM

What / Why / When / Where / How

WhatBackground reuses setup steps, Before/After hooks run setup/teardown, Data Tables pass tabular data, and Page Object Model keeps step defs clean.
WhyWithout these, feature files repeat, step defs grow huge, and a UI change breaks many places at once.
WhenAdd Background and POM as soon as you have more than one or two scenarios; add screenshots-on-failure from day one.
WhereBackground in .feature files; hooks and Page Objects in TypeScript modules under steps/ and pages/.
HowWrite Background once, use createBdd().Before/After in a hooks.ts, type DataTable parameters, and call Page Object methods from steps.

Keyword Glossary

Background
Gherkin steps that run before each scenario in the feature.
Before
playwright-bdd hook running before each scenario.
After
playwright-bdd hook running after each scenario.
Tagged Hook
Hook that runs only for scenarios with a matching tag.
Data Table
Tabular data passed inside a step.
DataTable.hashes()
Method returning rows as objects keyed by header.
Page Object Model
Pattern where each page is a TS class with locators and actions.
Locator
Playwright handle like page.getByRole(...) used inside Page Objects.
testInfo
Playwright object with test status, retry, attachments.
attach
testInfo.attach to add evidence into the HTML report.

Interview One-Liners

Background is for setup, not for behavior.
Hooks are perfect for storage state, screenshots on failure, and cleanup.
Tagged hooks let me reset DB only for @db scenarios.
Data Tables shine when a step needs many fields, not just one or two.
POM keeps step defs short and stable when the UI changes.

● steps/hooks.ts 
import { createBdd } from 'playwright-bdd';

const { Before, After } = createBdd();

// Runs before every scenario
Before(async ({ page }) => {
  // Example: set viewport, clear cookies, etc.
  await page.context().clearCookies();
});

// Runs after every scenario — attach screenshot on failure
After(async ({ page }, { testInfo }) => {
  if (testInfo.status === 'failed') {
    const png = await page.screenshot();
    await testInfo.attach('failure-screenshot', { body: png, contentType: 'image/png' });
  }
});

// Tagged hook — runs ONLY for scenarios tagged @db
Before({ tags: '@db' }, async () => {
  // Reset database / seed required data
});
● pages/login.page.ts 
import { Page, Locator, expect } from '@playwright/test';

export class LoginPage {
  readonly page: Page;
  readonly username: Locator;
  readonly password: Locator;
  readonly loginBtn: Locator;

  constructor(page: Page) {
    this.page = page;
    this.username = page.getByLabel('Username');
    this.password = page.getByLabel('Password');
    this.loginBtn = page.getByRole('button', { name: /login/i });
  }

  async open() {
    await this.page.goto('/login');
    await expect(this.username).toBeVisible();
  }

  // Business-friendly method used by step definitions
  async loginAs(user: string, pwd: string) {
    await this.username.fill(user);
    await this.password.fill(pwd);
    await this.loginBtn.click();
  }
}

● steps/login.steps.ts 
import { createBdd } from 'playwright-bdd';
import { expect } from '@playwright/test';
import { LoginPage } from '../pages/login.page';
import { HomePage } from '../pages/home.page';

const { Given, When, Then } = createBdd();

Given('the user is on the login page', async ({ page }) => {
  const loginPage = new LoginPage(page);
  await loginPage.open();
});

When('the user logs in with {string} and {string}', async ({ page }, user: string, pwd: string) => {
  const loginPage = new LoginPage(page);
  await loginPage.loginAs(user, pwd);
});

Then('the dashboard page should be displayed', async ({ page }) => {
  const homePage = new HomePage(page);
  expect(await homePage.isDashboardVisible()).toBeTruthy();
});

Best Practices Checklist

  • Keep Background under 4 steps and use it only when EVERY scenario needs that setup.
  • Use Before for setup, After for screenshot + cleanup.
  • Attach screenshots only on failure (check testInfo.status).
  • Use tagged Before('@db') for special-case setup.
  • Use dataTable.hashes() for headered tables — returns objects.
  • Move all UI actions into Page Objects — step defs should call methods.
  • Name page methods with business verbs (loginAs, addToCart).
  • Trust Playwright auto-waiting inside Page Objects — avoid manual waits.
  • Pass page via constructor — keep Page Objects stateless and reusable.
  • Keep assertions in steps, not in Page Objects.

Interview Prep Q&A

Q1. Why use Background?
To remove repetition when every scenario starts with the same setup steps.
Q2. How is a Data Table different from a Scenario Outline?
Data Table passes a table into one step; Scenario Outline runs a whole scenario per row.
Q3. Why use Page Object Model with playwright-bdd?
Because step defs should describe behavior, not click buttons — Page Objects handle the clicking.
Q4. How do you capture a screenshot on failure?
In After hook, check testInfo.status, then page.screenshot() and testInfo.attach().
Q5. What is a tagged hook?
A hook that runs only for scenarios with a matching tag, like Before('@db').

TEST RUNNER, TAGS, PARALLEL EXECUTION

Run only what matters with tags and scale the suite with Playwright projects and workers

What / Why / When / Where / How

WhatPlaywright Test is the runner; playwright-bdd configures it to read features; tags decide what runs; projects + workers handle parallel.
WhyWithout tags, everything runs; without parallel, big suites are slow; without projects, you cannot cover multiple browsers cleanly.
WhenSet up the basic config on day one, add tags when you have ~10 scenarios, add multi-project and workers when CI time matters.
WhereAll in playwright.config.ts and via --grep on the command line.
HowAdd projects + workers + retries in defineConfig; tag scenarios; use --grep '@smoke' to filter at run time.

Keyword Glossary

Playwright Test
The built-in runner that executes generated specs.
defineBddConfig
Helper from playwright-bdd to register features and steps.
Tag
Label like @smoke on Feature or Scenario.
--grep
Playwright CLI flag to run only matching titles/tags.
Project
Browser/device configuration in playwright.config.ts.
Worker
Parallel process running Playwright tests.
fullyParallel
Config flag that runs files and tests in parallel.
Retries
Number of times a flaky test will be re-run automatically.
Reporter
Output format like html, list, dot.
storageState
Saved login state reused across tests for speed.

Interview One-Liners

Tags + --grep is the cleanest way to choose what runs in CI.
Playwright runs files in parallel by default — fullyParallel makes tests inside files parallel too.
Projects are how I cover Chromium, Firefox, and WebKit cleanly.
Workers control resource usage on CI machines.
Retries are useful but should not hide real flakiness.

  steps:    'steps/**/*.ts',
  // Limit the run to certain tag expressions at generation time:
  tags:     '@smoke and not @wip',
  // Output folder for generated specs (gitignored):
  outputDir: '.features-gen',
});

Best Practices Checklist

  • Keep one playwright.config.ts as the single source of truth.
  • Use defineBddConfig() in the same file to register features and steps.
  • Add a project per browser you care about (chromium, firefox, webkit).
  • Use fullyParallel: true once tests are isolated.
  • Set retries: 1 or 2 only after investigating flakiness.
  • Use --grep '@smoke' for CI smoke runs; full suite for nightly.
  • Use --grep-invert '@wip' to skip work-in-progress.
  • Use storageState to skip repeated logins.
  • Set workers based on CI machine, not local laptop.
  • Document the team's tag vocabulary in TAGS.md.

Interview Prep Q&A

Q1. How do tags help in CI?
They let CI run only what is needed, like @smoke on every PR and full suite at night.
Q2. How does Playwright run in parallel?
By default it runs spec files in parallel; with fullyParallel: true it also runs tests inside a file in parallel.
Q3. What is a project in Playwright?
A named configuration in playwright.config.ts, usually one per browser or device.
Q4. How do you skip work-in-progress scenarios?
Tag them @wip and use --grep-invert '@wip' or filter via npm scripts.
Q5. Why set retries carefully?
Because retries can mask real bugs if used to hide instability instead of fixing it.

● steps/screenshot-attach.ts 
import { createBdd } from 'playwright-bdd';

const { After } = createBdd();

// Attach extra evidence on failure
After(async ({ page }, { testInfo }) => {
  if (testInfo.status === 'failed') {
    // Screenshot
    const png = await page.screenshot({ fullPage: true });
    await testInfo.attach('failure-screenshot', { body: png, contentType: 'image/png' });

    // Page HTML snapshot
    const html = await page.content();
    await testInfo.attach('page-source.html', { body: Buffer.from(html), contentType: 'text/html' });

    // Console + URL for context
    await testInfo.attach('final-url.txt', { body: Buffer.from(page.url()), contentType: 'text/plain' });
  }
});

// Log in once and save the storage state to a file
export default async function globalSetup(_config: FullConfig) {
  const browser = await chromium.launch();
  const context = await browser.newContext();
  const page = await context.newPage();

  await page.goto(process.env.BASE_URL + '/login');
  await page.getByLabel('Username').fill(process.env.QA_USER!);
  await page.getByLabel('Password').fill(process.env.QA_PASS!);
  await page.getByRole('button', { name: /login/i }).click();
  await page.waitForURL('**/dashboard');

  await context.storageState({ path: 'storage/qa.json' });
  await browser.close();
}

# BDD + Playwright-TS Framework — 10 Maintainable Rules

1. **playwright-bdd** for Gherkin -> Playwright generation.
2. **TypeScript strict mode** so step parameters are type-checked.
3. **Page Object Model** — all UI clicks live in page classes.
4. **Thin step definitions** — call page methods, do not click.
5. **Playwright auto-waiting** — no Thread.sleep, no manual waits.
6. **getByRole / getByLabel / getByText** first; CSS only when needed.
7. **storageState + global setup** for fast authenticated flows.
8. **Reporters**: html (debug) + allure (stakeholders) + json (dashboards).
9. **CI archives**: playwright-report/, test-results/, allure-results/.
10. **Living documentation** — every passing run is the spec.

Best Practices Checklist

  • Keep one source of truth in playwright.config.ts.
  • Use storageState for authenticated journeys — log in once, reuse everywhere.
  • Load BASE_URL and secrets from env vars / CI secrets, not from code.
  • Use .env files locally with dotenv; never commit real secrets.
  • Cache node_modules and Playwright browsers in CI for speed.
  • Archive playwright-report/, test-results/, and allure-results/ in CI.
  • Use publishHTML in Jenkins so reports surface on the build page.
  • Keep .features-gen/ out of Git — it is build output.
  • Document tag vocabulary (@smoke, @regression, @wip) in TAGS.md.
  • Run smoke pack on every PR and full regression nightly.

Interview Prep Q&A

Q1. Why use storageState?
To skip the login flow in most tests, which saves time and reduces flakiness.
Q2. How do you install Playwright on a CI agent?
Run `npx playwright install --with-deps` to fetch browsers and OS-level dependencies.
Q3. Why not hardcode BASE_URL?
Because the same suite runs against QA, UAT, and other environments — env vars switch them cleanly.
Q4. Why archive playwright-report/ in CI?
So failures can be debugged from the build page without rerunning anything.
Q5. Explain BDD + Playwright-TS CI/CD in interview?
Code triggers CI, playwright-bdd generates specs from Gherkin, Playwright Test runs them across browsers, and reports + traces are archived for release decisions.
