FOUNDATION — BDD with Cucumber + Selenium Java

Understand BDD philosophy and why Cucumber + Selenium Java is a popular combination for UI automation

What / Why / When / Where / How

WhatBDD (Behavior-Driven Development) describes what the system should do in plain English; Cucumber reads those feature files and runs them as Selenium Java tests.
WhyIt connects business language (Gherkin) to real browser actions (Selenium) so PO, dev, and QA share the same source of truth.
WhenUse it for any UI feature where the business cares about user flows — login, search, checkout, profile updates.
WhereIn a Maven Java project: feature files under src/test/resources, step definitions + runners under src/test/java.
HowDiscuss examples (Three Amigos) -> write Gherkin -> create Selenium step definitions -> wire a TestRunner -> run via Maven Surefire.

Keyword Glossary

BDD
Behavior-Driven Development — describing software with examples of behavior.
Cucumber
Tool that reads Gherkin feature files and runs them against Java step definitions.
Gherkin
Plain-English language used inside .feature files.
Selenium
Java library that drives Chrome, Firefox, Edge browsers.
Selenium Manager
Built-in Selenium 4 driver resolver — auto-downloads drivers.
Three Amigos
Product Owner + Developer + QA discussing examples before coding.
Feature File
File ending in .feature containing a Feature and Scenarios.
Step Definition
Java method mapped to a Gherkin step using @Given/@When/@Then.
Test Runner
Java class with @CucumberOptions that tells Cucumber what to run.
Living Documentation
Up-to-date docs that come from passing feature files.

Interview One-Liners

Cucumber gives the language; Selenium does the clicking.
Selenium Manager removed driver-management pain in Selenium 4.
Feature files are not test scripts — they are shared specifications.
Three Amigos is where most BDD value is created, not in tools.
BDD pays off long term through living documentation.

package com.learn.base;

import org.openqa.selenium.WebDriver;
import org.openqa.selenium.chrome.ChromeDriver;
import org.openqa.selenium.chrome.ChromeOptions;

public class DriverFactory {

    // ThreadLocal so each test gets its own driver — safe for parallel
    private static final ThreadLocal<WebDriver> tlDriver = new ThreadLocal<>();

    public static WebDriver getDriver() {
        if (tlDriver.get() == null) {
            // Selenium 4 auto-downloads the right driver
            ChromeOptions options = new ChromeOptions();
            options.addArguments("--start-maximized");
            tlDriver.set(new ChromeDriver(options));
        }
        return tlDriver.get();
    }

    public static void quitDriver() {
        if (tlDriver.get() != null) {
            tlDriver.get().quit();
            tlDriver.remove();
        }
    }
}

Best Practices Checklist

  • Start with one stable Feature file and one step definition class.
  • Use Selenium Manager — do not commit chromedriver.exe.
  • Use Cucumber Expressions ({string}, {int}) instead of raw regex.
  • Keep feature files focused on business behavior, not UI mechanics.
  • Use ThreadLocal WebDriver from day one to support parallel later.
  • Externalize URLs and credentials into config files.
  • Use TestNG Assert for verifications inside steps.
  • Use explicit waits (Duration.ofSeconds) — never Thread.sleep.
  • Keep step definitions short — push UI logic into Page Objects.
  • Document the team's tag vocabulary (e.g. @smoke, @regression) in TAGS.md.

GHERKIN + STEP DEFINITIONS FOR SELENIUM UI

Map Gherkin scenarios to Selenium 4 step definitions using Cucumber Expressions and explicit waits

What / Why / When / Where / How

WhatStep definitions are Java methods that turn each Gherkin step into a Selenium action or assertion.
WhyWithout step definitions, Cucumber cannot run any browser action — the Gherkin is just text.
WhenWrite them right after a feature file is reviewed and agreed by the Three Amigos.
WhereIn src/test/java under a stepdefs package; one class per business area is a good rule.
HowUse @Given/@When/@Then with Cucumber Expressions, call Selenium methods, and assert with TestNG.

Keyword Glossary

@Given
Cucumber annotation for a Given step.
@When
Cucumber annotation for a When step.
@Then
Cucumber annotation for a Then step.
Cucumber Expression
Modern parameter syntax like {string}, {int}, {word}.
By
Selenium class for locators like By.id, By.cssSelector.
WebDriverWait
Selenium class for explicit waits.
ExpectedConditions
Helper used with WebDriverWait — visibility, clickable, etc.
Duration
Java time class used as Duration.ofSeconds(10).
Assert
TestNG class for assertions inside step methods.
Scenario Outline
Gherkin template that runs once per Examples row.

Interview One-Liners

Step definitions should read like one short sentence in Java.
Cucumber Expressions are easier and safer than raw regex.
Use explicit waits with Duration.ofSeconds — never Thread.sleep.
Use By.id when possible; fall back to By.cssSelector before XPath.
Keep assertions in @Then, not in @Given or @When.

● src/test/java/com/learn/stepdefs/LoginSteps.java 
package com.learn.stepdefs;

import com.learn.base.DriverFactory;
import io.cucumber.java.en.Given;
import io.cucumber.java.en.When;
import io.cucumber.java.en.Then;
import org.openqa.selenium.By;
import org.openqa.selenium.WebDriver;
import org.openqa.selenium.support.ui.ExpectedConditions;
import org.openqa.selenium.support.ui.WebDriverWait;
import org.testng.Assert;

import java.time.Duration;

public class LoginSteps {

    private final WebDriver driver = DriverFactory.getDriver();
    private final WebDriverWait wait = new WebDriverWait(driver, Duration.ofSeconds(10));

    @Given("the user is on the login page")
    public void openLogin() {
        driver.get("https://qa.example.com/login");
    }

    // {string} captures any quoted text — no regex needed
    @When("the user logs in with {string} and {string}")
    public void loginWith(String username, String password) {
        driver.findElement(By.id("username")).sendKeys(username);
        driver.findElement(By.id("password")).sendKeys(password);
        driver.findElement(By.id("loginBtn")).click();
    }

    @Then("the dashboard page should be displayed")
    public void dashboardShown() {
        wait.until(ExpectedConditions.visibilityOfElementLocated(By.id("dashboardTitle")));
        Assert.assertTrue(driver.findElement(By.id("dashboardTitle")).isDisplayed());
    }
}

● cucumber-expressions-cheatsheet.md 
# Cucumber Expressions Cheatsheet

| Expression | Captures           | Java Type |
|------------|--------------------|-----------|
| {string}   | "quoted text"      | String    |
| {int}      | 42                 | int       |
| {float}    | 3.14               | float     |
| {word}     | single word        | String    |
| {bigdecimal}| 100.50            | BigDecimal|

## Rules of thumb
- Use {string} for any quoted business value.
- Use {int} for counts and amounts.
- Use {word} when no spaces or quotes are expected.
- Use Custom Parameter Types for domain words like {currency}.

## Tip
If your annotation starts with ^ and ends with $, you are using regex.
Otherwise you are using a Cucumber Expression.

Best Practices Checklist

  • Use one step definition class per business area, not per feature.
  • Prefer Cucumber Expressions over raw regex.
  • Use ExpectedConditions.visibilityOfElementLocated for read steps.
  • Use ExpectedConditions.elementToBeClickable for click steps.
  • Centralize wait timeouts in a constants class.
  • Use TestNG Assert.assertEquals / assertTrue inside @Then steps.
  • Avoid hardcoded URLs — read them from config.
  • Reuse the same DriverFactory across all step classes.
  • Move long Selenium logic into Page Objects (next session).
  • Run feature files locally before pushing to CI.

Interview Prep Q&A

Q1. Cucumber Expression vs Regex?
Cucumber Expression is simpler (uses {string}, {int}); regex is more powerful but harder to read.
Q2. Why not Thread.sleep?
It wastes time and still flakes — explicit waits are reliable and faster.
Q3. Where should assertions live?
Inside @Then steps so they describe the expected outcome.
Q4. How do step definitions get the WebDriver?
From a shared DriverFactory (ThreadLocal) so all steps use the same driver per scenario.
Q5. Can one step definition match many feature lines?
Yes — parameters let one method handle many similar Gherkin steps.

ADVANCED FEATURES — Background, Hooks, Data Tables, Page Object Model

Reuse setup, capture screenshots on failure, pass tabular data, and integrate Cucumber with POM

What / Why / When / Where / How

WhatBackground reuses setup steps, Hooks run setup/teardown automatically, Data Tables pass tabular data, and Page Object Model keeps step defs clean.
WhyWithout these, feature files repeat, step defs grow huge, and a UI change breaks ten places at once.
WhenAdd Background and POM as soon as you have more than one or two scenarios; add screenshots-on-failure from day one.
WhereBackground lives in .feature files; Hooks and Page Objects live in Java packages alongside step definitions.
HowWrite Background once, hook @Before/@After in a Hooks class, declare List<Map> for data tables, and call Page Object methods from step defs.

Keyword Glossary

Background
Steps that run before each scenario in the feature.
Hook
Method that runs before/after a scenario or step.
@Before
Cucumber hook running before each scenario.
@After
Cucumber hook running after each scenario.
Tagged Hook
Hook that runs only for matching tagged scenarios.
Data Table
Tabular data passed inside a step.
List<Map>
Common Java type for headered Data Tables.
Page Object Model
Pattern where each page is a Java class with locators and actions.
TakesScreenshot
Selenium interface to capture screenshots.
Scenario.attach
Cucumber method to attach evidence to the report.

Interview One-Liners

Background is for setup, not for behavior.
Hooks are perfect for browser launch and screenshot on failure.
Tagged hooks let me reset DB only for @db scenarios.
Data Tables shine when a step needs many fields, not just one or two.
POM keeps step defs short and stable when the UI changes.

● src/test/java/com/learn/hooks/Hooks.java 
package com.learn.hooks;

import com.learn.base.DriverFactory;
import io.cucumber.java.After;
import io.cucumber.java.Before;
import io.cucumber.java.Scenario;
import org.openqa.selenium.OutputType;
import org.openqa.selenium.TakesScreenshot;
import org.openqa.selenium.WebDriver;

public class Hooks {

    @Before
    public void beforeScenario(Scenario scenario) {
        // Driver is created lazily by DriverFactory.getDriver()
        System.out.println("Starting: " + scenario.getName());
    }

    @After
    public void afterScenario(Scenario scenario) {
        WebDriver driver = DriverFactory.getDriver();

        // Attach screenshot only when scenario fails
        if (scenario.isFailed()) {
            byte[] png = ((TakesScreenshot) driver).getScreenshotAs(OutputType.BYTES);
            scenario.attach(png, "image/png", scenario.getName());
        }
        DriverFactory.quitDriver();
    }
}

Best Practices Checklist

  • Keep Background under 4 steps and use it only when EVERY scenario needs that setup.
  • Use Hooks for browser launch, screenshot on failure, and browser quit.
  • Take screenshots only on failure to control disk usage.
  • Use tagged hooks like @Before("@db") for special-case setup.
  • Prefer List<Map<String,String>> for headered Data Tables.
  • Move all UI actions into Page Objects — step defs should call methods, not click directly.
  • Name page methods with business verbs (loginAs, addToCart).
  • Use explicit waits inside Page Objects, never in step definitions.
  • Centralize WebDriver creation in DriverFactory.
  • Avoid putting assertions in Page Objects — keep them in @Then steps.

Interview Prep Q&A

Q1. Why use Background?
To remove repetition when every scenario starts with the same setup steps.
Q2. How is a Data Table different from a Scenario Outline?
Data Table passes a table into one step; Scenario Outline runs a whole scenario per row.
Q3. Why use Page Object Model with Cucumber?
Because step defs should describe behavior, not click buttons — Page Objects handle the clicking.
Q4. How do you capture a screenshot on failure?
In @After, check scenario.isFailed(), then use TakesScreenshot and scenario.attach to attach the image.
Q5. What is a tagged hook?
A hook that runs only for scenarios with a matching tag, like @Before("@db").

TEST RUNNER, TAGS, PARALLEL EXECUTION

Wire features to TestNG, choose what runs with tag expressions, and speed up the suite with parallel execution

What / Why / When / Where / How

WhatThe TestRunner is the entry point Cucumber uses; tags decide what runs; parallel makes the suite faster.
WhyWithout a Runner, nothing runs; without tags, everything runs; without parallel, big suites take hours.
WhenSet up the Runner on day one, add tags when you have more than ~10 scenarios, add parallel once tests are isolated.
WhereRunner classes in src/test/java/.../runners; junit-platform.properties + testng.xml under src/test/resources.
HowAdd @CucumberOptions on the Runner, tag scenarios in features, override scenarios() with parallel DataProvider for TestNG.

Test Runner
Class with @CucumberOptions used to run Cucumber scenarios.
@CucumberOptions
Annotation that configures features, glue, tags, plugin.
Glue
Package path Cucumber scans for step definitions.
Tag
Label like @smoke on Feature or Scenario.
Tag Expression
Boolean like @smoke and not @wip for filtering.
AbstractTestNGCucumberTests
Cucumber base class for TestNG integration.
DataProvider
TestNG mechanism used here to enable parallel scenarios.
Parallel
Running scenarios at the same time using threads.
Surefire
Maven plugin that executes the Runner via testng.xml.
Thread Count
Number of parallel scenarios allowed at once.

Interview One-Liners

The TestRunner is the bridge between Maven Surefire and Cucumber.
Tags are the cleanest way to choose what runs in CI.
Tag expressions let me combine tags like @smoke and not @wip.
Parallel works only after tests are isolated and data-safe.
I keep one Runner per logical pack — smoke, sanity, regression.

# Cucumber Tag Expressions

| Expression                 | Meaning                                  |
|----------------------------|------------------------------------------|
| `@smoke`                   | Only @smoke scenarios                    |
| `@smoke and @login`        | Must have both tags                      |
| `@smoke and not @wip`      | @smoke but skip work-in-progress         |
| `@smoke or @sanity`        | Either one                               |
| `not @manual`              | Skip anything tagged manual              |

## Tips
- Keep tags lowercase and short.
- One tag per intent: pack (@smoke), area (@login), state (@wip).
- Combine with care — too many tags becomes a maze.

Best Practices Checklist

  • Use one Runner per pack (TestRunnerSmoke, TestRunnerRegression).
  • Set monochrome = true for clean console output.
  • Add plugin = pretty, html, json by default.
  • Use clear tag vocabulary: @smoke, @regression, @wip, @db.
  • Never run @wip in CI — only locally.
  • Use tag expressions for combinations like @smoke and not @wip.
  • Override scenarios() with parallel = true only after ThreadLocal driver is in place.
  • Pick thread count based on CPU and app capacity, not blindly high numbers.
  • Use junit-platform.properties for declarative parallel config.
  • Document tags in a TAGS.md so new joiners know the convention.

Extend AbstractTestNGCucumberTests, override scenarios() and annotate with @DataProvider(parallel = true).

A boolean filter like @smoke and not @wip used to choose scenarios.

To enable Cucumber parallel scenarios declaratively without changing Java code.

package com.learn.utils;

import io.cucumber.java.Scenario;
import org.openqa.selenium.OutputType;
import org.openqa.selenium.TakesScreenshot;
import org.openqa.selenium.WebDriver;

public class ScreenshotUtil {

    public static void attachOnFailure(Scenario scenario, WebDriver driver) {
        // Only attach when the scenario failed — saves disk and noise
        if (scenario.isFailed() && driver != null) {
            byte[] png = ((TakesScreenshot) driver).getScreenshotAs(OutputType.BYTES);
            scenario.attach(png, "image/png", scenario.getName());
        }
    }
}

import org.openqa.selenium.WebDriver;
import org.testng.annotations.AfterMethod;
import org.testng.annotations.BeforeMethod;

public class BaseTest {

    @BeforeMethod
    public void setUp() {
        // Driver launched lazily via DriverFactory.getDriver()
        DriverFactory.getDriver();
    }

    @AfterMethod
    public void tearDown() {
        DriverFactory.quitDriver();
    }

    protected WebDriver driver() {
        return DriverFactory.getDriver();
    }
}

● framework-best-practices.md 
# BDD + Selenium Framework — 10 Maintainable Rules

1. **Selenium Manager** for drivers — no manual chromedriver downloads.
2. **DriverFactory + ThreadLocal** for parallel-safe WebDriver.
3. **Page Object Model** — all UI clicks live in page classes.
4. **Thin step definitions** — call page methods, do not click.
5. **Explicit waits** with Duration.ofSeconds, never Thread.sleep.
6. **Cucumber Expressions** over raw regex.
7. **Hooks** for browser launch, screenshot on failure, browser quit.
8. **Tag vocabulary** documented in TAGS.md (@smoke, @regression, @wip).
9. **Reports archived** in CI (html + json + Spark).
10. **Living documentation** — every passing run is the spec.

Best Practices Checklist

  • Always configure at least html and json Cucumber plugins.
  • Add Extent Spark reporter for stakeholder-friendly reports.
  • Attach screenshots only on failure to save disk and time.
  • Treat every run's HTML report as living documentation.
  • Keep WebDriver creation centralized in DriverFactory (ThreadLocal).
  • Use Selenium Manager (Selenium 4) — no manual driver downloads.
  • Move all UI actions into Page Objects.
  • Keep step definitions short and business-focused.
  • Avoid Thread.sleep — use explicit waits in Page Objects.
  • Archive HTML + JSON reports in CI as build artifacts.
