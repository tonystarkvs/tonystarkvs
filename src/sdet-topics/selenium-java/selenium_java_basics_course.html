What

Selenium WebDriver is an open-source browser automation framework. With Java bindings, you control browsers programmatically — click buttons, fill forms, extract data — just like a real user.

Why

Manual regression testing is slow and error-prone. Selenium automates repetitive UI flows, giving teams confidence on every release without human effort per test run.

When

Use Selenium for end-to-end UI tests that verify user-facing functionality. Set it up during the project's QA phase or alongside CI/CD pipeline integration.

Where

Runs on local machines, CI servers (Jenkins, GitHub Actions, GitLab CI), Docker containers, and cloud grids like Sauce Labs or BrowserStack.

How

Install Java + Maven. Add selenium-java & webdrivermanager dependencies in pom.xml. Instantiate a ChromeDriver, navigate to a URL, find elements, perform actions, and assert results using TestNG/JUnit.

// FirstTest.java — Your very first Selenium test
import org.openqa.selenium.WebDriver;
import org.openqa.selenium.chrome.ChromeDriver;
import io.github.bonigarcia.wdm.WebDriverManager;
import org.testng.Assert;
import org.testng.annotations.Test;

public class FirstTest {

    @Test
    public void openGoogle() {
        // Step 1: Auto-download & configure ChromeDriver
        WebDriverManager.chromedriver().setup();

        // Step 2: Launch Chrome browser
        WebDriver driver = new ChromeDriver();

        // Step 3: Navigate to URL
        driver.get("https://www.google.com");

        // Step 4: Assert page title contains expected text
        String title = driver.getTitle();
        Assert.assertTrue(title.contains("Google"));

        // Step 5: Always close the browser
        driver.quit();
    }
}
// DriverSetup.java — Reusable base class for driver init
import org.openqa.selenium.WebDriver;
import org.openqa.selenium.chrome.ChromeDriver;
import io.github.bonigarcia.wdm.WebDriverManager;
import org.testng.annotations.BeforeMethod;
import org.testng.annotations.AfterMethod;

public class DriverSetup {

    // protected = accessible in all child test classes
    protected WebDriver driver;

    @BeforeMethod
    public void setUp() {
        // Auto-manages chromedriver binary version
        WebDriverManager.chromedriver().setup();
        driver = new ChromeDriver();

        // Maximize window to avoid element-not-visible issues
        driver.manage().window().maximize();
    }

    @AfterMethod
    public void tearDown() {
        // quit() kills the browser process completely
        if (driver != null) driver.quit();
    }
}

        // driver.quit()  → closes ALL windows + ends session

Q1. What is the difference between Selenium RC and Selenium WebDriver?
Selenium RC used a server to inject JavaScript into browsers, making it slow and unreliable. WebDriver uses native browser automation APIs (W3C protocol in Selenium 4), communicating directly with browser drivers. WebDriver is faster, more stable, and supports modern browser features natively.
Q2. What does WebDriverManager do and why is it preferred?
WebDriverManager (by BoniGarcia) automatically downloads the browser driver binary that matches the installed browser version at runtime. It eliminates manual driver management and version mismatch failures — a common source of "session not created" errors in CI when Chrome auto-updates.
Q3. Explain driver.close() vs driver.quit() with example.
driver.close() closes only the currently active browser window/tab. driver.quit() closes ALL windows and terminates the entire WebDriver session including the ChromeDriver process. Always use quit() in teardown to prevent process leaks, especially in parallel test execution.
Q4. Why is Selenium 4 considered a major upgrade?
Selenium 4 adopts the W3C WebDriver standard natively (no more JSON Wire Protocol translation), adds CDP (Chrome DevTools Protocol) support for network mocking and console capture, introduces relative locators, improves window/tab management APIs, and includes built-in grid improvements.
Q5. How do you configure Chrome to run in headless mode for CI?
Create a ChromeOptions object, add --headless=new, --no-sandbox, and --disable-dev-shm-usage arguments (the last two required for Docker/Linux), set window size with --window-size=1920,1080, then pass options to new ChromeDriver(options).

What

Locators are strategies used to find HTML elements on a web page. Selenium provides 8 locator types: id, name, className, tagName, linkText, partialLinkText, cssSelector, and xpath.

Why

Before interacting with any element (click, type, assert), Selenium must locate it precisely. Wrong or fragile locators are the #1 cause of flaky test failures in real projects.

When

Every time you interact with a UI element. Priority: id > name > CSS Selector > XPath. Use XPath only when CSS can't express the selector (e.g., parent traversal).

Where

Inspect element locators using Chrome DevTools (F12), copy as CSS/XPath. Tools like ChroPath, SelectorsHub, or XPath Helper assist during development.

How

Use driver.findElement(By.id("...")) or By.cssSelector("...") / By.xpath("..."). driver.findElements() returns a List for multiple matches. Prefer unique, stable attributes over dynamic class names.

import org.openqa.selenium.By;
import org.openqa.selenium.WebElement;

public class LocatorById {

    @Test
    public void locateByAllBasicStrategies() {
        driver.get("https://demo.seleniumeasy.com");

        // 1. By ID — fastest, always prefer when available
        WebElement field = driver.findElement(By.id("user-name"));

        // 2. By Name attribute
        WebElement pwd = driver.findElement(By.name("password"));

        // 3. By Class Name — avoid if class is dynamic
        WebElement btn = driver.findElement(By.className("login-btn"));

        // 4. By Tag Name — usually returns many elements
        WebElement header = driver.findElement(By.tagName("h1"));

        // 5. By Link Text — exact anchor text match
        WebElement link = driver.findElement(By.linkText("Sign In"));

        // 6. By Partial Link Text — substring match
        WebElement link2 = driver.findElement(By.partialLinkText("Sign"));
    }
}

// ByXPath.java — XPath absolute vs relative
public class XPathDemo {

    @Test
    public void xpathExamples() {

        // ABSOLUTE XPath — NEVER use in tests (breaks on DOM change)
        // /html/body/div[2]/form/input[1]

        // RELATIVE XPath — always start with //
        driver.findElement(By.xpath("//input[@id='username']"));

        // By attribute value
        driver.findElement(By.xpath("//input[@type='submit']"));

        // By text content (contains function)
        driver.findElement(By.xpath("//button[contains(text(),'Login')]"));

        // Exact text match
        driver.findElement(By.xpath("//a[text()='Sign In']"));

        // Parent traversal (not possible in CSS!)
        driver.findElement(By.xpath("//label[text()='Email']/following-sibling::input"));

        // Multiple attribute conditions with and
        driver.findElement(By.xpath("//input[@type='text' and @name='email']"));
    }
}

import static org.openqa.selenium.support.locators.RelativeLocator.*;

public class RelativeLocatorDemo {

    @Test
    public void relativeLocatorTest() {
        // Find password field BELOW the username field
        WebElement username = driver.findElement(By.id("username"));

        WebElement password = driver.findElement(
            with(By.tagName("input")).below(username)
        );

        // Find element to the RIGHT of a label
        WebElement label = driver.findElement(By.xpath("//label[text()='Email']"));
        WebElement emailInput = driver.findElement(
            with(By.tagName("input")).toRightOf(label)
        );

        // Other relative methods: above(), toLeftOf(), near()
        password.sendKeys("secret123");
        emailInput.sendKeys("user@test.com");
    }
}

  • ✅
    Never use absolute XPath

    Absolute XPaths like /html/body/div[2]/input break on every DOM change. Always use relative XPath with //.

    Ask devs to add data-testid="login-btn" attributes. These are stable, don't change with styling, and are automation-friendly.

  • ✅
    Use findElements() for existence checks

    Checking !findElements(By.id("x")).isEmpty() avoids NoSuchElementException and is cleaner than try-catch.

    Avoid dynamic class names

    CSS modules and frameworks generate classes like "Button__root--3x7Pd". These change between builds and break tests.

  • ✅
    Validate locators in browser console

    Use $('css') or $x('//xpath') in Chrome DevTools console before adding to code.

Q1. What is the difference between findElement() and findElements()?
findElement() returns a single WebElement and throws NoSuchElementException if not found. findElements() returns a List<WebElement> and returns an empty list if nothing matches — never throws. Use findElements() when checking if an element exists, to avoid exception handling overhead.
Q2. When would you choose XPath over CSS Selector?
XPath is chosen when you need to: (1) locate elements by text content using text() or contains(), (2) traverse upward to parent/ancestor nodes (CSS can only go down the DOM), (3) use complex boolean conditions combining multiple attributes, or (4) locate elements based on sibling relationships using following-sibling:: / preceding-sibling:: axes.
Q3. What are relative locators in Selenium 4?
Relative locators (also called "friendly locators") find elements by their visual position relative to another known element. Methods: above(), below(), toLeftOf(), toRightOf(), near(). Used with with(By.tagName("input")).below(usernameField). Useful for forms where elements lack unique attributes but have predictable visual layouts.
Q4. How do you handle elements with dynamic IDs?
Several strategies: (1) Use CSS attribute^= starts-with or *= contains for partial matching, (2) Use XPath contains(@id,'staticPart'), (3) Use sibling/relative positioning, (4) Use other stable attributes like data-testid, name, or placeholder, (5) Request dev team to add stable test IDs as data-testid attributes.
Q5. How can you validate a CSS selector or XPath in Chrome?
Open Chrome DevTools (F12) → Console tab. For CSS: type $$('your-css-selector') — returns NodeList. For XPath: type $x('//your/xpath') — returns array of matching nodes. Both show count and let you inspect matched elements. Elements tab also has Ctrl+F search supporting both syntaxes.

What

Browser interactions cover all actions performed on web elements: clicking, typing text, selecting dropdown options, handling JavaScript alerts, switching between iframes, and managing multiple browser windows/tabs.

Why

Real user journeys involve complex interactions beyond just finding elements. Handling alerts, frames, and windows correctly is essential for testing modern SPAs and multi-window workflows.

When

Used in virtually every test scenario. Dropdown handling is needed for forms; alert handling for confirmation dialogs; frame switching for embedded widgets; window switching for OAuth flows and new tab navigation.

Where

All interaction methods are on WebElement (click, sendKeys, clear) or WebDriver (navigate, manage, switchTo). The Actions class handles advanced mouse/keyboard interactions.

How

Use element.click(), element.sendKeys(), Select class for dropdowns, driver.switchTo().alert() for popups, driver.switchTo().frame() for iframes, and driver.switchTo().window() for tab/window management.

import java.util.Set;

public class WindowDemo {

    @Test
    public void handleMultipleWindows() {
        driver.get("https://the-internet.herokuapp.com/windows");

        // Save the current (parent) window handle
        String parentWindow = driver.getWindowHandle();

        // Click link that opens a new window
        driver.findElement(By.linkText("Click Here")).click();

        // Get all window handles
        Set<String> allWindows = driver.getWindowHandles();

        // Switch to the new window
        for (String handle : allWindows) {
            if (!handle.equals(parentWindow)) {
                driver.switchTo().window(handle);
                break;
            }
        }

        // Interact with new window
        Assert.assertEquals(driver.getTitle(), "New Window");

        // Close new window and return to parent
        driver.close();
        driver.switchTo().window(parentWindow);
    }
}

Q1. How do you handle a dropdown that is NOT a native HTML select element?
For custom dropdowns (built with div/ul/li), the approach is: (1) Click the dropdown trigger element to open it, (2) Wait for the options list to be visible, (3) Use findElement(By.xpath("//li[text()='Option']")) to locate the option, (4) Click the option element. Q2. What happens if you don't call switchTo().defaultContent() after working in an iframe?
WebDriver context remains inside the iframe. All subsequent findElement() calls will search within the iframe's DOM only, causing NoSuchElementException for elements outside it. Always call driver.switchTo().defaultContent() to return to the main document, or parentFrame() to go one frame level up in nested frames.
Q3. Explain the difference between accept() and dismiss() on an Alert.
accept() clicks the OK/Yes button — equivalent to user pressing Enter or clicking OK. dismiss() clicks the Cancel/No button — equivalent to pressing Escape. For simple alerts (no cancel button), only accept() is valid. For prompt alerts, always call sendKeys() before accept() if you need to input text.
Q4. How do you perform a right-click (context click) in Selenium?
Use the Actions class: new Actions(driver).contextClick(element).perform(). You cannot use element.click() for right-clicks. After triggering the context menu, find and click the desired menu option using a standard findElement().click().
Q5. How do you switch to a newly opened window when you don't know its handle in advance?
Save the current window handle before the action: String parent = driver.getWindowHandle(). After the action opens a new window, get all handles: Set<String> all = driver.getWindowHandles(). Iterate and switch to any handle that is NOT equal to the saved parent handle. This works for any number of windows.

What

Waits are synchronization mechanisms that tell Selenium to pause execution until a condition is met or a timeout expires. Types: Implicit Wait (global), Explicit Wait (condition-based), Fluent Wait (polling interval control).

Why

Modern web apps load asynchronously — elements appear after AJAX calls, animations, or API responses. Without waits, Selenium acts faster than the page loads, causing ElementNotVisibleException and StaleElementReferenceException.

When

Always. Explicit wait before every interaction with dynamically loaded content. Implicit wait set once in setup. Fluent wait for elements with irregular loading patterns like polling dashboards.

Where

Implicit wait is set on the WebDriver instance. Explicit wait uses WebDriverWait with ExpectedConditions. Fluent Wait is configured with FluentWait builder. Never mix implicit + explicit — causes unpredictable timeout behavior.

How

new WebDriverWait(driver, Duration.ofSeconds(10)).until(ExpectedConditions.visibilityOfElementLocated(By.id("x"))). ExpectedConditions provides 30+ pre-built condition checks.

Q1. What is the difference between Implicit, Explicit, and Fluent Wait?
Implicit wait is global — set once and applied to every findElement() call, polling DOM until element appears. Explicit wait uses WebDriverWait with ExpectedConditions for specific element conditions — more targeted. Fluent wait extends explicit wait with configurable polling intervals and exception-ignoring — ideal for irregular load patterns. Never combine implicit with explicit.
Q2. Why should you never use Thread.sleep() in Selenium tests?
Thread.sleep() is a hard-coded static delay that doesn't respond to actual page state. If the page loads in 1s but sleep is 5s, 4s are wasted. If load takes 6s and sleep is 5s, the test still fails. It makes tests slower AND flakier simultaneously. Condition-based waits like WebDriverWait exit as soon as the condition is met, optimizing execution time.
Q3. What is StaleElementReferenceException and how do you handle it?
Thrown when a previously located WebElement reference becomes invalid because the DOM was re-rendered (React re-renders, page navigation, AJAX update). Solutions: (1) Re-find the element after the DOM update, (2) Wait for staleness with ExpectedConditions.stalenessOf(element) then re-locate, (3) Use FluentWait with .ignoring(StaleElementReferenceException.class).
Q4. What does ExpectedConditions.presenceOfElementLocated differ from visibilityOfElementLocated?
presenceOfElementLocated waits until the element exists in the DOM — it may still be hidden (display:none, opacity:0, off-screen). visibilityOfElementLocated waits until the element is in DOM AND is visible (has positive dimensions, is not hidden by CSS). Use visibility before interactions, presence when you only need to read attributes of potentially-hidden elements.
Q5. How would you implement a custom ExpectedCondition?
Implement the ExpectedCondition<T> interface (or use a lambda). The apply(WebDriver d) method should return a non-null, non-false value when condition is met, or null/false to keep polling. Example lambda: wait.until(d -> d.findElements(By.cssSelector("tr")).size() > 5) — waits until table has more than 5 rows.

Q1. How does @DataProvider work in TestNG?
@DataProvider is a method that returns Object[][] — a 2D array where each inner array is one test run's parameters. The test method with @Test(dataProvider="name") receives one inner array per invocation. TestNG runs the test method N times (once per row). Parameters can be any Java type and the method can read from Excel, databases, or JSON.
Q2. What is the difference between @BeforeClass and @BeforeMethod?
@BeforeClass runs ONCE before the first test method in the class — used for setup shared across all tests (one-time browser launch for the class). @BeforeMethod runs before EACH test method — used for per-test setup like navigating to a start URL or clearing cookies between tests. @BeforeClass is more efficient; @BeforeMethod provides better test isolation.
Q3. How do you capture a screenshot on test failure in TestNG?
Implement ITestListener and override onTestFailure(ITestResult result). Inside: cast the test instance to get the WebDriver reference, use ((TakesScreenshot)driver).getScreenshotAs(OutputType.FILE), and copy to a named file (include test name and timestamp). Register the listener in testng.xml under <listeners>.
Q4. Explain how to run tests in parallel using TestNG.
In testng.xml, set parallel="methods" (or "classes"/"tests") and thread-count="4" on the <suite> tag. CRITICAL: each thread must have its own WebDriver instance. Implement IRetryAnalyzer and override retry(ITestResult) — return true to retry, false to fail. Apply via @Test(retryAnalyzer=RetryAnalyzer.class). Use for genuinely flaky tests (network timeouts, timing-sensitive animations). Do NOT use it to mask real bugs — an IRetryListener annotation factory can apply it globally. Limit retries to 1-2; more indicates a deeper problem to fix.

What

Page Object Model is a design pattern where each web page or component is represented as a Java class. The class contains locators and methods that perform actions on that page, separating test logic from UI interaction code.

Why

Without POM, locators are scattered across hundreds of test methods. One UI change requires updating every test. With POM, you update ONE page class, and all tests using it get fixed automatically — massive maintenance reduction.

When

Always — from day one of any real automation project. Even for small projects, POM pays off by the time you have 50+ tests. Critical when multiple team members write tests for the same application.

Where

How

Q1. What is Page Object Model and why is it important?
POM is a design pattern where each web page is represented as a Java class containing locators and action methods. It's important because it separates test logic (what to test) from UI interaction code (how to interact). When the UI changes, only the corresponding page class needs updating — not every test. This dramatically reduces maintenance effort and eliminates code duplication.
Q2. How does PageFactory work internally?
PageFactory.initElements(driver, this) uses Java's dynamic proxy pattern. Each @FindBy-annotated field is replaced with a proxy object. When the field is accessed, the proxy intercepts the call and executes driver.findElement() with the specified locator at that moment (lazy evaluation). This means the locator is NOT evaluated at object creation time.
Q3. Should page objects contain assertions?
No. Page objects should only perform actions (click, type, navigate) and return values (getText, isDisplayed). Assertions belong exclusively in test classes. This keeps page objects reusable — the same loginPage.login() method can be used in positive tests, negative tests, and as a prerequisite setup for other tests. Mixing assertions couples page objects to specific test scenarios.
Q4. Explain the "return page object" pattern in POM.
When a page method causes navigation to another page, it returns an instance of the destination page class. Example: LoginPage.clickLogin() returns new DashboardPage(driver). This enables natural method chaining in tests: loginPage.enterUsername("user").enterPassword("pass").clickLogin().getWelcomeText(). It also makes the flow self-documenting — the return type tells you which page you're on after the action.
Q5. When would you avoid using @FindBy and PageFactory?
Avoid PageFactory for React, Angular, or Vue SPAs that heavily re-render the DOM. The lazy proxy may hold a reference to a stale element after a re-render, causing StaleElementReferenceException. The safer alternative is manually declaring By locators and calling driver.findElement() inside each method — this guarantees a fresh DOM lookup every time the method is called.

import org.openqa.selenium.TakesScreenshot;
import org.openqa.selenium.OutputType;
import org.apache.commons.io.FileUtils;
import java.io.File;
import java.time.LocalDateTime;
import java.time.format.DateTimeFormatter;

public class ScreenshotUtil {

    public static void captureScreenshot(
            WebDriver driver, String testName) {

        // Cast WebDriver to TakesScreenshot interface
        TakesScreenshot ts = (TakesScreenshot) driver;

        // Capture as FILE type (also available: BASE64, BYTES)
        File srcFile = ts.getScreenshotAs(OutputType.FILE);

        // Create timestamped filename
        String timestamp = LocalDateTime.now()
            .format(DateTimeFormatter.ofPattern("yyyyMMdd_HHmmss"));

        String fileName = testName + "_" + timestamp + ".png";
        File destFile = new File("test-output/screenshots/" + fileName);

        try {
            FileUtils.copyFile(srcFile, destFile);
            System.out.println("Screenshot: " + destFile.getAbsolutePath());
        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}

        System.out.println("Title: " + title);

        // Highlight element for visual debugging
        js.executeScript(
            "arguments[0].style.border='3px solid red';", btn);

        // Check if page is fully loaded
        String state = (String) js.executeScript(
            "return document.readyState;");
        Assert.assertEquals(state, "complete");
    }

Q1. How do you capture a screenshot in Selenium and save it to a file?
Cast the WebDriver to TakesScreenshot: File src = ((TakesScreenshot)driver).getScreenshotAs(OutputType.FILE). Copy to destination using Apache Commons IO: FileUtils.copyFile(src, new File("screenshots/test_timestamp.png")). For Extent Reports use OutputType.BASE64 to embed the image directly without file path management.
Q2. What is JavascriptExecutor and when would you use it?
JavascriptExecutor is a WebDriver interface that executes JavaScript in the current browser context. Use it when: (1) element is not interactable via Selenium (hidden, disabled, inside shadow DOM), (2) reading-only fields need values set, (3) scrolling to specific positions or elements, (4) checking page load state via document.readyState, (5) direct DOM manipulation for test setup. Avoid as a first choice — it bypasses Selenium's visibility checks.
Q3. What is Extent Reports and how do you set it up?
Extent Reports is a third-party library for generating rich HTML test reports. Setup: (1) Add extentreports Maven dependency, (2) Create ExtentSparkReporter with output path, (3) Create ExtentReports and attach the reporter, (4) Call extent.createTest(name) per test, (5) Log with test.pass()/fail()/info(), (6) Call extent.flush() in @AfterSuite. Integrate via ITestListener for automated logging without modifying test methods.
Q4. How do you scroll to an element in Selenium?
Three approaches: (1) JS — js.executeScript("arguments[0].scrollIntoView(true)", element) — most reliable, works in nested scroll containers, (2) JS — js.executeScript("window.scrollTo(x, y)") — for coordinate-based scrolling, (3) Selenium 4 native — new Actions(driver).scrollToElement(element).perform() — no JS needed, cleaner syntax. Prefer scrollIntoView for element-specific scroll, especially for lazily-loaded content.
Q5. How do you embed screenshots in Extent Reports for failed tests?
In ITestListener.onTestFailure(): (1) Get screenshot as FILE or BASE64, (2) For FILE: call test.addScreenCaptureFromPath(filePath), (3) For BASE64: call test.addScreenCaptureFromBase64String(base64String). BASE64 embedding is preferred for CI because report is self-contained — screenshots are part of the HTML file itself, no separate image directory to manage or transfer.
