Selenium Java: Multiple Windows & Navigation (Sessions 12 to 16)

Selenium Grid allows running tests on multiple machines, browsers, and operating systems simultaneously. It uses a Hub-Node architecture where the Hub acts as a central router and Nodes are machines with browsers. Grid 4 (Selenium 4) is completely rewritten with Docker/K8s support, a built-in GraphQL API, and a live dashboard at localhost:4444.

Interview one-liners

Selenium Grid enables parallel cross-browser test execution across multiple physical or virtual machines.
The Hub is the central server that receives test requests and routes them to matching Node machines.
A Node is a machine registered to the Hub that has browsers available for test execution.
Grid 4 introduced Standalone mode — a single JAR runs Hub and Node together on one machine.
RemoteWebDriver is the client-side class used to send test commands to a Selenium Grid Hub.
Grid 4 natively supports Docker containers, Kubernetes, and has a live session dashboard at port 4444.

Best Practices

  • ✓Always use RemoteWebDriver with ChromeOptions/FirefoxOptions — never use deprecated DesiredCapabilities on Grid
  • ✓Use Standalone mode for local development and Hub+Node mode for CI/CD multi-machine setups
  • ✓Set "se:name" capability so Grid dashboard shows meaningful session names instead of random IDs
  • ✓Use Docker Compose for reproducible Grid environments — prevents "works on my machine" issues in CI
  • ✓Always call driver.quit() to release the Grid session slot — orphaned sessions block other tests
  • ✓Monitor sessions via Grid dashboard at http://hub-ip:4444 to debug capacity and queue issues

SDET Interview Q&A

Q1. What is the difference between Hub and Node in Selenium Grid?
A: Hub is the central server (Router + Distributor + SessionMap) that receives test requests and routes them. Node is a machine registered to the Hub that runs actual browsers. A Hub can have many Nodes, each with different OS/browser combinations.
Q2. What is Standalone mode in Grid 4?
A: Standalone mode starts both Hub and Node components in a single process on one machine: java -jar selenium-server.jar standalone. Ideal for local testing without separate Hub+Node setup. Listens on port 4444 by default.
Q3. How do you run tests in parallel on Selenium Grid using TestNG?
A: In testng.xml, set parallel="tests" and define multiple <test> blocks with different <parameter name="browser"> values. Each parallel thread creates its own RemoteWebDriver session on the Grid Hub, which routes to available Nodes.
Q4. What new components were introduced in Grid 4 compared to Grid 3?
A: Grid 4 introduced: Router (entry point), Distributor (session matching), SessionMap (session registry), SessionQueue (pending requests), and Event Bus (internal messaging). Grid 3 had only Hub and Node. Grid 4 also added GraphQL API and built-in UI dashboard.
Q5. What happens if you forget to call driver.quit() on a Grid session?
A: The browser process on the Node remains open, consuming the session slot. If the Grid has a max session limit (default: 1 per CPU core), the slot is occupied and new tests queue or timeout waiting. Always quit in a finally block or TestNG @AfterMethod.

Overview — JavascriptExecutor

JavascriptExecutor is a Selenium interface that allows executing raw JavaScript code directly in the browser context. It is used when standard WebDriver commands cannot interact with hidden elements, shadow DOMs, SPAs, or when you need to read computed values, scroll precisely, or manipulate the DOM in ways WebDriver does not natively support.

Interview one-liners

JavascriptExecutor is a Selenium interface — cast the driver to it to access executeScript() and executeAsyncScript().
executeScript() runs synchronous JavaScript and returns the result to Java as an Object.
executeAsyncScript() runs asynchronous JavaScript — must call the callback argument to return control.
arguments[0] in JavaScript refers to the first Java object (WebElement or value) passed to executeScript().
JavascriptExecutor is used when click() fails on hidden/overlapping elements — js.executeScript("arguments[0].click()", el).
scrollIntoView(true) aligns the element to the top of the viewport; false aligns to the bottom.

All JavascriptExecutor methods

executeScript(script, args...)Runs synchronous JS. Returns value (String, Long, Boolean, List, WebElement, null)
executeAsyncScript(script, args...)Runs async JS. Must call arguments[N] as callback to return. Used for AJAX waits
arguments[0], [1], [2]...Refers to Java objects passed after the script string — WebElements, Strings, numbers

JSAsyncScriptDemo.java

 1// ── executeAsyncScript() — Wait for async JS operations ──────────
 2// executeAsyncScript injects a callback as the LAST argument
 3// Your JS code MUST call this callback to signal completion
 4// Used for: AJAX waits, setTimeout, promises, animation completion
 5import org.openqa.selenium.JavascriptExecutor;
 6import org.openqa.selenium.WebDriver;
 7import org.openqa.selenium.chrome.ChromeDriver;
 8import java.time.Duration;
 9
10public class JSAsyncScriptDemo {
11  public static void main(String[] args) {
12    WebDriver driver = new ChromeDriver();
13    driver.get("https://www.example.com");
14    JavascriptExecutor js = (JavascriptExecutor) driver;
15
16    // scriptTimeout → how long to wait for the async callback to fire
17    driver.manage().timeouts().scriptTimeout(Duration.ofSeconds(10));
18
19    // EXAMPLE 1: Wait for 2 seconds using setTimeout via JS
20    // arguments[arguments.length-1] → the auto-injected callback
21    // JavaScript calls the callback after 2000ms, which returns "done"
22    Object result = js.executeAsyncScript(
23      "var cb = arguments[arguments.length - 1];" +
24      "setTimeout(function(){ cb('done after 2s'); }, 2000);"
25    );
26    System.out.println("Async result: " + result); // "done after 2s"
27
28    // EXAMPLE 2: Wait for page readyState to be "complete"
29    js.executeAsyncScript(
30      "var cb = arguments[arguments.length - 1];" +
31      "var check = setInterval(function(){" +
32      "  if(document.readyState === 'complete') { clearInterval(check); cb(); }" +
33      "}, 100);"
34    );
35    System.out.println("Page fully ready!");
36    driver.quit();
37  }

Best Practices

  • ✓Always cast driver to JavascriptExecutor — do NOT use instanceof check unless testing an interface
  • ✓Use JS click as a LAST RESORT — prefer standard WebElement.click() first; JS click bypasses accessibility checks
  • ✓Always set scriptTimeout before executeAsyncScript() — default timeout is 0 in some drivers
  • ✓Cast the return type explicitly ((String), (Long), (Boolean)) — storing as raw Object causes classCastException later
  • ✓Numbers returned from JS are always Long in Java — never Integer. Cast to Long or int after

SDET Interview Q&A

Q1. What is the difference between executeScript() and executeAsyncScript()?
A: executeScript() runs synchronous JS and returns immediately with the result. executeAsyncScript() runs asynchronous JS — Selenium waits for the injected callback to be called before proceeding. Use async for setTimeout, AJAX, or promise-based operations.
Q2. What does arguments[0] mean in JavascriptExecutor?
A: It refers to the first Java object passed as an argument after the script string. For example, executeScript("arguments[0].click();", myElement) — arguments[0] is myElement. You can pass multiple args: arguments[1], arguments[2], etc.
Q3. What types can executeScript() return?
A: String, Long (all JS numbers become Long), Boolean, List<Object> (JS arrays), WebElement (JS DOM elements), null. Never returns Integer — always Long for numbers. Cast explicitly before using.
Q4. When would you use JavascriptExecutor over standard WebDriver commands?
A: When standard Selenium fails: clicking hidden/overlapping elements, setting readonly/disabled field values, reading innerText/innerHTML, scrolling precisely, manipulating localStorage/sessionStorage, handling SPAs with pushState navigation, or removing attribute restrictions.
Q5. How do you check if a page is fully loaded using JavascriptExecutor?
A: Execute "return document.readyState;" — if it returns "complete", the page is fully loaded. Used in custom wait conditions when pageLoadTimeout isn't sufficient for SPA content that loads after initial DOM ready.

Overview — Actions Class (Mouse Operations)

The Actions class in Selenium provides an API for complex user gesture simulations that cannot be done with simple click() or sendKeys(). It builds an action chain and executes them all with perform(). This session covers all mouse-related operations: hover, drag-and-drop, right-click, double-click, click-and-hold, and coordinate-based interactions.

Interview one-liners

Actions class builds a chain of user gestures and executes them all at once with perform().
moveToElement() simulates hovering the mouse over an element — triggers CSS :hover state and tooltip display.
contextClick() simulates a right-click, opening the browser or application context menu.
doubleClick() simulates a double left-click — used for text selection, inline editing, and file rename.
dragAndDrop(source, target) picks up the source element and drops it on the target element.
clickAndHold() presses and holds the mouse button without releasing — paired with release() to drop.

All Actions mouse methods

moveToElement(element)Mouse hover — positions cursor over element
click(element)Left-click (same as WebElement.click but chainable)
doubleClick(element)Double left-click on element
contextClick(element)Right-click on element
clickAndHold(element)Press and hold left mouse button
release(element)Release held mouse button at element
dragAndDrop(src,tgt)Drag source element and drop on target
dragAndDropBy(el,x,y)Drag element by pixel offset from its current position
moveByOffset(x,y)Move mouse cursor by pixel offset from current position
perform()EXECUTE all queued actions — MUST be called at end of chain

Best Practices

  • ✓ALWAYS call perform() at the end — actions are queued and do nothing without it
  • ✓Scroll the element into view (via JS or scrollIntoView) before moveToElement() — hovering off-screen fails
  • ✓Prefer dragAndDrop(source, target) for most cases; use clickAndHold → moveToElement → release for animated drag interactions
  • ✓Chain multiple actions into one perform() call — separate perform() calls reset cursor state between them
  • ✓For moveByOffset, remember coordinates are relative to current cursor position, NOT page origin

SDET Interview Q&A

Q1. What is the Actions class and why is perform() mandatory?
A: The Actions class builds a sequence of user gestures (mouse/keyboard) into an action chain. perform() is mandatory because it actually executes the entire queued chain. Without it, nothing happens — actions are only described, not performed.
Q2. What is the difference between dragAndDrop() and clickAndHold + moveToElement + release?
A: dragAndDrop(src,tgt) is a shortcut that combines hold+move+release internally. The manual approach gives more control — you can add intermediate hover steps, pauses, or move through specific path coordinates — useful for slider controls and canvas interactions.
Q3. How does moveByOffset() differ from moveToElement()?
A: moveToElement(el) moves to the CENTER of a specific WebElement regardless of current cursor position. moveByOffset(x,y) moves from the CURRENT cursor position by the given pixel offsets — requires knowing where the cursor currently is.
Q4. Can you chain Actions with Keyboard operations?
A: Yes. The same Actions object supports chaining mouse AND keyboard operations: actions.moveToElement(el).click().sendKeys(Keys.ENTER).perform(). All keyboard and mouse methods can be freely mixed in one chain before a single perform().
Q5. What exception is thrown if you call moveToElement on an element not in the viewport?
A: MoveTargetOutOfBoundsException — thrown when the target coordinates are outside the current browser viewport. Always scroll the element into view before using Actions mouse methods.

Overview — Actions Class (Keyboard Operations)

The Actions class also handles keyboard interactions beyond simple sendKeys(). It can simulate individual key presses, holds, and releases — including modifier keys like Ctrl, Shift, and Alt. This is essential for keyboard shortcuts, text selection, copy-paste, and hotkey testing.

Interview one-liners

sendKeys(Keys.CONTROL, "a") via Actions selects all text — equivalent to pressing Ctrl+A.
keyDown(Keys.SHIFT) holds the Shift key pressed for all subsequent actions until keyUp() is called.
keyUp(Keys.SHIFT) releases a previously held modifier key.
Actions.sendKeys() is for key sequences on the focused element or the whole page.
Keys enum provides constants for all special keys: ENTER, TAB, ESCAPE, BACK_SPACE, CONTROL, ALT, SHIFT, F1-F12.
Keyboard shortcuts like Ctrl+C, Ctrl+V, Ctrl+Z can be simulated using keyDown+sendKeys+keyUp chain.

All Actions keyboard methods

keyDown(modifier)Press and HOLD a modifier key (CTRL, SHIFT, ALT, META)
keyUp(modifier)RELEASE a previously held modifier key
sendKeys(element, keys)Type keys on a specific WebElement (focus + type)
Keys enum constantsENTER, TAB, ESCAPE, SPACE, BACK_SPACE, DELETE, HOME, END, ARROW_UP/DOWN/LEFT/RIGHT, F1-F12, CONTROL, ALT, SHIFT, COMMAND

SDET Interview Q&A

Q1. What is the difference between sendKeys(Keys.ENTER) on a WebElement vs Actions.sendKeys(Keys.ENTER)?
A: element.sendKeys(Keys.ENTER) sends Enter to that specific focused element. actions.sendKeys(Keys.ENTER).perform() sends Enter to the currently focused element on the page — no explicit element target. Both work, but element-based is more explicit and reliable.
Q2. What is Keys.chord() and when do you use it?
A: Keys.chord(key1, key2...) creates a composite keystroke string combining multiple keys pressed simultaneously. Used as a shorthand: sendKeys(Keys.chord(Keys.CONTROL, "a")) is equivalent to keyDown(CTRL) + sendKeys("a") + keyUp(CTRL) but in a single method call.
Q3. What happens if you call keyDown(Keys.SHIFT) but forget keyUp()?
A: The SHIFT key remains held for all subsequent WebDriver operations in that session — all text typed becomes uppercase, all clicks become shift+clicks (selection), and navigation shortcuts change behavior. Always release modifier keys explicitly with keyUp().
Q4. How do you type uppercase letters without using toUpperCase() in Java?
A: Use actions.keyDown(Keys.SHIFT).sendKeys("hello").keyUp(Keys.SHIFT).perform() — holding SHIFT while typing makes the browser produce uppercase characters, exactly as a human typist would.
Q5. What is the difference between Keys.ENTER and Keys.RETURN?
A: They are functionally identical in Selenium — both send the Enter/Return key event. Keys.ENTER is typically used for form submission; Keys.RETURN is the carriage-return equivalent. In practice, browsers treat them the same for most elements.

Overview — Robot Class

The Java Robot class (java.awt.Robot) generates native OS-level keyboard and mouse events — below the browser. Unlike Selenium's Actions, Robot interacts with the entire operating system, making it essential for handling OS-level dialogs (file upload dialogs, Windows authentication, native OS popups) that Selenium cannot reach.

Interview one-liners

Robot class is a Java AWT class that generates native OS-level keyboard and mouse events at the system level.
Robot is used when Selenium cannot handle OS-level dialogs like file upload windows, Windows auth, and native popups.
robot.keyPress(KeyEvent.VK_ENTER) presses a key at the OS level — works even outside the browser.
robot.mouseMove(x, y) moves the system mouse cursor to absolute screen coordinates.
robot.delay(ms) pauses Robot execution for the specified milliseconds — not Thread.sleep().
Robot must be combined with StringSelection and Toolkit to type text into OS-level dialogs via clipboard.

All Robot class methods

new Robot()Instantiate — throws AWTException if platform doesn't support it
keyPress(int keycode)Press key down at OS level (KeyEvent.VK_* constants)
keyRelease(int keycode)Release key at OS level — always pair with keyPress
mouseMove(int x, int y)Move OS cursor to absolute screen pixel coordinates
mousePress(int buttons)Press mouse button (InputEvent.BUTTON1_DOWN_MASK = left)
mouseRelease(int buttons)Release mouse button — always pair with mousePress
mouseWheel(int amount)Scroll mouse wheel — positive=down, negative=up
delay(int ms)Pause Robot for N milliseconds before next action
createScreenCapture(rect)Capture screenshot of screen region as BufferedImage
getPixelColor(x, y)Returns java.awt.Color of pixel at given screen coordinates

 2// Selenium cannot interact with OS-native file chooser dialogs
 3// Robot + Clipboard paste is the standard approach
 4// StringSelection → Java clipboard entry for text
 5import java.awt.Robot;
 6import java.awt.Toolkit;                     // OS toolkit for clipboard
 7import java.awt.datatransfer.StringSelection; // clipboard text wrapper
 8import java.awt.event.KeyEvent;
 9import org.openqa.selenium.By;
10import org.openqa.selenium.WebDriver;
11import org.openqa.selenium.chrome.ChromeDriver;
12
13public class RobotFileUploadDemo {
14  public static void main(String[] args) throws Exception {
15    WebDriver driver = new ChromeDriver();
16    driver.get("https://the-internet.herokuapp.com/upload");
17
18    // Click the file input button to OPEN the OS file dialog
19    driver.findElement(By.id("file-upload")).click();
20    Thread.sleep(1500); // wait for OS dialog to open
21
22    Robot robot = new Robot();
23
24    // STEP 1: Set the file path text onto the OS CLIPBOARD
25    // StringSelection wraps text for clipboard use
26    StringSelection filePathSel = new StringSelection("C:\\TestFiles\\sample.pdf");
27    Toolkit.getDefaultToolkit().getSystemClipboard()
28           .setContents(filePathSel, filePathSel);
29
30    // STEP 2: Ctrl+V → paste the file path into the OS dialog's text field
31    robot.keyPress(KeyEvent.VK_CONTROL);
32    robot.keyPress(KeyEvent.VK_V);       // VK_V = the letter V
33    robot.keyRelease(KeyEvent.VK_V);
34    robot.keyRelease(KeyEvent.VK_CONTROL);
35
36    // STEP 3: Press ENTER to confirm file selection in the OS dialog
37    robot.keyPress(KeyEvent.VK_ENTER);
38    robot.keyRelease(KeyEvent.VK_ENTER);
39
40    System.out.println("File upload dialog handled via Robot!");
41    driver.quit();
42  }
43}
RobotMouseDemo.java

 1// ── Robot mouse operations — OS-level cursor control ─────────────
 2// mouseMove → move OS cursor to screen coordinates (ABSOLUTE pixels)
 3// mousePress/mouseRelease → OS-level mouse button events
 4import java.awt.Robot;
 5import java.awt.event.InputEvent; // InputEvent.BUTTON*_DOWN_MASK constants
 6
 7public class RobotMouseDemo {
 8  public static void main(String[] args) throws Exception {
 9    Robot robot = new Robot();
10
11    // mouseMove(x, y) → move OS cursor to ABSOLUTE screen coordinates
12    // (0,0) = top-left of primary monitor
13    robot.mouseMove(500, 400); // move cursor to x=500, y=400 on screen
14    robot.delay(500);
15
16    // BUTTON1_DOWN_MASK = left mouse button
17    // BUTTON2_DOWN_MASK = middle mouse button (scroll wheel click)
18    // BUTTON3_DOWN_MASK = right mouse button
19    robot.mousePress(InputEvent.BUTTON1_DOWN_MASK);   // left-click DOWN
20    robot.mouseRelease(InputEvent.BUTTON1_DOWN_MASK); // left-click UP
21    System.out.println("OS left-click at (500, 400)");
22
23    // Right-click via Robot
24    robot.mousePress(InputEvent.BUTTON3_DOWN_MASK);
25    robot.mouseRelease(InputEvent.BUTTON3_DOWN_MASK);
26    System.out.println("OS right-click at (500, 400)");
27
28    // mouseWheel(amount) → scroll the mouse wheel
29    // Positive = scroll DOWN; Negative = scroll UP
30    // Each unit = one notch of the scroll wheel
31    robot.mouseWheel(3);  // scroll down 3 notches
32    robot.mouseWheel(-2); // scroll up 2 notches
33
34    System.out.println("Robot mouse scroll completed!");
35  }
36}
RobotScreenshotDemo.java

 1// ── Robot Screenshot — Capture OS-level screen regions ───────────
 2// createScreenCapture() captures the ENTIRE SCREEN (not just browser)
 3// Useful for: OS dialogs, taskbar, multiple monitors, desktop content
 4import java.awt.AWTException;
 5import java.awt.Color;         // java.awt.Color → RGB color representation
 6import java.awt.Rectangle;     // Rectangle → screen capture region
 7import java.awt.Robot;
 8import java.awt.Toolkit;
 9import java.awt.image.BufferedImage; // in-memory image object
10import javax.imageio.ImageIO;
11import java.io.File;
12
13public class RobotScreenshotDemo {
14  public static void main(String[] args) throws Exception {
15    Robot robot = new Robot();
16
17    // Toolkit.getDefaultToolkit().getScreenSize() → get full screen dimensions
18    java.awt.Dimension screen = Toolkit.getDefaultToolkit().getScreenSize();
19
20    // Rectangle(x, y, width, height) → define capture area
21    Rectangle fullScreen = new Rectangle(0, 0, (int)screen.getWidth(), (int)screen.getHeight());
22
23    // createScreenCapture(rect) → returns BufferedImage of that screen region
24    BufferedImage screenshot = robot.createScreenCapture(fullScreen);
25
26    // ImageIO.write(image, format, file) → save to disk as PNG/JPG/BMP
27    ImageIO.write(screenshot, "png", new File("robot-screenshot.png"));
28    System.out.println("Full OS screen captured to robot-screenshot.png");
29
30    // getPixelColor(x, y) → returns Color at screen coordinate — useful for
31    // visual regression or checking OS dialog button states
32    Color pixel = robot.getPixelColor(500, 300);
33    System.out.println("Pixel at (500,300): R="+pixel.getRed()+" G="+pixel.getGreen()+" B="+pixel.getBlue());
34  }
35}
RobotWindowsAuthDemo.java

 1// ── Robot + Windows Authentication Dialog ────────────────────────
 2// When a site shows Basic/Windows Auth, a native OS dialog appears
 3// Selenium CANNOT handle this — Robot + clipboard is the solution
 4import java.awt.Robot;
 5import java.awt.Toolkit;
 6import java.awt.datatransfer.StringSelection;
 7import java.awt.event.KeyEvent;
 8import org.openqa.selenium.WebDriver;
 9import org.openqa.selenium.chrome.ChromeDriver;
10
11public class RobotWindowsAuthDemo {
12  public static void main(String[] args) throws Exception {
13
14    // APPROACH 1: URL-embedded credentials (Basic Auth — Selenium 4 way)
15    // Format: http://username:password@site.com
16    // Chrome may block this for security — use Robot for Windows/NTLM auth
17    WebDriver driver = new ChromeDriver();
18    driver.get("http://admin:admin@the-internet.herokuapp.com/basic_auth");
19
20    // APPROACH 2: Robot to type into Windows OS authentication dialog
21    Robot robot = new Robot();
22    robot.delay(2000); // wait for OS dialog to appear
23
24    // Type username via clipboard → Robot Ctrl+V
25    StringSelection user = new StringSelection("admin");
26    Toolkit.getDefaultToolkit().getSystemClipboard().setContents(user, user);
27    robot.keyPress(KeyEvent.VK_CONTROL); robot.keyPress(KeyEvent.VK_V);
28    robot.keyRelease(KeyEvent.VK_V);     robot.keyRelease(KeyEvent.VK_CONTROL);
29
30    // TAB to password field
31    robot.keyPress(KeyEvent.VK_TAB); robot.keyRelease(KeyEvent.VK_TAB);
32
33    // Type password via clipboard → Robot Ctrl+V
34    StringSelection pass = new StringSelection("admin");
35    Toolkit.getDefaultToolkit().getSystemClipboard().setContents(pass, pass);
36    robot.keyPress(KeyEvent.VK_CONTROL); robot.keyPress(KeyEvent.VK_V);
37    robot.keyRelease(KeyEvent.VK_V);     robot.keyRelease(KeyEvent.VK_CONTROL);
38
39    // Press ENTER to submit
40    robot.keyPress(KeyEvent.VK_ENTER); robot.keyRelease(KeyEvent.VK_ENTER);
41    driver.quit();
42  }
43}
RobotVsActionsComparison.java

 1// ── Robot vs Actions — When to use which ─────────────────────────
 2// RULE: Use Actions for web page elements; Robot for OS-level dialogs
 3// This demo shows the correct tool for different scenarios
 4import java.awt.Robot;
 5import java.awt.event.KeyEvent;
 6import org.openqa.selenium.By;
 7import org.openqa.selenium.Keys;
 8import org.openqa.selenium.WebDriver;
 9import org.openqa.selenium.WebElement;
10import org.openqa.selenium.chrome.ChromeDriver;
11import org.openqa.selenium.interactions.Actions;
12
13public class RobotVsActionsComparison {
14  public static void main(String[] args) throws Exception {
15    WebDriver driver = new ChromeDriver();
16    Actions  actions = new Actions(driver);
17    Robot    robot   = new Robot();
18
19    // USE CASE 1: Hover over a menu item → USE ACTIONS (browser element)
20    WebElement menu = driver.findElement(By.id("main-nav"));
21    actions.moveToElement(menu).perform(); // ACTIONS → browser DOM element
22
23    // USE CASE 2: Press ESCAPE on a web dialog → USE ACTIONS or sendKeys
24    actions.sendKeys(Keys.ESCAPE).perform(); // ACTIONS → browser-level event
25
26    // USE CASE 3: File upload OS dialog → USE ROBOT (OS-level)
27    driver.findElement(By.id("file-input")).click();
28    Thread.sleep(1000);
29    robot.keyPress(KeyEvent.VK_ESCAPE); // ROBOT → OS dialog dismiss
30    robot.keyRelease(KeyEvent.VK_ESCAPE);
31
32    // USE CASE 4: Windows Authentication dialog → USE ROBOT (OS-level)
33    // Windows Auth is a system dialog → only Robot or URL credentials work
34    // actions.sendKeys("password") would FAIL — focus is on OS dialog, not browser
35
36    // QUICK DECISION RULE:
37    // Can you inspect the element in DevTools? → ACTIONS / WebDriver
38    // Element is outside the browser? → ROBOT
39
40    driver.quit();
41  }
42}

Best Practices

  • ✓Always call keyRelease() immediately after every keyPress() — leaked OS keys affect all applications system-wide
  • ✓Use robot.delay() between actions, not Thread.sleep() — Robot's delay is designed for action sequencing
  • ✓Use clipboard + Ctrl+V approach (via StringSelection) for typing into OS dialogs — direct keystrokes require handling each character separately including special chars
  • ✓Robot does NOT work in headless mode — it requires an actual display. Use it only in GUI environments or with virtual displays (Xvfb on Linux)
  • ✓Prefer the URL-embedded credential approach (http://user:pass@site.com) over Robot for Basic Auth — simpler and more reliable
  • ✓Use sendKeys() on a visible file input element instead of Robot when possible — driver.findElement(By.id("file")).sendKeys("/path/to/file") avoids the OS dialog entirely

SDET Interview Q&A

Q1. What is the difference between Robot class and Actions class?
A: Actions class interacts with browser-level events — it works within the WebDriver session on DOM elements. Robot class generates native OS-level events that work on any window/dialog open on the screen, including OS file choosers and Windows Authentication dialogs that Selenium cannot reach.
Q2. How do you handle OS-level file upload dialogs using Robot?
A: (1) Click the file input button to open the OS dialog; (2) Put the file path into the OS clipboard using StringSelection and Toolkit.getSystemClipboard().setContents(); (3) Use Robot to press Ctrl+V to paste; (4) Press Enter to confirm. This avoids typing each character individually.
Q3. Why doesn't Robot work in headless mode?
A: Robot interacts with a physical or virtual display device. In headless mode, no display is initialized. new Robot() throws AWTException: headless environment. Use a virtual framebuffer (Xvfb on Linux) to simulate a display, or avoid Robot in headless CI pipelines by using Selenium's native file input sendKeys() approach instead.
Q4. What is KeyEvent.VK_* and how does it differ from Selenium's Keys enum?
A: KeyEvent.VK_* are Java AWT virtual key codes for OS-level events (e.g., KeyEvent.VK_ENTER = 10). Selenium.Keys are Unicode characters mapped to keyboard keys for browser-level WebDriver events. They both represent keyboard input but at completely different system layers.
Q5. What is the best way to handle file upload in Selenium WITHOUT using Robot?
A: For HTML <input type="file"> elements, simply call element.sendKeys("/absolute/path/to/file.pdf") on the file input directly — no clicking needed. Selenium's sendKeys on a file input sets the file path without triggering the OS dialog. Robot is only needed for non-standard file upload implementations.
