Selenium Java Learning Hub: Basic Topics

Selenium is a free, open-source browser automation framework that lets you control web browsers programmatically. Think of it as a "robot hand" that types, clicks, and navigates websites exactly like a human — but automated, fast, and repeatable. It is used for functional testing, regression testing, and UI automation of web applications.

Interview one-liners

Selenium is an open-source tool used to automate web browser interactions for testing purposes.
Selenium supports multiple browsers like Chrome, Firefox, Edge, and Safari via their native drivers.
Selenium is not a testing framework — it is a browser automation library used alongside TestNG or JUnit.
Selenium supports multiple languages: Java, Python, C#, JavaScript, Ruby, and Kotlin.
Selenium WebDriver directly communicates with the browser without any intermediate server in v3.
Selenium is platform-independent and can run on Windows, Linux, and macOS environments.

Best Practices

  • ✓Always call driver.quit() in a finally block — ensures browser closes even if test fails
  • ✓Never hardcode browser driver paths — use WebDriverManager or Selenium Manager (v4.6+)
  • ✓Set implicit wait once at test setup, not inside individual test methods
  • ✓Use ChromeOptions for headless mode in CI/CD environments
  • ✓Always assert page title or URL after navigation to confirm correct page loaded
  • ✓Use interface type WebDriver driver (not ChromeDriver driver) for browser flexibility

SDET Interview Q&A

Q1. What is the difference between driver.close() and driver.quit()?
A: close() closes only the currently focused tab. quit() closes ALL browser windows and terminates the entire WebDriver session, freeing all system resources.
Q2. What is WebDriver in Selenium?
A: WebDriver is an interface (API contract) in Selenium that provides methods to interact with a browser. Classes like ChromeDriver and FirefoxDriver implement this interface for specific browsers.
Q3. Is Selenium a testing framework?
A: No. Selenium is a browser automation library. For test management (assertions, reports, structure), you combine it with frameworks like TestNG or JUnit.
Q4. What languages does Selenium support?
A: Selenium officially supports Java, Python, C#, JavaScript (Node.js), Ruby, and Kotlin with maintained client bindings.
Q5. What happens if you don't call quit() after tests?
A: Browser processes remain open in OS memory. ChromeDriver orphan processes accumulate causing memory leaks and test instability in subsequent runs.

Overview — Selenium Architecture

Selenium's architecture defines HOW your Java code talks to the browser. In Selenium 3, it used the JSON Wire Protocol. In Selenium 4, it uses the native W3C WebDriver Protocol — a standardized HTTP-based specification. Understanding this flow is critical for any SDET interview.

Interview one-liners

Selenium 4 uses the W3C WebDriver standard protocol, making it more stable and browser-compatible.
In Selenium's architecture, Java client sends HTTP requests to the browser driver, which forwards to the actual browser.
ChromeDriver acts as a bridge (middleware) between Selenium WebDriver and the Chrome browser.
W3C protocol eliminates the need for JSON Wire Protocol translation layer used in Selenium 3.
Each browser has its own driver: ChromeDriver for Chrome, GeckoDriver for Firefox, EdgeDriver for Edge.
Selenium Grid extends the architecture to run tests remotely on different machines simultaneously.

SDET Interview Q&A

Q1. What protocol does Selenium 4 use and why is it better than Selenium 3?
A: Selenium 4 natively uses W3C WebDriver protocol, eliminating the JSON Wire Protocol translation layer from v3. This means fewer compatibility issues and more stable cross-browser behavior.
Q2. What is the role of ChromeDriver in Selenium architecture?
A: ChromeDriver is an executable that acts as a bridge between Selenium's HTTP commands and Chrome's DevTools Protocol (CDP). It translates WebDriver commands into Chrome-understandable instructions.
Q3. What is Selenium Grid and when do you use it?
A: Selenium Grid allows parallel test execution across multiple machines and browsers. Use it when you need to cut test execution time via parallelism or need cross-browser/cross-OS testing on different machines.
Q4. How does RemoteWebDriver differ from ChromeDriver?
A: ChromeDriver runs locally. RemoteWebDriver sends commands over HTTP to a remote machine (Grid Hub), enabling execution on remote/cloud infrastructure.
Q5. What changed in Selenium Grid 4 vs Grid 3?
A: Grid 4 introduced a new architecture: Router, Distributor, SessionMap, and Node. It also supports Docker, Kubernetes, and has a built-in GraphQL API for monitoring sessions.

Overview — Selenium Components

Selenium is a suite of 4 components, each serving a different purpose. Understanding what each component does, when to use it, and how they relate is foundational knowledge for any automation engineer.

Interview one-liners

Selenium WebDriver is the core component for programmatic browser control using code.
Selenium IDE is a browser plugin for record-and-playback test creation — no coding required.
Selenium Grid enables parallel test execution across multiple machines and browser configurations.
Selenium RC (Remote Control) is the predecessor to WebDriver and is now fully deprecated.
Selenium WebDriver replaced RC because it communicates directly with browsers without a proxy server.
Selenium Manager (v4.6+) auto-manages browser driver binaries without manual setup.

Best Practices

  • ✓Prefer By.id() and By.cssSelector() over XPath — faster and more readable
  • ✓Use findElements() (plural) to check element existence — returns empty list, not exception
  • ✓Use Actions class for hover/drag — don't use JavaScript unless WebDriver can't handle it
  • ✓Capture screenshots on test failure in @AfterMethod hooks for evidence
  • ✓Avoid XPath with absolute paths — they break if page structure changes even slightly

SDET Interview Q&A

Q1. What is the difference between findElement() and findElements()?
A: findElement() returns a single WebElement, throws NoSuchElementException if not found. findElements() returns a List<WebElement> and returns an empty list if nothing matches — no exception thrown.
Q2. Which locator strategy is best and why?
A: By.id() is fastest and most reliable per HTML spec (IDs must be unique). For complex scenarios, By.cssSelector() is preferred over XPath for speed and readability.
Q3. When would you use JavascriptExecutor?
A: When standard WebDriver interactions fail — e.g., clicking a hidden element, scrolling to an element, setting a readonly input value, or reading custom DOM properties.
Q4. What is the Actions class used for?
A: Complex user gestures: hover (mouse-over), right-click, double-click, drag-and-drop, keyboard combos. Regular .click() can't handle these.
Q5. Name all 4 Selenium components.
A: (1) Selenium WebDriver — core browser automation; (2) Selenium IDE — record-and-playback extension; (3) Selenium Grid — distributed parallel execution; (4) Selenium RC — legacy, fully deprecated.

Overview — v3 vs v4

Selenium 4 is not just a version bump — it is a complete architectural and API overhaul. Understanding what changed, what was removed, and what was added is essential for modern SDET roles. The shift from JSON Wire Protocol to W3C is the biggest fundamental change.

Interview one-liners

Selenium 4 fully adopted the W3C WebDriver standard, making it browser-vendor agnostic.
Selenium 4 introduced native CDP (Chrome DevTools Protocol) support for network interception.
Relative Locators (near, above, below, toLeftOf, toRightOf) are brand new in Selenium 4.
Selenium 4 deprecated DesiredCapabilities — replaced by browser-specific Options classes.
Selenium Grid 4 was completely rewritten with Docker support and a built-in observability dashboard.
Selenium 4 eliminated JSON Wire Protocol translation, reducing flakiness in test runs.

SDET Interview Q&A

Q1. What is the biggest change in Selenium 4?
A: Native W3C WebDriver protocol. Selenium 3 used its own JSON Wire Protocol requiring translation. Selenium 4 speaks W3C directly, eliminating compatibility issues between browser and driver versions.

A: New in Selenium 4 — find elements based on visual position relative to another element. Methods: near(), above(), below(), toLeftOf(), toRightOf(). Uses JavaScript getBoundingClientRect() internally.

A: Chrome DevTools Protocol integration — lets Selenium 4 intercept network requests, mock geolocation, capture performance metrics, and block URLs. Not available in Selenium 3.

Overview — Deprecated in Selenium 4

When upgrading from Selenium 3 to 4, certain APIs were deprecated or removed. Using deprecated APIs causes compiler warnings and may break in future Selenium releases. Knowing these ensures you write future-proof automation code.

Interview one-liners

DesiredCapabilities is deprecated in Selenium 4 — use browser-specific Options classes instead.
FindsByXPath and similar FindsBy interfaces were removed entirely from Selenium 4.
TimeUnit-based timeout methods are deprecated — use java.time.Duration objects instead.
System.setProperty for driver paths is deprecated — Selenium Manager handles this automatically.
setScriptTimeout() was renamed to scriptTimeout() in Selenium 4 — old name is deprecated.
The withTimeout(long, TimeUnit) overload in FluentWait is deprecated — use Duration.

BiDiDemo.java

1// ── WebDriver BiDi — Real-time Event Listening ───────────────────
2// BiDi = Bidirectional WebDriver Protocol (W3C draft standard)
3// PUSH events from browser to Java (vs old PULL/polling model)
4// BiDi works across Chrome AND Firefox — unlike CDP (Chrome-only)
5import org.openqa.selenium.WebDriver;
6import org.openqa.selenium.chrome.ChromeDriver;
7import org.openqa.selenium.chrome.ChromeOptions;
8import org.openqa.selenium.bidi.module.LogInspector;
9
10public class BiDiDemo {
11  public static void main(String[] args) {
12
13    // Enable BiDi by setting "webSocketUrl" capability to true
14    ChromeOptions options = new ChromeOptions();
15    options.setCapability("webSocketUrl", true); // activates BiDi mode
16
17    WebDriver driver = new ChromeDriver(options);
18
19    // LogInspector → BiDi module that listens for browser log events
20    // PUSH model — events fire as they happen, no polling needed
21    try (LogInspector inspector = new LogInspector(driver)) {
22
23      // onConsoleEntry() → fires on EVERY console.log/error/warn call
24      inspector.onConsoleEntry(entry ->
25        System.out.println("BiDi Console: " + entry.getText())
26      );
27
28      driver.get("https://www.example.com");
29      System.out.println("BiDi event listener active!");
30    }
31    driver.quit();
32  }
33}

SDET Interview Q&A

Q1. What new features does Selenium 4 offer for network testing?
A: Via CDP: intercept and block network requests, mock responses, monitor all outgoing HTTP calls, and simulate offline mode — from Java code without proxy tools like BrowserMob.
Q2. What is WebDriver BiDi and how is it different from CDP?
A: BiDi is a W3C draft standard for real-time browser-to-driver communication. CDP is Chrome-proprietary. BiDi works across Chrome, Firefox, and Edge. It enables push-based event listening rather than polling.
Q3. Can you take a full-page screenshot in Selenium 4?
A: Yes, but only in Firefox using HasFullPageScreenshot interface and getFullPageScreenshotAs() method. Chrome's standard getScreenshotAs() captures only the visible viewport.
Q4. How is Selenium Grid 4 different from Grid 3?
A: Grid 4 is completely rewritten. New components: Router, Distributor, SessionMap, Node. New features: native Docker/Kubernetes support, built-in GraphQL monitoring API, UI dashboard at port 4444, and Standalone mode.
Q5. How do you mock geolocation in Selenium 4?
A: Use CDP Emulation domain — devTools.send(Emulation.setGeolocationOverride(lat, lng, accuracy)). The browser reports those coordinates to any website requesting GPS location via the Geolocation API.
