Overview — Alerts, Dialogs & Popups

A dialog or alert is a browser-level popup that interrupts user interaction. Selenium handles three types: Alert (simple message), Confirm (OK/Cancel), and Prompt (text input). Web-level dialogs (custom HTML modals) use standard WebElement interactions. Failing to handle alerts before performing other actions throws UnhandledAlertException.

Interview one-liners

driver.switchTo().alert() switches Selenium focus to the currently active browser alert popup.
Alert.accept() clicks OK; Alert.dismiss() clicks Cancel — never swap them on a confirm dialog.
Alert.sendKeys() types text into a JavaScript prompt() dialog input field.
Alert.getText() retrieves the message text displayed inside the alert popup.
UnhandledAlertException is thrown if you interact with the page while an alert is still open.
Web-level dialogs (HTML modals) are NOT browser alerts — handle them using standard findElement() methods.

Best Practices

  • ✓Always use WebDriverWait + alertIsPresent() instead of directly calling switchTo().alert() — prevents NoAlertPresentException on slow pages
  • ✓Capture alert.getText() BEFORE calling accept() or dismiss() — once closed, the alert text is gone forever
  • ✓Never use switchTo().alert() for HTML modal dialogs — they are DOM elements, handled by findElement()
  • ✓For confirm() popups, use accept() for OK and dismiss() for Cancel — never swap them
  • ✓For prompt(), call sendKeys() BEFORE accept() — typing after accepting has no effect
  • ✓In TestNG @AfterMethod, add a try-catch to close any open alerts before the next test starts

SDET Interview Q&A

Q1. What is the difference between alert(), confirm(), and prompt() in Selenium?
A: alert() has only OK (use accept()). confirm() has OK and Cancel (accept() for OK, dismiss() for Cancel). prompt() has a text field + OK/Cancel — use sendKeys() before accept().
Q2. What exception is thrown when an alert is present and you try to interact with the page?
A: UnhandledAlertException — thrown when Selenium tries to perform any action on the main page while a JavaScript alert is still open. Always handle the alert first.
Q3. What exception is thrown when there is no alert but you call switchTo().alert()?
A: NoAlertPresentException — thrown when switchTo().alert() is called but no browser alert is currently open. Use WebDriverWait + alertIsPresent() to handle this safely.
Q4. How do you handle an HTML modal dialog differently from a JavaScript alert?
A: HTML modals are part of the page DOM — find and interact with them using findElement(By.id()), getText(), and click(). Never use switchTo().alert() for HTML modals.
Q5. Can you type into a browser alert using sendKeys() on a WebElement?
A: No. For a JavaScript prompt() dialog, you type via alert.sendKeys("text") on the Alert object itself — not via a WebElement, because the input inside an alert is not part of the DOM.

Overview — iFrame Handling

An iFrame (inline frame) is an HTML element that embeds another separate HTML document inside the current page. Selenium cannot directly access elements inside an iFrame without first switching context into it. After working inside the iFrame, you must switch back to the main document to access elements outside it.

Interview one-liners

switchTo().frame() switches Selenium's context INTO an iFrame so you can find elements inside it.
switchTo().defaultContent() returns Selenium's focus to the TOP-LEVEL main page document.
switchTo().parentFrame() moves to the IMMEDIATE parent frame (one level up, not all the way to top).
You can switch into an iFrame by index, name/id string, or WebElement reference — 3 ways.
Nested iFrames require multiple switchTo().frame() calls — one per level to go deeper.
NoSuchFrameException is thrown when the iFrame you try to switch into does not exist.

iFrame context switching flow

Selenium context levels
Main Page
defaultContent()
→ frame()
iFrame Level 1
switchTo().frame()
→ frame()
iFrame Level 2
nested iFrame
← parentFrame()
Back to Level 1
one level up
3 ways to switch into an iFrame
By Indexdriver.switchTo().frame(0) — switches to the first iFrame (0-based index)
By Name/IDdriver.switchTo().frame("myFrame") — uses the name or id HTML attribute
By Elementdriver.switchTo().frame(webElement) — most reliable, uses located WebElement

Best Practices

  • ✓Always call switchTo().defaultContent() after finishing work inside an iFrame — failure to do so causes NoSuchElementException when accessing the main page
  • ✓Prefer frameToBeAvailableAndSwitchToIt(By) over direct switchTo().frame() — it waits for the iFrame to load before switching
  • ✓Use WebElement-based switch (switchTo().frame(webElement)) instead of index — most stable and least affected by page changes
  • ✓For nested iFrames, always enter from the outermost level inward — you cannot jump directly to a deeply nested iFrame
  • ✓Use parentFrame() when you need to work between nested iFrame levels — use defaultContent() only when done with all iFrame work

SDET Interview Q&A

Q1. What is the difference between defaultContent() and parentFrame()?
A: defaultContent() jumps ALL the way back to the top-level main page from any nesting depth. parentFrame() goes exactly one level up — from child iFrame to its immediate parent iFrame.
Q2. What exception is thrown when you try to find an element inside an iFrame without switching context first?
A: NoSuchElementException — the element exists in the DOM but is inside an iFrame's separate document context. Selenium searches only the current active context (main page or current iFrame).
Q3. What are the 3 ways to switch into an iFrame?
A: (1) By index: frame(0); (2) By name or id: frame("myFrame"); (3) By WebElement: frame(webElement). WebElement is most reliable, index is least reliable.
Q4. How do you handle an iFrame that loads dynamically after page load?
A: Use WebDriverWait + ExpectedConditions.frameToBeAvailableAndSwitchToIt(By.id("iframeId")) — it waits for the iFrame to appear in the DOM then switches into it automatically.
Q5. What exception is thrown when the iFrame you try to switch into doesn't exist?
A: NoSuchFrameException — thrown when switchTo().frame() cannot find an iFrame matching the given index, name, id, or WebElement reference.

Overview — Windows & Tabs Handling

When a test causes a new browser window or tab to open (e.g., clicking a link with target="_blank"), Selenium stays focused on the original window. You must explicitly switch to the new window using its window handle — a unique string ID assigned to each open window/tab. All operations apply to whichever window Selenium is currently focused on.

Interview one-liners

getWindowHandle() returns the unique string ID of the currently focused browser window or tab.
getWindowHandles() returns a Set of all open window/tab handle IDs in the current session.
switchTo().window(handle) switches Selenium's focus to the window/tab with that handle ID.
driver.close() closes the CURRENTLY FOCUSED window/tab without ending the session.
After closing a child window, you MUST switch back to the parent window handle — driver is still open.
WindowType.TAB and WindowType.WINDOW in Selenium 4 open new tabs/windows natively without JavaScript.

→

then switch back

  • ✓Always save driver.getWindowHandle() as parentHandle BEFORE any action that opens a new window
  • ✓After calling driver.close() on a child window, ALWAYS call driver.switchTo().window(parentHandle) immediately
  • ✓Use WindowType.TAB and WindowType.WINDOW (Selenium 4) instead of JavaScript window.open() hacks

SDET Interview Q&A

Q1. What is a window handle in Selenium?
A: A unique opaque String ID assigned by the browser to each open window or tab. Selenium uses it to identify and switch focus between multiple open browser windows/tabs in the same WebDriver session.
Q2. What is the difference between getWindowHandle() and getWindowHandles()?
A: getWindowHandle() returns a single String — the ID of the CURRENTLY focused window. getWindowHandles() returns a Set<String> of ALL open window/tab IDs in the session.
Q3. What happens if you call driver.close() and then try to use driver without switching to another window?
A: NoSuchWindowException — after close(), the driver focus points to a closed window. You must call switchTo().window(someValidHandle) before doing anything else.
Q4. How do you switch to the newest opened window without knowing its handle?
A: Convert getWindowHandles() to an ArrayList and take the last element — new ArrayList<>(driver.getWindowHandles()).get(size-1) — as the most recently opened window is typically added last.
Q5. What is the difference between driver.close() and driver.quit() in multi-window scenarios?
A: close() closes only the currently focused window/tab; other windows remain open and the session stays active. quit() closes ALL windows, tabs, and terminates the entire WebDriver session.

Overview — All Navigation Handling

Navigation in Selenium covers all methods for controlling where the browser goes and how it gets there. This includes opening URLs, using browser history (back/forward/refresh), handling page load timeouts, and advanced URL navigation patterns used in real test frameworks.

Interview one-liners

driver.get(url) opens a URL and WAITS for the page to fully load before returning control.
driver.navigate().to(url) is functionally identical to get() but part of the navigate() object.
navigate().back() simulates clicking the browser Back button (uses browser history).
navigate().forward() simulates clicking the browser Forward button (uses browser history).
navigate().refresh() reloads the current page — equivalent to pressing F5 in the browser.
pageLoadTimeout sets the maximum time Selenium waits for a page to load before throwing TimeoutException.

Navigation methods architecture

All navigation entry points
driver.get(url)Direct URL open — waits for full page load (readyState = complete)
navigate().to(url)Same as get() but via navigate() API — prefer for navigate chaining
navigate().back()Browser history — one step back (URL must have been visited)
navigate().forward()Browser history — one step forward (must have gone back first)
navigate().refresh()Hard reload of current page — re-sends HTTP request, clears JS state
pageLoadTimeoutMax time to wait for page load — throws TimeoutException if exceeded

Best Practices

  • ✓Always set pageLoadTimeout to prevent tests from hanging indefinitely on slow pages
  • ✓After any navigation, use ExpectedConditions.urlContains() to verify the correct page loaded
  • ✓Prefer driver.get() over navigate().to() for simplicity — they behave identically for String URLs
  • ✓Use navigate().back() for testing browser history flows — do NOT call get() on the previous URL manually
  • ✓For SPA (Single Page Applications), use ExpectedConditions.urlToBe() or urlContains() instead of relying on page load events

SDET Interview Q&A

Q1. What is the difference between driver.get() and driver.navigate().to()?
A: Both load a URL and wait for the page to fully load. The key difference: navigate().to() also accepts a java.net.URL object in addition to String. get() accepts only String. Both are functionally equivalent for String URLs.
Q2. What exception is thrown when page load exceeds the pageLoadTimeout?
A: TimeoutException — thrown when the page does not reach document.readyState == "complete" within the configured timeout duration. The page may be partially usable even after this exception.
Q3. What happens if you call navigate().back() on the first page in history?
A: Nothing visible happens — the browser stays on the current page. There is no exception thrown. The command silently fails if there is no page to go back to in the session history.
Q4. How do you wait for a URL to change after a form submission?
A: Use WebDriverWait + ExpectedConditions.urlContains("expectedPath") or urlToBe("fullExpectedUrl"). This is far more reliable than using Thread.sleep() after form submit.
Q5. Does navigate().refresh() clear cookies or local storage?
A: No — refresh() only reloads the HTML/CSS/JS resources. Cookies, localStorage, and sessionStorage are preserved across a refresh. To clear those, use driver.manage().deleteAllCookies() or JavaScript localStorage.clear().

26
27    // METHOD 3: selectByIndex() → selects by position (0-based)
28    dropdown.selectByIndex(1); // selects the 2nd option (index starts at 0)
29
30    // getFirstSelectedOption() → returns the currently selected WebElement
31    System.out.println("Selected: " + dropdown.getFirstSelectedOption().getText());

34    List<WebElement> opts = dropdown.getOptions();

ExplicitWaitDemo.java
 1// ── Explicit Wait — WebDriverWait + ExpectedConditions ────────────
 2// Explicit Wait = targeted wait for a specific condition to become true
 3// Much better than Thread.sleep() — only waits as long as needed
 4import org.openqa.selenium.By;
 5import org.openqa.selenium.WebDriver;
 6import org.openqa.selenium.WebElement;
 7import org.openqa.selenium.chrome.ChromeDriver;
 8import org.openqa.selenium.support.ui.WebDriverWait;
 9import org.openqa.selenium.support.ui.ExpectedConditions;
10import java.time.Duration;
11
12public class ExplicitWaitDemo {
13  public static void main(String[] args) {
14    WebDriver driver = new ChromeDriver();
15    driver.get("https://the-internet.herokuapp.com/dynamic_loading/1");
16
17    // WebDriverWait → explicit wait with max 15 second timeout
18    WebDriverWait wait = new WebDriverWait(driver, Duration.ofSeconds(15));
19
20    // Click the Start button to trigger dynamic loading
21    driver.findElement(By.cssSelector("#start button")).click();
22
23    // visibilityOfElementLocated → waits until element EXISTS + is VISIBLE
24    WebElement finish = wait.until(
25      ExpectedConditions.visibilityOfElementLocated(By.id("finish")));
26    System.out.println("Loaded text: " + finish.getText());
27
28    // OTHER KEY ExpectedConditions:
29    // presenceOfElementLocated → element in DOM (not necessarily visible)
30    // elementToBeClickable → visible AND enabled (safe to click)
31    // textToBePresentInElement → waits for specific text to appear
32    // invisibilityOfElement → waits for element to DISAPPEAR (e.g. loader)
33    // titleContains("text") → waits for page title to contain a string
34
35    driver.quit();
36  }
37}

Best Practices

  • ✓Always call clear() before sendKeys() on input fields that might have pre-existing values
  • ✓Use getAttribute("value") to read input field contents — getText() returns empty for input elements
  • ✓Check isSelected() before clicking checkboxes/radios — prevents accidental unchecking
  • ✓Use Select class only for native HTML <select> elements — custom dropdowns require click() on list items
  • ✓Prefer ExpectedConditions.elementToBeClickable() over visibilityOfElementLocated() before calling click() — it checks both visibility AND enabled state
  • ✓Never mix implicit waits and explicit waits in the same test — they interact unpredictably and cause flaky timeouts

SDET Interview Q&A

Q1. What is the difference between getText() and getAttribute("value")?
A: getText() returns the visible rendered text between HTML tags (works on p, div, h1, span). getAttribute("value") reads the value attribute — used to get text typed into input fields, since getText() returns empty string for inputs.
Q2. What is the difference between isDisplayed(), isEnabled(), and isSelected()?
A: isDisplayed() — element is visible (not hidden by CSS). isEnabled() — element is not disabled (can be clicked/typed into). isSelected() — checkbox/radio/option is currently checked/selected. All return boolean.
Q3. How do you select an option in a dropdown if it doesn't have a native select element?
A: For custom CSS/JS dropdowns: (1) click() the dropdown trigger to open it; (2) use findElement() to locate the specific option in the list; (3) click() the option. The Select class only works with native HTML <select> elements.
Q4. What is the difference between implicit wait and explicit wait?
A: Implicit wait applies GLOBALLY to every findElement() call for the entire session. Explicit wait (WebDriverWait) applies to ONE specific condition at one specific point. Never mix them — they interact badly and can cause unpredictable wait times.
Q5. How do you scroll to an element that is not in the visible viewport?
A: Use JavaScript: ((JavascriptExecutor) driver).executeScript("arguments[0].scrollIntoView(true);", element). This scrolls the page so the element becomes visible. In Selenium 4, Actions.scrollToElement(element) is the native alternative.
