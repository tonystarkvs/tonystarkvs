Overview — Handling popups: browser level

A browser-level popup is a dialog generated natively by JavaScript using alert(), confirm(), or prompt(). These are not part of the HTML DOM — they are rendered by the browser engine and require Selenium's Alert interface, not regular element selectors. Attempting to interact with the page while one is open throws UnhandledAlertException.

What / Why / When / Where / How

WhatNative browser dialogs spawned by JS: alert (OK only), confirm (OK/Cancel), prompt (text input)
WhyFreeze all page interactions until dismissed — must be handled before any subsequent action
WhenOn form submit confirmations, session expiry warnings, delete confirmations, cookie consent banners
WhereIn test cleanup (@AfterMethod), after triggering UI actions that cause alerts
Howdriver.switchTo().alert() → read text / send keys / accept / dismiss

Keyword Glossary

Alert
Selenium interface — represents any open browser-native JS dialog (all 3 types)
switchTo().alert()
Moves Selenium focus to the currently open browser dialog
accept()
Clicks OK button on any alert/confirm/prompt — submits and closes it
dismiss()
Clicks Cancel button — only meaningful on confirm() and prompt()
getText()
Returns the message string shown inside the open alert dialog
sendKeys(text)
Types text into a prompt() input field — must be called before accept()
alertIsPresent()
ExpectedCondition — waits until an alert is present before switching
NoAlertPresentException
Thrown when switchTo().alert() is called but no alert is open
UnhandledAlertException
Thrown when a page action is attempted while an alert is still open
WebDriverWait
Explicit wait that polls until a condition is met — used to wait for alert

Interview one-liners

driver.switchTo().alert() moves Selenium focus to the currently open browser-native JS dialog.
alert() has only OK; confirm() has OK and Cancel; prompt() has a text input plus OK and Cancel.
accept() clicks OK on any alert type; dismiss() clicks Cancel — only valid on confirm() and prompt().
UnhandledAlertException is thrown when you interact with the page while an alert is still open.
Always use WebDriverWait with alertIsPresent() before switchTo().alert() to avoid NoAlertPresentException on slow pages.

Best Practices

  • ✓Always use WebDriverWait + alertIsPresent() instead of calling switchTo().alert() directly — prevents NoAlertPresentException on slow-loading dialogs
  • ✓Capture alert.getText() BEFORE calling accept() or dismiss() — the text is inaccessible after the dialog closes
  • ✓In @AfterMethod, wrap driver.switchTo().alert().dismiss() in a try-catch — silently swallows NoAlertPresentException when no alert is open
  • ✓For prompt(), call sendKeys() before accept() — this is the only way to type into a browser-native prompt (it is not a DOM element)
  • ✓Never call dismiss() on a simple alert() — it has no Cancel button and behavior varies across browsers

Interview Prep — 5 SDET Q&A

Q1. What is the difference between alert(), confirm(), and prompt() in Selenium handling?
A: All three use switchTo().alert(). alert() is info-only — use accept(). confirm() has OK/Cancel — use accept() or dismiss(). prompt() has a text input — call sendKeys(text) then accept() to submit, or dismiss() to cancel.
Q2. What exception is thrown when you interact with the page while an alert is still open?
A: UnhandledAlertException — Selenium throws this when you try to find elements, click, or navigate while a JS alert dialog is still blocking the page. Always handle the alert before continuing test execution.
Q3. What exception is thrown when you call switchTo().alert() but no alert is open?
A: NoAlertPresentException. Use WebDriverWait.until(ExpectedConditions.alertIsPresent()) to safely wait for the alert before switching, or wrap in try-catch to check for unexpected alerts.
Q4. Why must getText() be called before accept() or dismiss()?
A: Once accept() or dismiss() is called, the alert dialog is closed and the Alert object becomes stale. Any call to getText() after closing throws a NoAlertPresentException or returns null depending on the driver version.
Q5. How do you assert that a specific message appears in an alert?
A: Trigger the action that opens the alert, switch to it with switchTo().alert(), capture alert.getText() into a String, then use Assert.assertEquals(alertText, "expected message") before calling accept().

Overview — Web-level and OS-level popups

Web-level popups are custom HTML modal dialogs built with CSS/JS — they ARE part of the DOM and use standard findElement(). OS-level popups (Windows Authentication, native file pickers, print dialogs) live outside the browser and require the Java Robot class or URL-embedded credentials. Understanding which level a popup is at determines the correct Selenium strategy.

What / Why / When / Where / How

Web modalHTML <div> overlay with CSS — is a DOM element → use findElement(), click(), getText()
Auth popupWindows/Basic HTTP Auth dialog rendered by OS — Robot + clipboard or URL credentials
Print dialogOS print dialog — Robot sends keyboard shortcuts (Escape to close)
New windowPop-up window opened by target="_blank" or window.open() — handle with window handles
NotificationBrowser permission popup — disable via ChromeOptions args before session starts

Keyword Glossary

HTML modal
A <div> element styled as a dialog overlay — lives in the DOM, handled by standard WebElement methods
Windows Auth
HTTP Basic/NTLM authentication dialog rendered by the OS — Selenium cannot reach it directly
Robot class
java.awt.Robot — generates OS-level keyboard and mouse events to interact with native dialogs
StringSelection
java.awt.datatransfer.StringSelection — places a String into the OS clipboard
Toolkit.getSystemClipboard()
Returns the system clipboard object — used to set clipboard content for Robot paste
ChromeOptions arg
--disable-notifications — disables browser permission popups before the session opens
visibilityOfElementLocated
ExpectedCondition — waits until a HTML modal element is visible before interacting
URL credentials
http://user:pass@domain.com — embeds credentials in URL to bypass Basic Auth dialog
getWindowHandles()
Returns Set of all open browser window/tab IDs — used to switch to popup windows
KeyEvent.VK_ESCAPE
OS-level Escape keycode used with Robot to dismiss native dialogs (print, permission)

Interview one-liners

HTML modals are DOM elements — never use switchTo().alert() for them; use findElement() and click().
Disable browser notification popups before tests by passing --disable-notifications to ChromeOptions.
Windows Authentication dialogs are OS-level — use URL-embedded credentials or Robot class to handle them.
A popup window opened by target="_blank" needs getWindowHandles() and switchTo().window(handle) to access.
Robot class works at OS level — it cannot interact with DOM elements and does not work in headless mode.

Popup level decision diagram

Decision: which tool to use?
Can DevTools inspect it?YES → it is a DOM element → use findElement() and WebElement methods
Is it a JS alert/confirm/prompt?YES → use switchTo().alert() and the Alert interface
Is it a new browser window/tab?YES → use getWindowHandles() + switchTo().window()
Is it a Basic Auth dialog?YES → use URL credentials OR Robot class + StringSelection clipboard
Is it a browser notification ask?YES → add --disable-notifications to ChromeOptions before driver creation

Best Practices

  • ✓Determine popup level first: open DevTools — if you can inspect it, it is a DOM element → use findElement(); if not, it is browser or OS level
  • ✓Disable notification/permission popups globally in @BeforeSuite via ChromeOptions.addArguments("--disable-notifications")
  • ✓Prefer URL-embedded credentials over Robot for Basic Auth — simpler, no display requirement, works in headless mode
  • ✓Always save the parent window handle before clicking any link that might open a popup window
  • ✓After driver.close() on a popup window, always switchTo().window(parentHandle) immediately — failure causes NoSuchWindowException

Interview Prep — 5 SDET Q&A

Q1. How do you differentiate between a JS alert and an HTML modal?
A: Open browser DevTools — if you can see and inspect the element in the Elements panel, it is an HTML modal (DOM element → use findElement()). If DevTools cannot inspect it and it freezes the browser, it is a JS alert → use switchTo().alert().
Q2. Why can't Selenium handle Windows Authentication dialogs directly?
A: Windows Authentication dialogs are rendered by the OS, not the browser. Selenium's WebDriver protocol only communicates with the browser's rendering engine. OS-level windows are outside its scope. Solutions: URL-embedded credentials, Robot class, or CDP network header injection.
Q3. How do you suppress Chrome's "Allow notifications" popup in Selenium tests?
A: Two ways: (1) options.addArguments("--disable-notifications") disables entirely; (2) Set Chrome prefs via options.setExperimentalOption("prefs", map) where profile.default_content_setting_values.notifications = 2 (block). Both must be applied before new ChromeDriver(options).
Q4. What is the difference between driver.close() and driver.quit() in multi-window scenarios?
A: close() closes only the currently focused window — session stays alive and other windows remain open. quit() closes ALL windows and terminates the entire WebDriver session. After close() on a popup, always switch back to the parent window handle.
Q5. How does Robot class differ from Selenium's Actions class for keyboard input?
A: Actions class works at the browser/DOM level — it sends events to web elements. Robot works at the OS level — it generates actual hardware-level keyboard/mouse events that work on any application window including native OS dialogs. Robot does not work in headless mode (no display).

Overview — File upload — all ways

File upload testing is one of the most common real-world Selenium challenges. There are five distinct approaches: (1) sendKeys() on the file input directly, (2) Robot class for OS dialog, (3) JavaScript DOM manipulation, (4) AutoIt on Windows, and (5) Sikuli for image-based recognition. Each method targets a different layer of the OS/browser stack. Knowing all five and when to apply them is essential for SDET interviews.

What / Why / When / Where / How

WhatInteracting with <input type="file"> element or OS file picker to choose a file for upload

Interview one-liners

The simplest and most reliable way to upload a file in Selenium is sendKeys() with the absolute file path on the file input element.
sendKeys() on input[type="file"] bypasses the OS file dialog entirely — Selenium sets the file path directly in the DOM.
If the file input is hidden, use JavascriptExecutor to remove the display:none or visibility:hidden attribute before calling sendKeys().
Robot class handles OS-level file pickers by pasting the file path into the OS dialog via clipboard and pressing Enter.
AutoIt is Windows-only — it executes an AutoIt script that handles the native Windows file picker by window title.

Five file upload methods — decision map

Method 1: sendKeys() Recommended
Works on visible HTML input[type=file]. Simplest, fastest, no external tools. Cross-platform.
Method 2: JS Executor Hidden inputs
Unhides file input via JS, then sendKeys(). Use when input has display:none or visibility:hidden.
Method 3: Robot class OS dialog
For custom upload buttons that open OS dialog. Clipboard paste. Requires display (no headless).
Method 4: AutoIt Windows only
External Windows tool. Runs compiled .exe script. Handles Windows-native file picker by window title.
Method 5: Sikuli Image recognition
Uses screen screenshots to find and interact with OS elements visually. Platform-independent but fragile to UI changes.

HiddenInputJSUploadDemo.java
 1// ── METHOD 2: JavascriptExecutor — Handle hidden file inputs ──────
 2// Some sites hide the file input with CSS (display:none / visibility:hidden)
 3// and show a custom styled button instead. sendKeys() fails on hidden elements.
 4// Solution: use JS to make the input visible/enabled, then sendKeys()
 5import org.openqa.selenium.By;
 6import org.openqa.selenium.JavascriptExecutor;
 7import org.openqa.selenium.WebDriver;
 8import org.openqa.selenium.WebElement;
 9import org.openqa.selenium.chrome.ChromeDriver;
10
11public class HiddenInputJSUploadDemo {
12  public static void main(String[] args) {
13    WebDriver driver = new ChromeDriver();
14    driver.get("https://the-internet.herokuapp.com/upload");
15    JavascriptExecutor js = (JavascriptExecutor) driver;
16
17    // Locate the hidden file input element
18    WebElement fileInput = driver.findElement(By.id("file-upload"));
19
20    // TECHNIQUE 1: Remove display:none — makes element visible to sendKeys()
21    js.executeScript("arguments[0].style.display = 'block';", fileInput);
22
23    // TECHNIQUE 2: Remove visibility:hidden
24    js.executeScript("arguments[0].style.visibility = 'visible';", fileInput);
25
26    // TECHNIQUE 3: Remove opacity:0 (common in drag-and-drop upload zones)
27    js.executeScript("arguments[0].style.opacity = '1';", fileInput);
28
29    // TECHNIQUE 4: Remove the 'disabled' attribute if present
30    js.executeScript("arguments[0].removeAttribute('disabled');", fileInput);
31
32    // Now sendKeys() will work — element is visible to Selenium
33    fileInput.sendKeys("C:\\TestFiles\\sample.pdf");
34    System.out.println("Hidden input handled — file set via JS unhide");
35
36    // Alternative: set value directly without sendKeys (skips validation events)
37    // js.executeScript("arguments[0].value = 'filename'", fileInput); ← limited use
38
39    driver.quit();
40  }
41}
RobotFileUploadDemo.java
 1// ── METHOD 3: Robot class — Handle OS-level file picker dialog ────
 2// When a custom button triggers the native OS file picker dialog
 3// Selenium cannot interact with OS dialogs — use Robot + clipboard paste
 4// Does NOT work in headless mode (requires a physical or virtual display)
 5import java.awt.Robot;
 6import java.awt.Toolkit;
 7import java.awt.datatransfer.StringSelection;
 8import java.awt.event.KeyEvent;
 9import org.openqa.selenium.By;
10import org.openqa.selenium.WebDriver;
11import org.openqa.selenium.chrome.ChromeDriver;
12
13public class RobotFileUploadDemo {
14  public static void main(String[] args) throws Exception {
15    WebDriver driver = new ChromeDriver();
16    driver.get("https://the-internet.herokuapp.com/upload");
17
18    // STEP 1: Click the upload button — this opens the OS file picker dialog
19    driver.findElement(By.id("file-upload")).click();
20    Thread.sleep(2000); // wait for the OS dialog to appear on screen
21
22    Robot robot = new Robot();
23
24    // STEP 2: Put the file path string into the system clipboard
25    // StringSelection → wrapper that puts a String onto the OS clipboard
26    StringSelection sel = new StringSelection("C:\\TestFiles\\sample.pdf");
27    Toolkit.getDefaultToolkit().getSystemClipboard().setContents(sel, null);
28
29    // STEP 3: Ctrl+V → paste the path into the OS dialog's file name field
30    robot.keyPress(KeyEvent.VK_CONTROL);
31    robot.keyPress(KeyEvent.VK_V);
32    robot.keyRelease(KeyEvent.VK_V);
33    robot.keyRelease(KeyEvent.VK_CONTROL);
34    robot.delay(500); // brief pause for paste to register in OS dialog
35
36    // STEP 4: Press ENTER → clicks Open / OK button in the OS file dialog
37    robot.keyPress(KeyEvent.VK_ENTER);
38    robot.keyRelease(KeyEvent.VK_ENTER);
39    Thread.sleep(1000); // wait for dialog to close and file to be selected
40
41    // Now submit the form with the selected file
42    driver.findElement(By.id("file-submit")).click();
43    System.out.println("File uploaded via Robot OS dialog handling!");
44    driver.quit();
45  }
46}

Best Practices

  • ✓Always use sendKeys(absoluteFilePath) as the primary method — it is the simplest, most reliable, cross-platform approach and avoids OS dialog entirely
  • ✓Use absolute paths — relative paths like "./files/test.pdf" do not work with Selenium's sendKeys() on file inputs
  • ✓If the file input is hidden (display:none), use JavascriptExecutor to unhide it before calling sendKeys() — do not click the custom button
  • ✓Robot class is a fallback for OS dialogs but will not work in headless mode — use it only in non-headless CI environments with a virtual display (Xvfb on Linux)
  • ✓After upload, always wait for the success confirmation element before asserting — file uploads take network time and assertions may run before the response arrives
  • ✓For test data files, store them in src/test/resources/ and get the path via System.getProperty("user.dir") + "/src/test/resources/sample.pdf" — avoids hardcoded drive letters

Interview Prep — 5 SDET Q&A

Q1. What is the simplest way to upload a file in Selenium?
A: Call driver.findElement(By.id("file-input")).sendKeys("/absolute/path/to/file.pdf"). This sets the file path directly on the <input type="file"> element without opening the OS file picker dialog. Only works with the native HTML file input — not custom JavaScript-driven upload zones.
Q2. Why does sendKeys() fail on a hidden file input and how do you fix it?
A: Selenium's sendKeys() throws ElementNotInteractableException on hidden elements. Fix: use JavascriptExecutor to remove the display:none or visibility:hidden CSS — js.executeScript("arguments[0].style.display='block';", fileInput) — then call sendKeys() on the now-visible element.
Q3. How many ways are there to upload a file in Selenium and what are they?
A: Five ways: (1) sendKeys(path) on visible file input — simplest and recommended; (2) JavascriptExecutor to unhide input then sendKeys; (3) Robot class with clipboard paste for OS dialogs; (4) AutoIt script execution (Windows only); (5) Sikuli image-based recognition (platform-independent but fragile).
Q4. What is the limitation of using Robot class for file upload?
A: Robot does not work in headless mode — it requires an actual display device. In headless CI (e.g., GitHub Actions), use Xvfb (X Virtual Framebuffer) on Linux to create a virtual display for Robot. Also, Robot interacts with whichever OS dialog is currently focused — if another dialog appears, Robot may type into the wrong window.
Q5. How do you get the file path in a platform-independent way for test data files?
A: Use System.getProperty("user.dir") to get the project root, then append the resource path: System.getProperty("user.dir") + File.separator + "src" + File.separator + "test" + File.separator + "resources" + File.separator + "sample.pdf". This avoids hardcoded Windows drive letters and works on all OS.
