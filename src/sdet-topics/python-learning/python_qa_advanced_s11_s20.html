Example 4 — Full Page Object Model
# pages/login_page.py
from selenium.webdriver.common.by import By
from selenium.webdriver.support.ui import WebDriverWait
from selenium.webdriver.support import expected_conditions as EC

class LoginPage:
    USERNAME  = (By.ID, "username")
    PASSWORD  = (By.ID, "password")
    LOGIN_BTN = (By.CSS_SELECTOR, "button[type='submit']")
    ERROR_MSG = (By.CSS_SELECTOR, ".alert-error")

    def __init__(self, driver):
        self.driver = driver
        self.wait = WebDriverWait(driver, 10)

    def login(self, username, password):
        self.wait.until(EC.visibility_of_element_located(self.USERNAME))
        self.driver.find_element(*self.USERNAME).send_keys(username)
        self.driver.find_element(*self.PASSWORD).send_keys(password)
        self.driver.find_element(*self.LOGIN_BTN).click()

    def get_error(self):
        return self.driver.find_element(*self.ERROR_MSG).text

Interview one-liners
  • Q: Implicit vs explicit wait?Implicit sets a global default; explicit waits for a specific condition. Always prefer explicit — it's more reliable and readable.
  • Q: Why POM over direct locators in tests?Locator changes break 1 page file, not 50 test scripts. POM = maintainability at scale.
  • Q: find_element vs find_elements?Singular throws NoSuchElementException if missing; plural returns an empty list. Use plural when checking element count.
Common mistakes + fixes
❌ Wrongtime.sleep(3) # hard sleep — flaky!
✅ FixWebDriverWait(driver,10).until(EC.visibility_of_element_located(...))
❌ Wrongdriver.find_element(By.XPATH, "//div[1]/span[2]/input") # brittle
✅ Fixdriver.find_element(By.ID, "username") # stable ID locator

Example 5 — API helper class (reusable)
class APIClient:
    def __init__(self, base_url, token=None):
        self.session = requests.Session()
        self.session.headers["Content-Type"] = "application/json"
        if token:
            self.session.headers["Authorization"] = f"Bearer {token}"
        self.base = base_url

    def get(self, path, **kw):
        r = self.session.get(f"{self.base}{path}", **kw)
        r.raise_for_status()   # raises on 4xx/5xx
        return r.json()

    def post(self, path, payload):
        r = self.session.post(f"{self.base}{path}", json=payload)
        r.raise_for_status()
        return r.json()
Interview one-liners
  • Q: requests.get() vs requests.Session()?Session reuses TCP connections and persists cookies/headers. Use Session for multi-step API test flows.
  • Q: How to validate JSON schema?Use jsonschema library or manually assert required keys. Always validate both structure and values.
  • Q: What is raise_for_status()?Auto-raises HTTPError for 4xx/5xx responses — cleaner than if resp.status_code != 200 in every test.
Best practices ✓
  • Assert status code before reading body
  • Use Session for authenticated test suites
  • Store base_url in config, not hardcoded
  • Use resp.json() not json.loads(resp.text)
  • Test negative cases: 401, 403, 404, 400

Interview one-liners
  • Q: What is BDD?Behaviour-Driven Development — tests written in business language (Gherkin) so devs, QA, and business all share a common understanding.
  • Q: Given/When/Then?Given = precondition (setup), When = action (trigger), Then = expected outcome (assertion). And/But = continuation of previous step.
  • Q: Scenario vs Scenario Outline?Scenario is a single test case. Scenario Outline + Examples table = data-driven BDD — runs once per row.
Best practices ✓
  • Keep scenarios short: 3–5 steps max
  • Use Background for repeated Given steps
  • Write feature files before code (BDD intent)
  • Tag scenarios: @smoke @regression @slow
  • Step text must match exactly — no variation

Example 1 — Decorator basics
import functools

def log_test(func):
    """Decorator: log when each test starts/ends."""
    @functools.wraps(func)   # preserves function name
    def wrapper(*args, **kwargs):
        print(f"▶ Starting: {func.__name__}")
        result = func(*args, **kwargs)
        print(f"✅ Done: {func.__name__}")
        return result
    return wrapper

@log_test
def test_login():
    assert True

test_login()  # ▶ Starting: test_login  ✅ Done: test_login
Example 2 — Retry decorator (flaky test handler)
import time, functools

def retry(times=3, delay=1):
    """Retry decorator for flaky network tests."""
    def decorator(func):
        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            for attempt in range(1, times+1):
                try:
                    return func(*args, **kwargs)
                except Exception as e:
                    print(f"Attempt {attempt} failed: {e}")
                    if attempt < times: time.sleep(delay)
            raise Exception(f"Failed after {times} retries")
        return wrapper
    return decorator

@retry(times=3, delay=2)
def test_flaky_api(): pass
Example 3 — Timer decorator
import time, functools

def timer(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)
        end = time.perf_counter()
        duration = end - start
        print(f"[PERF] {func.__name__}: {duration:.3f}s")
        if duration > 2.0:
            print(f"⚠️  SLOW: exceeded 2s threshold")
        return result
    return wrapper

@timer
def test_page_load():
    time.sleep(1.5)   # simulate test
Example 4 — Context manager (with statement custom)
from contextlib import contextmanager

@contextmanager
def managed_driver(url):
    """Auto-quit driver even if test fails."""
    driver = None
    try:
        driver = webdriver.Chrome(...)
        driver.get(url)
        yield driver          # test code runs here
    finally:
        if driver:
            driver.quit()

# Usage — browser guaranteed to close:
with managed_driver("https://example.com") as d:
    assert "Example" in d.title
Example 5 — Generator for large test data
def generate_test_ids(prefix, count):
    """Memory-efficient test ID generator."""
    for i in range(1, count + 1):
        yield f"{prefix}_{i:04d}"   # TC_0001, TC_0002

# Use in test generation — doesn't load all in memory
for tc_id in generate_test_ids("TC", 1000):
    print(tc_id)   # TC_0001 ... TC_1000

# List version (loads all at once — avoid for large sets)
ids = list(generate_test_ids("SMOKE", 50))
Interview one-liners
  • Q: What does @functools.wraps do?Preserves the wrapped function's __name__ and __doc__. Without it, all decorated functions show as "wrapper" in logs.
  • Q: Generator vs list?Generators are lazy — yield one item at a time. Use for large test data sets to avoid loading everything into memory.
  • Q: What is a context manager?Ensures setup and teardown happen via with block. __enter__ sets up, __exit__ always cleans up — even on exception.
Best practices ✓
  • Always use @functools.wraps in decorators
  • Use @retry for network-dependent tests
  • Use @timer to track performance regression
  • Context managers for resource cleanup (DB, files)
  • Prefer generators over lists for >1000 items

results = {}

Example 3 — ThreadPoolExecutor (modern threading)
from concurrent.futures import ThreadPoolExecutor
import requests

urls = [f"https://jsonplaceholder.typicode.com/posts/{i}"
        for i in range(1, 11)]

def fetch(url):
    r = requests.get(url)
    return {"url": url, "status": r.status_code}

# Fetch 10 URLs in 4 parallel threads

Example 4 — async/await (modern Python)
import asyncio
import aiohttp   # pip install aiohttp

async def fetch_async(session, url):
    async with session.get(url) as resp:
        return {"status": resp.status, "url": url}

async def run_all(urls):
    async with aiohttp.ClientSession() as session:
        tasks = [fetch_async(session, u) for u in urls]
        return await asyncio.gather(*tasks)

urls = ["https://jsonplaceholder.typicode.com/posts/1"]
results = asyncio.run(run_all(urls))

Interview one-liners
  • Q: Thread vs process?Threads share memory (GIL-limited in Python for CPU work). Processes have separate memory — use multiprocessing for CPU-heavy tasks, threading for I/O.

  • Ensure tests are fully independent
  • Use unique test data per parallel worker

Interview one-liners
  • Q: Why test the DB layer separately?API tests validate the surface; DB tests verify data persisted correctly. UI can pass while DB has corrupt/missing data.

Best practices ✓

  • Test DB constraints: NOT NULL, UNIQUE, FK

# conftest.py — capture screenshot on every failure
import pytest, os

@pytest.hookimpl(tryfirst=True, hookwrapper=True)
def pytest_runtest_makereport(item, call):
    outcome = yield
    report  = outcome.get_result()

    if report.when == "call" and report.failed:
        driver = item.funcargs.get("driver")
        if driver:
            os.makedirs("screenshots", exist_ok=True)
            name = f"screenshots/{item.name}.png"
            driver.save_screenshot(name)
            print(f"Screenshot saved: {name}")

# conftest.py — print summary after test run
import pytest

results = {"passed": 0, "failed": 0, "skipped": 0}

@pytest.hookimpl(hookwrapper=True)
def pytest_runtest_makereport(item, call):
    outcome = yield
    rep = outcome.get_result()
    if rep.when == "call":
        results[rep.outcome] = results.get(rep.outcome, 0) + 1

def pytest_terminal_summary(terminalreporter):
    print(f"\n📊 SUMMARY: ✅ {results['passed']} | ❌ {results['failed']}")

SDET interview question bank
  • Q: Explain your test automation framework architecture."We use a Python + pytest framework with Page Object Model for UI, requests-based API client for backend, CSV/JSON data files for DDT, conftest fixtures for shared setup, and GitHub Actions for CI. Reports via Allure with screenshot on failure."
  • Q: What is the testing pyramid?Unit tests (many, fast, cheap) → Integration/API tests → UI/E2E tests (few, slow, expensive). Focus most automation at API layer — faster, more stable than UI.
  • Q: How do you handle flaky tests?Root-cause first: is it timing (add explicit waits), data (use unique test data), or environment? Add @retry decorator as last resort, not first fix.
  • Q: pytest fixture scopes?function (default, per test), class, module (per file), session (once per run). Use session for expensive setup (browser, DB connection).
  • Q: Difference between mock, stub, spy?Stub returns fake data. Mock verifies calls were made. Spy wraps real object to observe. Use unittest.mock or pytest-mock in Python.
  • Q: How to test an API that doesn't exist yet?Mock it with responses library or unittest.mock.patch. Or use WireMock/Mockoon as a mock server — test your client code independently.
