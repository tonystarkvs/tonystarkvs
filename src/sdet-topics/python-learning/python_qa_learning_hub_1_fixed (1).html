Session 1 — Python Basics & Setup
Variables · Print · Input · Comments · Data type overview · Python as interpreted language
Python is interpreted — no compile step needed. You run python file.py and the interpreter handles everything.
What is Python?
Python is a high-level, dynamically-typed, interpreted language. For QA/SDET roles, it's the #1 language for automation scripts, test frameworks (pytest), and API testing tools.
Why Python for QA?
Simple readable syntax → faster test writing. Huge ecosystem (pytest, selenium, requests). Cross-platform. Used in Selenium, Appium, Robot Framework, and CI/CD pipelines.

Session 6 — Object-Oriented Programming (OOP)
Class · Object · __init__ · self · Inheritance · Encapsulation · Polymorphism · Page Object Model
OOP is the foundation of the Page Object Model (POM) — the industry-standard design pattern for Selenium/Playwright test automation.
Hands-On Examples
Example 1 — Class & __init__ (basic)
class TestCase:
    """Represents a single test case."""
    
    def __init__(self, tc_id, title, priority="Medium"):
        self.tc_id    = tc_id
        self.title    = title
        self.priority = priority
        self.status   = "NOT RUN"   # default
    
    def run(self):
        self.status = "PASSED"
        print(f"[{self.tc_id}] {self.title} → {self.status}")

tc = TestCase("TC001", "Login with valid creds", "High")
tc.run()   # [TC001] Login with valid creds → PASSED
Example 2 — Inheritance (BasePage pattern)
class BasePage:
    """All page objects inherit from this."""
    def __init__(self, driver):
        self.driver = driver
    
    def get_title(self):
        return self.driver.title()    # shared method
    
    def click(self, locator):
        self.driver.find_element(locator).click()

class LoginPage(BasePage):          # inherits BasePage
    def login(self, user, pwd):
        self.click("#username")    # uses parent method
        self.click("#password")
Example 3 — Class method & static method
class TestReport:
    total = 0   # class variable (shared)
    
    def __init__(self, name):
        self.name = name
        TestReport.total += 1
    
    @classmethod
    def get_total(cls):
        return cls.total    # access class variable
    
    @staticmethod
    def validate_name(name):
        return bool(name) and name.startswith("TC")

print(TestReport.validate_name("TC001"))  # True
Example 4 — __str__ and __repr__
class APIResponse:
    def __init__(self, status, body):
        self.status = status
        self.body   = body
    
    def __str__(self):     # for print()
        return f"Response({self.status}): {self.body}"
    
    def __repr__(self):    # for debugging
        return f"APIResponse(status={self.status!r})"

resp = APIResponse(200, "{'user':'admin'}")
print(resp)   # Response(200): {'user':'admin'}
Example 5 — Real Page Object Model (POM)
# ─── pages/login_page.py ───────────────────
class LoginPage:
    # Locators — hidden from test scripts
    USERNAME = "#username"
    PASSWORD = "#password"
    SUBMIT   = "#login-btn"
    ERROR_MSG= ".error-text"
    
    def __init__(self, driver):
        self.driver = driver
    
    def login(self, username, password):
        """High-level action used in test scripts."""
        self.driver.find(self.USERNAME).send_keys(username)
        self.driver.find(self.PASSWORD).send_keys(password)
        self.driver.find(self.SUBMIT).click()
Interview One-Liners
  • Q: What is self in Python?self refers to the current instance. It's how methods access instance variables — Python passes it automatically.
  • Q: Why use OOP in test automation?POM pattern — encapsulates locators, reuses code via inheritance, makes tests readable and maintainable.
  • Q: @classmethod vs @staticmethod?@classmethod receives cls (class reference); @staticmethod is a plain utility function — no class/instance access.
Best Practices ✓
  • Always implement Page Object Model (POM)
  • Never hardcode locators in test scripts
  • Use BasePage for shared driver utilities
  • Add __str__ to make debug logs readable
  • Keep constructors simple — no logic in __init__

def parse_test_data(data, key):

    except ZeroDivisionError:

# Custom exception for test framework errors
class TestDataError(Exception):
    """Raised when test data file is invalid."""
    def __init__(self, filename, reason):
        self.filename = filename
        super().__init__(f"Invalid data in '{filename}': {reason}")

def load_data(filename):
    if not filename.endswith(".csv"):
        raise TestDataError(filename, "only .csv files are supported")

    load_data("users.xlsx")

Interview One-Liners
  • Q: When does "finally" NOT run?Only when os._exit() or a hard crash occurs. In all normal cases — even with exceptions — finally runs.
  • Q: except Exception vs bare except?Always use except Exception — bare except also catches SystemExit and KeyboardInterrupt, which is dangerous.
  • Q: What is assert used for in tests?To verify expected outcomes. In pytest, a failing assert raises AssertionError and marks the test as FAILED.
Best Practices ✓
  • Always catch specific exceptions first
  • Use finally for browser/DB cleanup
  • Never silence exceptions with empty except: pass
  • Add context to raise messages

Interview One-Liners
  • Q: import X vs from X import Y?import X loads the module; use as X.func(). from X import Y imports directly; use as Y().
  • Q: Why use logging over print()?Logging has levels (DEBUG/INFO/ERROR), can write to files, and is configurable — print can't be controlled in CI/CD pipelines.
  • Q: What is sys.exit(1) used for?Exits the script with code 1 — CI/CD systems (Jenkins, GitHub Actions) treat non-zero exit as a build failure.
Best Practices ✓
  • Replace print() with logging in production tests
  • Set logging.INFO for CI, DEBUG locally
  • Use sys.exit(1) to signal failure to CI/CD
  • Use random.choice() for browser cross-testing
  • Use re for dynamic error code extraction

Interview One-Liners
  • Q: What is a pytest fixture?A reusable setup function injected into tests by name. Replaces setUp/tearDown from unittest. Supports scope (function/class/module/session).
  • Q: What is conftest.py?A special pytest file for sharing fixtures across test files. No import needed — pytest auto-discovers it.
  • Q: What is @pytest.mark.parametrize?Runs a test function multiple times with different data sets. The foundation of Data-Driven Testing (DDT) in pytest.
Best Practices ✓
  • Prefix all test functions with test_
  • One assert concept per test function
  • Use fixtures for all setup/teardown
  • Use parametrize for data-driven tests
