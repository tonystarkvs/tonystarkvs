Overview — All Test Hooks in Playwright

Test hooks are lifecycle functions that run automatically at specific points in a test suite's execution — before/after each test or once for a group. Playwright provides beforeAll, afterAll, beforeEach, afterEach with full async/await support, access to fixtures, and guaranteed execution order. Each hook receives the same fixture system as tests — page, context, request, testInfo, and custom fixtures are all available.

What / Why / When / Where / How

WhatLifecycle callbacks that wrap test execution — beforeEach/afterEach run before/after every individual test, beforeAll/afterAll run once for the entire describe block. They share fixtures and can perform setup and teardown.
WhyEliminate duplication — instead of repeating page.goto('/login') in every test, put it in beforeEach. Share expensive setup like DB seeding in beforeAll. Guarantee cleanup even when tests fail.
WhenUse beforeEach for anything that must be fresh per test (navigation, login). Use beforeAll for expensive shared setup (DB seed, server start). Use afterEach for per-test cleanup. Use afterAll for shared teardown.
WhereInside test.describe() blocks or at the top of a spec file. They are scoped — a beforeEach inside a nested describe only runs for tests in that describe. Outer hooks run before inner hooks.
HowHooks receive the same fixtures as tests via destructuring. Execution order: outer beforeAll → outer beforeEach → inner beforeEach → TEST → inner afterEach → outer afterEach → inner afterAll → outer afterAll.

Keyword Glossary

test.beforeEach()
Runs before every individual test in the current scope. Receives all fixtures. Use for per-test navigation, login, data setup. Creates fresh state per test.
test.afterEach()
Runs after every individual test, even if the test failed. Receives fixtures AND testInfo.status. Use for per-test cleanup, conditional screenshots, logging.
test.beforeAll()
Runs ONCE before the first test in a describe block. Worker-scoped — each parallel worker runs its own beforeAll. Use for DB seeding, server start.
test.afterAll()
Runs ONCE after the last test in a describe block. Use for shared cleanup: delete seeded data, close DB connections. Worker-scoped.
testInfo
Second parameter in hooks. Contains: title, status, errors, retry, testId, outputDir, attach(), annotation. Useful in afterEach for conditional teardown.
testInfo.status
Actual result of the test: 'passed', 'failed', 'timedOut', 'skipped', 'interrupted'. Available in afterEach/afterAll for conditional behavior.
testInfo.expectedStatus
What the test was expected to return (usually 'passed'). Compare with testInfo.status: if status !== expectedStatus, the test failed unexpectedly.
Scope
Hooks apply to all tests in their enclosing describe block or file. Nested hooks only affect tests in their own describe. Outer hooks always run for inner tests too.
Hook fixtures
Hooks receive fixtures via destructuring: test.beforeEach(async ({ page, context, request }) => {...}). Same fixture system as tests.
Worker isolation
beforeAll/afterAll run once per worker, not once globally. If 4 workers run tests in parallel, beforeAll runs 4 times. For truly global once-only setup, use globalSetup.

Interview One-Liners

Use beforeEach when each test needs fresh isolated state — like navigating to a page. Use beforeAll when all tests share expensive setup like seeding a database, which only needs to happen once per worker.
afterEach is guaranteed to run even if the test fails — put conditional screenshot logic there: if testInfo.status !== testInfo.expectedStatus, capture a full-page screenshot.
A critical misunderstanding is thinking beforeAll is global — it runs once PER WORKER, not once globally. For truly once-only global setup, use globalSetup in playwright.config.ts.
Playwright hooks follow FIFO nesting: outer beforeEach runs before inner beforeEach, but outer afterEach runs AFTER inner afterEach. Setup goes outside-in, teardown goes inside-out.
testInfo.status in afterEach is powerful — use it to log test duration, conditionally attach traces, decide whether to clean up or preserve state for debugging.
Hooks have access to the same fixtures as tests — use the request fixture in afterEach to delete test data, and the page fixture in beforeEach to pre-navigate.

beforeAll vs globalSetup — When to Use Which
beforeAllRuns once per WORKER (not truly global). Has access to page/context/request fixtures. Scoped to test file or describe block.
globalSetupRuns ONCE globally before any worker starts. No fixtures — plain Node.js function. Use for: save storageState, DB migration, env setup.
Anti-patternUsing beforeAll for login (runs N times per worker). Correct: login in globalSetup, restore in test.use(storageState).
Hook failureIf beforeAll fails → ALL tests in block are skipped/failed. If beforeEach fails → that test fails, afterEach still runs.

all-hooks-basic.ts

import { test, expect } from '@playwright/test';

// ── FILE-LEVEL HOOKS — run for EVERY test in this file ────────────

// beforeAll: runs ONCE before the first test (per worker)
test.beforeAll(async ({ request }) => {
  console.log('beforeAll: seed reference data');
  await request.post('/api/categories', {
    data: { name: 'Electronics', slug: 'electronics' },
  });
});

// afterAll: runs ONCE after the last test (per worker)
test.afterAll(async ({ request }) => {
  console.log('afterAll: remove seeded data');
  await request.delete('/api/categories/electronics');
});

// beforeEach: runs before EVERY test — fresh state per test
test.beforeEach(async ({ page }) => {
  console.log('beforeEach: navigate to homepage');
  await page.goto('https://myapp.com');
  await expect(page).toHaveURL(/myapp\.com/);
});

// afterEach: runs after EVERY test (even on failure)
test.afterEach(async ({ page }, testInfo) => {
  console.log(`afterEach: test "${testInfo.title}" → ${testInfo.status}`);
  if (testInfo.status !== testInfo.expectedStatus) {
    const path = testInfo.outputPath('failure.png');
    await page.screenshot({ path, fullPage: true });
    await testInfo.attach('failure-screenshot', {
      path, contentType: 'image/png',
    });
  }
});

test('homepage title is correct', async ({ page }) => {
  await expect(page).toHaveTitle(/MyApp/);
});

test('electronics category exists', async ({ page }) => {
  await expect(page.getByRole('link', { name: 'Electronics' }))
    .toBeVisible();
});

aftereach-conditional.ts

import { test, expect } from '@playwright/test';
import fs from 'fs';

test.afterEach(async ({ page }, testInfo) => {
  const failed = testInfo.status !== testInfo.expectedStatus;

  if (failed) {
    // 1. Full-page screenshot → attached to HTML report
    const ssPath = testInfo.outputPath('failure-screenshot.png');
    await page.screenshot({ path: ssPath, fullPage: true });
    await testInfo.attach('failure-screenshot', {
      path: ssPath, contentType: 'image/png',
    });

    // 2. Log current URL for context
    await testInfo.attach('failing-url', {
      body: page.url(), contentType: 'text/plain',
    });

    // 3. Capture DOM as HTML for offline inspection
    const html = await page.content();
    const htmlPath = testInfo.outputPath('dom-snapshot.html');
    fs.writeFileSync(htmlPath, html);
    await testInfo.attach('dom-snapshot', {
      path: htmlPath, contentType: 'text/html',
    });

    console.error(`TEST FAILED: ${testInfo.title} on URL: ${page.url()}`);
  }
  console.log(`${testInfo.title}: ${testInfo.duration}ms [${testInfo.status}]`);
});

test('passing test — no artifacts', async ({ page }) => {
  await page.goto('https://myapp.com');
  await expect(page.locator('h1')).toBeVisible();
});

Best Practices

  • ✓Use beforeEach for per-test navigation and state reset — never share mutable state between tests through variables set in beforeEach
  • ✓Keep beforeAll for truly shared, read-only setup (seeding reference data). If tests mutate the shared state, they'll interfere with each other in parallel
  • ✓Always put teardown in afterEach/afterAll, never at the end of a test body — end-of-test cleanup is skipped on test failure, hook teardown is not
  • ✓Use testInfo.status in afterEach for conditional debugging: only capture screenshots/logs when the test failed — saves CI artifact space significantly
  • ✓Prefer fixtures over beforeEach for shared setup that needs to be reused across multiple spec files — a fixture is composable; beforeEach is file-scoped
  • ✓For login setup: never use beforeAll for browser login. Use globalSetup to login once and save storageState, then test.use({storageState:'auth.json'})
  • ✓When using beforeAll to create shared data, always have a corresponding afterAll to delete it — orphaned test data accumulates and causes interference
  • ✓Add a custom timeout to beforeAll when it does network operations: test.beforeAll(async () => {...}, {timeout: 60_000})

Interview Prep — 5 SDET Q&A

Q1. What is the execution order when you have nested describe blocks with multiple hooks?
A: Execution order follows an inside-out pattern for teardown and outside-in for setup. For a test inside an inner describe: (1) Outer beforeAll runs once. (2) Outer beforeEach runs. (3) Inner beforeEach runs. (4) TEST EXECUTES. (5) Inner afterEach runs. (6) Outer afterEach runs. (7) Inner afterAll runs once after last test. (8) Outer afterAll runs once after all tests. Setup goes outside-in (wrapping), teardown goes inside-out (unwrapping).
Q2. Why is using beforeAll for browser login an anti-pattern?
A: Two problems: (1) Worker isolation — beforeAll runs once per Playwright worker, not once globally. With 4 workers, beforeAll executes 4 separate login flows. (2) Shared page state — if you login in beforeAll and share the page, tests are no longer isolated. The correct approach: run login once in globalSetup, save cookies/storage via context.storageState({path:'auth.json'}), then restore per test with test.use({storageState:'auth.json'}).
Q3. How do you pass data from beforeAll to individual tests?
A: Declare a variable in the describe block scope: let createdUser: {id:number, email:string}. In beforeAll, assign to it via API call. Each test reads createdUser.id without re-creating it. Important: only safe if tests READ the shared variable without modifying it. If tests mutate shared state, run the describe block in serial mode: test.describe.configure({mode:'serial'}).
Q4. How do you capture a screenshot only when a test fails, using hooks?
A: In afterEach, check testInfo.status against testInfo.expectedStatus: if (testInfo.status !== testInfo.expectedStatus) { await page.screenshot({path: testInfo.outputPath('failure.png'), fullPage: true}); await testInfo.attach('failure-screenshot', {path, contentType:'image/png'}); }. testInfo.outputPath() creates a unique file path preventing parallel tests from overwriting each other's screenshots.
Q5. What happens if beforeEach throws an error?
A: The test it was preparing is marked as failed with the hook's error. The test body never executes. However, afterEach still runs — Playwright guarantees afterEach execution regardless. If beforeAll throws, ALL tests in that describe block are marked as failed and skipped — only afterAll still runs for cleanup.

Overview — test.describe() Configurations & All Keys

test.describe() groups related tests together and provides a scope for hooks, configurations, and modifiers. Every test in a describe block shares its context — and test.describe.configure() controls HOW those tests run: in parallel, serially, with retries, or with specific timeouts. Playwright also provides modifier methods like test.only(), test.skip(), test.fixme(), test.fail(), test.slow(), and test.step().

What / Why / When / Where / How

WhatA named block that groups related tests. Creates a scope for hooks, test.use() options, and modifiers. Appears in test reports as a parent heading. Can be nested to any depth.
WhyOrganization — group tests by feature, page, or user role. Scoping — hooks and test.use() inside a describe only affect tests in that block. Configuration — describe.configure() controls execution mode.
WhenWhen configuring test infrastructure, optimizing CI pipeline speed, debugging flaky tests, setting up cross-browser testing, and managing test parallelism.
WhereIn playwright.config.ts for global configuration, in globalSetup/globalTeardown files for one-time operations, and in test files for lifecycle hooks.
Howmode:'parallel' — all tests run simultaneously. mode:'serial' — tests run in order; if one fails, subsequent tests are skipped. Default is parallel across files, serial within a file.

Keyword Glossary

test.describe()
Groups related tests under a named block. Creates scope for hooks, test.use(), and modifiers. Renders as parent heading in HTML report.
test.describe.configure()
Configures behavior for all tests in the describe block. Options: mode ('parallel'|'serial'|'default'), retries (number), timeout (ms).
mode:'parallel'
All tests in the describe block run simultaneously in parallel workers. Each test gets its own browser context.
mode:'serial'
Tests run one at a time in registration order. If one test fails, ALL subsequent tests in the block are SKIPPED. Use for wizard flows.
test.only()
Marks one test to run exclusively. NEVER commit — CI checks for .only with forbidOnly:true.
test.skip()
Marks a test or block as skipped. Can be conditional: test.skip(condition, 'reason'). Skipped tests show in reports as skipped (not failed).
test.fixme()
Marks a known-broken test. Like skip but communicates "this is broken and needs fixing". Appears with a fixme icon in reports.
test.fail()
Marks a test that is EXPECTED to fail. The test passes if it fails (inverted assertion). Use to document known bugs.
test.slow()
Multiplies the test timeout by 3. Use for inherently slow tests (video processing, email delivery).
test.step()
Groups actions into a named logical step. Visible in trace viewer and HTML report. Can be nested. Returns a value.

Interview One-Liners

Use test.describe.configure({mode:'serial'}) for wizard-style checkout flows where step 2 depends on state created in step 1 — if add-to-cart fails, checkout should be skipped.
test.only() is a development tool, never a commit — configure CI with forbidOnly:true in playwright.config.ts so the pipeline fails if anyone commits a .only().
test.fail() marks a test that SHOULD fail — the test PASSES when it fails as expected. When the bug is fixed, the test turns red reminding you to remove the fail annotation.
test.slow() triples the timeout instead of hardcoding test.setTimeout(90000). It communicates intent and if the global timeout changes, slow tests automatically scale.
test.describe.configure() must be called at the TOP of the describe block — it's evaluated when the test plan is built, before any execution begins.
Nested test.describe() blocks create a hierarchy in reports — outer > inner > test name. This makes it easy to filter and understand coverage at a glance.

Best Practices

  • ✓Always set forbidOnly: !!process.env.CI in playwright.config.ts — prevents accidentally committed test.only() from silently running only one test in CI
  • ✓Use mode:'serial' sparingly — only for genuine wizard/multi-step flows where tests truly depend on each other's state. Default to parallel
  • ✓Prefer test.skip(condition, 'reason') over removing a test — the reason string appears in reports and communicates WHY the test is skipped
  • ✓Use test.fail() to track known bugs — creates a failing test record that PASSES until the bug is fixed, then turns red automatically
  • ✓Put test.describe.configure() as the FIRST line inside a describe block — it's declarative and must be evaluated during planning
  • ✓Use test.slow() instead of test.setTimeout(large_number) — it communicates intent, scales proportionally with config changes
  • ✓Structure describe blocks to mirror your app's information architecture: describe('Feature') → describe('Sub-feature') → test('specific behavior')
  • ✓Use test.describe.configure({retries: 2}) for groups of known-flaky tests instead of setting global retries — avoids wasting retry time on stable tests

Interview Prep — 5 SDET Q&A

Q1. What is the difference between test.describe.configure({mode:'serial'}) and test.describe.serial()?
A: They are identical in behavior — test.describe.serial() is a shorthand for test.describe.configure({mode:'serial'}). The configure() version is more powerful because you can pass multiple options: mode, retries, and timeout all at once. Similarly, test.describe.parallel() is shorthand for test.describe.configure({mode:'parallel'}).
Q2. When would you use test.describe.configure({mode:'serial'}) in a real project?
A: Real use cases: (1) E-commerce checkout wizard — add-to-cart → fill address → payment → confirmation. Step 4 cannot be tested if step 3 failed. (2) Onboarding flow — account creation → email verification → profile setup. (3) Data import workflow — upload → validate → process → verify. Important: set retries:0 for serial describe blocks — retrying a failed step causes state confusion.
Q3. How does test.fail() work and when should you use it?
A: test.fail() inverts the test's pass condition. The test PASSES when its assertions fail, and FAILS when its assertions unexpectedly pass. Use case: documenting confirmed bugs. When the developer fixes the bug, the test suddenly PASSES — which means test.fail() now fails the test, creating an automatic alert to remove the annotation.
Q4. What is the difference between test.use() and test.describe.configure()?
A: They configure completely different things. test.use({options}) sets fixture values — browser settings like viewport, storageState, isMobile, locale. It's about WHAT the test gets. test.describe.configure({mode, retries, timeout}) controls execution behavior — HOW tests run. They're complementary and can be used together in the same describe block.
Q5. How do you organize a large spec file with hundreds of tests using describe blocks?
A: Strategy: (1) Top-level by major feature area. (2) Second level by user role or scenario. (3) Third level by sub-feature. (4) Use test.use() at each level for role-specific auth and viewport. (5) Use test.describe.configure({mode:'parallel'}) at top level. (6) Use mode:'serial' only for inherently sequential groups. The resulting structure mirrors the product specification, making it easy for non-engineers to understand coverage.
