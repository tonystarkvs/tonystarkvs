Session 8 · Assertions

Playwright's expect() function provides auto-retrying assertions that poll the condition repeatedly (every ~100ms) until it passes or the timeout expires. This makes tests resilient to async DOM updates, loading states, and animations without explicit waits.

Two Categories

Locator assertions: expect(locator).toBeVisible() — async, auto-retrying, for DOM checks. Value assertions: expect(value).toBe() — sync, one-shot snapshot. Always prefer locator assertions for DOM state.

Soft Assertions

expect.soft() records a failure but continues the test. All soft failures are reported at the end. Perfect for checking multiple fields at once without stopping at the first failure.

Custom Messages

Pass a string to expect(locator, 'message'). The message appears in test reports, making CI failures self-documenting without reading test code.

Negation

Any assertion can be negated with .not: expect(el).not.toBeVisible(). The .not form also auto-retries — waits until the element disappears.

Glossary
toBeVisible()Element is rendered and visible — attached, not hidden by CSS. Auto-retries. Most common assertion.
toBeHidden()Element is CSS-hidden or detached from DOM. Opposite of toBeVisible. Retries until hidden or timeout.
toHaveText()Full text match (trimmed). Accepts string or regex. Strict — entire visible text must match.
toContainText()Substring match. Passes if element text includes the expected string. Accepts array for multiple elements.
toHaveValue()Asserts .value of input/select/textarea. For checkboxes use toBeChecked() instead.
toBeChecked()Checkbox/radio is checked. Accepts {checked:false} for inverse. More semantic than toHaveValue.
toBeEnabled()Element does not have the disabled attribute. Inverse: toBeDisabled().
toHaveCount()Locator matches exactly N elements. Auto-retries until count matches. Never use await locator.count() === N without expect.
toHaveURL()Page URL matches string glob, regex, or exact URL. Called on page object: expect(page).toHaveURL().
toHaveTitle()document.title matches string or regex. Called on page: expect(page).toHaveTitle().
toHaveAttribute()HTML attribute equals expected value. e.g. expect(link).toHaveAttribute('href', '/home'). Accepts regex for value.
toHaveClass()Element has the specified CSS class. Accepts string or regex. Useful for state changes via class names.
toHaveCSS()Computed CSS property equals expected value. e.g. toHaveCSS('color','rgb(255,0,0)'). Always use computed values.
expect.soft()Non-stopping assertion. Records failure, continues test. All soft failures surface at test end. Perfect for batch field checks.
toBeFocused()Element is the current document.activeElement. Use after Tab navigation or .focus() calls.
toHaveScreenshot()Visual regression assertion. Compares against a stored baseline PNG. Fails if pixel diff exceeds threshold.

Core Concepts

Locator assertions vs value assertions

✅ Locator (auto-retrying)
await expect(page.locator('.status')).toHaveText('Saved')
→retries ~100ms until text='Saved' or 30s timeout
❌ Value snapshot (no retry)
expect(await locator.textContent()).toBe('Saved')
→reads once — race condition on async updates

Assertion priority order

1sttoBeVisible / toBeHidden→element presence
2ndtoHaveText / toContainText→content
3rdtoHaveURL / toHaveTitle→navigation
4thtoHaveValue / toBeChecked→form state
5thtoHaveAttribute / toHaveClass→DOM properties
lasttoHaveScreenshot→visual regression
Best Practices
  • ✓
    Always use locator assertions (expect(locator).toXxx()) for DOM checks — never snapshot values with await locator.textContent() then compare.
  • ✓
    Use expect.soft() for form validation, profile field checks, and any multi-field verification — collect all failures in one run.
  • ✓
    Always add custom failure messages to business-critical assertions: expect(el, 'Price must update after coupon').toHaveText('₹450').
  • ✓
    Prefer toContainText() over toHaveText() when text includes dynamic values (user names, timestamps, counts).
  • ✓
    Use expect(page).toHaveURL() after navigation — it auto-retries through redirect chains. Never use expect(page.url()).toBe().
  • ✓
    Update visual regression baselines intentionally: run npx playwright test --update-snapshots after deliberate UI changes, not on every test run.
  • ✓
    For list order assertions, use expect(locator).toHaveText(['a','b','c']) — one call checks count AND content AND order simultaneously.
Interview Prep — 5 SDET Q&A
Q1: Why should you use expect(locator).toHaveText() instead of expect(await locator.textContent()).toBe()? ▶
The locator assertion form auto-retries every ~100ms until the text matches or the timeout expires — it handles all async DOM updates automatically. The value snapshot form reads the text once at that exact moment; if React hasn't re-rendered or the API hasn't responded yet, it reads the old value and fails. This is a race condition baked into the test. The rule: any value that might change asynchronously must be asserted via a locator assertion, never via an awaited value compared with expect().toBe().
Q2: When do you choose toHaveText() vs toContainText()? ▶
toHaveText() does a full text match after trimming whitespace — the element's entire text content must equal the expected value exactly. If the element says "3 items in cart" and you assert toHaveText('items in cart'), it fails. toContainText() checks if the text includes the expected as a substring — toContainText('items in cart') passes. I use toHaveText() when I know the complete static text (e.g. heading text, button labels). I use toContainText() when the text includes dynamic parts like user names, numbers, timestamps, or when I'm checking error messages that may have prefixes.
Q3: How do soft assertions help in SDET interviews and real projects? ▶
In real projects: form validation tests benefit most — if a registration form has 8 required fields and all fail validation, a hard expect stops at field 1. With expect.soft() all 8 failures are collected and reported in one test run, making the bug report complete. In interviews: SDETs who know expect.soft() demonstrate awareness of test efficiency and report quality. The pattern — run all soft assertions, then optionally add a hard assertion at the end that checks test.info().errors.length === 0 — shows mature test design thinking.
Q4: How does visual regression testing work with toHaveScreenshot()? ▶
On the first run, toHaveScreenshot() captures a PNG and saves it as the baseline in a __screenshots__ folder. On subsequent runs, it captures a new screenshot and diffs it pixel-by-pixel against the baseline. If the pixel difference percentage exceeds the threshold, the test fails and saves a diff image. Configuration: you can set maxDiffPixels (absolute count) or maxDiffPixelRatio (percentage). To update baselines after intentional UI changes: npx playwright test --update-snapshots. Best practice: only run visual tests on specific components, not full pages — full-page screenshots are fragile due to font rendering differences across OSes.
Q5: How do you assert multiple items in a list are in correct order? ▶
Single call: await expect(page.locator('ul li')).toHaveText(['First','Second','Third']) — asserts count, content, and order simultaneously. For partial text per item: .toContainText(['item1', 'item2', 'item3']). For order verification when items are sorted dynamically: const texts = await page.locator('li').allTextContents(); const sorted = [...texts].sort(); expect(texts).toEqual(sorted) — compares the actual array to its sorted copy. For count only: expect(locator).toHaveCount(N). The key principle: always use the locator assertion form (which retries) rather than collecting values into a variable and comparing statically.
visibility.ts
// visibility.ts — toBeVisible, toBeHidden, toBeAttached
import { test, expect } from '@playwright/test';

test('visibility assertions', async ({ page }) => {
  await page.goto('https://myapp.com/dashboard');

  // toBeVisible — element must be visible in the viewport (not hidden via CSS)
  await expect(page.locator('.welcome-banner')).toBeVisible();

  // toBeHidden — element exists in DOM but is CSS-hidden
  await expect(page.locator('.loading-overlay')).toBeHidden();

  // Visible after an async action (auto-retry handles the wait)
  await page.click('#open-modal-btn');
  await expect(page.getByRole('dialog')).toBeVisible();

  // Hidden after dismiss
  await page.keyboard.press('Escape');
  await expect(page.getByRole('dialog')).not.toBeVisible();

  // toBeAttached — element is in DOM (even if visibility:hidden)
  await expect(page.locator('#hidden-form')).toBeAttached();

  // not.toBeAttached — element removed from DOM entirely
  await page.click('#delete-widget');
  await expect(page.locator('.widget-card')).not.toBeAttached();

  // toBeInViewport — element is inside the browser viewport
  await expect(page.locator('h1')).toBeInViewport();
});
text-assert.ts
// text-assert.ts — toHaveText, toContainText, regex, arrays
import { test, expect } from '@playwright/test';

test('text content assertions', async ({ page }) => {
  await page.goto('https://myapp.com/products');

  // Exact text match (trimmed)
  await expect(page.locator('h1')).toHaveText('All Products');

  // Regex match — dynamic content
  await expect(page.locator('.result-count')).toHaveText(/\d+ products found/);

  // Substring match — ignore surrounding text
  await expect(page.locator('.promo-banner')).toContainText('Free delivery');

  // Array form — checks all list items in ORDER
  await expect(page.locator('.breadcrumb span'))
    .toHaveText(['Home', 'Products', 'Laptops']);

  // toContainText array — each item must contain its substring
  await expect(page.locator('.price-tag'))
    .toContainText(['₹', '₹', '₹']);

  // After dynamic update — cart counter
  await page.click('.add-to-cart');
  await expect(page.locator('.cart-count')).toHaveText('1');

  // Ignore whitespace differences — normalize:true
  await expect(page.locator('.desc'))
    .toHaveText('A   spaced   text', { normalizeWhitespace: true });

  // Inner text only (ignores hidden child text)
  await expect(page.locator('.label')).toHaveText('Sale', { useInnerText: true });
});

test('page-level assertions', async ({ page }) => {
  await page.goto('https://myapp.com/login');

  // Title — string and regex
  await expect(page).toHaveTitle('Login | MyApp');

  await page.fill('#email', 'test@example.com');
  await page.fill('#password', 'Pass123!');
  await page.click('button[type="submit"]');

  // Waits for redirect to complete — auto-retrying
  await expect(page).toHaveURL(/dashboard/);
  await expect(page).toHaveTitle(/Dashboard/);

  // Not on login page
  await expect(page).not.toHaveURL(/login/);

  // Page-level screenshot — full viewport capture
  await expect(page).toHaveScreenshot('dashboard-full.png');

  // With threshold for minor rendering differences
  await expect(page).toHaveScreenshot('dashboard-threshold.png', {
    maxDiffPixelRatio: 0.02,  // 2% pixel diff tolerance
  });
});

  await expect.soft(
    page.getByRole('checkbox', { name: 'Newsletter' }),
    'Newsletter must be unchecked by default'
  ).not.toBeChecked();

  await expect.soft(page.getByLabel('Country'), 'Default country is India')
    .toHaveValue('IN');

  // Hard assertion at the end — stops if ANY soft failed
  await expect(page.getByRole('button', { name: 'Save Changes' }))
    .toBeVisible(); // critical — must always be present

  // Log total soft failures (zero = all passed)
  const softFails = test.info().errors.length;
  console.log(`Soft failures: ${softFails}`);
  expect(softFails, 'All profile fields must be correct').toBe(0);
});

custom-expect.ts
// custom-expect.ts — extending expect with custom matchers
import { test, expect, Locator } from '@playwright/test';

// Extend expect with a custom matcher
expect.extend({
  async toBeValidEmail(locator: Locator) {
    const value = await locator.inputValue();
    const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
    const pass = emailRegex.test(value);
    return {
      message: () => `Expected "${value}" to be a valid email address`,
      pass,
    };
  },
  async toBePriceFormat(locator: Locator, currency: string = '₹') {
    const text = await locator.textContent() ?? '';
    const pass = new RegExp(`^\\${currency}\\d+(\\.\\d{2})?$`).test(text.trim());
    return {
      message: () => `Expected "${text}" to be a price in format ${currency}XXX.XX`,
      pass,
    };
  },
});

// Extend type declarations
declare module '@playwright/test' {
  interface Matchers<R, T> {
    toBeValidEmail(): Promise<R>;
    toBePriceFormat(currency?: string): Promise<R>;
  }
}

test('use custom matchers', async ({ page }) => {
  await page.goto('https://myapp.com/checkout');

  await expect(page.getByLabel('Email')).toBeValidEmail();
  await expect(page.locator('.total-price')).toBePriceFormat('₹');
});
Session 9 · Waiting Strategies

Playwright auto-waits for actionability on every action — but sometimes you need explicit waits for specific conditions: network responses, URL changes, element states, JS conditions, or custom events. Knowing when and how to wait is what separates flaky tests from reliable ones.

Auto-waiting

Every action (click, fill, hover) auto-waits for: attached, visible, stable, enabled, receives-events. No explicit wait needed before most interactions. Powered by the locator retry mechanism.

waitForResponse

page.waitForResponse(urlPattern) waits for a specific network response. Use after form submits or search actions to wait for the API call to complete before asserting on results.

waitForFunction

page.waitForFunction(fn) polls a JavaScript expression inside the browser until it returns truthy. Use for complex JS state conditions that no DOM-based locator can check.

Never use timeout

page.waitForTimeout(2000) is an anti-pattern — it always waits the full duration even if the condition is met sooner. It makes tests slow and still flaky. Always use condition-based waits.

Glossary
Auto-waitingBuilt-in mechanism on every action. Retries until element is actionable (attached, visible, stable, enabled, receives events) or timeout expires. No code needed.
waitForSelector()Waits for element matching selector to be present and match state option: 'attached','detached','visible','hidden'. Returns ElementHandle. Legacy — prefer expect(locator).toBeVisible().
waitForURL()Waits until page.url() matches string glob, regex, or predicate. After clicks that trigger redirects. Faster than waitForNavigation.
waitForLoadState()Waits for 'load', 'domcontentloaded', or 'networkidle'. Use after navigation actions when you need explicit load confirmation.
waitForResponse()Waits for a network response matching URL string/regex/predicate. Returns the Response object. Use to wait for API calls to complete before asserting.
waitForRequest()Waits for a network request to be sent matching URL. Use to verify the correct API was called with the right parameters.
waitForFunction()Polls a JS function in the browser context until it returns truthy. Use for complex JS state: animation complete, window property set, third-party SDK initialized.
waitForEvent()Waits for a page/context event: 'popup', 'dialog', 'download', 'filechooser', 'console', 'pageerror'. Returns a Promise resolving with the event data.
waitForTimeout()Hard sleep in ms. Anti-pattern — always waits the full duration. Only acceptable for debugging or very rare third-party timing issues with no observable state change.
page.route()Intercepts network requests matching a URL pattern. Can fulfill, abort, or continue them. Use to mock API responses or block tracking scripts.
expect(locator).toBeVisible()The correct replacement for most explicit waits. Auto-retries until visible or timeout. Combines waiting AND assertion in one call.
networkidleLoad state where no network connections have fired for ≥500ms. Best for SPAs. Caution: long-polling apps never reach networkidle.

Core Concepts

Waiting strategy decision tree

Need to wait for...
DOM element visible/hidden→expect(locator).toBeVisible/Hidden()
URL to change→page.waitForURL() or expect(page).toHaveURL()
API response→page.waitForResponse(urlPattern)
Page to load→page.waitForLoadState('networkidle')
JS condition→page.waitForFunction(() => window.flag === true)
Popup/dialog→page.waitForEvent('popup'/'dialog')
Hard sleep→NEVER use waitForTimeout in production

Promise.all — race-condition-safe waiting

❌ Wrong — action before listener
await page.click('#open') ← popup fires here
await page.waitForEvent('popup') ← too late!
✅ Correct — both start simultaneously
const [popup] = await Promise.all([
page.waitForEvent('popup'),
page.click('#open'),
]);
waitForLoadState — Understanding the 3 load states
Timeline — When each state fires relative to page load
① FASTEST domcontentloaded

Fires when the HTML is fully parsed and the DOM tree is built. Does not wait for images, stylesheets, fonts, or API calls to finish. 

await page.waitForLoadState('domcontentloaded');
✅ Use when you only need DOM elements — inputs, buttons, text Example: Login form — fill username/password right after DOM loads
② MEDIUM load (default)

Fires when all page resources are loaded — HTML, CSS, images, scripts. Does not wait for XHR/fetch calls made after page load (API-driven content). 

await page.waitForLoadState('load');
// Same as calling with no argument:
await page.waitForLoadState(); // 'load' is the default
✅ Use when you need CSS, images or scripts to be present Example: Product page — assert product image is visible
③ SLOWEST networkidle

Fires when there are fewer than 2 network requests for at least 500ms. Waits for all lazy loads, API calls, and dynamic content to fully settle. 

await page.waitForLoadState('networkidle');
✅ Use when page is a SPA or loads data via API after mount ⚠️ Avoid with WebSockets / long-polling — page never becomes idle
Speed & Use Case — Side by Side
State Speed What it waits for Best for
domcontentloaded ⚡ Fastest HTML parsed · DOM tree ready Login forms, form fills, simple interactions

Page Load Flow — What Fires & When
START Browser receives the URL — begins downloading and parsing the HTML document byte by byte.
① DOM
domcontentloaded fires — HTML is fully parsed. DOM nodes exist. Buttons and inputs are reachable. Images and CSS may still be downloading.
② LOAD
load fires — All CSS, JS, images and stylesheets are downloaded. Page looks visually complete. API calls triggered by JS may still be running.
③ IDLE
networkidle fires — No more than 1 active network request for 500ms. All API data rendered. SPA fully mounted. Lazy-loaded content finished.
💡 Rule: Always use the fastest state that satisfies your test. Faster state = less wait time = less flakiness = faster CI.
Best Practices
  • ✓
    Never use waitForTimeout() in production tests. Replace with condition-based waits — expect(locator).toBeVisible() for DOM, waitForResponse() for network, waitForFunction() for JS state.
  • ✓
    Use Promise.all() pattern for event-based waits (popup, dialog, download) — start the listener and trigger action simultaneously to prevent race conditions.
  • ✓
    Use waitForResponse() with a status check: resp.status() === 200 — waiting for ANY response to a URL catches error responses that still technically complete the wait.
  • ✓
    Avoid networkidle on apps with WebSockets, Server-Sent Events, or long-polling — the page never becomes idle. Use element-based assertions instead.
  • ✓
    Set specific timeouts on waits that can legitimately take longer: waitForResponse(url, { timeout: 60_000 }) — better than increasing the global timeout for everything.
  • ✓
    Use page.route() to mock slow API responses in tests that verify loading states — controls timing deterministically without real server delays.
Interview Prep
Q1: What is auto-waiting in Playwright and why does it reduce flakiness? ▶
Auto-waiting means every Playwright action (click, fill, hover, check) automatically retries until the target element meets 5 actionability conditions: attached to DOM, visible (not hidden by CSS), stable (not animating), receives pointer events (not covered by overlay), and enabled (not disabled). This eliminates the most common source of test flakiness — clicking elements that are loading, animating, or behind a spinner. Before auto-waiting, you'd write waitForSelector + click in Selenium. In Playwright, locator.click() handles both. The same applies to assertions — expect(locator).toBeVisible() retries until visible, so there's no need to wait then assert separately.
Q2: How do you wait for an API call to complete after a form submit? ▶
I use waitForResponse() — it resolves the moment a response matching the URL pattern is received. The critical pattern: set up the wait BEFORE the action, then trigger the action, then await the wait. Example: const responsePromise = page.waitForResponse(resp => resp.url().includes('/api/users') && resp.status() === 200); await page.click('#create-user-btn'); const response = await responsePromise; expect(response.ok()).toBeTruthy(). I also check response.status() === 200 in the predicate to avoid resolving on error responses. After the response, I assert on the DOM changes that the API call triggered.
Q3: When would you use waitForFunction()? ▶
waitForFunction() evaluates a JavaScript expression in the browser context and retries until it returns truthy. I use it for conditions that have no DOM representation: (1) Third-party SDK initialization — await page.waitForFunction(() => window.analytics !== undefined && window.analytics.ready === true). (2) Complex animation state — await page.waitForFunction(() => document.querySelector('.chart').getAttribute('data-rendered') === 'true'). (3) Local storage/session state — await page.waitForFunction(() => localStorage.getItem('auth_token') !== null). (4) Custom application events — await page.waitForFunction(() => window.appState.initialized === true). It's the escape hatch for anything that can't be expressed as a locator or URL condition.
Q4: Why is waitForTimeout() considered an anti-pattern? ▶
Two fundamental problems: (1) It always sleeps the full duration — if the condition is met at 50ms, waitForTimeout(3000) still waits the full 3 seconds. In a suite with 200 tests, if each has one sleep, that's 600 seconds of dead time. (2) It's still flaky on slow CI — if you pick 3000ms because the app usually takes 2500ms, it fails when CI is slower than usual. A properly written condition-based wait never has this problem. Every waitForTimeout can be replaced: waiting for a DOM change → expect(locator).toBeVisible(). Waiting for a network response → waitForResponse(). Waiting for JS state → waitForFunction(). The only acceptable use: a deliberate pause to observe a visual effect during local debugging, always removed before committing.
Q5: How do you handle tests that need to mock a slow API to test loading states? ▶
I use page.route() to intercept the API request and add a delay before responding. Example: await page.route('**/api/products', async route => { await new Promise(r => setTimeout(r, 2000)); // 2s delay await route.fulfill({ status: 200, contentType: 'application/json', body: JSON.stringify({ products: [] }) }); }). Then: await page.click('#load-products'). await expect(page.locator('.loading-spinner')).toBeVisible(); // assert spinner shows. await expect(page.locator('.loading-spinner')).toBeHidden(); // waits for spinner to disappear. await expect(page.locator('.products-grid')).toBeVisible(); // grid appears. This approach is deterministic — the delay is exact, not dependent on server speed.

// polling-custom.ts — custom polling + waitForCondition helper
import { test, expect, Page } from '@playwright/test';

// Custom helper: poll until condition or timeout
async function waitUntilTrue(
  page: Page,
  condition: () => Promise<boolean>,
  opts: { timeout?: number; interval?: number } = {}
): Promise<void> {
  const { timeout = 30_000, interval = 500 } = opts;
  const deadline = Date.now() + timeout;
  while (Date.now() < deadline) {
    if (await condition()) return;
    await page.waitForTimeout(interval);
  }
  throw new Error(`Condition not met within ${timeout}ms`);
}

test('custom polling for complex conditions', async ({ page }) => {
  await page.goto('https://myapp.com/batch-job');

  // Start a long-running job
  await page.click('#start-job-btn');

  // Poll status every 2s until job completes
  await waitUntilTrue(page, async () => {
    const status = await page.locator('.job-status').textContent();
    return status?..includes('Completed') ?? false;
  }, { timeout: 60_000, interval: 2000 });

  await expect(page.locator('.job-status')).toContainText('Completed');

  // waitForFunction with polling option (Playwright built-in)
  await page.waitForFunction(
    () => document.querySelector('.progress-bar')
      ?.getAttribute('aria-valuenow') === '100',
    { polling: 1000, timeout: 30_000 }
  );
});

Page Object Model

Interview Prep
Q1: How do you implement Page Object Model in Playwright TypeScript? ▶
I create a BasePage class that receives the page fixture in its constructor. Each page class extends BasePage and defines locators as getter properties (never regular class fields) so they're re-queried on every access: get submitButton() { return this.page.getByRole('button', {name:'Submit'}); }. Actions are async methods: async login(email, password) { await this.emailInput.fill(email); await this.passwordInput.fill(password); await this.submitButton.click(); }. Tests instantiate POMs and call methods: const loginPage = new LoginPage(page); await loginPage.login('user@test.com','Pass123'); await loginPage.expectDashboard(). The POM contains all locators and actions; the test contains only flow and assertions.
Q2: What are Playwright fixtures and how do they differ from beforeEach? ▶
Fixtures are a dependency injection system — each fixture is a function that sets up resources, yields them to the test, and tears them down after. Unlike beforeEach which runs for every test in a describe block, fixtures are composed: a test declares which fixtures it needs and Playwright injects them. Key advantages: (1) Composable — a loggedInPage fixture can internally use the page and context fixtures. (2) Auto-teardown — anything after yield in a fixture runs after the test, even if it throws. (3) Type-safe — custom fixtures are typed, giving IntelliSense. (4) Scoped — fixtures can have 'test', 'worker', or 'global' scope. beforeEach can't be shared across files without a helper function; fixtures are reusable across any test that imports them.
Q3: How do you run tests in parallel safely? ▶
Playwright runs each test in a separate browser context (process-isolated by default), so tests don't share DOM or cookies. The key to parallel safety: (1) Each test creates its own test data — never share a single user account across parallel tests because concurrent actions cause conflicts. (2) Use unique identifiers per test — append test ID to email/username: user_${test.info().testId}@test.com. (3) Avoid shared global state — no shared files, no shared database rows without transactions/locking. (4) For tests with genuine order dependency, use test.describe.configure({mode:'serial'}) on that describe block only — not globally. fullyParallel:true in the config means each test file also runs in parallel (default is parallel across files, serial within a file).
Q4: How do you set up Playwright in a CI/CD pipeline? ▶
GitHub Actions example: (1) actions/checkout + actions/setup-node. (2) npm ci to install dependencies. (3) npx playwright install --with-deps to get browser binaries + system dependencies. (4) npx playwright test --reporter=html,junit. Key config settings for CI: retries:2 (stability), workers: process.env.CI ? 4 : undefined (scale to CI runner cores), trace:'on-first-retry' (capture on failure), video:'on-first-retry'. (5) Upload artifacts: actions/upload-artifact for playwright-report/ and test-results/. (6) Publish JUnit XML to the CI test results dashboard for trend analysis. For Allure: install allure-playwright, run allure generate after tests, publish the allure-report as an artifact or deploy to GitHub Pages.
Q5: How do you combine API testing with UI testing in Playwright? ▶
Playwright's request fixture gives you an APIRequestContext to make HTTP requests without a browser. Pattern: (1) Setup — create test data via API (fast, reliable): const user = await request.post('/api/users', {data:{name:'Test'}}).then(r=>r.json()). (2) UI test — verify the created user appears in the UI: await page.goto('/users'); await expect(page.locator('.user-row').filter({hasText:user.name})).toBeVisible(). (3) Teardown — delete via API: await request.delete('/api/users/'+user.id). This hybrid approach is faster than creating data through UI flows, more reliable (UI bugs don't break setup), and gives true end-to-end coverage. The request context in globalSetup can also be used to seed databases with realistic test data before the suite starts.

  // Getters — re-queried on every access (no stale handles)
  get emailInput() { return this.page.getByLabel('Email'); }
  get passwordInput() { return this.page.getByLabel('Password'); }
  get submitButton() { return this.page.getByRole('button', { name: 'Sign In' }); }
  get errorMessage() { return this.page.locator('.auth-error'); }

  // Actions

    await this.page.goto(this.url);
  }

    await this.emailInput.fill(email);
    await this.passwordInput.fill(password);
    await this.submitButton.click();
  }

  async expectError(msg: string): Promise<void> {
    await expect(this.errorMessage).toContainText(msg);
  }
}

  get welcomeMsg() { return this.page.locator('.welcome-msg'); }
  get statsGrid() { return this.page.locator('.stats-grid'); }

    await expect(this.page).toHaveURL(/dashboard/);

// global-setup.ts — runs ONCE before all tests
import { chromium, FullConfig, request } from '@playwright/test';
import fs from 'fs';

export default async function globalSetup(config: FullConfig) {

  // ── 1. Browser Login — save storageState ─────────────────────────
  const browser = await chromium.launch();
  const context = await browser.newContext();
  const page = await context.newPage();

  await page.goto('https://myapp.com/login');
  await page.fill('#email',    process.env.TEST_EMAIL ?? 'admin@test.com');
  await page.fill('#password', process.env.TEST_PASS ?? 'Admin123!');
  await page.click('button[type="submit"]');
  await page.waitForURL('**/dashboard');

  // Save auth state to file
  fs.mkdirSync('playwright/.auth', { recursive: true });
  await context.storageState({ path: 'playwright/.auth/user.json' });
  await browser.close();

  // ── 2. API Setup — seed test data ────────────────────────────────
  const apiContext = await request.newContext({
    baseURL: 'https://myapp.com',
  });

  // Create test categories needed by tests
  await apiContext.post('/api/categories', {
    data: { name: 'Test Category', code: 'TEST_CAT' },
  });

  await apiContext.dispose();
  console.log('✅ Global setup complete — auth saved, test data seeded');
}
