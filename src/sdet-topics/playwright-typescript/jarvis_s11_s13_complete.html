Overview — Network Mocking & Interception

page.route() intercepts every outgoing network request matching a URL pattern. You can fulfill it with mock data, abort it to simulate failures, or continue it after modifying headers/body. This is the foundation of fast, deterministic, offline-capable Playwright tests — removing external API dependencies entirely.

What / Why / When / Where / How

Whatpage.route() registers a request interceptor. route.fulfill() returns mock data, route.abort() cancels requests, route.continue() modifies and forwards — all without touching the real server
WhyRemove external API dependencies, test edge cases (empty lists, 500 errors, slow responses), run tests offline, achieve 10–50× speed improvement by eliminating real network round-trips
WhenWhen testing UI components that depend on API responses, error handling flows, loading states, empty states, and when you need deterministic test data regardless of server state
WhereIn test setup before page.goto(), in fixtures for global mocks (analytics blocking), in individual tests for scenario-specific mocks
Howawait page.route('**/api/**', handler) — handler receives route and request objects. Call route.fulfill({json}), route.abort(), or route.continue({headers})

Keyword Glossary

page.route()
Registers a request interceptor. First arg: URL pattern (string glob, regex, predicate). Second arg: handler receiving (route, request)
route.fulfill()
Responds with mock data. Options: status, contentType, body, headers, json (auto-sets content-type). Does NOT hit real server
route.abort()
Cancels the request with a network error: 'failed', 'timedout', 'connectionrefused', 'blockedbyclient'
route.continue()
Forwards request to real server with optional overrides: url, method, headers, postData
route.fetch()
Fetches real response, lets you inspect/modify it, then fulfill with the modified version — real data with surgical modifications
page.unroute()
Removes a previously registered route handler — use to remove mocks after a specific test step
context.route()
Same as page.route() but applies to ALL pages in the BrowserContext — use for global interceptors
routeFromHAR()
Replays network responses from a saved HAR file — great for complex multi-request flows without individual mocking
page.on('request')
Passive event listener — observes requests without intercepting. Use for logging, verifying API calls were made
HAR file
HTTP Archive — a JSON recording of all network traffic. Playwright can record and replay HAR files for full offline playback

Interview One-Liners

I use page.route() to remove external API dependencies — every API call becomes deterministic, tests run offline, and I can test edge cases like 500 errors that are impossible to trigger reliably on a real server.
route.fulfill() gives the browser a fake successful response for injecting test data. route.abort() cancels the request at the network level for testing error handling like 'No internet connection' screens.
route.fetch() makes the real network call, gives me the response to inspect and modify, then I fulfill with the modified version — real data with surgical modifications without full mocking.
I use context.route() instead of page.route() for global interceptors — blocking analytics trackers, injecting auth headers, and mocking shared API endpoints across the entire test context.
For complex flows with dozens of API calls, I record a HAR file once from a real session using page.routeFromHAR({update:true}), then replay it in all subsequent test runs.
Always set up page.route() BEFORE page.goto() — routes registered after navigation won't intercept requests that already fired during page load.

Best Practices

  • ✓Always set up page.route() BEFORE page.goto() — routes registered after navigation won't intercept requests that already fired during page load
  • ✓Use route.fulfill({json: data}) shorthand — it automatically sets Content-Type: application/json and serializes the object
  • ✓Use context.route() for cross-page mocks (analytics blocking, global auth) and page.route() for test-specific mocks
  • ✓Use route.fetch() + route.fulfill() when you want real data but need to augment it rather than replacing it entirely
  • ✓Block third-party scripts (analytics, ads, chat widgets) globally with context.route() in your fixture — they add flakiness and slow down tests
  • ✓Use routeFromHAR() for integration-style tests with complex multi-request flows — record once, replay forever

Interview Prep — 5 SDET Q&A

Q1. What is the difference between page.route() and page.on('response')?
A: page.route() is an interceptor — it INTERCEPTS the request before it reaches the server. You can fulfill, abort, or continue it. page.on('response') is a passive observer — it listens to real responses AFTER they come back. Use route() for mocking/control, on('response') for spying.
Q2. How do you test an empty state (no data) UI component?
A: Intercept the API call and return an empty array: await page.route('**/api/products', route => route.fulfill({ json: { products: [], total: 0 } })). Then navigate and assert the empty state UI is visible and product cards have count 0.
Q3. How do you simulate a 500 server error?
A: await page.route('**/api/users', route => route.fulfill({ status: 500, json: { error: 'Internal Server Error' } })). Then assert the error banner is visible and retry button appears. Test multiple error codes: 401 (redirect to login), 403 (permission denied), 404 (not found).
Q4. How does route.fetch() differ from route.continue()?
A: route.continue() forwards the request and you never see the response. route.fetch() makes the actual call and returns a Response object — you can read, modify, then fulfill with the modified version. It's the most powerful pattern for real data with surgical modifications.
Q5. How do you block analytics and tracking scripts in tests?
A: Add a context-level route in a fixture: await context.route(url => trackers.some(d => url.includes(d)), route => route.abort()). Benefits: faster tests, no flakiness from external CDN, no third-party errors in console, clean test output.

Overview — Custom Fixtures & Dependency Injection

Playwright fixtures are a dependency injection system built into @playwright/test. They handle setup, yield resources to tests via the use callback, then auto-teardown — even if the test throws. Custom fixtures extend the base test object with reusable setup: page objects, authenticated users, seeded databases.

What / Why / When / Where / How

WhatA DI container built into @playwright/test. Fixtures are declared as async generator functions using test.extend(). Code before await use() = setup, code after = teardown
WhyEliminate repeated beforeEach/afterEach across files. DRY, composable, type-safe. Fixtures can depend on other fixtures. Auto-cleanup guaranteed even on test failure
WhenWhen you need reusable setup across test files: authenticated pages, API clients, seeded data, page objects, custom browser configurations
WhereIn a dedicated fixtures file (e.g., fixtures/index.ts). All test files import the custom test from this file instead of @playwright/test
Howexport const test = base.extend<MyFixtures>({...}). Each fixture is an async function receiving ({deps}, use). Call await use(value) to yield to the test

Keyword Glossary

test.extend()
Creates a new test object with added fixture definitions. Returns a new test function that replaces the base import
use callback
The function parameter in a fixture. Call await use(value) to yield the fixture value to the test. Code after use() runs as teardown
fixture scope
'test' (default) — new instance per test. 'worker' — shared across tests in same worker process
auto fixture
A fixture with { auto: true } runs for EVERY test even if the test doesn't declare it as a parameter
worker fixture
Scope: 'worker'. Shared across all tests in the same Playwright worker. E.g.: browser instance, DB connection
fixture dependency
Fixtures can receive other fixtures as parameters. Playwright resolves the dependency graph automatically
override fixture
Re-declare a built-in fixture (page, context) in test.extend() to modify its behavior globally
TestInfo
Object passed to fixture/test as second arg. Contains: title, status, errors, retry count, outputDir, attach()
mergeTests()
Combines multiple fixture files into one unified test object. Import from @playwright/test
test.use()
Configures options for all tests in a describe block or file. E.g., test.use({ storageState: 'auth.json' })

Interview One-Liners

Fixtures are superior to beforeEach because they're composable across files, type-safe, lazy (only instantiated for tests that need them), and auto-teardown even when tests throw exceptions.
The yield pattern: code before await use() is setup, await use() runs the test, code after is guaranteed teardown — atomically tied to its setup unlike beforeEach/afterEach.
I use worker-scoped fixtures for expensive one-time setup: launching a browser, starting a test server. Test-scoped fixtures create fresh contexts and pages per test.
Overriding the built-in page fixture lets me inject behavior into EVERY test's page — like blocking trackers, setting viewport — invisible to individual tests.
Auto fixtures are powerful for cross-cutting concerns: I have an auto fixture that captures all browser console errors and attaches them to the test report if the test fails.
TestInfo inside a fixture lets me write conditional teardown: if (testInfo.status !== 'passed') { await page.screenshot({...}) } — screenshots only on failure.

import { test as base, expect } from '@playwright/test';

const BLOCKED_DOMAINS = [
  'google-analytics.com', 'hotjar.com', 'segment.io',
  'facebook.net', 'doubleclick.net', 'clarity.ms',

export const test = base.extend({

    const consoleErrors: string[] = [];
    page.on('console', msg => {
      if (msg.type() === 'error') consoleErrors.push(msg.text());

    page.on('pageerror', err => consoleErrors.push(`PAGE ERROR: ${err.message}`));
    await use(page); // test uses the enhanced page
    if (consoleErrors.length > 0) {
      console.warn(`Browser errors:\n${consoleErrors.join('\n')}`);
    }
  },
});

  • ✓Always export your custom test and expect from a single fixtures file — all test files import from this one file, never from @playwright/test directly
  • ✓Put code before await use() for setup and code after for teardown — the fixture lifecycle guarantees teardown even if the test throws

Interview Prep — 5 SDET Q&A

Q1. Why are fixtures better than beforeEach/afterEach?
A: Five advantages: (1) Composability — fixtures depend on other fixtures. (2) Type safety — TypeScript knows fixture types. (3) Laziness — unused fixtures are never instantiated. (4) Guaranteed teardown — code after await use() always runs. (5) File-level sharing — available in any test file that imports the custom test.
Q2. When would you use worker scope vs test scope?
A: Test scope (default) creates fresh instance per test — perfect for pages, contexts. Worker scope creates one instance per worker process — perfect for expensive operations like browser launch, DB connection. Rule: cheap and stateful → test scope. Expensive and shareable → worker scope.
Q3. How do you create fixtures for different user roles?
A: Create separate fixtures for each role, each restoring the corresponding storageState. A multi-role test uses both: test('admin approves order', async ({adminPage, buyerPage}) => {...}). Both run in parallel inside the same test with their own context.
Q4. How do you override the built-in page fixture?
A: Re-declare page in test.extend(): page: async ({page}, use) => { await page.route(...); await use(page); }. Every test using this custom test object gets a pre-configured page with trackers blocked — invisible to individual tests.
Q5. How do you use TestInfo for failure screenshots?
A: TestInfo is the second parameter: page: async ({page}, use, testInfo) => { await use(page); if (testInfo.status !== testInfo.expectedStatus) { await page.screenshot({path: testInfo.outputPath('failure.png')}); await testInfo.attach('failure-screenshot', {path, contentType:'image/png'}); } }

Overview — Screenshots, Trace, Video & CLI

Playwright's debugging toolkit — screenshots (on-demand and on-failure), trace viewer (full test recording), video (MP4 of browser), and the complete CLI reference for running, debugging, generating, and configuring tests from the command line.

What / Why / When / Where / How

WhatScreenshots: captured via page.screenshot() or toHaveScreenshot(). Trace: full test recording with DOM snapshots. Video: MP4 recording. CLI: npx playwright commands
WhyDebug CI failures without reproducing locally. Trace viewer shows every action, network call, console log, and DOM snapshot. Visual regression testing with toHaveScreenshot()
WhenSet trace:'on-first-retry' and screenshot:'only-on-failure' in CI. Use --ui mode for local debugging. Use codegen to record test flows
WhereIn playwright.config.ts use:{} block for global settings. Per-project overrides in projects[].use:{}. Per-file with test.use()
HowConfig hierarchy: global use:{} → project use:{} → file test.use() → per-test options → CLI flags (highest priority)

Keyword Glossary

page.screenshot()
Manual capture. Options: path, fullPage, clip, mask (hide dynamic content), quality, type (png/jpeg)
toHaveScreenshot()
Visual regression assertion. Compares against baseline PNG. First run creates baseline. Options: maxDiffPixelRatio, mask
trace option
Config: 'off', 'on', 'on-first-retry' (recommended CI), 'retain-on-failure'. Generates .zip viewable in trace viewer
test.step()
Groups actions into named steps visible in trace viewer and HTML report. Makes complex tests readable
video option
Config: 'off', 'on', 'on-first-retry', 'retain-on-failure'. Saves .webm converted to .mp4
trace viewer
Built-in inspector: npx playwright show-trace trace.zip. Shows timeline, action log, network, console, DOM, screenshots per step
testInfo.attach()
Programmatically attach files, buffers, or text to the test report. Screenshots, logs, response bodies all attachable
page.pause()
Pauses test and opens Playwright Inspector for interactive debugging. Only works in headed mode (PWDEBUG=1)
--shard
Split test suite across machines: --shard=1/4 runs first 25%. Used with blob reporter to merge results
codegen
Record browser interactions as Playwright TypeScript code: npx playwright codegen https://myapp.com

Interview One-Liners

The trace viewer is my first stop for CI failures — it shows a complete timeline of every action, network request, and DOM state change. I use trace:'on-first-retry' so traces are only captured when tests fail.
I use three screenshot strategies: screenshot:'only-on-failure' in config for automatic capture, page.screenshot() in fixtures for teardown capture, and toHaveScreenshot() for visual regression.
test.step() groups actions into named steps visible in the trace viewer — makes failures immediately point to which step failed instead of individual method calls.
PWDEBUG=1 npx playwright test --headed --timeout=0 opens the Playwright Inspector with the test paused at page.pause() — I can step through actions and inspect locators live.
Playwright's config is hierarchical: global use:{} → per-project → per-file test.use() → per-test inline → CLI flags. Lower levels always override higher.
npx playwright codegen myapp.com opens the browser and records interactions as Playwright TypeScript code in real-time — fastest way to get working locators.

sharding-ci.yml

# .github/workflows/playwright-sharded.yml
# Split tests across 4 CI machines for 4x speed

name: Playwright Sharded Tests
on: [push, pull_request]

jobs:
  test:
    name: Shard ${{ matrix.shardIndex }}/${{ matrix.shardTotal }}
    runs-on: ubuntu-latest
    strategy:
      matrix:
        shardIndex: [1, 2, 3, 4]
        shardTotal: [4]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20', cache: 'npm' }
      - run: npm ci
      - run: npx playwright install --with-deps
      - run: npx playwright test
               --shard=${{ matrix.shardIndex }}/${{ matrix.shardTotal }}
               --reporter=blob
        env: { CI: 'true' }
      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: blob-report-${{ matrix.shardIndex }}
          path: blob-report/

  merge-reports:
    needs: [test]
    runs-on: ubuntu-latest
    if: always()
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20', cache: 'npm' }
      - run: npm ci
      - uses: actions/download-artifact@v4
        with: { path: all-blob-reports, pattern: blob-report-* }
      - run: npx playwright merge-reports
               --reporter html ./all-blob-reports
      - uses: actions/upload-artifact@v4
        with: { name: unified-report, path: playwright-report/ }

Best Practices

  • ✓Set trace:'on-first-retry' and screenshot:'only-on-failure' in CI config — captures debugging data only when needed without bloating artifacts
  • ✓Use test.step() to group logical actions — steps appear as named blocks in the trace viewer, making failures instantly understandable
  • ✓For visual regression: mask dynamic content (timestamps, user IDs) with mask: [page.locator('.timestamp')] to prevent false failures
  • ✓Use --shard=1/4 in CI to split tests across machines. Combine with blob reporter and merge-reports for a unified report
  • ✓Use --last-failed during local development to only re-run failing tests — much faster than running the full suite
  • ✓Use testInfo.outputPath('filename') for all file artifacts — creates a unique path per test run, preventing overwrite conflicts in parallel

Interview Prep — 5 SDET Q&A

Q1. When do you use trace vs video vs screenshot for debugging?
A: Trace is primary — interactive .zip with every action, network request, console message, DOM snapshot. Video shows visual browser experience as MP4 — useful for stakeholders. Screenshots are for quick checks and visual regression. Cost: trace > video > screenshot.
Q2. How does the Playwright config hierarchy work?
A: Settings flow global to local: (1) playwright.config.ts use:{} — global. (2) projects[n].use:{} — per-project. (3) test.use({}) — per-file. (4) Per-test inline options. (5) CLI flags — highest priority. Lower levels override higher.
Q3. How do you debug a flaky test using the trace viewer?
A: Set trace:'on-first-retry'. Let test fail in CI. Download trace.zip. Run npx playwright show-trace trace.zip. Examine Actions timeline, click failed step to see DOM snapshot, check Network tab for slow/missing API calls, check Console for JS errors.
Q4. How does test sharding work?
A: --shard=1/4 runs first 25% of tests deterministically. Each machine uses blob reporter. A final merge job runs: npx playwright merge-reports --reporter html ./all-blobs producing a unified HTML report. 4 shards = ~4× speed improvement.
Q5. How do you perform visual regression testing?
A: Use expect(locator).toHaveScreenshot('name.png'). First run creates baselines. Subsequent runs pixel-diff against baselines. Mask dynamic content. Set maxDiffPixelRatio:0.02 for tolerance. Update with --update-snapshots after intentional UI changes.
