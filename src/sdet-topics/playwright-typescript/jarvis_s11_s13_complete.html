Session 11 · Network Mocking & Interception

page.route() intercepts every outgoing network request matching a URL pattern. You can fulfill it with mock data, abort it to simulate failures, or continue it after modifying headers/body. This is the foundation of fast, deterministic, offline-capable tests.

Why Mock Networks?

Remove external dependencies. Test edge cases (empty lists, 500 errors, slow responses, malformed data) that are impossible to trigger reliably on a real API. Speed up tests by 10–50× by eliminating real network round-trips.

route.fulfill()

Responds to the intercepted request with your own data — status, headers, body (JSON / HTML / binary). The browser receives the mock as if the real server responded.

route.abort()

Cancels the request. Browser receives a network error. Use to test error states: no-internet, CORS failures, CDN timeouts, ad/tracker blocking.

route.continue()

Passes the request to the real server, optionally modifying headers, URL, or body before forwarding. Use for auth token injection, URL rewriting, A/B testing overrides.

Glossary
page.route()Registers a request interceptor. First arg: URL pattern (string glob, regex, predicate). Second arg: handler function receiving (route, request). Fires for every matching request.
route.fulfill()Responds to request with mock data. Options: status (HTTP code), contentType, body (string/Buffer), headers, json (auto-sets content-type). Does NOT hit real server.
route.abort()Cancels the request with a network error code: 'failed', 'timedout', 'connectionrefused', 'blockedbyclient', 'connectionaborted'. Browser sees a network error.
route.continue()Forwards request to real server, with optional overrides: url, method, headers, postData. Use for partial modification without full mocking.
route.fetch()Fetches the real response, lets you inspect/modify it, then fulfill with the modified version. Perfect for response transformation without full mocking.
page.unroute()Removes a previously registered route handler. Use to remove mocks after a specific test step without tearing down the whole context.
context.route()Same as page.route() but applies to ALL pages in the BrowserContext. Use for auth headers, analytics blocking, global error states.
route.request()Returns the Request object inside a route handler. Access: url(), method(), headers(), postData(), postDataJSON().
page.on('request')Event listener (not interceptor) — observes requests passively without intercepting. Use for logging, verifying calls were made.
page.on('response')Event listener for responses. Access status, headers, body. Use alongside route() to spy on real responses.
HAR fileHTTP Archive — a JSON recording of all network traffic. Playwright can record (page.routeFromHAR) and replay HAR files for full offline test playback.
routeFromHAR()Replays network responses from a saved HAR file. Great for complex multi-request flows that would be tedious to mock individually.

Core Concepts

Route handler decision tree

What do you want to do?
Return fake data→route.fulfill({json:{...}})
Simulate network failure→route.abort('failed')
Modify request then send→route.continue({headers:{...}})
Fetch real + modify response→const r=await route.fetch(); route.fulfill({...r, body:modified})
Block tracking scripts→route.abort('blockedbyclient')
Add delay (slow network)→setTimeout then route.fulfill()
Replay recorded traffic→page.routeFromHAR('recording.har')

Priority and multiple routes

Multiple routes can match the same URL. Playwright calls them in LIFO (last registered = first called) order. The first handler that calls fulfill/abort/continue wins — subsequent handlers are skipped.

page.route('**/api/**', globalHandler)registered 1st
page.route('**/api/users', specificHandler)registered 2nd → called FIRST
→ specificHandler runs first (LIFO)
→ if it calls route.continue(), globalHandler still runs
Best Practices
  • ✓
    Always set up page.route() BEFORE page.goto() — routes registered after navigation won't intercept requests that already fired during page load.
  • ✓
    Use route.fulfill({json: data}) shorthand — it automatically sets Content-Type: application/json and serializes the object. No need for JSON.stringify() and manual header setting.
  • ✓
    Use context.route() for cross-page mocks (analytics blocking, global auth) and page.route() for test-specific mocks.
  • ✓
    Use route.fetch() + route.fulfill() when you want real data but need to augment it (add a field, change a status) rather than replacing it entirely.
  • ✓
    Call page.unroute() or use route.fulfill({times:1}) (one-shot) to remove mocks that should only fire once — prevents accidental interference with later test steps.
  • ✓
    Use routeFromHAR() for integration-style tests with complex multi-request flows. Record once, replay forever — no maintenance per API change unless the contract changes.
  • ✓
    Block third-party scripts (analytics, ads, chat widgets) globally with context.route() in your fixture — they add flakiness and slow down tests without contributing to coverage.
Interview Prep
Q1: What is the difference between page.route() and page.on('response')? ▶
page.route() is an interceptor — it INTERCEPTS the request before it reaches the server (or instead of it). You can fulfill (fake the response), abort (kill it), or continue (modify and forward it). It has full control. page.on('response') is a passive observer — it listens to real responses AFTER they come back from the server. You can read the response data but cannot modify or block it. Use page.route() when you want to control the response (mocking, error simulation). Use page.on('response') when you want to spy on what the real server returned without interfering.
Q2: How do you test an empty state (no data) UI component? ▶
I intercept the API call and return an empty array: await page.route('**/api/products', route => route.fulfill({ json: { products: [], total: 0 } })); then await page.goto('/products'). Now the page receives an empty response. I assert: await expect(page.locator('.empty-state')).toBeVisible(); await expect(page.locator('.product-card')).toHaveCount(0); await expect(page.locator('.empty-msg')).toContainText('No products found'). This tests the empty state UI deterministically — no need to have an empty database, no risk of real data appearing unexpectedly.
Q3: How do you simulate a 500 server error to test error handling UI? ▶
await page.route('**/api/users', route => route.fulfill({ status: 500, json: { error: 'Internal Server Error', message: 'Database connection failed' } })); After triggering the action that calls the API, I assert the error UI: await expect(page.locator('.error-banner')).toBeVisible(); await expect(page.locator('.error-banner')).toContainText('Something went wrong'); await expect(page.locator('.retry-btn')).toBeVisible(). I test multiple error codes — 401 (redirect to login), 403 (permission denied UI), 404 (not found screen), 429 (rate limit UI) — each with a separate test using a different route.fulfill() status.
Q4: How does route.fetch() differ from route.continue()? ▶
route.continue() forwards the request to the real server and you never see the response — you just optionally modify the request (headers, body, URL) before sending. You can't inspect what came back. route.fetch() makes the actual network call and returns a Response object to your handler. You can read the real response, modify it (add/remove fields, change status), then call route.fulfill() with the modified version. Example: const realResp = await route.fetch(); const body = await realResp.json(); body.isAdmin = true; // inject a field; await route.fulfill({json: body}). This is the most powerful pattern — real data with surgical modifications.
Q5: How do you block all analytics and tracking scripts in tests? ▶
I add a context-level route in a fixture that aborts requests matching known tracker domains: await context.route(url => ['google-analytics.com','hotjar.com','segment.io','facebook.net','doubleclick.net'].some(d => url.includes(d)), route => route.abort()); This runs for every test using that fixture. Benefits: (1) Tests run faster — no waiting for third-party scripts. (2) No flakiness from external CDN availability. (3) No test interference from analytics events firing during test actions. (4) Console is clean — no third-party errors polluting test output. I put this in a base fixture that every test inherits from.

spy-network.ts
// spy-network.ts — passively spy on requests/responses without blocking
import { test, expect } from '@playwright/test';

test('spy on all API calls — verify calls were made', async ({ page }) => {

  const requests: { url: string; method: string }[] = [];
  const responses: { url: string; status: number }[] = [];

  // Passive listener — does NOT intercept, just observes
  page.on('request', (req) => {
    if (req.url().includes('/api/')) {
      requests.push({ url: req.url(), method: req.method() });
    }
  });

  page.on('response', (resp) => {
    if (resp.url().includes('/api/')) {
      responses.push({ url: resp.url(), status: resp.status() });
    }
  });

  await page.goto('https://myapp.com/dashboard');
  await page.waitForLoadState('networkidle');

  // Assert specific API calls were made
  const userCall = requests.find(r => r.url.includes('/api/user'));
  expect(userCall).toBeDefined();
  expect(userCall!.method).toBe('GET');

  // Assert no 4xx/5xx errors occurred
  const errors = responses.filter(r => r.status >= 400);
  expect(errors, 'No API errors should occur').toHaveLength(0);

  // Log full call inventory for debugging
  console.log(`API calls made: ${requests.length}`);
  requests.forEach(r => console.log(`  ${r.method} ${r.url}`));
});

Session 12 · Custom Fixtures

Playwright fixtures are a dependency injection system built into @playwright/test. They handle setup, yield resources to tests, then auto-teardown — even if the test throws. Custom fixtures extend the base test object with your own reusable setup, from page objects to authenticated users to seeded databases.

Why Fixtures?

Eliminate repeated beforeEach/afterEach across files. DRY, composable, type-safe. Fixtures can depend on other fixtures, forming a setup chain. Auto-cleanup guaranteed.

test.extend()

Creates a new test object with added fixtures. Tests import this custom test instead of base. Fixtures are declared as async generator functions using use callback.

Scopes

'test' (default) — fresh instance per test. 'worker' — shared across tests in the same worker process. 'global' — set via globalSetup, not test.extend.

Composing Fixtures

Fixtures can use other fixtures as parameters. A loggedInPage fixture can depend on page and context built-ins, composing them into a higher-level abstraction.

fixture scope'test' (default) — new instance per test. 'worker' — shared across tests in same worker process (e.g., browser launch). Declared in fixture options object.

Best Practices
  • ✓
    Always export your custom test and expect from a single fixtures file. All test files import from this one file — never from @playwright/test directly.
  • ✓
    Put code before await use() for setup and code after for teardown. Never use try/finally — the fixture lifecycle guarantees teardown even if the test throws.
  • ✓
    Use worker scope only for truly expensive operations (browser launch, DB connect, server start). Everything else should be test scope for isolation.
  • ✓
    Use auto:true fixtures for cross-cutting concerns: error logging, performance timing, analytics mock. Tests don't need to declare them.
  • ✓
    Create role-specific auth fixtures: adminPage, buyerPage, guestPage — each restores the correct storageState. Multi-role tests can use multiple fixtures simultaneously.
  • ✓
    Use TestInfo in fixtures for conditional teardown — capture screenshots or traces only when tests fail, to avoid bloating CI artifact storage.
  • ✓
    Merge multiple fixture files using chained test.extend() — one fixture file for auth, one for API, one for POM. Import the final merged test in all spec files.
Interview Prep
Q1: Why are fixtures better than beforeEach/afterEach? ▶
Five key advantages: (1) Composability — fixtures can depend on other fixtures, forming a setup chain. beforeEach can't be composed across files cleanly. (2) Type safety — custom fixtures are typed, giving IntelliSense for fixture values. (3) Laziness — fixtures are only instantiated for tests that declare them. An unused fixture costs nothing. (4) Guaranteed teardown — code after await use() always runs, even if the test throws, because it's inside the fixture's execution scope. beforeEach has no built-in link to afterEach — a failing test can leave afterEach in a broken state. (5) File-level sharing — fixtures defined once are available in any test file that imports the custom test object, without any explicit import per file.
Q2: When would you use worker scope vs test scope for a fixture? ▶
Test scope (default) creates a fresh instance per test — perfect for pages, contexts, API clients. It gives isolation but costs more time. Worker scope creates one instance per worker process and shares it across all tests in that worker — perfect for expensive operations. Real examples: (1) Browser launch — test.extend({ browser: [{async({}, use) { const b=await chromium.launch(); await use(b); await b.close(); }}, {scope:'worker'}] }). All tests share one browser process but get fresh contexts per test. (2) Test database connection — one connection per worker, shared across tests. (3) Local dev server — started once per worker. The rule: anything cheap and stateful → test scope. Anything expensive and can be shared safely → worker scope.
Q3: How do you create a fixture for different user roles? ▶
I create separate fixtures for each role, each restoring the corresponding storageState: adminPage: async ({browser}, use) => { const ctx = await browser.newContext({storageState:'playwright/.auth/admin.json'}); const pg = await ctx.newPage(); await use(pg); await ctx.close(); }. Similarly buyerPage, guestPage. A multi-role test: test('admin approves order', async ({adminPage, buyerPage}) => { await buyerPage.goto('/products'); await buyerPage.click('.buy'); await adminPage.goto('/admin/orders'); await expect(adminPage.locator('.pending-order')).toBeVisible(); }). Both fixtures run in parallel inside the same test, each with their own context and page. Teardown closes both contexts when the test ends.
Q4: How do you override the built-in page fixture? ▶
Re-declare page in test.extend() and use the built-in via the fixture parameter: export const test = base.extend({ page: async ({page}, use) => { // Intercept trackers for every test's page await page.route(url => trackers.some(t=>url.includes(t)), r=>r.abort()); // Set consistent locale await page.evaluate(() => document.documentElement.lang = 'en'); await use(page); // page is now the modified version } }). Every test using this custom test object gets a pre-configured page with trackers blocked. This is invisible to individual tests — they just use page normally. The same pattern works for context to inject storageState or viewport globally.
Q5: How do you use TestInfo inside a fixture for failure screenshots? ▶
TestInfo is the second parameter of every fixture and test: page: async ({page}, use, testInfo) => { await use(page); if (testInfo.status !== testInfo.expectedStatus) { const screenshotPath = testInfo.outputPath('failure-screenshot.png'); await page.screenshot({path: screenshotPath, fullPage:true}); await testInfo.attach('failure-screenshot', {path: screenshotPath, contentType:'image/png'}); } }. Code after use() is teardown — it sees the final test status. testInfo.status is 'passed', 'failed', 'timedOut', or 'skipped'. testInfo.outputPath() returns a path inside the test's dedicated output folder. testInfo.attach() adds the file to the HTML report and trace viewer as a named attachment. This approach is more efficient than screenshot:'on' because it only captures when needed.

  'google-analytics.com', 'hotjar.com', 'segment.io',

    }
  },
});

Session 13 · Screenshots · Trace · Video · Global→Local Setup · CLI

Playwright's debugging toolkit — screenshots (on-demand and on-failure), trace viewer (full test recording), video (MP4 of browser), and the complete CLI reference for running, debugging, generating, and configuring tests from the command line.

Screenshots

Captured via page.screenshot() (manual), screenshot:'only-on-failure' (auto), or toHaveScreenshot() (visual regression). Saved as PNG, attachable to reports.

Trace

Full test recording: every action, network call, console log, DOM snapshot, screenshots. Viewable in trace viewer (npx playwright show-trace). Best debugging tool for CI failures.

Video

MP4 recording of the browser window. Slower than trace. Use video:'on-first-retry' for CI. Saved in test-results folder. Shows EXACTLY what the browser did visually.

Config Hierarchy

Settings cascade from playwright.config.ts (global) → project use: (per browser) → test.use() (per file) → individual test options (per test). Lower levels override higher.

Interview Prep
Q1: When do you use trace vs video vs screenshot for debugging? ▶
Trace is my primary debugging tool — it records every action, network request, console message, and DOM snapshot in one interactive .zip file. I can jump to any moment and inspect the exact DOM state. I use trace:'on-first-retry' for CI. Video shows the visual browser experience as an MP4 — useful for sharing with stakeholders or understanding complex animation/timing issues that don't show up in DOM snapshots. I use video:'retain-on-failure'. Screenshots are for quick checks — visual regression with toHaveScreenshot(), automatic failure capture with screenshot:'only-on-failure', or manual captures inside test logic. Each has a different cost: trace > video > screenshot in terms of file size and processing overhead.
Q2: How does the Playwright config hierarchy work? ▶
Settings flow from global to local, with lower levels overriding higher: (1) playwright.config.ts use:{} — global defaults for all tests and all projects. (2) projects[n].use:{} — overrides global for a specific project (e.g., Firefox gets different viewport). (3) test.use({}) inside a spec file — overrides project config for all tests in that file. (4) Per-test options: test('name', {screenshot:'on'}, async...) — inline options for one specific test. (5) CLI flags (--trace=on, --headed) — highest priority, override everything in the config file. This hierarchy means you can set 'trace:off' globally, 'trace:on-first-retry' for CI project, and 'trace:on' for a specific test file you're debugging, all without touching other tests.
Q3: How do you debug a flaky test using the trace viewer? ▶
Workflow: (1) Set trace:'on-first-retry' in config so traces are captured when the flaky test retries. (2) Let the test run in CI until it fails. (3) Download the trace.zip from CI artifacts. (4) Run npx playwright show-trace trace.zip to open the interactive viewer. (5) In the viewer: examine the Actions timeline to find the step that failed, click on it to see the DOM snapshot at that exact moment, check the Network tab to see if an API call was slow/missing, look at the Console tab for any JS errors. (6) Common findings: element not found = slow network caused content to load after locator timed out; element covered = animation/overlay was still visible; unexpected URL = redirect happened before assertion. Fix based on the specific finding.
Q4: How do you split tests across multiple CI machines using sharding? ▶
Playwright's --shard flag splits the test suite deterministically: --shard=1/4 runs the first 25%, --shard=2/4 the second 25%, etc. In GitHub Actions, use a matrix strategy: matrix: { shardIndex: [1,2,3,4], shardTotal: [4] }. Each job runs: npx playwright test --shard=${{ matrix.shardIndex }}/${{ matrix.shardTotal }}. Each shard produces its own blob report. After all shards complete, a merge job runs: npx playwright merge-reports --reporter html ./all-blob-reports/ to create a unified HTML report from all shards. This gives a single report showing all test results even though tests ran across multiple machines. Time reduction: 4 shards on 4 machines = ~4× faster CI.
Q5: How do you perform visual regression testing and maintain baselines? ▶
Initial setup: Write tests using expect(locator).toHaveScreenshot('name.png'). Run once with --update-snapshots to create baselines. Commit the generated PNG files in __screenshots__ folder to git. Every subsequent run: Playwright takes a new screenshot, pixel-diffs it against the baseline, fails if diff exceeds threshold. For maintenance: When you intentionally change UI, run npx playwright test --update-snapshots after reviewing the diff. Best practices: (1) Mask dynamic content (timestamps, avatars, random IDs) with the mask option. (2) Use element-level screenshots (locator.toHaveScreenshot) not full-page, for stability. (3) Set maxDiffPixelRatio:0.02 (2% tolerance) to handle minor font rendering differences across environments. (4) Run visual tests in a dedicated project with a fixed OS/browser combination — screenshots differ across OSes.

  await testInfo.attach('dashboard-view', {

# sharding-ci.yml — split tests across 4 CI machines for 4x speed
# File: .github/workflows/playwright-sharded.yml

name: Playwright Sharded Tests

on: [push, pull_request]

jobs:
  test:
    name: Shard ${{ matrix.shardIndex }}/${{ matrix.shardTotal }}
    runs-on: ubuntu-latest
    timeout-minutes: 30
    strategy:
      matrix:
        shardIndex: [1, 2, 3, 4]  # 4 parallel machines
        shardTotal: [4]

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20', cache: 'npm' }
      - run: npm ci
      - run: npx playwright install --with-deps

      # Run one shard on this machine
      - run: >
          npx playwright test
          --shard=${{ matrix.shardIndex }}/${{ matrix.shardTotal }}
          --reporter=blob
        env: { CI: 'true' }

      # Upload shard's blob report
      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: blob-report-${{ matrix.shardIndex }}
          path: blob-report/
          retention-days: 1

  merge-reports:
    needs: [test]  # runs AFTER all shards complete
    runs-on: ubuntu-latest
    if: always()
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20', cache: 'npm' }
      - run: npm ci
      - name: Download all blob reports
        uses: actions/download-artifact@v4
        with: { path: all-blob-reports, pattern: blob-report-* }
      - name: Merge into one HTML report
        run: >
          npx playwright merge-reports
          --reporter html ./all-blob-reports
      - uses: actions/upload-artifact@v4
        with: { name: unified-report, path: playwright-report/ }
