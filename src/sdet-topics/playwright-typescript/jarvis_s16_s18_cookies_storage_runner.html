Overview — Cookie Handling in Playwright

Cookies are small key-value data pieces stored by the browser and sent with every HTTP request to their domain. Playwright exposes full cookie control through context.addCookies(), context.cookies(), context.clearCookies(), and storageState — enabling auth injection, session testing, and cookie-based state verification. Each BrowserContext has its own isolated cookie jar.

What / Why / When / Where / How

WhatBrowser-stored key-value pairs with domain, path, expiry, httpOnly, secure, and sameSite attributes — managed per BrowserContext in Playwright
WhyBypass login UI by injecting session cookies, test cookie-based features (remember-me, GDPR consent), verify cookies are set correctly after auth, test cookie expiry and secure attributes
WhenWhen restoring auth state without re-logging in, testing cookie-dependent features, verifying server sets correct cookies, testing cross-domain cookie behavior
WhereIn test fixtures for auth injection, in beforeEach/afterEach hooks, inside test assertions for cookie verification
Howcontext.addCookies([]) injects cookies, context.cookies() reads all cookies, context.clearCookies() removes them, storageState saves/restores the full cookie jar

Keyword Glossary

context.addCookies()
Injects an array of cookie objects into the BrowserContext. Each cookie must have name, value, domain (or url). Optional: path, expires, httpOnly, secure, sameSite.
context.cookies()
Async method returning all cookies in the context as an array. Accepts optional URL filter: context.cookies('https://myapp.com').
context.clearCookies()
Removes all cookies from the context. Pass optional filter: {name, domain, path} to selectively clear specific cookies.
storageState
JSON snapshot containing all cookies + localStorage. Saved via context.storageState({path}), restored via browser.newContext({storageState}).
httpOnly
Cookie attribute that prevents JavaScript from accessing the cookie via document.cookie. Must be read through Playwright's context API.
secure
Cookie attribute requiring HTTPS to send the cookie. In Playwright tests with HTTP localhost, set secure:false or use ignoreHTTPSErrors:true.
sameSite
Controls cross-site request behavior: 'Strict' (never sent cross-site), 'Lax' (sent on top-level navigation), 'None' (always sent, requires secure:true).
domain
The domain the cookie applies to. Use .myapp.com (dot prefix) for subdomain cookies. Must match the actual domain you're testing.
expires
Unix timestamp (seconds) when the cookie expires. -1 = session cookie (deleted when browser closes). Future timestamp = persistent cookie.
page.evaluate()
Runs JavaScript inside the browser — can access document.cookie for non-httpOnly cookies. For httpOnly cookies, must use context.cookies().

Interview One-Liners

Instead of driving the full login UI flow in every test, inject the session cookie directly using context.addCookies() — the browser sends it with every HTTP request as if the user just logged in.
httpOnly cookies cannot be read by JavaScript on the page — in Playwright, use context.cookies() from the test code side to read them, bypassing the browser's JS sandbox.
storageState saves and restores the entire cookie jar plus localStorage in one JSON file — the complete session snapshot. addCookies() injects individual cookies for precise control.
The most common cookie injection failure is a domain mismatch — always verify by calling context.cookies() immediately after addCookies() to confirm injection worked.
To test the logged-out state from an authenticated context, call context.clearCookies() instead of creating a new context — it simulates the user clearing their browser cookies.
For apps with GDPR cookie consent banners, inject the consent cookie directly in a fixture using addCookies() — this bypasses the consent dialog entirely.

add-cookies.ts

import { test, expect } from '@playwright/test';

// PURPOSE: Inject cookies programmatically before navigation
// Shows all cookie properties and verification pattern

test('inject cookies with all properties', async ({ context, page }) => {

  // Inject BEFORE navigation — cookies must exist before page loads
  await context.addCookies([
    {
      name:     'session_id',          // cookie identifier key
      value:    'test-session-abc123',  // cookie value data
      domain:   'myapp.com',           // exact domain match required
      path:     '/',                   // URL scope
      httpOnly: true,                  // JS cannot read via document.cookie
      secure:   false,                 // allow HTTP (set true for HTTPS only)
      sameSite: 'Lax',                 // 'Strict' | 'Lax' | 'None'
      expires: Math.floor(Date.now() / 1000) + 3600, // 1 hour from now
    },
    // Multiple cookies in one call
    {
      name:   'user_preferences',
      value:  '{"theme":"dark","lang":"en"}',
      domain: 'myapp.com',
      path:   '/',
    },
    // Subdomain cookie — dot prefix for all subdomains
    {
      name:   'cross_subdomain',
      value:  'shared',
      domain: '.myapp.com',            // note the leading dot
      path:   '/',
    },
  ]);

  // Verify injection succeeded (domain mismatch = silent failure)
  const cookies = await context.cookies();
  expect(cookies.find(c => c.name === 'session_id')).toBeDefined();
  expect(cookies).toHaveLength(3);

  // Navigate — browser sends cookies automatically in request headers
  await page.goto('https://myapp.com/dashboard');
  await expect(page.locator('.user-greeting')).toBeVisible();
});
read-cookies.ts

import { test, expect } from '@playwright/test';

// PURPOSE: Read and inspect cookies — context.cookies() vs document.cookie
// KEY: httpOnly cookies only visible via context.cookies() not document.cookie

test('read all cookies after login', async ({ context, page }) => {
  await page.goto('https://myapp.com/login');
  await page.fill('#email',    'user@test.com');
  await page.fill('#password', 'Pass123!');
  await page.click('button[type="submit"]');
  await page.waitForURL('**/dashboard');

  // Method 1: context.cookies() — ALL cookies including httpOnly
  const allCookies = await context.cookies();
  console.log(`Total cookies: ${allCookies.length}`);
  allCookies.forEach(c => console.log(`  ${c.name}=${c.value} httpOnly=${c.httpOnly}`));

  // Filter cookies for a specific URL
  const appCookies = await context.cookies('https://myapp.com');
  console.log(`App-specific cookies: ${appCookies.length}`);

  // Find specific cookie by name
  const sessionCookie = allCookies.find(c => c.name === 'session_id');
  expect(sessionCookie).toBeDefined();
  expect(sessionCookie!.httpOnly).toBe(true);
  expect(sessionCookie!.secure).toBe(true);

  // Method 2: page.evaluate — ONLY non-httpOnly cookies visible in JS
  const jsCookies = await page.evaluate(() => document.cookie);
  console.log(`JS-visible cookies: ${jsCookies}`);
  // session_id does NOT appear here because httpOnly:true prevents JS access
  expect(jsCookies).not.toContain('session_id');
});

test('simulate logout by clearing session cookies', async ({ context, page }) => {

  // Start authenticated — inject session cookie
  await context.addCookies([
    { name: 'session_id',   value: 'abc123', domain: 'myapp.com', path: '/' },
    { name: 'remember_me',  value: 'true',   domain: 'myapp.com', path: '/' },
    { name: 'analytics_id', value: 'ga123',  domain: 'myapp.com', path: '/' },
  ]);
  await page.goto('https://myapp.com/dashboard');
  await expect(page.locator('.user-greeting')).toBeVisible();

  // Clear ONLY the session cookie — simulate session expiry
  await context.clearCookies({ name: 'session_id' });

  // Verify only session_id removed, others remain
  const remaining = await context.cookies();
  expect(remaining.find(c => c.name === 'session_id')).toBeUndefined();
  expect(remaining.find(c => c.name === 'remember_me')).toBeDefined();

  // Reload — should redirect to login since session_id gone
  await page.reload();
  await expect(page).toHaveURL(/login/);

  // Clear ALL cookies — full logout
  await context.clearCookies();
  const afterClear = await context.cookies();
  expect(afterClear).toHaveLength(0);
});

test.describe('Cookie expiry scenarios', () => {

  test('expired cookie → redirect to login', async ({ context, page }) => {
    const oneHourAgo = Math.floor(Date.now() / 1000) - 3600;

Best Practices

  • ✓Always verify cookie injection worked by calling context.cookies() immediately after addCookies() — domain mismatches cause silent failures
  • ✓Use storageState for auth reuse and addCookies() for precise attribute testing — don't use the full login UI flow in tests that aren't testing login
  • ✓Inject GDPR consent cookies in a global fixture using addCookies() — prevents consent banners from blocking test interactions in every test
  • ✓Test httpOnly and secure attributes via context.cookies() — never via page.evaluate(document.cookie) which cannot see httpOnly cookies
  • ✓Use expires: Math.floor(Date.now()/1000) + 3600 for 1-hour persistent cookies, expires: -1 for session cookies, and a past timestamp to test expiry behavior
  • ✓When testing cross-subdomain cookie sharing, use the domain: '.myapp.com' (dot-prefixed) form — this makes the cookie available on all subdomains
  • ✓After each test that modifies cookies, use context.clearCookies() in afterEach or rely on per-test fresh contexts (recommended) to prevent state leakage

Interview Prep — 5 SDET Q&A

Q1. How do you inject a session cookie to bypass login in Playwright?
A: I use context.addCookies() before any navigation: await context.addCookies([{ name: 'session_id', value: 'known-test-token', domain: 'myapp.com', path: '/', httpOnly: true, secure: false }]). Then await page.goto('/dashboard'). The page loads as if the user is logged in because the browser sends the cookie with every request. Important: the domain must exactly match the app's domain. I verify it worked with: const cookies = await context.cookies(); expect(cookies.find(c => c.name === 'session_id')).toBeTruthy().
Q2. How do you test that a server sets cookies correctly after login?
A: After the login action, I read all cookies using context.cookies('https://myapp.com') and assert on their properties: const sessionCookie = cookies.find(c => c.name === 'session_id'). Then I verify: expect(sessionCookie.httpOnly).toBe(true), expect(sessionCookie.secure).toBe(true), expect(sessionCookie.sameSite).toBe('Strict'), and expect(sessionCookie.expires).toBeGreaterThan(Date.now()/1000). This tests the server's security configuration.
Q3. What is the difference between context.cookies() and document.cookie in evaluate()?
A: They have fundamentally different visibility. document.cookie (accessed via page.evaluate()) only returns cookies that do NOT have the httpOnly flag — this is intentional to prevent XSS attacks. context.cookies() is a Playwright API call that reads from the browser's internal cookie store directly, returning ALL cookies including httpOnly ones. For reading session tokens and auth cookies (which are almost always httpOnly), always use context.cookies().
Q4. How do you test cookie expiry behavior?
A: Two scenarios: (1) Expired cookie — inject a cookie with a past expires timestamp: const pastTime = Math.floor(Date.now()/1000) - 3600. Navigate and verify the app treats the user as logged out. (2) Approaching expiry — set expires to 5 minutes from now and assert the app shows a "session expiring soon" warning. For session cookies (expires: -1), close and reopen the context to test they're cleared.
Q5. How do you handle third-party cookies and SameSite testing?
A: sameSite:'None' cookies require secure:true, which requires HTTPS. For testing in localhost HTTP environments, set ignoreHTTPSErrors:true in context options. To test SameSite behavior: for 'Strict' — verify the cookie is NOT sent when navigating from an external domain. For 'Lax' — verify it IS sent on top-level navigation but not on cross-origin subresource requests. For 'None' with cross-origin iframes, verify the iframe receives the cookie.

Overview — LocalStorage Handling in Playwright

LocalStorage is a browser storage mechanism that persists key-value strings across page loads within the same origin. Playwright accesses it via page.evaluate() (running JavaScript in the browser context) and through the storageState API which captures and restores all localStorage keys along with cookies.

What / Why / When / Where / How

WhatA synchronous key-value store in the browser — data persists across page reloads and browser restarts (unlike sessionStorage). Scoped to origin. Stores only strings — objects need JSON.stringify()
WhyApps store auth tokens (JWT), user preferences, cart state, dark-mode settings, feature flags, and form drafts in localStorage — tests must set, verify, and test app behavior with these values
WhenWhen testing localStorage-based auth (JWTs), app initialization from stored preferences, cart persistence across refreshes, feature flag overrides, and app recovery from corrupted storage
WhereIn addInitScript() for pre-navigation seeding, in page.evaluate() for post-load reading, and in storageState for complete session snapshots
HowOnly via page.evaluate() — Playwright doesn't have a dedicated localStorage API. Inject state BEFORE navigation via addInitScript() or AFTER via evaluate(). storageState snapshots include localStorage

Keyword Glossary

localStorage
Browser's persistent key-value string store. Origin-scoped. Survives page refresh and browser restart. Max ~5MB per origin. No expiry. Accessed via window.localStorage in JS.
sessionStorage
Similar to localStorage but tab-scoped — cleared when tab closes. Not shared between tabs/windows. Use page.evaluate() to access.
page.evaluate()
Executes a JavaScript function in the browser's page context. Primary way to interact with localStorage. Can read, write, remove, and clear.
addInitScript()
Injects a script that runs BEFORE any other script on the page — on every navigation. Critical for localStorage-based auth seeding.
storageState
JSON snapshot of both cookies AND localStorage for the context. The 'origins' array contains localStorage key-value pairs per origin.
localStorage.setItem()
JS method: window.localStorage.setItem('key', 'value'). Value must be a string — use JSON.stringify() for objects.
localStorage.getItem()
JS method: window.localStorage.getItem('key'). Returns string or null. Call inside page.evaluate() to read back to Node.js test code.
localStorage.removeItem()
Removes a specific key from localStorage. Use to test app behavior when a required key is missing.
localStorage.clear()
Removes ALL keys from localStorage for the current origin. Use to simulate a fresh user with no stored data.
context.addInitScript()
Context-level version — runs on ALL pages opened in the context. Use for global localStorage state across all navigation.

Interview One-Liners

page.evaluate() runs AFTER the page loads — the app's JS may have already read localStorage. page.addInitScript() runs BEFORE any app code, making it the correct way to seed localStorage for app initialization.
Many SPAs store JWT tokens in localStorage — to bypass login, use addInitScript to inject the token before the app's auth guard checks it.
localStorage only stores strings — objects must be JSON.stringify()'d before storing and JSON.parse()'d when reading back. Forgetting this creates '[object Object]' bugs.
storageState captures both cookies AND localStorage — when restored, the app sees both the session cookie AND the stored user preferences exactly as left.
Test app resilience by injecting malformed data into localStorage: localStorage.setItem('user_config', '{broken json}') and asserting the app gracefully falls back to defaults.
sessionStorage is tab-scoped and cleared when the tab closes — test it via page.evaluate() the same way as localStorage but it won't persist across contexts.

read-localstorage.ts

import { test, expect } from '@playwright/test';

// PURPOSE: Read localStorage values and all keys via page.evaluate()

test('read and verify localStorage after user action', async ({ page }) => {
  await page.goto('https://myapp.com/settings');
  await page.getByLabel('Language').selectOption('hi');
  await page.click('#save-preferences');

  // Read single key — raw string
  const lang = await page.evaluate(
    () => window.localStorage.getItem('language')
  );
  expect(lang).toBe('hi');

  // Read and parse JSON object
  const prefs = await page.evaluate(() => {
    const raw = window.localStorage.getItem('user_preferences');
    return raw ? JSON.parse(raw) : null;
  });
  expect(prefs).toMatchObject({ language: 'hi' });

  // Read ALL localStorage keys
  const allStorage = await page.evaluate(() => {
    const result: Record = {};
    for (let i = 0; i < window.localStorage.length; i++) {
      const key = window.localStorage.key(i)!;
      result[key] = window.localStorage.getItem(key)!;
    }
    return result;
  });
  expect(Object.keys(allStorage).length).toBeGreaterThan(0);

  const missing = await page.evaluate(
    () => window.localStorage.getItem('nonexistent_key')
  );
  expect(missing).toBeNull();
});

Best Practices

  • ✓Use addInitScript() to seed localStorage for app initialization — NOT evaluate() after goto(). The app reads localStorage on mount; evaluate() is too late
  • ✓Always JSON.stringify() objects before setItem() and JSON.parse() when reading back — localStorage only stores strings
  • ✓Test localStorage-based auth by injecting JWT/token via initScript, then asserting the app loads the authenticated state without touching the login page
  • ✓Use localStorage.clear() via evaluate to test app behavior with empty storage — simulates a new user's first visit. Assert app shows onboarding, not a crash
  • ✓Test corrupted localStorage by injecting malformed JSON — verify the app catches parse errors and falls back gracefully instead of crashing
  • ✓For storageState auth reuse, run the real login once in globalSetup which captures both cookies AND localStorage. Tests restoring storageState get the complete app session state
  • ✓Prefer storageState over manual addInitScript for auth when the app uses a combination of cookies + localStorage — storageState captures both atomically from a real session

Interview Prep — 5 SDET Q&A

Q1. How do you set localStorage values before the page app code runs?
A: The critical pattern is addInitScript(), not evaluate(). When you call page.goto(), the browser loads HTML, then JavaScript. If you use page.evaluate() after goto(), the app's JS has already run. addInitScript() registers a script that runs BEFORE any other JavaScript on the page: await page.addInitScript(() => { window.localStorage.setItem('auth_token', 'my-test-jwt'); }). Then await page.goto('/dashboard'). The app initializes and reads auth_token from localStorage — it sees the pre-seeded value.
Q2. How does storageState capture and restore localStorage?
A: When you call context.storageState({path:'auth.json'}), Playwright snapshots the entire browser storage state into a JSON file with two sections: "cookies" and "origins" (containing localStorage per origin). When you restore with browser.newContext({storageState:'auth.json'}), Playwright pre-populates both the cookie jar AND localStorage for each origin before any page is opened.
Q3. How do you test that an app correctly persists user preferences in localStorage?
A: Test the full read-write-restore cycle: (1) Verify initial state is null. (2) Trigger the preference change via UI. (3) Verify localStorage was updated via page.evaluate(() => localStorage.getItem('theme')). (4) Reload the page and verify the app reads and applies the stored preference on load.
Q4. How do you test app resilience when localStorage has corrupted data?
A: Inject corrupted data using addInitScript (so it's set before app initialization): window.localStorage.setItem('user_config', '{broken json}'). Then navigate and assert: (1) No JavaScript errors via page.on('pageerror'). (2) Graceful fallback to defaults. (3) User message if needed. This tests defensive programming in the app.
Q5. What is the difference between localStorage and sessionStorage in Playwright testing?
A: Both are accessed identically via page.evaluate(). The behavioral difference: localStorage persists across page.reload(), page.goto(), and context restores via storageState. sessionStorage is tab-scoped — cleared when the browser tab closes. sessionStorage is NOT included in storageState snapshots because it's inherently non-persistent.

Overview — Test Runner & Its Concepts in Playwright

The Playwright Test Runner (@playwright/test) is a full-featured test framework built specifically for browser automation. It manages test discovery, parallel execution, fixture injection, retries, reporters, and tracing — all configured through playwright.config.ts.

What / Why / When / Where / How

WhatThe test runner engine that discovers test files (*.spec.ts), builds a test plan, distributes tests across workers, injects fixtures, captures artifacts, and generates reports
WhyUnderstanding the runner's internals — worker architecture, fixture lifecycle, retry mechanism, project system — lets you optimize suite speed, debug flaky tests, and configure CI correctly
WhenWhen configuring test infrastructure, optimizing CI pipeline speed, debugging flaky tests, setting up cross-browser testing, and managing test parallelism
WhereIn playwright.config.ts for global configuration, in globalSetup/globalTeardown files for one-time operations, and in test files for lifecycle hooks
Hownpx playwright test — Workers (parallel processes), Projects (browser configs), Fixtures (DI system), Reporters (output formats), Retries (auto re-runs), Sharding (CI splitting)

Keyword Glossary

The official Playwright test runner package. Exports: test, expect, defineConfig, devices, and all fixture types.
playwright.config.ts
Central config file exported from defineConfig(). Controls: testDir, timeout, retries, workers, reporter, use (global browser options), projects.
Worker
An isolated Node.js process that runs a subset of tests. Each worker has its own browser instance. Workers run in parallel.
Project
A named configuration targeting a specific browser/device/environment. Each project runs the full test suite independently.

Config flag: when true, ALL tests across ALL files run in parallel. When false (default), tests within a file run serially.

Config: number of times to re-run a failed test before marking it failed. Set to 0 locally, 2 in CI.

Path to a TypeScript file exporting an async default function. Runs ONCE before any worker starts. No fixtures available.
globalTeardown
Path to TS file with async default function. Runs ONCE after all workers finish. Use for cleanup, final report generation.
reporter
Output format: 'html' (rich interactive), 'dot' (CI dots), 'json', 'junit' (XML for CI), 'blob' (for shard merging).
shard
Split the test suite across multiple machines: --shard=1/4 runs the first 25% of tests. Used with blob reporter to merge results.
timeout
Per-test timeout in ms (default 30s). actionTimeout: per-action timeout. navigationTimeout: for page.goto().
testMatch / testDir
testDir: directory to search for tests. testMatch: glob patterns for test files (default: **/*.spec.ts).

Interview One-Liners

A Playwright worker is an isolated Node.js process — not just an isolated test. It has its own browser instance and runs multiple tests serially within that browser.
By default, Playwright runs different spec files in parallel but tests within a file serially. Setting fullyParallel:true makes every individual test run in parallel.
globalSetup runs once before any worker starts — it's truly global. beforeAll runs once per worker, so with 4 workers it runs 4 times.
Playwright Projects let you run the exact same test suite against Chromium, Firefox, WebKit, and mobile devices by adding entries to the projects array.
Set retries:2 in CI and retries:0 locally. In CI, transient infrastructure failures cause legitimate retries. Locally, retries mask real bugs.
Sharding splits the test plan across multiple machines without any code changes — --shard=2/4 deterministically runs the second 25% of tests.

import { chromium, FullConfig, request } from '@playwright/test';
import fs from 'fs';

// PURPOSE: globalSetup runs ONCE before workers; globalTeardown after

export default async function globalSetup(config: FullConfig) {
  console.log('globalSetup: starting — runs ONCE before any worker');

  // 1. API health check
  const apiCtx = await request.newContext({ baseURL: 'https://myapp.com' });
  const health = await apiCtx.get('/api/health');
  if (!health.ok()) throw new Error(`API down: ${health.status()}`);
  await apiCtx.dispose();

  // 2. Browser login → save storageState
  fs.mkdirSync('playwright/.auth', { recursive: true });
  const browser = await chromium.launch();

  for (const { email, password, file } of [
    { email: 'admin@test.com', password: 'Admin123!', file: 'admin.json' },
    { email: 'buyer@test.com', password: 'Buyer123!', file: 'buyer.json' },
  ]) {
    const ctx  = await browser.newContext();
    const page = await ctx.newPage();
    await page.goto('https://myapp.com/login');
    await page.fill('#email', email);
    await page.fill('#password', password);
    await page.click('button[type="submit"]');
    await page.waitForURL('**/dashboard');
    await ctx.storageState({ path: `playwright/.auth/${file}` });
    await ctx.close();
  }
  await browser.close();
}

export async function globalTeardown() {
  const apiCtx = await request.newContext({ baseURL: 'https://myapp.com' });
  await apiCtx.delete('/api/test-data/cleanup');
  await apiCtx.dispose();
}

Interview Prep — 5 SDET Q&A

Q1. Explain the Playwright worker architecture and how parallel execution works.
A: Playwright uses a multi-process architecture. The main process reads playwright.config.ts, collects all tests, builds a test plan, and spawns N worker processes. Each worker is an independent Node.js process that: (1) Launches its own browser instance. (2) Receives its subset of tests. (3) Runs each test sequentially within its browser. (4) Reports results back. Workers share no memory, no browser state, and no fixtures. With fullyParallel:true, individual tests are distributed across workers for maximum parallelism.
Q2. What is the difference between globalSetup, beforeAll, and beforeEach?
A: globalSetup: runs ONCE before any worker starts — plain Node.js, no fixtures. Use for login once and save storageState. beforeAll: runs once per WORKER — with 4 workers, it runs 4 times. Has access to fixtures. Use for seeding data specific to a test group. beforeEach: runs before every individual test. Has access to all fixtures. Use for navigation to starting URL, creating per-test data.
Q3. How do you configure Playwright for multiple browsers in one CI run?
A: Use the projects array in playwright.config.ts: projects: [{ name:'chromium', use:{...devices['Desktop Chrome']} }, { name:'firefox', use:{...devices['Desktop Firefox']} }, { name:'webkit', use:{...devices['Desktop Safari']} }]. Each project runs the complete test suite with its browser configuration. For auth with project dependencies, create a setup project that runs globalSetup login, then all browser projects declare dependencies.
Q4. How does test sharding work and how do you merge reports from multiple shards?
A: Sharding splits the test plan across machines. --shard=1/4 runs the first 25% of tests deterministically. Each machine uses blob reporter which saves a binary report file. A final merge job downloads all blobs and runs: npx playwright merge-reports --reporter html ./all-blobs, producing a single unified HTML report showing all test results even though they ran on different machines.
Q5. What is the fixture DI system and how does it differ from beforeEach?
A: Playwright's fixture system is a dependency injection container. Fixtures are declared as async generator functions in test.extend(). Key differences from beforeEach: (1) Composability — fixtures can depend on other fixtures. (2) Laziness — unused fixtures are never instantiated. (3) Cross-file sharing — fixtures are importable from any test file. (4) Scope control — fixtures can be 'worker' scope or 'test' scope. (5) Type safety — TypeScript knows the fixture types.
