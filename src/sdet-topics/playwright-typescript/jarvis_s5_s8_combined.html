Session 5 · Navigation

Navigation in Playwright means programmatically moving a browser page between URLs — forward, back, reload, or to a new URL — and waiting for it to reach a specific load state before your test proceeds.

What / Why / When / Where / How

What page.goto(url) loads a URL. page.goBack()/page.goForward() use the browser history. page.reload() refreshes. Each returns a Response or null.
Why Tests must control page state precisely. Navigation APIs let you set up starting conditions, simulate user history traversal, and verify redirect behavior.
When Every test starts with goto(). Use waitForURL() after actions that redirect (login, form submit). Use waitForLoadState() before interacting with dynamic content.
How Playwright's goto() auto-waits for 'load' by default. Pass {waitUntil} to change: 'domcontentloaded', 'networkidle', or 'commit'.

Keyword Glossary

page.goto(url)
Navigates to the URL. Returns Response. Waits for 'load' by default. Options: waitUntil, timeout, referer.
page.goBack()
Navigates back in browser history. Returns Response or null (null if no history). Accepts same waitUntil options as goto.
page.goForward()
Navigates forward in browser history. Returns Response or null. Only works if goBack() was previously called.
page.reload()
Reloads the current page. Useful after server-side changes or to test cache-busting behavior. Accepts waitUntil.
page.waitForURL()
Waits until page.url() matches a string, regex, or predicate function. Used after redirects (login → dashboard). Accepts timeout.
page.waitForLoadState()
Waits for the page to reach: 'load' (all resources), 'domcontentloaded' (HTML parsed), or 'networkidle' (no requests for 500ms).
waitUntil
Option on goto/goBack/goForward/reload. Values: 'load' (default), 'domcontentloaded' (fast), 'networkidle' (dynamic apps), 'commit' (fastest — response headers received).
networkidle
Load state where no network connections for ≥500ms. Best for SPAs that load data after initial HTML. Slightly slower but most reliable for dynamic content.
page.url()
Returns the current page URL string synchronously. Use inside waitForURL predicate or assertions.
page.title()
Async. Returns document.title string. Use with expect(page).toHaveTitle() for navigation assertions.

Interview One-Liners

waitUntil"page.goto() defaults to waitUntil:'load', but for SPAs that fetch data after HTML, I use 'networkidle' — it waits until no network requests fire for 500ms, ensuring the dynamic content is fully loaded before my locators run."
waitForURL"After submitting a login form, I use await page.waitForURL('**/dashboard') instead of a hard sleep — it resolves the moment the URL matches, making the test as fast as the app allows while still being reliable."
goBack/Forward"page.goBack() and goForward() return null if there's no history to navigate to — I always null-check the response or use waitForLoadState() after calling them, because the navigation may be a no-op."
Response"goto() returns a Response object — I can check response.status() to assert the page loaded with a 200 and not a 404 or 500, which lets me catch broken routes in my tests without needing a separate API call."
Hash routing"For single-page apps using hash routing (/#/products), waitForURL(/\#\/products/) uses a regex to match the hash fragment — a plain string match won't work because URL patterns use glob syntax that doesn't cover # by default."
Redirect chain"To test a redirect chain (e.g. /old-page → /new-page → /dashboard), I use page.waitForURL('**/dashboard') and then assert page.url() contains the final destination — Playwright follows all redirects automatically inside goto()."

Flows & Core Concepts

Load States — Which to Choose

Choosing the wrong load state causes either slow tests (networkidle when not needed) or flaky tests (domcontentloaded when JS hasn't run yet).

commit
Response headers received (fastest)
domcontentloaded
HTML parsed, inline scripts run
load
Images, CSS, JS loaded (default)
networkidle
0 network requests for ≥500ms (SPA safe)

waitForURL Patterns

Use dynamic waits after clicks or actions that trigger routing changes.

String Glob
page.waitForURL('**/dashboard')
Regex
page.waitForURL(/\/order\/\d+/)
Predicate
waitForURL(u => u.includes('success'))
Avoid
page.waitForTimeout(2000) (brittle)

import { test, expect } from '@playwright/test';

test('basic page navigation with response check', async ({ page }) => {

  // goto() returns a Response object — check status
  const response = await page.goto('https://myapp.com');

  await expect(page).toHaveURL(/dashboard/);

  // Reload — e.g. to test persistence after save
  await page.reload({ waitUntil: 'domcontentloaded' });
  await expect(page.locator('.dashboard-widget')).toBeVisible();
});

loadstates.ts 
// loadstates.ts — choosing the right waitUntil / loadState

import { test, expect } from '@playwright/test';

test('load state selection guide', async ({ page }) => {

  // 'commit' — fastest: response headers received, content streaming
  // Use: checking status codes, starting to intercept early
  await page.goto('https://myapp.com', { waitUntil: 'commit' });

  // 'domcontentloaded' — HTML parsed, scripts run, no images/fonts waited
  // Use: server-rendered pages, fast assertions on HTML content
  await page.goto('https://myapp.com/about', { waitUntil: 'domcontentloaded' });
  await expect(page.locator('h1')).toContainText('About');

  // 'load' — all resources loaded (default). Use for most standard pages
  await page.goto('https://myapp.com/products'); // waitUntil:'load' default

  // 'networkidle' — no requests for 500ms. Use for SPAs with async data
  await page.goto('https://myapp.com/analytics', { waitUntil: 'networkidle' });

  // waitForLoadState() — wait for a state AFTER already being on a page
  await page.click('#load-more-btn'); // triggers more network requests
  await page.waitForLoadState('networkidle'); // wait for them to settle
  await expect(page.locator('.product-card')).toHaveCount(20);
});

test('browser back/forward navigation', async ({ page }) => {

  // Build history: navigate through 3 pages
  await page.goto('https://myapp.com/');          // page 1

  await page.goBack();

  await page.goBack();
  await expect(page).toHaveURL(/myapp\.com\/$/);

  const resp = await page.goBack();

  // Go forward back to products
  await page.goForward();
  await expect(page).toHaveURL(/products/);
});
redirect-chain.ts 
// redirect-chain.ts — test 301/302 redirect chains

import { test, expect } from '@playwright/test';

test('captures full redirect chain', async ({ page }) => {

  const redirects: { url: string; status: number }[] = [];

  // Capture every response including redirects
  page.on('response', (resp) => {
    if ([301, 302, 307, 308].includes(resp.status())) {
      redirects.push({ url: resp.url(), status: resp.status() });
    }
  });

  // goto() follows ALL redirects automatically
  const finalResponse = await page.goto('https://myapp.com/old-path');

  // Final response is the last page (after all redirects)
  expect(finalResponse?.status()).toBe(200);
  await expect(page).toHaveURL(/new-path/);

  // Verify redirect happened (at least one 301/302 in chain)
  expect(redirects.length).toBeGreaterThan(0);
  console.log('Redirect chain:', redirects);
});

// spa-navigation.ts — handle SPA route changes without full page load

import { test, expect } from '@playwright/test';

test('SPA client-side navigation', async ({ page }) => {

  await page.goto('https://myapp.com', { waitUntil: 'networkidle' });

  // Click React Router / Next.js link — no full page load, URL changes
  await page.click('a[href="/products"]');

  // waitForURL catches client-side history.pushState() changes
  await page.waitForURL('**/products');

  // Wait for the new view to render (SPA re-renders component)
  await expect(page.locator('h1')).toContainText('Products');

  // Hash routing: /#/cart
  await page.click('a[href="#/cart"]');
  await page.waitForURL(/\#\/cart/); // regex for hash routes

  // When URL doesn't change (tab switch, modal open):
  // wait for the DOM change instead
  await page.click('#details-tab');
  await expect(page.locator('.details-panel')).toBeVisible();
});

Best Practices

  • ✓
    Never use page.waitForTimeout() for navigation — always use waitForURL() or waitForLoadState(). Hard sleeps make tests slow and brittle.
  • ✓
    Use waitUntil:'networkidle' for SPAs that load data via fetch/XHR after initial HTML. Use 'domcontentloaded' for server-rendered pages to save time.
  • ✓
    Assert response.status() from goto() for critical pages — ensures you're not testing on a 500 error page silently.
  • ✓
    Use page.waitForURL() with a timeout parameter for slow redirects — the default 30s timeout is often too long for CI; set a tighter expected timeout.
  • ✓
    For browser history testing, always navigate to at least 2 pages before calling goBack() — a fresh context has no history and goBack() returns null.

Interview Prep — Q&A

Q1: What is the difference between waitForURL and waitForLoadState? ▼
waitForURL waits for the URL to change to a specific pattern — it's about WHERE the page is. It resolves as soon as page.url() matches the pattern, regardless of whether the new page has loaded. waitForLoadState waits for HOW MUCH of the page has loaded — 'load', 'domcontentloaded', or 'networkidle'. In practice, I use both in sequence after a redirect: await page.waitForURL('**/dashboard') confirms we're on the right route, then await page.waitForLoadState('networkidle') ensures the dashboard data has loaded before I start asserting on content.
Q2: Why does goto() sometimes fail with a timeout in CI but not locally? ▼
The most common cause is using waitUntil:'networkidle' (or default 'load') on pages that keep long-polling connections open — the browser is never truly idle in CI where the app is running on a slow container. The solution is to either use 'domcontentloaded' and rely on explicit element assertions, or to intercept/abort the long-polling routes using page.route() so networkidle can resolve. Also check that the timeout is generous enough for CI's slower network — set timeout: 60000 for pages that load third-party resources.
Q3: How do you test a multi-step redirect chain? ▼
goto() follows all redirects automatically and returns the final Response. I call: const response = await page.goto('/old-url'). Then assert the final URL: expect(page.url()).toContain('/final-destination'). For asserting intermediate redirects happened, I use page.on('response', r => responses.push(r.url())) before goto() — this captures every response in the chain including 301/302 redirects. Then I can assert the redirect chain contained all expected hops.
Q4: When would you use page.reload() in a test? ▼
Three main scenarios: (1) Testing persistence — after saving data, reload to verify it was persisted to the server, not just held in front-end state. (2) Testing cache behavior — reload after a CDN invalidation to verify the new content appears. (3) Testing session recovery — reload to simulate a user returning to a form mid-completion, verifying the app restores their draft. I always pair reload() with waitForLoadState('networkidle') so the test doesn't proceed before the reloaded content finishes rendering.
Q5: How do you handle navigation in SPAs that don't change the URL? ▼
Some SPAs use state-based routing without URL changes (modals, tabs, wizard steps). In these cases, I can't use waitForURL. Instead, I use: (1) await expect(page.locator('.step-2-content')).toBeVisible() to wait for the new view to appear, (2) page.waitForResponse() to wait for the API call the navigation triggers, or (3) page.waitForFunction(() => window.appState.currentStep === 2) to poll JavaScript state. The key principle: identify what changes when the "navigation" happens — DOM, network, or JS state — and wait for that specific signal.

Session 6 · Locators

A Locator in Playwright is a lazy, auto-retrying reference to one or more DOM elements. Unlike CSS selectors that find elements once, locators re-query the DOM on every interaction, making tests resilient to dynamic content changes.

What / Why / When / Where / How

What Locators wrap a selector strategy — CSS, XPath, ARIA role, text, label, placeholder, test-id. They are lazy (not resolved until used) and retry automatically until the timeout.
Why Hard-coded CSS selectors break when classes change. Playwright's semantic locators (getByRole, getByLabel) target elements the way users see them — by meaning, not implementation detail.
When Always. Every interaction in Playwright goes through a Locator. The question is which strategy — prefer getByRole → getByLabel → getByText → getByTestId → CSS in that order of preference.
How Use the semantic locator methods directly: page.getByRole('button', {name: 'Submit'}). Scope search using parent chaining, or filter by text/child elements using .filter().

Keyword Glossary

getByRole()
Locates by ARIA role. Most resilient — survives class/structure changes. Options: name (accessible name), exact (boolean). e.g. getByRole('button', {name:'Submit'})
getByLabel()
Finds form input associated with a <label> element by label text. Best for form fields. Works with for/id, aria-labelledby, and wrapping label.
getByText()
Finds element by visible text content. Exact match by default. Use exact:false for substring. Good for paragraphs, headings, list items.
getByPlaceholder()
Finds input/textarea by placeholder attribute text. Use when no label is present.
getByTestId()
Finds by data-testid attribute (configurable). Most explicit — add data-testid attributes to your app for stable test hooks that survive UI refactors.
getByAltText()
Finds img, area, or input elements by their alt attribute. Best for image assertions and icon buttons with meaningful alt text.
locator()
CSS selector or XPath locator. Use xpath= prefix for XPath. Fallback when no semantic locator is available. Accepts :has(), :not(), :nth-child() pseudo-selectors.
.filter()
Filters a locator result by text or by having another locator inside it. e.g. page.locator('.row').filter({hasText:'Alice'}) finds only rows containing 'Alice'.
.nth()
Selects the Nth element (0-indexed) from a set of matching locators. e.g. locator('.card').nth(2) selects the third card.
.first() / .last()
Shortcuts for .nth(0) and .nth(-1). Cleaner for first/last element selection from a list.
.and()
Combines two locators with AND logic. e.g. page.getByRole('button').and(page.getByText('Submit')) — must match both conditions.
.or()
Combines locators with OR logic. e.g. page.getByRole('button').or(page.getByRole('link')) — useful when an element can be either type.

Interview One-Liners

Resilience"I always prefer getByRole over CSS selectors — a button's class might change during refactoring, but its ARIA role and accessible name stay stable, making getByRole('button', {name:'Add to Cart'}) far more resilient."
Auto-retry"Playwright locators are lazy and auto-retrying — every action (click, fill, check) retries finding the element until the default timeout. This means I don't need explicit waits before most interactions."
filter()"The filter() method is powerful for lists — page.locator('tr').filter({hasText: 'Alice'}) finds the row containing Alice without needing complex XPath. I can chain multiple filters for multi-condition matching."
getByTestId"For complex UI elements where no semantic locator works cleanly, I add data-testid='checkout-btn' to the element in the source code — it's a contract between developers and QA that survives all CSS/structure changes."
Chaining"Locators can be scoped to a parent — page.locator('.product-card').getByRole('button', {name:'Buy'}) first narrows to product cards, then finds the Buy button inside, avoiding false matches from other buttons on the page."
Strict mode"Playwright locators are strict by default — if multiple elements match, the action throws a strict mode violation. This is intentional: it forces me to be more specific rather than silently acting on the wrong element."

// and-or-all.ts — .and(), .or(), .all(), .count()

import { test, expect } from '@playwright/test';

test('advanced locator composition', async ({ page }) => {

  await page.goto('https://myapp.com');

  // .and() — element must match BOTH locators
  const disabledBtn = page
    .getByRole('button')
    .and(page.locator('[disabled]'));

    .getByRole('button', { name: 'Login' })

  // .count() — how many elements match
  const count = await page.locator('.product-card').count();
  console.log(`Found ${count} products`);

  // .all() — iterate over all matching elements
  const cards = await page.locator('.product-card').all();
  for (const card of cards) {
    const price = await card.locator('.price').textContent();
    console.log(`Card price: ${price}`);
  }
});

Best Practices

  • ✓
    Prefer semantic locators (getByRole, getByLabel, getByText) over CSS selectors. They reflect user intent, survive refactors, and are self-documenting.
  • ✓
    Always scope locators to their nearest unique parent to prevent strict mode violations. page.locator('.user-table').getByRole('row') is better than page.getByRole('row').
  • ✓
    Use filter({hasText}) for list items — it's readable, chainable, and far cleaner than nth() when the position of an item can change.
  • ✓
    Add data-testid attributes to critical UI elements in collaboration with developers. This creates stable test anchors that survive all UI refactors.
  • ✓
    Never store a locator in a variable and reuse it across page navigations — locators are tied to the current DOM state. Create a new locator after navigation.

Interview Prep — Q&A

Q1: Why is getByRole preferred over CSS selectors? ▼
getByRole targets elements by their semantic ARIA role and accessible name — the same way screen readers and users experience the page. A button's CSS class might change from .btn-primary to .cta-button during a design refactor, but its role ('button') and accessible name ('Submit Order') don't change. This makes getByRole vastly more resilient to UI changes. Additionally, it doubles as an accessibility check — if getByRole can't find the element, it means the element lacks proper ARIA semantics, which is itself a bug worth fixing.
Q2: What is a strict mode violation and how do you fix it? ▼
A strict mode violation occurs when a locator matches more than one element and you try to perform a single-element action (click, fill) on it. Playwright throws: "strict mode violation: locator('button') resolved to 2 elements". Fixes: (1) Be more specific: getByRole('button', {name:'Submit'}) instead of getByRole('button'). (2) Scope to parent: page.locator('#checkout-form').getByRole('button'). (3) Use filter(): page.getByRole('button').filter({hasText:'Pay'}). (4) If you genuinely need all matches, use .all() to get an array and iterate.
Q3: How does locator chaining work for nested elements? ▼
Locator chaining narrows the DOM scope step by step. When you write page.locator('.card').getByRole('button'), Playwright first finds all elements matching .card, then searches for role='button' WITHIN each of those elements. This prevents false matches from buttons elsewhere on the page. You can chain multiple levels: page.locator('.user-table').locator('tbody').locator('tr').filter({hasText:'Alice'}).getByRole('button', {name:'Edit'}) — each step narrows to only elements inside the previous match. Under the hood, this generates a CSS descendant selector, but it reads as clear English logic.
Q4: How do you select the 3rd row in a dynamic table? ▼
Two approaches depending on what I need: (1) By position: page.locator('table tbody tr').nth(2) — selects the 3rd row (0-indexed). (2) By content — better for dynamic tables where row positions can change: page.locator('table tbody tr').filter({hasText: 'Alice Johnson'}) — finds the row containing Alice regardless of its position. The content-based approach is more resilient because sorted/filtered tables shuffle row order. If I need to assert all rows, I use: const rows = await page.locator('tbody tr').all() which returns an array of Locators for each row.
Q5: What is the difference between locator.all() and locator.count()? ▼
locator.count() returns a number — the total count of matching elements. It's used for assertions: await expect(page.locator('.item')).toHaveCount(5). locator.all() returns an array of individual Locator objects, one per matching element. It's used when you need to iterate and interact with each element individually: const items = await page.locator('.item').all(); for (const item of items) { await item.click(); }. Important: .all() is only safe to call after the elements are fully rendered — it doesn't retry like other locator actions. For count assertions, always use expect().toHaveCount() with auto-retry rather than raw .count().

Session 7 · Actions

Actions in Playwright are the methods that interact with the browser — clicking, typing, hovering, selecting, dragging, uploading files. Every action auto-waits for the element to be visible, enabled, stable, and receive events before executing.

What / Why / When / Where / How

What Mouse: click, dblclick, hover, dragAndDrop. Keyboard: fill, type, press, clear. Form: check, uncheck, selectOption. Scroll: scrollIntoViewIfNeeded. Files: setInputFiles.
Why Playwright waits for 5 conditions before acting: visible, attached to DOM, stable (not animating), receives pointer events, enabled. This eliminates most explicit waits needed in Selenium.
When Use fill() for fast typing, and type() character-by-character for searching autocomplete. Use check() for checkboxes to guarantee states.
How Call the action method directly on your locator: await page.getByRole('button').click(). Customize actions using options like {force: true} or keyboard modifiers.

Keyword Glossary

click()
Clicks the element. Options: button, clickCount, modifiers, position, force, noWaitAfter, timeout. Auto-waits for actionability checks.
dblclick()
Double-clicks. Fires two click events. Use for editable cells, file rename, text selection.
fill()
Clears the field and sets value in one operation. Fastest way to fill inputs. Triggers input and change events. Use for most form inputs.
type()
Simulates real keyboard typing character by character. Triggers keydown, keypress, keyup, input per character. Use for autocomplete/search-as-you-type fields.
clear()
Clears the value of an input field. Equivalent to fill(''). Use before re-filling when you need to clear first.
press()
Presses a key or key combination. e.g. press('Enter'), press('Tab'), press('Control+A'), press('ArrowDown'). Can also be called on page directly.
hover()
Moves mouse over element. Triggers mouseover and mouseenter events. Use before clicking tooltips, dropdown menus, or hover-revealed buttons.
check() / uncheck()
Checks/unchecks a checkbox or radio. Auto-waits until the input is not already in the target state. Use instead of click() for checkboxes — it's idempotent.
selectOption()
Selects an option in a <select> dropdown. Accepts value, label, or index. Can select multiple options for multi-select.
dragAndDrop()
Drags source element and drops on target. page.dragAndDrop(source, target) or locator.dragTo(target). Simulates full mouse down/move/up sequence.
setInputFiles()
Sets files for a file input element. Accepts path string, array of paths, or file-like objects. Bypasses the native file picker dialog entirely.
scrollIntoViewIfNeeded()
Scrolls the page until the element is in the viewport. Automatically called by most actions, but can be called explicitly to make elements visible.

Interview Prep — Q&A

Q1: What are Playwright's actionability checks and why do they matter? ▼
Before performing any action (click, fill, hover), Playwright verifies five conditions: (1) Attached — element exists in the DOM. (2) Visible — not hidden via CSS (display:none, visibility:hidden, opacity:0, or zero dimensions). (3) Stable — not currently animating or moving (waits for CSS transitions to complete). (4) Receives pointer events — not covered by another element like an overlay or spinner. (5) Enabled — not disabled attribute. These checks retry automatically until the timeout. This eliminates the most common class of test failures: clicking a button that's mid-animation, clicking behind a loading overlay, or filling a disabled field.
Q2: When would you use type() instead of fill()? ▼
fill() sets the value of an input atomically — it clears and sets in one operation, fast. type() simulates keystroke-by-keystroke input, firing keydown, keypress, keyup, and input events for each character, which is slower. I use type() (or the newer pressSequentially()) when the field has JavaScript that listens to individual keystroke events — like a live search/autocomplete that calls an API on each key press, a character counter that updates per keystroke, or a rich text editor that transforms text as you type. For standard form inputs like email, password, or phone number, fill() is always preferred.
Q3: How do you test a file upload in Playwright? ▼
Playwright's setInputFiles() bypasses the native OS file picker dialog and directly sets files on the input element. For a standard <input type="file">: await page.locator('input[type="file"]').setInputFiles('tests/fixtures/test-image.png'). For multiple files: setInputFiles(['file1.pdf', 'file2.pdf']). For a custom upload button that triggers the file input via JS, I first listen for the file chooser: const [fileChooser] = await Promise.all([page.waitForEvent('filechooser'), page.click('#upload-btn')]); then: await fileChooser.setFiles('test-file.pdf'). After upload, I assert the file name appears in the preview and the upload progress/success indicator shows.
Q4: How do you test drag and drop in Playwright? ▼
Playwright has two drag APIs. High-level: await page.dragAndDrop('.drag-item', '.drop-target') — recommended for most cases. Locator-level: await page.locator('.drag-item').dragTo(page.locator('.drop-target')). For precise position control: dragTo(target, { targetPosition: {x:10, y:10} }) drops at a specific coordinate within the target. For complex custom DnD libraries (React DnD, SortableJS): sometimes I need the low-level approach — mouse down, move with steps, and mouse up. The steps parameter makes the move gradual, which some DnD implementations require to register drag intent.
Q5: How do you click a button that is only visible on hover? ▼
I use hover() first to reveal the button, then click it: await page.locator('.row').hover() — this triggers CSS :hover and shows the hidden action buttons. Then await page.locator('.row').getByRole('button', {name:'Delete'}).click(). If the button disappears when I move the mouse to click it (it's on a different element from what I hovered), I use the force option: .click({force:true}). Alternatively, I trigger the hover and then check if the button has transitioned to visible: await expect(page.locator('.row .delete-btn')).toBeVisible() before clicking.

Session 8 · Assertions

Assertions in Playwright use the expect() function from @playwright/test. They are auto-retrying — Playwright polls the condition repeatedly (every ~100ms) until it passes or the timeout expires, making tests resilient to asynchronous DOM updates.

What / Why / When / Where / How

What Two categories: Locator assertions (expect(locator).toBeVisible()) — auto-retrying, async. General assertions (expect(value).toBe()) — synchronous, no retry. Always use locator assertions for DOM checks.
Why DOM updates are async. Without retry, asserting immediately after a click would fail because React/Vue hasn't re-rendered yet. Playwright's expect() retries the check every ~100ms until it passes or times out.
When Always use locator assertions for asserting states like visible, hidden, checked, enabled, or text content. Use general assertions for API values and JSON files.
How Write assertions programmatically: await expect(page.locator('.cart-count')).toHaveText('1'). To continue execution after a check fails, use expect.soft().

Keyword Glossary

toBeVisible()
Asserts element is visible — attached to DOM and has visible CSS (not hidden/display:none). Retries until visible or timeout.
toBeHidden()
Asserts element is not visible or not attached. Opposite of toBeVisible(). Use to verify elements disappear after dismiss/close.
toHaveText()
Asserts element's text content equals the expected string. Accepts string, regex. Trims whitespace. Use when you need exact match.
toContainText()
Asserts element's text contains the expected substring. More lenient than toHaveText(). Use when surrounding text might change.
toHaveValue()
Asserts an input/select/textarea has a specific value. For inputs: checks the .value property. For checkboxes: use toBeChecked() instead.
toBeChecked()
Asserts checkbox or radio is checked. Pass {checked:false} for the inverse. More semantic than toHaveValue('true').
toBeEnabled() / toBeDisabled()
Asserts element is interactable (not disabled attribute). toBeDisabled() is the inverse. Use before form submission checks.
toHaveURL()
Asserts page.url() matches string glob, regex, or equals the expected URL. Called on page: expect(page).toHaveURL(). Retries.
toHaveTitle()
Asserts document.title matches string or regex. Called on page: expect(page).toHaveTitle(/Dashboard/).
toHaveCount()
Asserts a locator matches exactly N elements. Retries until count matches or timeout. Use for list length assertions.
toHaveAttribute()
Asserts element has an HTML attribute with a specific value. e.g. expect(link).toHaveAttribute('href', '/dashboard').
expect.soft()
Non-throwing assertion — records the failure but continues test execution. All soft failures are reported at the end. Useful for batch-checking multiple fields.

Interview One-Liners

Auto-retry"Playwright's expect() assertions auto-retry until the condition passes or the timeout expires — this is why I never need a waitForSelector before an expect().toBeVisible(), because the assertion itself handles the wait."
Locator vs Value"I distinguish between locator assertions (expect(locator).toBeVisible()) which are async and retry, and value assertions (expect(string).toBe()) which are synchronous snapshots. Using expect(await locator.textContent()).toBe() is wrong — it captures the value once with no retry and fails on async updates."
Soft assertions"I use expect.soft() for form validation tests where I want to verify ALL field errors in one test run — if I used regular expect(), the test stops at the first failed field and I miss all subsequent validation failures."
Custom message"I always add a descriptive message to complex assertions: expect(locator, 'Price should update after coupon applied').toHaveText('₹450'). This makes the failure report self-explanatory without needing to read the test code."
Negative assertions"For negative assertions I use .not: expect(locator).not.toBeVisible() rather than manually checking if the element is absent. The .not modifier also benefits from auto-retry — it waits until the element disappears."
toHaveText vs toContainText"toHaveText() does an exact text match (after trimming) — it fails if the element has any extra text. toContainText() checks if the element's text includes the expected string as a substring — safer when surrounding text can vary."

Flows & Core Concepts

Locator Assertions vs Value Assertions

Always prefer locator assertions for DOM nodes to leverage auto-retry.

✅ Locator Assertion
await expect(locator).toHaveText('Saved')
Retries until text matches or timeout
❌ Value Assertion
expect(await locator.textContent()).toBe('Saved')
Reads once. Fails if dynamic updates are pending

Soft Assertions Pattern

Soft assertions let you run all assertions and fail the test at the very end.

Assertion 1 (Soft)
expect.soft(l1).toBeVisible()
(Fails, but continues)
Assertion 2 (Soft)
expect.soft(l2).toHaveValue('x')
(Fails, but continues)
Finish Test
Fails test showing BOTH recorded errors

Best Practices

  • ✓
    Always use locator assertions (expect(locator).toXxx()) for DOM checks — never expect(await locator.textContent()).toBe(). Locator assertions retry; value snapshots don't.

Interview Prep — Q&A

Q1: Why should you use expect(locator).toHaveText() instead of expect(await locator.textContent()).toBe()? ▼
The key difference is retry behavior. expect(locator).toHaveText('Saved') is an auto-retrying assertion — Playwright polls the DOM every ~100ms and retries the check until the text equals 'Saved' or the timeout expires. This handles async DOM updates seamlessly. The anti-pattern expect(await locator.textContent()).toBe('Saved') reads the text once at the exact moment that line runs — if the DOM update hasn't happened yet (React re-render, API response pending), it reads the OLD value and fails. Always use the locator assertion form for any value that might change asynchronously.
Q2: When would you use soft assertions? Give a real example. ▼
Soft assertions are ideal when you want to check multiple independent conditions in one test without stopping at the first failure. Real example: validating a user profile form has all required fields correctly pre-populated. expect.soft(page.getByLabel('Name'), 'Name should be pre-filled').toHaveValue('Alice'); expect.soft(page.getByLabel('Email'), 'Email should be pre-filled').toHaveValue('alice@example.com');. If Name is correct but Email is wrong, a regular expect would stop and I'd never know subsequent field states. With soft assertions, all failures are collected and reported together.
Q3: How do you assert that an element is NOT present on the page? ▼
Three approaches: (1) expect(locator).toBeHidden() — asserts the element exists in the DOM but is CSS-hidden (display:none, visibility:hidden). (2) expect(locator).not.toBeVisible() — same as toBeHidden() essentially. (3) await expect(locator).toHaveCount(0) — asserts zero elements match the selector, meaning the element was removed from the DOM entirely. Important: all three auto-retry — they wait for the element to disappear, not just snapshot the current state. Don't use expect(await locator.count()).toBe(0).
Q4: What is the difference between toHaveText() and toContainText()? ▼
toHaveText() performs a full text match (after whitespace normalization/trimming) — the element's entire text content must equal the expected value. If the element says "3 items in cart" and you assert toHaveText('items in cart'), it fails. toContainText() checks if the full text INCLUDES the expected substring — toContainText('items in cart') on "3 items in cart" passes. I use toHaveText when I know the complete text. I use toContainText when the text has dynamic parts (numbers, timestamps, user names).
Q5: How would you assert a list of items in a specific order? ▼
Use toHaveText() with an array of expected values: await expect(page.locator('ul li')).toHaveText(['First item', 'Second item', 'Third item']). Playwright asserts each array element against each locator match in order. For partial matching in a list: await expect(page.locator('.product-name')).toContainText(['iPhone', 'Samsung', 'OnePlus']).
