Session 1 — Project Setup & Framework Structure
What is Playwright?
Playwright is a Node.js library by Microsoft for end-to-end browser automation. It supports Chromium, Firefox, and WebKit in a single API. TypeScript gives you type safety, autocomplete, and better IDE support.

Page Object Model (POM) — BasePage
// pages/BasePage.ts import { Page, Locator, expect } from '@playwright/test'; export class BasePage { readonly page: Page; constructor(page: Page) { this.page = page; } async navigate(path: string = '') { await this.page.goto(path); } async waitForPageLoad() { await this.page.waitForLoadState('networkidle'); } async getTitle() { return this.page.title(); } async takeScreenshot(name: string) { await this.page.screenshot({ path: `test-results/${name}.png` }); } }
POM — LoginPage.ts
// pages/LoginPage.ts import { Page, Locator, expect } from '@playwright/test'; import { BasePage } from './BasePage'; export class LoginPage extends BasePage { // All locators declared as readonly properties readonly emailInput: Locator; readonly passwordInput: Locator; readonly submitButton: Locator; readonly errorMessage: Locator; readonly forgotPasswordLink: Locator; constructor(page: Page) { super(page); // Best practice: use getByRole and getByLabel first this.emailInput = page.getByLabel('Email'); this.passwordInput = page.getByLabel('Password'); this.submitButton = page.getByRole('button', { name: 'Sign in' }); this.errorMessage = page.getByTestId('error-alert'); this.forgotPasswordLink = page.getByText('Forgot password?'); } async goto() { await this.page.goto('/login'); } async login(email: string, password: string) { await this.emailInput.fill(email); await this.passwordInput.fill(password); await this.submitButton.click(); } async expectError(message: string) { await expect(this.errorMessage).toBeVisible(); await expect(this.errorMessage).toHaveText(message); } async expectRedirectToDashboard() { await expect(this.page).toHaveURL('/dashboard'); } }
Using POM in Tests

Global Setup (authentication)
// global-setup.ts — runs once before the whole suite import { chromium, FullConfig } from '@playwright/test'; async function globalSetup(config: FullConfig) { const browser = await chromium.launch(); const page = await browser.newPage(); // Login and save auth state await page.goto('https://myapp.com/login'); await page.fill('#email', 'admin@test.com'); await page.fill('#password', 'secret'); await page.click('[type="submit"]'); await page.waitForURL('/dashboard'); // Save cookies + localStorage to file await page.context().storageState({ path: 'playwright/.auth/user.json' }); await browser.close(); } export default globalSetup;
Save auth state once in globalSetup, then reuse it with storageState in your config. This avoids logging in before every test.

Q1. What is Playwright and how is it different from Selenium?
Playwright is a modern browser automation framework by Microsoft. Unlike Selenium, it has built-in auto-waiting (no explicit waits needed), supports multiple browsers with one API, can test network requests/responses, has built-in trace viewer, and is much faster. Selenium requires WebDriver protocol; Playwright communicates directly with browser DevTools Protocol.
Q2. What is Page Object Model (POM) and why use it?
POM is a design pattern where each web page is represented as a class with locators as properties and user actions as methods. Benefits: removes duplicate code, centralizes locator maintenance (change in one place), improves readability, and makes tests closer to business language. If a locator changes, you fix it in the page class, not in 50 test files.
Q3. What is a fixture in Playwright?
A fixture is a reusable piece of setup provided to tests via dependency injection. Built-in fixtures include page, browser, context. You can create custom fixtures that extend these — for example, a loginPage fixture that automatically creates a LoginPage object, or an authenticatedPage that pre-logs-in before the test runs. Fixtures handle setup AND teardown automatically.
Q4. What is the difference between page, context, and browser?
Browser is the actual browser process (Chrome, Firefox). Context is like an incognito window — isolated cookies, storage, and sessions. Page is a single tab inside a context. For multi-user scenarios, create two contexts, each with their own page. This lets you test admin and user simultaneously without cookie conflicts.
Q5. How does Playwright handle waiting? What is auto-wait?
Playwright auto-waits before every action. When you call click(), it waits for: element to be attached to DOM, visible, stable (not animating), enabled, and receives events. When you use expect(), it retries the assertion until it passes or times out. This eliminates most explicit waits. Never use waitForTimeout() — it doesn't wait for any real condition.
Q6. What are flaky tests and how do you fix them?
Flaky tests pass sometimes and fail sometimes without code changes. Causes: race conditions, hardcoded waits, shared test data, order-dependent tests, environment differences. Fixes: use proper auto-wait assertions, isolate test data per test, make tests independent, use retries only as safety net not as a fix, use trace viewer to record failures and diagnose root cause.
Q7. How do you handle authentication in Playwright?
Use storageState. In globalSetup, log in once and save cookies/localStorage to a JSON file with page.context().storageState(). In playwright.config.ts, set storageState to that file path in the use block. Every test then starts pre-authenticated. This avoids logging in before every test, making the suite much faster.
Q8. What is the best locator strategy priority?
Priority: 1) getByRole (ARIA semantics, accessibility-friendly) → 2) getByLabel (for form inputs) → 3) getByText (for visible text) → 4) getByTestId (data-testid, most stable for SPAs) → 5) CSS selectors (tag, id, class) → 6) XPath (last resort, avoid). Prefer locators that reflect how users interact with the page, not implementation details like CSS class names.
Q9. What is trace in Playwright and when is it useful?
Trace captures a full recording of test execution: every action, screenshot at each step, network requests, console logs. Set trace: 'on-first-retry' in config. After a failure, run npx playwright show-trace to open an interactive timeline. It's the most powerful debugging tool — lets you see exactly what happened in CI without re-running.
Q10. How do you run tests in parallel and what are workers?
Workers are parallel processes that run tests simultaneously. Set workers: 4 in config to run 4 tests at once, dramatically reducing total runtime. With fullyParallel: true, even tests within the same file run in parallel. For shared state (e.g., same user account), limit workers or use unique data per test. In CI, typically use workers: '50%' to use half the available cores.
Q11. What is the difference between test.skip, test.only, test.fixme?
test.skip() — skips the test, shows as skipped in report. Optionally conditional: test.skip(browserName === 'firefox', 'Not supported'). test.only() — runs ONLY this test (and other .only tests) in the file. Great for debugging. test.fixme() — marks test as expected to fail (known broken). test.slow() — triples the timeout for that test. Never commit test.only() to main branch!
Q12. How do you do API testing in Playwright?
Use the built-in request fixture for pure API tests, or use page.request for browser-context API calls. Example: const response = await request.get('/api/users'); expect(response.ok()).toBeTruthy(); const body = await response.json(); expect(body.users).toHaveLength(10). You can also intercept browser requests with page.route() to mock or modify API responses.
