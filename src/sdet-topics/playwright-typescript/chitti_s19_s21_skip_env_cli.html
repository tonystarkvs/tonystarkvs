Session 19 · test.skip — Complete Guide

test.skip() tells Playwright to completely bypass a test or group without running it. The test is recorded as skipped (not passed, not failed) in reports. It can be used unconditionally, conditionally at runtime, and at different levels — test, describe, or inside a test body. Placement matters critically: inside the test body vs as the first argument changes when and how it is evaluated.

WHAT is test.skip?

A modifier that marks a test as "not to be executed." Playwright skips its setup, execution, and teardown. It shows as skipped (⚠️) in HTML reports — not as a failure. Completely different from test.fail() which still runs the test.

WHY use it?

Skip tests for: unimplemented features, known flaky third-party tests, browser-specific bugs, environment-specific tests (no email server in CI), tests under active development, or temporarily bypassing broken infrastructure.

WHEN and placement?

Top-level (static): test.skip('name', fn) — evaluated at collection time, before any browser starts. Inside body (dynamic): test.skip(condition) — evaluated at runtime, can use fixture values like browserName, page, testInfo.

WHERE it affects

At test level: skips one test. At describe level: test.describe.skip() skips ALL tests in that block. At file level: test.skip() at top of file skips ALL tests. Inside test body: skips from that point (hooks for the test still ran up to that point).

NO→use skip or fixme
YES→use fail (runs test, inverts assertion)

Interview Prep — 5 SDET Q&A
Q1: What is the exact difference between calling test.skip() outside vs inside a test body? ▶
Outside (static): test.skip('my test', async () => {...}) — this is evaluated during Playwright's test collection phase before any browser starts. The test is never added to the execution queue. No hooks run at all — not beforeEach, not afterEach, not the test body. Inside (dynamic): test('my test', async ({browserName}) => { test.skip(browserName==='webkit', 'not supported'); ...}) — this is evaluated at runtime AFTER fixtures are injected, AFTER beforeEach ran, and AFTER entering the test body. When skip fires inside the body: afterEach STILL runs (cleanup must happen), but the remainder of the test body after the skip call does NOT execute. The key interview answer: static = plan time, no hooks run. Dynamic = runtime, beforeEach and afterEach STILL run.
Q2: How does test.skip() affect the CI pipeline exit code? ▶
Skipped tests do NOT cause pipeline failure. The Playwright process exits with code 0 when all non-skipped tests pass, regardless of how many tests are skipped. The summary shows: "10 passed, 2 skipped" — exit code 0. If ANY test fails: exit code 1 — pipeline fails. This is fundamentally different from failing — a skip means a conscious decision was made not to run the test. Important implication: if you accidentally skip tests that should be running (e.g., wrong condition), CI appears green but coverage is reduced. Teams should monitor the skip count in CI metrics and alert when it rises unexpectedly.
Q3: How do you skip tests only on a specific browser? ▶
Use the dynamic form inside the test body with the built-in browserName fixture: test('drag-drop test', async ({page, browserName}) => { test.skip(browserName === 'webkit', 'BUG-789: Drag and drop broken on WebKit 17'); await page.dragAndDrop(...); }). For skipping in a describe block: test.describe('Payment flow', () => { test.beforeEach(async ({browserName}) => { test.skip(browserName === 'firefox', 'Stripe widget not supported on FF'); }); }). The beforeEach skip approach skips all tests in the describe for that browser without touching each test. The browserName values are: 'chromium', 'firefox', 'webkit' — always lowercase. For multiple browsers: test.skip(['webkit','firefox'].includes(browserName), 'Desktop-only feature').
Q4: What is the difference between test.skip() and test.fixme()? ▶
Both skip the test — the test body never executes, both show exit code 0. The difference is semantic/communicative: test.skip() means "we consciously chose not to run this" — reasons: unimplemented feature, missing environment, conditional platform. test.fixme() means "this is broken and needs fixing" — it signals technical debt, a known broken test that someone should address. In HTML reports, fixme tests appear with a distinct fixme icon vs a generic skip icon. Some teams configure CI alerts when fixme count rises. In practice: if a test is skipped because the feature isn't built yet → test.skip(). If a test is skipped because a recent code change broke it and needs a developer to fix it → test.fixme().
Q5: How do you skip all tests in a file without modifying every test? ▶
Three approaches: (1) File-level skip at top of file — call test.skip() with no arguments at the very top (outside any describe): test.skip(). This skips ALL tests in the file. (2) Wrap everything in a skipped describe: test.describe.skip('Entire checkout module', () => { all your tests here }). The describe name appears in reports making it clear what was skipped. (3) Conditional file skip: test.skip(process.env.SKIP_PAYMENT_TESTS === 'true', 'Payment tests disabled via env flag'). The best approach for a temporarily disabled feature is option 2 — the describe name in reports tells teammates exactly which feature is disabled. Option 1 is fine for a file that is completely irrelevant in the current environment.
static-skip-forms.ts
// static-skip-forms.ts
// PURPOSE: All static/unconditional forms of test.skip()
// KEY: Evaluated at PLAN BUILD TIME — hooks never run for these tests

import { test, expect } from '@playwright/test';

// ── FORM 1: Static skip — test body is the 2nd argument ─────────
// test.skip replaces test() — first arg = name, second = async fn
// beforeEach, afterEach do NOT run for this test
test.skip('payment feature not implemented yet', async ({ page }) => {
  // This body is NEVER executed — plan build time skip
  await page.goto('/checkout/payment');
  await expect(page.locator('.payment-form')).toBeVisible();
});

// ── FORM 2: Static skip with options (timeout, retries) ──────────
test.skip('dark mode persistence — pending design',
  { timeout: 60_000 },  // options object is 2nd arg
  async ({ page }) => {
    // Also never runs — static skip with options
    await page.goto('/');
  }
);

// ── FORM 3: File-level skip — skips ALL tests in this file ────────
// Call with NO arguments at the very top of the file (outside describe)
// test.skip(); ← uncomment to skip entire file

// ── Comparison: normal test — runs normally ───────────────────────
test('homepage loads — this runs normally', async ({ page }) => {
  await page.goto('https://myapp.com');
  await expect(page).toHaveTitle(/MyApp/);
});

// Report output:
// ✅ homepage loads — this runs normally        [passed]
// ⚠️ payment feature not implemented yet         [skipped]
// ⚠️ dark mode persistence — pending design      [skipped]
// Summary: 1 passed, 0 failed, 2 skipped  ← exit code 0
dynamic-conditional-skip.ts
// dynamic-conditional-skip.ts
// PURPOSE: Conditional skip INSIDE test body — runtime evaluation
// KEY: beforeEach RUNS before this skip fires (opposite of static skip)

import { test, expect } from '@playwright/test';

// ── PATTERN 1: Simple condition ───────────────────────────────────
test('skip based on boolean condition', async ({ page }) => {

  // Called INSIDE body — runtime evaluation (fixtures available)
  // test.skip(condition) — no reason
  test.skip(process.env.CI === 'true');
  // If CI=true: test stops here. afterEach still runs for cleanup.

  await page.goto('https://myapp.com');
  await expect(page.locator('h1')).toBeVisible();
});

// ── PATTERN 2: With reason message ───────────────────────────────
test('email test — requires local SMTP', async ({ page }) => {

  // test.skip(condition, 'reason') — reason appears in HTML report
  test.skip(
    process.env.CI === 'true',
    'Requires local SMTP server — not available in CI pipeline'
  );

  await page.goto('https://myapp.com/send-email');
  await page.click('#send-test-email');
  await expect(page.locator('.email-sent-success')).toBeVisible();
});

// ── PATTERN 3: Multi-condition skip ──────────────────────────────
test('complex multi-condition skip', async ({ page }, testInfo) => {

  // Combine multiple conditions with logical operators
  const skipConditions = [
    process.env.CI === 'true',            // in CI pipeline
    process.env.TEST_ENV === 'dev',        // on dev environment
    testInfo.project.name === 'mobile',   // mobile project
  ];

  test.skip(
    skipConditions.some(c => c),   // ANY condition = true → skip
    'Skipped: CI or Dev env or mobile project detected'
  );

  await page.goto('https://myapp.com/admin');
  await expect(page.locator('.admin-panel')).toBeVisible();
});

// skip-vs-fixme-fail.ts
// PURPOSE: Side-by-side comparison of skip, fixme, and fail modifiers
// Shows the right modifier for each scenario

import { test, expect } from '@playwright/test';

// ── test.skip() — Feature not built yet / env not available ───────
// Behavior: test body NEVER runs, exit code 0
// Use: conscious choice to not run
test.skip('SKIP: analytics dashboard — feature in sprint 15',
  async ({ page }) => {
  await page.goto('/analytics'); // never reached
});

// ── test.fixme() — Broken test that needs developer attention ─────
// Behavior: identical to skip (body never runs), but shows fixme icon
// Use: signals "someone broke this, needs fixing"
test.fixme('FIXME: dark mode toggle crashes React — regression from PR #234',
  async ({ page }) => {
  await page.click('#dark-toggle'); // never reached
});

// ── test.fail() — Known bug: test RUNS and is expected to fail ─────
// Behavior: test body RUNS, assertion fails → test PASSES (inverted)
// Use: document bugs that exist in production right now
// If bug is fixed: assertion passes → test.fail() makes it FAIL → alert!
test.fail('FAIL: BUG-789 checkout button disabled on iOS Safari 17',
  async ({ page }) => {
  await page.goto('/checkout');
  // This assertion FAILS due to the bug — test.fail() makes the suite pass
  // When bug is fixed: assertion PASSES → test.fail() marks test as FAILED
  // → that's your signal to remove test.fail() annotation!
  await expect(page.getByRole('button', { name: 'Place Order' }))
    .toBeEnabled(); // fails on buggy build → test.fail() = suite passes
});

// ── Correct pairing guide ─────────────────────────────────────────
// QUESTION: Should the test body run?
// NO  → test.skip()  (not built) or test.fixme() (broken infra)
// YES → test.fail()  (known bug, test documents it)
//
// QUESTION: Is this a bug or a missing feature?
// Missing feature → test.skip('feature sprint 15')
// Broken by code change → test.fixme('regression from PR #234')
// Confirmed production bug → test.fail('BUG-123 known issue')

test('normal comparison test — always runs', async ({ page }) => {
  await page.goto('https://myapp.com');
  await expect(page).toHaveTitle(/MyApp/);
});
Session 20 · Environment Handling — Dev / Test / UAT URLs

Environment handling means making your Playwright tests point to the correct base URL (Dev, Test, UAT, Production) based on where they're running — without hardcoding URLs in test files. This is achieved through process.env variables, playwright.config.ts baseURL, and environment-specific config files.

WHAT are environments?

Dev (developer laptop), Test/QA (shared testing server), UAT (User Acceptance Testing — pre-prod), Production. Each has a different base URL. Tests must point to the right one without code changes between runs.

WHY handle environments?

Hardcoded URLs break when the environment changes. Environment handling lets the same test suite run against Dev, Test, and UAT automatically — just by changing an env variable or passing a flag. CI pipelines use this heavily.

HOW it works

Set BASE_URL environment variable → playwright.config.ts reads process.env.BASE_URL → sets baseURL → all page.goto('/path') calls resolve against that base. One config, all environments.

WHERE it's configured

The baseURL in playwright.config.ts use:{} block is the central switch. It's read from process.env.BASE_URL. CI pipelines set this in workflow YAML. Developers set it in a .env.local file. Never hardcode URLs in individual test files.

Interview Prep — 5 SDET Q&A
Q1: How do you configure Playwright to run the same tests against Dev, Test, and UAT? ▶
The standard approach: in playwright.config.ts, set baseURL: process.env.BASE_URL ?? 'http://localhost:3000'. In all test files, use relative paths: page.goto('/dashboard') — never absolute URLs. To run against different environments: npx BASE_URL=https://test.myapp.com playwright test (Test env). npx BASE_URL=https://uat.myapp.com playwright test (UAT). npx playwright test (defaults to localhost Dev). In CI, set BASE_URL as a pipeline environment variable or secret. On developer machines, create .env.local with BASE_URL=http://localhost:3000 and load it with dotenv. This way: zero test files change between environments, only the environment variable changes.
Q2: How do you handle different credentials for each environment? ▶
I create a test-accounts.ts file that exports credentials per environment: const TEST_ACCOUNTS = { dev: { email:'dev-user@test.com', password:'DevPass123' }, test: { email:'qa-user@test.com', password:'QAPass123' }, uat: { email:'uat-user@test.com', password:'UATPass123' } }. In tests: const env = (process.env.TEST_ENV ?? 'dev') as keyof typeof TEST_ACCOUNTS; const {email, password} = TEST_ACCOUNTS[env]. In CI: set TEST_ENV=uat and the actual credential values as secrets (process.env.UAT_EMAIL, process.env.UAT_PASSWORD). Never hardcode real passwords. For globalSetup: it reads TEST_ENV and uses the right account to login and save storageState.
Q3: What is dotenv and how do you use it with Playwright? ▶
dotenv is an npm package that reads a .env file and loads each KEY=VALUE line into process.env. In Playwright: npm install -D dotenv. In playwright.config.ts: import dotenv from 'dotenv'; dotenv.config({path:'.env.local'}). The .env.local file (NOT committed to git): BASE_URL=http://localhost:3000, TEST_ENV=dev, TEST_PASSWORD=localdevpassword. The .env.example file (committed, placeholder values): BASE_URL=, TEST_ENV=dev, TEST_PASSWORD=. This workflow: new developer joins, copies .env.example to .env.local, fills in their local values, runs tests — they work without any shared config changes. CI doesn't use dotenv — it has real env variables set in the pipeline YAML or secrets manager.
Q4: How do you prevent tests from accidentally running against production? ▶
Multiple layers: (1) Skip destructive tests on prod: test.skip(process.env.TEST_ENV === 'prod', 'No data-mutating tests on production'). (2) Separate config file for prod: playwright.config.prod.ts with testMatch:['**/smoke/**/*.spec.ts'] — only smoke tests. (3) Global check in globalSetup: if (process.env.TEST_ENV === 'prod' && process.env.ALLOW_PROD_TESTS !== 'true') throw new Error('Production tests require explicit ALLOW_PROD_TESTS=true flag'). (4) CI pipeline gate: the prod smoke test job only runs on specific tags/branches (release tags), never on every PR. (5) Read-only test accounts on prod: prod test user has no write permissions, so even if a destructive test runs it fails with 403 before mutating data.
Q5: How do you structure multiple environment config files? ▶
Create a base config and environment-specific overrides: base.config.ts exports shared settings (projects, timeout, reporters). Environment-specific configs import and override: playwright.config.uat.ts — import base, spread it, override use:{baseURL:'https://uat.myapp.com', storageState:'playwright/.auth/uat-user.json'}, retries:1. playwright.config.prod.ts — import base, override testMatch:['**/smoke/**'], retries:0, use:{baseURL:'https://myapp.com'}. Run with: npx playwright test --config=playwright.config.uat.ts. In CI: uat-job uses --config=playwright.config.uat.ts, smoke-job uses --config=playwright.config.prod.ts. This avoids a single giant config.ts with dozens of if/else branches.

Session 21 · Complete Playwright CLI Reference

The Playwright Command Line Interface (CLI) is the entry point for all test operations — running, debugging, recording, inspecting, and managing browsers. Every command starts with npx playwright. Understanding the full CLI lets you control every aspect of test execution from a terminal or CI pipeline.

WHAT is the CLI?

A Node.js binary installed with @playwright/test. Entry point: npx playwright. Provides commands for: running tests, debugging, generating code, installing browsers, showing reports, and more.

WHY master the CLI?

CI/CD pipelines are 100% CLI-driven. Debugging, running specific tests, filtering by project/browser, sharding across machines, updating snapshots — all done via CLI flags. No GUI required.

Main commands

test (run), show-report, show-trace, codegen (record), install (browsers), debug, merge-reports, --ui mode. Each has its own sub-options.

Flag priority

CLI flags override playwright.config.ts. So --headed overrides headless:true in config. --retries=0 overrides retries:2 in config. CLI is the highest priority override layer.

Keyword Glossary — Session 21
npx playwright testRun all tests. The main command. Reads playwright.config.ts by default. Exits 0 on pass, 1 on failure.
--headedRun tests with a visible browser window. Opposite of headless. Use for debugging locally. Slow — each test opens a visible window.
--debugShorthand for PWDEBUG=1. Opens Playwright Inspector, pauses at page.pause() calls, enables step-through debugging.

Interview Prep — 5 SDET Q&A
Q1: How do you run a single test from the command line? ▶
Three approaches: (1) By file: npx playwright test tests/checkout.spec.ts — runs all tests in that file. (2) By test title pattern: npx playwright test -g "checkout button works" — runs all tests whose title matches. Use quotes for multi-word titles. (3) By file and line number: npx playwright test tests/checkout.spec.ts:42 — runs the specific test defined at line 42 of that file. This is the most precise. (4) Combined: npx playwright test checkout.spec.ts -g "payment" --project=chromium --headed --workers=1 — runs payment tests in checkout.spec.ts on Chromium in headed mode with a single worker, ideal for debugging. The --workers=1 flag is important during debugging because parallel workers interleave output making it hard to follow.
Q2: What is the difference between --debug, --ui, and PWDEBUG=1? ▶
--debug and PWDEBUG=1 are identical — both launch the test in headed mode with the Playwright Inspector open. The test pauses at every page.pause() call in the test code. You can step through actions, inspect elements, and use the locator picker. --ui is different and more powerful — it opens the Playwright UI Mode, which is a full GUI application showing: a tree of all tests, the ability to click any test to run it, a live browser panel showing what's happening, time-travel scrubbing through the test's action history, and detailed assertion information. --ui doesn't require page.pause() in the test — it's always interactive. Best practice: use --ui for general debugging, use --debug only when you need to step through specific page.pause() points in the test code.
Q3: How does --shard work and why is it used in CI? ▶
--shard=x/n tells Playwright to run the x-th slice of a total n slices of the test plan. Playwright deterministically divides all collected tests into equal groups. --shard=1/4 runs tests 1-25%, --shard=2/4 runs 26-50%, etc. In CI: set up a matrix job (GitHub Actions matrix, GitLab parallel) that runs multiple machines simultaneously, each with a different --shard value. Example: 100 tests × 4 shards = 25 tests per machine → ~4× speed improvement. The workflow: each shard saves results as blob reporter output. A final merge job downloads all blob files and runs: npx playwright merge-reports --reporter html ./all-blobs to create a unified HTML report showing all 100 test results. This gives both speed AND a single unified report.
Q4: How do you use codegen and what are its limitations? ▶
npx playwright codegen https://myapp.com opens two windows: the browser and a code panel. As you interact in the browser (click, fill, navigate), Playwright generates TypeScript code in real time. Useful for: quickly getting working locators for elements you've never tested, generating the skeleton of a new test flow, recording auth flows to extract cookie/localStorage patterns. Limitations: (1) Generated locators are often CSS-based (.class names) rather than semantic (getByRole) — always review and improve them. (2) Generated code doesn't handle async waits intelligently — it records actions but not the page's loading state between them. (3) It can't generate assertions — it only records actions. (4) Code often needs significant cleanup for production use: add proper waits, replace brittle locators, add assertions, extract page objects. Think of codegen as a starting scaffold, not production code.
Q5: What package.json scripts do you set up for a typical Playwright project? ▶
Typical scripts: "test": "playwright test" — run all tests, "test:dev": "BASE_URL=http://localhost:3000 playwright test" — run against local dev, "test:uat": "BASE_URL=https://uat.myapp.com TEST_ENV=uat playwright test --config=playwright.config.uat.ts" — UAT run, "test:smoke": "playwright test --config=playwright.config.prod.ts" — production smoke, "test:debug": "playwright test --debug --headed", "test:ui": "playwright test --ui" — UI mode, "test:headed": "playwright test --headed", "test:chrome": "playwright test --project=chromium", "test:last-failed": "playwright test --last-failed", "codegen": "playwright codegen", "report": "playwright show-report", "install:browsers": "playwright install --with-deps". Package.json scripts let teammates run complex CLI commands with simple npm run commands without memorizing flag combinations.

npx playwright test --config=playwright.config.uat.ts

# Fail if test.only() is found anywhere
npx playwright test --forbid-only

#!/bin/bash
# cli-debug-commands.sh
# PURPOSE: All debugging-related CLI commands

# ── PLAYWRIGHT INSPECTOR (Step-through debugger) ──────────────────

# Open Inspector — pauses at page.pause() calls in test code
npx playwright test --debug

# Same via environment variable (Linux/Mac)
PWDEBUG=1 npx playwright test

# On Windows (PowerShell)
# $env:PWDEBUG=1; npx playwright test

# Debug a specific test only
npx playwright test login.spec.ts --debug

# Debug with no timeout (needed when you're slow stepping through)
npx playwright test --debug --timeout=0

# ── UI MODE (Best local debugging experience) ─────────────────────

# Open Playwright UI Mode — interactive test runner with browser panel
npx playwright test --ui

# UI mode on a specific port
npx playwright test --ui --ui-port=8080

# UI mode with specific host (for remote access)
npx playwright test --ui --ui-host=0.0.0.0

# ── HEADED MODE (Show browser window) ────────────────────────────

# Run with visible browser
npx playwright test --headed

# Slow motion — delay between each action (ms)
# Configure in playwright.config.ts: use: { launchOptions: { slowMo: 500 } }
# Or via CLI with PWDEBUG=console
PWDEBUG=console npx playwright test --headed

# ── CODE GENERATION (Record test flows) ───────────────────────────

# Start recording — opens browser + code panel
npx playwright codegen https://myapp.com

# Record with TypeScript output explicitly
npx playwright codegen --target=typescript https://myapp.com

# Save generated code to a file automatically
npx playwright codegen -o tests/generated/new-test.spec.ts https://myapp.com

# Record with a saved auth session loaded
npx playwright codegen --load-storage=playwright/.auth/admin.json https://myapp.com

# Record and save the session at the end
npx playwright codegen --save-storage=playwright/.auth/recorded.json https://myapp.com

# Record on a specific viewport size
npx playwright codegen --viewport-size=375,812 https://myapp.com

# Record emulating a device
npx playwright codegen --device="iPhone 14 Pro" https://myapp.com

# ── TRACE VIEWER ─────────────────────────────────────────────────

# Open a trace file in the interactive viewer
npx playwright show-trace test-results/my-test/trace.zip

# Show trace for a specific test-results folder
npx playwright show-trace test-results/login-chromium/trace.zip

#!/bin/bash
# cli-reporters-artifacts.sh
# PURPOSE: Reporter configuration, artifact capture, report viewing

# ── REPORTERS ─────────────────────────────────────────────────────

# HTML report (rich interactive — saved to playwright-report/)
npx playwright test --reporter=html

# Dot reporter (CI-friendly — one dot per test)
npx playwright test --reporter=dot

# Line reporter (one line per test)
npx playwright test --reporter=line

# JSON report (machine-readable — for custom processing)
npx playwright test --reporter=json
npx playwright test --reporter=json --output=test-results/results.json

# JUnit XML (for CI dashboards — GitHub, Jenkins, GitLab)
npx playwright test --reporter=junit
npx playwright test --reporter=junit --output=test-results/junit.xml

# Multiple reporters simultaneously
npx playwright test --reporter=html,dot

# Blob reporter (for shard merging)
npx playwright test --reporter=blob

# ── ARTIFACTS ─────────────────────────────────────────────────────

# Override trace setting
npx playwright test --trace=on              # always capture trace
npx playwright test --trace=on-first-retry  # trace only on retry
npx playwright test --trace=off             # disable trace

# Override screenshot setting
npx playwright test --screenshot=on               # always capture
npx playwright test --screenshot=only-on-failure  # only on fail
npx playwright test --screenshot=off

# Override video recording
npx playwright test --video=on               # always record
npx playwright test --video=on-first-retry   # only on retry
npx playwright test --video=off

# ── VISUAL REGRESSION ─────────────────────────────────────────────

# Update toHaveScreenshot() baselines after intentional UI changes
npx playwright test --update-snapshots

# Update only specific test's snapshots
npx playwright test checkout.spec.ts --update-snapshots

# ── REPORT VIEWING ────────────────────────────────────────────────

# Open HTML report in browser (default: playwright-report/)
npx playwright show-report

# Open a specific report folder
npx playwright show-report ./custom-report/

# Merge shard blob reports into unified HTML
npx playwright merge-reports --reporter html ./all-blob-reports

# Merge with specific output folder
npx playwright merge-reports --reporter html \
  --output ./merged-report \
  ./all-blob-reports

# Then in steps:

npx playwright install

npx playwright install chromium

npx playwright install chromium firefox          # multiple browsers

npx playwright install --with-deps

npx playwright install chromium --with-deps
