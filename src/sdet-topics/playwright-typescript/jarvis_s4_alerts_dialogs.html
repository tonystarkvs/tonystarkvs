Overview — Alerts, Dialogs & Popups Handling in Playwright

Browser dialogs are native browser pop-ups triggered by JavaScript: alert(), confirm(), prompt(), and the page unload warning (beforeunload). They block page execution until dismissed. Playwright handles them via the page.on('dialog', handler) event listener. If no handler is registered, Playwright auto-dismisses dialogs by default — your test won't hang, but you won't verify dialog content either.

What / Why / When / Where / How

WhatNative browser dialogs (alert, confirm, prompt, beforeunload) that block JavaScript execution until the user responds. They are NOT DOM elements — they're browser chrome.
WhyIf a dialog appears and no handler is registered, Playwright auto-dismisses it. If your test needs to verify dialog content, accept it, or input text into a prompt — you must register a handler.
WhenDelete confirmations (confirm()), save notifications (alert()), user input collection (prompt()), and unsaved changes warnings (beforeunload).
WhereRegister page.on('dialog', handler) BEFORE the action that triggers the dialog. The handler receives a Dialog object with accept(), dismiss(), message(), type().
Howpage.on('dialog', async (dialog) => { await dialog.accept(); }) for persistent handlers, or page.waitForEvent('dialog') for one-time dialogs.

Keyword Glossary

page.on('dialog')
Registers an event listener that fires whenever a browser dialog (alert/confirm/prompt/beforeunload) appears. Must be registered before the triggering action.
Dialog
The Playwright object representing an open browser dialog. Provides: type(), message(), defaultValue(), accept(), dismiss().
dialog.accept()
Clicks the OK/Accept button. For prompt dialogs, pass a string: dialog.accept('my input text') to fill the text field before accepting.
dialog.dismiss()
Clicks the Cancel/Dismiss button. For alert (no Cancel), dismiss() behaves like accept(). For confirm/prompt, returns false/null to JS.
dialog.message()
Returns the text content of the dialog. Use to assert the dialog shows the correct message before accepting/dismissing.
dialog.type()
Returns the type as a string: 'alert', 'confirm', 'prompt', or 'beforeunload'. Useful in a single handler that routes different types.
dialog.defaultValue()
Returns the default value pre-filled in a prompt dialog's input. e.g. prompt('Name', 'John') → defaultValue() returns 'John'.
waitForEvent('dialog')
Alternative to page.on() — returns a Promise that resolves with the Dialog object when it appears. Useful for one-time dialogs.
beforeunload
A special dialog type that fires when the user navigates away from a page with unsaved changes. accept() lets the user leave; dismiss() keeps them on the page.
page.removeListener()
Removes a previously registered event listener. Use to remove dialog handlers after a specific test scenario.

Interview One-Liners

Playwright auto-dismisses dialogs by default if no handler is registered — your test won't hang on an unexpected alert, but you won't verify its content unless you set up page.on('dialog').
Just like popup detection, dialog handling requires the listener to be set up BEFORE the action that triggers it — otherwise you miss the event and the dialog auto-dismisses.
There are four browser dialog types: alert (one button), confirm (OK/Cancel), prompt (text input), and beforeunload (leave/stay) — dialog.type() tells you which one.
For prompt dialogs, dialog.accept('my text') fills the text field and clicks OK in one call — you don't interact with the input separately.
I always assert dialog.message() in my handler before calling accept() or dismiss() — this verifies the app shows the right message text, not just that a dialog appeared.
If a test involves multiple sequential dialogs, I use page.on() with a response queue — each call to the handler consumes one dialog and pops the next expected response.
Testing beforeunload is tricky — I register the handler, then call page.close() or navigate away. The handler dismiss() keeps the user on the page, accept() lets them leave.

Dialog Types — Accept vs Dismiss Behavior
alert('msg')accept() → OK clicked, no return. dismiss() → same as accept (no Cancel button).
confirm('sure?')accept() → JS gets true. dismiss() → JS gets false.
prompt('name?')accept('Alice') → JS gets 'Alice'. dismiss() → JS gets null.
beforeunloadaccept() → user leaves page. dismiss() → user stays on page.
One-time vs Persistent Handler
page.on('dialog')Persistent — fires every time a dialog appears. Remove with page.removeListener().
waitForEvent('dialog')One-time — resolves once, returns Dialog object. Cleaner for single-use scenarios.

alert-handling.ts
// alert-handling.ts
// PURPOSE: Handle window.alert() dialogs in Playwright
// alert() has one button (OK) — it just notifies the user

import { test, expect } from '@playwright/test';

test('handle alert dialog', async ({ page }) => {

  await page.goto('https://myapp.com/notifications');

  // Track dialog message for assertion
  let dialogMessage = '';

  // STEP 1: Register handler BEFORE the action that triggers it
  page.on('dialog', async (dialog) => {
    // Capture the dialog message for assertion
    dialogMessage = dialog.message();
    console.log(`📢 Dialog type: ${dialog.type()}`);    // 'alert'
    console.log(`📢 Message: ${dialog.message()}`);         // 'Success!'

    // accept() = click OK on the alert
    await dialog.accept();
  });

  // STEP 2: Trigger the action that fires the alert
  await page.click('#save-settings-btn');

  // STEP 3: Assert the dialog showed the correct message
  expect(dialogMessage).toBe('Settings saved successfully!');

  // STEP 4: Assert page state after dialog is dismissed
  await expect(page.locator('.settings-status'))
    .toContainText('Saved');
});
confirm-dialog.ts
// confirm-dialog.ts
// PURPOSE: Handle window.confirm() — OK returns true, Cancel returns false
// USE CASE: Delete confirmations, dangerous action warnings

import { test, expect } from '@playwright/test';

test.describe('Delete confirmation dialog', () => {

  test('accept confirm → item deleted', async ({ page }) => {

    await page.goto('https://myapp.com/items');
    await expect(page.locator('.item-row')).toHaveCount(3);

    // Register handler: accept the confirm dialog
    page.on('dialog', async (dialog) => {
      expect(dialog.type()).toBe('confirm');
      expect(dialog.message()).toContain('Are you sure you want to delete');
      await dialog.accept(); // ← clicks OK → JS confirm() returns true
    });

    await page.click('.delete-btn');

    // Item was deleted — now 2 remain
    await expect(page.locator('.item-row')).toHaveCount(2);
  });

  test('dismiss confirm → item NOT deleted', async ({ page }) => {

    await page.goto('https://myapp.com/items');
    await expect(page.locator('.item-row')).toHaveCount(3);

    // Register handler: dismiss the confirm dialog (click Cancel)
    page.on('dialog', async (dialog) => {
      await dialog.dismiss(); // ← clicks Cancel → JS confirm() returns false
    });

    await page.click('.delete-btn');

    // Nothing deleted — still 3 items
    await expect(page.locator('.item-row')).toHaveCount(3);
  });
});
prompt-dialog.ts
// prompt-dialog.ts
// PURPOSE: Handle window.prompt() — passes text input to dialog
// KEY: dialog.accept('text') fills input AND clicks OK in one call

import { test, expect } from '@playwright/test';

test('handle prompt dialog with user input', async ({ page }) => {

  await page.goto('https://myapp.com/rename');

  // Use waitForEvent for one-time dialogs — cleaner than page.on()
  const dialogPromise = page.waitForEvent('dialog');

  // Trigger the prompt (app calls: prompt('Enter new name:', 'Default Name'))
  await page.click('#rename-btn');

  // Await and capture the dialog
  const dialog = await dialogPromise;

  // Verify dialog properties
  expect(dialog.type()).toBe('prompt');
  expect(dialog.message()).toBe('Enter new name:');
  expect(dialog.defaultValue()).toBe('Default Name'); // pre-filled value

  // Accept with custom text → JS receives 'My New Project' as return value
  await dialog.accept('My New Project');

  // Page should update with the new name
  await expect(page.locator('.project-title'))
    .toHaveText('My New Project');
});

test('dismiss prompt → no rename happens', async ({ page }) => {

  await page.goto('https://myapp.com/rename');

  page.on('dialog', async (dialog) => {
    await dialog.dismiss(); // Cancel → JS prompt() returns null
  });

  await page.click('#rename-btn');

  // Title unchanged since user cancelled
  await expect(page.locator('.project-title'))
    .toHaveText('Default Name');
});

// beforeunload.ts
// PURPOSE: Handle the beforeunload "Leave site?" dialog
// TRIGGER: User navigates away from a page with unsaved form data

import { test, expect } from '@playwright/test';

test('beforeunload: dismiss keeps user on page', async ({ page }) => {

  await page.goto('https://myapp.com/edit-article');

  // Start editing (creates unsaved state → beforeunload will fire)
  await page.fill('textarea#content', 'My unsaved article content');

  // Register handler for beforeunload
  page.on('dialog', async (dialog) => {
    expect(dialog.type()).toBe('beforeunload');
    // dismiss() = "Stay on page" / "Cancel"
    await dialog.dismiss();
  });

  // Try to navigate away — triggers beforeunload
  await page.goBack();

  // Should still be on edit page (navigation was blocked)
  await expect(page).toHaveURL(/edit-article/);
  await expect(page.locator('textarea#content'))
    .toHaveValue('My unsaved article content');
});

test('beforeunload: accept lets user leave', async ({ page }) => {

  await page.goto('https://myapp.com/edit-article');
  await page.fill('textarea#content', 'Unsaved content');

  page.on('dialog', async (dialog) => {
    // accept() = "Leave page" / "OK"
    await dialog.accept();
  });

  // Navigate away — accepts the warning → user leaves
  await page.goBack();

  // Should be on previous page now
  await expect(page).toHaveURL(/\/articles/);
});
multi-dialog.ts
// multi-dialog.ts
// PURPOSE: Handle multiple sequential dialogs in one test
// PATTERN: Response queue — shifts one response per dialog

import { test, expect, Dialog } from '@playwright/test';

type DialogResponse = {
  action: 'accept' | 'dismiss';
  text?: string;           // only for prompt dialogs
  expectedMsg?: string;   // expected message to assert
};

test('multi-step flow with sequential dialogs', async ({ page }) => {

  await page.goto('https://myapp.com/batch-actions');

  // Define responses queue in the ORDER dialogs will appear
  const responses: DialogResponse[] = [
    { action: 'accept', expectedMsg: 'Archive 5 items?' },       // Dialog 1
    { action: 'accept', text: 'archive-2024', expectedMsg: 'Archive name:' }, // Dialog 2 (prompt)
    { action: 'dismiss', expectedMsg: 'Also delete originals?' }, // Dialog 3
  ];

  // Single handler manages all dialogs using the queue
  page.on('dialog', async (dialog: Dialog) => {
    const response = responses.shift(); // pop first response
    if (!response) throw new Error('Unexpected dialog!');

    if (response.expectedMsg) {
      expect(dialog.message()).toContain(response.expectedMsg);
    }

    if (response.action === 'accept') {
      await dialog.accept(response.text ?? '');
    } else {
      await dialog.dismiss();
    }
  });

  // Trigger the action that fires all 3 dialogs in sequence
  await page.click('#batch-archive-btn');

  // All dialogs handled — verify final state
  await expect(page.locator('.archive-status'))
    .toContainText('Archived as: archive-2024');
  expect(responses.length).toBe(0); // all responses were consumed
});
dialog-types-router.ts
// dialog-types-router.ts
// PURPOSE: Single smart handler that routes all dialog types
// USE CASE: Reusable utility for test frameworks

import { test, expect, Page, Dialog } from '@playwright/test';

// ── Reusable dialog handler factory ─────────────────────────────
function createDialogHandler(options: {
  alert?: 'accept' | 'dismiss';
  confirm?: 'accept' | 'dismiss';
  prompt?: { action: 'accept' | 'dismiss'; text?: string };
  beforeunload?: 'accept' | 'dismiss';
  onMessage?: (msg: string, type: string) => void;
}) {
  return async (dialog: Dialog) => {
    const type = dialog.type();
    const msg  = dialog.message();

    options.onMessage?.(msg, type);
    console.log(`[Dialog] type=${type} message="${msg}"`);

    switch (type) {
      case 'alert':
        await (options.alert === 'dismiss'
          ? dialog.dismiss() : dialog.accept());
        break;
      case 'confirm':
        await (options.confirm === 'dismiss'
          ? dialog.dismiss() : dialog.accept());
        break;
      case 'prompt':
        const p = options.prompt;
        if (p?.action === 'dismiss') { await dialog.dismiss(); }
        else { await dialog.accept(p?.text ?? ''); }
        break;
      case 'beforeunload':
        await (options.beforeunload === 'accept'
          ? dialog.accept() : dialog.dismiss());
        break;
    }
  };
}

// ── Usage in test ────────────────────────────────────────────────
test('smart dialog router handles all types', async ({ page }) => {

  await page.goto('https://myapp.com');

  const messages: string[] = [];

  page.on('dialog', createDialogHandler({
    alert:       'accept',
    confirm:     'accept',
    prompt:      { action: 'accept', text: 'Test User' },
    beforeunload: 'dismiss',
    onMessage: (msg) => messages.push(msg),
  }));

  await page.click('#trigger-all-dialogs');
  console.log(`All dialog messages: ${messages.join(' | ')}`);
});

Best Practices

  • ✓Always register dialog handlers before the action that triggers them. The 'dialog' event fires synchronously in the browser — a late-registered handler will never catch it.
  • ✓Always assert dialog.message() inside the handler before accepting/dismissing — this verifies the dialog shows the correct text, not just that one appeared.
  • ✓Use dialog.type() to write a single handler that routes different dialog types: alert → accept(), confirm → check expected flow, prompt → accept with test data.
  • ✓For one-time dialogs, prefer page.waitForEvent('dialog') over page.on() — it's cleaner, doesn't need cleanup with removeListener().
  • ✓If your test has multiple dialogs in sequence, maintain a response queue array in the handler and shift() one response per dialog call.
  • ✓Always await dialog.accept() and dialog.dismiss() — they return Promises even though the dialog appears synchronous. Not awaiting causes test instability.
  • ✓For beforeunload testing, use page.close() with a registered handler for reliable triggering — page.goto() may not always trigger it depending on browser behavior.

Interview Prep — 5 SDET Q&A

Q1: What happens if a dialog appears and no handler is registered in Playwright?
A: Playwright auto-dismisses dialogs if no 'dialog' event handler is registered. For alert — it's like clicking OK. For confirm — it's like clicking Cancel (returns false to JS). For prompt — returns null. For beforeunload — stays on the page. This means tests won't hang on unhandled dialogs, but you also can't verify dialog content or test acceptance flows without an explicit handler.
Q2: How do you test a delete confirmation flow that uses window.confirm()?
A: I set up the handler before clicking Delete: page.on('dialog', async (dialog) => { expect(dialog.type()).toBe('confirm'); expect(dialog.message()).toContain('Are you sure'); await dialog.accept(); }). Then click the delete button. Then assert the item is gone. For the dismiss case (user clicks Cancel), I use dialog.dismiss() and verify the item still exists. Two separate test cases cover both paths.
Q3: How do you handle a window.prompt() dialog in Playwright?
A: prompt() dialogs have an input field. dialog.accept('Alice') fills the prompt's input field with 'Alice' and confirms — the calling JS code receives 'Alice' as the return value. I also check dialog.defaultValue() to verify the pre-filled value. Calling dialog.dismiss() instead makes JS receive null regardless of any input.
Q4: How do you handle multiple sequential dialogs in one test?
A: I use a response queue pattern: const responses = [{action:'accept'}, {action:'dismiss'}, {action:'accept', text:'data'}]. In page.on('dialog'), each invocation calls responses.shift() to pop one response. This is clean and deterministic. It scales to any number of dialogs in any order.
Q5: How do you test the beforeunload dialog — the "Leave site?" warning?
A: Beforeunload dialogs appear when navigating away from a page with unsaved changes. I navigate to the form, start filling it (triggering unsaved state), register the handler, then call page.close() or page.goBack(). dialog.dismiss() keeps the user on the page. dialog.accept() lets them leave. Note: Some frameworks use custom confirm() dialogs instead of real beforeunload — those are handled identically.
