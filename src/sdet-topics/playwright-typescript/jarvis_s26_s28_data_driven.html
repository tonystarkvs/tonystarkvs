Session 26 · External JSON Test Data

External JSON test data means storing all test inputs, expected outputs, and test case metadata in .json files outside your test code. Tests read this data at runtime using Node.js fs.readFileSync() or require()/import. This separates what to test (data) from how to test (code) — the foundation of Data-Driven Testing.

WHAT

JSON files in a test-data/ folder hold arrays of objects — each object is one test case with inputs and expected values. Tests iterate over the array and run once per object.

WHY external JSON?

Non-engineers can add/modify test cases without touching TypeScript. Test count scales without code changes. Data is reusable across test files. Easy to generate from APIs or databases.

WHEN to use?

Login scenarios, form validation (valid/invalid inputs), API endpoint testing with multiple payloads, search with different queries, price calculations, currency formats. Anywhere you'd copy-paste the same test with different values.

HOW to read JSON?

Three ways: JSON.parse(fs.readFileSync(path,'utf-8')), TypeScript import data from './data.json' with resolveJsonModule:true, or async fs.promises.readFile(). All work identically in Playwright tests.

Core Concepts
⬡ Data-Driven Test Structure
test-data/login.json→array of test cases (data)
tests/login.spec.ts→for...of loop — one test() per case
JSON row → test()→test name uses data.description or data.role
JSON row → page.fill()→inputs from data.email, data.password
JSON row → expect()→assertions use data.expectedURL, data.expectedMsg
⬡ JSON Data File — Schema Design

Every JSON test data file should follow these conventions:

Top-level array→[ { case 1 }, { case 2 }, ... ]
id field→unique identifier — appears in test name
description→human-readable test intent
input fields→email, password, query, amount — test inputs
expected fields→expectedURL, expectedMsg, shouldPass — assertions
tags (optional)→["smoke","regression"] — for filtering
Best Practices
✓
Always define a TypeScript interface matching the JSON shape and cast parsed data to it. Compile-time safety prevents silent typos in property names.
✓
Load JSON data at the module level (outside test/describe blocks) using fs.readFileSync() — data is loaded once when the file is collected, not on every test execution.
✓
Always include a description or id field in each test case object and use it in the test name — reports become self-documenting.
✓
Use path.join(__dirname, '../test-data/file.json') for paths — never relative strings like './data.json' which depend on the current working directory.
✓
Keep JSON data files small and focused — one file per feature (login.json, registration.json, search.json). Don't create one giant all-data.json file.
✓
Add a skip: boolean field per test case and filter before looping: loginData.filter(d => !d.skip) — disables specific cases without deleting them from the file.
Interview Prep
Q1: What is Data-Driven Testing and how do you implement it in Playwright? ▶
Data-Driven Testing (DDT) separates test logic from test data. The same test code runs multiple times, once per data set. In Playwright: (1) Create test-data/login.json with an array of test case objects. (2) In the spec file: const loginData = JSON.parse(fs.readFileSync(path.join(__dirname,'../test-data/login.json'),'utf-8')) as LoginData[]. (3) Loop: for (const tc of loginData) { test(`${tc.description}`, async ({page}) => { await page.fill('#email', tc.email); await page.fill('#password', tc.password); await page.click('button[type="submit"]'); await expect(page).toHaveURL(tc.expectedURL); }); }. Adding a new test case requires only adding a new JSON object — no TypeScript changes.
Q2: How do you handle type safety when reading JSON test data? ▶
Define a TypeScript interface matching the JSON shape exactly: interface LoginTestCase { id: string; description: string; email: string; password: string; expectedURL: string; shouldPass: boolean; }. Then cast: const data = JSON.parse(fs.readFileSync(fp,'utf-8')) as LoginTestCase[]. Now TypeScript enforces that you only access valid properties — accessing data[0].emailAddress would be a compile error. For import-style access: add "resolveJsonModule": true to tsconfig.json, then: import loginData from '../test-data/login.json'; TypeScript infers the type automatically from the JSON structure. The interface approach is safer for large files because it explicitly documents the expected schema.
Q3: Where should test data files live in the project structure? ▶
Standard structure: project root/test-data/ for all JSON/CSV data files, OR tests/fixtures/ for smaller fixture data. I prefer: tests/test-data/login.json, tests/test-data/registration.json, tests/test-data/products.json — grouped by feature, same level as the spec files. Key rules: (1) Committed to git — not .gitignored. (2) Never in node_modules. (3) Use path.join(__dirname) to reference, never hardcoded absolute paths. (4) Separate from fixture files (PDFs, images) — those go in tests/fixtures/. For large suites: tests/test-data/auth/login.json, tests/test-data/ecommerce/checkout.json — hierarchical by domain.
Q4: How do you skip specific test cases in a JSON data array without deleting them? ▶
Add a skip field to the JSON schema: { "id":"TC001","description":"admin login","email":"admin@test.com","skip":false } and { "id":"TC002","description":"broken scenario","skip":true }. In the test file: const activeData = loginData.filter(tc => !tc.skip). Loop over activeData instead of loginData. Alternatively, add a reason field: { "skip":true,"skipReason":"BUG-456: broken until sprint 12" }. Log skipped cases: const skipped = loginData.filter(tc => tc.skip); if (skipped.length) console.log('Skipped:', skipped.map(s => s.id)); This approach preserves the skipped data for documentation and lets reviewers see what's being skipped in PRs.
Q5: How do you make JSON test data environment-aware? ▶
Two approaches: (1) Environment-specific files: tests/test-data/login.dev.json, tests/test-data/login.uat.json. In the spec: const env = process.env.TEST_ENV ?? 'dev'; const dataFile = path.join(__dirname, `../test-data/login.${env}.json`); const data = JSON.parse(fs.readFileSync(dataFile,'utf-8')). (2) Environment field within each case: { "email":"admin@test.com","environments":["dev","test","uat"] }. Filter: const relevantData = allData.filter(d => d.environments.includes(process.env.TEST_ENV ?? 'dev')). Approach 1 is cleaner for cases where Dev and UAT have completely different data sets. Approach 2 is better when most data is shared but a few cases only apply to specific environments.

tests/login-data-driven.spec.ts
// tests/login-data-driven.spec.ts
// PURPOSE: Read login.json and run one test per data entry
// PATTERN: fs.readFileSync at module level + for...of loop

import { test, expect } from '@playwright/test';
import fs   from 'fs';
import path from 'path';

// ── Load JSON data at module level (runs once at collection time) ─
const DATA_FILE = path.join(__dirname, '../test-data/login.json');
const loginCases = JSON.parse(fs.readFileSync(DATA_FILE, 'utf-8'));

// ── Run tests inside describe for clean report grouping ──────────
test.describe('Login — Data-Driven from JSON', () => {

  for (const tc of loginCases) {

    // Skip disabled test cases without deleting them
    if (tc.skip) {
      console.log(`⚠️  Skipping [${tc.id}]: ${tc.skipReason ?? 'no reason'}`);
      test.skip(`[${tc.id}] ${tc.description}`, async () => {});
      continue;
    }

    // Test name includes ID + description — makes report readable
    test(`[${tc.id}] ${tc.description}`, async ({ page }) => {

      await page.goto('/login');
      await page.fill('#email',    tc.email);
      await page.fill('#password', tc.password);
      await page.click('button[type="submit"]');

      if (tc.shouldPass) {
        // Positive case: assert redirect URL and page title
        await expect(page).toHaveURL(new RegExp(tc.expectedURL));
        await expect(page).toHaveTitle(tc.expectedTitle);
      } else {
        // Negative case: stay on login, show error message
        await expect(page).toHaveURL(/login/);
        await expect(page.locator('.error-msg'))
          .toContainText(tc.expectedMsg);
      }
    });
  }
});
tests/typed-json-data.spec.ts
// typed-json-data.spec.ts — TypeScript interface for type-safe JSON
import { test, expect } from '@playwright/test';
import fs   from 'fs';
import path from 'path';

// ── TypeScript interface — defines the exact JSON schema ──────────
// Compile-time error if test code accesses wrong field name
interface LoginTestCase {
  id:            string;
  description:   string;
  email:         string;
  password:      string;
  role:          string;
  expectedURL:   string;
  expectedTitle?: string;       // optional field
  expectedMsg?:  string;        // only present for negative cases
  shouldPass:    boolean;
  skip:          boolean;
  skipReason?:   string;
  tags:          string[];
}

// Load and cast with the interface — type-safe access
const loginData: LoginTestCase[] = JSON.parse(
  fs.readFileSync(path.join(__dirname, '../test-data/login.json'), 'utf-8')
);

// ── Filter: only run smoke tests from command line flag ───────────
const smokeOnly = process.env.SMOKE === 'true';
const activeData = loginData
  .filter(d => !d.skip)                                      // skip disabled
  .filter(d => !smokeOnly || d.tags.includes('smoke'));  // smoke filter

test.describe('Typed Login Tests', () => {
  for (const tc of activeData) {
    test(`[${tc.id}] ${tc.description}`, async ({ page }) => {
      await page.goto('/login');

      // TypeScript knows tc.email is string — autocomplete works
      await page.fill('#email',    tc.email);
      await page.fill('#password', tc.password);
      await page.click('button[type="submit"]');

      if (tc.shouldPass) {
        await expect(page).toHaveURL(new RegExp(tc.expectedURL));
        if (tc.expectedTitle) {
          await expect(page).toHaveTitle(tc.expectedTitle);
        }
      } else {
        await expect(page).toHaveURL(/login/);
        // tc.expectedMsg is string | undefined — check before using
        if (tc.expectedMsg) {
          await expect(page.locator('.error-msg'))
            .toContainText(tc.expectedMsg);
        }
      }
    });
  }
});

// registration-driven.spec.ts — form validation driven by JSON
import { test, expect } from '@playwright/test';
import fs   from 'fs';
import path from 'path';

interface RegCase {
  id: string; description: string; name: string;
  email: string; password: string; confirmPassword: string;
  phone: string; shouldPass: boolean; expectedURL?: string;
  expectedError?: string; errorField?: string; skip: boolean;
}

const regData: RegCase[] = JSON.parse(
  fs.readFileSync(path.join(__dirname, '../test-data/registration.json'), 'utf-8')
).filter((d: RegCase) => !d.skip);

test.describe('Registration — JSON Data Driven', () => {

  for (const tc of regData) {
    test(`[${tc.id}] ${tc.description}`, async ({ page }) => {
      await page.goto('/register');
      await page.fill('#name',             tc.name);
      await page.fill('#email',            tc.email);
      await page.fill('#password',         tc.password);
      await page.fill('#confirm-password', tc.confirmPassword);
      await page.fill('#phone',            tc.phone);
      await page.click('button[type="submit"]');

      if (tc.shouldPass) {
        await expect(page).toHaveURL(new RegExp(tc.expectedURL!));
      } else {
        if (tc.errorField && tc.expectedError) {
          await expect(page.locator(tc.errorField))
            .toContainText(tc.expectedError);
        }
      }
    });
  }
});
Session 27 · Read & Use Test Data in Multiple Scenarios

One JSON/CSV data file can feed multiple different test scenarios — login tests, API tests, UI tests — all reading from the same source. A DataLoader utility class centralizes all data-reading logic. Tests simply call DataLoader.getLoginData() and receive a typed array. This avoids repetitive fs.readFileSync calls scattered across spec files.

WHAT is a DataLoader?

A utility class or module in utils/DataLoader.ts that encapsulates all JSON/CSV/Excel reading. Exports typed methods: getLoginData(), getProductData(), getSearchData(). Tests import and call these methods.

WHY centralize data loading?

Single source of truth — change the data path in one place. Caching — load once, reuse across tests. Type safety — return typed arrays. Filtering — return only active cases. If you move a file, fix one line not 20 spec files.

CSV data

CSV files are simpler for non-engineers to edit (Excel-readable). Use csv-parser or papaparse npm package to parse. Each row becomes an object. Headers become property names. Parse once, use everywhere.

Reuse across scenarios

Same login data → used in login UI tests, login API tests, auth fixture setup. Same product data → used in search tests, cart tests, checkout tests. Data shared, test logic separate.

Keyword Glossary
DataLoaderUtility class in utils/DataLoader.ts. Static methods that read and return typed test data. Caches data after first read. Single location for all data-reading logic in the project.
CSV parserLibrary to convert CSV text into an array of objects. csv-parse/sync (npm) provides synchronous parsing. papaparse.parse() is also popular. Each row → one object, column headers → property names.
data cachingPattern: store loaded data in a module-level variable. If already loaded (not null/undefined), return cached version. Prevents reading the same file 50 times when 50 tests use the same data.
data filteringDataLoader methods can accept filter parameters: getLoginData('smoke') returns only smoke-tagged cases. getLoginData('positive') returns only shouldPass:true cases. Filtering in one place, not repeated in each spec file.
shared fixture dataPattern where a beforeAll or custom fixture calls DataLoader to load shared data. All tests in the describe use the loaded data. Avoids loading data inside each individual test function.
parameterized testA test that takes parameters (from a data object) instead of hardcoded values. The test function signature is the same; only the data changes per execution.
csv-parse/syncnpm package for synchronous CSV parsing. Usage: import {parse} from 'csv-parse/sync'; const rows = parse(csvString, {columns:true, skip_empty_lines:true}). Returns an array of objects keyed by column headers.

Best Practices
✓
Create a single utils/DataLoader.ts class that all spec files import. No spec file should ever contain fs.readFileSync or direct path resolution for test data.
✓
Add data caching to DataLoader — module-level Map keyed by filename. File is read once per test run, not once per test.
✓
Use CSV format for test data that business stakeholders maintain — they can edit in Excel and save as CSV without touching code. Parse with csv-parse/sync.
✓
DataLoader methods should accept an optional filter parameter ('smoke', 'positive', 'negative') — filter logic lives in DataLoader, not scattered across spec files.
✓
Validate data on load — if a required field is missing in a JSON object, DataLoader should throw a descriptive error: 'TC003 is missing required field: expectedURL'. Catches bad data before tests run.
✓
Same test data should feed both UI tests AND API tests — verify the same scenario works at both layers. If UI login passes but API login fails with the same credentials, that's a bug.
Interview Prep
Q1: How do you build a DataLoader utility and why? ▶
DataLoader is a class (or module) in utils/DataLoader.ts that centralizes all data file reading for the project. Why: (1) DRY — fs.readFileSync + JSON.parse + path.join is written once, not repeated in 20 spec files. (2) Caching — data loaded once per test run, not per test. (3) Filtering — getLoginData('smoke') applies filter logic once. (4) Environment awareness — internally selects the right file based on TEST_ENV. (5) Refactoring safety — if test-data/ folder moves, fix one path in DataLoader. Implementation: export class DataLoader { private static cache = new Map(); static get(file: string) { if (!this.cache.has(file)) this.cache.set(file, JSON.parse(fs.readFileSync(path.join(DATA_DIR, file),'utf-8'))); return this.cache.get(file); } static getLoginData(filter?: string) { const data = this.get('login.json') as LoginTestCase[]; return filter ? data.filter(d => d.tags.includes(filter)) : data; } }
Q2: How do you parse CSV test data in Playwright TypeScript? ▶
Install: npm install -D csv-parse. In DataLoader: import { parse } from 'csv-parse/sync'; import fs from 'fs'; static getProductData(): ProductData[] { const csvString = fs.readFileSync(path.join(DATA_DIR,'products.csv'),'utf-8'); const rows = parse(csvString, { columns: true, // use first row as property names skip_empty_lines: true, cast: true, // auto-convert numbers and booleans trim: true, // remove whitespace }); return rows as ProductData[]; }. The cast:true option converts '42' to 42 and 'true' to true automatically. columns:true maps column headers to property names. In the CSV: id,name,price,inStock → { id:'1', name:'Laptop', price:75000, inStock:true }. Each row returns as one object — same usage pattern as JSON arrays.
Q3: How do you reuse the same test data across UI and API tests? ▶
Login.json has test cases with email, password, role, expectedURL. UI test: for (const tc of DataLoader.getLoginData()) { test('UI: ' + tc.description, async ({page}) => { await page.fill('#email', tc.email); ... }); }. API test using the same data: for (const tc of DataLoader.getLoginData()) { test('API: ' + tc.description, async ({request}) => { const resp = await request.post('/api/auth/login', {data:{email:tc.email, password:tc.password}}); if (tc.shouldPass) expect(resp.status()).toBe(200); else expect(resp.status()).toBe(401); }); }. Both test types share one data source. If you change an email in login.json, both UI and API tests automatically use the new value. This gives you "test pyramid" coverage of the same scenario at both layers.
Q4: How do you handle errors when required fields are missing from JSON data? ▶
Add validation in DataLoader: static validateLoginData(data: any[]): LoginTestCase[] { const requiredFields = ['id','description','email','password','expectedURL','shouldPass']; data.forEach((item, index) => { requiredFields.forEach(field => { if (item[field] === undefined || item[field] === null) { throw new Error(`login.json row ${index} (id: ${item.id ?? 'unknown'}) is missing required field: "${field}". Fix the data file before running tests.`); } }); }); return data as LoginTestCase[]; }. Call in getLoginData(): return this.validateLoginData(raw). This throws before any test starts, with a clear error pointing exactly to the problem row and field. Without this, tests would silently fail with confusing errors like "Cannot read properties of undefined".
Q5: How do you filter data by tags in DataLoader? ▶
Each JSON test case has a tags array: "tags":["smoke","regression"]. DataLoader filter: static getLoginData(tag?: string): LoginTestCase[] { const all = this.get('login.json') as LoginTestCase[]; const active = all.filter(d => !d.skip); if (!tag) return active; return active.filter(d => d.tags?.includes(tag)); }. In spec: for (const tc of DataLoader.getLoginData('smoke')) — only smoke tests. In CI: SMOKE=true npx playwright test runs only smoke-tagged cases. This tag-based filtering is more powerful than file-per-environment because one data file can have smoke, regression, and negative tags — the caller decides which subset to run. For complex filtering: DataLoader.getLoginData('positive') filters shouldPass:true. DataLoader.getLoginData('negative') filters shouldPass:false.
utils/DataLoader.ts
// utils/DataLoader.ts
// PURPOSE: Centralized data loading utility for all test files
// FEATURES: caching, filtering, validation, environment-aware

import fs   from 'fs';
import path from 'path';
import { parse } from 'csv-parse/sync'; // npm install -D csv-parse

// Path to all test data files
const DATA_DIR = path.join(__dirname, '../test-data');

// ── Types ──────────────────────────────────────────────────────────
export interface LoginCase {
  id: string; description: string; email: string;
  password: string; role: string; expectedURL: string;
  expectedMsg?: string; shouldPass: boolean;
  skip: boolean; tags: string[];
}
export interface ProductCase {
  id: string; name: string; price: number;
  category: string; inStock: boolean; searchQuery: string;
}

// ── DataLoader class ───────────────────────────────────────────────
export class DataLoader {
  // Module-level cache — file read once per test run
  private static cache = new Map<string, unknown>();

  // Generic JSON loader with caching
  private static loadJSON<T>(fileName: string): T[] {
    if (!this.cache.has(fileName)) {
      const fp = path.join(DATA_DIR, fileName);
      const raw = JSON.parse(fs.readFileSync(fp, 'utf-8'));
      this.cache.set(fileName, raw);
    }
    return this.cache.get(fileName) as T[];
  }

  // Generic CSV loader with caching
  private static loadCSV<T>(fileName: string): T[] {
    if (!this.cache.has(fileName)) {
      const fp = path.join(DATA_DIR, fileName);
      const rows = parse(fs.readFileSync(fp, 'utf-8'), {
        columns: true,           // header row → property names
        skip_empty_lines: true,  // ignore blank rows
        cast: true,               // auto-cast numbers + booleans
        trim: true,               // strip whitespace
      });
      this.cache.set(fileName, rows);
    }
    return this.cache.get(fileName) as T[];
  }

  // ── Public API ────────────────────────────────────────────────────

  // Login data — filter by tag or get all active
  static getLoginData(tag?: string): LoginCase[] {
    const all = this.loadJSON<LoginCase>('login.json');
    const active = all.filter(d => !d.skip);
    return tag ? active.filter(d => d.tags.includes(tag)) : active;
  }

  // Product data from CSV
  static getProductData(): ProductCase[] {
    return this.loadCSV<ProductCase>('products.csv');
  }

  // Clear cache (useful between test runs in watch mode)
  static clearCache(): void { this.cache.clear(); }
}

Session 28 · Full DDT + POM Framework from Scratch

A production-grade Playwright framework combines Page Object Model (POM) — classes that encapsulate page interactions — with Data-Driven Testing (DDT) — external JSON/CSV/Excel data files in a utils/ folder. Tests become one-liner calls: await loginPage.loginWith(tc). Maintenance is minimal: UI changes → fix one POM class, data changes → fix one JSON file.

Keyword Glossary

BasePageParent class extended by all page objects. Contains: constructor(page:Page), common methods (navigate, waitForLoad, getTitle, getURL). All page objects extend BasePage to inherit shared behavior.
POM action methodA method on a page object that performs a complete user action: loginPage.loginWith({email,password}) internally calls fill, fill, click, waitForURL. Tests call one method, page object handles the steps.
ExcelReaderUtility class that reads .xlsx files using the xlsx npm package. Converts spreadsheet rows to typed JavaScript objects. Used when business data lives in Excel workbooks that are shared via email.

custom fixturesPlaywright fixtures that inject page objects: test.extend({ loginPage: async ({page}, use) => use(new LoginPage(page)) }). Tests declare loginPage as a fixture parameter and get a fully typed page object.
DDT + POM togetherThe combination: data comes from DataLoader (DDT), actions are performed via page objects (POM). for (const tc of data) { test(tc.description, async ({loginPage}) => { await loginPage.loginWith(tc); }) }. Tests are 5 lines, not 20.
Interview One-Liners
POM + DDT together"POM and DDT solve two different maintenance problems: POM means UI changes only require fixing one Page class, not every test. DDT means adding test scenarios only requires editing a data file, not writing new test functions. Together they give you a framework where both UI refactors and new test requirements are handled without touching test logic."
BasePage pattern"BasePage is the parent class all page objects extend — it holds the page fixture reference and common methods like navigate(), getTitle(), waitForLoad(). This prevents duplication: if I add a global navigation method, I add it once in BasePage and all page objects inherit it."
fixture injection"I inject page objects through Playwright fixtures — test.extend({loginPage: async ({page}, use) => use(new LoginPage(page))}). Tests declare loginPage as a parameter and get a fully constructed, ready-to-use page object. It's cleaner than constructing page objects in beforeEach."
Excel reading"When clients provide test data in Excel workbooks (.xlsx), I use the xlsx npm package in an ExcelReader utility — XLSX.utils.sheet_to_json() converts spreadsheet rows directly to JavaScript objects that my DataLoader returns to spec files. The business team maintains data in Excel, tests consume it automatically."
test lines vs framework lines"A well-built DDT+POM framework inverts the ratio: 80% of the code is in page objects, utilities, and data files — 20% is in actual test spec files. Tests become 5-line data-driven loops. When something breaks, you fix the page object or data file, not 50 spec files."
Core Concepts
⬡ Framework Layers
Layer 1: Test Data→JSON / CSV / Excel in test-data/
Layer 2: Data Access→utils/DataLoader.ts + ExcelReader.ts
Layer 3: Page Objects→pages/LoginPage.ts, ProductPage.ts
Layer 4: Fixtures→fixtures/fixtures.ts — inject POMs
Layer 5: Tests→tests/*.spec.ts — thin, data-driven loops
Rule→each layer only talks to the layer below it
⬡ POM Design Principles
Locators as private fields→readonly emailInput = this.page.locator('#email')
Actions as public methods→async loginWith(data) → complete action
Assertions in page object→async assertLoggedIn() → reusable assertion
Return page objects→loginWith() returns DashboardPage (fluent chain)
NO test logic in POM→no test() calls, no describe() in page objects
NO direct API calls→page objects use page fixture only
Best Practices
✓
Always have a BasePage class. Never repeat the constructor(protected page: Page) in every page object — inherit it.
✓
Keep locators as readonly private fields in the page class, not inline strings in methods. Selector changes require only one field update.
✓
Action methods should be complete user flows, not single clicks — loginWith(data) fills email + password + clicks + waits. Tests call one method per action.
✓
Inject page objects via Playwright fixtures (test.extend) — not via beforeEach. Fixtures are composable, typed, and auto-teardown.
✓
Keep spec files thin — 5-10 lines per test. All complex logic belongs in page objects or utils. If a spec file is growing large, extract logic to a page object.
✓
The utils/ExcelReader.ts should convert Excel data to the same format as JSON data — page objects and tests should not know whether data came from JSON or Excel.
Interview Prep
Q1: What is the Page Object Model and what problem does it solve? ▶
POM is a design pattern where each page/component of the application has a corresponding TypeScript class. The class holds: (1) Locators as private readonly properties. (2) Action methods that perform complete user flows. (3) Assertion methods for reusable checks. Problem solved: without POM, if the selector for the email input changes from #email to #user-email, you'd search all 50 spec files and update each one. With POM, you update the single locator in LoginPage.ts and all tests using it are fixed. Tests become readable: await loginPage.loginWith({email, password}) vs 10 lines of fill/click/waitForURL in every test. POM also enables reuse — loginWith() is called from login tests, checkout tests (which need auth), and profile tests.
Q2: How do you read Excel (.xlsx) test data in Playwright? ▶
Install: npm install xlsx. In utils/ExcelReader.ts: import * as XLSX from 'xlsx'; export class ExcelReader { static read(filePath: string, sheetName?: string): Record[]; { const wb = XLSX.readFile(filePath); const wsName = sheetName ?? wb.SheetNames[0]; const ws = wb.Sheets[wsName]; const raw = XLSX.utils.sheet_to_json(ws, {defval:null}); return raw as Record[]; } }. In DataLoader: static getCheckoutData() { const fp = path.join(DATA_DIR,'checkout.xlsx'); const rows = ExcelReader.read(fp, 'CheckoutCases'); return rows.filter(r => !r.skip) as CheckoutTestCase[]; }. The data is then used identically to JSON/CSV: for (const tc of DataLoader.getCheckoutData()) { test(tc.description, ...) }. Tests don't know or care that the data came from an Excel file.
Q3: How do you combine POM with data-driven tests in fixtures? ▶
Create a custom fixtures file: import {test as base} from '@playwright/test'; import {LoginPage} from '../pages/LoginPage'; import {DashboardPage} from '../pages/DashboardPage'; import {DataLoader} from '../utils/DataLoader'; type Fixtures = { loginPage: LoginPage; dashboardPage: DashboardPage; loginData: LoginTestCase[]; }; export const test = base.extend({ loginPage: async ({page}, use) => use(new LoginPage(page)), dashboardPage: async ({page}, use) => use(new DashboardPage(page)), loginData: async ({}, use) => use(DataLoader.getLoginData()), }); export {expect} from '@playwright/test'. In spec file: import {test, expect} from '../fixtures/fixtures'; test.describe('Login DDT+POM', () => { test('login scenarios', async ({loginPage, dashboardPage, loginData}) => { for (const tc of loginData) { await loginPage.loginWith(tc); if (tc.shouldPass) await dashboardPage.assertLoggedIn(tc.role); } }); }).
Q4: How do you structure a BasePage class? ▶
BasePage is the foundation all page objects extend: export abstract class BasePage { constructor(protected readonly page: Page) {} async navigate(path: string): Promise { await this.page.goto(path); } async getTitle(): Promise { return this.page.title(); } async getURL(): string { return this.page.url(); } async waitForLoad(): Promise { await this.page.waitForLoadState('networkidle'); } async getErrorMessage(): Promise { const el = this.page.locator('.error-message'); await el.waitFor({state:'visible', timeout:3000}).catch(()=>{}); return el.textContent(); } }. All page objects: export class LoginPage extends BasePage { private readonly emailInput = this.page.locator('#email'); async loginWith(data: LoginCase) { await this.navigate('/login'); await this.emailInput.fill(data.email); ... } }. Protected page gives all pages access to the Playwright page fixture. Abstract prevents instantiating BasePage directly.
Q5: What makes a DDT+POM framework enterprise-grade? ▶
Enterprise-grade checklist: (1) BasePage with shared methods — no duplication across page objects. (2) DataLoader with caching — one file read per run, not per test. (3) Data validation — DataLoader validates required fields and throws descriptive errors. (4) Type safety everywhere — TypeScript interfaces for all data, typed method signatures. (5) Custom fixtures — page objects injected via test.extend, not constructed in beforeEach. (6) Environment-aware data — DataLoader reads TEST_ENV to select the right data file. (7) Tag filtering — getLoginData('smoke') for fast CI, getLoginData() for full run. (8) Multiple data formats — JSON, CSV, and Excel all handled in DataLoader, transparent to tests. (9) Thin spec files — tests are 5-10 lines, all logic in page objects/utils. (10) Consistent structure — every page, util, and data file follows the same pattern making it easy for new team members to contribute.

// pages/BasePage.ts
// PURPOSE: Parent class for all page objects
// All page objects extend this — shared methods inherited by all

import { Page, Locator } from '@playwright/test';

export abstract class BasePage {

  // Protected: accessible in child page objects
  // Readonly: page reference never changes after construction
  constructor(protected readonly page: Page) {}

  // ── Navigation ────────────────────────────────────────────────────
  async navigate(path: string): Promise<void> {
    await this.page.goto(path);
  }

  // ── Page Info ─────────────────────────────────────────────────────
  async getTitle(): Promise<string> { return this.page.title(); }
  getURL(): string                   { return this.page.url(); }

  // ── Wait helpers ──────────────────────────────────────────────────
  async waitForLoad(): Promise<void> {
    await this.page.waitForLoadState('networkidle');
  }

  async waitForURL(pattern: string | RegExp): Promise<void> {
    await this.page.waitForURL(pattern);
  }

  // ── Error / Success messages ──────────────────────────────────────
  async getErrorMessage(): Promise<string | null> {
    const el = this.page.locator('.error-message, .alert-error, .toast-error');
    await el.waitFor({ state: 'visible', timeout: 3000 }).catch(() => {});
    return el.textContent().catch(() => null);
  }

  async getSuccessMessage(): Promise<string | null> {
    const el = this.page.locator('.success-message, .alert-success, .toast-success');
    await el.waitFor({ state: 'visible', timeout: 5000 }).catch(() => {});
    return el.textContent().catch(() => null);
  }

  // ── Screenshot helper ─────────────────────────────────────────────
  async screenshot(name: string): Promise<Buffer> {
    return this.page.screenshot({ path: `screenshots/${name}.png`, fullPage: true });
  }
}

import { Page, expect } from '@playwright/test';
import { BasePage }     from './BasePage';
import { DashboardPage } from './DashboardPage';
import type { LoginCase } from '../utils/DataLoader';

export class LoginPage extends BasePage {

  // ── Locators as readonly private fields ───────────────────────────
  // Change selector in ONE place — all methods use the field
  private readonly emailInput    = this.page.locator('#email');
  private readonly passwordInput = this.page.locator('#password');
  private readonly submitBtn     = this.page.getByRole('button', { name: /sign in/i });
  private readonly errorMsg      = this.page.locator('.login-error');
  private readonly rememberMe    = this.page.locator('#remember-me');

  // ── Action: navigate to login page ────────────────────────────────
  async goto(): Promise<void> {
    await this.navigate('/login');
    await expect(this.emailInput).toBeVisible();
  }

  // ── Action: complete login flow with data object ──────────────────
  // Returns DashboardPage — fluent chaining pattern
  async loginWith(data: Pick<LoginCase, 'email' | 'password'>): Promise<DashboardPage> {
    await this.emailInput.fill(data.email);
    await this.passwordInput.fill(data.password);
    await this.submitBtn.click();
    return new DashboardPage(this.page);  // return next page
  }

  // ── Action: login with "remember me" checked ──────────────────────
  async loginWithRememberMe(data: Pick<LoginCase, 'email' | 'password'>): Promise<DashboardPage> {
    await this.rememberMe.check();
    return this.loginWith(data);
  }

  // ── Assertion: verify error message shown ─────────────────────────
  async assertErrorMessage(expectedMsg: string): Promise<void> {
    await expect(this.errorMsg).toContainText(expectedMsg);
  }

  // ── Assertion: verify form is visible (on login page) ────────────
  async assertOnLoginPage(): Promise<void> {
    await expect(this.page).toHaveURL(/login/);
    await expect(this.emailInput).toBeVisible();
  }

// utils/ExcelReader.ts
// PURPOSE: Read .xlsx / .xls test data files
// npm install xlsx

import * as XLSX from 'xlsx'; // npm install xlsx
import path from 'path';

export class ExcelReader {

  // ── Read all rows from first sheet (or named sheet) ───────────────
  static read<T = Record<string, unknown>>(
    filePath: string,
    sheetName?: string
  ): T[] {
    const workbook = XLSX.readFile(filePath);

    // Use named sheet or default to first sheet
    const targetSheet = sheetName ?? workbook.SheetNames[0];
    const worksheet  = workbook.Sheets[targetSheet];

    if (!worksheet) {
      throw new Error(`Sheet "${targetSheet}" not found in ${filePath}. Available: ${workbook.SheetNames.join(', ')}`);
    }

    // sheet_to_json: row 1 = headers, rows 2+ = data objects
    const rows = XLSX.utils.sheet_to_json<T>(worksheet, {
      defval: null,      // empty cells → null (not undefined)
      raw: false,         // formatted strings not raw numbers
    });

    console.log(`📊 ExcelReader: loaded ${rows.length} rows from "${targetSheet}"`);
    return rows;
  }

  // ── List all sheet names in a workbook ────────────────────────────
  static listSheets(filePath: string): string[] {
    return XLSX.readFile(filePath).SheetNames;
  }
}

test.describe('✅ Positive Login (DDT + POM)', () => {

  for (const tc of positives) {

    // Full test in 6 lines using POM action methods
    test(`[${tc.id}] ${tc.description}`,
      async ({ loginPage, dashboardPage }) => {

      await loginPage.goto();

      // One method call — all fill/click/wait encapsulated in LoginPage
      await loginPage.loginWith(tc);

      // Dashboard assertions from DashboardPage methods
      await dashboardPage.assertLoggedIn();
      await dashboardPage.assertUserRole(tc.role);
      await expect(loginPage.getURL()).toContain(tc.expectedURL);
    });
  }
});

test.describe('❌ Negative Login (DDT + POM)', () => {

  for (const tc of negatives) {

    test(`[${tc.id}] ${tc.description}`,
      async ({ loginPage }) => {

      await loginPage.goto();
      await loginPage.loginWith(tc);

      // POM assertion methods — no expect() in tests!
      await loginPage.assertOnLoginPage();
      if (tc.expectedMsg) {
        await loginPage.assertErrorMessage(tc.expectedMsg);
      }
    });
  }
});
