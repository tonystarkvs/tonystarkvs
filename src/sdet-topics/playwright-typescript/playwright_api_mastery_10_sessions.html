Key terms glossary
APIRequestContext
Playwright's built-in HTTP client. Created per test via the request fixture or playwright.request.newContext().
request fixture
Auto-provided fixture in every test — gives you a pre-configured HTTP client with baseURL, headers etc. from your config.
APIResponse
The object returned by request.get/post/put/delete. Has .status(), .json(), .text(), .headers(), .ok() methods.
baseURL
Set in playwright.config.ts use block. All relative paths are resolved against it — e.g. request.get('/posts') becomes the full URL.
schema validation
Verifying that a JSON response has the exact shape (keys, types) your contract specifies. Use zod or ajv libraries.
extraHTTPHeaders
Default headers added to every request — e.g. Authorization, Content-Type, x-api-key.
response.ok()
Returns true if status is 200–299. Convenience shortcut for expect(response.status()).toBe(200).

Session 3 — Real CRUD tests against JSONPlaceholder
GET — fetch a single post
// tests/api/posts.spec.ts import { test, expect } from '@playwright/test'; test.describe('Posts API — GET', () => { test('GET /posts returns 100 posts', async ({ request }) => { const response = await request.get('/posts'); expect(response.status()).toBe(200); expect(response.ok()).toBeTruthy(); const posts = await response.json(); expect(posts).toHaveLength(100); expect(posts[0]).toHaveProperty('id'); expect(posts[0]).toHaveProperty('title'); expect(posts[0]).toHaveProperty('body'); expect(posts[0]).toHaveProperty('userId'); }); test('GET /posts/1 returns correct post', async ({ request }) => { const response = await request.get('/posts/1'); expect(response.status()).toBe(200); const post = await response.json(); expect(post.id).toBe(1); expect(post.userId).toBe(1); expect(typeof post.title).toBe('string'); expect(post.title.length).toBeGreaterThan(0); }); test('GET /posts/999 returns 404 for non-existent post', async ({ request }) => { const response = await request.get('/posts/999'); expect(response.status()).toBe(404); }); });
POST — create a resource
test('POST /posts creates a new post', async ({ request }) => { const newPost = { title: 'Playwright API Testing', body: 'This is an automated test post', userId: 1, }; const response = await request.post('/posts', { data: newPost, // automatically serialized to JSON }); expect(response.status()).toBe(201); const created = await response.json(); // JSONPlaceholder echoes back your data + assigns id: 101 expect(created.id).toBe(101); expect(created.title).toBe(newPost.title); expect(created.body).toBe(newPost.body); expect(created.userId).toBe(newPost.userId); });
PUT — full update
test('PUT /posts/1 updates the full post', async ({ request }) => { const updatedPost = { id: 1, title: 'Updated Title', body: 'Updated body content', userId: 1, }; const response = await request.put('/posts/1', { data: updatedPost }); expect(response.status()).toBe(200); const result = await response.json(); expect(result.title).toBe('Updated Title'); expect(result.id).toBe(1); });
PATCH — partial update
test('PATCH /posts/1 partially updates a post', async ({ request }) => { const response = await request.patch('/posts/1', { data: { title: 'Only title changed' }, }); expect(response.status()).toBe(200); const result = await response.json(); expect(result.title).toBe('Only title changed'); // body and userId remain unchanged (in real APIs) });
DELETE — remove a resource
test('DELETE /posts/1 returns 200', async ({ request }) => { const response = await request.delete('/posts/1'); // JSONPlaceholder returns 200 {} for deletes expect(response.status()).toBe(200); });
Query parameters
test('GET /comments filtered by postId', async ({ request }) => { const response = await request.get('/comments', { params: { postId: 1 }, // adds ?postId=1 to URL }); expect(response.status()).toBe(200); const comments = await response.json(); expect(comments).toHaveLength(5); comments.forEach((c: any) => expect(c.postId).toBe(1)); });
Session 4 — Assertions — every pattern you need
Status & response object assertions
const response = await request.get('/posts/1'); // Status code expect(response.status()).toBe(200); expect(response.status()).toBeGreaterThanOrEqual(200); expect(response.status()).toBeLessThan(300); // Convenience: ok = status 200-299 expect(response.ok()).toBeTruthy(); // Headers expect(response.headers()['content-type']).toContain('application/json'); // Response time (Playwright exposes timing via request) // Use Date.now() pattern: const start = Date.now(); const r = await request.get('/posts'); const duration = Date.now() - start; expect(duration).toBeLessThan(3000); // must respond in under 3s
JSON body assertions
const post = await response.json(); // Exact value expect(post.id).toBe(1); expect(post.title).toBe('sunt aut facere repellat'); // Type checks expect(typeof post.id).toBe('number'); expect(typeof post.title).toBe('string'); expect(Array.isArray(post.tags)).toBe(true); // Existence expect(post).toHaveProperty('userId'); expect(post).toHaveProperty('id', 1); // key AND value // Partial object match expect(post).toMatchObject({ userId: 1, id: 1, }); // Array assertions const posts = await response.json(); expect(posts).toHaveLength(100); expect(posts.length).toBeGreaterThan(0); // Each item in array posts.forEach((p: any) => { expect(p).toHaveProperty('id'); expect(p).toHaveProperty('userId'); expect(typeof p.title).toBe('string'); });
Schema validation with Zod
// schemas/postSchema.ts import { z } from 'zod'; export const PostSchema = z.object({ id: z.number().positive(), userId: z.number().positive(), title: z.string().min(1), body: z.string().min(1), }); export const PostsArraySchema = z.array(PostSchema); export type Post = z.infer<typeof PostSchema>;
// in your test: import { PostSchema, PostsArraySchema } from '../../schemas/postSchema'; test('response matches Post schema', async ({ request }) => { const response = await request.get('/posts/1'); const body = await response.json(); // Throws detailed ZodError if shape is wrong const result = PostSchema.safeParse(body); expect(result.success).toBeTruthy(); if (!result.success) { console.error(result.error.format()); } }); test('all posts match schema', async ({ request }) => { const body = await (await request.get('/posts')).json(); const result = PostsArraySchema.safeParse(body); expect(result.success).toBeTruthy(); });
Soft assertions (don't stop on first failure)
test('multiple assertions continue after failure', async ({ request }) => { const response = await request.get('/posts/1'); const post = await response.json(); // soft assertions — all run even if one fails expect.soft(response.status()).toBe(200); expect.soft(post.id).toBe(1); expect.soft(post.userId).toBe(1); expect.soft(post).toHaveProperty('title'); expect.soft(post).toHaveProperty('body'); // hard assertion at end — stops test here expect(response.ok()).toBeTruthy(); });
Use toMatchObject() when you only care about SOME fields. Use exact assertions for critical fields like id, status, type.

Session 7 — Fixtures, hooks & API client pattern
API Client — like POM but for APIs
// api-clients/BaseApiClient.ts import { APIRequestContext, APIResponse, expect } from '@playwright/test'; export class BaseApiClient { protected request: APIRequestContext; protected baseUrl: string; constructor(request: APIRequestContext, baseUrl: string = '') { this.request = request; this.baseUrl = baseUrl; } protected async get(path: string, options = {}): Promise<APIResponse> { const response = await this.request.get(path, options); return response; } protected async post(path: string, data: object): Promise<APIResponse> { return this.request.post(path, { data }); } protected async put(path: string, data: object): Promise<APIResponse> { return this.request.put(path, { data }); } protected async patch(path: string, data: object): Promise<APIResponse> { return this.request.patch(path, { data }); } protected async delete(path: string): Promise<APIResponse> { return this.request.delete(path); } async assertStatus(response: APIResponse, expected: number) { expect(response.status()).toBe(expected); } }
// api-clients/PostsApiClient.ts import { APIRequestContext, APIResponse } from '@playwright/test'; import { BaseApiClient } from './BaseApiClient'; export interface Post { id?: number; userId: number; title: string; body: string; } export class PostsApiClient extends BaseApiClient { constructor(request: APIRequestContext) { super(request); } async getAllPosts(): Promise<Post[]> { const res = await this.get('/posts'); await this.assertStatus(res, 200); return res.json(); } async getPostById(id: number): Promise<Post> { const res = await this.get(`/posts/${id}`); await this.assertStatus(res, 200); return res.json(); } async createPost(post: Omit<Post, 'id'>): Promise<Post> { const res = await this.post('/posts', post); await this.assertStatus(res, 201); return res.json(); } async updatePost(id: number, post: Post): Promise<Post> { const res = await this.put(`/posts/${id}`, post); await this.assertStatus(res, 200); return res.json(); } async deletePost(id: number): Promise<void> { const res = await this.delete(`/posts/${id}`); await this.assertStatus(res, 200); } }

Q1. What is APIRequestContext in Playwright and how does it differ from using fetch()?

APIRequestContext is Playwright's built-in HTTP client, fully integrated with the test framework. It automatically picks up baseURL and extraHTTPHeaders from playwright.config.ts, supports all HTTP methods, and returns APIResponse objects with helper methods like .ok(), .json(), .headers(). Unlike fetch(), it integrates with Playwright's fixture system, respects test timeouts, and can share session state with browser contexts — critical for auth flows combining UI and API.

Q2. What is the request fixture and what does it provide?

The request fixture is a built-in Playwright fixture that gives each test its own APIRequestContext pre-configured with baseURL and extraHTTPHeaders from your config. Its scope is per-test — a fresh context for each test ensures isolation. For shared contexts across tests in a suite, create one with playwright.request.newContext() in beforeAll and always dispose() it in afterAll.

Q3. How do you structure an API automation framework in Playwright?

Use the API Client pattern (equivalent of POM for UI). Create a BaseApiClient with generic get/post/put/patch/delete methods. Create resource-specific clients like PostsApiClient that extend it and expose domain methods like createPost(), getPostById(). Wire these into custom fixtures so tests receive typed client objects. Store schemas in a /schemas folder (Zod), test data in /test-data, and helpers in /helpers. Tests then read like business language: const post = await postsClient.createPost({...}).

Q4. How do you handle authentication in API tests?

Three approaches depending on the case. (1) For all tests: set Authorization header in extraHTTPHeaders in playwright.config.ts. (2) For test-specific auth: pass headers in individual request calls. (3) For login flows: use globalSetup to call the login endpoint once, save the token (or cookies via storageState), and reuse it across all tests. Always store tokens in environment variables, never hardcode them. Use dotenv to load .env files.

Q5. What is schema validation and why is it important?

Schema validation verifies that an API response has the exact structure (keys, data types, required fields) that your contract defines. It catches issues like a field being renamed, changing from string to number, or being accidentally removed — issues that don't break a status code check but would break the consuming frontend. Use Zod in TypeScript: define a schema with z.object(), use safeParse() to validate, and assert result.success is true. This acts as a contract test.

Q6. How do you run only API tests and skip UI tests in Playwright?

Use projects in playwright.config.ts. Create a project named 'api-tests' with testMatch pointing to '**/api/**/*.spec.ts'. Run it with --project=api-tests. Alternatively, use folder structure and npx playwright test tests/api/. You can also use --grep to match a tag like @api if you label your test names. For CI, define separate npm scripts like test:api and test:ui pointing to different project names.

Q7. What is the difference between PUT and PATCH in API testing?

PUT is a full replacement — you send the entire resource and it overwrites everything. If you omit a field, it becomes null or is removed. PATCH is a partial update — you send only the fields you want to change and the rest remain. In tests, verify PUT by checking all fields in the response match what you sent. Verify PATCH by checking the changed field updated while other fields remain untouched. Both typically return 200 with the updated resource.

Q8. How do you do data-driven API testing in Playwright?

Use a for...of loop around test() calls with an array of test cases — each case has inputs and expected output. Load test data from JSON files in /test-data using import with resolveJsonModule:true in tsconfig. For dynamic unique data, use @faker-js/faker to generate random but valid payloads. This avoids test pollution and makes tests independent. Label each dynamic test clearly: test(`create post userId=${data.userId}`) so reports are readable.

Q9. How do you mock an API response in Playwright?

For UI tests that make API calls, use page.route(urlPattern, handler). The handler receives the intercepted route and you call route.fulfill() with a custom status, contentType, and body. This lets you test error states (500, 404), slow responses, or specific edge-case payloads without needing the real backend to reproduce those states. For pure API tests, don't mock — test the real API. Mocking is for UI-layer isolation only.

Q10. How do you assert response time / performance in API tests?

Capture Date.now() before and after the request call and compare the difference. Example: const start = Date.now(); await request.get('/posts'); const ms = Date.now() - start; expect(ms).toBeLessThan(2000). This is a basic SLA check. For more precise timing, Playwright's HAR recording captures detailed network timing. Set recordHar in the context config and analyze the HAR file for TTFB, download time etc. Always add performance assertions as soft assertions so they warn rather than block.

Q11. What is the difference between test-level and suite-level fixtures?

Test-level scope (default) creates a fresh fixture instance for every test — perfect for request contexts, page objects, and anything that must be isolated. Worker-level scope (scope: 'worker') creates one instance per parallel worker and shares it across multiple tests in that worker — good for expensive setup like database seeding. Suite-level setup uses beforeAll/afterAll hooks. Always dispose() custom contexts in afterAll or in the fixture teardown (after the use() call) to prevent memory leaks.

Q12. How do you run Playwright API tests in a CI pipeline?

Set environment variables (API_BASE_URL, API_TOKEN etc.) as CI secrets. Use process.env.CI to switch config values: retries:2 on CI vs 0 locally, workers:'50%' to use available cores. Use --reporter=junit to output results.xml for CI dashboards. For large suites, use sharding: run --shard=1/3, 2/3, 3/3 on parallel machines and merge with npx playwright merge-reports. Use blob reporter for merge, then html reporter for the final combined report.
