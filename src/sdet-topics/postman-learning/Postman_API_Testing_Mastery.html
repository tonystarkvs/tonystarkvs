S1 · Postman Basics & HTTP Fundamentals

Postman is a GUI client that lets you craft, send, and inspect HTTP requests without writing code. It speaks the same protocol as curl but adds collections, environments, tests, and team workflows. Mastering HTTP verbs, status codes, and headers is the foundation of every API test you will ever write.

What / Why / When / Where / How

WhatA desktop/web tool to design, send & test HTTP/HTTPS requests against REST, SOAP, GraphQL APIs.
WhyFaster than curl, visual viewer, built-in tests, environments, team sharing via workspaces.
WhenAPI exploration, manual testing, contract validation, backend debugging, before automation.
WhereLocal desktop app, web app, or CLI (Newman) inside CI/CD pipelines.
HowWorkspace → Collection → Request → Method/URL/Headers/Body → Send → assert in Tests tab.

Keyword Glossary

Workspace
Top container for collections, environments, mocks shared with teammates.
Collection
Group of saved requests + tests + scripts, exportable as JSON v2.1.
Request
One HTTP call = method + URL + headers + body + auth + tests.
Response
Server reply containing status, headers, body, cookies, timings.
HTTP Method
Verb: GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS.
Status Code
3-digit result: 1xx info, 2xx ok, 3xx redirect, 4xx client, 5xx server.
Endpoint
Specific URL path like /api/v1/users/42.
Base URL
Reusable host prefix as variable {{baseUrl}}.
Header
Key:value metadata such as Content-Type, Authorization.
Idempotent
Same call repeated = same result. GET, PUT, DELETE are idempotent.

Interview One-Liners

Postman is essentially a friendly UI wrapper around the raw HTTP protocol.
GET is safe and idempotent; POST is neither — that rule clears half the confusion.
A 2xx means success, 4xx blames the client, and 5xx blames the server.
Always store host, token, and IDs as Postman variables — never hard-code them.
Collections turn ad-hoc clicks into a repeatable, shareable test suite.
Headers travel as metadata; the body carries the actual payload.

Best Practices Checklist

  • ✅ Always use {{baseUrl}} variable — never hard-code hosts.
  • ✅ Name requests verb-resource style: GET-User-By-Id.
  • ✅ Group related requests inside a Collection folder.
  • ✅ Add at least a status-code test to every request.
  • ✅ Use the Postman Console to debug headers/variables.
  • ✅ Pin one environment per stage: dev / qa / staging / prod.
  • ✅ Commit collection JSON exports to git for versioning.
  • ✅ Never share personal API tokens — use the team vault.
  • ✅ Document each request in the description field (Markdown).
  • ✅ Pre-fill examples — they auto-generate API docs.

Interview Prep Q&A

Q1. Difference between Postman and curl?
A: Postman is GUI + scripting + collections + team features; curl is single-shot CLI. Postman exports to curl for reproducibility.
Q2. Is GET idempotent? Why?
A: Yes — N calls yield the same server state. POST is not because each typically creates a new resource.
Q3. 401 vs 403?
A: 401 = not authenticated; 403 = authenticated but forbidden (lacks permission).
Q4. Where do path params live?
A: Inside the URL itself like /users/:id. Query params follow ? as key=value pairs.
Q5. How to parameterize a host?
A: Define {{baseUrl}} in Environment and reference it in every request URL.

S2 · Requests, Responses & HTTP Methods

Every REST API revolves around a small set of verbs — GET, POST, PUT, PATCH, DELETE. Each has a different intent, body shape, and expected status code. Knowing which one to use prevents bugs and clears 90% of API interview questions.

What / Why / When / Where / How

WhatHTTP methods are verbs that tell the server what action to perform on a resource.
WhyCorrect verb choice gives predictable behaviour, caching, retries, and clean REST design.
WhenChoose verb at design time per CRUD operation; reuse in Postman tests.
WhereMethod dropdown beside URL in Postman; first line of every HTTP request.
HowGET=read, POST=create, PUT=full replace, PATCH=partial update, DELETE=remove.

Keyword Glossary

GET
Safe & idempotent read; no body; cacheable.
POST
Creates a new resource; not idempotent; returns 201.
PUT
Full replace of resource; idempotent; usually 200 or 204.
PATCH
Partial update; usually not idempotent; 200.
DELETE
Removes resource; idempotent; usually 204 No Content.
HEAD
Same as GET but only headers — existence/size checks.
OPTIONS
Lists allowed methods (CORS preflight).
201 Created
Success status for POST that created a resource.
204 No Content
Success with empty body — typical for DELETE/PUT.
Safe Method
Method that does not modify server state (GET, HEAD, OPTIONS).

Interview One-Liners

PUT replaces the entire resource; PATCH only the fields you send.
DELETE should be idempotent — calling it twice should not error.
POST is the only verb that is neither safe nor idempotent.
A successful DELETE typically returns 204 with no body.
Use OPTIONS to debug CORS preflight failures in browsers.
Always verify both status and body shape — never just status.

Best Practices Checklist

  • ✅ Pick the verb that matches intent — never overload POST for everything.
  • ✅ Return 201 with Location header on POST creation.
  • ✅ Return 204 No Content for successful DELETE.
  • ✅ Make DELETE idempotent — 2nd call should not error.
  • ✅ Use PATCH only for partial updates; PUT for full replacement.
  • ✅ Validate both status code AND response body shape.
  • ✅ Negative-test every endpoint with bad ids and bad payloads.
  • ✅ Document expected status codes per request in description.
  • ✅ Group CRUD verbs into one folder per resource in collection.
  • ✅ Use HEAD for cheap existence/size probes instead of GET.

Interview Prep Q&A

Q1. PUT vs PATCH?
A: PUT replaces the entire resource (idempotent); PATCH updates only sent fields (often not idempotent).
Q2. Why does DELETE return 204?
A: Because there is nothing meaningful to send back after deletion.
Q3. Is POST idempotent?
A: No. Each call usually creates a new resource with a new id.
Q4. What is OPTIONS used for?
A: CORS preflight — the browser asks the server which methods/headers are allowed.
Q5. Safe vs idempotent?
A: Safe = no state change (GET/HEAD). Idempotent = same result on retry (GET/PUT/DELETE).

S3 · Headers, Params, Path Variables & Body Types

Headers describe metadata, query/path params travel in the URL, and the body carries the payload. Postman supports raw, form-data, x-www-form-urlencoded, binary, and GraphQL bodies — each maps to a different Content-Type and shape on the wire.

What / Why / When / Where / How

WhatDifferent ways of attaching data to an HTTP request — header, query, path, body.
WhyEach carries different intent and the server picks it apart accordingly.
WhenPick body type based on API contract — REST usually JSON, upload uses multipart.
WhereHeaders tab, Params tab, Body tab inside Postman request builder.
HowSet Content-Type header to match the body type you choose.

Keyword Glossary

Content-Type
Header that tells server the body media type (application/json, multipart/form-data…).
Accept
Header asking the server which media type the client wants back.
Authorization
Header carrying credentials — Bearer token, Basic, etc.
Query Param
Key=value pairs after ? in the URL — ?page=2&size=10.
Path Variable
Placeholder inside URL — /users/:id replaced at send time.
raw
Postman body mode for free-text payload (JSON, XML, plain text).
form-data
Multipart body for file uploads + simple key-value fields.
x-www-form-urlencoded
HTML-form-style URL-encoded body.
binary
Send a single raw file as request body — used for image/PDF uploads.
multipart
Combines multiple parts (file + JSON metadata) in one request.

Interview One-Liners

Headers are envelope metadata; the body is the letter inside.
Query params are for filters & paging; path params identify a specific resource.
Use form-data when you must upload a file along with text fields.
x-www-form-urlencoded is the default for HTML form submissions.
Always pair raw JSON body with the Content-Type application/json header.
GraphQL still rides on HTTP POST with a JSON body containing query and variables.

Best Practices Checklist

  • ✅ Match Content-Type with body type — always.
  • ✅ Use path params for resource identity, query params for filters.
  • ✅ Never log Authorization headers in console.log.
  • ✅ Keep large uploads in form-data with streaming.
  • ✅ Validate Content-Type of response, not just status.
  • ✅ URL-encode special characters in query params.
  • ✅ Prefer raw JSON for REST; form-data for uploads.
  • ✅ Use {{$guid}} for unique X-Request-Id traces.
  • ✅ Document required vs optional headers per endpoint.
  • ✅ Set default headers at collection level to DRY.

Interview Prep Q&A

Q1. Query vs Path params?
A: Query filters/pages a collection; path identifies a specific resource.
Q2. form-data vs urlencoded?
A: form-data supports files (multipart); urlencoded is plain key=value HTML-form style.
Q3. Why set Content-Type?
A: Server parses body based on it. Wrong header = 415 Unsupported Media Type.
Q4. How to upload a file in Postman?
A: Body → form-data → key type=File → choose file → send.
Q5. What is content negotiation?
A: Client sends Accept; server returns matching representation; mismatch = 406.

S4 · Environments, Variables, Globals & Collections

Variables make collections reusable across environments. Postman has 5 variable scopes — data, local, environment, collection, global — resolved in that priority order. Mastering scopes prevents the classic 'works on my machine' bug.

What / Why / When / Where / How

WhatNamed placeholders inside {{}} that Postman substitutes at send time.
WhyAvoid hard-coding hosts, tokens, IDs; switch env per stage.
WhenAny value that changes across dev/qa/staging/prod or per iteration.
WhereEnvironment quick-look, Variables tab inside collection, globals.
HowDefine key/value once, reference everywhere with {{key}}.

Keyword Glossary

Variable
Named placeholder referenced as {{name}} inside requests/scripts.
Scope
Visibility level: data, local, environment, collection, global.
Environment
Set of variables grouped by stage (dev, qa, prod).
Global
Cross-environment variable available everywhere.
Collection Variable
Variable tied to one collection — survives env switch.
Local Variable
Set in script with pm.variables.set — request-scope only.
Data Variable
Provided by Collection Runner / Newman iterationData.
replaceIn
pm.variables.replaceIn renders a template string.
Initial vs Current
Initial = synced value; Current = local only.
Export
Save collection/environment as JSON for sharing or git.

Interview One-Liners

Variable precedence (highest → lowest): Data > Local > Environment > Collection > Global.
Use {{baseUrl}} pattern to switch hosts in one click.
Never store secrets in Globals — they sync across the team workspace.
pm.variables.replaceIn lets you template any string at runtime.
Initial value is shared; Current value is private to your machine.
Collection variables travel with the collection — perfect for fixtures.

Best Practices Checklist

  • ✅ Never hard-code hosts or tokens — always variable-ize.
  • ✅ Use Environment for stage-specific values, Collection vars for fixtures.
  • ✅ Store secrets only in Current value (not Initial) to avoid syncing.

  • ✅ Use {{$guid}}/{{$timestamp}} for unique request ids.

  • ✅ Use pm.collectionVariables for values shared across requests in one collection.
  • ✅ Avoid Globals for anything sensitive or stage-specific.
  • ✅ Validate variable existence in pre-request before using.

Interview Prep Q&A

Q1. What is variable precedence?
A: Data > Local > Environment > Collection > Global (highest first).
Q2. Initial vs Current value?
A: Initial syncs to teammates; Current stays on your machine — use Current for secrets.
Q3. When to use collection variable?
A: For values tied to the API contract itself (apiVersion, default page size).
Q4. How to template a string in script?
A: Use pm.variables.replaceIn('... {{key}} ...').
Q5. How does Newman pick env?
A: Via the -e flag pointing to an exported environment JSON file.

S5 · Authentication — Basic, Bearer, API Key, OAuth2

Authentication answers 'who are you?' before the API answers 'what do you want?'. Postman supports Basic, API Key, Bearer, OAuth1, OAuth2, AWS Sig, NTLM, Digest and Hawk. Modern REST APIs are 95% Bearer + OAuth2.

What / Why / When / Where / How

WhatMechanisms that prove identity to the API server before granting access.
WhyWithout auth, anyone could read or modify your data.
WhenEvery protected endpoint — set auth at collection level for DRY.
WhereAuthorization tab in Postman; or as a Header via script.
HowPick auth type, supply credentials, Postman injects header before send.

Keyword Glossary

Basic Auth
base64(username:password) sent in Authorization header.
Bearer Token
Authorization: Bearer <jwt or opaque token>.
API Key
Static key passed via header (X-API-Key) or query (?api_key=).
OAuth2
Delegated auth standard with flows: client_credentials, auth_code, password, refresh.
JWT
Signed JSON token: header.payload.signature, often used as Bearer.
Refresh Token
Long-lived token used to mint new access tokens.
Scope
String that limits what a token can do (read:users, write:orders).
Token Expiry
exp claim — once past, server returns 401.
Client Credentials
M2M flow: client_id + client_secret → access_token.
Authorization Code
Browser-redirect flow used by user-facing apps.

Interview One-Liners

Authentication = who you are; Authorization = what you can do.
Always store tokens in environment Current value, never in Initial.
OAuth2 client_credentials is the go-to for server-to-server testing.
A refresh token is just a longer-lived ticket for new access tokens.
Set auth at collection level — children inherit unless overridden.
Validate 401 vs 403 paths separately in your test suite.

S10 · Mock Servers, Monitors, Postman Flows & Advanced Patterns

Mock servers let frontends and tests work against an API before it exists. Monitors run collections on a schedule (uptime canaries). Flows visually wire requests together. Advanced patterns include setNextRequest chaining, dynamic variables, and retry logic.

What / Why / When / Where / How

WhatTools that extend Postman beyond ad-hoc requests — mocks, monitors, flows, chaining.
WhyDecouple frontend/backend, monitor production, build complex test flows visually.
WhenMocks during design, monitors in prod, chaining in regression suites.
WherePostman cloud (mocks + monitors) and runtime (chaining + dynamic vars).
HowAdd Examples → publish Mock; schedule Monitor; use postman.setNextRequest for chains.

Keyword Glossary

Mock Server
Cloud URL that serves your saved Examples as a fake API.
Example
A saved request/response pair used by the mock server.
Monitor
Scheduled Newman-like run that uses your collection.
Postman Flows
Visual canvas to chain requests with logic blocks.
setNextRequest
Override default execution order: postman.setNextRequest('name').
Dynamic Variable
Built-in like {{$randomEmail}}, {{$timestamp}}, {{$guid}}.
Retry
Loop a request until it succeeds — usually via setNextRequest.
Contract Test
Validate provider matches consumer expectations.
Performance Run
Stress run using Postman Runner with virtual users.
Visualize
pm.visualizer.set for custom HTML charts in response panel.

Interview One-Liners

Mocks let frontends ship before backends are ready.
Monitors are your always-on uptime canary.
setNextRequest is the secret to retry and dynamic flow control.
Dynamic variables ({{$randomEmail}}) eliminate flaky duplicate-data tests.
Postman Flows replaces brittle scripts with a visual canvas.
Contract tests catch breaking changes before they hit production.

● 07_RetryLogic.postman.js 
// Retry up to 3 times on flaky endpoint
const tries = +pm.environment.get("tries") || 0;
if (pm.response.code >= 500 && tries < 3) {
    pm.environment.set("tries", tries + 1);
    postman.setNextRequest(pm.info.requestName);   // same request
} else {
    pm.environment.unset("tries");
}

Best Practices Checklist

  • ✅ Use mocks to unblock frontend developers waiting on backend.
  • ✅ Schedule monitors at 5-minute intervals for critical uptime endpoints.
  • ✅ Wire on-failure notifications to your team channel/email.
  • ✅ Use dynamic variables to keep data-driven tests independent.
  • ✅ Limit setNextRequest retries with a counter to prevent infinite loops.
  • ✅ Version Examples in git alongside the collection JSON.
  • ✅ Reserve Flows for complex multi-step business journeys.
  • ✅ Treat contract tests as a separate suite, runnable per provider.
  • ✅ Cap virtual users in performance runs to avoid IP blocking.
  • ✅ Document your chaining order in the collection description.

Interview Prep Q&A

Q1. What is a Postman Mock?
A: Cloud URL that returns your saved Examples — fakes the API for frontend dev.
Q2. What is a Monitor?
A: Scheduled Newman-style runner that pings your collection and alerts on failure.
Q3. How to retry a request?
A: Use postman.setNextRequest with the same request name and a counter.
Q4. What is setNextRequest(null)?
A: Stops the run after the current request — terminates the chain.
Q5. What are dynamic variables?
A: Built-ins like {{$randomEmail}}, {{$timestamp}}, {{$guid}} for uniqueness.
