Interview Q&A
Step 1 What are Postman Variables?
Variables store reusable data (tokens, IDs, URLs). Postman has 5 variable scopes — each overrides the previous.
// Scope hierarchy (highest → lowest priority) Global > Collection > Environment > Data > Local
Step 2 Extract from response using Tests tab
After a request runs, the "Tests" script fires. Parse the JSON body and store values into environment variables.
// In Tests tab (after Login API call) const json = pm.response.json(); // Store token from response body pm.environment.set("authToken", json.token); // Store a nested value pm.environment.set("userId", json.data.user.id); // Store from response header const loc = pm.response.headers.get("Location"); pm.environment.set("createdUrl", loc);
💡 pm.response.json() parses the entire response body as JSON. Always call this before accessing nested fields.
Step 3 Verify the variable was stored
After running the request, click Environment (eye icon, top-right) → you'll see authToken with its current value populated automatically.
Concept Using variables in requests
Reference any variable anywhere in Postman using double-curly syntax: {{variableName}}
In URL
{{baseUrl}}/users/{{userId}}
In Header
Authorization: Bearer {{authToken}}
Pattern Full chain: Login → use token in next request
// Request 1: POST /login → Tests tab const res = pm.response.json(); pm.environment.set("authToken", res.access_token); // Request 2: GET /profile → Headers tab Authorization: Bearer {{authToken}} // Request 3: GET /orders → URL {{baseUrl}}/users/{{userId}}/orders
💡 Postman resolves {{var}} at runtime. If the variable is empty, Postman sends the literal text — watch for that in logs.
Tip Pre-request Scripts for dynamic values
Use the "Pre-request Script" tab to set variables before the request fires.
// Pre-request Script tab const ts = new Date().toISOString(); pm.environment.set("timestamp", ts); // Use {{timestamp}} in your request body
Concept What is Collection Runner?
Collection Runner executes all requests in a collection in order — automatically — with shared variables, data files, and iteration support. No manual clicking.
How to run Step-by-step
1
Open Collection → click the 3-dot menu → "Run Collection"
2
Set Order — drag requests into the correct execution order (Login first, then use token)
3
Select Environment — pick the environment containing your base URL, credentials etc.
4
Iterations — set how many times to loop (useful with data files)
5
Data file (optional) — upload a CSV/JSON to run the same flow for multiple users/inputs
6
Click "Run" — Postman executes every request, passes variables, and shows pass/fail per assertion
Data file CSV example
// users.csv username,password alice,pass123 bob,secret456 // In Pre-request / body use: { "username": "{{username}}", "password": "{{password}}" }
💡 Postman reads one CSV row per iteration. With 3 rows, it runs the full collection 3 times with different data each time.
Core API pm.test() — basic assertion
// Tests tab — runs after response arrives // 1. Status code check pm.test("Status is 200", () => { pm.response.to.have.status(200); }); // 2. Body contains key pm.test("Token exists", () => { const json = pm.response.json(); pm.expect(json.token).to.be.a("string"); }); // 3. Response time pm.test("Responds under 500ms", () => { pm.expect(pm.response.responseTime).to.be.below(500); });
Pattern Validate against environment variable
// Check returned ID matches what we stored pm.test("User ID matches", () => { const json = pm.response.json(); const expected = pm.environment.get("userId"); pm.expect(json.id).to.eql(expected); }); // Schema shape check pm.test("Has required fields", () => { const json = pm.response.json(); pm.expect(json).to.have.all.keys("id", "name", "email"); });
⚠️ Use .eql() for deep equality (objects/arrays). Use .equal() for primitives. Mixing them is a common bug.
Cheatsheet Common assertions
pm.response.to.have.status(201) pm.expect(json.active).to.be.true pm.expect(json.items).to.be.an("array").with.lengthOf(3) pm.expect(json.email).to.include("@") pm.expect(json.score).to.be.within(0, 100) pm.expect(res.headers.get("Content-Type")).to.include("application/json")
Architecture Complete automated flow
Structure your collection as a self-contained pipeline. Each request feeds into the next via environment variables.
Collection: E2E User Flow ├── Request 1: POST /auth/login │ Pre-req : set baseUrl │ Tests : save authToken, userId │ ├── Request 2: POST /users/{{userId}}/profile │ Header : Bearer {{authToken}} │ Tests : save profileId, assert 201 │ ├── Request 3: GET /users/{{userId}}/profile │ Header : Bearer {{authToken}} │ Tests : assert data matches, save │ └── Request 4: DELETE /users/{{userId}} Header : Bearer {{authToken}} Tests : assert 204, cleanup vars
Cleanup Remove variables after test
// Last request Tests tab — cleanup pm.environment.unset("authToken"); pm.environment.unset("userId");
CLI Newman — run from terminal / CI
Newman is the command-line runner for Postman. Run your entire suite in CI/CD with one command.
# Install npm install -g newman # Run collection with environment newman run MyCollection.json \ -e Production.json \ --reporters cli,junit \ --reporter-junit-export results.xml # With data file newman run MyCollection.json \ -e Staging.json \ -d users.csv \ --iteration-count 5
💡 Export collection via: Collection → 3-dot menu → Export. Export environment from Environment → 3-dot menu → Export.
Common interview questions
Q1. How do you pass data between requests in Postman?
Extract values from a response in the Tests tab using pm.environment.set("key", value). Reference them in subsequent requests with {{key}} in the URL, headers, or body. Postman resolves these at runtime before sending the next request.
Q2. Difference between environment and global variables?
Global variables are available across all collections and environments — use for truly shared constants. Environment variables are scoped to a selected environment (Dev/Staging/Prod) — use for configs that change per environment like baseUrl, credentials, tokens.
Q3. How does Collection Runner help in automation?
It executes all requests in a collection sequentially without manual intervention, supports iteration with CSV/JSON data files, shares environment variables across requests, and shows a pass/fail report for every assertion. It mimics end-to-end user flows in a single click.
Q4. What is Newman and when would you use it?
Newman is Postman's CLI runner. You'd use it to integrate Postman tests into a CI/CD pipeline (Jenkins, GitHub Actions, GitLab CI). It accepts the exported collection JSON and environment JSON, and outputs results as JUnit XML or HTML for test reports.
Q5. How do you validate response schema in Postman?
Use pm.expect(json).to.have.all.keys("id","name") for field presence, or integrate the tv4/ajv library in the Tests tab for full JSON Schema validation by requiring it from CDN or bundling it in a pre-request script.
Q6. What is the Pre-request Script and when is it useful?
It runs before the request is sent. Useful for: generating timestamps or UUIDs dynamically, refreshing an OAuth token if expired, computing an HMAC signature, or setting up test data the request needs in its body.
