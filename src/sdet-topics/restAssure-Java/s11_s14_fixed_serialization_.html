🌐 Overview — What / Why / When / Where / How
❓
WHAT
Serialization = converting a Java object (Map/POJO) into JSON string. Deserialization = JSON string → Java object. Both are needed for every POST/PUT test.
💡
WHY
Avoids error-prone string concatenation. Type-safe, IDE-friendly, maintainable for any nesting depth. Jackson auto-handles serialization.
⏰
WHEN
Every POST/PUT/PATCH request needing a JSON body. From a 3-field flat object up to 4-level deep nested order payloads.
📍
WHERE
Inside REST Assured's given().body(). Jackson sits between your Java code and the HTTP request, converting silently.
⚙️
HOW
5 approaches: raw String, HashMap, JSONObject, POJO+Jackson, POJO+Gson. Each shown simple → complex in this module.

🎤 Interview One-Liners
Serialization = converting a Java object (Map/POJO) into a JSON string to send as an HTTP request body via REST Assured's body().
Deserialization = converting a JSON response string back into a typed Java object using Jackson's readValue() or REST Assured's as(Class).
HashMap approach = using Map<String,Object> as the body — REST Assured + Jackson auto-converts it to JSON when contentType is application/json.
contentType("application/json") = MANDATORY trigger — without it REST Assured won't serialize a Map/POJO; it sends raw form-encoded params instead.
Nested JSON in HashMap = put another Map as the value of a key — Jackson serializes inner Map as a nested JSON object automatically.
Array in HashMap = put a List<String> or List<Map> as the value — Jackson serializes it as a JSON array of strings or array of objects.

✅ Best Practices
✔ Always set contentType("application/json") — without it, REST Assured won't trigger Jackson serialization for Map/POJO bodies.
✔ Use Map<String,Object> with type parameters — not raw Map — to avoid ClassCastException at runtime.
✔ Use log().body() in the given() phase during development to print exact JSON being sent and verify structure.
✔ For lists in HashMap: use Arrays.asList() for fixed immutable lists, new ArrayList<>() when you need to add items dynamically.
✔ Never use raw String JSON for complex bodies — switch to HashMap or POJO beyond 3 fields.
🎯 SDET Interview Q&A
Q1: Why does contentType("application/json") matter for HashMap serialization? +
Without it, REST Assured doesn't know you want JSON — it may send the Map as URL-encoded form parameters (key=value&key2=value2). Setting contentType("application/json") signals REST Assured to delegate to Jackson's ObjectMapper, which converts the Map into a proper JSON string for the request body.
Q2: How do you represent a JSON array of objects using HashMap? +
Create each object as Map<String,Object>. Add all objects to a List<Map<String,Object>>. Put the list as a value in the parent map: parentMap.put("items", listOfMaps). Jackson serializes List<Map> as a JSON array of objects: "items":[{...},{...}].
Q3: What serialization library does REST Assured use by default? +
REST Assured auto-detects the library on classpath in priority order: Jackson (jackson-databind) first, then Gson, then JAXB for XML. If Jackson is present, all Map/POJO body serialization uses Jackson's ObjectMapper automatically. Configure explicitly with RestAssured.config().objectMapperConfig().
Q4: Difference between Arrays.asList() and new ArrayList()? +
Arrays.asList() returns a fixed-size List — you cannot add/remove elements (throws UnsupportedOperationException). Use it for complete, hardcoded lists. new ArrayList<>() is mutable — you can add, remove, sort. For JSON array building in tests, use ArrayList when items are added dynamically.
Q5: How do you verify the exact JSON being sent in a REST Assured request? +
Add .log().body() in the given() phase — it prints the serialized request body to console before sending. For full request including headers: .log().all(). For CI pipelines: .log().ifValidationFails() prints only when assertions fail, keeping build output clean.
🌐 Overview — JSONObject & Gson
❓
WHAT
org.json's JSONObject/JSONArray = explicit JSON builders with chainable put(). Gson = Google's library that converts Java POJOs to JSON via toJson().
💡
WHY
JSONObject gives explicit JSON construction with preserved key order. Gson is lighter than Jackson and handles nested POJOs without many annotations.
⏰
WHEN
Use JSONObject for explicit JSON building. Use Gson when your team uses it as standard or Jackson is unavailable.
📍
WHERE
JSONObject → body(obj.toString()). Gson → gson.toJson(pojo) → body(jsonString). Both pass String to body().
⚙️
HOW
org.json: new JSONObject().put(k,v). Gson: new Gson().toJson(obj). GsonBuilder for custom options.
🎤 Interview One-Liners
JSONObject = class from org.json library that builds JSON objects via put(key,value) — call toString() to get the final JSON string for body().
JSONArray = org.json class for building JSON arrays — put(value) appends elements, toString() gives the array string.
Gson = Google's open-source Java serialization library — simpler than Jackson, fewer annotations needed for basic POJO mapping.
toJson() = Gson method converting Java object → JSON string (serialization). Equivalent to Jackson's writeValueAsString().
fromJson() = Gson method converting JSON string → Java object (deserialization). Equivalent to Jackson's readValue().
@SerializedName = Gson annotation mapping a Java field name to a different JSON key — equivalent to Jackson's @JsonProperty.

✅ Best Practices
✔ Always call body(jsonObj.toString()) for JSONObject — be explicit about converting to String before passing to body().
✔ Create one Gson instance per test class — Gson is thread-safe after construction, reuse saves memory.
✔ Use @SerializedName("json_key") when Java camelCase fields map to JSON snake_case keys (e.g. productName → "product_name").
✔ Use GsonBuilder().serializeNulls() when the API requires null fields to be present in the request body.
✔ Prefer Jackson over Gson for REST Assured — Jackson is auto-detected; Gson requires manual toJson() calls in every test.
🎯 SDET Interview Q&A
Q1: Difference between JSONObject (org.json) and HashMap for request body? +
HashMap is Java's built-in map — REST Assured+Jackson auto-serializes it. JSONObject is org.json's explicit builder — you call toString() to get JSON. JSONObject preserves insertion order; HashMap does not. JSONObject provides getInt/getString helpers. If you have Jackson on classpath, use HashMap — fewer dependencies.
Q2: How do you configure REST Assured to use Gson instead of Jackson? +
Configure globally: RestAssured.config = RestAssuredConfig.config().objectMapperConfig(new ObjectMapperConfig(GSON)). Or simply call gson.toJson(pojo) manually and pass the String to body() — this always works regardless of REST Assured config, and is the most reliable approach.
Q3: How does @SerializedName work in Gson? +
@SerializedName("json_key") on a POJO field tells Gson to use that name when serializing to JSON and when deserializing from JSON. Example: @SerializedName("first_name") String firstName — Gson serializes as "first_name":"value" and maps "first_name" from JSON to firstName. Equivalent to Jackson's @JsonProperty.
Q4: What does serializeNulls() do in GsonBuilder? +
By default, Gson omits null fields from JSON. GsonBuilder().serializeNulls() overrides this and includes them as "field": null. Use when the API's schema validation requires all fields to be present even if null — without it those fields simply don't appear in the request body.
Q5: When would you choose Gson over Jackson? +
Gson when: existing project already uses Gson, need simpler code with fewer annotations, Android project (Gson is lighter). Jackson when: new REST Assured project, need @JsonIgnore/@JsonInclude/@JsonCreator annotations, need REST Assured auto-detection, or need advanced polymorphic deserialization.
🌐 Overview — POJO + Jackson Full Mastery
❓
WHAT
POJO + Jackson = industry standard. Design Java classes mirroring JSON structure. Jackson auto-serializes via REST Assured body() and deserializes via as(Class).
💡
WHY
Type safety, IDE autocomplete, refactoring support. Jackson annotations give fine-grained control over every field's JSON behavior.
⏰
WHEN
Use POJO for any recurring request structure, data-driven tests, and complex nested APIs. Essential for enterprise test frameworks.
📍
WHERE
POJO classes in src/test/java/model/. In tests: given().body(pojo). Response: extract().as(MyClass.class) or TypeRef for lists.
⚙️
HOW
Create POJO mirroring JSON. Add Jackson annotations. Pass to body() or extract with as(). Jackson's ObjectMapper handles all conversion.
🎤 Interview One-Liners
@JsonProperty("key") = maps Java field name to a different JSON key — e.g. Java field firstName ↔ JSON key "first_name".
@JsonIgnore = Jackson annotation to completely exclude a field from both serialization (request) and deserialization (response).
@JsonIgnoreProperties(ignoreUnknown=true) = class-level annotation to silently ignore extra JSON fields not present in the POJO.
@JsonInclude(NON_NULL) = exclude null fields from JSON output during serialization — keeps request body clean.
@JsonAlias = accepts multiple JSON key names for deserialization — useful when API migrates field names.
TypeRef = REST Assured generic type token for deserializing List<User> — solves Java type erasure problem.

✅ Best Practices
✔ Always add @JsonIgnoreProperties(ignoreUnknown = true) to every POJO — prevents test failures when the API adds new fields.
✔ Always provide a no-arg constructor in POJO — Jackson requires it for deserialization via reflection.
✔ Use TypeRef<List<MyClass>> not List.class for deserializing JSON arrays — plain List.class loses generic type info.
✔ Use @JsonInclude(NON_NULL) to keep request bodies clean — avoids sending fields the API didn't ask for.
✔ Use ObjectMapper.writerWithDefaultPrettyPrinter() during debugging to confirm exact JSON payload structure.
🎯 SDET Interview Q&A
Q1: Why does Jackson need a no-arg constructor in POJO? +
Jackson's default deserialization creates an instance using the no-arg constructor via reflection, then populates fields using setters. Without it: InvalidDefinitionException. Workaround: use @JsonCreator on a parameterized constructor, annotating each parameter with @JsonProperty to map JSON keys to constructor arguments.
Q2: What is Java type erasure and how does TypeRef solve it? +
Type erasure means generic types (List<User>) lose their parameter at runtime — JVM sees raw List. So List<User>.class doesn't exist. TypeRef creates an anonymous subclass that preserves generic type info via Class hierarchy reflection. REST Assured uses this to tell Jackson to deserialize into List<User> specifically, not just List.
Q3: How do @JsonIgnore and @JsonIgnoreProperties differ? +
@JsonIgnore is field-level — marks ONE field to skip entirely in both directions. @JsonIgnoreProperties(ignoreUnknown=true) is class-level — silently skips JSON fields that have no matching POJO field (defensive deserialization). Different problems: @JsonIgnore controls what you send; @JsonIgnoreProperties controls what you accept from the API.
Q4: What is ObjectNode and when would you use it over POJO? +
ObjectNode is Jackson's dynamic JSON tree node — you build JSON without creating a POJO class. Use ObjectNode when JSON structure changes at runtime, for one-off payloads, or when the schema is variable. Use POJO when the structure is fixed and recurring — POJOs give compile-time safety, IDE support, and refactoring tools.
Q5: How does @JsonAlias differ from @JsonProperty for deserialization? +
@JsonProperty("key") sets ONE mapping used for both serialization (output key) and deserialization (input key). @JsonAlias({"key1","key2"}) adds ALTERNATIVE names for DESERIALIZATION ONLY — the field still serializes with its primary name. Use @JsonAlias when an API is migrating field names and may send either "name" or "full_name" for the same field.

Bottom-up approach = always create the deepest nested object first, then wrap upward to root — prevents NullPointerException during build.
Array of objects = JSON pattern [{...},{...}] — Java: List<Map> or List<POJO> depending on which approach is used.
Mixed nesting = same JSON has nested objects AND arrays at the same level — e.g. order has customer (object) AND items (array of objects).

ObjectNode+ArrayNode = Jackson's tree model for dynamic JSON — no POJO needed, build structures at runtime with createObjectNode/createArrayNode.

✅ Best Practices
✔ Always build complex JSON bottom-up — deepest nested object first, then wrap outward to root.
✔ For 3+ levels use POJO+Jackson — HashMap code becomes unreadable and error-prone at deep nesting.
✔ Use ObjectNode/ArrayNode when JSON structure is dynamic at runtime — avoids creating rigid POJO classes for variable schemas.
✔ For deep response validation use both body() assertions AND jsonPath() extraction — assert critical fields and extract for reporting.
✔ For double-array access: items[0].tags[0] gets first element of items, then first element of its tags array.
🎯 SDET Interview Q&A
Q1: How do you navigate a 4-level deep JSON path in REST Assured? +
Use dot notation in body(): body("customer.address.geo.lat", equalTo(28.6)). Each dot = one level deeper. For arrays: body("items[0].tags[0]", equalTo("java")) — [0] accesses array element by zero-based index. REST Assured evaluates this as a GPath expression using Groovy's navigation engine internally.
Q2: ObjectNode vs POJO — when to choose which? +
POJO for fixed/recurring structures — type-safe, IDE support, reusable. ObjectNode for dynamic structures where keys/values change at runtime, or one-off payloads where creating a dedicated class is overkill. Both integrate with REST Assured body(). ObjectNode avoids extra class files for simple, non-recurring JSON shapes.
Q3: How do you assert a specific element inside a nested array? +
Use array indexing in GPath: body("items[0].productId", equalTo(101)) accesses first element's productId. For "any element matches": body("items.productId", hasItem(101)). For "all elements": body("items.qty", everyItem(greaterThan(0))). For filtered: body("items.findAll{it.qty>1}.name", hasItem("REST Assured Pro")).
Q4: Can you mix POJOs and Maps in the same request body? +
Yes — Jackson handles mixed structures seamlessly. Example: a root Map containing a POJO as a value: map.put("address", addressPojo). Jackson serializes the root Map as a JSON object, and when it encounters the AddressModel POJO value, it recursively serializes it using the POJO's getters. Mix freely based on what's convenient.
Q5: What is the difference between ArrayNode.add() and ObjectNode.set() in Jackson? +
ArrayNode.add(value) appends an element to the end of the array — used to build JSON arrays. It's chainable (returns ArrayNode). ObjectNode.set(key, node) embeds a JsonNode (ObjectNode or ArrayNode) as the value for a key in the object — used for nesting. ObjectNode.put(key, value) is for primitive values (String, int, boolean); set() is for nested nodes.
