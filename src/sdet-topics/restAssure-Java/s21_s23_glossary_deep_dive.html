Overview — What / Why / When / Where / How
📦
What
JSON, POJO, HTTP, and Web Services are the four foundational pillars every SDET must master — they are the data format, Java object model, transport layer, and service concept behind every REST Assured test.
💡
Why
Without understanding these, you're copy-pasting code blindly. Every REST Assured line you write is grounded in one of these four concepts. Interviewers test all four.
⏰
When
Applied every single day — JSON is the body of every POST request; POJO is your deserialization target; HTTP is the protocol behind every given().when().then() call.
📍
Where
JSON lives in request/response bodies. POJO lives in your test Java project. HTTP lives between your test and the server. Web Services are the deployed APIs you test.
⚙️
How
REST Assured ties them together: you write a POJO → serialize to JSON → send over HTTP → receive JSON → deserialize back to POJO. That's the complete test data pipeline.
Keyword Glossary
JSON
JavaScript Object Notation — lightweight, human-readable text format using key-value pairs. The dominant data format for REST APIs.
POJO
Plain Old Java Object — a simple Java class with private fields, public getters/setters, and a no-arg constructor. Maps to JSON fields by name.
HTTP
HyperText Transfer Protocol — application-layer protocol defining how clients send requests and servers send responses over TCP/IP networks.
Web Service
A software system exposed over a network via a standard protocol (HTTP/HTTPS) allowing machine-to-machine communication.
Serialization
Converting a Java POJO → JSON string for sending as an HTTP request body. Jackson's writeValueAsString().
Deserialization
Converting a JSON response string → Java POJO. Jackson's readValue() or REST Assured's as(Class).
HTTP Request
Message from client to server: method (GET/POST) + URL + headers + optional body. REST Assured's given().when() builds this.
HTTP Response
Message from server to client: status code + headers + body. REST Assured's then() asserts on this.
TCP/IP
Transmission Control Protocol / Internet Protocol — the network-layer protocol HTTP sits on top of. Handles reliable data delivery.
HTTPS
HTTP Secure — HTTP encrypted with TLS/SSL. All production APIs use HTTPS on port 443. REST Assured handles it transparently.
Concept 1 — JSON: Structure, Types, and Nesting
JSON complete structure reference — every type with SDET context
JSON = text-based data interchange format Rule 1: Keys must be STRINGS in double quotes Rule 2: Values can be 6 types: string, number, boolean, null, object, array { "id": 1, ← NUMBER (no quotes) → Java: int/long/double "name": "Chitti", ← STRING (double quotes) → Java: String "active": true, ← BOOLEAN (no quotes) → Java: boolean "score": 9.75, ← DECIMAL → Java: double/float/BigDecimal "notes": null, ← NULL → Java: null reference "roles": ["admin","sdet"], ← ARRAY → Java: List<String> "address": { ← OBJECT → Java: nested POJO or Map "city": "Delhi", "geo": { "lat": 28.6 } ← NESTED OBJECT (3 levels) }, "orders": [ ← ARRAY OF OBJECTS → List<OrderModel> { "id":1, "total":599.0 }, { "id":2, "total":299.0 } ] } REST Assured body() GPath expressions for above JSON: body("id", equalTo(1)) body("roles[0]", equalTo("admin")) body("address.city", equalTo("Delhi")) body("address.geo.lat",equalTo(28.6f)) body("orders[0].total",equalTo(599.0f))
Concept 2 — POJO: Design Rules and Jackson Mapping
POJO ↔ JSON mapping — the complete mental model
POJO RULES (Jackson requires these to work): 1. Private fields ← encapsulation 2. Public getters + setters ← Jackson uses getters for serialization 3. No-arg constructor ← Jackson needs this to create instance 4. @JsonIgnoreProperties ← defensive: ignore unknown JSON fields JSON ←→ Java field name matching rules: JSON key "name" ← exact match → Java field String name JSON key "user_id" ← no match → Java field @JsonProperty("user_id") int userId JSON key "firstName" ← match → Java field String firstName (camelCase matches) POJO LIFECYCLE in a REST Assured test: REQUEST SIDE (Serialization): UserModel user = new UserModel("Chitti", "c@test.com"); given().body(user) ← Jackson: UserModel → {"name":"Chitti","email":"c@test.com"} .when().post("/users"); RESPONSE SIDE (Deserialization): .then().extract().as(UserModel.class); ← Jackson: {"id":1,"name":"Chitti"} → UserModel{id=1, name="Chitti"} WHY POJO > HashMap for SDET tests: HashMap: (String) map.get("address") → runtime ClassCastException risk POJO: user.getAddress().getCity() → compile-time safety + IDE autocomplete

Concept 4 — Web Services: Types and How They Relate
Web Services ecosystem — where REST Assured fits
WEB SERVICE = any software system exposed over a network via HTTP Key property: language-agnostic — Java test calls Python API = fine Types of Web Services: 1. REST API → uses HTTP methods, JSON/XML, URL as resource identifier 2. SOAP API → XML envelope, WSDL contract, strict protocol 3. GraphQL API→ single endpoint, client chooses fields, over HTTP/POST 4. gRPC → binary protocol (Protocol Buffers), bidirectional streaming Web Service components: Endpoint = specific URL a service exposes: /api/v1/users/{id} Resource = the entity the service manages: User, Product, Order Client = caller (browser, mobile app, test script) Server = machine hosting the service (EC2, GCP, Kubernetes pod) Contract = agreed schema / OpenAPI spec between client and server REST Assured position in the Web Services ecosystem: Test Script (Java) → REST Assured → HTTP → Web Service → Database ↑ Acts as the HTTP CLIENT — exactly like a browser/app
Interview One-Liners
JSON = JavaScript Object Notation — a text-based key-value format where keys must be strings and values can be string, number, boolean, null, object, or array.
POJO = Plain Old Java Object — a simple Java class with private fields + getters + setters + no-arg constructor. Jackson maps JSON field names to POJO field names automatically.
HTTP = HyperText Transfer Protocol — the application-layer request/response protocol used by all REST APIs. Every REST Assured test sends an HTTP request and receives an HTTP response.
Web Service = any software function exposed over a network via HTTP — language-agnostic, meaning a Java test can call a Python API, a Node.js API, or a Go API identically.
Serialization = Java POJO → JSON string (for request body). Deserialization = JSON response → Java POJO. Jackson handles both directions automatically in REST Assured.
No-arg constructor is MANDATORY in a POJO for Jackson deserialization — Jackson creates an empty instance first, then populates fields via setters. Without it: InvalidDefinitionException.

} @Test public void extractTypedValues() { JsonPath jp = given() .baseUri("https://jsonplaceholder.typicode.com") .when().get("/users/1") .then().extract().response().jsonPath(); int id = jp.getInt("id"); // POJODesign.java // Demonstrates POJO design rules — WHY each rule exists // Covers: no-arg constructor, getters/setters, Jackson annotations import com.fasterxml.jackson.annotation.*; import static io.restassured.RestAssured.given; import org.testng.Assert; import org.testng.annotations.Test; @JsonIgnoreProperties(ignoreUnknown = true) // @JsonIgnoreProperties(ignoreUnknown=true) // WHY: API may add new fields; without this, Jackson throws // UnrecognizedPropertyException and your test breaks unnecessarily class UserPOJO { // RULE 1: Private fields — encapsulation, Jackson uses getters not fields private int id; private String name; private String email; @JsonProperty("username") // @JsonProperty("username") — WHY: JSON key "username" matches field "username" // If JSON key was "user_name": @JsonProperty("user_name") String username private String username; // RULE 2: No-arg constructor — WHY: Jackson creates instance with this, // then calls setters. Without it: InvalidDefinitionException at runtime. public UserPOJO() {} // Optional: parameterized constructor for test setup public UserPOJO(String name, String email) { this.name = name; this.email = email; } // RULE 3: Public getters — WHY: Jackson reads these during serialization // (Java POJO → JSON). Getter name convention: getFieldName() public int getId() { return id; } public String getName() { return name; } public String getEmail() { return email; } public String getUsername() { return username; } // RULE 4: Public setters — WHY: Jackson writes these during deserialization // (JSON → Java POJO). Setter name: setFieldName(Type value) public void setId (int v) { id = v; } public void setName (String v) { name = v; } public void setEmail (String v) { email = v; } public void setUsername(String v) { username = v; } } public class POJODesign { @Test public void pojoRoundTrip() { // SERIALIZE: UserPOJO → JSON body UserPOJO req = new UserPOJO("Chitti", "c@test.com"); given().baseUri("https://jsonplaceholder.typicode.com") .contentType("application/json").body(req) .when().post("/users") .then().statusCode(201); // DESERIALIZE: JSON response → UserPOJO UserPOJO res = given().baseUri("https://jsonplaceholder.typicode.com") .when().get("/users/1") .then().extract().as(UserPOJO.class); Assert.assertEquals(res.getId(), 1); System.out.println("POJO round-trip: " + res.getName()); } }
// HTTPMethodsFlow.java // Every HTTP method — meaning, idempotency, safe flag, typical status codes // Idempotent = calling N times gives same server state as calling once // Safe = does NOT modify server data import static io.restassured.RestAssured.given; import static org.hamcrest.Matchers.*; import org.testng.annotations.Test; public class HTTPMethodsFlow { static final String BASE = "https://jsonplaceholder.typicode.com"; @Test public void getMethod() { // GET = READ | safe=YES | idempotent=YES | has body=NO // Expected status: 200 OK given().baseUri(BASE).when().get("/users/1") .then().statusCode(200).body("id", equalTo(1)); } @Test public void postMethod() { // POST = CREATE | safe=NO | idempotent=NO | has body=YES // NOT idempotent: 3 POSTs create 3 records // Expected status: 201 Created given().baseUri(BASE).contentType("application/json") .body("{ \"title\":\"Test\", \"userId\":1 }") .when().post("/posts") .then().statusCode(201).body("id", notNullValue()); } @Test public void putMethod() { // PUT = FULL UPDATE | safe=NO | idempotent=YES | has body=YES // Replaces entire resource — MUST send all fields // Expected status: 200 OK given().baseUri(BASE).contentType("application/json") .body("{ \"id\":1, \"title\":\"Updated\", \"userId\":1 }") .when().put("/posts/1") .then().statusCode(200).body("title", equalTo("Updated")); } @Test public void patchMethod() { // PATCH = PARTIAL UPDATE | safe=NO | idempotent=DEPENDS | body=YES // Only send fields you want to change // Expected status: 200 OK given().baseUri(BASE).contentType("application/json") .body("{ \"title\":\"Patched Only Title\" }") .when().patch("/posts/1") .then().statusCode(200).body("title", equalTo("Patched Only Title")); } @Test public void deleteMethod() { // DELETE = REMOVE | safe=NO | idempotent=YES | body=NO // Expected status: 200 OK or 204 No Content given().baseUri(BASE) .when().delete("/posts/1") .then().statusCode(anyOf(equalTo(200), equalTo(204))); } }
// SerializationRoundTrip.java // Full round-trip: POJO → JSON (serialize) → HTTP POST → HTTP GET → POJO (deserialize) // Shows the complete data flow an SDET must understand import com.fasterxml.jackson.databind.ObjectMapper; import static io.restassured.RestAssured.given; import org.testng.Assert; import org.testng.annotations.Test; public class SerializationRoundTrip { private final ObjectMapper om = new ObjectMapper(); @Test public void demonstrateSerializationSteps() throws Exception { // ── STEP 1: Create Java POJO (the starting object) ───────────── UserPOJO requestPojo = new UserPOJO("Chitti", "chitti@sdet.com"); // ── STEP 2: Serialization — POJO → JSON string ───────────────── // Jackson ObjectMapper.writeValueAsString() = explicit serialization // REST Assured does this automatically with given().body(pojo) String jsonString = om.writeValueAsString(requestPojo); System.out.println("Serialized JSON: " + jsonString); // Output: {"name":"Chitti","email":"chitti@sdet.com"} // ── STEP 3: HTTP transport — JSON string travels over network ──── // REST Assured puts jsonString in HTTP request body // HTTP layer: encodes as bytes, sends over TCP/IP given() .baseUri("https://jsonplaceholder.typicode.com") .contentType("application/json") .body(requestPojo) // ← REST Assured auto-serializes .when() .post("/users") .then() .statusCode(201); // ── STEP 4: HTTP response arrives ──────────────────────────────── // Server processes request, stores data, returns response String responseBody = given() .baseUri("https://jsonplaceholder.typicode.com") .when().get("/users/1") .then().extract().asString(); System.out.println("Response JSON: " + responseBody.substring(0,50)); // ── STEP 5: Deserialization — JSON string → POJO ───────────────── // Jackson ObjectMapper.readValue() = explicit deserialization // REST Assured does this with extract().as(UserPOJO.class) UserPOJO responsePojo = om.readValue(responseBody, UserPOJO.class); System.out.println("Deserialized: id=" + responsePojo.getId() + " name=" + responsePojo.getName()); Assert.assertEquals(responsePojo.getId(), 1); Assert.assertNotNull(responsePojo.getName()); } }

Best Practices
✔ Always prefer POJO over Map for deserialization in production test frameworks — compile-time type safety and IDE autocomplete prevent typo-based bugs that only surface at runtime.
✔ Keep JSON schema and POJO in sync — when an API adds a new field, update both the JSON Schema file and the POJO class. @JsonIgnoreProperties(ignoreUnknown=true) buys you time but not forever.
✔ Never use RestAssured.baseURI for the server address in real tests — use given().baseUri() or RequestSpecBuilder to avoid thread-safety issues in parallel runs.
✔ Understand HTTP status code families — 2xx = success, 4xx = client error (your code), 5xx = server error (their code). Asserting wrong status codes leads to false-positive tests.
✔ Know that HTTP is stateless — each request is independent. If test step 2 depends on test step 1's data, pass it explicitly (e.g. extract the created ID and store it in a variable).
Interview Q&A
Q1: What are the 6 JSON value types and their Java equivalents? +
string → String, number → int/long/double, boolean → boolean, null → null, object → Map or POJO, array → List. In REST Assured: body() assertions automatically handle type matching — equalTo(1) for integers, equalTo("text") for strings. Use getInt(), getString(), getList() in JsonPath for explicit typed extraction.
Q2: Why does a POJO need a no-arg constructor for Jackson? +
Jackson's default deserialization strategy: (1) call no-arg constructor to instantiate the object, (2) iterate JSON fields, (3) call matching setter for each field. Without a no-arg constructor, step 1 fails with InvalidDefinitionException. The workaround is @JsonCreator on a parameterized constructor where each param gets @JsonProperty mapping. Lombok's @NoArgsConstructor auto-generates no-arg constructors.
Q3: What is the difference between HTTP PUT and PATCH? +
PUT replaces the ENTIRE resource — you must send all fields, even unchanged ones. PATCH does a PARTIAL update — send only the fields to change. Example: updating just a user's email → use PATCH with {"email":"new@email.com"}. Using PUT would require sending all user fields. PUT is idempotent (N calls = same result). PATCH idempotency depends on implementation.
Q4: What makes REST "stateless" and why does it matter for test automation? +
Stateless = the server stores ZERO client-session information between requests. Every request must be self-contained with all needed data. For test automation this means: (1) you must send auth tokens in every request — the server won't remember you're logged in, (2) you can't rely on previous request's side-effects unless you explicitly verify them, (3) tests can run in any order on any server instance — enabling parallel execution on distributed server farms.
Q5: What is the difference between Content-Type and Accept headers? +
Content-Type is on the REQUEST — tells the server the format of the request BODY you're sending (e.g. application/json). Accept is on the REQUEST — tells the server the format you WANT in the RESPONSE (e.g. application/json, application/xml). In REST Assured: contentType("application/json") sets Content-Type; header("Accept","application/json") sets Accept. If the server can't satisfy Accept, it returns 406 Not Acceptable.

Keyword Glossary
Groovy
JVM-based dynamic programming language — superset of Java. REST Assured uses it for GPath expressions and Hamcrest DSL internally.
GPath
Groovy Path — Groovy's expression language for navigating XML/JSON trees. Powers body() assertions in REST Assured.
Closure
Groovy anonymous function block: {it.field == value}. Used in findAll{}, find{} for filtering JSON arrays in REST Assured.
Network Port
A 16-bit number (0–65535) identifying a specific process/service on a machine. Appended to IP: 192.168.1.1:8080.
Port 80
Default HTTP port — browsers and REST clients use it automatically. URL http://example.com = http://example.com:80.
Port 443
Default HTTPS port — all production APIs use this. URL https://example.com = https://example.com:443.
Port 8080
Convention for local HTTP dev servers — Spring Boot default, Tomcat default, Jenkins default. Requires explicit port in URL: http://localhost:8080.
JSON Wire Protocol
Selenium's original HTTP-based protocol for browser automation — used by Selenium WebDriver before 2018.
W3C Protocol
WebDriver W3C standard — replaced JSON Wire Protocol. All modern browsers (Chrome 75+, Firefox 60+) implement W3C WebDriver natively.
WebDriver
API standard for automating browsers. Selenium sends HTTP requests to a browser driver (ChromeDriver) which controls the browser.

Concept 3 — JSON Wire Protocol vs W3C Protocol
JSON Wire Protocol vs W3C — the Selenium/WebDriver context
CONTEXT: Both protocols control HOW Selenium talks to browser drivers This is NOT REST API testing — it's WebDriver browser automation protocol But SDETs must know this because Selenium is part of the SDET toolkit JSON WIRE PROTOCOL (old — pre-2018): → Created by Selenium team (not a standard) → HTTP + JSON messages between Selenium and ChromeDriver/GeckoDriver → Each browser driver implemented it DIFFERENTLY (inconsistencies) Selenium → HTTP POST /session → ChromeDriver → opens Chrome Selenium → HTTP GET /element → ChromeDriver → finds element → Non-standard: Appium, WinAppDriver extended it incompatibly W3C WEBDRIVER PROTOCOL (current — 2018+): → Official W3C standard (World Wide Web Consortium) → Standardized HTTP endpoints all drivers MUST implement identically → All modern browsers implement it natively in the browser itself Selenium → HTTP POST /session → W3C ChromeDriver → opens Chrome Selenium → HTTP GET /element/{id} → W3C driver → finds element → Standard: Appium 2.0, WinAppDriver all align to W3C KEY DIFFERENCES: JSON Wire: capabilities sent as desiredCapabilities{} W3C: capabilities sent as alwaysMatch{} + firstMatch[] JSON Wire: actions API was limited and inconsistent W3C: actions API (mouse, keyboard, touch) standardized WHY SDET MUST KNOW THIS: Selenium 4 = W3C ONLY (dropped JSON Wire support) If your grid uses old drivers: W3C incompatibility causes session errors Interview question: "Why did Selenium 4 drop JSON Wire Protocol?"
Interview One-Liners
Groovy is a JVM language that REST Assured uses internally — you write Java code but the GPath expressions inside body() are actually evaluated as Groovy, enabling closures like findAll{it.id > 5}.
Port 8080 is the Spring Boot and Tomcat default port for local development — REST Assured tests hitting a local API must explicitly include it: given().baseUri("http://localhost:8080").
Port 443 is the default HTTPS port — you never see it in URLs because browsers and REST clients add it automatically. All production APIs run on HTTPS:443.
JSON Wire Protocol = Selenium's old proprietary HTTP protocol for browser automation — replaced by the W3C WebDriver standard in Selenium 4. Not related to REST API testing.
W3C WebDriver = the official World Wide Web Consortium standard that defines exactly how Selenium communicates with browser drivers — mandatory in Selenium 4 and all modern drivers.
GPath closures: findAll{it.userId == 1}.title — this Groovy closure filters array elements where userId equals 1, then extracts their title fields — impossible with simple dot notation alone.

Best Practices
✔ Never hardcode port numbers in test strings — read from System.getProperty("port","8080") so the same test runs against local:8080, staging:8081, and prod:443 without code changes.
✔ Use Groovy's findAll{} for server-side filtering in assertions instead of extracting the full list to Java — it's more concise and the filtering happens in the Groovy engine, not your Java code.
✔ Use unique().size() in GPath to assert field uniqueness across an array — e.g. body("id.unique().size()", equalTo(100)) verifies all IDs are distinct in one assertion.
✔ Create one RequestSpecification per microservice (per port) in @BeforeClass — avoids repeating port/baseUri in every test method and makes environment switching trivial.
✔ For Selenium project interviews: clearly state that JSON Wire Protocol is deprecated and Selenium 4 uses W3C exclusively — this distinguishes you from candidates using outdated knowledge.
Interview Q&A
Q1: What role does Groovy play in REST Assured? +
REST Assured is built on top of Groovy and Java. The GPath expressions inside body() are evaluated by Groovy's parser — not Java. This enables Groovy's concise syntax like findAll{it.id > 5}, size(), unique(), min(), max(). You write Java code but Groovy runs the expressions. You don't need to know Groovy syntax deeply — just understand that closures like {it.field == value} are Groovy anonymous functions where 'it' is the current element.
Q2: Why does port 8080 need to be explicitly specified but port 443 doesn't? +
Port 443 is the IANA-registered default for HTTPS — browsers and HTTP clients automatically append :443 to https:// URLs, so you never see it. Port 8080 is a convention (Spring Boot/Tomcat default for local dev) but NOT the HTTP default (which is 80). Any port other than 80 for HTTP or 443 for HTTPS must be explicitly included in the URL: http://localhost:8080/api. Without it, REST Assured would try port 80 by default.
Q3: What is a Groovy closure and how does 'it' work? +
A Groovy closure is an anonymous function block written as {params -> body} or just {body} where 'it' is the implicit parameter. In findAll{it.userId == 1}: the closure is called once per array element; 'it' holds the current element (a Map/JSON object); the closure returns true to keep the element or false to discard it. findAll collects all elements where the closure returns true. 'it' is exactly like Java lambda's implicit parameter in single-parameter lambdas.
Q4: What is the difference between JSON Wire Protocol and W3C WebDriver Protocol? +
JSON Wire Protocol = Selenium's original unofficial HTTP protocol for browser automation (2007–2018). W3C WebDriver = the standardized W3C Recommendation replacing it (2018+). Key differences: capabilities format changed (desiredCapabilities → alwaysMatch/firstMatch), actions API standardized, URL paths simplified. Selenium 4 dropped JSON Wire entirely — only W3C. Relevant to SDETs maintaining Selenium WebDriver grids. Not related to REST API testing (REST Assured uses plain HTTP).
Q5: How do you configure REST Assured to test microservices on different ports? +
Create a separate RequestSpecification per service using RequestSpecBuilder, setting baseUri and port independently: new RequestSpecBuilder().setBaseUri("http://localhost").setPort(8080).build(). Store each spec in a constant or @BeforeClass variable. Use given(userServiceSpec) for user calls and given(orderServiceSpec) for order calls. This cleanly separates service configurations and makes it trivial to switch ports per environment using System.getProperty() lookups.
Overview — What / Why / When / Where / How
🌐
What
REST API, SOAP API, GraphQL API, and API Chaining — the four API types and patterns every SDET encounters. Understanding all four positions you as a senior automation engineer.
💡
Why
Most companies use REST but legacy systems use SOAP and modern APIs use GraphQL. API Chaining is the core pattern of E2E test automation — every CRUD test is an API chain.
⏰
When
REST Assured covers REST + SOAP. GraphQL needs special handling. API Chaining is used in every test that spans multiple API calls (create → read → update → delete).
📍
Where
REST: most modern APIs. SOAP: banking, government, enterprise ERP. GraphQL: Facebook Graph API, GitHub API v4, Shopify. API Chaining: every E2E test scenario.
⚙️
How
REST Assured handles REST natively. SOAP via POST with XML body. GraphQL via POST with JSON body containing a query string. Chaining by extracting values from one response and passing to the next request.
Keyword Glossary
REST API
Representational State Transfer — architectural style using HTTP methods, URL as resource identifier, stateless, JSON/XML format.
SOAP API
Simple Object Access Protocol — XML-based protocol with mandatory WSDL contract, envelope structure, and strict error handling.
GraphQL API
Query language for APIs — client specifies exactly which fields to return. Single endpoint, over HTTP POST, returns only requested data.
API Chaining
Pattern of using output data from one API call as input to the next — extracting IDs/tokens and passing them to subsequent requests.
WSDL
Web Services Description Language — XML contract for SOAP APIs defining available operations, message formats, and bindings.
SOAP Envelope
XML wrapper around every SOAP message: Header (optional) + Body (required, contains the actual operation call).
GraphQL Query
Client-defined string specifying exactly which fields to return — e.g. { user(id:1) { name email address { city } } }.
GraphQL Mutation
GraphQL equivalent of POST/PUT/DELETE — modifies data. Sent in the same POST body as a query but uses the mutation keyword.
extract().path()
REST Assured method to pull a specific value from response and use it in the next API call — the mechanical implementation of API chaining.
idempotent
Calling the same operation N times gives the same result as calling it once. GET, PUT, DELETE are idempotent. POST is NOT.

Stateless: YES
Learning: Medium

API Chaining — the core E2E test pattern
API CHAINING = output of one call becomes input of the next REAL WORLD E2E SCENARIO: Order placement flow Step 1: Register User POST /users { name:"Chitti" } → 201 { id:42, token:"eyJ..." } EXTRACT: userId=42, token="eyJ..." ↓ chain: pass userId + token to next call Step 2: Create Cart POST /carts { userId:42 } → 200 { cartId:99 } using userId=42 from step 1 EXTRACT: cartId=99 ↓ chain: pass cartId to next call Step 3: Add Items to Cart POST /carts/99/items { productId:101, qty:2 } → 200 using cartId=99 from step 2 ↓ Step 4: Place Order POST /orders { cartId:99, userId:42 } → 201 { orderId:"ORD-001" } EXTRACT: orderId="ORD-001" ↓ Step 5: Verify Order GET /orders/ORD-001 → 200 { status:"PLACED", total:598 } using orderId from step 4 Each step's extracted value feeds the next → that's API CHAINING
Interview One-Liners
REST API = Representational State Transfer — uses HTTP methods (GET/POST/PUT/DELETE), URL as resource identifier, JSON format, stateless. Tested natively with REST Assured.
SOAP API = Simple Object Access Protocol — all calls are HTTP POST with an XML envelope body; requires a WSDL contract; tested with REST Assured by setting body(xmlString) and parsing XML response.
GraphQL API = single /graphql endpoint, all operations via POST; client defines exactly which fields to return in a query string; mutations modify data; tested with REST Assured by setting the query as request body.
API Chaining = extracting a value from one API response (e.g. created user ID) and passing it as input to the next API call — the foundation of every E2E integration test scenario.
WSDL = Web Services Description Language — the mandatory XML contract for SOAP APIs that defines all available operations, input/output message formats, and the endpoint URL.
GraphQL Mutation = GraphQL's equivalent of POST/PUT/DELETE — modifies server data. Sent as POST to the same /graphql endpoint, just with mutation keyword instead of query.

// RESTApiTest.java // REST API = Representational State Transfer // Core principles: stateless, HTTP methods, URL as resource, JSON format // This is what REST Assured is designed for — native support import static io.restassured.RestAssured.given; import static org.hamcrest.Matchers.*; import org.testng.annotations.*; public class RESTApiTest { static final String BASE = "https://jsonplaceholder.typicode.com"; // REST Principle 1: UNIFORM INTERFACE — use standard HTTP methods @Test public void restCRUD() { // CREATE — POST (not idempotent, creates new resource) int id = given().baseUri(BASE).contentType("application/json") .body("{ \"title\":\"REST Test\", \"userId\":1 }") .when().post("/posts") .then().statusCode(201).extract().path("id"); // READ — GET (safe, idempotent, no body) given().baseUri(BASE).when().get("/posts/"+id) .then().statusCode(200).body("id", equalTo(id)); // UPDATE (full) — PUT (idempotent, replaces whole resource) given().baseUri(BASE).contentType("application/json") .body("{ \"id\":"+id+", \"title\":\"Updated\", \"userId\":1 }") .when().put("/posts/"+id) .then().statusCode(200).body("title", equalTo("Updated")); // UPDATE (partial) — PATCH (only changed fields) given().baseUri(BASE).contentType("application/json") .body("{ \"title\":\"Patched\" }") .when().patch("/posts/"+id) .then().statusCode(200).body("title", equalTo("Patched")); // DELETE (idempotent, removes resource) given().baseUri(BASE) .when().delete("/posts/"+id) .then().statusCode(anyOf(equalTo(200), equalTo(204))); } @Test public void restQueryParams() { // REST Principle 2: RESOURCE IDENTIFICATION via URL + query params given().baseUri(BASE) .queryParam("userId", 1) // filter by userId .queryParam("_limit", 5) // pagination .when().get("/posts") .then().statusCode(200).body("size()", equalTo(5)); } }

Best Practices
✔ For GraphQL: always check the "errors" field in the response body — GraphQL returns HTTP 200 even for failures, putting error details in data.errors[] rather than using 4xx status codes.
✔ In API chains, store extracted values in local variables (not static fields) unless sharing across test methods is intentional — static fields cause cross-test data pollution.
✔ For SOAP tests, always match the namespace URIs exactly from the WSDL — a wrong namespace in the XML envelope causes silent failures where the server rejects the message without a clear error.
✔ Use extract().response() (not just extract().path()) when you need multiple values from the same response — it avoids making duplicate HTTP calls to extract different fields separately.
✔ Design API chains to be independent: each chain should create its own test data at the start and clean it up at the end — never depend on data left by a previous test run or another test class.
Interview Q&A
Q1: What are the key differences between REST and SOAP? +
REST: uses HTTP methods (GET/POST/PUT/DELETE), JSON/XML format, multiple endpoints, stateless, lightweight, no formal contract required (optional OpenAPI). SOAP: all operations via HTTP POST, XML-only format, single endpoint, mandatory WSDL contract, WS-Security, WS-Transaction support, can be stateful. SOAP is more complex but has stronger formal contracts. REST dominates modern APIs (95%); SOAP remains in banking, government, and enterprise ERP systems.
Q2: Why does GraphQL always return HTTP 200 even when an operation fails? +
GraphQL specification defines that the HTTP layer is just a transport. All application-level errors (validation failures, permission denied, not found) are returned in the response body's "errors" array, regardless of outcome — always HTTP 200. Only true HTTP-level problems (server crash, network timeout, authentication middleware failure) return non-200. SDETs must explicitly check .body("errors", nullValue()) or .body("data", notNullValue()) for GraphQL error detection.
Q3: What is API Chaining and how is it implemented in REST Assured? +
API Chaining = using the output of one API call as input to the next. Implemented in REST Assured by: (1) extract().path("field") to get a single value, (2) extract().response().jsonPath() for multiple values, (3) extract().as(POJO.class) for full object deserialization. Store extracted values in local variables and pass them to the next request's URL path, query params, or body. Example: int userId = post("/users").then().extract().path("id"); then: get("/users/" + userId).
Q4: How do you test a GraphQL API with REST Assured? +
GraphQL over HTTP = POST to single /graphql endpoint with JSON body containing "query" (the GraphQL query string) and optionally "variables" (parameterized values). In REST Assured: Map body = new HashMap<>(); body.put("query", "{ users { id name } }"); given().contentType("application/json").body(body).when().post("/graphql").then().statusCode(200).body("data.users", notNullValue()). For mutations: use the same pattern but with the mutation keyword in the query string.
Q5: What makes a good E2E API chain test? +
A good E2E chain test: (1) creates all its own test data in step 1 — no dependency on external data, (2) extracts every value needed for the next step and verifies it's not null before using it, (3) asserts at each step that the chained data flows correctly (e.g. check userId matches expected after step 2), (4) cleans up created data in @AfterMethod to prevent data pollution, (5) uses dependsOnMethods in TestNG so later steps are skipped if earlier ones fail rather than proceeding with invalid data.
