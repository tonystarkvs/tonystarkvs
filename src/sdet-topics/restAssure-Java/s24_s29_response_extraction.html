Overview — What / Why / When / Where / How
🦊
What
Groovy's GPath $ syntax = the most powerful inline extraction and assertion language in REST Assured. $ = root document, $.key = field, $[0] = array index.
💡
Why
GPath expressions let you assert AND filter directly inside body() without writing any Java code — the most concise validation approach in REST Assured.
⏰
When
Use GPath inside then().body() for every structural assertion — single fields, nested objects, array elements, filtered subsets, and aggregate functions.
📍
Where
Inside REST Assured's then().body(expression, matcher) chain. GPath is evaluated by Groovy's JsonSlurper engine on the response body.
⚙️
How
REST Assured passes the JSON body to Groovy's parser. GPath navigates the parsed tree using dot notation, index notation, and Groovy closures. $ = root node.
Keyword Glossary
$
Root of the JSON document in GPath. body("$", ...) refers to the entire response. body("$.key") = top-level field.
$ (root array)
When the response is a JSON array, "$" or "" references the whole array. "$[0]" = first element.
$[n]
Array index access on root. "$[0].orderId" = orderId of first element in a root array response.
dot notation
Navigate nested objects: "order.customer.address.city" — each dot descends one level into the object.
size()
Groovy method returning length of array or string. "items.size()" = number of items in the array field.
findAll{}
Groovy closure filtering array elements. "items.findAll{it.qty>1}.productId" = productIds where qty > 1.
find{}
Returns only the FIRST element matching the closure condition — singular result, not a list.
collect{}
Transforms each element — "items.collect{it.price}" = list of all prices from every item.
it
Implicit Groovy closure parameter — refers to the current element being evaluated in findAll{}, find{}, collect{}.
sum()
Groovy aggregate — "items.price.sum()" = total sum of all price values in the items array.

"$" in GPath = the root of the JSON document — body("$", notNullValue()) asserts the response body is not empty; body("$.orderId", equalTo("ORD-2024-001")) is identical to body("orderId", ...).
"$[0]" = first element of a root-level JSON array — used when the entire response body is an array, not an object.
"items.size()" = Groovy's size() on a JSON array field — returns the count of elements without extracting the array to Java.

"items[0].tags[1]" = nested array indexing — first item's second tag. GPath supports chained index notation for arrays within arrays.

Best Practices
✔ Use "$" prefix when the expression might be ambiguous — e.g. "$.status" is unambiguous about navigating from root; without it, Groovy still works but reads less clearly.
✔ Prefer findAll{}.size() over extracting to Java and calling list.size() — it keeps the assertion inline and avoids an extra step of pulling data out of the response.
✔ Use everyItem() for asserting all elements in a collected array satisfy a condition — body("items.qty", everyItem(greaterThan(0))) is more concise than individual index assertions.
✔ Combine path params (pathParam()), query params (queryParam()), and custom headers in a RequestSpecBuilder for reuse across all 6 session examples.
✔ Never use flatten() without knowing the data shape — nested arrays of different depths may produce unexpected results; log the response first to confirm structure.
Interview Q&A
Q1: What does "$" mean in a GPath expression? +
"$" represents the ROOT of the JSON document in Groovy's GPath syntax. body("$", notNullValue()) asserts the response body exists. body("$.orderId", equalTo("X")) is identical to body("orderId", equalTo("X")) — the $ prefix is optional for root-level fields. When the response is a JSON ARRAY, "$" refers to the entire array; "$[0]" accesses the first element and "$[0].field" navigates into it.
Q2: How do you access a nested array inside an array element using GPath? +
Use chained index notation: "items[0].tags[0]" = first element of items, then first element of its tags array. "items[0].tags" returns the entire tags array for the first item. "items.tags" (without index) collects tags from ALL items as a list of lists — use .flatten() to merge them: "items.tags.flatten()" produces a single flat list of all tags across all items.
Q3: What is the difference between find{} and findAll{} in GPath? +
findAll{condition} returns ALL elements where the closure returns true — the result is a List. find{condition} returns only the FIRST element where the closure is true — the result is a single element (not a list). Use findAll when you want to assert count or check multiple matches; use find when you want to navigate into one specific element and assert its fields.
Q4: How do you assert the sum of all prices in an items array with GPath? +
Use Groovy's sum(): body("items.price.sum()", equalTo(948.0f)). This evaluates as: collect all "price" values from items → [599.0, 349.0] → call .sum() on that Groovy List → 948.0. Similarly: body("items.price.min()", equalTo(349.0f)) and body("items.price.max()", equalTo(599.0f)). These work on any numeric field collected from an array.
Q5: When would you NOT use GPath and prefer jsonPath() instead? +
Use jsonPath() when you need to EXTRACT a value to USE in subsequent logic — e.g. store it in a variable for the next API call, log it, or perform Java-based computation. GPath inside body() is for ASSERTIONS only — it doesn't return values to Java code. Also use jsonPath() when you need strongly typed collection extraction: getList("items.price", Double.class) vs getList which returns Object.
Overview — response.path(expression)
🔍
What
response.path(expr) = the quickest way to extract a single typed value from a stored Response object using a GPath expression. Returns Object — Java auto-casts.
💡
Why
After storing a Response, you often need to pull one or two fields out quickly. response.path() is more concise than creating a full JsonPath object for simple extractions.
⏰
When
Use when you have a Response object already stored and need to read one or a few values. Also available inside the fluent chain as then().extract().path().
📍
Where
On a Response object: res.path("customer.name"). In the chain: given()...then().extract().path("orderId"). Both use GPath internally.
⚙️
How
REST Assured internally calls Groovy's JsonSlurper.parseText(body) and navigates the parsed tree using the GPath expression you provide. Returns the matching node as Object.
Keyword Glossary
response.path(expr)
Method on Response interface — evaluates a GPath expression and returns the matching value as Object (auto-casts to String, int, List etc.).
extract().path()
Fluent chain version — terminates the then() chain and returns a single value. Cannot add more assertions after this call.
extract().response()
Terminates the chain and returns the full Response object — allowing both assertions AND multiple path() calls on the same response.
auto-cast
Java's implicit type cast when assigning an Object return to a typed variable. e.g. String name = res.path("name") works only if the JSON value is a string.
Response interface
REST Assured interface wrapping the complete HTTP response: status code, headers, cookies, body — and provides path(), jsonPath(), xmlPath() methods.
getStatusCode()
Method on Response returning the int HTTP status code (200, 201, 404 etc.).
getTime()
Method on Response returning response time in milliseconds — used for SLA performance assertions.
getHeader(name)
Method on Response returning the value of a specific response header by name.
asString()
Method on ResponseBody (accessed via res.getBody()) — returns the full response body as a raw String.
nested path
GPath expression with dots for nested navigation: "customer.address.city" — path() evaluates as deeply as specified.
Interview One-Liners
response.path("orderId") = the fastest one-liner to extract a single value from a stored Response — internally evaluates as a GPath expression, returns Object that Java auto-casts.
extract().path("orderId") = terminates the fluent then() chain and returns the value — cannot add any more .body() assertions after it in the same chain.
extract().response() = preferred over extract().path() when you need multiple values — store the Response once, then call path() as many times as needed without extra HTTP calls.
response.path("items[0].price") = path() fully supports GPath array indexing and nested dot notation — identical capability to body() assertions but returns the value instead of asserting it.
response.path("items.productId") = when the path expression resolves to multiple elements (array field without index), path() returns a List — always declare the variable as List in that case.

Best Practices
✔ Always prefer extract().response() over extract().path() in the chain when you need more than one value — one HTTP call, unlimited path() calls on the stored Response.
✔ When path() returns a List (multi-element collection), always declare the variable as List<String> or List<Float> — assigning to a non-List type causes ClassCastException at runtime.
✔ Use res.getStatusCode(), res.getTime(), res.getHeader(name) on the stored Response for HTTP-level assertions alongside body path() extractions.
✔ All then().body() assertions must come BEFORE extract() in the fluent chain — once you call extract(), the chain terminates and no further assertions can be added.
✔ For numeric fields: declare as float not int when the JSON value has decimals (599.0) — Jackson deserializes JSON numbers with decimals as Float/Double, not Integer.
Interview Q&A
Q1: What is the difference between response.path() and jsonPath().getString()? +
response.path("field") returns Object — Java infers the type. jsonPath().getString("field") explicitly returns String and is type-safe. For numbers: path("qty") may require a cast; jsonPath().getInt("qty") always returns int. Both use GPath internally. Prefer path() for quick single-value extraction; prefer jsonPath() typed methods when type safety matters or when you need getList() for arrays.
Q2: Why can't you add body() assertions after extract().path() in the chain? +
extract() terminates the ValidatableResponse fluent chain and returns an ExtractableResponse. Once you call extract(), REST Assured finalizes the validation phase. You can't add more validators after extraction. Solution: put ALL body() assertions before extract(), then call extract().response() to get the full Response object. Then use res.then().body() for additional assertions or res.path() for extractions — both work independently on the stored Response.
Q3: What happens when path() expression matches multiple values? +
When the GPath expression resolves to multiple elements — e.g. "items.productId" collects productId from every array element — path() returns a List<Object> (or List<String> when auto-cast). If you try to assign to a String variable, you get a ClassCastException. Always check: if the path selects from an array without an index, it returns a List. If it navigates into a single object, it returns a scalar value.
Q4: How do you extract HTTP response metadata (not just body) using the Response object? +
The Response object exposes: res.getStatusCode() for HTTP status, res.getTime() for response time in milliseconds, res.getHeader("Content-Type") for a specific header, res.getHeaders() for all headers, res.getCookies() for cookie map, res.getBody().asString() for raw body text, res.getContentType() for content type string. These are separate from path() which only accesses the body content.
Q5: How do you access a 4-level deep field like customer.address.geo.lat using path()? +
Use dot notation for every level: res.path("customer.address.geo.lat"). REST Assured evaluates this as a GPath expression: start at root, navigate to "customer" object, then to "address" object, then to "geo" object, then return "lat" value. The depth has no practical limit — just chain as many dots as needed. For numbers with decimals, declare as float: float lat = res.path("customer.address.geo.lat");
Overview — jsonPath() Object
🗺️
What
response.jsonPath() = a dedicated JsonPath object providing typed getter methods: getString(), getInt(), getFloat(), getBoolean(), getList(), getMap().
💡
Why
Eliminates the unsafe implicit casting of path(). Typed methods prevent ClassCastException and make extraction intent explicit and IDE-friendly.
⏰
When
Use when extracting multiple fields from the same response — create the JsonPath object once, then call typed getters for each field without the overhead of re-parsing.
📍
Where
JsonPath jp = response.jsonPath() — then jp.getString("field"), jp.getList("items.price", Float.class). Also: JsonPath.from(rawString) for String-based parsing.
⚙️
How
JsonPath wraps the response body and provides typed extraction. Internally uses same GPath engine as path() but surfaces it through typed method signatures.
Keyword Glossary
jsonPath()
Method on Response returning a JsonPath object pre-loaded with the response body — ready for typed extractions.
JsonPath.from(String)
Static factory — creates a JsonPath from a raw JSON string. Useful when you have the body as String and want typed access.
getString(path)
Extracts a String value — returns null if the field is absent, empty string if the value is null.
getInt(path)
Extracts an int — throws NumberFormatException if the value is not numeric.
getFloat(path)
Extracts a float — use for decimal JSON numbers. getDouble() available for higher precision.
getBoolean(path)
Extracts a boolean — maps JSON true/false to Java boolean primitives.
getList(path)
Returns a List<Object> from an array field. Use getList(path, Class) for typed list.
getList(path, Class)
Returns a typed List — e.g. getList("items.price", Float.class) returns List<Float>. Prevents ClassCastException.
getMap(path)
Returns a Map<String,Object> for a JSON object field — useful for extracting a whole sub-object.
getObject(path, Class)
Deserializes a JSON object at the path into a typed POJO. Requires Jackson/Gson on classpath.
Interview One-Liners
jp.getString("customer.name") = type-safe String extraction — returns null if field absent, never throws ClassCastException unlike path() which returns Object.
jp.getList("items.price", Float.class) = typed list extraction — the Class parameter tells JsonPath which type to cast each element to, producing a clean List<Float>.
jp.getMap("customer.address") = extracts a whole JSON object as Map<String,Object> — gives key-based access to all fields without needing to extract them individually.
JsonPath.from(rawString) = static factory method to parse ANY JSON string into a JsonPath object — no REST Assured Response needed, works on strings from files, DB, or other sources.
jp.getObject("fulfillment", FulfillmentModel.class) = deserializes a JSON sub-object directly into a typed POJO — the most powerful JsonPath method combining extraction and deserialization.

Best Practices
✔ Always use typed getList(path, String.class) over untyped getList(path) — the Class parameter prevents runtime ClassCastException when iterating the extracted list.
✔ Create the JsonPath object ONCE and call multiple typed getters — the JSON body is parsed only once, all subsequent getter calls operate on the pre-parsed in-memory tree.
✔ Use getObject(path, MyClass.class) when extracting a complex sub-object — it's cleaner than extracting each field individually and lets Jackson handle the mapping.
✔ Use getMap("field") when the sub-object structure is dynamic or unknown — it's safer than getObject() when you don't have a matching POJO class.
✔ Groovy closures work inside getList() path arguments — jp.getList("items.findAll{it.qty>1}.productId", String.class) combines filtering and typed extraction in one call.
Interview Q&A
Q1: What is the advantage of getList(path, Class) over getList(path)? +
getList(path) returns List<Object> — each element is untyped, requiring explicit casting when iterating (e.g. (String) element). This can cause ClassCastException if the actual type differs. getList(path, String.class) returns List<String> directly — no casting needed, compile-time type safety, IDE code completion works. Always use the Class parameter form in production test code.
Q2: When would you use JsonPath.from(String) instead of response.jsonPath()? +
Use JsonPath.from(String) when: (1) you already extracted the body as a raw String with asString() and don't have the original Response, (2) parsing JSON from a test resource file (expected response JSON), (3) parsing JSON from a database query result, (4) testing a JsonPath expression in isolation without making an HTTP call. response.jsonPath() is a convenience that calls JsonPath.from(response.asString()) internally.
Q3: How do you use getObject() to deserialize only part of the response? +
jp.getObject("fulfillment", FulfillmentModel.class) — the first argument is the JSON path to a sub-object, the second is the target POJO class. Jackson maps only the JSON fields at that path to the POJO. The POJO must have @JsonIgnoreProperties(ignoreUnknown=true) to handle fields it doesn't declare. This is more precise than extract().as(FullModel.class) when you only need one section of a large response.
Q4: Can you use Groovy closures in getList() path expressions? +
Yes — the path argument in any JsonPath method is a GPath expression evaluated by Groovy. jp.getList("items.findAll{it.discount > 0}.productId", String.class) first filters items where discount > 0 using a Groovy closure, then extracts productId from each matching element, returning a typed List<String>. The Groovy evaluation happens before the typed extraction — they compose seamlessly.
Q5: What does getMap() return and when is it preferred over getObject()? +
getMap(path) returns Map<String,Object> — a dynamic key-value representation of a JSON object. Prefer it over getObject() when: (1) you don't have a matching POJO class, (2) the sub-object structure is dynamic/unknown, (3) you need to iterate over all keys, (4) for one-off extractions that don't warrant creating a POJO class. Prefer getObject(path, Class) when you have a matching POJO and want compile-time type safety with method-based field access.
Overview — extract().as(Map.class)
🗂️
What
extract().as(Map.class) = deserializes the ENTIRE response JSON object into a Java Map<String,Object>. Nested objects become nested LinkedHashMaps.
💡
Why
Gives you the complete response as a navigable Java Map tree — useful for dynamic validation, iterating over all keys, or when you don't have a matching POJO class.
⏰
When
Use for dynamic/unknown response schemas, schema-less validation, or when you want to assert presence of all expected keys without creating individual body() assertions.
📍
Where
In the chain: then().extract().as(Map.class). For typed lists: extract().as(new TypeRef<List<Map<String,Object>>>(){}).
⚙️
How
REST Assured delegates to Jackson's ObjectMapper. Jackson maps JSON objects → LinkedHashMap, JSON arrays → ArrayList, JSON numbers → Integer/Long/Double.
Keyword Glossary
as(Map.class)
REST Assured method — Jackson deserializes the JSON response body into Map<String,Object>. Root must be a JSON object.
LinkedHashMap
Jackson's default Map implementation for JSON objects — preserves key insertion order. What you actually get when casting a nested object.
unchecked cast
Casting Object to a generic type like Map<String,Object> — compiler warns but safe when Jackson always returns Maps for JSON objects.
ArrayList
Jackson's default List implementation for JSON arrays — what nested array fields become in the Map tree.
Number casting
JSON numbers become Integer (small) or Long (large) or Double (decimal). Safe pattern: ((Number) map.get("price")).doubleValue().
keySet()
Java Map method returning all keys as a Set — useful for dynamic validation that all expected fields are present.
forEach()
Java 8 Map method iterating over all key-value pairs — useful for printing or logging all response fields.
containsKey()
Map method — asserts a specific key exists. Used for flexible field-presence validation without strict value checks.
Stream + filter
Java stream operations on the ArrayList from a JSON array field — map-and-filter on response items using typed streams.
TypeRef
REST Assured's generic type token — needed for List<Map> extraction to preserve generic type through Java erasure.
Interview One-Liners
as(Map.class) = Jackson deserializes the complete response JSON into a Map<String,Object> where nested JSON objects become LinkedHashMaps and arrays become ArrayLists.
Numeric safety: always cast JSON numbers via ((Number) map.get("price")).doubleValue() — Jackson may produce Integer or Double depending on the value, so Number is the safe parent type.
map.keySet() = use to validate all expected fields are present in the response without writing individual body() assertions for each field — iterate and assert containsAll.
Nested access: (Map) map.get("customer") then (Map) customer.get("address") — each nested JSON object requires a separate (Map) cast to access its fields.
Array iteration: (List) map.get("items") returns ArrayList; cast each element to Map to access its fields: ((Map) item).get("productId").

Best Practices
✔ Always cast numeric values via ((Number) map.get("price")).doubleValue() — never directly cast to int or double, as Jackson chooses Integer vs Long vs Double based on value magnitude.
✔ Use Map + containsKey() for dynamic schema validation — iterate over a list of required fields and assert each is present without writing individual body() assertions per field.
✔ Print map.keySet() during initial API exploration to discover all response fields before writing assertions — faster than reading API docs.
✔ Prefer as(POJO.class) over as(Map.class) in production frameworks — Map requires manual casting everywhere and has no compile-time safety. Map is best for quick exploratory tests.
✔ Use Java Streams on the extracted List to filter and aggregate — items.stream().filter().mapToDouble().sum() is cleaner than manual iteration loops.
Interview Q&A
Q1: What Java types does Jackson produce when deserializing into Map.class? +
JSON object → LinkedHashMap (preserves insertion order). JSON array → ArrayList. JSON string → String. JSON integer (fits in 32 bits) → Integer. JSON integer (large) → Long. JSON decimal → Double. JSON boolean → Boolean. JSON null → null. Understanding this mapping is critical for safe casting: always use (Number).intValue() or (Number).doubleValue() for numeric fields, never cast directly to int or double.
Q2: How do you navigate 4 levels deep using Map access? +
Cast each level: Map customer = (Map) resp.get("customer"); Map address = (Map) customer.get("address"); Map geo = (Map) address.get("geo"); double lat = ((Number) geo.get("lat")).doubleValue(); Each (Map) cast is needed because get() returns Object. This is the main disadvantage vs POJO deserialization where user.getAddress().getGeo().getLat() is fully typed. For deeply nested access, jsonPath().getDouble("customer.address.geo.lat") is significantly cleaner.
Q3: How do you validate that all required fields are present using as(Map.class)? +
Define a List of required field names, then iterate: List.of("orderId","status","customer").forEach(f -> Assert.assertTrue(resp.containsKey(f), "Missing: " + f)). This catches missing fields without asserting specific values — useful when you want structural validation (all fields present) rather than value validation. It catches API regressions where the backend removes a response field.
Q4: When would you prefer Map over POJO for response extraction? +
Prefer Map when: (1) the response schema is dynamic or unknown at design time, (2) quickly exploring a new API without writing POJO classes, (3) need to iterate over all keys dynamically, (4) the response structure varies by request parameters. Prefer POJO when: (1) schema is fixed and well-known, (2) multiple tests use the same response structure, (3) nested access is frequent, (4) compile-time safety is important for maintainability.
Q5: How do you access items[0].tags[0] using the Map approach? +
List items = (List) resp.get("items"); Map item0 = (Map) items.get(0); List tags = (List) item0.get("tags"); String firstTag = (String) tags.get(0); — four lines and three casts vs GPath "items[0].tags[0]" which is one expression. This illustrates why Map-based access is verbose for deeply nested array elements and why GPath or jsonPath() is preferred for extraction-heavy tests.
Overview — JSONObject(rawString)
📦
What
new JSONObject(rawString) = org.json library's explicit JSON parser — wraps the response body string into a JSONObject providing typed accessor methods.
💡
Why
Provides null-safe optString(), explicit getJSONObject() and getJSONArray() methods — cleaner than Map casting, no Groovy knowledge needed.
⏰
When
Useful in teams that already depend on org.json, or when you prefer explicit JSON building/parsing APIs over Jackson's automatic mapping.
📍
Where
First extract body with then().extract().asString(). Then: new JSONObject(rawBody). Navigate nested objects via getJSONObject(key) and arrays via getJSONArray(key).
⚙️
How
org.json's parser reads the raw JSON string character-by-character and builds an in-memory tree of JSONObject and JSONArray instances — no Jackson involved.
Keyword Glossary
JSONObject(String)
Constructor that parses a raw JSON string into a JSONObject. Throws JSONException on invalid JSON.
getString(key)
Returns String value for key. Throws JSONException if key is missing or value is not a String.
optString(key)
Returns String or "" if key missing — NULL SAFE. optString(key, default) returns specified default if missing.
getInt(key)
Returns int for numeric field. Throws JSONException if missing or non-numeric.
getDouble(key)
Returns double for decimal fields. Safer than getFloat for large decimal numbers.
getBoolean(key)
Returns boolean for true/false fields. Throws JSONException if missing or not boolean.
getJSONObject(key)
Returns nested JSONObject for an object field — used to navigate deeper levels.
getJSONArray(key)
Returns JSONArray for array fields. Use length() for count and getJSONObject(n) for element access.
JSONArray(String)
Constructor for root JSON arrays. length() = element count. getJSONObject(n) = nth element.
has(key)
Returns boolean — true if the key exists in the JSONObject. Use before getString to avoid JSONException.
Interview One-Liners
new JSONObject(rawBody) = parses raw JSON string into org.json's JSONObject — requires two steps: extract body as String, then wrap in JSONObject.
optString("key","N/A") = null-safe alternative to getString() — returns the specified default value if the key is absent, preventing JSONException for optional fields.
getJSONObject("address") = extracts a nested JSON object as its own JSONObject instance — enabling chained navigation: obj.getJSONObject("address").getJSONObject("geo").getDouble("lat").
getJSONArray("items").length() = counts elements in a JSON array — JSONArray.length() is the org.json equivalent of array.size().
has("key") = null-safe key existence check before accessing a field — use instead of try/catch around getString() when a field may or may not be present.

Best Practices
✔ Always use optString(key, defaultValue) for optional fields — getString() throws JSONException on missing keys, causing test failures for fields that are legitimately absent.
✔ Use has(key) before accessing nested objects that may be absent — prevents JSONException when conditional response fields are missing.
✔ Chain getJSONObject() calls for deep navigation: jo.getJSONObject("payment").getJSONObject("card").getString("network") — more readable than creating intermediate variables for each level.
✔ Use a try-catch around new JSONObject(rawBody) if the API might return non-JSON (e.g. HTML error pages) — catches JSONException before it becomes a confusing test failure.
✔ Prefer REST Assured's jsonPath() over JSONObject for new projects — jsonPath() is already in REST Assured's dependency tree; JSONObject requires an additional org.json dependency.
Interview Q&A
Q1: What is the two-step process for using JSONObject with REST Assured? +
Step 1: Extract body as raw String using then().extract().asString() — this terminates the REST Assured chain and gives you the response body text. Step 2: Parse the string using new JSONObject(rawBody) — org.json parses the JSON text into a navigable JSONObject. Then use getString, getJSONObject, getJSONArray etc. to access fields. Unlike jsonPath() which wraps this automatically, JSONObject requires manual extraction first.
Q2: What is the difference between getString() and optString() in JSONObject? +
getString(key) throws JSONException if the key is absent from the JSON object. optString(key) returns an empty string "" if the key is missing — never throws. optString(key, "default") returns the specified default string if missing. Rule: use getString() for mandatory fields where absence indicates a real bug; use optString() for optional/conditional fields that may legitimately be absent depending on the API scenario.
Q3: How do you access a 4-level deep field using JSONObject? +
Chain getJSONObject() calls: JSONObject geo = jo.getJSONObject("customer").getJSONObject("address").getJSONObject("geo"); double lat = geo.getDouble("lat"); Each call returns a new JSONObject allowing further navigation. For deep one-time access, chain all in one line: double lat = jo.getJSONObject("customer").getJSONObject("address").getJSONObject("geo").getDouble("lat"); This is similar to POJO chaining but with explicit method names at each level.
Q4: How do you iterate a JSONArray? +
Use a for loop with length(): JSONArray items = jo.getJSONArray("items"); for(int i=0; i<items.length(); i++) { JSONObject item = items.getJSONObject(i); String id = item.getString("productId"); }. length() = count of elements (not size()). getJSONObject(n) = nth element as JSONObject. getString(n) for string array elements, getInt(n) for integer array elements. No Java stream support — only traditional for loops with index.
Q5: When would you choose JSONObject over jsonPath() for response extraction? +
Choose JSONObject when: (1) your project already uses org.json as a build dependency (saves adding jsonPath overhead), (2) you need optString null-safe access for many optional fields (jsonPath returns null for missing fields but doesn't have a default-value option), (3) team is more familiar with explicit getters than GPath expressions, (4) code review requires explicitly typed getters at every level. Choose jsonPath() when you want concise dot-notation paths without intermediate variable creation.

Best Practices
✔ Create Gson once and reuse — it is thread-safe after construction. Avoid creating new instances per test method.
✔ Use @SerializedName only when JSON keys differ from Java field names — default exact-name matching is cleaner and needs no annotation.
✔ Prefer Gson over Jackson when POJOs are simple (no getters/setters) — Gson field-access reduces boilerplate significantly.
✔ Use TypeToken for any generic type deserialization — List<T> and Map<String,T> require it due to Java type erasure.
✔ Use GsonBuilder().setLenient() when the API returns non-strict JSON — prevents MalformedJsonException on edge-case responses.
Interview Q&A
Q1: How does Gson differ from Jackson for POJO deserialization? +
Gson accesses fields directly via reflection — no getters/setters or no-arg constructor needed. Jackson requires getter methods and a no-arg constructor. Gson silently ignores unknown JSON fields; Jackson throws UnrecognizedPropertyException unless @JsonIgnoreProperties is set. Gson is lighter but slower for large payloads; Jackson is faster but more configuration-heavy. Choose Gson for simplicity; Jackson for performance-critical or enterprise projects.
Q2: What is TypeToken and why is it needed? +
TypeToken solves Java type erasure. At runtime, List<GsonItem> becomes just List — Gson cannot know what type to deserialize each element into. TypeToken creates an anonymous subclass that captures the generic type: new TypeToken<List<GsonItem>>(){}.getType() retains List<GsonItem> at runtime. Without it, Gson would create List<LinkedTreeMap> instead of List<GsonItem>, causing ClassCastException when accessing elements.
Q3: How do you handle JSON keys that don't match Java field names? +
Use @SerializedName annotation: @SerializedName("order_id") String orderId; maps JSON key "order_id" to Java field orderId. Works for both deserialization (fromJson) and serialization (toJson). Alternative: use GsonBuilder().setFieldNamingPolicy(FieldNamingPolicy.LOWER_CASE_WITH_UNDERSCORES) to apply a global naming strategy instead of annotating each field.
Q4: How do you use Gson with REST Assured for request body building? +
Create a POJO representing the request body, populate its fields, then serialize: OrderRequest req = new OrderRequest(); req.orderId = "ORD-001"; String body = new Gson().toJson(req); Then pass to REST Assured: given().body(body).post(url). This is cleaner than building JSON strings manually — the POJO acts as a type-safe request builder, and Gson toJson handles serialization including nested objects and arrays.
Q5: When would you NOT use Gson for response extraction? +
Don't use Gson when: (1) you only need 1-2 fields — jsonPath().getString() is simpler than creating a full POJO class, (2) the response schema changes frequently — Map-based access is more flexible, (3) you need Groovy GPath closures for filtering (findAll, collect) — only REST Assured native body() and jsonPath() support GPath, (4) performance is critical for very large payloads — Jackson is faster, (5) your project already uses Jackson — avoid mixing two JSON libraries.
