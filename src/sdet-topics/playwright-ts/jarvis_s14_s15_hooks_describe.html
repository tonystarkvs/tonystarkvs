Session 14 · All Test Hooks

Test hooks are lifecycle functions that run automatically at specific points in a test suite's execution — before/after each test or once for a group. Playwright provides beforeAll, afterAll, beforeEach, afterEach with full async/await support, access to fixtures, and guaranteed execution order.

WHAT are hooks?

Lifecycle callbacks that wrap test execution. beforeEach / afterEach run before/after every individual test. beforeAll / afterAll run once for the entire describe block. They share fixtures and can perform setup and teardown.

WHY use hooks?

Eliminate duplication. Instead of repeating page.goto('/login') in every test, put it in beforeEach. Share expensive setup like DB seeding in beforeAll. Guarantee cleanup even when tests fail.

WHEN to use which?

Use beforeEach for anything that must be fresh per test (navigation, login). Use beforeAll for expensive shared setup (DB seed, server start). Use afterEach for per-test cleanup. Use afterAll for shared teardown.

WHERE do they run?

Inside test.describe() blocks or at the top of a spec file. They are scoped — a beforeEach inside a nested describe only runs for tests in that describe. Outer hooks run before inner hooks.

HOW do they work?

Hooks receive the same fixtures as tests (page, context, request, testInfo, custom fixtures). They are fully async with await support. Execution order: outer beforeAll → outer beforeEach → inner beforeEach → TEST → inner afterEach → outer afterEach → inner afterAll → outer afterAll.

Keyword Glossary — Session 14
test.beforeEach()Runs before every individual test in the current scope (file or describe block). Receives all fixtures. Use for per-test navigation, login, data setup. Creates fresh state per test.
test.afterEach()Runs after every individual test, even if the test failed. Receives fixtures AND testInfo.status. Use for per-test cleanup, conditional screenshots, logging.
test.beforeAll()Runs ONCE before the first test in a describe block. All tests in the block share the result. Worker-scoped — each parallel worker runs its own beforeAll. Use for DB seeding, server start, one-time API setup.
test.afterAll()Runs ONCE after the last test in a describe block. Use for shared cleanup: delete seeded data, close DB connections, shut down test servers. Worker-scoped.
testInfoSecond parameter in hooks (after fixtures). Contains: title, status, errors, retry, testId, outputDir, attach(), annotation. Especially useful in afterEach for conditional teardown logic.
testInfo.statusActual result of the test: 'passed', 'failed', 'timedOut', 'skipped', 'interrupted'. Available in afterEach/afterAll hooks for conditional behavior based on outcome.
testInfo.expectedStatusWhat the test was expected to return (usually 'passed'). Compare with testInfo.status: if status !== expectedStatus, the test failed unexpectedly.
ScopeHooks apply to all tests in their enclosing describe block or file. Nested hooks only affect tests in their own describe block. Outer hooks always run for inner tests too.
Hook fixturesHooks receive fixtures via destructuring: test.beforeEach(async ({ page, context, request }) => {...}). Same fixture system as tests — page, context, custom fixtures all available.
test.use()Not a hook but closely related. Sets fixture options for all tests in a file or describe block. Placed at the top of a describe or file. e.g. test.use({storageState:'auth.json'}).
Execution orderHooks run in nesting order: outermost-first for before hooks, innermost-first for after hooks. Multiple beforeEach in same scope run in registration order (top to bottom).
Worker isolationbeforeAll/afterAll run once per worker, not once globally. If 4 workers run tests in parallel, beforeAll runs 4 times (once per worker). For truly global once-only setup, use globalSetup.

Core Concepts — Session 14

⬡ Full Hook Execution Order

Understanding execution order is critical. For nested describes, hooks wrap from outside in. This is deterministic — never random.

1. outer beforeAll→runs once (first test in outer)
2. outer beforeEach→runs before EVERY test
3. inner beforeEach→runs before inner tests only
4. TEST RUNS ✓
5. inner afterEach→runs after inner tests only
6. outer afterEach→runs after EVERY test
7. inner afterAll→once (after last inner test)
8. outer afterAll→once (after last outer test)

⬡ beforeAll vs globalSetup — When to Use Which

beforeAll→runs once per WORKER (not truly global)
beforeAll→has access to page/context/request fixtures
beforeAll→scoped to test file or describe block
globalSetup→runs ONCE globally before any worker starts
globalSetup→no fixtures — plain Node.js function
globalSetup→use for: save storageState, DB migration, env setup
Anti-pattern→using beforeAll for login (runs N times per worker)
Correct→login in globalSetup, restore in test.use(storageState)

⬡ Hook State Sharing — Variables Between Hooks and Tests

Variables declared in the describe block scope are shared between hooks and tests. This is how you pass data from beforeAll/beforeEach to tests.

describe block scope→let userId: number — accessible everywhere
beforeAll→userId = await api.createUser()
test 1→uses userId (created in beforeAll)
test 2→uses same userId (same worker)
afterAll→await api.deleteUser(userId)
⚠ Warning→shared state = parallel tests UNSAFE (use mode:'serial')

⬡ Hook Timeout and Failure Behavior

Hooks respect the global timeout. If a hook fails, tests in that scope are marked as failed with the hook's error. afterEach and afterAll still run even when tests fail.

beforeAll fails→ALL tests in block are skipped/failed
beforeEach fails→that test fails, afterEach still runs
test fails→afterEach still runs (cleanup guaranteed)
afterEach fails→test is marked failed even if test passed
custom timeout→test.beforeAll(async () => {...}, 60_000)
Best Practices — Session 14
  • ✓
    Use beforeEach for per-test navigation and state reset. Never share mutable state between tests through variables set in beforeEach — that breaks parallel isolation.
  • ✓
    Keep beforeAll for truly shared, read-only setup (seeding reference data) or append-only state. If tests mutate the shared state, they'll interfere with each other when run in parallel.
  • ✓
    Always put teardown in afterEach / afterAll, never at the end of a test body. End-of-test cleanup is skipped on test failure — hook teardown is not.
  • ✓
    Use testInfo.status in afterEach for conditional debugging: only capture screenshots/logs when the test failed. Saves CI artifact space significantly.
  • ✓
    Prefer fixtures over beforeEach for shared setup that needs to be reused across multiple spec files. A fixture is composable; beforeEach is file-scoped.
  • ✓
    For login setup: never use beforeAll for browser login. Use globalSetup to login once and save storageState, then test.use({storageState:'auth.json'}) in your file. This runs once globally, not per worker.
  • ✓
    When using beforeAll to create shared data, always have a corresponding afterAll to delete it. Orphaned test data accumulates and causes interference between runs.
  • ✓
    Add a custom timeout to beforeAll when it does network operations: test.beforeAll(async () => {...}, {timeout: 60_000}). The global test timeout is often too short for complex setup.
Interview Prep — 5 SDET Q&A
Q1: What is the execution order when you have nested describe blocks with multiple hooks? ▶
Execution order follows an inside-out pattern for teardown and outside-in for setup. For a test inside an inner describe: (1) Outer beforeAll runs once at the start. (2) Outer beforeEach runs. (3) Inner beforeEach runs. (4) TEST EXECUTES. (5) Inner afterEach runs. (6) Outer afterEach runs. (7) Inner afterAll runs once after last test in inner describe. (8) Outer afterAll runs once after last test in outer describe. This is deterministic and documented. The key insight: setup goes outside-in (wrapping), teardown goes inside-out (unwrapping). If you have multiple beforeEach in the same scope, they run in the order they were registered (top to bottom in the file).
Q2: Why is using beforeAll for browser login an anti-pattern? ▶
Two problems: (1) Worker isolation — beforeAll runs once per Playwright worker, not once globally. In a suite with 4 parallel workers, beforeAll executes 4 separate login flows, negating the performance benefit. (2) Shared page state — if you login in beforeAll and share the page across tests (which requires mode:'serial'), tests are no longer isolated. One test can leave the browser in a state that breaks the next test. The correct approach: run login once in globalSetup (truly global, runs before any worker starts), save the auth cookies/storage to a file with context.storageState({path:'auth.json'}), then restore per test with test.use({storageState:'auth.json'}). Each test gets a fresh authenticated context without re-running the login UI flow.
Q3: How do you pass data from beforeAll to individual tests? ▶
Declare a variable in the describe block scope (outside hooks and tests): let createdUser: {id:number, email:string}. In beforeAll, assign to it: test.beforeAll(async ({request}) => { const r = await request.post('/api/users', {data:{...}}); createdUser = await r.json(); }). Each test can then read createdUser.id without re-creating it. The variable lives in the describe block's closure scope — all hooks and tests in that block share it. Important caveat: this is safe only if tests READ the shared variable without modifying it. If tests mutate the shared state, run the describe block in serial mode: test.describe.configure({mode:'serial'}). In afterAll, clean up: await request.delete('/api/users/' + createdUser.id).
Q4: How do you capture a screenshot only when a test fails, using hooks? ▶
In afterEach, check testInfo.status against testInfo.expectedStatus: test.afterEach(async ({page}, testInfo) => { if (testInfo.status !== testInfo.expectedStatus) { const screenshotPath = testInfo.outputPath('failure.png'); await page.screenshot({path: screenshotPath, fullPage: true}); await testInfo.attach('failure-screenshot', {path: screenshotPath, contentType:'image/png'}); console.error('Test failed:', testInfo.title, 'on URL:', page.url()); } }). testInfo.expectedStatus is normally 'passed', so the condition fires when the test actually failed, timed out, or was unexpectedly skipped. testInfo.outputPath() creates a unique file path inside the test's dedicated output folder, preventing parallel tests from overwriting each other's screenshots. testInfo.attach() makes the screenshot visible in the HTML report.
Q5: What happens if beforeEach throws an error? ▶
When beforeEach throws, the test it was preparing is marked as failed with the hook's error as the failure reason. The test body never executes. However, afterEach still runs for that test — Playwright guarantees afterEach execution regardless of what happened in beforeEach or the test. This is critical: if beforeEach creates partial state (started a DB transaction, created a user), afterEach can still clean it up. If beforeAll throws, ALL tests in that describe block are marked as failed and skipped — none of their individual before/afterEach hooks run, only afterAll still runs for cleanup. This is why beforeAll failures are severe: they take down the entire test group. Keep beforeAll lean and fault-tolerant.
all-hooks-basic.ts
// all-hooks-basic.ts
// PURPOSE: Demonstrates all 4 hooks in one file with clear comments
// Shows: beforeAll, afterAll, beforeEach, afterEach with fixtures

import { test, expect } from '@playwright/test';

// ── FILE-LEVEL HOOKS — run for EVERY test in this file ────────────

// beforeAll: runs ONCE before the first test (per worker)
test.beforeAll(async ({ request }) => {
  console.log('🔧 beforeAll: seed reference data');
  // Seed product categories used by all tests
  await request.post('/api/categories', {
    data: { name: 'Electronics', slug: 'electronics' },
  });
});

// afterAll: runs ONCE after the last test (per worker)
test.afterAll(async ({ request }) => {
  console.log('🧹 afterAll: remove seeded data');
  await request.delete('/api/categories/electronics');
});

// beforeEach: runs before EVERY test — fresh state per test
test.beforeEach(async ({ page }) => {
  console.log('⚡ beforeEach: navigate to homepage');
  await page.goto('https://myapp.com');
  // Verify we're on the correct page before each test runs
  await expect(page).toHaveURL(/myapp\.com/);
});

// afterEach: runs after EVERY test (even on failure)
test.afterEach(async ({ page }, testInfo) => {
  console.log(`✅ afterEach: test "${testInfo.title}" → ${testInfo.status}`);
  // Screenshot only on failure
  if (testInfo.status !== testInfo.expectedStatus) {
    const path = testInfo.outputPath('failure.png');
    await page.screenshot({ path, fullPage: true });
    await testInfo.attach('failure-screenshot', {
      path, contentType: 'image/png',
    });
  }
});

// ── TESTS — they get all hooks above for free ─────────────────────

test('homepage title is correct', async ({ page }) => {
  // beforeEach already navigated here — just assert
  await expect(page).toHaveTitle(/MyApp/);
});

test('electronics category exists', async ({ page }) => {
  // beforeAll seeded the category — it should appear in nav
  await expect(page.getByRole('link', { name: 'Electronics' }))
    .toBeVisible();
});

beforeall-data-sharing.ts
// beforeall-data-sharing.ts
// PURPOSE: Share data from beforeAll to all tests via describe-scope variables
// IMPORTANT: Only safe when tests do NOT mutate the shared data

import { test, expect } from '@playwright/test';

test.describe('Product CRUD — shared product ID', () => {

  // ── Describe-scope variable — shared across all hooks + tests ─────
  let productId: number;
  let productSlug: string;

  // SETUP: create ONE product — all tests will read it
  test.beforeAll(async ({ request }) => {
    const response = await request.post('/api/products', {
      data: { name: 'Test Laptop', price: 75000, stock: 100 },
    });
    expect(response.ok()).toBeTruthy();
    const body = await response.json();
    productId   = body.id;
    productSlug = body.slug;
    console.log(`Created product: id=${productId}, slug=${productSlug}`);
  });

  // TEARDOWN: delete after all tests — NOT after each
  test.afterAll(async ({ request }) => {
    await request.delete(`/api/products/${productId}`);
    console.log(`Deleted product: id=${productId}`);
  });

  // All tests READ the shared productId — none modify it
  test('product detail page loads', async ({ page }) => {
    await page.goto(`/products/${productSlug}`);
    await expect(page.locator('h1')).toContainText('Test Laptop');
  });

  test('product appears in search results', async ({ page }) => {
    await page.goto('/search?q=Test+Laptop');
    await expect(page.locator(`.product-card[data-id="${productId}"]`))
      .toBeVisible();
  });

  test('product API returns correct data', async ({ request }) => {
    const r = await request.get(`/api/products/${productId}`);
    const data = await r.json();
    expect(data.name).toBe('Test Laptop');
    expect(data.price).toBe(75000);
  });
});
aftereach-conditional.ts
// aftereach-conditional.ts
// PURPOSE: Conditional logic in afterEach — screenshot + log on failure
// PATTERN: testInfo.status !== testInfo.expectedStatus = test failed

import { test, expect } from '@playwright/test';
import fs from 'fs';

test.afterEach(async ({ page }, testInfo) => {

  const failed = testInfo.status !== testInfo.expectedStatus;

  if (failed) {
    // 1. Full-page screenshot → attached to HTML report
    const ssPath = testInfo.outputPath('failure-screenshot.png');
    await page.screenshot({ path: ssPath, fullPage: true });
    await testInfo.attach('failure-screenshot', {
      path: ssPath, contentType: 'image/png',
    });

    // 2. Log current URL for context
    await testInfo.attach('failing-url', {
      body: page.url(),
      contentType: 'text/plain',
    });

    // 3. Capture DOM as HTML for offline inspection
    const html = await page.content();
    const htmlPath = testInfo.outputPath('dom-snapshot.html');
    fs.writeFileSync(htmlPath, html);
    await testInfo.attach('dom-snapshot', {
      path: htmlPath, contentType: 'text/html',
    });

    // 4. Add Jira annotation to test report
    testInfo.annotations.push({
      type: 'environment',
      description: `Failed URL: ${page.url()}`,
    });

    // 5. Console output for CI logs
    console.error([
      `❌ TEST FAILED: ${testInfo.title}`,
      `   Status:  ${testInfo.status}`,
      `   Retry:   ${testInfo.retry}`,
      `   URL:     ${page.url()}`,
    ].join('\n'));
  }

  // Always log duration regardless of pass/fail
  console.log(`⏱  ${testInfo.title}: ${testInfo.duration}ms [${testInfo.status}]`);
});

test('passing test — no artifacts', async ({ page }) => {
  await page.goto('https://myapp.com');
  await expect(page.locator('h1')).toBeVisible(); // passes → no screenshot
});

test.fail('intentionally failing test — artifacts captured', async ({ page }) => {
  await page.goto('https://myapp.com');
  await expect(page.locator('.nonexistent')).toBeVisible(); // fails → screenshots
});

// beforeall-api-seed.ts
// PURPOSE: Seed complex test data in beforeAll — clean up in afterAll
// PATTERN: API-first setup for reliable, fast test data creation

import { test, expect } from '@playwright/test';

test.describe('Order management suite', () => {

  // Shared state — describe block scope
  const seededData: {
    userId: number;
    orderId: number;
    productId: number;
  } = {} as any;

  test.beforeAll(async ({ request }) => {
    console.log('🌱 Seeding order test data...');

    // Create user
    const userResp = await request.post('/api/users', {
      data: { email: 'ordertest@example.com', role: 'buyer' },
    });
    seededData.userId = (await userResp.json()).id;

    // Create product
    const prodResp = await request.post('/api/products', {
      data: { name: 'Test Widget', price: 1000, stock: 99 },
    });
    seededData.productId = (await prodResp.json()).id;

    // Create order linking user + product
    const orderResp = await request.post('/api/orders', {
      data: { userId: seededData.userId, productId: seededData.productId, qty: 2 },
    });
    seededData.orderId = (await orderResp.json()).id;

    console.log(`✅ Seeded: user=${seededData.userId}, order=${seededData.orderId}`);
  });

  test.afterAll(async ({ request }) => {
    console.log('🗑  Cleaning up order test data...');
    await request.delete(`/api/orders/${seededData.orderId}`);
    await request.delete(`/api/products/${seededData.productId}`);
    await request.delete(`/api/users/${seededData.userId}`);
  });

  test('order appears in orders list', async ({ page }) => {
    await page.goto(`/admin/orders/${seededData.orderId}`);
    await expect(page.locator('.order-id'))
      .toContainText(String(seededData.orderId));
  });

  test('order has correct product details', async ({ page }) => {
    await page.goto(`/admin/orders/${seededData.orderId}`);
    await expect(page.locator('.product-name')).toContainText('Test Widget');
  });
});
hooks-vs-fixtures.ts
// hooks-vs-fixtures.ts
// PURPOSE: Show when to use hooks vs fixtures — comparison guide
// RULE: Fixtures for reusable cross-file setup; Hooks for in-file logic

import { test as base, expect } from '@playwright/test';

// ── APPROACH 1: Fixtures (preferred for reusability) ──────────────
const fixtureTest = base.extend({
  // Fixture: creates user, yields to test, deletes after
  freshUser: async ({ request }, use, testInfo) => {
    const r = await request.post('/api/users', {
      data: { email: `user_${testInfo.testId}@test.com` },
    });
    const user = await r.json();
    await use(user);                           // test runs here
    await request.delete(`/api/users/${user.id}`); // teardown
  },
});

// ✅ Fixture test — freshUser auto-created and deleted
fixtureTest('edit user via fixture', async ({ page, freshUser }) => {
  await page.goto(`/admin/users/${freshUser.id}`);
  await expect(page.locator('.user-email')).toContainText(freshUser.email);
});

// ── APPROACH 2: Hooks (for file/describe-scoped logic) ────────────
test.describe('User profile — hook approach', () => {

  let userId: number;

  // Hook: runs before EACH test in this describe — fresh user each time
  test.beforeEach(async ({ request }, testInfo) => {
    const r = await request.post('/api/users', {
      data: { email: `hook_${testInfo.testId}@test.com` },
    });
    userId = (await r.json()).id;
  });

  // Hook: cleanup after each test — pairs with beforeEach
  test.afterEach(async ({ request }) => {
    await request.delete(`/api/users/${userId}`);
  });

  test('edit user via hooks', async ({ page }) => {
    await page.goto(`/admin/users/${userId}`);
    await expect(page.locator('.user-id'))
      .toContainText(String(userId));
  });
});

// nested-hooks.ts
// PURPOSE: Complex nested hooks with multiple levels of setup
// Shows: scope isolation, override behavior, test.use() with hooks

import { test, expect } from '@playwright/test';

// ── TOP-LEVEL HOOKS (run for ALL tests in this file) ──────────────
test.beforeEach(async ({ page }, testInfo) => {
  console.log(`[FILE] beforeEach → test: ${testInfo.title}`);
  await page.goto('https://myapp.com');
});

test.afterEach(async ({}, testInfo) => {
  console.log(`[FILE] afterEach → ${testInfo.status}`);
});

// ── DESCRIBE 1: Admin tests (adds admin auth + navigation) ─────────
test.describe('Admin area', () => {

  // Admin-specific: navigates to /admin after file-level goto('/') 
  test.beforeEach(async ({ page }) => {
    console.log('[ADMIN] beforeEach → navigate to /admin');
    await page.goto('https://myapp.com/admin');
  });

  test('admin dashboard visible', async ({ page }) => {
    // Execution: [FILE]beforeEach → [ADMIN]beforeEach → TEST → [FILE]afterEach
    await expect(page).toHaveURL(/admin/);
    await expect(page.locator('.admin-dashboard')).toBeVisible();
  });

  // ── NESTED DESCRIBE: specific admin section ──────────────────────
  test.describe('Admin Users section', () => {

    // Additional navigation for this specific section
    test.beforeEach(async ({ page }) => {
      console.log('[USERS] beforeEach → click Users nav');
      await page.click('nav a[href="/admin/users"]');

Session 15 · test.describe() Configurations & All Keys

test.describe() groups related tests together and provides a scope for hooks, configurations, and modifiers. Every test in a describe block shares its context — and test.describe.configure() controls HOW those tests run: in parallel, serially, with retries, or with specific timeouts.

WHAT is test.describe?

A named block that groups related tests. It creates a scope for hooks, test.use() options, and modifiers. It appears in test reports as a parent heading. Can be nested to any depth.

WHY use it?

Organization — group tests by feature, page, or user role. Scoping — hooks and test.use() inside a describe only affect tests in that block. Configuration — describe.configure() controls execution mode.

Execution modes

mode:'parallel' — all tests in the block run simultaneously. mode:'serial' — tests run in order, one at a time; if one fails, subsequent tests are skipped. Default is parallel across files, serial within a file.

Modifier methods

test.describe.only() — run only this block. test.describe.skip() — skip this block. test.describe.fixme() — mark block as broken. test.describe.serial() — shorthand for configure({mode:'serial'}).

All test.* keys

Individual test keys: test.only(), test.skip(), test.fixme(), test.fail(), test.slow(), test.step(), test.setTimeout(), test.use(), test.info(), test.extend(). These can be called on individual tests or describe blocks.

Keyword Glossary — Session 15
test.describe()Groups related tests under a named block. Creates scope for hooks, test.use(), and modifiers. Renders as parent heading in HTML report. Accepts optional function as only argument.

test.only()Marks one test to run exclusively. All other tests in the suite are skipped. Also: test.describe.only(). NEVER commit — CI checks for .only with forbidOnly:true.
test.skip()Marks a test or block as skipped. Can be conditional: test.skip(condition, 'reason'). Skipped tests show in reports as skipped (not failed). Runtime-safe — evaluated per run.
test.fixme()Marks a known-broken test. Like skip but communicates "this is broken and needs fixing". Appears with a fixme icon in reports. Conditional form: test.fixme(condition, 'reason').
test.fail()Marks a test that is EXPECTED to fail. The test passes if it fails (inverted assertion). Use to document known bugs without removing the test. Fails if the test unexpectedly passes.
test.slow()Multiplies the test timeout by 3. Use for inherently slow tests (video processing, email delivery). Better than globally increasing timeout for all tests.
test.setTimeout()Sets explicit timeout in ms for a specific test or describe block: test.setTimeout(60_000). Overrides the global timeout for that test only.
test.step()Groups actions into a named logical step. Visible in trace viewer and HTML report. Can be nested. Returns a value (the step callback's return value). Async/await supported.

PARALLEL (default)
Tests are independent — no shared state mutation
Each test creates its own data
Faster execution — 3–10× speedup with workers
SERIAL (configure mode:'serial')
Tests DEPEND on previous test state
Wizard flows: step 1 → step 2 → step 3
Test 2 SKIPPED if test 1 fails
Runs in ONE worker only — no parallel benefit

Best Practices — Session 15
  • ✓
    Always set forbidOnly: !!process.env.CI in playwright.config.ts. This makes CI fail if anyone pushes a test.only() accidentally — the most common cause of "no tests ran" in CI.
  • ✓
    Use mode:'serial' sparingly — only for genuine wizard/multi-step flows where tests truly depend on each other's state. Default to parallel. Serial tests are slower and skip on first failure.
  • ✓
    Prefer test.skip(condition, 'reason') over removing a test. The reason string appears in reports and communicates WHY the test is skipped, not just that it is.
  • ✓
    Use test.fail() to track known bugs — it creates a failing test record that PASSES until the bug is fixed, then turns red automatically, reminding you to remove the annotation.
  • ✓
    Put test.describe.configure() as the FIRST line inside a describe block — before any hooks or tests. It's declarative and must be evaluated during planning, not during execution.
  • ✓
    Use test.slow() instead of test.setTimeout(large_number). It communicates intent, scales proportionally with config changes, and is more self-documenting.
  • ✓
    Structure describe blocks to mirror your app's information architecture: describe('Feature') → describe('Sub-feature') → test('specific behavior'). Reports become self-documenting.
  • ✓
    Use test.describe.configure({retries: 2}) for groups of known-flaky tests (e.g., third-party widget tests) instead of setting global retries — avoids wasting retry time on stable tests.
Interview Prep — 5 SDET Q&A
Q1: What is the difference between test.describe.configure({mode:'serial'}) and test.describe.serial()? ▶
They are identical in behavior — test.describe.serial() is a shorthand for test.describe.configure({mode:'serial'}). The configure() version is more powerful because you can pass multiple options: mode, retries, and timeout all at once. test.describe.serial() is more readable when you ONLY want to set serial mode. Similarly, test.describe.parallel() is shorthand for test.describe.configure({mode:'parallel'}). I use configure() when I need to combine options: test.describe.configure({mode:'serial', retries:0, timeout:120_000}) — no retries for serial flows (retry causes state confusion) with a generous timeout for multi-step scenarios.
Q2: When would you use test.describe.configure({mode:'serial'}) in a real project? ▶
Real use cases: (1) E-commerce checkout wizard — add-to-cart → fill address → payment → confirmation. Step 4 literally cannot be tested if step 3 (payment) failed because there's no order. Running them serially and skipping on failure prevents meaningless failures. (2) Onboarding flow — account creation → email verification → profile setup → first action. (3) Data import workflow — upload file → validate → process → verify results. The key signal: test B reads/depends on side effects that test A must create first. Important caveat: set retries:0 for serial describe blocks. Retrying a failed step in a wizard can cause data duplication or corrupted state, since the previous steps' side effects persist between retries.
Q3: How does test.fail() work and when should you use it? ▶
test.fail() inverts the test's pass condition. The test PASSES when its assertions fail, and FAILS when its assertions unexpectedly pass. Use case: documenting confirmed bugs. When you find a bug in production, write a test that exposes it, mark it test.fail('BUG-1234: button is broken'). The test suite remains green (because the bug makes the test fail as expected). When the developer fixes the bug, the test suddenly PASSES its assertions — which means test.fail() now fails the test (because it expected failure but got success). This creates an automatic alert: "remove the test.fail() annotation because the bug is fixed." It prevents the common problem of finding broken tests that nobody noticed were broken for months.
Q4: What is the difference between test.use() and test.describe.configure()? ▶
They configure completely different things. test.use({options}) sets fixture values — browser settings like viewport, storageState, isMobile, locale, userAgent, geolocation, extraHTTPHeaders. It's about WHAT the test gets (the environment). test.describe.configure({mode, retries, timeout}) controls execution behavior — HOW tests run (in what order, how many retries, how long they have). They're complementary: you can use both in the same describe block. Example: test.describe('Mobile admin', () => { test.use({isMobile:true, storageState:'admin.json'}); // WHAT test.describe.configure({retries:1, timeout:45_000}); // HOW test.beforeEach(async ({page}) => {...}); test('mobile admin test', ...); }). The mobile admin tests get a mobile browser with admin auth (test.use), run with 1 retry and 45s timeout (configure).
Q5: How do you organize a large spec file with hundreds of tests using describe blocks? ▶
My strategy for large spec files: (1) Top-level by major feature area: describe('Product Catalog'), describe('Shopping Cart'), describe('Checkout'). (2) Second level by user role or scenario: describe('Guest user'), describe('Logged-in user'), describe('Admin'). (3) Third level by sub-feature: describe('Search and Filter'), describe('Product Detail'). (4) Use test.use() at each level for role-specific auth and viewport. (5) Use test.describe.configure({mode:'parallel'}) at the top level to run feature groups in parallel. (6) Use test.describe.configure({mode:'serial'}) only for inherently sequential groups. (7) Keep each test independent within parallel groups — no shared mutable state. The resulting structure in HTML report shows: Checkout > Admin > Order Management > bulk cancel works. This mirrors the product specification structure, making it easy for non-engineers to understand coverage.

import { test, expect } from '@playwright/test';

// ── test.describe() variants ──────────────────────────────────────
test.describe('All test.describe() methods demo', () => {

  // test.describe.configure() — set mode/retries/timeout for block
  test.describe.configure({
    mode: 'parallel', // 'parallel' | 'serial' | 'default'
    retries: 1,
    timeout: 45_000,
  });

  // test.use() — fixture options for this describe block
  test.use({
    viewport: { width: 1280, height: 720 },
    locale: 'en-IN',
    timezoneId: 'Asia/Kolkata',
  });

  // test() — normal test
  test('normal passing test', async ({ page }) => {
    await page.goto('/');
    await expect(page).toHaveTitle(/MyApp/);
  });

  // test.skip() — unconditional skip
  test.skip('skipped test', async () => { /* never runs */ });

  // test.fail() — expected to fail (inverted pass)
  test.fail('known bug BUG-123', async ({ page }) => {
    await page.goto('/broken-feature');
    await expect(page.locator('.broken')).toBeVisible();
  });

  // test.fixme() — broken, needs fixing (skip variant)
  test.fixme('FIXME: refactor needed', async () => { /* never runs */ });

  // test with inline options (overrides describe config)
  test('inline options override',
    { timeout: 60_000, retries: 0 }, // per-test inline options
    async ({ page }) => {

      // test.slow() — multiply timeout × 3
      test.slow(); // or test.slow(true, 'reason')

      // test.setTimeout() — explicit ms override
      test.setTimeout(90_000);

      // test.step() — named action group
      await test.step('navigate', async () => {
        await page.goto('/');
      });

      // test.info() — access TestInfo without parameter
      const info = test.info();
      console.log(`Running: ${info.title} retry=${info.retry}`);

      // conditional skip inside test body
      test.skip(info.retry > 1, 'Give up after 2 retries');

      // conditional fail inside test body
      test.fail(info.retry > 0, 'Expected failure on retry');
    }
  );
});
