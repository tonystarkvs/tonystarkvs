Introduction to Playwright + TypeScript

Set up your automation environment and write your very first test from scratch.

Overview — What / Why / When / Where / How
What
Playwright is a Node.js library by Microsoft for end-to-end browser automation — Chromium, Firefox, WebKit.
Why
Faster than Selenium, auto-waits built-in, supports multiple browsers in one run, works headless or headed.
When
Use for UI regression, smoke tests, E2E testing of web apps before every release or PR merge.
Where
Runs locally, in Docker, GitHub Actions, Azure Pipelines — any Node.js environment.
How
npm init → install Playwright → write test() blocks → run with npx playwright test.
Interview One-liners
"Playwright is a Microsoft-maintained E2E testing framework that supports Chromium, Firefox, and WebKit out of the box."
"Unlike Selenium, Playwright auto-waits for elements to be actionable before interacting — no Thread.sleep() needed."
"TypeScript gives us compile-time type checking on our test code, catching errors before we even run the tests."
"Playwright runs tests in true browser contexts, not simulated DOM environments like jsdom used by Jest."
"The playwright.config.ts file is the single source of truth for browser targets, base URL, retries, and timeouts."
Architecture Diagram
Your Test Code (.spec.ts) ↓ calls Playwright Test Runner ←── playwright.config.ts ↓ launches Browser (Chromium / Firefox / WebKit) ↓ opens Browser Context (isolated session — cookies, storage) ↓ creates Page (tab — has URL, DOM, network) ↓ provides Locator API → Actions → Assertions
Code Examples
Terminal — Install Playwright
# Step 1: Create a new project folder and initialise npm
mkdir my-pw-tests && cd my-pw-tests
npm init -y

# Step 2: Install Playwright with TypeScript support
# --yes accepts all defaults: chromium + firefox + webkit browsers
npm init playwright@latest

# Step 3: Run the sample test Playwright generated for you
npx playwright test

# Step 4: Open the HTML test report in browser
npx playwright show-report
tests/first.spec.ts — Your First Test
// 'test' and 'expect' are Playwright's core imports
import { test, expect } from '@playwright/test';

// test() defines a single test case
// First arg: test name (shows in report)
// Second arg: async callback receives { page } — the browser tab object
test('homepage has correct title', async ({ page }) => {

  // page.goto() navigates the browser to a URL
  // await = wait for full navigation before next line
  await page.goto('https://playwright.dev');

  // expect() + toHaveTitle() asserts the page title matches
  // Playwright auto-retries this assertion for up to 5 seconds
  await expect(page).toHaveTitle(/Playwright/);
});
playwright.config.ts — Central Configuration
import { defineConfig, devices } from '@playwright/test';

// defineConfig gives TypeScript auto-complete on config options
export default defineConfig({

  // testDir: where Playwright looks for .spec.ts files
  testDir: './tests',

  // fullyParallel: run ALL tests at the same time
  fullyParallel: true,

  // retries: auto-retry failing tests in CI (not locally)
  retries: process.env.CI ? 2 : 0,

  // reporter: 'html' creates a rich clickable report
  reporter: 'html',

  // use: settings shared across ALL projects
  use: {
    // baseURL lets you write page.goto('/login') instead of full URL
    baseURL: 'https://myapp.com',
    // trace: record a full trace on first retry for debugging
    trace: 'on-first-retry',
  },
});
playwright.config.ts — Multi-Browser Projects
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  projects: [
    // Each project = one browser target
    // Playwright runs ALL your tests on EACH project
    {
      name: 'chromium',
      // devices['Desktop Chrome'] sets viewport, userAgent automatically
      use: { ...devices['Desktop Chrome'] },
    },
    {
      name: 'firefox',
      use: { ...devices['Desktop Firefox'] },
    },
    {
      name: 'mobile-chrome',
      // Emulates a Pixel 5 screen size and touch events
      use: { ...devices['Pixel 5'] },
    },
  ],
});
tests/screenshot.spec.ts — Capture & Visual Debug
import { test, expect } from '@playwright/test';

test('capture screenshot for debugging', async ({ page }) => {
  await page.goto('https://playwright.dev');

  // page.screenshot() saves a PNG of the current viewport
  // path: where to save it (relative to project root)
  await page.screenshot({ path: 'screenshots/homepage.png' });

  // fullPage: true captures the ENTIRE scrollable page, not just visible area
  await page.screenshot({
    path: 'screenshots/full-page.png',
    fullPage: true,
  });

  // toHaveScreenshot() does VISUAL COMPARISON — fails if pixels differ
  // Saves a baseline image on first run, compares on subsequent runs
  await expect(page).toHaveScreenshot('homepage-baseline.png');
});
Best Practices Checklist
Always use async/await — never mix with callbacks or .then() chains in tests
Set baseURL in config so tests aren't coupled to a hardcoded domain
Enable trace: 'on-first-retry' in config to get click-by-click debug replays in CI
Name tests clearly: describe WHAT is tested, not HOW (e.g. "user can log in" not "click login button")
Commit playwright.config.ts to version control — it IS your test strategy documented in code

Locators — Finding Elements Like a Pro

Master Playwright's smart locator API — the foundation of every reliable test.

Overview
What
Locators are lazy, auto-retrying handles to DOM elements. They find and interact with elements on your page.
Why
Playwright auto-waits until element is visible, enabled, stable — eliminating flaky timing issues.
When
Every time you need to click, type, read, or assert anything on the page.
Where
In your test files and Page Object Model classes — anywhere you reference a UI element.
How
page.getByRole(), getByText(), getByLabel(), getByTestId() — prefer semantic over CSS/XPath.
Interview One-liners
"Playwright locators are lazy — they don't hit the DOM until you call an action, so stale element exceptions are impossible."
"getByRole() is my preferred locator because it mirrors how assistive technologies see the page — robust and accessible."
"I avoid CSS selectors with class names in tests because class names change during refactoring; data-testid attributes are stable."
"Playwright's auto-wait polls the DOM up to the timeout limit — no explicit waits needed for standard interactions."
"Chaining locators like page.getByRole('list').getByRole('listitem') scopes the search to a subtree, preventing wrong-element matches."
Locator Priority (Best → Avoid)
🏆 1. getByRole() — semantic, accessible, most resilient ✅ 2. getByLabel() — form inputs tied to their label text ✅ 3. getByPlaceholder() — inputs without visible label ✅ 4. getByText() — buttons, links, headings by visible text ✅ 5. getByTestId() — data-testid attr, team-agreed standard ⚠️ 6. locator('css') — OK for stable selectors, fragile otherwise ❌ 7. locator('xpath') — brittle, hard to read, last resort only
Code Examples
getByRole — Semantic Element Targeting
import { test, expect } from '@playwright/test';

test('login using semantic locators', async ({ page }) => {
  await page.goto('/login');

  // getByRole('button') finds <button> elements
  // { name: 'Sign in' } filters by accessible name (text content)
  // This matches: <button>Sign in</button>
  await page.getByRole('button', { name: 'Sign in' }).click();

  // getByRole('heading') finds <h1>–<h6>
  // { level: 1 } narrows to only <h1>
  await expect(
    page.getByRole('heading', { level: 1 })
  ).toHaveText('Welcome back');

  // getByRole('link') targets <a href> tags
  await page.getByRole('link', { name: 'Forgot password?' }).click();
});
getByLabel — Form Inputs
test('fill login form by label', async ({ page }) => {
  await page.goto('/login');

  // getByLabel('Email') finds the INPUT associated with
  // <label>Email</label> via for="..." or wrapping
  // fill() clears field then types the value
  await page.getByLabel('Email').fill('user@example.com');

  // For password fields — same pattern
  await page.getByLabel('Password').fill('Secret123');

  // getByRole for submit button — exact:true prevents partial matches
  await page.getByRole('button', { name: 'Login', exact: true }).click();

  // Assert we're redirected to dashboard
  await expect(page).toHaveURL(/dashboard/);
});
getByTestId — Stable Custom Attribute
// In your HTML: <button data-testid="submit-btn">Submit</button>
// In playwright.config.ts: use: { testIdAttribute: 'data-testid' }

test('use data-testid for stable selectors', async ({ page }) => {
  await page.goto('/checkout');

  // getByTestId() reads the data-testid attribute
  // Unaffected by styling changes, text changes, or restructuring
  await page.getByTestId('submit-btn').click();

  // Checking a data-testid element's visibility
  await expect(
    page.getByTestId('success-banner')
  ).toBeVisible();

  // Getting text content from a testid element
  const msg = await page.getByTestId('success-banner').textContent();
  console.log(msg); // "Order placed successfully!"
});
Chaining Locators — Scope to a Section
test('interact inside a specific card', async ({ page }) => {
  await page.goto('/products');

  // Problem: page may have MANY "Add to Cart" buttons
  // Solution: first scope to the right product card

  // Step 1: Find the container element (the product card)
  const productCard = page
    .getByTestId('product-card')
    .filter({ hasText: 'Wireless Headphones' });

  // Step 2: Search INSIDE that card for the button
  // This won't accidentally click "Add to Cart" on another product
  await productCard
    .getByRole('button', { name: 'Add to Cart' })
    .click();

  // Step 3: Assert cart count updated
  await expect(page.getByTestId('cart-count')).toHaveText('1');
});
filter() — Narrow Down Multiple Matches
test('use filter to find the right list item', async ({ page }) => {
  await page.goto('/todo');

  // All todo items match getByRole('listitem')
  // filter({ hasText: ... }) keeps only the one containing that text
  const todoItem = page
    .getByRole('listitem')
    .filter({ hasText: 'Buy groceries' });

  // Now interact only with that specific todo
  await todoItem.getByRole('checkbox').check();

  // hasNot: filters OUT items containing the text
  const activeTodos = page
    .getByRole('listitem')
    .filter({ hasNotText: 'completed' });

  // count() returns the number of matched elements
  await expect(activeTodos).toHaveCount(3);
});
Best Practices
Follow locator priority — getByRole > getByLabel > getByTestId > CSS. Never start with XPath.
Add data-testid attributes to key UI elements as part of the developer–QA contract.
Always chain locators to scope searches when the same element pattern appears multiple times on a page.
Use exact: true when partial text matching could accidentally match the wrong element.
Avoid nth() positional selectors — they break when the page order changes.

Assertions — Verifying What Matters

Write precise, auto-retrying assertions that prove your app behaves correctly.

Overview
What
Assertions validate expected vs actual state using expect(). Playwright auto-retries async assertions until they pass or timeout.
Why
Without assertions, tests can pass even when the app is broken. Assertions are your proof the feature works.
When
After every meaningful action — navigate, click, submit, API call. Every test should end with at least one assertion.
Where
Use web-first assertions (expect(locator)) for DOM checks; non-retrying assertions (expect(value)) for plain JS values.
How
await expect(page).toHaveTitle() / expect(locator).toBeVisible() / expect(locator).toHaveText()
Interview One-liners
"Playwright's web-first assertions auto-retry for the full timeout period — I never need to add explicit sleep() calls."
"I always use toHaveURL() over checking page.url() manually because toHaveURL() waits for navigation to complete."
"soft assertions with expect.soft() let the test continue after a failure so you can see ALL broken things in one run."
"toHaveText() checks the visible text — toHaveValue() checks the input value — mixing them up is a common beginner mistake."
"I wrap related assertions in a test.step() block so the HTML report shows exactly which logical step failed."
Assertion Types Map
Page Assertions → expect(page).toHaveTitle() / toHaveURL() Locator Assertions → expect(locator).toBeVisible() / toHaveText() / toBeEnabled() Input Assertions → expect(locator).toHaveValue() / toBeChecked() Count Assertions → expect(locator).toHaveCount(n) Attribute Assertions → expect(locator).toHaveAttribute('href', '/home') Soft Assertions → expect.soft(locator) — test continues after failure API/Value Assertions → expect(value).toBe() / toEqual() / toContain() [NOT auto-retried]
Code Examples
Page-Level Assertions
test('page level assertions', async ({ page }) => {
  await page.goto('/');

  // toHaveTitle: checks <title> tag — supports string or regex
  await expect(page).toHaveTitle('My App — Home');
  await expect(page).toHaveTitle(/My App/); // regex = partial match

  await page.getByRole('link', { name: 'About' }).click();

  // toHaveURL: waits for navigation then checks the URL
  // Much safer than checking page.url() immediately
  await expect(page).toHaveURL('/about');
  await expect(page).toHaveURL(/about/);
});
Visibility & State Assertions
test('element state assertions', async ({ page }) => {
  await page.goto('/dashboard');

  // toBeVisible: element exists AND is displayed (not hidden)
  await expect(page.getByTestId('welcome-banner')).toBeVisible();

  // toBeHidden: opposite of visible — display:none or not in DOM
  await expect(page.getByTestId('loading-spinner')).toBeHidden();

  // toBeEnabled: form element is NOT disabled
  await expect(page.getByRole('button', { name: 'Save' })).toBeEnabled();

  // toBeDisabled: the submit button is disabled until form is valid
  await expect(page.getByRole('button', { name: 'Submit' })).toBeDisabled();

  // toBeChecked: checkbox or radio is selected
  await expect(page.getByLabel('Remember me')).toBeChecked();
});
Text, Value & Attribute Assertions
test('text and value assertions', async ({ page }) => {
  await page.goto('/profile');

  // toHaveText: checks VISIBLE text content of element
  // Use for <p>, <span>, <div>, <button> display text
  await expect(page.getByTestId('user-name'))
    .toHaveText('Alice Johnson');

  // toHaveValue: checks INPUT field value (not its label/placeholder)
  // Use for <input>, <textarea>, <select>
  await expect(page.getByLabel('Username'))
    .toHaveValue('alice_j');

  // toHaveAttribute: checks an HTML attribute's value
  await expect(page.getByRole('link', { name: 'Profile' }))
    .toHaveAttribute('href', '/profile');

  // toContainText: partial text match — element can have more text
  await expect(page.getByTestId('bio'))
    .toContainText('Software Engineer');
});
Soft Assertions — See All Failures At Once
test('soft assertions — collect all failures', async ({ page }) => {
  await page.goto('/home');

  // expect.soft() records failure but DOES NOT stop the test
  // Great for checking multiple UI elements in one test
  await expect.soft(page.getByTestId('logo')).toBeVisible();
  await expect.soft(page.getByTestId('nav-menu')).toBeVisible();
  await expect.soft(page.getByTestId('hero-banner')).toBeVisible();
  await expect.soft(page.getByTestId('cta-button')).toHaveText('Get Started');

  // After all soft assertions, Playwright reports ALL that failed
  // Without soft: first failure stops test → you miss other problems
  // With soft: ALL broken elements reported in ONE test run
});
test.step — Grouped Assertions in Report
test('checkout flow with named steps', async ({ page }) => {

  // test.step() groups actions+assertions into a named block
  // The HTML report shows each step with pass/fail — very readable

  await test.step('Navigate to cart', async () => {
    await page.goto('/cart');
    await expect(page).toHaveURL(/cart/);
  });

  await test.step('Verify cart items', async () => {
    await expect(page.getByTestId('cart-item')).toHaveCount(2);
    await expect(page.getByTestId('total-price')).toHaveText('$59.98');
  });

  await test.step('Proceed to checkout', async () => {
    await page.getByRole('button', { name: 'Checkout' }).click();
    await expect(page).toHaveURL(/checkout/);
  });
});
Best Practices
Always await assertions — skipping await causes tests to pass silently even when they should fail
Prefer web-first assertions (expect(locator)) over manually checking element attributes — they auto-retry
Use expect.soft() when validating entire page layouts — catch all broken elements in one run
Never assert things you already set yourself — asserting that fill('alice') sets value 'alice' adds no value
Use test.step() to give meaningful names to assertion groups — makes CI reports instantly readable

Hooks — beforeEach, afterEach, beforeAll, afterAll

Control test lifecycle to eliminate duplicate setup code and ensure clean test isolation.

Overview
What
Hooks are lifecycle functions that run automatically before/after each test or the entire test suite block.
Why
DRY principle — write login/setup once, reuse across all tests. Hooks ensure every test starts in a clean, known state.
When
beforeEach for per-test setup (navigate to URL, log in). afterEach for cleanup. beforeAll for one-time expensive setup (DB seed).
Where
Inside test.describe() blocks to scope hooks to a group, or at file level to apply to all tests in that file.
How
test.beforeEach(async ({ page }) => { ... }) — receives the same fixtures as tests.
Interview One-liners
"beforeEach runs before every single test in its scope — perfect for navigating to a page or logging in."
"beforeAll runs only once before the entire describe block — I use it for database seeding or API token generation."
"afterEach is my cleanup hook — I reset state, delete test data, or take a failure screenshot there."
"Hooks in a nested describe() are scoped to that group only — they don't bleed into sibling describe blocks."
"If beforeAll is used with workers, be careful — it runs once per worker, not once globally across all workers."
Hook Execution Order
describe('Suite') ├── beforeAll() ← runs ONCE before all tests in this block │ ├── beforeEach() ← runs before EVERY test ──────────────┐ ├── test('Test 1') ← test runs │ ├── afterEach() ← runs after EVERY test ←────────────┘ │ ├── beforeEach() ← again before Test 2 ├── test('Test 2') ├── afterEach() │ └── afterAll() ← runs ONCE after all tests in this block
Code Examples
beforeEach — Navigate Before Every Test
import { test, expect } from '@playwright/test';

// test.describe() groups related tests AND their hooks
test.describe('Dashboard tests', () => {

  // beforeEach runs BEFORE every test() in this describe block
  // page fixture here is the SAME page the test will receive
  test.beforeEach(async ({ page }) => {
    // Every test starts at /dashboard — no duplication!
    await page.goto('/dashboard');
    // Wait for content to load before test starts
    await page.getByTestId('dashboard-title').waitFor();
  });

  test('shows user greeting', async ({ page }) => {
    // page is already at /dashboard — beforeEach handled it
    await expect(page.getByTestId('greeting')).toBeVisible();
  });

  test('shows stats widgets', async ({ page }) => {
    // Also starts at /dashboard automatically
    await expect(page.getByTestId('stats-widget')).toHaveCount(4);
  });
});
afterEach — Cleanup & Screenshot on Failure
test.describe('Form submission', () => {

  test.afterEach(async ({ page }, testInfo) => {
    // testInfo.status = 'passed' | 'failed' | 'skipped' | 'timedOut'
    // testInfo.expectedStatus = what was expected

    if (testInfo.status !== testInfo.expectedStatus) {
      // Only take screenshot when test FAILS — save storage
      const screenshot = await page.screenshot();

      // testInfo.attach() adds screenshot TO the HTML report
      // You can click it in the report to see exactly what failed
      await testInfo.attach('failure-screenshot', {
        body: screenshot,
        contentType: 'image/png',
      });
    }
  });

  test('submits contact form', async ({ page }) => {
    await page.goto('/contact');
    await page.getByLabel('Name').fill('Test User');
    await page.getByRole('button', { name: 'Send' }).click();
    await expect(page.getByTestId('success-msg')).toBeVisible();
  });
});
beforeAll / afterAll — One-Time Setup
import { test, expect, request } from '@playwright/test';

// Shared variable: created in beforeAll, used in all tests
let authToken: string;

test.describe('API-authenticated tests', () => {

  // beforeAll: runs ONCE for the entire describe block
  // Use for expensive operations: DB seed, API auth, browser setup
  test.beforeAll(async () => {
    const ctx = await request.newContext();
    const res = await ctx.post('/api/auth/token', {
      data: { user: 'admin', pass: 'secret' }
    });
    const body = await res.json();
    authToken = body.token; // Store token for all tests
  });

  // afterAll: clean up DB records, revoke tokens, etc.
  test.afterAll(async () => {
    console.log('Revoking token...', authToken);
    // cleanup API call here
  });

  test('uses shared auth token', async ({ page }) => {
    await page.setExtraHTTPHeaders({ 'Authorization': `Bearer ${authToken}` });
    await page.goto('/admin');
  });
});
Nested describe — Scoped Hook Execution
test.describe('User profile', () => {

  test.beforeEach(async ({ page }) => {
    await page.goto('/profile'); // Runs before ALL tests below
  });

  test('shows profile picture', async ({ page }) => {
    await expect(page.getByTestId('avatar')).toBeVisible();
  });

  // Nested describe: gets the OUTER beforeEach PLUS its own
  test.describe('Edit mode', () => {

    // This runs AFTER the outer beforeEach
    // Order: outer beforeEach → inner beforeEach → test
    test.beforeEach(async ({ page }) => {
      await page.getByRole('button', { name: 'Edit' }).click();
    });

    test('shows edit form', async ({ page }) => {
      // page is at /profile AND edit mode is open
      await expect(page.getByLabel('Display name')).toBeVisible();
    });
  });
});
test.skip / test.only — Control Execution
test.describe('Payment tests', () => {

  // test.skip(): temporarily skip a test (shown in report as skipped)
  // Don't delete tests — skip them with a reason
  test.skip('PayPal checkout', async ({ page }) => {
    // Skipped: PayPal integration not ready in staging
  });

  // test.only(): run ONLY this test in this file (useful for debugging)
  // WARNING: never commit .only() — it blocks all other tests!
  test.only('credit card checkout', async ({ page }) => {
    await page.goto('/checkout');
    // ... test code
  });

  // test.fixme(): mark as broken — runs but is expected to fail
  test.fixme('Apple Pay checkout', async ({ page }) => {
    // Known bug: PW-4521 — Apple Pay button not rendering
  });
});
Best Practices
Use beforeEach over beforeAll when possible — each test gets a fresh state, preventing test pollution
Never share mutable state (like an array of created items) between tests — each test must be independent
Add failure screenshots in afterEach with testInfo.attach() — critical for debugging CI failures
Never commit test.only() — add a lint rule or pre-commit hook to catch it automatically
Use test.skip() with a comment explaining WHY — "skipped because X" is better than silently deleted

Parallel Testing — Run Everything at Once

Cut your test suite time from 30 minutes to 3 minutes using Playwright's parallel execution engine.

Overview
What
Parallel testing runs multiple test files simultaneously — each in its own isolated browser context.
Why
Without parallelism, 100 tests at 3s each = 5 minutes. With 4 workers = ~75 seconds. Speed directly impacts developer feedback loop.
When
Always enable in CI. Enable locally once your suite grows beyond ~20 tests.
Where
Configured in playwright.config.ts via workers and fullyParallel options.
How
Set fullyParallel: true for tests-within-files to run parallel too. Set workers: N to control CPU usage.
Interview One-liners
"By default Playwright runs test FILES in parallel but tests WITHIN a file serially — fullyParallel:true changes that."
"I set workers: '50%' so Playwright uses half the CPU cores — leaving headroom for the app under test."
"Tests must be fully independent for parallelism to work — shared global state is the root cause of parallel test flakiness."
"test.describe.serial() forces a group of tests to run sequentially inside a parallel suite — useful for ordered workflows."
"In CI I use workers: process.env.CI ? 4 : 2 to cap parallelism based on available runner resources."
Parallel Execution Model
Test Suite (10 files) ↓ Playwright splits work across workers ┌──────────────────────────────────────────────────────┐ │ Worker 1 │ Worker 2 │ Worker 3 │ │ file1.spec.ts │ file3.spec.ts │ file5.spec.ts │ │ file2.spec.ts │ file4.spec.ts │ file6.spec.ts │ │ Browser Context │ Browser Context │ Browser Ctx │ │ [Isolated] │ [Isolated] │ [Isolated] │ └──────────────────────────────────────────────────────┘ ↓ All workers finish Aggregate Results → HTML Report
Code Examples
playwright.config.ts — Enable Full Parallelism
import { defineConfig } from '@playwright/test';

export default defineConfig({

  // fullyParallel: true  → tests WITHIN a file also run in parallel
  // fullyParallel: false → tests within a file run one-by-one (DEFAULT)
  // Files always run in parallel regardless of this setting
  fullyParallel: true,

  // workers: number of parallel browser instances
  // '50%' = use half of available CPU cores (good for CI)
  // undefined = Playwright chooses (half of logical CPUs)
  workers: process.env.CI ? 4 : undefined,

  // retries: retry flaky tests automatically in CI
  retries: process.env.CI ? 2 : 0,
});
Workers Configuration — CPU Control
export default defineConfig({

  // OPTION 1: Fixed number — predictable, good for known CI machines
  workers: 4,

  // OPTION 2: Percentage of CPU cores
  // On 8-core machine: '50%' = 4 workers
  workers: '50%',

  // OPTION 3: Environment-aware (BEST practice for teams)
  // CI has dedicated runners, local devs have other apps running
  workers: process.env.CI ? 4 : 2,

  // OPTION 4: Single worker = sequential (like old Selenium)
  // Use only for debugging weird parallel issues
  // workers: 1,

  // timeout: max time per test (ms) — needed when parallelism
  // can cause resource contention on slow machines
  timeout: 30000, // 30 seconds per test
});
test.describe.serial — Force Sequential Order
import { test, expect } from '@playwright/test';

// test.describe.serial() runs these tests ONE AT A TIME
// in ORDER, even when fullyParallel: true is set globally
// Use for: user registration → login → profile → logout flows
test.describe.serial('user onboarding flow', () => {

  // Test 1: MUST run first
  test('1. register new user', async ({ page }) => {
    await page.goto('/register');
    // fill form and submit...
    await expect(page).toHaveURL(/verify-email/);
  });

  // Test 2: runs AFTER Test 1 completes
  test('2. verify email and login', async ({ page }) => {
    await page.goto('/login');
    // login with the just-registered user...
    await expect(page).toHaveURL(/dashboard/);
  });

// ❌ BAD: Tests share a global counter — parallel tests corrupt it

  testCount++; // Race condition in parallel!
});

// ✅ GOOD: Each test generates unique data independently

  await page.getByLabel('Email').fill(uniqueEmail);
  // No collision with other parallel tests!
});

// Each parallel worker gets its own isolated fixture instance

Workers — Deep Dive into Playwright's Execution Engine

Understand exactly what a worker is, how it manages browser resources, and how to configure it for speed vs stability.

Overview
What
A worker is a separate Node.js process that Playwright spawns. Each worker owns one browser instance and runs test files assigned to it.
Why
Process isolation means a crash in one worker doesn't kill others. True parallelism — not just async concurrency.
When
Workers kick in the moment you run npx playwright test. Playwright's scheduler assigns test files to available workers.
Where
Each worker = separate OS process + browser instance. They don't share memory or browser state.
How
Playwright test runner → spawns N worker processes → each opens a browser → runs assigned test files → exits.
Interview One-liners
"A worker is a separate Node.js process — workers don't share memory, so there are no shared-state race conditions between them."
"Playwright reuses a worker for multiple test files — it doesn't spawn a new browser for every single test file."
"workerIndex is a zero-based fixture that tells you which worker a test is running on — useful for allocating unique test data."
"test.describe.configure({ mode: 'parallel' }) overrides the global setting for a specific group without touching config."
"On CI with 2 CPU cores, setting workers: 4 doesn't give you 4x speed — you'll be context-switching and may slow down."
Worker Architecture
npx playwright test ↓ Test Runner (main process) • Reads playwright.config.ts • Builds test file list • Spawns N worker processes ↓ spawns ┌────────────────────────────────────────────┐ │ Worker 0 (PID 1234) │ │ → Launches Chromium browser │ │ → Runs: login.spec.ts │ │ → Runs: checkout.spec.ts │ │ → beforeAll() runs ONCE per worker │ │ → Receives: workerIndex = 0 │ └────────────────────────────────────────────┘ ┌────────────────────────────────────────────┐ │ Worker 1 (PID 5678) │ │ → Launches separate Chromium instance │ │ → Runs: search.spec.ts │ │ → Runs: profile.spec.ts │ │ → beforeAll() runs ONCE per worker │ │ → Receives: workerIndex = 1 │ └────────────────────────────────────────────┘ ↓ Results merged → HTML Report

  // testInfo.workerIndex: 0-based index of the WORKER PROCESS
  // Never changes for a worker across its lifetime

    // Cleanup: runs when the worker shuts down
    console.log('Worker cleanup: revoking token');
  }, { scope: 'worker' }],
});

Worker-Scoped beforeAll — Runs Once Per Worker
// KEY INSIGHT: With parallel workers, beforeAll() runs
// ONCE PER WORKER, not once globally!

// If workers: 4, beforeAll runs 4 times total
// (once in each worker process)

let dbConnection: any;

    // Cleanup: runs when worker is done with all tests in this file

    // dbConnection is available here (set by beforeAll in THIS worker)
  });
});
Best Practices
Use workerIndex to partition test data — assign different pre-created test accounts to each worker to avoid login conflicts
Use worker-scoped fixtures for expensive resources (DB connections, auth tokens) shared across tests in a worker

Remember: beforeAll is per-worker, not globally once — design your setup code with this in mind
Test the worker count setting in CI first before setting it in config — run with --workers=1 vs --workers=4 and compare times

Page Object Model (POM) — Architecture for Maintainability

Design a scalable test architecture where UI changes only require updating one file, not 50 tests.

Overview
What
POM is a design pattern where each web page is represented as a TypeScript class with locators and action methods.
Why
Without POM: change a button's selector → update 50 tests. With POM: change it once in the page class → all tests fixed.
When
Use POM from day 1 when your suite exceeds 5–10 test files. Essential for teams and long-lived projects.
Where
Create a pages/ directory. One file per page: pages/LoginPage.ts, pages/DashboardPage.ts, etc.
How
Class with constructor(page), locators as private properties, business-logic methods as public async functions.
Interview One-liners
"POM encapsulates selectors and actions in a class — tests call methods like loginPage.login(), not raw page.fill() calls."
"With POM, my tests read like user stories — loginPage.login('admin') then dashboardPage.verifyWelcome() — not implementation details."
"I keep assertions OUT of page objects — they belong in tests. Page objects provide actions; tests provide assertions."
"Component objects are a variation — reusable classes for repeated UI components like navbars, modals, and tables."
"POM combined with custom fixtures lets me inject page objects directly into tests like test({ loginPage }, () => {})"
POM Architecture
Without POM With POM tests/login.spec.ts tests/login.spec.ts → page.fill('#email', ...) → → loginPage.login(email, password) → page.fill('#password', ...) → page.click('.btn-login') pages/LoginPage.ts → locators: emailInput, passwordInput tests/checkout.spec.ts loginBtn → page.fill('#email', ...) → methods: login(email, pass) → page.fill('#password', ...) expectError(msg) → page.click('.btn-login') pages/DashboardPage.ts Change: .btn-login → #login-btn → locators: greeting, statsWidgets ❌ Update EVERY test file → methods: verifyLoaded() navigateTo(section) Change selector? ✅ Update LoginPage.ts ONLY

pages/LoginPage.ts — The Page Object Class
import { Page, Locator } from '@playwright/test';

// Each page = one TypeScript class
// The class owns ALL locators and actions for that page
export class LoginPage {

  // readonly: locators don't change after construction
  // private: tests should use methods, not access locators directly
  private readonly emailInput: Locator;
  private readonly passwordInput: Locator;
  private readonly loginButton: Locator;
  private readonly errorMessage: Locator;

  // Constructor receives the page object from Playwright
  constructor(private readonly page: Page) {
    // Define locators ONCE here — used by ALL methods below
    this.emailInput = page.getByLabel('Email');
    this.passwordInput = page.getByLabel('Password');
    this.loginButton = page.getByRole('button', { name: 'Sign in' });
    this.errorMessage = page.getByTestId('login-error');
  }

  // navigate(): go to the login page URL
  async navigate() {
    await this.page.goto('/login');
  }

  // login(): high-level action — fill form and click submit
  // Tests call this instead of raw fill() + click() calls
  async login(email: string, password: string) {
    await this.emailInput.fill(email);
    await this.passwordInput.fill(password);
    await this.loginButton.click();
  }

  // getErrorMessage(): returns error text for test assertion
  // Page object RETURNS data; tests ASSERT on it
  async getErrorMessage(): Promise<string | null> {
    return this.errorMessage.textContent();
  }
}

pages/BasePage.ts — Shared Functionality
import { Page, Locator, expect } from '@playwright/test';

// BasePage: the parent class ALL page objects extend
// Contains methods needed by EVERY page in the app
export abstract class BasePage {

  // abstract: each subclass MUST define its own URL
  protected abstract readonly url: string;

  // Constructor stores page — subclasses call super(page)
  constructor(protected readonly page: Page) {}

  // navigate(): every page can go to its own URL
  async navigate(): Promise<this> {
    await this.page.goto(this.url);
    return this; // returns 'this' for fluent chaining
  }

  // getTitle(): every page can check its own document title
  async getTitle(): Promise<string> {
    return this.page.title();
  }

  // waitForNetworkIdle(): wait until all requests settle
  async waitForNetworkIdle(): Promise<this> {
    await this.page.waitForLoadState('networkidle');
    return this;
  }

  // takeScreenshot(): attach screenshot with timestamp
  async takeScreenshot(name: string) {
    await this.page.screenshot({
      path: `screenshots/${name}_${Date.now()}.png`
    });
  }
}

// Data Factory: generates unique, typed test data
// Separates DATA concerns from TEST logic concerns

// Define a TypeScript interface for the shape of a User
export interface User {
  name: string;
  email: string;
  password: string;
  role: 'admin' | 'user' | 'viewer';
}

export class UserFactory {

  // create(): generate a complete unique User object
  // Partial<User> allows overriding specific fields
  static create(overrides: Partial<User> = {}): User {
    const timestamp = Date.now();
    return {
      // Unique name and email for every test — no conflicts!
      name: `Test User ${timestamp}`,
      email: `user_${timestamp}@test.com`,
      password: 'TestPass123!',
      role: 'user',
      // Spread overrides LAST so they take precedence
      ...overrides,
    };
  }

  // createAdmin(): shorthand for creating an admin user
  static createAdmin(): User {
    return this.create({ role: 'admin' });
  }
}

API Testing — HTTP Without a Browser
import { test, expect, request } from '@playwright/test';

test('API: create user via HTTP POST', async ({ request }) => {
  // request fixture: Playwright's built-in HTTP client
  // No browser launched — much faster for pure API tests

  // POST request with JSON body
  const response = await request.post('/api/users', {
    data: {
      name: 'Test User',
      email: `user_${Date.now()}@test.com`,
    }
  });

  // Assert HTTP status code
  expect(response.status()).toBe(201);

  // Parse response body as JSON
  const user = await response.json();
  expect(user).toHaveProperty('id');
  expect(user.name).toBe('Test User');
});

test('API setup + UI test', async ({ page, request }) => {
  // Create test data via API (FAST) then verify UI
  const res = await request.post('/api/products', {
    data: { name: 'Test Widget', price: 9.99 }
  });
  const { id } = await res.json();

  // Now navigate to the UI and verify the product appears
  await page.goto(`/products/${id}`);
  await expect(page.getByRole('heading')).toHaveText('Test Widget');
});
page.route() — Network Mocking
test('show error banner when API returns 500', async ({ page }) => {

  // page.route(): intercept matching URL and handle it
  // No real API call is made — Playwright intercepts it
  await page.route('**/api/user/profile', async (route) => {
    // fulfill(): respond with our custom mock response
    await route.fulfill({
      status: 500,
      contentType: 'application/json',
      body: JSON.stringify({ error: 'Internal Server Error' }),
    });
  });

  await page.goto('/profile');

  // Assert that the UI shows an error state
  await expect(page.getByTestId('error-banner')).toBeVisible();
  await expect(page.getByTestId('error-banner'))
    .toContainText('Something went wrong');
});

test('block images to speed up test', async ({ page }) => {
  // Abort image requests — makes navigation 2x faster
  await page.route('**/*.{png,jpg,jpeg,svg}', route => route.abort());
  await page.goto('/');
  // Test text content — images not needed for this test
});

Best Practices
Use storageState for login — tests are 5–10x faster when they skip the login UI for every test
Create test data via API in beforeEach, not by clicking through the UI — UI setup is slow and creates flaky dependencies
Use page.route() to test error states — never rely on the real backend to be in a broken state for your test
Enable trace: 'on-first-retry' — the trace viewer is the most powerful debugging tool in Playwright's arsenal
Track test metrics: flake rate, avg execution time, coverage % — present these to your team to show QA value
