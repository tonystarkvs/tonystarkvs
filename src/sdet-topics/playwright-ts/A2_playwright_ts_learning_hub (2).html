Session 1 — What is Playwright & Why TypeScript?

Phase 1 · Fundamentals ~60 min Setup + Core Concepts
Overview — What / Why / When / Where / How
Playwright
What: An open-source browser automation library by Microsoft that lets you control real browsers (Chromium, Firefox, WebKit) programmatically.
Why: Faster, more reliable, auto-waits for elements, built-in parallelism — no more flaky Selenium tests.
When: Use it for end-to-end (E2E) testing of web applications, UI regression, and API testing.
Where: Runs on Windows, Mac, Linux — locally or in CI/CD pipelines (GitHub Actions, Jenkins, etc.).
How: You write test scripts in TypeScript/JavaScript, Playwright drives the browser, asserts UI behavior.
TypeScript
What: A superset of JavaScript with static types — meaning you declare what data type each variable holds.
Why: Catches bugs at compile time (before running), gives you autocomplete in VS Code, makes large test codebases maintainable.
When: Always — the Playwright team recommends TypeScript for SDET professionals.
How: TypeScript code gets compiled (.ts → .js) before execution. The compiler is called tsc.
Interview One-Liners (say exactly these in interviews)
"Playwright is a Microsoft-owned E2E testing framework that supports Chromium, Firefox, and WebKit with a single API."
"Unlike Selenium, Playwright has built-in auto-waiting, which eliminates most flaky test failures caused by timing issues."
"I use TypeScript with Playwright because static typing catches type mismatches at compile time, not at runtime."
"Playwright can test across all major browser engines in parallel, which significantly reduces overall test suite execution time."
"Playwright's Page Object Model integration with TypeScript classes gives our test suite strong maintainability and readability."

example-1.ts — Your first Playwright test
// 'import' — brings external code into this file
// 'test' — Playwright's function to define a test case
// 'expect' — Playwright's assertion utility (checks if something is true)
import { test, expect } from '@playwright/test';

// test() takes two arguments:
// 1. A string — the test name (shows in reports)
// 2. An async arrow function — the test body
// { page } — destructured from Playwright's fixture (auto-provided browser page)
test('homepage title is correct', async ({ page }) => {

  // page.goto() — navigates the browser to a URL
  // 'await' — waits for the async operation to finish before proceeding
  await page.goto('https://playwright.dev');

  // page.title() — reads the document title from the browser tab
  // expect(...).toContain(...) — asserts the title includes this text
  await expect(page).toHaveTitle(/Playwright/);
});

example-3.ts — async / await explained
// 'async' — marks a function as asynchronous (it can do time-taking work)
// 'await' — pauses execution until the async operation completes
// Without await, your code runs without waiting → test fails randomly

import { test, expect } from '@playwright/test';

test('login flow with async/await', async ({ page }) => {

  // Step 1: Navigate — browser must finish loading before next line runs
  await page.goto('https://example.com/login');

  // Step 2: Find input by placeholder text, then type into it
  // getByPlaceholder() — Playwright locator (finds elements on page)
  await page.getByPlaceholder('Email').fill('user@test.com');

  // Step 3: Fill password field
  await page.getByPlaceholder('Password').fill('secret123');

  // Step 4: Click the login button (found by its visible text)
  await page.getByRole('button', { name: 'Log in' }).click();

  // Step 5: Assert dashboard heading is visible after login
  await expect(page.getByRole('heading', { name: 'Dashboard' })).toBeVisible();
});

// playwright.config.ts — the central configuration file for your test suite

import { defineConfig, devices } from '@playwright/test';

// 'export default' — makes this config available to the Playwright runner
export default defineConfig({

  timeout: 30000,

  // use: default settings applied to every test
  use: {

    screenshot: 'only-on-failure',   // capture screenshots when test fails
    video: 'retain-on-failure',       // record video only for failures
    headless: true,                   // run without opening browser window
  },

  // projects: run tests across multiple browsers
  projects: [
    { name: 'chromium', use: { ...devices['Desktop Chrome'] } },
    { name: 'firefox',  use: { ...devices['Desktop Firefox'] } },
  ],
});

  • Use TypeScript strict mode ("strict": true in tsconfig.json)

Interview Q&A — SDET Style
Q1: What is the difference between Playwright and Selenium?
Playwright uses CDP (Chrome DevTools Protocol) and WebSocket to control browsers directly, giving it built-in auto-waiting, network interception, and parallel execution out of the box. Selenium uses WebDriver protocol and requires explicit waits. Playwright is faster, more reliable, and natively supports modern browser features like service workers and shadow DOM.
Q2: Why would you choose TypeScript over JavaScript for automation?
TypeScript catches type errors at compile time, not at runtime. In a large test suite, this prevents bugs like passing a number where a string is expected. It also gives IDE autocomplete and refactoring support, making the codebase more maintainable as the team scales.
Q3: What is async/await and why is it critical in Playwright?
async/await is JavaScript's way of handling asynchronous operations (operations that take time, like network requests or DOM interactions). Every Playwright action (click, fill, goto) is async. Without await, the next line runs before the action completes, causing unpredictable test failures.
Q4: What is playwright.config.ts and what goes in it?
It is the central configuration file for your entire test suite. It controls: test directory location, browser projects, timeouts, retries, base URL, screenshot/video capture settings, and environment-specific options. It is loaded automatically when you run npx playwright test.
Q5: What does npx playwright install do?
It downloads Playwright's own browser binaries (Chromium, Firefox, WebKit) into a local cache on your machine. Playwright does NOT use your system-installed Chrome — it uses its own controlled browser versions to ensure tests are reproducible across different machines and CI environments.

Session 2 — Locators & Selectors

Phase 1 · Fundamentals ~60 min Finding Elements on Page
Overview
Locator
What: A Playwright object that represents a way to find one or more elements on a webpage. It is lazy — it only searches when you perform an action on it.
Why: Playwright locators auto-retry finding the element (up to the timeout) before failing. This prevents flakiness caused by slow-loading elements.
When: Every time you need to interact with or assert on a page element.
Key difference from Selenium: Playwright locators have built-in waiting; Selenium findElement throws immediately if not found.
Interview One-Liners
"Playwright locators are lazy and auto-retrying — they keep trying to find an element until the timeout expires."
"I prefer getByRole() because it mirrors how users and screen readers perceive the page — it tests behavior, not implementation."
"CSS selectors like #id or .class are fragile — a UI redesign breaks them. Role-based locators survive UI changes."
"getByTestId() is the most stable locator in CI/CD — it requires developers to add data-testid attributes, which never change for business reasons."
"I use locator chaining — like page.getByRole('list').getByRole('listitem') — to scope searches within a specific section of the page."
Locator Priority Hierarchy
Code Examples
example-1.ts — getByRole (most recommended)
import { test, expect } from '@playwright/test';

test('role-based locators', async ({ page }) => {
  await page.goto('https://example.com');

  // getByRole() — finds element by its ARIA role + visible name
  // role: 'button' matches <button> tags and elements with role="button"
  // { name: 'Submit' } — the visible text or aria-label of the element
  await page.getByRole('button', { name: 'Submit' }).click();

  // role: 'heading' — finds <h1>, <h2>, etc.
  // { level: 1 } — specifically finds <h1>
  await expect(page.getByRole('heading', { level: 1 }))
    .toHaveText('Welcome');

  // role: 'textbox' — finds <input type="text"> or <textarea>
  await page.getByRole('textbox', { name: 'Search' }).fill('Playwright');

  // role: 'checkbox' — finds <input type="checkbox">
  await page.getByRole('checkbox', { name: 'Accept terms' }).check();

  // role: 'link' — finds <a> tags
  await page.getByRole('link', { name: 'Privacy Policy' }).click();
});
example-2.ts — getByLabel & getByPlaceholder
test('label and placeholder locators', async ({ page }) => {
  await page.goto('https://example.com/register');

  // getByLabel() — finds input by its associated <label> text
  // HTML: <label for="email">Email Address</label> <input id="email">
  await page.getByLabel('Email Address').fill('test@email.com');

  // getByPlaceholder() — finds input by its placeholder attribute
  // HTML: <input placeholder="Enter your name">
  await page.getByPlaceholder('Enter your name').fill('John Doe');

  // getByText() — finds ANY element containing this visible text
  // Useful for finding paragraphs, spans, divs with specific text
  await expect(page.getByText('Already have an account?')).toBeVisible();
});
example-3.ts — getByTestId (most stable in CI)
// getByTestId() requires developers to add data-testid attributes in HTML
// HTML: <button data-testid="submit-btn">Submit</button>
// These attributes survive redesigns — the most stable selector in large teams

test('test ID locators', async ({ page }) => {
  await page.goto('https://example.com');

  // Finds: <button data-testid="login-button">
  await page.getByTestId('login-button').click();

  // You can configure the attribute name in playwright.config.ts:
  // use: { testIdAttribute: 'data-qa' } — then use data-qa="..." in HTML
  await page.getByTestId('error-message').waitFor();

  // Chaining: narrow search scope by parent element first
  // Only looks for 'nav-link' INSIDE the element with data-testid="sidebar"
  await page.getByTestId('sidebar')
    .getByTestId('nav-link')
    .first()  // .first() — gets the first matching element
    .click();
});
example-4.ts — CSS & XPath (use rarely)
test('CSS and XPath selectors', async ({ page }) => {
  await page.goto('https://example.com');

  // page.locator() — accepts CSS selectors as a string
  // '#login' = element with id="login"
  await page.locator('#login').click();

  // '.btn-primary' = element with class="btn-primary"
  await page.locator('.btn-primary').click();

  // CSS attribute selector: [attribute="value"]
  await page.locator('input[type="email"]').fill('test@test.com');

  // XPath — use only when CSS cannot reach the element
  // xpath= prefix tells Playwright to use XPath mode
  // //button[contains(text(),'Log')] = button whose text contains 'Log'
  await page.locator("xpath=//button[contains(text(),'Log')]").click();
});
example-5.ts — Locator filtering & chaining
test('advanced locator techniques', async ({ page }) => {
  await page.goto('https://example.com/products');

  // .filter() — narrows locator results using additional conditions
  // Finds all list items, but only those that contain the text 'In Stock'
  const inStockItems = page.getByRole('listitem')
    .filter({ hasText: 'In Stock' });

  // .count() — returns how many elements match the locator
  const count: number = await inStockItems.count();
  console.log(`Found ${count} in-stock items`);

  // .nth(index) — picks the Nth element (0-based index)
  // .nth(0) = first element, .nth(1) = second element
  await inStockItems.nth(0).getByRole('button', { name: 'Add to cart' }).click();

  // .all() — returns an array of all matching locators
  // Use for loops — process each matching element
  const allItems = await page.getByRole('listitem').all();
  for (const item of allItems) {
    const text = await item.textContent();
    console.log(text);  // textContent() returns the text inside the element
  }
});
Best Practices
  • Prefer getByRole() → getByLabel() → getByTestId() in that order
  • Never use XPath unless absolutely necessary — it breaks on minor DOM changes
  • Never use index-based selectors like nth-child(3) — they break when UI order changes
  • Ask developers to add data-testid attributes to dynamic elements
  • Use locator chaining to scope searches and avoid ambiguous matches
Interview Q&A
Q1: What is the difference between a Locator and ElementHandle?
A Locator is a lazy, auto-retrying descriptor — it finds the element fresh each time you interact with it. An ElementHandle is a direct JS reference to a DOM element captured at a moment in time — if the element is re-rendered by React/Vue, the handle becomes stale. Always use Locators in Playwright.
Q2: Why is getByRole() preferred over CSS selectors?
getByRole() is resilient to UI redesigns. If a developer changes a button's class or ID, role-based locators still work. It also aligns tests with accessibility — if the role is wrong, both users with screen readers AND your tests fail, giving you double coverage.
Q3: How do you handle a situation where multiple elements match your locator?
Use chaining to scope: page.getByRole('list').getByRole('listitem'). Or use .filter() to narrow by text content. Or use .first(), .last(), or .nth(index). If the page genuinely has duplicate elements that you can't distinguish, ask the developer to add data-testid attributes.
Q4: What is auto-waiting in Playwright?
Before performing any action (click, fill, etc.), Playwright automatically waits for the element to be: attached to DOM, visible, stable (not moving), enabled (not disabled), and not obscured. This replaces manual waitForElement() calls from Selenium and eliminates most timing-related test failures.
Q5: What is the purpose of .filter() on a locator?
.filter() narrows a collection of matching elements by additional criteria such as hasText (element contains specific text) or has (element contains a child matching another locator). It is used when you want a subset of a list of similar elements — for example, all product cards that show "In Stock".

Session 3 — Assertions & expect()

Phase 1 · Fundamentals ~60 minValidating UI State
Overview
Assertion
What: A statement that checks whether the actual outcome matches the expected outcome. If it fails, the test fails with a clear message.
Why: Without assertions, a test only verifies the code runs without crashing — not that the UI shows the correct result.
Playwright's expect(): Auto-retrying — it keeps checking until the condition passes or timeout expires.
Interview One-Liners
"Playwright's web-first assertions auto-retry, so I don't need explicit waits before asserting on dynamic content."
"toBeVisible() checks that the element is both present in the DOM and visible to the user — not just attached to the DOM."
"I use soft assertions when I want a test to collect multiple failures in one run rather than stopping at the first."
"toHaveURL() is a page-level assertion — it asserts the browser's current URL after a navigation."
"expect.poll() lets me assert on a custom async condition that Playwright doesn't have a built-in matcher for."
Code Examples
example-1.ts — Core locator assertions
import { test, expect } from '@playwright/test';

test('core assertion methods', async ({ page }) => {
  await page.goto('https://example.com');

  // toBeVisible() — passes if element is in DOM AND visible on screen
  await expect(page.getByRole('heading')).toBeVisible();

  // toHaveText() — passes if element's text content matches exactly
  await expect(page.getByRole('heading')).toHaveText('Example Domain');

  // toContainText() — passes if element contains this text (partial match)
  await expect(page.getByRole('paragraph')).toContainText('illustrative');

  // toBeEnabled() — passes if form element is not disabled
  await expect(page.getByRole('button')).toBeEnabled();

  // toBeHidden() — passes if element is NOT visible (opposite of toBeVisible)
  await expect(page.getByTestId('error-msg')).toBeHidden();
});
example-2.ts — Page-level assertions
test('page-level assertions', async ({ page }) => {
  await page.goto('https://example.com');

  // toHaveURL() — asserts the current browser URL
  // Accepts string, regex, or URLSearchParams
  await expect(page).toHaveURL('https://example.com');
  await expect(page).toHaveURL(/example\.com/);

  // toHaveTitle() — asserts the browser tab title (<title> tag)
  await expect(page).toHaveTitle(/Example/);

  // Navigate and verify redirect — URL changes after action
  await page.getByRole('link', { name: 'More information' }).click();
  await expect(page).toHaveURL(/iana\.org/);
});
example-3.ts — Input & attribute assertions
test('input and attribute assertions', async ({ page }) => {
  await page.goto('https://example.com/form');

  const emailInput = page.getByLabel('Email');
  await emailInput.fill('test@test.com');

  // toHaveValue() — checks what is typed inside an input field
  await expect(emailInput).toHaveValue('test@test.com');

  // toHaveAttribute() — checks an element's HTML attribute value
  // Useful for checking href, src, disabled, type, etc.
  await expect(emailInput).toHaveAttribute('type', 'email');

  // toHaveClass() — checks element has this CSS class
  await expect(page.getByRole('button', { name: 'Submit' }))
    .toHaveClass(/btn-primary/);

  // toBeChecked() — for checkboxes and radio buttons
  await page.getByRole('checkbox', { name: 'Remember me' }).check();
  await expect(page.getByRole('checkbox', { name: 'Remember me' }))
    .toBeChecked();
});
example-4.ts — Soft assertions
test('soft assertions — collect all failures', async ({ page }) => {
  await page.goto('https://example.com');

  // expect.soft() — records failure but CONTINUES the test
  // Normal expect() stops the test immediately on failure
  // Use soft assertions to check multiple fields in one test run
  await expect.soft(page.getByRole('heading')).toHaveText('Example Domain');
  await expect.soft(page).toHaveTitle('Example Domain');
  await expect.soft(page.getByRole('link')).toBeVisible();

  // At the END of the test, Playwright reports ALL soft failures together
  // This is more efficient than fixing one failure, rerunning, finding next
});
example-5.ts — Custom timeout on assertions
test('assertion timeout override', async ({ page }) => {
  await page.goto('https://example.com');

  // By default, assertions retry for 5 seconds (configured in playwright.config.ts)
  // { timeout: N } — override just for this assertion (milliseconds)
  // Use when you know this specific element takes longer to appear
  await expect(page.getByRole('heading'))
    .toBeVisible({ timeout: 10000 });

  // expect.poll() — assert on a custom condition that auto-retries
  // Use when Playwright has no built-in matcher for what you need
  await expect.poll(async () => {
    const count = await page.getByRole('listitem').count();
    return count;
  }, { timeout: 10000 }).toBeGreaterThan(3);
});
Best Practices
  • Always await expect() calls — missing await makes assertions unreliable
  • Use web-first assertions (toBeVisible, toHaveText) — they auto-retry unlike manual waits
  • Use soft assertions for form validation testing — check all fields in one test run
  • Set global expect: { timeout: 5000 } in config — override per-assertion only when needed
  • Use regex in assertions for dynamic text: toHaveText(/Welcome, .+/)
Interview Q&A
Q1: What is the difference between toBeVisible() and toBeAttached()?
toBeAttached() only checks that the element exists in the DOM. toBeVisible() checks that the element is in the DOM AND visible to the user (not hidden via CSS display:none, visibility:hidden, or opacity:0). Always use toBeVisible() when verifying UI behavior.
Q2: When would you use soft assertions?
When testing a form or a page with multiple fields that all need verification. Instead of fixing one assertion failure, rerunning, finding the next failure — soft assertions let you see ALL failures in one test run. Useful in smoke tests and regression suites.
Q3: How does Playwright's auto-retrying assertion work?
When you call expect(locator).toBeVisible(), Playwright repeatedly evaluates the condition every 100ms until it passes OR the timeout (default 5s) expires. This is why you don't need waitForElement() before assertions in Playwright — the assertion itself is the wait.
Q4: What is the difference between toHaveText() and toContainText()?
toHaveText() matches the ENTIRE text content of the element. toContainText() passes if the element's text includes your string as a substring. Use toContainText() for dynamic text like "Welcome, John" when you only want to verify "Welcome".
Q5: What is expect.poll() and when do you use it?
expect.poll() lets you assert on a custom async function that Playwright retries until the assertion passes or times out. Use it when you need to assert on something Playwright has no built-in matcher for — like a database count, an API response value, or a complex computed state.

Session 4 — Page Object Model (POM)

Phase 1 · Fundamentals ~75 minDesign Pattern
Overview
Page Object Model (POM)
What: A design pattern where each web page (or component) is represented as a TypeScript class. The class contains locators and methods for that page.
Why: When a locator changes, you update it in ONE place (the page class) instead of hunting across 50 test files.
How: Create a class per page with a constructor receiving page, define locators as properties, define actions as methods.
Interview One-Liners
"Page Object Model separates test logic from page interaction logic — tests read like user stories, not like CSS selectors."
"With POM, when a UI change breaks a locator, I fix it in one class file — not in every test that uses it."
"I define locators as readonly properties in the constructor so they're initialized once and reused across all methods."
"Page objects should expose actions (login, search, addToCart) — not raw locators — to tests."
"I follow the rule: assertions belong in tests, not in page objects — page objects only perform actions."
Code Examples
LoginPage.ts — Page Object class
// 'import' the Page type from Playwright for TypeScript type checking
import { Page, Locator } from '@playwright/test';

// 'class' — a blueprint for creating page objects
// TypeScript class groups related locators and actions for one page
export class LoginPage {

  // 'readonly' — these properties are set once and never changed
  // 'Locator' — Playwright type for element references
  readonly emailInput: Locator;
  readonly passwordInput: Locator;
  readonly loginButton: Locator;
  readonly errorMessage: Locator;

  // 'constructor' — runs when you create a new LoginPage instance
  // 'page: Page' — receives the Playwright page object as a parameter
  constructor(private page: Page) {
    // Initialize all locators here — defined once, reused everywhere
    this.emailInput    = page.getByLabel('Email');
    this.passwordInput  = page.getByLabel('Password');
    this.loginButton    = page.getByRole('button', { name: 'Log in' });
    this.errorMessage   = page.getByTestId('login-error');
  }

  // 'async' method — encapsulates the full login action
  // Takes username and password as parameters (TypeScript types: string)
  async login(email: string, password: string): Promise<void> {
    await this.page.goto('/login');       // navigate to login page
    await this.emailInput.fill(email);      // type email
    await this.passwordInput.fill(password); // type password
    await this.loginButton.click();          // click login button
  }
}
login.spec.ts — Test using the Page Object
import { test, expect } from '@playwright/test';
import { LoginPage } from '../pages/LoginPage';

test('successful login redirects to dashboard', async ({ page }) => {

  // Instantiate the page object — 'new' creates an instance of the class
  const loginPage = new LoginPage(page);

  // Call the page object method — test reads like a user story
  await loginPage.login('user@test.com', 'Password@123');

  // Assertion stays in the test file — not in the page object
  await expect(page).toHaveURL(/dashboard/);
});

test('invalid credentials shows error message', async ({ page }) => {
  const loginPage = new LoginPage(page);

  await loginPage.login('bad@email.com', 'wrongpassword');

  // errorMessage is a locator defined in the page object
  await expect(loginPage.errorMessage).toBeVisible();
  await expect(loginPage.errorMessage).toContainText('Invalid credentials');
});
BasePage.ts — Base class for shared behavior
import { Page } from '@playwright/test';

// 'BasePage' — parent class with methods shared by ALL page objects
// Avoids repeating common code (waitForLoader, scrollToTop) in every page
export class BasePage {

  // 'protected' — accessible in BasePage AND in child classes (not outside)
  constructor(protected page: Page) {}

  // Method available to ALL child page classes
  async waitForPageLoad(): Promise<void> {
    // waitForLoadState('networkidle') — waits until no network requests for 500ms
    await this.page.waitForLoadState('networkidle');
  }

  async getPageTitle(): Promise<string> {
    return await this.page.title();
  }
}

// 'extends' — LoginPage inherits all methods from BasePage
import { Locator } from '@playwright/test';
export class LoginPageV2 extends BasePage {
  readonly emailInput: Locator;

  constructor(page: Page) {
    super(page);  // 'super()' calls the BasePage constructor — required when extending
    this.emailInput = page.getByLabel('Email');
  }

  async login(email: string, password: string) {
    await this.page.goto('/login');
    await this.emailInput.fill(email);
    await this.page.getByLabel('Password').fill(password);
    await this.page.getByRole('button', { name: 'Log in' }).click();
    await this.waitForPageLoad();  // inherited from BasePage
  }
}
example-4.ts — Returning page objects from methods
import { Page, Locator } from '@playwright/test';
import { DashboardPage } from './DashboardPage';

export class LoginPageV3 {
  constructor(private page: Page) {}

  // Return type is 'Promise<DashboardPage>' — after login we land on dashboard
  // This technique chains page objects — login returns the next page object
  async loginAndGoToDashboard(email: string, pwd: string): Promise<DashboardPage> {
    await this.page.goto('/login');
    await this.page.getByLabel('Email').fill(email);
    await this.page.getByLabel('Password').fill(pwd);
    await this.page.getByRole('button', { name: 'Log in' }).click();
    return new DashboardPage(this.page);  // return next page object
  }
}

example-5.ts — Component objects (reusable UI parts)
import { Locator } from '@playwright/test';

// Component Object — like a Page Object but for a reusable UI component
// Example: a navigation bar that appears on EVERY page
export class NavBar {

  // 'container' — the root locator scoping all nav interactions
  readonly container: Locator;

  // Takes a Locator (not Page) — works within any page's nav element
  constructor(container: Locator) {
    this.container = container;
  }

  async clickLink(linkName: string) {
    // All locators scoped to container — won't match links outside the nav
    await this.container
      .getByRole('link', { name: linkName })
      .click();
  }
}

Best Practices
  • One class per page — do not put multiple pages in one file
  • Assertions belong in tests — page objects only perform actions
  • Use readonly for locators — prevents accidental reassignment
  • Extend a BasePage class for shared utility methods
  • Return the next page object from navigation methods for clean chaining
  • Create component objects for reusable UI parts (nav bar, modal, table)
Interview Q&A
Q1: What is the Page Object Model and why do you use it?
POM is a design pattern that creates a class for each web page. Locators and page interactions are defined in the class. Tests only call class methods. This means when a UI changes, you update one class file, not every test that interacts with that page.
Q2: Should assertions go inside page objects?
No. Page objects should only perform actions. Assertions belong in tests. If you put assertions in page objects, a page object becomes tightly coupled to specific test scenarios and cannot be reused in tests that expect different outcomes.
Q3: What is the difference between 'private' and 'protected' in a TypeScript class?
'private' members are only accessible within the class where they're defined. 'protected' members are accessible in the class AND in any class that extends (inherits from) it. Use 'protected' for the 'page' property in BasePage so child page objects can access it.
Q4: How do you share a page object across multiple tests?
Use Playwright fixtures. You define a custom fixture that creates the page object and inject it into tests. This is more scalable than creating 'new LoginPage(page)' in every test — the fixture handles instantiation.
Q5: What is the 'super()' call in a TypeScript class?
super() calls the constructor of the parent class. When a class extends another, TypeScript requires you to call super() in the child constructor before accessing 'this'. It passes arguments to the parent's constructor — in our case, the Playwright 'page' object is passed to BasePage.

Session 5 — Fixtures, Hooks & Test Organization

Phase 1 · Fundamentals~75 minTest Setup & Teardown
Overview
Fixtures
What: Playwright's dependency injection system — fixtures provide pre-built objects (page, browser, context) to tests automatically.
Why: Eliminates boilerplate setup in every test. You can create custom fixtures that set up a logged-in state, a database seed, or a page object.
Built-in fixtures: page, browser, context, request, browserName.
Hooks
What: Functions that run automatically before/after tests — beforeAll, afterAll, beforeEach, afterEach.
Why: Set up shared state (login session, test data) once for a group of tests rather than repeating it in each test.
Interview One-Liners
"beforeEach runs before every test in its describe block — I use it to navigate to the page being tested."
"Custom fixtures let me inject a pre-authenticated page into tests — avoiding login steps in every test."
"beforeAll runs once before all tests in a describe block — ideal for expensive operations like database seeding."
"test.describe() groups related tests — failures in one describe block don't affect another."
"I use test.use() at the describe level to set browser context options (viewport, locale) for a specific test group."
Code Examples
example-1.ts — beforeEach and afterEach hooks
import { test, expect } from '@playwright/test';

// test.describe() — groups related tests into a named suite
// Like a folder for tests — organizes output in reports
test.describe('Product Search', () => {

  // test.beforeEach() — runs BEFORE each test in this describe block
  // Receives the same { page } fixture as tests
  test.beforeEach(async ({ page }) => {
    await page.goto('/products');  // navigate once — shared by all tests below
  });

  test('search by product name', async ({ page }) => {
    // page is already at /products — beforeEach navigated for us
    await page.getByPlaceholder('Search products').fill('laptop');
    await page.keyboard.press('Enter');
    await expect(page.getByRole('listitem')).toHaveCount(5);
  });

  test('empty search shows all products', async ({ page }) => {
    await expect(page.getByRole('listitem')).toHaveCount(20);
  });

  // test.afterEach() — runs AFTER each test — good for cleanup
  test.afterEach(async ({ page }) => {
    // Clear local storage after each test to avoid state leakage
    await page.evaluate(() => localStorage.clear());
  });
});
example-2.ts — Custom fixture (logged-in page)
import { test as base, expect } from '@playwright/test';
import { LoginPage } from './pages/LoginPage';

// Extend the base 'test' object with a custom 'loggedInPage' fixture
// 'ExtendedFixtures' interface defines what our custom fixture provides
type ExtendedFixtures = { loggedInPage: import('@playwright/test').Page };

// test.extend() — adds our fixture to Playwright's fixture system
export const test = base.extend<ExtendedFixtures>({

  // 'loggedInPage' fixture — auto-logs in before giving page to test
  loggedInPage: async ({ page }, use) => {
    // SETUP: perform login before test
    const loginPage = new LoginPage(page);
    await loginPage.login('user@test.com', 'Password@123');

    // 'use' — hand the prepared page to the test
    await use(page);

    // TEARDOWN (after 'use'): runs after the test finishes
    await page.goto('/logout');
  },
});

example-3.ts — beforeAll for expensive setup
test.describe('Admin Panel Tests', () => {

  // test.beforeAll() — runs ONCE before all tests in this describe block
  // Use for: database seeding, API setup, creating test accounts
  // Note: does NOT receive 'page' fixture — runs outside browser context
  test.beforeAll(async ({ request }) => {
    // 'request' fixture — Playwright's API testing client (no browser needed)
    // Seed test data via API before any test opens a browser
    await request.post('/api/seed-test-data', {
      data: { type: 'admin-tests' }
    });
  });

  // test.afterAll() — runs ONCE after all tests in this describe block
  test.afterAll(async ({ request }) => {
    // Clean up test data after all tests finish
    await request.delete('/api/seed-test-data');
  });

  test('admin can view all users', async ({ page }) => {
    await page.goto('/admin/users');
    await expect(page.getByRole('row')).toHaveCount(10);
  });
});

example-5.ts — test.skip, test.only, test.fail
// test.skip() — marks a test to be skipped (not run)
// Use when: known bug exists, feature not yet built
test.skip('payment flow', async ({ page }) => {
  // This test is skipped — shown as 'skipped' in report
});

// test.only() — runs ONLY this test (all others are skipped)
// NEVER commit test.only() to git — it skips your entire test suite!
test.only('isolated debug test', async ({ page }) => {
  // Only this test runs during local debugging
});

// test.fail() — marks a test as EXPECTED to fail
// If it passes, that's the failure! Used for known bugs with open tickets
test.fail('known bug - BUG-1234', async ({ page }) => {
  await page.goto('/checkout');
  // This is expected to fail until the bug is fixed
});

// Conditional skip — skip based on browser or environment
test('webkit specific test', async ({ page, browserName }) => {
  // test.skip() inside a test — condition evaluated at runtime
  test.skip(browserName === 'firefox', 'This test only runs in Chrome and WebKit');
  await page.goto('/');
});
Best Practices
  • Use beforeEach for navigation — keeps tests independent
  • Use beforeAll only for expensive external setup (API seeding) — not browser actions
  • Never commit test.only() — it silently disables your entire test suite in CI
  • Use custom fixtures for authentication — saves login time across the suite
  • Group tests with test.describe() — makes reports readable and failures scoped
Interview Q&A
Q1: Difference between beforeEach and beforeAll?
beforeEach runs before EVERY test — good for navigation and resetting state. beforeAll runs ONCE before all tests in the block — good for expensive setup like database seeding or creating a user via API that all tests share.
Q2: What is a Playwright fixture?
A fixture is a dependency that Playwright automatically injects into your test function. Built-in fixtures include page, browser, context, and request. You can create custom fixtures using test.extend() to provide pre-authenticated pages, database connections, or page object instances.
Q3: Why is test.only() dangerous in CI/CD?
test.only() causes Playwright to skip all other tests and run only the marked one. If accidentally committed to the main branch, your entire CI pipeline runs only that one test while reporting success — giving false confidence. Always use it locally only and remove before committing.
Q4: How do you run tests in parallel and control worker count?
Playwright runs test files in parallel by default. Control worker count in playwright.config.ts with workers: 4. Tests within a single file run sequentially by default. Add test.describe.parallel() to run tests within a describe block in parallel. In CI, set workers based on available CPU cores.
Q5: What is test.use() and where is it valid?
test.use() overrides fixture options for the current scope — either at the file level (affects all tests in the file) or inside a describe block (affects only tests in that group). Common uses: viewport size, locale, timezone, storageState (for pre-authenticated sessions), and baseURL overrides.
