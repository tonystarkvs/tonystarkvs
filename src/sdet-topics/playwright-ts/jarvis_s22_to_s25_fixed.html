Session 22 · File Upload

locator.setInputFiles() sets files directly on <input type="file"> bypassing the OS file picker dialog entirely. For custom upload buttons that trigger a hidden input, use page.waitForEvent('filechooser') inside a Promise.all(). Both patterns avoid the OS dialog which cannot be automated by any browser tool.

WHAT

Automating file selection in browser UIs. Playwright sets file data directly at DOM level — no OS dialog involvement. Works on visible and hidden inputs.

WHY bypass OS dialog?

Native OS file pickers (Windows Explorer, macOS Finder) live outside the browser. No automation tool can interact with them. Playwright's APIs bypass this limitation entirely.

setInputFiles vs FileChooser

setInputFiles() — direct input element. waitForEvent('filechooser') — when a custom button triggers a hidden input via JavaScript. Both achieve the same result.

Verification

After upload: assert filename in preview, progress bar visible then hidden, success message shown, API responded correctly, uploaded file accessible in the list.

Keyword Glossary
setInputFiles()Playwright method on Locator. Sets files on <input type="file"> at DOM level. Accepts: path string, path array, or {name, mimeType, buffer} object. Triggers input/change events. Works on hidden inputs.
waitForEvent('filechooser')Waits for the FileChooser event that fires when browser would open OS file picker. Must be registered BEFORE the triggering click. Always used inside Promise.all().
FileChooserObject returned by waitForEvent('filechooser'). Methods: setFiles(paths), isMultiple() → boolean, acceptMimeTypes() → string[].
buffer uploadUpload a file created in memory: { name:'test.csv', mimeType:'text/csv', buffer:Buffer.from('content') }. No disk file needed. Perfect for edge cases — empty files, wrong types, malformed content.
multiple filessetInputFiles(['f1.pdf','f2.jpg']) uploads an array. Works on <input type="file" multiple>. setInputFiles([]) clears all selections.
input[type="file"]The HTML element that holds the file selection. Often hidden when custom upload buttons are used. setInputFiles() works regardless of display:none or visibility:hidden.
drag-and-drop uploadFor drop-zones: use page.evaluate() with DataTransfer API or dispatchEvent('drop'). Different from setInputFiles() which targets the file input directly.
Promise.all patternCritical pattern: Promise.all([page.waitForEvent('filechooser'), page.click('#btn')]). Both start simultaneously — prevents the race condition of missing the filechooser event.
Interview One-Liners
OS Dialog bypass"Native OS file pickers are operating system windows — no browser automation tool can interact with them. Playwright's setInputFiles() circumvents this entirely by setting the file at the DOM level, triggering the same input and change events as if the user had selected the file through the dialog."
Promise.all rule"The critical rule for waitForEvent('filechooser') is that the listener must be registered BEFORE the click — I use Promise.all so both start simultaneously. If I await the click first, the chooser event fires before the listener is ready and the test hangs forever."
Buffer uploads"For testing edge cases like wrong file type, empty file, or malformed content, I use buffer-based uploads: setInputFiles({name:'virus.exe', mimeType:'application/x-msdownload', buffer:Buffer.from('fake')}) — no disk files needed, any content I want."
Hidden input"Many modern upload UIs hide the actual file input and style a custom button — setInputFiles() works on hidden inputs because it operates at the DOM level, not by simulating a click on the visible element."
Clear selection"setInputFiles([]) with an empty array removes all previously selected files — this tests the clear-selection behavior and verifies the app correctly resets to its empty upload state before letting the user re-submit."

Best Practices
✓
Store fixture files in tests/fixtures/ committed to git. Use path.join(__dirname, '../fixtures/test.pdf') for cross-platform paths.
✓
Always Promise.all([waitForEvent('filechooser'), click()]) — never await click first. The race condition will cause the test to hang.
✓
Use buffer-based uploads for edge case testing — wrong file type, empty file, oversized content — without creating multiple fixture files on disk.
✓
Use page.route('**/api/upload', route => route.fulfill({status:413})) to test server rejection of oversized files — mock the server response rather than creating gigabyte test files.
✓
Use page.waitForResponse('**/api/upload') before asserting upload success UI — the UI updates only after the server responds.
✓
Always assert AFTER upload — verify filename in preview AND success state. Don't assume setInputFiles() not throwing means the upload worked.
Interview Prep
Q1: Why can't Playwright automate the native OS file picker? ▶
The native OS file picker is a window managed by the operating system, not the browser. Browser automation tools only control the browser process — they cannot interact with OS-level windows like Windows Explorer or macOS Finder. Playwright solves this with two approaches: setInputFiles() sets file data directly on the DOM element at the JavaScript level (triggering the same input/change events), and the FileChooser API intercepts the browser's intent to open the OS picker and replaces it programmatically. Both approaches bypass the OS dialog entirely.
Q2: How do you upload a file to a drag-and-drop upload zone? ▶
First check if the drop zone has a hidden input — if so, use setInputFiles() directly. If it's a pure JS drag-drop zone: await page.evaluate(async () => { const dt = new DataTransfer(); const file = new File(['content'], 'test.txt', {type:'text/plain'}); dt.items.add(file); const zone = document.querySelector('.drop-zone'); zone.dispatchEvent(new DragEvent('dragover', {dataTransfer:dt, bubbles:true})); zone.dispatchEvent(new DragEvent('drop', {dataTransfer:dt, bubbles:true})); }). Always verify the drop zone's framework implementation first since different frameworks handle drop events differently.
Q3: How do you test file type validation? ▶
Client-side: await page.locator('input[type="file"]').setInputFiles({name:'virus.exe', mimeType:'application/x-msdownload', buffer:Buffer.from('fake content')}). Then assert: await expect(page.locator('.file-error')).toContainText('Only PDF files allowed'). Server-side: combine with waitForResponse — const resp = await page.waitForResponse('**/api/upload'); expect(resp.status()).toBe(415) for Unsupported Media Type. Buffer-based uploads let you create files with any name and MIME type without real disk files, making them ideal for all validation edge cases.
Q4: How do you test upload progress bar? ▶
Use page.route() to add artificial delay: await page.route('**/api/upload', async route => { await new Promise(r => setTimeout(r, 2000)); await route.fulfill({status:201, json:{fileId:'abc'}}); }). Then trigger upload and assert: await expect(page.locator('.upload-progress')).toBeVisible() — visible during 2s delay. await expect(page.locator('.upload-success')).toBeVisible() — appears after delay. await expect(page.locator('.upload-progress')).toBeHidden() — gone after success. This tests the loading state which is impossible to test reliably without controlling the API timing.
Q5: How do you upload multiple files and test max file limit? ▶
For multiple files: await page.locator('input[type="file"][multiple]').setInputFiles(['img1.jpg','img2.png','img3.webp']). Verify all appear: await expect(page.locator('.file-preview-item')).toHaveCount(3). Test clear: await fileInput.setInputFiles([]); await expect(page.locator('.empty-state')).toBeVisible(). Test limit exceeded: setInputFiles with 6 files when max is 5 — await expect(page.locator('.max-warning')).toContainText('Maximum 5 files'). await expect(page.locator('button[type="submit"]')).toBeDisabled(). FileChooser: const [fc] = await Promise.all([page.waitForEvent('filechooser'), page.click('#btn')]); fc.isMultiple() returns true for multiple inputs.

// basic-upload.ts — direct input element upload + full verification
import { test, expect } from '@playwright/test';
import path from 'path';

test('upload PDF to standard file input', async ({ page }) => {
  await page.goto('https://myapp.com/documents/upload');

  // Locate the file input — works even if hidden (display:none)
  const fileInput = page.locator('input[type="file"]');

  // Absolute path — avoids CWD issues
  const filePath = path.join(__dirname, '../fixtures/sample.pdf');
  await fileInput.setInputFiles(filePath);

  // Filename appears in preview immediately
  await expect(page.locator('.selected-file-name'))
    .toContainText('sample.pdf');

  // Wait for API response before asserting success UI
  const respPromise = page.waitForResponse(
    r => r.url().includes('/api/upload') && r.status() === 201
  );
  await page.click('button[type="submit"]');
  const resp = await respPromise;
  expect(resp.ok()).toBeTruthy();

  // Assert success state
  await expect(page.locator('.upload-success')).toBeVisible();
  await expect(page.locator('.files-list')).toContainText('sample.pdf');
});

Session 23 · File Download

File downloads are intercepted using page.waitForEvent('download') inside Promise.all() with the triggering action. The Download object provides: suggestedFilename(), saveAs(path), failure(), path(), and createReadStream(). Always verify the download didn't fail and validate the file content — not just that the download started.

WHAT

Playwright intercepts browser download events before the OS download manager handles them. The Download object gives full access to filename, content, and failure state.

WHY test downloads?

Verify: correct filename pattern, file isn't empty, content matches (CSV headers, JSON structure, PDF validity), download triggered on correct action, failure handled gracefully.

WHEN does event fire?

Any download trigger: <a download href>, Content-Disposition:attachment response, window.open() with blob URL, or programmatic Blob API generation.

HOW to use

Promise.all([waitForEvent('download'), click()]). Then: saveAs() for persistence, failure() to check success, read with fs.readFileSync() for content verification.

Keyword Glossary
waitForEvent('download')Listens for the download event which fires when browser starts receiving a file. Must be registered BEFORE the triggering action. Always inside Promise.all() to avoid race conditions.
Download objectReturned by waitForEvent('download'). Represents one file download. Provides all methods to inspect and save the file.
suggestedFilename()Returns the filename from Content-Disposition header or URL path segment. e.g. 'report-2024-01.csv'. Use regex assertions for dynamic dates.
saveAs(path)Async: copies downloaded file to specified path. Creates parent directories. Use testInfo.outputPath() for parallel-safe paths.
failure()Async: returns null if download succeeded, error string if failed. Always check — downloads can fail silently without throwing exceptions.
path()Returns temp path where Playwright saved the download. Deleted when context closes. Use saveAs() for persistent storage beyond the test.
createReadStream()Returns a Node.js Readable stream of file content. Use for streaming large file verification without loading entire file into memory.
Content-DispositionHTTP header that triggers downloads: Content-Disposition: attachment; filename="report.csv". Playwright detects this and fires the download event automatically.
Interview One-Liners
Promise.all mandatory"The download event fires the moment the browser starts receiving the file — if I await the click first then call waitForEvent, the event has already fired and I've missed it. Promise.all ensures listener and click start atomically, preventing this race condition."
failure() check"I always assert download.failure() === null — a download can silently fail due to a network interruption or auth error without throwing an exception. Without this check, my test proceeds happily even if the file was never received."
Content verification"Asserting the download happened is not enough — I read the saved file and verify: CSV headers are correct, JSON has expected structure, PDF starts with '%PDF-' magic bytes. A corrupted file that downloads successfully is still a bug."
temp path lifecycle"The download.path() temp file is only valid during the browser context's lifetime. When context.close() is called, temp files are deleted. I call saveAs() before teardown if I need to read content after the context closes."
filename regex"I always use regex for download.suggestedFilename() — the filename usually includes a dynamic date, so: expect(download.suggestedFilename()).toMatch(/^report-\\d{4}-\\d{2}-\\d{2}\\.csv$/) — this verifies the naming convention without hardcoding a specific date."
Core Concepts
⬡ Download Event Flow
1. Promise.all starts→listener registered + click fires simultaneously
2. Click triggers→browser requests download URL
3. Server responds→Content-Disposition: attachment; filename="file.csv"
4. 'download' event fires→Promise.all resolves with Download object
5. download.saveAs(path)→file written to test output folder
6. download.failure() === null→confirm success
7. fs.readFileSync(path)→verify content
⬡ All Download Object Methods
suggestedFilename()→string — from Content-Disposition or URL
saveAs(path)→Promise<void> — persist to disk
path()→Promise<string|null> — temp path (context-scoped)
createReadStream()→Promise<Readable> — stream content
failure()→Promise<null|string> — null=success, string=error
cancel()→Promise<void> — cancel in-progress download
url()→string — the download URL (blob: or https:)
page()→Page — page that initiated the download
Best Practices
✓
Always Promise.all([page.waitForEvent('download'), triggerAction()]) — never await the click first. Race condition will cause test to hang.
✓
Always call await download.failure() and assert it is null — downloads can fail silently (network error, auth error) without throwing.
✓
Use testInfo.outputPath('filename.csv') as save path — unique per test, prevents parallel tests from overwriting each other's downloaded files.
✓
Assert suggestedFilename() with a regex pattern — filename often contains a dynamic date or timestamp that varies between runs.
✓
Read and validate file content after saveAs() — check CSV headers, JSON structure, PDF magic bytes. A successful download of a corrupt file is still a bug.
✓
Test download cancellation — call download.cancel() and verify the app handles it gracefully: shows appropriate message, allows retry.
Interview Prep
Q1: How do you verify the content of a downloaded CSV file? ▶
After download and saveAs(): const content = fs.readFileSync(filePath, 'utf-8'); const lines = content.trim().split('\n'); expect(lines[0]).toBe('Name,Email,Date,Amount') — exact header check. expect(lines.length).toBeGreaterThan(1) — has data rows. expect(content).toMatch(/\d{4}-\d{2}-\d{2}/) — date format present. expect(content).not.toContain('\x00') — not binary garbage. expect(fs.statSync(filePath).size).toBeGreaterThan(0) — not empty file. This verifies the download contains correctly structured, valid data — not just that a file arrived.
Q2: How do you assert the downloaded filename when it contains a dynamic date? ▶
Use regex with download.suggestedFilename(): expect(download.suggestedFilename()).toMatch(/^report-\d{4}-\d{2}-\d{2}\.csv$/) — matches 'report-2024-01-15.csv' regardless of the specific date. Other patterns: extension only: .toMatch(/\.csv$/). Prefix only: .toStartWith('export-'). Includes today's date: const today = new Date().toISOString().split('T')[0]; expect(download.suggestedFilename()).toContain(today). Exact assertion only when the filename is truly static and never contains dates or IDs.
Q3: How do you test a failed download? ▶
Mock the download endpoint: await page.route('**/api/export', r => r.fulfill({status:403, json:{error:'Not authorized'}})); const [download] = await Promise.all([page.waitForEvent('download'), page.click('#export-btn')]); const failure = await download.failure(); expect(failure).not.toBeNull(). Also assert UI error: await expect(page.locator('.download-error')).toContainText('Not authorized'). Note: if the server returns a non-download response (no Content-Disposition header), the download event may not fire — instead assert the error toast appears after the click without a Promise.all.
Q4: How do you handle downloads triggered from a popup window? ▶
Listen on the popup Page object, not the parent page: const [popup] = await Promise.all([page.waitForEvent('popup'), page.click('#open-export-dialog')]); await popup.waitForLoadState(); const [download] = await Promise.all([popup.waitForEvent('download'), popup.click('#confirm-download')]); await download.saveAs(testInfo.outputPath('export.xlsx')). The download event is scoped to the page that initiated it — always listen on the page that contains the download trigger, not the parent page that opened the popup.
Q5: What is the difference between download.path() and download.saveAs()? ▶
download.path() returns the path to the temp file Playwright auto-saved during download. This temp file is deleted when the browser context closes (context.close()). Use path() only for quick reads within the same test. download.saveAs(path) copies the file to a location you control — it persists after the context closes. Use saveAs() when: you need to read content after teardown, in fixtures where context closes in teardown, or when you want to attach the file to the test report. Best practice: always use saveAs(testInfo.outputPath(filename)) so the file has a unique per-test path safe for parallel execution.
basic-download.ts
// basic-download.ts — trigger, capture, save, verify
import { test, expect } from '@playwright/test';

test('download CSV report and verify filename + failure', async ({ page }, testInfo) => {
  await page.goto('https://myapp.com/reports');

  // MANDATORY: Promise.all — listener before click, both start at once
  const [download] = await Promise.all([
    page.waitForEvent('download'),     // ← listener first
    page.click('#export-csv-btn'),       // ← trigger simultaneously
  ]);

  // Assert filename pattern (dynamic date in filename)
  expect(download.suggestedFilename())
    .toMatch(/^report-\d{4}-\d{2}-\d{2}\.csv$/);

  // Assert download URL
  expect(download.url()).toContain('/api/reports/export');

  // Save to test-specific output folder (parallel-safe)
  const savePath = testInfo.outputPath(download.suggestedFilename());
  await download.saveAs(savePath);

  // CRITICAL: check failure — null means success
  const err = await download.failure();
  expect(err, 'Download must succeed').toBeNull();

  // Attach to test report for manual review
  await testInfo.attach('report-csv', { path: savePath, contentType: 'text/csv' });

  // Assert post-download UI state
  await expect(page.locator('.download-success-toast'))
    .toContainText('Downloaded successfully');
});

Session 24 · Viewport & Device Emulation

Playwright can emulate any device — mobile, tablet, desktop — by setting viewport size, deviceScaleFactor, touch support, user agent, and orientation. The built-in devices dictionary contains 50+ pre-configured device profiles. Set in playwright.config.ts, per project, per file via test.use(), or per test inline. Combine with network throttling for realistic mobile testing.

WHAT is viewport emulation?

Setting the browser window size and device characteristics (pixel ratio, touch, user agent) to simulate a specific device or screen size without physical hardware.

WHY test with emulation?

Responsive layouts break at different breakpoints. Mobile-specific features (touch, hamburger menu, bottom nav) only appear on small viewports. Pixel ratio affects image sharpness. Test all breakpoints without real devices.

devices dictionary

Playwright ships with 50+ device profiles: iPhone 14, Pixel 7, iPad, Galaxy S22 etc. Spread into test.use(): ...devices['iPhone 14 Pro']. Each profile sets viewport, userAgent, deviceScaleFactor, isMobile, hasTouch together.

What does emulation change?

viewport (width×height), deviceScaleFactor (pixel density), isMobile (navigator.userAgentData.mobile), hasTouch (touch events enabled), userAgent (browser identifier string), orientation (portrait/landscape).

Keyword Glossary
viewportObject {width, height} in CSS pixels. Sets the visible browser area. Affects CSS media queries, responsive breakpoints, and element layouts. Default: 1280×720.
devicesPlaywright's built-in dictionary of 50+ device profiles. Import from '@playwright/test'. Spread into test.use() or project config: {...devices['iPhone 14 Pro']}.
isMobileBoolean in context options. When true, sets navigator.userAgentData.mobile=true, scales viewport to device size, and applies touch emulation. Required for mobile-specific JavaScript behavior.
hasTouchBoolean — enables touch events (touchstart, touchmove, touchend). When true, the browser fires touch events instead of mouse events. Required for testing swipe, pinch-zoom, and tap behaviors.
deviceScaleFactorCSS pixel ratio (DPR). iPhone 14 = 3, most Android = 2.75. Affects visual rendering resolution. At DPR=2, a 375px viewport renders at 750 physical pixels — reveals blurry images and small tap targets.
userAgentThe browser identifier string sent in HTTP headers and exposed via navigator.userAgent. Set to match the target device. Some apps serve different content based on user agent detection.
page.setViewportSize()Changes viewport during test execution. Use to test viewport resize behavior, responsive layout transitions, and orientation changes mid-test.
emulateMedia()Emulates CSS media features: page.emulateMedia({colorScheme:'dark'}) triggers prefers-color-scheme:dark. Also: reducedMotion, forcedColors, media:'print' for print preview.

Q1: What is the difference between viewport and isMobile? ▶
viewport sets the CSS pixel dimensions of the browser window — it affects CSS media queries and responsive breakpoints. isMobile:true changes JavaScript APIs: navigator.userAgentData.mobile becomes true, touch events are enabled, and mobile-specific browser behaviors activate. Many React/Vue/Angular apps check both the viewport width AND isMobile to decide which UI to render. Setting only the viewport might make the CSS layout show mobile styles but the JavaScript still renders desktop components because isMobile is false. Use ...devices['iPhone 14'] to set both correctly at once — it's the only way to reliably emulate mobile behavior.
Q2: How do you test responsive breakpoint behavior? ▶
Three approaches: (1) Separate test files with test.use({viewport:{width:375,height:812}}) at the top. (2) Separate describe blocks within one file, each with test.use() for different viewpoints. (3) Mid-test resize: test('responsive nav', async ({page}) => { await page.goto('/'); await page.setViewportSize({width:1280,height:720}); await expect(page.locator('.desktop-nav')).toBeVisible(); await expect(page.locator('.hamburger-menu')).toBeHidden(); await page.setViewportSize({width:375,height:812}); await expect(page.locator('.hamburger-menu')).toBeVisible(); await expect(page.locator('.desktop-nav')).toBeHidden(); }). Option 3 is the most efficient — one test covers both breakpoints and verifies the transition.
Q3: How do you test dark mode with Playwright? ▶
await page.emulateMedia({colorScheme:'dark'}) — this tells the browser to report prefers-color-scheme:dark in CSS media queries and the matchMedia JavaScript API. Your app's CSS with @media (prefers-color-scheme: dark) rules will activate. Then: await expect(page.locator('body')).toHaveCSS('background-color', 'rgb(18, 18, 18)'). For visual regression: await expect(page).toHaveScreenshot('homepage-dark.png', {fullPage:true}). For light: await page.emulateMedia({colorScheme:'light'}). Also available: reducedMotion:'reduce' (disables animations), forcedColors:'active' (high-contrast mode), media:'print' (print preview CSS).
Q4: How do you create a mobile project in playwright.config.ts? ▶
import {defineConfig, devices} from '@playwright/test'; export default defineConfig({ projects: [ {name:'desktop-chrome', use:{...devices['Desktop Chrome']}}, {name:'iphone-14', use:{...devices['iPhone 14 Pro']}}, {name:'pixel-7', use:{...devices['Pixel 7']}}, {name:'ipad', use:{...devices['iPad (gen 7)']}}, ] }). Run: npx playwright test --project=iphone-14. Or all: npx playwright test. The devices spread sets viewport, isMobile, hasTouch, deviceScaleFactor, and userAgent all at once. View all available devices: console.log(Object.keys(devices)) — shows 50+ entries from 'Blackberry PlayBook' to 'Nexus 10' to modern iPhones and Android phones.
Q5: How do you test touch gestures like swipe? ▶
With hasTouch:true enabled (via device profile or explicit setting), Playwright fires touch events. For swipe: await page.locator('.carousel').evaluate(el => { const touch = (x,y) => new Touch({identifier:1, target:el, clientX:x, clientY:y, pageX:x, pageY:y, radiusX:1, radiusY:1, rotationAngle:0, force:1}); el.dispatchEvent(new TouchEvent('touchstart', {touches:[touch(300,200)], changedTouches:[touch(300,200)]})); el.dispatchEvent(new TouchEvent('touchend', {touches:[], changedTouches:[touch(50,200)]})); }). Then assert: await expect(page.locator('.carousel-slide.active')).toHaveAttribute('data-index','1'). For simpler tap: page.tap('.button') works when hasTouch:true is set.

Session 25 · Page Information Methods

Playwright provides rich methods to inspect the current page's state: page.url() (current URL), page.title() (document title), page.content() (full HTML source), page.evaluate() (run JS), and assertions like toHaveURL() and toHaveTitle(). These verify navigation worked, SEO metadata is correct, and page state matches expectations.

WHAT are page info methods?

APIs that read information about the current page state: URL, title, full HTML content, viewport size, cookies visible to JS, and arbitrary JS evaluation. Essential for navigation verification and SEO testing.

WHY use them?

Verify navigation landed on the correct page. Assert SEO meta tags are present and correct. Check canonical URLs. Read HTML structure for accessibility audits. Get JS values from the page runtime.

WHEN to use?

After every navigation to confirm correct URL. After login to verify redirect. After form submit to check new URL. After dynamic content loads to verify title updates. In SEO test suites to verify all meta tags.

HOW they work

page.url() and page.title() are synchronous reads from Playwright's internal state. page.content() is async — it asks the browser for current DOM. page.evaluate() runs JS in the browser context and returns the result.

Keyword Glossary
page.url()Synchronous method — returns the current page URL as a string. Reflects the final URL after any redirects. Use for navigation verification and after form submissions.
page.title()Async method — returns document.title as a string. Use for SEO testing and verifying dynamic title updates (SPA route changes, product names in title).
page.content()Async method — returns the full outer HTML of the page as a string. Includes everything: head, body, scripts. Use for accessibility audits, meta tag verification, HTML structure testing.
page.evaluate()Runs a JavaScript function in the browser context, returns the result to Node.js. Use to read any JavaScript value: window.location, document.cookie, localStorage, custom app state, computed styles.
toHaveURL()Playwright auto-retrying assertion. Accepts string (exact match), regex (pattern), or predicate function. Retries until the URL matches, useful for async navigation.
toHaveTitle()Auto-retrying assertion on page title. Accepts string (exact) or regex. Retries — good for SPAs where the title updates asynchronously after data loads.
page.waitForURL()Waits (not an assertion) for the URL to match a pattern. Accepts string, regex, or predicate. Use after clicks that trigger navigation to wait before proceeding.
page.evaluate() vs page.locator()locator() finds DOM elements and interacts with them (Playwright-side). evaluate() runs arbitrary JS in the browser context — use when you need values that aren't accessible through the DOM (window variables, computed state).

Interview Prep
Q1: What is the difference between page.url() and page.waitForURL()? ▶
page.url() is a synchronous read of the current URL — it returns whatever the URL is at that exact moment. If navigation is in progress, it might return the old URL. page.waitForURL(pattern) is an async method that waits until the URL matches the pattern before resolving. Use page.waitForURL() after triggering navigation to wait for it to complete: await page.click('#login-btn'); await page.waitForURL('**/dashboard'). Then assert content. page.url() is for reading the final URL after you know navigation is complete. expect(page).toHaveURL() is the assertion form — it auto-retries with timeout, making it more reliable than page.url() for flaky navigation timing.
Q2: How do you verify SEO meta tags using page.content()? ▶
const html = await page.content(); // Verify title tag expect(html).toContain('<title>MyApp — Dashboard</title>'); // Verify canonical URL expect(html).toMatch(/<link rel="canonical" href="https:\/\/myapp\.com\/dashboard"\/>/); // Verify og:title expect(html).toMatch(/<meta property="og:title" content="Dashboard — MyApp"\/>/); // Verify meta description expect(html).toMatch(/<meta name="description" content=".{50,160}"\/>/) — 50-160 chars. // Verify robots not noindex expect(html).not.toContain('robots" content="noindex'); // Check structured data expect(html).toContain('application/ld+json'). page.content() gets everything including the head section making it ideal for comprehensive SEO testing.
Q3: How do you assert a URL that changes dynamically (contains an ID)? ▶
After creating a resource, the URL often contains a dynamic ID: /orders/123456. Use regex: await expect(page).toHaveURL(/\/orders\/\d+/). Or predicate: await expect(page).toHaveURL(url => { const path = new URL(url).pathname; return path.startsWith('/orders/') && path.split('/').pop()?.length > 0; }). For URL parameters: await expect(page).toHaveURL(url => new URL(url).searchParams.get('status') === 'confirmed'). Extract the ID for later use: const orderId = page.url().split('/').pop(); console.log('Created order:', orderId). The regex approach handles dynamic IDs, timestamps, and UUIDs in URLs without hardcoding specific values.
Q4: When should you use page.evaluate() vs page.locator()? ▶
page.locator() should be your default for anything in the DOM — it has auto-waiting, retry logic, and Playwright's full assertion support. Use page.evaluate() only when you need values from the JavaScript runtime that aren't accessible through the DOM: window.ENV (app environment config), window.analytics.dataLayer (analytics events), document.cookie (for non-httpOnly cookies), navigator.language, navigator.userAgent, localStorage values, sessionStorage values, performance.timing data, custom app state objects like window.store.getState(). Also use evaluate() when you need to trigger JS that doesn't have a corresponding DOM interaction: simulating events, calling app APIs directly, or reading computed JS values after interactions.
Q5: How do you test browser history navigation (back/forward)? ▶
Playwright exposes goBack() and goForward() mirroring browser buttons: await page.goto('/products'); await page.click('.product-card a'); await expect(page).toHaveURL(/\/products\/\d+/); await page.goBack(); await expect(page).toHaveURL('https://myapp.com/products'); await expect(page.locator('.product-card')).toBeVisible() — verify page restored; await page.goForward(); await expect(page).toHaveURL(/\/products\/\d+/) — forward also works. Verify scroll position restoration: const scrollY = await page.evaluate(() => window.scrollY); expect(scrollY).toBeGreaterThan(0). Test that the browser back button from a protected page redirects to login: await page.goto('/dashboard'); await page.goBack(); await expect(page).toHaveURL(/login/).
