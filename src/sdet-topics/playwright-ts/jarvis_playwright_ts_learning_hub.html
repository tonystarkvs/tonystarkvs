Session 1 · Core Definition

A BrowserContext in Playwright is an isolated browser session — like an incognito profile — that has its own cookies, cache, storage, and authentication state, completely separate from other contexts running in the same browser instance.

WHAT is it?

A lightweight container inside a Browser object. Each context is fully isolated: separate cookies, localStorage, sessionStorage, network intercepts and auth tokens. Multiple contexts can run inside one browser process simultaneously.

WHY use it?

Avoid test pollution between test cases. Run multi-user scenarios simultaneously (e.g. admin + buyer). Test incognito behavior. Reuse saved authentication state across sessions to save time.

WHEN to use it?

Every test should have its own context. Use multiple contexts when testing multi-user flows, role-based access, or when you need parallel isolated sessions in one browser process.

WHERE does it live?

Context lives inside the Browser object. A Browser can have many Contexts. Each Context can have many Pages (tabs). Structure: Browser → Context → Page

HOW to create one?

Use browser.newContext(options) to create a standard context, or browser.newContext({storageState}) to restore auth. For incognito-like behavior, each newContext is already isolated. Playwright Test auto-manages context per test when using fixtures. Always call await context.close() in cleanup to prevent resource leaks.

viewportSets browser window size for the context. e.g. {width:1280, height:720}. Affects responsive layout tests.
permissionsArray of browser permissions to grant: 'geolocation', 'notifications', 'camera', 'microphone'. Bypasses native browser prompts.
context.addCookies()Programmatically injects cookies into the context. Useful for pre-seeding session cookies without a real login flow.

⬡ Context Isolation Model

Each BrowserContext maintains its own: Cookies, LocalStorage, SessionStorage, IndexedDB, Service Workers, HTTP Auth, Network Intercepts, and Geolocation. Nothing leaks between contexts.

This is why Playwright tests are independent by default — each test gets a fresh context, meaning no leftover login sessions, no stale cookies, no shared state pollution from previous tests.

Best Practices Checklist — Session 1
  • ✓
    Always create a new BrowserContext per test — never share contexts between tests. This is the foundation of test isolation in Playwright.
  • ✓
    Use storageState for authentication — log in once in globalSetup, save state to JSON, restore in each test. Eliminates login overhead and test flakiness from login UI changes.
  • ✓
    Always await context.close() in a finally block or use Playwright Test fixtures. Unclosed contexts are silent memory leaks in long CI runs.
  • ✓
    Set viewport explicitly in context options for visual consistency. Never rely on browser defaults — they differ across environments and break responsive tests.
  • ✓
    Use context.grantPermissions() instead of clicking native browser permission dialogs — this is deterministic and works in all headless environments.
  • ✓
    Prefer browser.newContext() over launching a new browser per test. Launching a browser is ~1s overhead; creating a context is ~5ms.
  • ✓
    For multi-role tests, use descriptive variable names: adminContext, buyerContext — not context1, context2. Makes test logic readable.
  • ✓
    Set locale, timezoneId, geolocation in context options for internationalization tests — never mock at the code level when the browser itself can be configured.
Interview Prep — 5 SDET Q&A
Q1: What is the difference between a Browser, BrowserContext, and Page in Playwright? ▶
Browser is the actual browser process (Chromium/Firefox/WebKit). BrowserContext is an isolated session within that browser — it has its own cookies, storage, and auth state. Page is a single tab or window inside a context. The hierarchy is Browser → Context → Page. One browser can run many contexts; each context can have many pages. This design lets you test multi-user scenarios with one browser process instead of launching multiple browsers, saving significant time and resources.
Q2: How do you avoid logging in on every test in Playwright? ▶
I use storageState. In globalSetup.ts, I launch a browser, log in once through the full UI flow, then call await context.storageState({path: 'playwright/.auth/user.json'}). This saves all cookies and localStorage to a file. Each test then creates a new context with: browser.newContext({ storageState: 'playwright/.auth/user.json' }) — restoring the session without any login steps. The test starts already authenticated, making the suite significantly faster and decoupled from login UI changes.
Q3: How would you test a multi-user real-time feature — e.g., one user sends a message, another receives it? ▶
I create two separate BrowserContexts in the same test. adminContext gets created with admin storageState, buyerContext with buyer storageState. Each context gets its own page. I navigate both pages to the chat feature. Admin page sends a message via adminPage.fill() and click. Then on buyerPage, I use await expect(buyerPage.locator('.message')).toContainText('Hello') to verify the buyer receives it in real-time. Closing both contexts in a finally block at the end ensures cleanup. This avoids launching two separate browsers and runs in one test process.
Q4: What happens if you don't close a BrowserContext? What are the implications? ▶
Unclosed contexts remain in memory for the lifetime of the browser process. In large test suites, this causes gradual memory growth, eventually leading to OOM (out of memory) crashes in CI environments. Additionally, open contexts may hold open network connections and service workers. In Playwright Test, the built-in context fixture handles cleanup automatically via its teardown lifecycle. In raw API usage, I always use try/finally: const context = await browser.newContext(); try { ... } finally { await context.close(); } — the finally block guarantees cleanup even when the test throws an error.
Q5: How do you test browser geolocation permissions in Playwright? ▶
Playwright lets you set geolocation at the context level, which is far more reliable than trying to interact with native browser permission dialogs. I create a context with: const context = await browser.newContext({ geolocation: { latitude: 28.6139, longitude: 77.2090 }, permissions: ['geolocation'] }). This immediately grants the geolocation permission and provides a mock location. The page's navigator.geolocation.getCurrentPosition() call returns this fake location. I can also change geolocation mid-test using await context.setGeolocation({ latitude: 19.0760, longitude: 72.8777 }) to simulate the user moving.

basic-context.ts
// ─────────────────────────────────────────────────────────────────
// basic-context.ts
// PURPOSE: Shows how to create and use a BrowserContext manually
// USE CASE: Raw API usage (without Playwright Test fixture)
// ─────────────────────────────────────────────────────────────────

import { chromium, Browser, BrowserContext, Page } from '@playwright/test';

// Wrap in an immediately invoked async function for top-level await
(async () => {

  // STEP 1: Launch the browser — this opens the actual Chromium process
  // headless: false → shows the browser window (useful for debugging)
  const browser: Browser = await chromium.launch({ headless: false });

  // STEP 2: Create a BrowserContext — isolated session with its own cookies/storage
  const context: BrowserContext = await browser.newContext({
    viewport: { width: 1280, height: 720 },   // fixed browser window size
    locale: 'en-US',                              // browser language setting
    timezoneId: 'Asia/Kolkata',                   // IST timezone for date/time tests
  });

  // STEP 3: Create a Page (tab) inside our context
  const page: Page = await context.newPage();

  // STEP 4: Navigate to a URL
  await page.goto('https://playwright.dev');

  // STEP 5: Verify the title to confirm successful navigation
  const title = await page.title();
  console.log(`Page title: ${title}`);

  // STEP 6: Always close the context in teardown — this closes all its pages too
  await context.close();

  // STEP 7: Close the browser process — only done once at end of session
  await browser.close();

})();
incognito-context.ts
// ─────────────────────────────────────────────────────────────────
// incognito-context.ts
// PURPOSE: Demonstrates that every newContext() IS incognito by default
// KEY INSIGHT: Playwright contexts are always isolated — no shared cookies
// ─────────────────────────────────────────────────────────────────

import { test, expect } from '@playwright/test';

test.describe('Incognito / Private Context Demo', () => {

  test('fresh context has no cookies from other tests', async ({ browser }) => {

    // Context A: Set a cookie manually to simulate a logged-in session
    const contextA = await browser.newContext();
    await contextA.addCookies([{
      name: 'session_token',
      value: 'abc123',
      domain: 'example.com',
      path: '/',
    }]);

    // Verify Context A has the cookie we added
    const cookiesA = await contextA.cookies();
    console.log('Context A cookies:', cookiesA.length); // → 1

    // Context B: Brand new context — starts completely empty (true incognito)
    const contextB = await browser.newContext();
    const cookiesB = await contextB.cookies();
    console.log('Context B cookies:', cookiesB.length); // → 0

    // Assertion: Context B must be completely empty
    await expect(cookiesB.length).toBe(0);

    // Cleanup: always close contexts you created manually
    await contextA.close();
    await contextB.close();
  });
});
storage-state.ts
// ─────────────────────────────────────────────────────────────────
// storage-state.ts
// PURPOSE: Save and restore auth session using storageState
// PATTERN: globalSetup saves → each test restores (no re-login needed)
// ─────────────────────────────────────────────────────────────────

import { chromium, FullConfig } from '@playwright/test';

// This is globalSetup — runs ONCE before all tests
export default async function globalSetup(config: FullConfig) {

  // Launch browser for the setup step only
  const browser = await chromium.launch();
  const context = await browser.newContext();
  const page = await context.newPage();

  // Perform actual login through the UI (full real login flow)
  await page.goto('https://myapp.com/login');
  await page.fill('#email', 'admin@example.com');
  await page.fill('#password', 'SecurePass123');
  await page.click('button[type="submit"]');

  // Wait until we're redirected to dashboard (login is complete)
  await page.waitForURL('**/dashboard');

  // ✅ SAVE the cookies + localStorage to a JSON file
  await context.storageState({ path: 'playwright/.auth/admin.json' });
  console.log('✅ Auth state saved to playwright/.auth/admin.json');

  await browser.close();
}

multi-context.ts
// ─────────────────────────────────────────────────────────────────
// multi-context.ts
// PURPOSE: Two users interact simultaneously in one test
// USE CASE: Testing real-time features like chat, notifications, shared docs
// ─────────────────────────────────────────────────────────────────

import { test, expect } from '@playwright/test';

test('admin sends message → buyer receives it in real-time', async ({ browser }) => {

  // ── Context 1: Admin user (restore their saved session) ──────────
  const adminContext = await browser.newContext({
    storageState: 'playwright/.auth/admin.json',
  });
  const adminPage = await adminContext.newPage();

  // ── Context 2: Buyer user (completely separate session) ──────────
  const buyerContext = await browser.newContext({
    storageState: 'playwright/.auth/buyer.json',
  });
  const buyerPage = await buyerContext.newPage();

  // Both navigate to the chat room simultaneously
  await Promise.all([
    adminPage.goto('https://myapp.com/chat/room-1'),
    buyerPage.goto('https://myapp.com/chat/room-1'),
  ]);

  // Admin types and sends a message
  await adminPage.fill('.message-input', 'Hello from Admin!');
  await adminPage.click('.send-button');

  // Buyer's page should receive the message in real-time (WebSocket push)
  await expect(buyerPage.locator('.chat-message').last())
    .toContainText('Hello from Admin!');

  // ── Cleanup: close both contexts ──────────────────────────────────
  await adminContext.close();
  await buyerContext.close();
});
permissions.ts
// ─────────────────────────────────────────────────────────────────
// permissions.ts
// PURPOSE: Grant browser permissions at context level
// AVOIDS: Clicking native browser permission popups (impossible in headless)
// ─────────────────────────────────────────────────────────────────

import { test, expect } from '@playwright/test';

test('location-based feature works with mocked geolocation', async ({ browser }) => {

  // Create context with geolocation + permission granted (New Delhi coordinates)
  const context = await browser.newContext({
    // Grant the 'geolocation' permission — bypasses native browser dialog
    permissions: ['geolocation'],
    // Provide mock coordinates (New Delhi, India)
    geolocation: { latitude: 28.6139, longitude: 77.2090 },
  });

  const page = await context.newPage();
  await page.goto('https://myapp.com/nearby-stores');

  // Page calls navigator.geolocation.getCurrentPosition() → gets our fake coords
  await page.click('#find-nearby-btn');

  // Should show Delhi-based stores
  await expect(page.locator('.store-city').first())
    .toContainText('Delhi');

  // Change geolocation mid-test to simulate user moving to Mumbai
  await context.setGeolocation({ latitude: 19.0760, longitude: 72.8777 });
  await page.click('#find-nearby-btn'); // refresh results

  // Now Mumbai stores should appear
  await expect(page.locator('.store-city').first())
    .toContainText('Mumbai');

  await context.close();
});
context-options.ts
// ─────────────────────────────────────────────────────────────────
// context-options.ts
// PURPOSE: Full reference of ALL BrowserContextOptions
// USE CASE: Context factory function used across your test framework
// ─────────────────────────────────────────────────────────────────

import { chromium, BrowserContextOptions } from '@playwright/test';

// Reusable factory: creates a context with any override options
async function createContext(overrides?: Partial<BrowserContextOptions>) {

  const browser = await chromium.launch({ headless: true });

  // Default context options — override any at call site
  const defaultOptions: BrowserContextOptions = {

    // ── Window & Display ────────────────────────────────────────
    viewport: { width: 1280, height: 720 },  // browser window size
    deviceScaleFactor: 1,                    // retina = 2, standard = 1
    isMobile: false,                          // enables mobile emulation mode
    hasTouch: false,                          // enables touch events

    // ── Localization ────────────────────────────────────────────
    locale: 'en-IN',                          // browser Accept-Language header
    timezoneId: 'Asia/Kolkata',               // affects Date() in JS

    // ── Permissions (bypass native browser dialogs) ─────────────
    permissions: ['notifications', 'clipboard-read'],

    // ── Network ──────────────────────────────────────────────────
    offline: false,                           // simulate offline mode
    ignoreHTTPSErrors: true,                  // ignore SSL cert errors (staging)

    // ── HTTP Auth (basic auth popup) ─────────────────────────────
    httpCredentials: {
      username: 'admin',
      password: 'secret',
    },

    // ── Extra HTTP Headers (added to every request) ───────────────
    extraHTTPHeaders: {
      'X-Test-Environment': 'playwright',
    },

    // ── Auth State (restore saved session) ───────────────────────
    // storageState: 'playwright/.auth/user.json',

    // ── Video & Tracing (debugging) ───────────────────────────────
    recordVideo: { dir: 'videos/' },        // saves MP4 per test
  };

  return browser.newContext({ ...defaultOptions, ...overrides });
}
