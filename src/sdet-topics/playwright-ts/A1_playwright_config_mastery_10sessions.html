Playwright config mastery — 10 structured learning sessions from fundamentals to real-world

Session 1 — What is playwright.config.ts and why does it exist?
Think of playwright.config.ts as the brain of your entire test framework. Every decision Playwright makes — which browser to use, how long to wait, where to save screenshots, whether to retry — comes from this single file.
The mental model — one file controls everything
Without configWith config
Must pass flags every time: --browser=firefox --retries=2Set once in config, just run npx playwright test
Each dev uses different settings → inconsistent resultsEveryone uses identical settings → reproducible results
CI needs a massive shell script of flagsCI just runs one command
Timeout changes require editing every test fileChange one line in config → affects all tests
Where does it live and how is it found?
# Playwright looks for config in this order: playwright.config.ts ← default (TypeScript, recommended) playwright.config.js ← JavaScript alternative playwright.config.mjs ← ESM alternative # Override the location: npx playwright test --config=./configs/staging.config.ts
Three levels of config — how they interact
LevelLocationScopeOverride priority
Globaluse:{} in config rootAll tests, all projectsLowest
Projectprojects[].use:{}One project onlyMiddle
Test-leveltest.use({}) inside test fileOne test or describe blockHighest
Exercise 1 — create your first config
// playwright.config.ts — minimal working config import { defineConfig } from '@playwright/test'; export default defineConfig({ testDir: './tests', // where to find test files timeout: 30_000, // 30 seconds per test use: { baseURL: 'https://playwright.dev', headless: true, }, });
Exercise 2 — test-level override beats config
// In a test file — overrides the global use block for this file only import { test } from '@playwright/test'; test.use({ baseURL: 'https://staging.myapp.com', // overrides config baseURL viewport: { width: 375, height: 812 }, // mobile size }); test('runs on staging with mobile viewport', async ({ page }) => { await page.goto('/'); // resolves to staging URL });
Exercise 3 — environment-aware config
// playwright.config.ts — different settings per environment import { defineConfig } from '@playwright/test'; const ENV = process.env.ENV ?? 'staging'; const urls = { local: 'http://localhost:3000', staging: 'https://staging.myapp.com', prod: 'https://myapp.com', }; export default defineConfig({ use: { baseURL: urls[ENV], // picks URL by ENV variable headless: ENV !== 'local', // headed only on local }, retries: ENV === 'prod' ? 0 : 2, // no retries on prod });
Interview one-liner: "playwright.config.ts is the single source of truth — it controls browser selection, timeouts, reporters, retries, and environment settings so every developer and CI run is identical."

Exercise 1 — fully annotated skeleton
import { defineConfig, devices } from '@playwright/test'; export default defineConfig({ // ── SECTION 1: TEST DISCOVERY ───────────────────────────── testDir: './tests', testMatch: ['**/*.spec.ts'], testIgnore: ['**/node_modules/**'], // ── SECTION 2: EXECUTION ────────────────────────────────── fullyParallel: true, workers: 4, retries: 2, timeout: 30_000, globalTimeout: 3_600_000, // ── SECTION 3: OUTPUT ───────────────────────────────────── outputDir: 'test-results/', reporter: [[ 'html', { open: 'never' }]], // ── SECTION 4: GLOBAL use{} BLOCK ───────────────────────── use: { baseURL: 'https://myapp.com', headless: true, screenshot: 'only-on-failure', video: 'retain-on-failure', trace: 'on-first-retry', }, // ── SECTION 5: PROJECTS ─────────────────────────────────── projects: [ { name: 'chromium', use: { ...devices['Desktop Chrome'] } }, { name: 'firefox', use: { ...devices['Desktop Firefox'] } }, ], });
Exercise 2 — defineConfig wrapper — why use it?
// WITHOUT defineConfig — works but no TypeScript help export default { testDir: './tests', // no autocomplete, no type errors caught }; // WITH defineConfig — full TypeScript IntelliSense + validation export default defineConfig({ testDir: './tests', // IDE shows all valid keys + their allowed values // misspelling 'testdir' becomes a TypeScript error });
Always wrap your config in defineConfig(). It gives you autocomplete for every key and TypeScript will catch typos before you even run the tests.
Exercise 3 — splitting a large config into multiple files
// configs/base.config.ts — shared foundation export const baseConfig = { testDir: './tests', timeout: 30_000, reporter: [['html', { open: 'never' }]], }; // playwright.config.ts — UI tests extend base import { defineConfig, devices } from '@playwright/test'; import { baseConfig } from './configs/base.config'; export default defineConfig({ ...baseConfig, // spread the shared config use: { baseURL: 'https://myapp.com', headless: true }, projects: [{ name: 'chrome', use: { ...devices['Desktop Chrome'] } }], });
Interview one-liners
"playwright.config.ts has five sections: discovery, execution, use block, output, and projects — each controlling a distinct concern."
"defineConfig() gives TypeScript type-safety for the config — it catches invalid keys and wrong value types at compile time."
"You can split a large config into a base file and environment-specific files, then spread the base using ES6 spread syntax."
Session 3 — Test discovery keys: testDir, testMatch, testIgnore
These three keys tell Playwright where to look for tests and which files to treat as specs. Getting these wrong means tests silently never run.
All discovery keys with every possible value
testDir
String path relative to config file. Default: '.' (project root). Example: './tests', './src/__tests__', './e2e'
testMatch
String glob, RegExp, or array of either. Default: '**/*.{test,spec}.{js,ts,mjs,mts,cjs,cts}'. Examples: '**/*.spec.ts', [/.*\.spec\.ts/, '**/*.test.ts']
testIgnore
String glob, RegExp, or array. Skips matching files. Default: '**/node_modules/**'. Example: ['**/*.skip.ts', '**/drafts/**']
snapshotDir
String path. Where screenshot snapshots live. Default: next to test file. Example: './snapshots'
snapshotPathTemplate
Template string with {testDir}, {testFilePath}, {arg}, {ext}, {projectName} tokens.
Exercise 1 — testMatch patterns explained
export default defineConfig({ testDir: './tests', // Pattern 1: only .spec.ts files (most common for TypeScript) testMatch: '**/*.spec.ts', // Pattern 2: spec.ts AND test.ts files testMatch: ['**/*.spec.ts', '**/*.test.ts'], // Pattern 3: only files in the e2e subfolder testMatch: '**/e2e/**/*.spec.ts', // Pattern 4: use a RegExp for complex matching testMatch: /.*\.(spec|test)\.(ts|js)/, // Pattern 5: different per project (override in projects array) projects: [ { name: 'ui-tests', testMatch: '**/e2e/**/*.spec.ts', }, { name: 'api-tests', testMatch: '**/api/**/*.spec.ts', }, ], });
Exercise 2 — testIgnore for skipping files
export default defineConfig({ testDir: './tests', testMatch: '**/*.spec.ts', // Ignore multiple patterns testIgnore: [ '**/node_modules/**', // always ignore node_modules '**/*.skip.spec.ts', // skip files named *.skip.spec.ts '**/drafts/**', // skip all files in any drafts/ folder '**/wip/**', // skip work-in-progress folder /.*smoke.*\.spec\.ts/, // RegExp: skip any file with 'smoke' in name ], });
Exercise 3 — common mistake: testDir without testMatch
Mistake: setting testDir but forgetting tests are nested deeper
// ❌ WRONG — testDir points to tests/ but specs are in tests/e2e/api/ // If you set testMatch: '*.spec.ts' (no **) only the root is searched export default defineConfig({ testDir: './tests', testMatch: '*.spec.ts', // only finds spec files directly in tests/ });
Fix: always use ** globstar to recurse into subdirectories
// ✅ CORRECT — ** recurses all subdirectories export default defineConfig({ testDir: './tests', testMatch: '**/*.spec.ts', // finds specs anywhere under tests/ });
Run npx playwright test --list to see exactly which files Playwright found. If a test is missing, your testMatch pattern is the first place to check.
Session 4 — All timeout keys — what they are, what they control
Playwright has five different timeouts. Each controls a different scope. Beginners confuse them constantly. This session maps each one clearly.
The 5 timeout types — visual hierarchy
KeyDefaultWhat it limitsWhere set
globalTimeout0 (no limit)Entire test suite runRoot config
timeout30 000msOne single testRoot config or per-test
expect.timeout5 000msOne assertion retry windowRoot config or per-expect
actionTimeout0 (uses test timeout)One action (click, fill, etc.)use{} block
navigationTimeout0 (uses test timeout)page.goto() and page.reload()use{} block
Exercise 1 — all timeout keys in one config
export default defineConfig({ // Entire suite must finish within 1 hour globalTimeout: 60 * 60 * 1_000, // 3 600 000 ms // Each individual test gets 30 seconds timeout: 30_000, // Each expect() retries for up to 10 seconds expect: { timeout: 10_000, }, use: { // Each click/fill/type must complete in 10 seconds actionTimeout: 10_000, // page.goto() must complete in 30 seconds navigationTimeout: 30_000, }, });
Exercise 2 — override timeout at test or describe level
import { test } from '@playwright/test'; // Override for one describe block test.describe('slow admin import flow', () => { test.setTimeout(120_000); // this describe gets 2 minutes test('imports large CSV file', async ({ page }) => { // gets 120 000ms not the global 30 000ms }); }); // Mark individual test as slow — triples its timeout test('exports PDF report', async ({ page }) => { test.slow(); // 30 000 × 3 = 90 000ms automatically // rest of test... }); // Override per-expect for one assertion test('wait for email to arrive', async ({ page }) => { await expect(page.locator('.inbox')).toHaveText( 'New email', { timeout: 60_000 } // this assertion waits 60 seconds ); });
Exercise 3 — common mistakes with timeouts
Mistake 1: using waitForTimeout() instead of proper waits
// ❌ WRONG — hardcoded sleep hides real timing issues await page.waitForTimeout(3000); // always waits 3s even if ready in 0.1s await page.click('#submit'); // ✅ CORRECT — wait for the actual condition await expect(page.locator('#submit')).toBeEnabled(); // retries until ready await page.click('#submit');
Mistake 2: test timeout too short for slow CI environments
// ❌ WRONG — 5s timeout fails on slow CI even for fast tests timeout: 5_000 // ✅ CORRECT — 30s local, but use retries on CI instead of huge timeouts timeout: 30_000, retries: process.env.CI ? 2 : 0,
Never set timeout: 0 — it means no limit and a hung test will block your entire CI pipeline forever.
Interview one-liners
"There are five timeouts: globalTimeout (whole suite), timeout (one test), expect.timeout (one assertion), actionTimeout (one click/fill), navigationTimeout (page.goto). Each has a different scope."
"test.slow() triples the configured timeout — use it for known-slow tests instead of setting an arbitrarily high number."
"Never use waitForTimeout() — it always waits the full duration regardless of whether the page is ready."
Session 5 — The use{} block — every key with all possible values
The use{} block is where you configure the browser context and page defaults. Every key here can be set globally in config, overridden per project, or overridden per test file using test.use().
Complete use{} reference
baseURL
URL string. Allows relative paths in page.goto('/'). Example: 'https://staging.myapp.com'
headless
true | false. Default: true. Set false to see the browser during debugging.
slowMo
number (ms). Slows every action by N ms. 0 = normal speed. Use 500 to watch what's happening.
viewport
{ width: number, height: number } | null (no fixed viewport). Default: { width: 1280, height: 720 }
screenshot
'on' | 'off' | 'only-on-failure' | 'retain-on-failure'. Default: 'off'
video
'on' | 'off' | 'retain-on-failure' | 'on-first-retry'. Default: 'off'
trace
'on' | 'off' | 'retain-on-failure' | 'on-first-retry' | 'on-all-retries'. Default: 'off'
storageState
Path string | object. Pre-set cookies + localStorage for auth. Example: 'playwright/.auth/user.json'
actionTimeout
number (ms). Max time for click/fill/type etc. Default: 0 (inherits test timeout).
navigationTimeout
number (ms). Max time for page.goto(), waitForURL(). Default: 0.
locale
BCP 47 string. Example: 'en-IN', 'de-DE', 'fr-FR', 'ja-JP'
timezoneId
IANA timezone. Example: 'Asia/Kolkata', 'America/New_York', 'Europe/London'
colorScheme
'light' | 'dark' | 'no-preference'. Default: 'light'
reducedMotion
'reduce' | 'no-preference'. Default: 'no-preference'
forcedColors
'active' | 'none'. Default: 'none'
ignoreHTTPSErrors
true | false. Default: false. Set true for self-signed certs in staging.
permissions
Array of strings: 'geolocation', 'camera', 'microphone', 'notifications', 'clipboard-read', 'clipboard-write'
geolocation
{ latitude: number, longitude: number, accuracy?: number }
extraHTTPHeaders
Object. Added to every request: { 'Authorization': 'Bearer token', 'X-API-Key': '123' }
acceptDownloads
true | false. Default: true
bypassCSP
true | false. Bypasses Content Security Policy. Default: false
offline
true | false. Simulate offline mode. Default: false
userAgent
String. Custom user agent string. Default: browser default.
javaScriptEnabled
true | false. Default: true
channel
'chrome' | 'msedge' | 'chrome-beta' | 'msedge-beta'. Use installed Chrome/Edge instead of bundled Chromium.
Exercise 1 — screenshot / video / trace decision guide
// Recommended settings for different environments: // LOCAL development — capture everything to help debug use: { screenshot: 'only-on-failure', // save screenshot when test fails video: 'retain-on-failure', // save video when test fails trace: 'on-first-retry', // record trace on retry (for debug) } // CI pipeline — minimal storage, trace is most valuable use: { screenshot: 'only-on-failure', video: 'off', // videos are large, skip on CI trace: 'retain-on-failure', // keep traces only when tests fail } // DEBUGGING a specific test — capture everything always use: { screenshot: 'on', // every test gets a screenshot video: 'on', // every test gets a video trace: 'on', // every test gets a trace slowMo: 500, // 500ms delay per action headless: false, // watch it in the browser }
Exercise 2 — geolocation and permissions
export default defineConfig({ use: { // Grant geolocation + clipboard permissions automatically permissions: ['geolocation', 'clipboard-read', 'clipboard-write'], geolocation: { latitude: 28.6139, longitude: 77.2090 }, // New Delhi locale: 'en-IN', timezoneId: 'Asia/Kolkata', }, });
Exercise 3 — dark mode and accessibility testing
// Test your app in dark mode export default defineConfig({ projects: [ { name: 'light-mode', use: { colorScheme: 'light' }, }, { name: 'dark-mode', use: { colorScheme: 'dark' }, }, { name: 'reduced-motion', use: { reducedMotion: 'reduce', colorScheme: 'no-preference' }, }, { name: 'forced-colors', // high-contrast mode use: { forcedColors: 'active' }, }, ], });
Session 6 — Reporters — all options, all values, real examples
Reporters control what happens to results after tests run. You can use multiple reporters simultaneously — each gets the same results and formats them differently.
All built-in reporters and their options
list
Verbose console output. One line per test with status. Best for local debugging. No options needed.
line
Compact console: one line per test, updates in-place. Good for watching a long run. No options.
dot
Minimal: one dot per test (. pass, F fail). Good for CI where you want minimal noise. No options.
html
Full interactive HTML report. Options: open: 'always'|'never'|'on-failure', outputFolder: string, host: string, port: number
json
Machine-readable JSON. Options: outputFile: string
junit
JUnit XML (for Jenkins, GitHub Actions, Azure DevOps). Options: outputFile: string, suiteName: string
github
Annotates GitHub PR diff view. No options. Only useful when running inside GitHub Actions.
blob
Binary output for merging sharded reports. Options: outputDir: string
Exercise 1 — multiple reporters at once
export default defineConfig({ reporter: [ // Always show live output in console ['list'], // Generate HTML report (never auto-open in CI) ['html', { open: 'never', outputFolder: 'playwright-report' }], // JUnit XML for Azure DevOps / Jenkins ['junit', { outputFile: 'test-results/junit.xml' }], // JSON for custom dashboards ['json', { outputFile: 'test-results/results.json' }], // Blob for sharded CI (merge later) ['blob', { outputDir: 'blob-report' }], ], });
Exercise 2 — environment-aware reporter
const isCI = !!process.env.CI; export default defineConfig({ reporter: isCI ? [ ['dot'], // minimal CI output ['github'], // annotate PR ['junit', { outputFile: 'results.xml' }], // for CI dashboard ['html', { open: 'never' }], // archive the report ] : [ ['list'], // verbose for local dev ['html', { open: 'on-failure' }], // auto-open on failure ], });
Exercise 3 — merging sharded reports (CI matrix strategy)
# In GitHub Actions — three parallel machines each run one shard npx playwright test --shard=1/3 --reporter=blob npx playwright test --shard=2/3 --reporter=blob npx playwright test --shard=3/3 --reporter=blob # After all shards finish — merge blob reports into one HTML npx playwright merge-reports --reporter=html ./blob-report # Open the merged report npx playwright show-report
In real projects: use list locally, dot + github + junit + html on CI. The HTML report is your best debugging tool — it shows screenshots, traces, and video for every failed test.
Interview one-liners
"You can configure multiple reporters simultaneously — list for live console output, html for a visual report, and junit for CI XML integration."
"The blob reporter enables sharding — each CI machine writes its own blob, then you merge them into one HTML report after all shards finish."
"Set open: 'never' on the HTML reporter in CI to avoid Playwright trying to open a browser on a headless server."
Session 7 — Projects: multi-browser, multi-env, dependencies
The projects array is the most powerful feature in playwright.config.ts. Each project is a named test matrix entry with its own browser, settings, and test filter. This is how you run the same tests across Chrome, Firefox, and mobile in one command.
Every key inside a project object
name
Required string. Used with --project=name. Example: 'chromium', 'api-staging'
use
Overrides the global use{} for this project only. All same keys as global use block.
testMatch
Glob / RegExp / array. Only this project runs matching files.
testIgnore
Glob / RegExp / array. This project skips matching files.
dependencies
Array of project names that must pass before this project runs. Used for setup projects.
retries
number. Overrides global retries for this project.
timeout
number (ms). Overrides global timeout for this project.
outputDir
string path. Project-specific output folder.
teardown
Name of another project to run after this project finishes (cleanup project).
Exercise 1 — multi-browser matrix
import { defineConfig, devices } from '@playwright/test'; export default defineConfig({ projects: [ // ── Desktop browsers ────────────────────────────────────── { name: 'chromium', use: { ...devices['Desktop Chrome'] } }, { name: 'firefox', use: { ...devices['Desktop Firefox'] } }, { name: 'webkit', use: { ...devices['Desktop Safari'] } }, { name: 'edge', use: { ...devices['Desktop Edge'], channel: 'msedge' } }, // ── Mobile emulation ───────────────────────────────────── { name: 'mobile-chrome', use: { ...devices['Pixel 5'] } }, { name: 'mobile-safari', use: { ...devices['iPhone 13'] } }, { name: 'mobile-android',use: { ...devices['Galaxy S9+'] } }, // ── Tablet emulation ───────────────────────────────────── { name: 'tablet-ipad', use: { ...devices['iPad Pro'] } }, ], });
Exercise 2 — setup project (auth before tests)
export default defineConfig({ projects: [ // Step 1: login and save auth state { name: 'setup', testMatch: /global\.setup\.ts/, // only runs global.setup.ts }, // Step 2: run UI tests using saved auth { name: 'chromium', use: { ...devices['Desktop Chrome'], storageState: 'playwright/.auth/user.json', // pre-authenticated! }, dependencies: ['setup'], // setup must pass first }, // Step 3: cleanup project runs after chromium tests { name: 'cleanup', testMatch: /global\.teardown\.ts/, teardown: 'chromium', // runs after chromium project }, ], });
Exercise 3 — multi-environment projects
export default defineConfig({ projects: [ // API tests against staging { name: 'api-staging', testMatch: '**/api/**/*.spec.ts', use: { baseURL: 'https://staging-api.myapp.com' }, }, // API tests against production { name: 'api-prod', testMatch: '**/api/**/*.spec.ts', use: { baseURL: 'https://api.myapp.com' }, retries: 0, // no retries on production }, // UI smoke tests — chromium only, tagged tests { name: 'smoke-chrome', testMatch: '**/e2e/**/*.spec.ts', use: { ...devices['Desktop Chrome'], baseURL: 'https://staging.myapp.com', }, }, ], });

Session 9 — Common mistakes, fixes & best practices checklist
These are the exact mistakes beginners make in their first real project. Each one has been seen in real codebases.
Mistake 1 — not using baseURL (hardcoding URLs in tests)
Every test has the full URL hardcoded — changing environment means editing 50 files
// ❌ WRONG — URL hardcoded in test await page.goto('https://staging.myapp.com/login'); await page.goto('https://staging.myapp.com/dashboard');
Set baseURL in config, use relative paths everywhere
// ✅ CORRECT — config has baseURL use: { baseURL: 'https://staging.myapp.com' } // Every test uses relative paths await page.goto('/login'); await page.goto('/dashboard');
Mistake 2 — screenshot/video/trace all set to 'on' in CI
Storing a video for every passing test creates GB of files in CI and slows the pipeline
// ❌ WRONG — captures everything for every test use: { screenshot: 'on', video: 'on', trace: 'on' } // ✅ CORRECT — only capture when something fails use: { screenshot: 'only-on-failure', video: 'retain-on-failure', trace: 'on-first-retry', }
Mistake 3 — setting retries too high to mask flaky tests
retries: 5 makes tests "pass" but hides flakiness — you're shipping unreliable code
// ❌ WRONG — masking flaky tests with high retries retries: 5 // ✅ CORRECT — retries 2 for CI safety, PLUS trace to diagnose retries: process.env.CI ? 2 : 0, use: { trace: 'on-first-retry' } // trace captures WHY it failed
Mistake 4 — workers too high for the machine
// ❌ WRONG — 16 workers on a 2-core CI machine causes thrashing workers: 16 // ✅ CORRECT — use percentage of available cores workers: process.env.CI ? 2 : '50%' // 50% of CPU cores locally
Mistake 5 — missing globalSetup for auth (logging in before every test)
Every test in a 200-test suite starts with a login — that's 200 login flows, slow and fragile
// ❌ WRONG — login repeated in beforeEach of every spec file test.beforeEach(async ({ page }) => { await page.goto('/login'); await page.fill('#email', 'user@test.com'); await page.fill('#password', 'secret'); await page.click('[type=submit]'); }); // ✅ CORRECT — login once in globalSetup, reuse storageState // global.setup.ts const page = await browser.newPage(); await page.goto('/login'); await page.fill('#email', 'user@test.com'); await page.click('[type=submit]'); await page.context().storageState({ path: 'playwright/.auth/user.json' }); // playwright.config.ts globalSetup: './global.setup.ts', use: { storageState: 'playwright/.auth/user.json' },
Best practices checklist
  • Always wrap config in defineConfig() for TypeScript validation
  • Set baseURL — never hardcode full URLs in test files
  • Use retries: process.env.CI ? 2 : 0 — retry on CI, not locally
  • Set trace: 'on-first-retry' — captures exactly what failed
  • Use screenshot: 'only-on-failure' and video: 'retain-on-failure'
  • Store tokens in env variables — never hardcode in config
  • Use globalSetup for auth — login once, not before every test
  • Use projects for multi-browser — never copy-paste the same tests
  • Set workers: '50%' locally — avoids overloading your machine
  • Use webServer config to auto-start your local app
  • Add .gitignore entries for playwright-report/, test-results/, .auth/
  • Run npx playwright test --list to verify test discovery before running
Session 10 — Interview Q&A — click to reveal answers

Q1. What is playwright.config.ts and what are its five main sections?

It is the single configuration file that controls every aspect of how Playwright runs. The five sections are: (1) test discovery — testDir, testMatch, testIgnore; (2) execution — workers, fullyParallel, retries, timeout; (3) use block — browser context defaults like baseURL, headless, screenshot, trace; (4) output — reporter, outputDir; (5) projects — the multi-browser/multi-environment matrix. Every decision Playwright makes traces back to one of these five sections.

Q2. What is the difference between timeout, actionTimeout, navigationTimeout, and expect.timeout?

timeout limits the entire test (default 30s). actionTimeout limits one action like click() or fill() — if not set, inherits from timeout. navigationTimeout limits page.goto() and waitForURL() — again inherits from timeout if not set. expect.timeout controls how long an assertion retries before failing (default 5s). In interviews: say "there are five timeouts: globalTimeout, timeout, expect.timeout, actionTimeout, navigationTimeout, each at a different scope."

Q3. How do you run tests on multiple browsers in Playwright?

Use the projects array. Each project entry has a name and a use block that spreads from the devices object: { name: 'firefox', use: { ...devices['Desktop Firefox'] } }. When you run npx playwright test, all projects run. To run only one: --project=firefox. The devices object contains predefined browser+viewport combos for all major browsers and mobile devices. This avoids any duplication — the same test files run across all browsers automatically.

Q4. What does storageState do and why is it used in config?

storageState saves the browser's cookies and localStorage to a JSON file and reloads them for subsequent tests. In config, you set it in the use block so every test starts already authenticated. The workflow is: globalSetup runs once, logs in, saves state with page.context().storageState({ path: 'user.json' }), then every test context loads that file and skips the login UI entirely. This is the standard pattern for auth in production frameworks — login once, reuse everywhere.

Q5. What is the difference between fullyParallel and workers?

workers sets how many parallel worker processes run simultaneously — each worker handles one test at a time. fullyParallel: true means tests within the same file also run in parallel, not just tests across files. By default (fullyParallel: false), tests within one file run sequentially in a single worker, preserving order. Setting fullyParallel: true on a suite where tests share state causes race conditions — make sure each test is independent before enabling it.

Q6. How do you configure Playwright for a staging vs production environment?

Two approaches. First: read from an environment variable — baseURL: process.env.BASE_URL — then pass the value in CI: BASE_URL=https://staging.myapp.com npx playwright test. Second: use the projects array — create a 'staging' project and a 'prod' project each with their own baseURL, then run --project=staging or --project=prod. The projects approach is cleaner because it captures all environment-specific settings (headers, auth state, retries) in one named entry.

Q7. What does trace: 'on-first-retry' do and when would you use 'on' vs 'retain-on-failure'?

trace: 'on-first-retry' starts recording a trace when a test fails and is being retried — you get a trace for failures without the overhead of tracing every passing test. Use 'on' only when actively debugging a specific test — it records everything, creates large files. Use 'retain-on-failure' to record always but delete traces for passing tests automatically. In CI, 'on-first-retry' is the recommended setting — it captures the failure evidence without huge storage costs.

Q8. How do project dependencies work in Playwright?

The dependencies array in a project lists project names that must run and pass before this project starts. The classic use case is a 'setup' project that handles authentication. You define a setup project with testMatch pointing to global.setup.ts. The main test project lists dependencies: ['setup']. If setup fails, the dependent project is skipped entirely. You can also use teardown — a project that runs cleanup after another project finishes, even if that project had failures.

Q9. How does the use{} block at root level interact with use{} inside projects?

Project-level use overrides root-level use for that project only. The merge is deep — if root sets baseURL and screenshot, and a project sets only baseURL, the project inherits screenshot from root. The priority order is: root use (lowest) → project use (middle) → test.use() in a test file (highest). This lets you set sensible defaults globally and only override what differs per browser or environment — no need to repeat every key in every project.

Q10. What is the webServer config and when should you use it?

webServer tells Playwright to start your application server before running tests and stop it after. It has a command (npm run start), a url to poll until the server is ready, and reuseExistingServer to skip starting if one is already running. Use it when your tests run against a local development server. In CI, it ensures tests never start before the app is ready. Set reuseExistingServer: !process.env.CI so locally you can pre-start the server yourself, but in CI it always starts fresh.
