Session 16 · Cookie Handling

Cookies are small key-value data pieces stored by the browser and sent with every HTTP request to their domain. Playwright exposes full cookie control through context.addCookies(), context.cookies(), context.clearCookies(), and storageState — enabling auth injection, session testing, and cookie-based state verification.

WHAT are cookies?

Browser-stored key-value pairs with domain, path, expiry, httpOnly, secure, and sameSite attributes. Sent automatically in request headers. Managed per BrowserContext in Playwright — each context has its own cookie jar.

WHY handle cookies in tests?

Bypass login UI by injecting session cookies. Test cookie-based features (remember-me, GDPR consent). Verify cookies are set correctly after auth. Test cookie expiry behavior and secure attributes.

WHEN to use cookie APIs?

When restoring auth state without re-logging in, when testing cookie-dependent features, when verifying server sets correct cookies, when testing cross-domain cookie behavior, or when needing to simulate specific session states.

HOW does it work?

context.addCookies([]) injects cookies programmatically. context.cookies() reads all cookies. context.clearCookies() removes them. storageState saves/restores the full cookie jar including all attributes.

Keyword Glossary — Session 16
context.addCookies()Injects an array of cookie objects into the BrowserContext. Each cookie must have name, value, domain (or url). Optional: path, expires, httpOnly, secure, sameSite. Applied immediately — subsequent requests include these cookies.
context.cookies()Async method returning all cookies in the context as an array. Accepts optional URL filter: context.cookies('https://myapp.com') — returns only cookies for that domain.
context.clearCookies()Removes all cookies from the context. Pass optional filter: {name, domain, path} to selectively clear. Use to simulate logged-out state or test cookie expiry flows.
storageStateJSON snapshot containing all cookies + localStorage. Saved via context.storageState({path}), restored via browser.newContext({storageState}). The standard pattern for auth reuse across tests.
httpOnlyCookie attribute that prevents JavaScript from accessing the cookie via document.cookie. Must be set/read through Playwright's context API — not page.evaluate(). Server-set auth cookies are usually httpOnly.
secureCookie attribute requiring HTTPS to send the cookie. In Playwright tests with HTTP localhost, set secure:false or use ignoreHTTPSErrors:true in context options.
sameSiteControls cross-site request behavior: 'Strict' (never sent cross-site), 'Lax' (sent on top-level navigation), 'None' (always sent, requires secure:true). Important for CSRF and third-party cookie tests.
domainThe domain the cookie applies to. Use .myapp.com (dot prefix) for subdomain cookies. Must match the actual domain you're testing — mismatched domains cause cookies to be silently ignored.
expiresUnix timestamp (seconds) when the cookie expires. -1 = session cookie (deleted when browser closes). Future timestamp = persistent cookie. Set to a past timestamp to test cookie expiry behavior.
page.evaluate()Runs JavaScript inside the browser — can access document.cookie for non-httpOnly cookies. For httpOnly cookies, must use context.cookies() from the Playwright API side.

Interview Prep — 5 SDET Q&A
Q1: How do you inject a session cookie to bypass login in Playwright? ▶
I use context.addCookies() before any navigation: await context.addCookies([{ name: 'session_id', value: 'known-test-token', domain: 'myapp.com', path: '/', httpOnly: true, secure: false }]). Then await page.goto('/dashboard'). The page loads as if the user is logged in because the browser sends the cookie with every request. Important: the domain must exactly match the app's domain. I verify it worked with: const cookies = await context.cookies(); expect(cookies.find(c => c.name === 'session_id')).toBeTruthy(). For persistent auth across many tests, I use storageState which captures the full cookie jar from a real login.
Q2: How do you test that a server sets cookies correctly after login? ▶
After the login action, I read all cookies using context.cookies() and assert on their properties: const cookies = await context.cookies('https://myapp.com'). const sessionCookie = cookies.find(c => c.name === 'session_id'). expect(sessionCookie).toBeDefined(). expect(sessionCookie.httpOnly).toBe(true) — verifies JS can't steal it. expect(sessionCookie.secure).toBe(true) — verifies HTTPS only. expect(sessionCookie.sameSite).toBe('Strict') — verifies CSRF protection. expect(sessionCookie.expires).toBeGreaterThan(Date.now()/1000) — verifies it's not already expired. This tests the server's security configuration, not just that login worked.
Q3: What is the difference between context.cookies() and document.cookie in evaluate()? ▶
They have fundamentally different visibility. document.cookie (accessed via page.evaluate()) is subject to browser security rules — it only returns cookies that do NOT have the httpOnly flag. This is intentional: httpOnly cookies are invisible to JavaScript to prevent XSS attacks from stealing session tokens. context.cookies() is a Playwright API call that reads from the browser's internal cookie store directly, bypassing the JS sandbox. It returns ALL cookies including httpOnly ones. In practice: for reading session tokens and auth cookies (which are almost always httpOnly), always use context.cookies(). For testing that JavaScript can/cannot access a cookie, use page.evaluate(() => document.cookie) and verify the httpOnly cookie does NOT appear there.
Q4: How do you test cookie expiry behavior? ▶
Two scenarios: (1) Expired cookie — inject a cookie with a past expires timestamp: const pastTime = Math.floor(Date.now()/1000) - 3600 — that's 1 hour ago. await context.addCookies([{...cookie, expires: pastTime}]). Then navigate and verify the app treats the user as logged out. (2) Approaching expiry — set expires to 5 minutes from now: expires: Math.floor(Date.now()/1000) + 300. Navigate to a page that checks cookie age and assert it shows a "session expiring soon" warning. For session cookies (expires: -1), close and reopen the context to test they're cleared. Note: Playwright doesn't automatically expire cookies during a test — you need to either set a past timestamp or clear cookies manually to simulate expiry.
Q5: How do you handle third-party cookies and SameSite testing? ▶
sameSite:'None' cookies require secure:true, which requires HTTPS. For testing in localhost HTTP environments, set ignoreHTTPSErrors:true in context options. To test SameSite behavior: (1) For sameSite:'Strict' — verify the cookie is NOT sent when navigating from an external domain by checking the request headers via page.on('request', ...). (2) For sameSite:'Lax' — verify it IS sent on top-level navigation but not on cross-origin subresource requests. (3) For sameSite:'None' with cross-origin iframes, use context.addCookies with sameSite:'None', secure:true and verify the iframe receives the cookie. In modern browsers, third-party cookies are being phased out — Playwright lets you test both the legacy cookie behavior and the new partitioned cookie behavior.
add-cookies.ts
// add-cookies.ts
// PURPOSE: Inject cookies programmatically before navigation
// Shows all cookie properties and verification pattern

import { test, expect } from '@playwright/test';

test('inject cookies with all properties', async ({ context, page }) => {

  // Inject BEFORE navigation — cookies must exist before page loads
  await context.addCookies([
    {
      // Required fields
      name:     'session_id',
      value:    'test-session-abc123',
      domain:   'myapp.com',    // exact domain match required
      path:     '/',

      // Security attributes
      httpOnly: true,           // JS cannot read via document.cookie
      secure:   false,          // allow HTTP (set true for HTTPS only)
      sameSite: 'Lax',          // 'Strict' | 'Lax' | 'None'

      // Expiry: Unix timestamp (seconds from epoch)
      // -1 = session cookie, future = persistent cookie
      expires: Math.floor(Date.now() / 1000) + 3600, // 1 hour from now
    },

    // Multiple cookies in one call
    {
      name:   'user_preferences',
      value:  '{"theme":"dark","lang":"en"}',
      domain: 'myapp.com',
      path:   '/',
    },

    // Subdomain cookie — dot prefix makes it available on all subdomains
    {
      name:   'cross_subdomain',
      value:  'shared',
      domain: '.myapp.com',    // note the leading dot
      path:   '/',
    },
  ]);

  // Verify injection succeeded (domain mismatch = silent failure)
  const cookies = await context.cookies();
  expect(cookies.find(c => c.name === 'session_id')).toBeDefined();
  expect(cookies).toHaveLength(3);

  // Navigate — browser sends cookies automatically in request headers
  await page.goto('https://myapp.com/dashboard');

  // App sees session cookie → user is "logged in"
  await expect(page.locator('.user-greeting')).toBeVisible();
});

// read-cookies.ts
// PURPOSE: Read and inspect cookies — context.cookies() vs document.cookie
// KEY: httpOnly cookies only visible via context.cookies() not document.cookie

import { test, expect } from '@playwright/test';

test('read all cookies after login', async ({ context, page }) => {
  await page.goto('https://myapp.com/login');
  await page.fill('#email',    'user@test.com');
  await page.fill('#password', 'Pass123!');
  await page.click('button[type="submit"]');
  await page.waitForURL('**/dashboard');

  // Method 1: context.cookies() — ALL cookies including httpOnly
  const allCookies = await context.cookies();
  console.log(`Total cookies: ${allCookies.length}`);
  allCookies.forEach(c => console.log(`  ${c.name}=${c.value} httpOnly=${c.httpOnly}`));

  // Filter cookies for a specific URL
  const appCookies = await context.cookies('https://myapp.com');
  console.log(`App-specific cookies: ${appCookies.length}`);

  // Find specific cookie by name
  const sessionCookie = allCookies.find(c => c.name === 'session_id');
  expect(sessionCookie).toBeDefined();
  expect(sessionCookie!.httpOnly).toBe(true);
  expect(sessionCookie!.secure).toBe(true);

  // Method 2: page.evaluate — ONLY non-httpOnly cookies visible in JS
  const jsCookies = await page.evaluate(() => document.cookie);
  console.log(`JS-visible cookies: ${jsCookies}`);
  // session_id does NOT appear here because httpOnly:true prevents JS access
  expect(jsCookies).not.toContain('session_id');
});

test('simulate logout by clearing session cookies', async ({ context, page }) => {

    { name: 'session_id',   value: 'abc123', domain: 'myapp.com', path: '/' },

  await context.clearCookies({ name: 'session_id' });

  // Reload — should redirect to login since session_id gone
  await page.reload();
  await expect(page).toHaveURL(/login/);

  // Clear ALL cookies — full logout
  await context.clearCookies();
  const afterClear = await context.cookies();
  expect(afterClear).toHaveLength(0);

  // Filter by domain — clear only analytics cookies
  // await context.clearCookies({ domain: 'analytics.com' });
});

// Tests security configuration: httpOnly, secure, sameSite, expiry

import { test, expect } from '@playwright/test';

test('login sets correctly configured session cookie', async ({ context, page }) => {

  // No cookies before login
  expect(await context.cookies()).toHaveLength(0);

  // Perform login
  await page.goto('https://myapp.com/login');
  await page.fill('#email', 'admin@test.com');
  await page.fill('#password', 'Admin123!');
  await page.click('button[type="submit"]');
  await page.waitForURL('**/dashboard');

  // Read all cookies set by the server
  const cookies = await context.cookies('https://myapp.com');
  const session = cookies.find(c => c.name === 'session_id');

  // ── Security assertions ───────────────────────────────────────────
  expect(session, 'session_id cookie must exist').toBeDefined();

  // httpOnly: JS cannot steal the token via XSS
  expect(session!.httpOnly, 'Must be httpOnly to prevent XSS theft').toBe(true);

  // secure: only sent over HTTPS — prevents sniffing
  expect(session!.secure, 'Must be secure (HTTPS only)').toBe(true);

  // sameSite: CSRF protection
  expect(session!.sameSite, 'Must be Strict for CSRF protection')
    .toBe('Strict');

  // Path: scoped to the entire app
  expect(session!.path).toBe('/');

  // Expiry: must be a future timestamp (not session cookie)
  expect(session!.expires).toBeGreaterThan(Date.now() / 1000);

  // Token value should be non-trivial (at least 32 chars)
  expect(session!.value.length).toBeGreaterThanOrEqual(32);

  // httpOnly cookie NOT visible to JavaScript
  const jsCookies = await page.evaluate(() => document.cookie);
  expect(jsCookies).not.toContain('session_id');
});

Session 17 · LocalStorage Handling

LocalStorage is a browser storage mechanism that persists key-value strings across page loads within the same origin. Playwright accesses it via page.evaluate() (running JavaScript in the browser context) and through the storageState API which captures and restores all localStorage keys along with cookies.

WHAT is localStorage?

A synchronous key-value store in the browser. Data persists across page reloads and browser restarts (unlike sessionStorage). Scoped to origin (protocol + hostname + port). Stores only strings — objects need JSON serialization.

WHY test localStorage?

Apps store auth tokens, user preferences, cart state, dark-mode settings, feature flags, and form drafts in localStorage. Tests must set initial state, verify apps write correct values, and test app behavior when localStorage is empty, corrupted, or missing keys.

WHEN to access it?

When testing: localStorage-based auth (JWTs), app initialization from stored preferences, cart persistence across refreshes, feature flag overrides, and testing app recovery from corrupted storage data.

HOW to access?

Only via page.evaluate() — Playwright doesn't have a dedicated localStorage API. Inject state BEFORE navigation (via addInitScript) or AFTER (via evaluate). storageState snapshots include localStorage.

Keyword Glossary — Session 17
localStorageBrowser's persistent key-value string store. Origin-scoped. Survives page refresh and browser restart. Max ~5MB per origin. No expiry (unlike cookies). Accessed via window.localStorage in JS.
sessionStorageSimilar to localStorage but tab-scoped — cleared when tab closes. Not shared between tabs/windows. Use page.evaluate() to access. Useful for testing single-tab session state.
page.evaluate()Executes a JavaScript function in the browser's page context. Primary way to interact with localStorage. Can read, write, remove, and clear localStorage. Returns serializable values back to Node.js.
addInitScript()Injects a script that runs BEFORE any other script on the page — on every navigation. Use to set localStorage values before the app's JavaScript initializes and reads storage. Critical for localStorage-based auth.
storageStateJSON snapshot of both cookies AND localStorage for the context. The 'origins' array in the JSON contains localStorage key-value pairs per origin. Restore with browser.newContext({storageState}).
localStorage.setItem()JS method: window.localStorage.setItem('key', 'value'). Called inside page.evaluate(). Value must be a string — use JSON.stringify() for objects. Returns undefined.
localStorage.getItem()JS method: window.localStorage.getItem('key'). Returns string or null. Call inside page.evaluate() to read back to Node.js test code.
localStorage.removeItem()Removes a specific key from localStorage. Use to test app behavior when a required key is missing.
localStorage.clear()Removes ALL keys from localStorage for the current origin. Use to simulate a fresh user with no stored data.
context.addInitScript()Context-level version — runs on ALL pages opened in the context. Use for global localStorage state that must be present on every page load across all navigation.
Interview One-Liners — Session 17
evaluate vs initScript"The critical difference: page.evaluate() runs AFTER the page loads and the app's JS may have already read localStorage — so changes via evaluate() may be too late. page.addInitScript() runs BEFORE any app code, making it the correct way to seed localStorage for app initialization."

Interview Prep — 5 SDET Q&A
Q1: How do you set localStorage values before the page app code runs? ▶
The critical pattern is addInitScript(), not evaluate(). When you call page.goto(), the browser loads HTML, then JavaScript. If you use page.evaluate() after goto(), the app's JavaScript has already run — it already read localStorage and may have initialized state from the empty/wrong values. addInitScript() registers a script that runs BEFORE any other JavaScript on the page: await page.addInitScript(() => { window.localStorage.setItem('auth_token', 'my-test-jwt'); window.localStorage.setItem('user', JSON.stringify({id:1, role:'admin'})); }). Then await page.goto('/dashboard'). The app initializes and reads auth_token from localStorage — it sees the pre-seeded value as if the user previously logged in. This is the foundation of localStorage-based auth bypass.
Q2: How does storageState capture and restore localStorage? ▶
When you call context.storageState({path:'auth.json'}), Playwright snapshots the entire browser storage state into a JSON file. The file has two sections: "cookies" (all cookies in the context) and "origins" (localStorage per origin). The origins section looks like: {"origin":"https://myapp.com","localStorage":[{"name":"auth_token","value":"jwt..."},{"name":"preferences","value":"{theme:dark}"}]}. When you restore with browser.newContext({storageState:'auth.json'}), Playwright pre-populates both the cookie jar AND localStorage for each origin before any page is opened. The app on first navigation sees the restored state as if the user had previously been on the site. This is why storageState is the gold standard for auth reuse.
Q3: How do you test that an app correctly persists user preferences in localStorage? ▶
Example: testing dark mode persistence. Step 1 — verify initial state: const theme = await page.evaluate(() => localStorage.getItem('theme')); expect(theme).toBeNull() — no preference stored initially. Step 2 — trigger the preference change: await page.click('#dark-mode-toggle'); await expect(page.locator('body')).toHaveClass(/dark/). Step 3 — verify localStorage was updated: const newTheme = await page.evaluate(() => localStorage.getItem('theme')); expect(newTheme).toBe('dark'). Step 4 — verify persistence across reload: await page.reload(); await expect(page.locator('body')).toHaveClass(/dark/) — app read localStorage on load and applied the stored theme. This tests the full read-write-restore cycle.
Q4: How do you test app resilience when localStorage has corrupted data? ▶
I inject corrupted data using addInitScript (so it's set before app initialization): await page.addInitScript(() => { window.localStorage.setItem('user_config', '{broken json without closing brace'); window.localStorage.setItem('cart', 'not-valid-json-at-all'); }). Then await page.goto('/') — the app starts and tries to JSON.parse() these values. I assert: (1) No JavaScript errors: const errors: string[] = []; page.on('pageerror', e => errors.push(e.message)); await page.goto('/'); expect(errors).toHaveLength(0) — no uncaught errors. (2) Graceful fallback: await expect(page.locator('.cart-count')).toHaveText('0') — cart falls back to empty. (3) User message if needed: await expect(page.locator('.config-reset-notice')).toBeVisible(). This tests defensive programming in the app.
Q5: What is the difference between localStorage and sessionStorage in Playwright testing? ▶
Both are accessed identically via page.evaluate() — swap 'localStorage' for 'sessionStorage'. The behavioral difference: localStorage persists across page.reload(), page.goto() to other pages, and context restores via storageState. sessionStorage is tab-scoped — cleared when the browser tab closes (i.e., when page context is destroyed). In Playwright: sessionStorage is cleared when page.close() is called. sessionStorage is NOT included in storageState snapshots because it's inherently non-persistent. In tests: I use sessionStorage.setItem via evaluate/initScript for testing wizard-step state that shouldn't persist. I verify sessionStorage.getItem returns null after a page.close() + reopen cycle to confirm the app's session handling is correct. sessionStorage is visible in DevTools under Application > Session Storage.

Session 18 · Test Runner & Its Concepts in Playwright

The Playwright Test Runner (@playwright/test) is a full-featured test framework built specifically for browser automation. It manages test discovery, parallel execution, fixture injection, retries, reporters, and tracing — all configured through playwright.config.ts.

WHAT is the runner?

The test runner is the engine that discovers test files (*.spec.ts), builds a test plan, distributes tests across workers, injects fixtures, captures artifacts, and generates reports. It's the npx playwright test command.

WHY it matters

Understanding the runner's internals — worker architecture, fixture lifecycle, retry mechanism, project system — lets you optimize suite speed, debug flaky tests, and configure CI correctly. Most test bugs are runner-config issues.

Key concepts

Workers (parallel processes), Projects (browser configurations), Fixtures (DI system), Reporters (output formats), globalSetup/Teardown (once-per-suite), Retries (automatic re-runs), Sharding (CI splitting).

Test lifecycle

globalSetup → Worker starts → beforeAll → beforeEach → Test → afterEach → afterAll → Worker ends → globalTeardown. Each worker is an independent Node.js process with its own browser instance.

Interview Prep — 5 SDET Q&A
Q1: Explain the Playwright worker architecture and how parallel execution works. ▶
Playwright uses a multi-process architecture. The main process (the test runner) reads playwright.config.ts, collects all tests, builds a test plan, and then spawns N worker processes. Each worker is an independent Node.js process that: (1) Launches its own browser instance (one Chromium/Firefox/WebKit per worker). (2) Receives its subset of tests from the main process. (3) Runs each test sequentially within its browser. (4) Reports results back to the main process. Workers are stateless from each other — they share no memory, no browser state, and no fixtures. By default (fullyParallel:false), entire spec files are distributed across workers, so all tests within a file run serially on the same worker. With fullyParallel:true, individual tests are distributed, so different tests from the same file can run in different workers simultaneously — maximum parallelism but requires complete test independence.
Q2: What is the difference between globalSetup, beforeAll, and beforeEach? When do you use each? ▶
globalSetup: A file referenced in playwright.config.ts. Runs ONCE before any worker starts — plain Node.js, no fixtures. Use for: login once and save storageState, database migration, starting a test server. Perfect for operations that should happen exactly once for the entire suite. beforeAll: A hook inside a test file or describe block. Runs once per WORKER for that scope — with 4 workers, beforeAll runs 4 times. Has access to fixtures (request, browser). Use for: seeding data specific to a test group, one-time expensive setup for a set of related tests. beforeEach: Runs before every individual test. Has access to all fixtures. Use for: navigation to starting URL, creating per-test data, resetting page state between tests. Rule of thumb: auth/global-once → globalSetup. Shared-data-per-group → beforeAll. Per-test-setup → beforeEach. Cross-file-reusable-setup → fixtures.
Q3: How do you configure Playwright for multiple browsers in one CI run? ▶
I use the projects array in playwright.config.ts: projects: [{ name:'chromium', use:{...devices['Desktop Chrome']} }, { name:'firefox', use:{...devices['Desktop Firefox']} }, { name:'webkit', use:{...devices['Desktop Safari']} }, { name:'mobile-chrome', use:{...devices['Pixel 7']} }]. Each project runs the complete test suite with its browser configuration. Running npx playwright test executes all projects. For a specific project: --project=chromium. For auth with project dependencies: create a setup project that runs globalSetup login, then all browser projects declare dependencies:[{project:'setup'}] so storageState is ready before browser tests start. In CI, I might skip webkit on Linux workers (no native WebKit) and only run it on macOS runners.
Q4: How does test sharding work and how do you merge reports from multiple shards? ▶
Sharding splits the test plan (not files) across machines. --shard=1/4 runs the first 25% of tests deterministically on machine 1. Each machine uses blob reporter: reporter:[['blob']], which saves a binary report file. GitHub Actions matrix: shardIndex:[1,2,3,4] creates 4 parallel jobs, each running: npx playwright test --shard=${{matrix.shardIndex}}/4 --reporter=blob. Each job uploads its blob-report/ folder as an artifact. A final merge job downloads all blobs and runs: npx playwright merge-reports --reporter html ./all-blobs, producing a single unified HTML report showing all 100% of test results even though they ran on 4 different machines. This gives approximately 4× speed improvement for large suites without any test code changes.
Q5: What is the fixture DI system and how does it differ from beforeEach? ▶
Playwright's fixture system is a dependency injection container. Fixtures are declared as async generator functions in test.extend(): the code before yield/use() is setup, yield/use() runs the test, code after is guaranteed teardown. Tests declare which fixtures they need as function parameters — Playwright resolves the dependency graph and instantiates only the needed fixtures. Key differences from beforeEach: (1) Composability — fixtures can depend on other fixtures (page depends on context, context depends on browser). beforeEach cannot be meaningfully composed. (2) Laziness — unused fixtures are never instantiated. beforeEach always runs. (3) Cross-file sharing — fixtures defined in a custom test object are importable from any test file. beforeEach is file-scoped unless extracted to a helper. (4) Scope control — fixtures can be 'worker' scope (shared across tests in a worker) or 'test' scope (fresh per test). (5) Type safety — TypeScript knows the fixture types, giving IntelliSense.

// global-setup-teardown.ts
// PURPOSE: globalSetup runs ONCE before workers; globalTeardown after
// No fixtures — plain Node.js async functions

import { chromium, FullConfig, request } from '@playwright/test';
import fs from 'fs';

// ── global-setup.ts ───────────────────────────────────────────────
export default async function globalSetup(config: FullConfig) {
  console.log('🚀 globalSetup: starting — runs ONCE before any worker');

  // 1. API health check — abort entire suite if server is down
  const apiCtx = await request.newContext({ baseURL: 'https://myapp.com' });
  const health = await apiCtx.get('/api/health');
  if (!health.ok()) throw new Error(`API down: ${health.status()}`);
  await apiCtx.dispose();

  // 2. Browser login → save storageState
  fs.mkdirSync('playwright/.auth', { recursive: true });
  const browser = await chromium.launch();

  for (const { email, password, file } of [
    { email: 'admin@test.com', password: 'Admin123!', file: 'admin.json' },
    { email: 'buyer@test.com', password: 'Buyer123!', file: 'buyer.json' },
  ]) {
    const ctx  = await browser.newContext();
    const page = await ctx.newPage();
    await page.goto('https://myapp.com/login');
    await page.fill('#email', email);
    await page.fill('#password', password);
    await page.click('button[type="submit"]');
    await page.waitForURL('**/dashboard');
    await ctx.storageState({ path: `playwright/.auth/${file}` });
    await ctx.close();
  }

  await browser.close();
  console.log('✅ globalSetup complete — auth states saved');
}

// ── global-teardown.ts ────────────────────────────────────────────
export async function globalTeardown() {
  console.log('🧹 globalTeardown: runs ONCE after all workers finish');
  // Clean up seeded data, close servers, delete test users, etc.
  const apiCtx = await request.newContext({ baseURL: 'https://myapp.com' });
  await apiCtx.delete('/api/test-data/cleanup');
  await apiCtx.dispose();
}

  // testInfo.retry: 0 = first run, 1 = first retry, 2 = second retry
  console.log(`Running attempt ${testInfo.retry + 1}`);

  // Skip test if exceeded retries — stop wasting time on pathological flakiness
  test.skip(testInfo.retry > 2, 'Give up after 2 retries');

  // On retry: capture extra diagnostic info
  if (testInfo.retry > 0) {
    console.warn(`⚠️ Retry ${testInfo.retry}: capturing diagnostics...`);

    // Start manual trace on retry (supplementary to auto trace)
    const consoleMessages: string[] = [];
    page.on('console', msg => consoleMessages.push(`[${msg.type()}] ${msg.text()}`));

    await page.goto('https://myapp.com/dashboard');

    // Attach console log to report
    await testInfo.attach('console-log', {
      body: consoleMessages.join('\n'),
      contentType: 'text/plain',
    });
  } else {
    await page.goto('https://myapp.com/dashboard');
  }

  await expect(page.locator('.dashboard-content')).toBeVisible();
});
