❓ Interview Q&A
How does ArrayList grow internally? ▼
When the internal array is full, ArrayList creates a new array of size oldCapacity + (oldCapacity >> 1) (1.5x) and copies all elements using Arrays.copyOf(). This is O(n) but happens rarely, so amortized add is O(1).
ArrayList vs LinkedList — when to use which? ▼
ArrayList: fast random access O(1), slow insert/delete in middle O(n). LinkedList: slow random access O(n), fast insert/delete at head/tail O(1). Use ArrayList for most cases — its memory locality makes it faster in practice even for iteration.
What is ConcurrentModificationException and how to avoid it? ▼
It occurs when you modify a list while iterating over it with a for-each loop. Fix: use Iterator.remove(), list.removeIf(predicate), or iterate over a copy, or use CopyOnWriteArrayList.
What is the difference between ArrayList and Array? ▼
Array: fixed size, can hold primitives and objects. ArrayList: dynamic size, holds only objects (autoboxing for primitives), provides utility methods (sort, search, contains). ArrayList is more flexible but has slightly more memory overhead.
What does trimToSize() do? ▼
trimToSize() reduces the internal array capacity to the current size, freeing unused memory. Useful after bulk removal operations when the list won't grow again.

Example 1 — Basic list operations
import java.util.LinkedList;

  • Use Deque<T> deque = new LinkedList<>() for queue/stack — ArrayDeque is faster but LinkedList works.
  • Avoid index-based access (get(i)) in LinkedList — it's O(n). Use iterator instead.
  • Prefer ArrayDeque over LinkedList as a pure stack or queue — better cache performance.
  • Use offer/poll/peek (Queue methods) instead of add/remove/element — they return null instead of throwing exceptions.
  • Don't use LinkedList when most operations are random reads — ArrayList will outperform it.
❓ Interview Q&A
What interfaces does LinkedList implement? ▼
LinkedList implements List, Deque, Queue, Cloneable, and Serializable. Because it implements Deque, it can function as a stack, queue, or double-ended queue simultaneously.
What is the internal node structure of LinkedList? ▼
Each node is an instance of the private static class Node<E> with three fields: E item (the data), Node<E> next, and Node<E> prev. The list holds references to the first and last nodes.
ArrayList vs LinkedList — memory comparison? ▼
ArrayList stores a compact Object[] array — low overhead per element. LinkedList stores a Node object per element with two extra pointers (prev + next), so memory overhead is roughly 3× per element. For large lists, ArrayList uses significantly less memory.
When would you choose LinkedList over ArrayList? ▼
When you have many insertions/deletions at the beginning or middle of the list and very few random accesses. Common use cases: implementing a queue, a deque, or when processing a list from front to back while removing elements during processing.
What is the difference between poll() and remove() in LinkedList? ▼
poll() retrieves and removes the head element, returning null if the list is empty. remove() does the same but throws NoSuchElementException if empty. Prefer poll() in production code to avoid exception handling.

  • Never use Stack — use Deque<T> stack = new ArrayDeque<>() which is faster and more feature-rich.
  • If working with legacy code using Vector, understand that all methods are synchronized — avoid nested synchronization blocks to prevent deadlocks.

❓ Interview Q&A
What is the difference between Vector and ArrayList? ▼
Vector: synchronized (every method), doubles capacity when full, legacy (Java 1.0), slower. ArrayList: not synchronized, grows by 50%, introduced in Java 1.2, faster. Both are dynamic arrays. Use ArrayList unless you specifically need the built-in synchronization.
Why is Vector considered legacy? ▼
Vector predates the Java Collections Framework (Java 1.2). Its synchronized-by-default approach causes unnecessary overhead in single-threaded apps. Modern alternatives (CopyOnWriteArrayList, synchronizedList) offer more granular control.
How does Vector's capacity grow vs ArrayList? ▼
Vector doubles its capacity (×2) when full if no capacityIncrement is set. ArrayList grows by 50% (×1.5). Vector's doubling wastes more memory but causes fewer resizing operations.
Is Vector truly thread-safe? ▼
Each individual method is synchronized, but compound operations are not atomic. For example, check-then-act patterns like "if (!v.contains(x)) v.add(x)" are not thread-safe without external synchronization because another thread can modify the vector between the contains() and add() calls.
What is the difference between Enumeration and Iterator? ▼
Enumeration (legacy): hasMoreElements()/nextElement(), no remove() support, fail-safe (doesn't throw ConcurrentModificationException). Iterator (modern): hasNext()/next()/remove(), fail-fast (throws CME if collection modified). Always prefer Iterator.

"HashSet is backed by a HashMap internally — elements are stored as keys with a dummy constant value (PRESENT)."

"add(), remove(), and contains() are O(1) on average; worst case O(n) due to hash collisions."

No order

🏗 Architecture — Hash Buckets
💻 Code examples
Example 1 — Add, contains, no duplicates
import java.util.HashSet;
import java.util.Set;
Set<String> fruits = new HashSet<>();
fruits.add("Apple"); fruits.add("Banana");
fruits.add("Apple");  // duplicate — ignored
fruits.add(null);       // one null allowed
System.out.println(fruits.size());          // 3
System.out.println(fruits.contains("Banana")); // true
Example 2 — Custom object — must override hashCode + equals
class Student {
    int id; String name;
    Student(int id, String name) { this.id=id; this.name=name; }
    @Override public int hashCode() { return Objects.hash(id); }
    @Override public boolean equals(Object o) {
        if (!(o instanceof Student)) return false;
        return id == ((Student) o).id;
    }
}
Set<Student> set = new HashSet<>();
set.add(new Student(1, "Riya"));
set.add(new Student(1, "Riya")); // duplicate, not added
System.out.println(set.size()); // 1
Example 3 — Set operations (union, intersection, difference)
Set<Integer> a = new HashSet<>(Set.of(1,2,3,4));
Set<Integer> b = new HashSet<>(Set.of(3,4,5,6));
Set<Integer> union = new HashSet<>(a); union.addAll(b);
Set<Integer> inter = new HashSet<>(a); inter.retainAll(b);
Set<Integer> diff = new HashSet<>(a); diff.removeAll(b);
System.out.println("Union: " + union); // [1,2,3,4,5,6]
Example 4 — De-duplicate a list
List<String> withDups = List.of("A","B","A","C");
Set<String> unique = new HashSet<>(withDups);
System.out.println(unique); // {A, B, C} — order not guaranteed
Example 5 — Load factor and initial capacity
Set<String> optimized = new HashSet<>(64, 0.8f);
// Pre-size: capacity = expectedSize / loadFactor + 1
Set<String> presized = new HashSet<>(100 / 3 * 4 + 1);
✅ Best practices
  • Always override both hashCode() and equals() when storing custom objects — inconsistency leads to silent bugs.
  • Use Objects.hash(field1, field2) for a clean, null-safe hashCode implementation.
  • Pre-size with new HashSet<>(expectedSize * 4 / 3 + 1) to avoid premature rehashing.
  • Declare as Set<T> interface, not HashSet<T>, to keep implementation swappable.
  • Use Set.of() for small, immutable sets — it's more memory-efficient and unmodifiable.
❓ Interview Q&A
How does HashSet detect duplicates? ▼
When you call add(element), HashSet computes element.hashCode() to find the bucket. If the bucket is non-empty, it uses element.equals() to check each existing entry. If equals() returns true for any entry, the element is a duplicate and is not added. This is why both hashCode and equals must be consistently overridden.
What is the load factor in HashSet? ▼
Load factor (default 0.75) determines when the hash table resizes. When size > capacity × loadFactor, the table doubles in size and all elements are rehashed. A lower load factor reduces collisions but wastes memory; a higher load factor saves memory but increases collisions.
What happens if two objects have the same hashCode but different equals? ▼
They both go into the same bucket (hash collision) and are stored as separate entries in the bucket's linked list (or tree if ≥8 entries in Java 8+). Both are retained since equals() distinguishes them. Performance degrades to O(n) for that bucket.
What is the difference between HashSet, LinkedHashSet, and TreeSet? ▼
HashSet: unordered, O(1) ops, fastest. LinkedHashSet: insertion-ordered, O(1) ops, slightly more memory. TreeSet: sorted (natural or Comparator), O(log n) ops, uses Red-Black tree. Choose based on ordering needs.
Is HashSet thread-safe? ▼
No. HashSet is not synchronized. For thread-safe sets: use Collections.synchronizedSet(new HashSet<>()), or ConcurrentHashMap.newKeySet() (better performance), or Collections.newSetFromMap(new ConcurrentHashMap<>()).

  • Override hashCode() and equals() in custom objects — same requirement as HashSet.

❓ Interview Q&A
How does LinkedHashSet maintain insertion order? ▼
It extends LinkedHashMap internally. Each entry node has prev/next pointers forming a doubly-linked list across all buckets. The iteration follows this linked list rather than scanning buckets sequentially, giving consistent insertion-order traversal.
What is the performance cost vs HashSet? ▼
Both are O(1) average for add/remove/contains. LinkedHashSet has slightly higher memory usage (two extra pointers per entry for the doubly-linked list) and marginally slower adds due to linked list pointer maintenance. The difference is usually negligible.
Can LinkedHashSet have null elements? ▼
Yes, exactly one null — same as HashSet. Attempting to add a second null returns false and the null count stays at one.
What happens to insertion order when you remove and re-add an element? ▼
The element moves to the end of the iteration order. Removing it detaches it from the linked list; re-adding it appends it to the tail of the list. The original position is not preserved — this is a common interview trap.
When would you choose TreeSet over LinkedHashSet? ▼
TreeSet when you need sorted order (natural or custom Comparator) and O(log n) operations are acceptable. LinkedHashSet when you need insertion order and O(1) operations. If ordering doesn't matter at all, use HashSet — it's the fastest.

❓ Interview Q&A
What data structure backs TreeSet? ▼
TreeSet is backed by a TreeMap, which itself uses a self-balancing Red-Black tree. Elements are stored as keys in the TreeMap with a dummy constant value. The tree guarantees O(log n) for all basic operations and keeps keys in sorted order at all times.
Why doesn't TreeSet allow null? ▼
TreeSet compares elements during insertion to maintain sorted order. Calling compareTo() or compare() on null throws NullPointerException because null cannot be naturally compared. HashSet/LinkedHashSet allow null because they use hashCode() and equals(), not comparison.
What is the difference between floor() and lower() in TreeSet? ▼
floor(e) returns the greatest element ≤ e (inclusive). lower(e) returns the greatest element strictly < e (exclusive). Similarly, ceiling(e) is ≥ e (inclusive) and higher(e) is strictly > e.
How does TreeSet determine element uniqueness? ▼
TreeSet uses compareTo() (or Comparator.compare()) — NOT equals(). If compareTo() returns 0, the element is considered a duplicate and not added. This means your Comparator must be consistent with equals() otherwise a TreeSet may silently drop elements that are equal() but compare to non-zero.

  • Always override hashCode() and equals() for custom key objects — or unexpected behaviour will occur.
  • Use getOrDefault(), putIfAbsent(), computeIfAbsent(), and merge() to write cleaner, null-safe Map code.
  • Use ConcurrentHashMap instead of Collections.synchronizedMap() for multi-threaded code — better throughput.
  • Pre-size: new HashMap<>(expectedSize * 4 / 3 + 1) to avoid rehashing.
  • Declare as Map<K,V> interface type — not HashMap<K,V> — for flexibility.
❓ Interview Q&A
How does HashMap work internally (put and get)? ▼
put(k,v): compute k.hashCode() → apply bit-mixing → find bucket index (hash & (n-1)) → traverse bucket chain with equals() to find existing key → update if found, append new node if not. get(k): same bucket lookup → return value if key found, null otherwise.
What is treeification in HashMap (Java 8)? ▼
When a single bucket accumulates ≥8 entries (due to hash collisions), Java 8 converts the linked list into a Red-Black tree, changing worst-case lookup from O(n) to O(log n). It untreeifies back to a linked list when the count drops to ≤6.
Why is HashMap not thread-safe? ▼
Multiple threads can simultaneously modify the bucket array — causing data loss, infinite loops (pre-Java 8 rehashing), or incorrect size. The structural modification count (modCount) enables fail-fast iterators but doesn't prevent concurrent modification. Use ConcurrentHashMap for multi-threaded access.
What is the difference between HashMap and Hashtable? ▼
Hashtable: synchronized (every method), no null keys/values, legacy (Java 1.0), slower. HashMap: not synchronized, allows one null key and multiple null values, Java 1.2, faster. Hashtable is legacy — prefer ConcurrentHashMap for thread-safe map operations.
What is the difference between HashMap, LinkedHashMap, and TreeMap? ▼
HashMap: unordered, O(1). LinkedHashMap: insertion-ordered (or access-ordered), O(1). TreeMap: key-sorted (Red-Black tree), O(log n). Choose: HashMap for raw performance, LinkedHashMap for ordered iteration, TreeMap for sorted keys and range queries.

Session 18 — LinkedHashMap Map

LinkedHashMap extends HashMap and maintains a doubly-linked list of all entries to preserve insertion order (default) or access order (LRU cache mode). All HashMap operations at O(1).

What
Ordered key-value map (insertion or access order)
Why
Predictable iteration + LRU cache support
When
Ordered maps, LRU caches, config preservation
⚡ Interview one-liners

"LinkedHashMap maintains a doubly-linked list across all entries to preserve insertion order — unlike HashMap which has no guaranteed order."

Ordering

"In access-order mode (new LinkedHashMap<>(16, 0.75f, true)), the most recently accessed entry moves to the end — enabling LRU eviction."

Access order

"Override removeEldestEntry() to automatically evict the oldest entry — the standard way to build a fixed-size LRU cache."

LRU cache

"All operations are O(1) on average — same as HashMap — with slightly more memory for linked list pointers."

Complexity

"LinkedHashMap allows one null key and multiple null values, same as HashMap."

Null handling
💻 Code examples
Example 1 — Insertion order preserved
import java.util.LinkedHashMap;
Map<String,Integer> lhm = new LinkedHashMap<>();
lhm.put("Banana", 2); lhm.put("Apple", 5); lhm.put("Cherry", 1);
lhm.put("Apple", 9); // update — order unchanged
System.out.println(lhm); // {Banana=2, Apple=9, Cherry=1}
Example 2 — Access order mode
Map<String,Integer> ao = new LinkedHashMap<>(16, 0.75f, true);
ao.put("A", 1); ao.put("B", 2); ao.put("C", 3);
ao.get("A"); // move A to end
System.out.println(ao); // {B=2, C=3, A=1}
Example 3 — LRU Cache with removeEldestEntry
class LRUCache<K,V> extends LinkedHashMap<K,V> {
    private final int maxSize;
    LRUCache(int maxSize) {
        super(16, 0.75f, true);
        this.maxSize = maxSize;
    }
    protected boolean removeEldestEntry(Map.Entry<K,V> eldest) {
        return size() > maxSize;
    }
}
LRUCache<Integer,String> cache = new LRUCache<>(3);
cache.put(1, "One"); cache.put(2, "Two"); cache.put(3, "Three");
cache.get(1); cache.put(4, "Four"); // evicts LRU (key=2)
System.out.println(cache); // {3=Three, 1=One, 4=Four}
Example 4 — vs HashMap order comparison
Map<String,Integer> hm = new HashMap<>();
Map<String,Integer> lhm = new LinkedHashMap<>();
hm.put("C",3); hm.put("A",1); lhm.put("C",3); lhm.put("A",1);
System.out.println("HashMap: " + hm);       // {A=1, C=3} random
System.out.println("LinkedHashMap: " + lhm); // {C=3, A=1} insertion order
Example 5 — Convert to insertion-ordered List of entries
lhm.entrySet().forEach(e -> System.out.println(e.getKey() + " → " + e.getValue()));
✅ Best practices
  • Use access-order mode (true third constructor arg) only when you need LRU semantics — insertion order is the safe default.
  • Always extend LinkedHashMap and override removeEldestEntry() to build a bounded LRU cache — don't implement from scratch.
  • Override hashCode() and equals() in custom key objects — same requirement as HashMap.
  • Use LinkedHashMap when building APIs that must return entries in the order they were added (e.g., configuration, ordered JSON).
  • Declare as Map<K,V> interface unless you specifically need LinkedHashMap's ordering guarantee in the API contract.
❓ Interview Q&A
What is the difference between insertion order and access order in LinkedHashMap? ▼
Insertion order (default): entries iterate in the order they were first added. put() of an existing key does not change its position. Access order (accessOrder=true): each get() or put() moves the accessed entry to the tail of the linked list — making the head always the least recently used (LRU) entry.
How do you implement an LRU cache using LinkedHashMap? ▼
Extend LinkedHashMap with accessOrder=true and override removeEldestEntry() to return true when size() exceeds maxCapacity. The eldest entry (head of linked list = least recently used) is automatically removed. This is the classic interview implementation of an O(1) LRU cache.
Performance comparison: LinkedHashMap vs HashMap? ▼
Both are O(1) average for get/put/remove. LinkedHashMap has slightly higher constant cost because each entry insertion/access must also update two pointer fields in the doubly-linked list. Memory is also slightly higher per entry (~2 extra pointers). The difference is usually negligible for most applications.
Can LinkedHashMap have null keys and values? ▼
Yes. LinkedHashMap inherits HashMap's null handling: one null key is allowed (goes to bucket 0) and multiple null values are allowed. This is unlike TreeMap (no null keys) and ConcurrentHashMap (no null keys or values).

Session 19 — TreeMap Map (Sorted)

TreeMap is a Red-Black tree–backed NavigableMap that keeps entries sorted by key (natural order or custom Comparator). All operations are O(log n). No null keys allowed (values can be null).

What
Sorted key-value Red-Black tree
Why
Sorted keys, range queries, floor/ceiling
When
Sorted maps, phone books, range scans
⚡ Interview one-liners

"TreeMap is backed by a self-balancing Red-Black tree — keys are always kept in sorted order automatically."

Internals

"All operations (get, put, remove, containsKey) are O(log n) — slower than HashMap O(1) but always sorted."

Complexity

"TreeMap does NOT allow null keys — it throws NullPointerException because null cannot be compared."

No null key

"TreeMap provides NavigableMap methods: floorKey(), ceilingKey(), headMap(), tailMap(), subMap(), firstKey(), lastKey()."

Navigation API

"Keys must implement Comparable or a Comparator must be provided — otherwise ClassCastException at runtime."

Comparable required
🏗 Architecture — Red-Black Tree
💻 Code examples
Example 1 — Natural key sort order
import java.util.TreeMap;
Map<String,Integer> tm = new TreeMap<>();
tm.put("Banana", 2); tm.put("Apple", 5); tm.put("Cherry", 1);
System.out.println(tm); // {Apple=5, Banana=2, Cherry=1}
Example 2 — Custom Comparator (reverse key order)
Map<Integer,String> rev = new TreeMap<>(Comparator.reverseOrder());
rev.put(1,"One"); rev.put(3,"Three"); rev.put(2,"Two");
System.out.println(rev); // {3=Three, 2=Two, 1=One}
Example 3 — NavigableMap range methods
TreeMap<Integer,String> tm = new TreeMap<>();
for (int i = 10; i <= 60; i+=10) tm.put(i, "V"+i);
System.out.println(tm.floorKey(35));  // 30
System.out.println(tm.ceilingKey(35)); // 40
System.out.println(tm.headMap(30));     // {10=V10, 20=V20}
Example 4 — headMap, tailMap, subMap
TreeMap<Integer,String> tm = new TreeMap<>(Map.of(1,"a",2,"b",3,"c",4,"d"));
System.out.println(tm.subMap(2, 4)); // {2=b, 3=c}
Example 5 — pollFirstEntry / pollLastEntry
TreeMap<Integer,String> tm = new TreeMap<>(Map.of(10,"A",20,"B",30,"C"));
Map.Entry<Integer,String> min = tm.pollFirstEntry(); // 10=A removed
System.out.println(tm); // {20=B, 30=C}
✅ Best practices
  • Never use null as a TreeMap key — it throws NullPointerException. Only null values are allowed.
  • Use NavigableMap<K,V> as the declared type to expose range query methods to callers.
  • Use subMap(), headMap(), tailMap() for range scans instead of iterating the full map and filtering.
  • Use pollFirstEntry()/pollLastEntry() for priority processing — they remove and return in O(log n).
  • Ensure your Comparator is consistent with equals() — inconsistency causes silent element loss (same issue as TreeSet).
❓ Interview Q&A
What data structure backs TreeMap? ▼
TreeMap uses a self-balancing Red-Black tree. Each node stores a key-value pair plus color (RED/BLACK), and left/right/parent references. The tree auto-balances on every insertion and deletion to guarantee O(log n) height, ensuring O(log n) for all operations.
Why doesn't TreeMap allow null keys? ▼
TreeMap compares keys using compareTo() or Comparator.compare() to position them in the tree. Calling these on null throws NullPointerException. TreeMap can have null values since values are not compared — only keys determine position in the tree.
HashMap vs TreeMap — when to choose which? ▼
HashMap: when order doesn't matter and you need maximum performance — O(1). TreeMap: when you need sorted keys, range queries (headMap, tailMap, subMap), or NavigableMap operations — O(log n). For most applications, HashMap wins on raw speed.
What is the difference between floorKey() and lowerKey()? ▼
floorKey(k) returns the greatest key ≤ k (inclusive). lowerKey(k) returns the greatest key strictly < k (exclusive). Similarly, ceilingKey(k) is ≥ k (inclusive) and higherKey(k) is strictly > k. These four navigation methods are the most common TreeMap interview questions.
Is TreeMap thread-safe? ▼
No. TreeMap is not synchronized. For a thread-safe sorted map use Collections.synchronizedSortedMap(new TreeMap<>()) for coarse-grained locking, or use ConcurrentSkipListMap for highly concurrent access — it provides O(log n) and is fully thread-safe without global locking.
