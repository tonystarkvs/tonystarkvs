Session 01 — The static Keyword

The static keyword makes a field or method belong to the class itself, not to any object. It's loaded once by the JVM when the class is first loaded, and shared across all instances.

What
Class-level member (field, method, block, nested class)
Why
Share data across all instances; utility methods
When
Counters, constants, factory methods
Where
Inside a class, before return type
How
Add static modifier before type
Interview one-liners

"A static member belongs to the class, not to any particular instance of it."

Memory model

"Static variables are stored in the Method Area (Metaspace) and are initialised when the class is loaded."

JVM internals

"You cannot use the this keyword inside a static method because there is no object context."

Common pitfall

"A static block runs exactly once, before any constructor, when the class is first loaded by the ClassLoader."

Execution order

"Overriding is not possible for static methods — it is called method hiding, not overriding."

Polymorphism boundary
Memory layout
Code examples
Example 1 — Static field as object counter
public class Counter {
    // static: belongs to class, shared by all objects
    static int count = 0;
    String name;

    Counter(String name) {
        this.name = name;
        count++; // increments the single shared counter
    }

    public static void main(String[] args) {
        new Counter("Alpha");
        new Counter("Beta");
        new Counter("Gamma");
        System.out.println(Counter.count); // 3
    }
}
Example 2 — Static method (utility)
public class MathUtils {
    public static int square(int n) { return n * n; }

    public static void main(String[] args) {
        // No object needed — called directly on class
        System.out.println(MathUtils.square(5)); // 25
    }
}
Example 3 — Static block (one-time init)
public class Config {
    static String dbUrl;

    // Runs ONCE when class is loaded — before any constructor
    static {
        dbUrl = "jdbc:mysql://localhost:3306/mydb";
        System.out.println("Static block executed");
    }

    public static void main(String[] args) {
        System.out.println(Config.dbUrl);
    }
}
Example 4 — static final constant
public class Circle {
    public static final double PI = 3.14159265;

    public static double area(double r) { return PI * r * r; }

    public static void main(String[] args) {
        System.out.println(Circle.area(5)); // 78.53...
    }
}
Example 5 — Static nested class
public class Outer {
    static int x = 10;

    // Does NOT hold reference to Outer instance
    static class Inner {
        void show() { System.out.println("x = " + x); }
    }

    public static void main(String[] args) {
        new Outer.Inner().show(); // x = 10
    }
}
Best practices
  • Access static members via class name (ClassName.member), never via object reference.
  • Name static final constants in UPPER_SNAKE_CASE.
  • Never store mutable shared state in static fields in multi-threaded apps without synchronisation.
  • Avoid overusing static — it breaks testability and violates OOP.
  • Utility classes with only static methods should have a private constructor.
Interview Q&A
Q1. Can we override a static method in Java?
No. Static methods are resolved at compile time using the reference type — this is called method hiding, not overriding. The subclass can declare a static method with the same name, but polymorphism does not apply.
Q2. Why can't a static method access instance variables?
A static method belongs to the class and can be called without any object existing. Since there may be no object in memory, there is no this reference — so no instance variable to access.
Q3. What is the execution order: static block vs constructor?
Static block runs first — exactly once when the class is loaded. Constructor runs every time a new object is created. For 3 objects: static block runs 1 time, constructor runs 3 times.
Q4. Where are static variables stored in JVM memory?
In the Method Area (called Metaspace from Java 8 onwards). This is separate from the Heap (objects) and Stack (local variables).
Q5. Can a static method be synchronised?
Yes. public static synchronized void method() acquires a lock on the Class object itself, not on any instance. Only one thread can execute that static method at a time, class-wide.

Session 02 — Constructors

A constructor is called automatically when an object is created with new. It initialises the object's state. It has the same name as the class and no return type (not even void).

What
Special method for object initialisation
Why
Ensure every object starts in a valid state
When
Every time new ClassName() is called
Where
Inside the class body
How
Same name as class, no return type
Interview one-liners

"A constructor has the same name as the class, no return type, and is called automatically when an object is created."

Definition

"If you don't write any constructor, the Java compiler provides a no-argument default constructor automatically."

Default constructor

"Constructor chaining with this() allows one constructor to call another inside the same class."

Chaining

"Constructors are not inherited by subclasses, but a subclass constructor can call the parent constructor using super()."

Inheritance link

"A copy constructor creates a new object by copying values from an existing object of the same class."

Copy constructor
Code examples
Example 1 — Default constructor
public class Student {
    String name; int age;

    Student() {           // default: no parameters
        name = "Unknown";
        age  = 0;
    }

    public static void main(String[] args) {
        Student s = new Student();
        System.out.println(s.name + ", " + s.age); // Unknown, 0
    }
}
Example 2 — Parameterised constructor
public class Student {
    String name; int age;

    Student(String name, int age) {
        this.name = name; // 'this' disambiguates param vs field
        this.age  = age;
    }

    public static void main(String[] args) {
        Student s1 = new Student("Riya", 21);
        Student s2 = new Student("Arjun", 23);
        System.out.println(s1.name); // Riya
    }
}
Example 3 — Constructor overloading
public class Rectangle {
    int width, height;

    Rectangle()             { width=1; height=1; }
    Rectangle(int side)      { width=side; height=side; }
    Rectangle(int w, int h) { width=w; height=h; }

    int area() { return width * height; }

    public static void main(String[] args) {
        System.out.println(new Rectangle().area());    // 1
        System.out.println(new Rectangle(5).area());   // 25
        System.out.println(new Rectangle(3,4).area()); // 12
    }
}
Example 4 — Constructor chaining with this()
public class Employee {
    String name; int id; String dept;

    Employee(String name, int id, String dept) {
        this.name=name; this.id=id; this.dept=dept;
    }
    Employee(String name, int id) {
        this(name, id, "General"); // MUST be first statement
    }
    Employee(String name) {
        this(name, 0);
    }
}
Example 5 — Copy constructor
public class Point {
    int x, y;
    Point(int x, int y) { this.x=x; this.y=y; }

    // Copy constructor — clones the object
    Point(Point other) { this.x=other.x; this.y=other.y; }

    public static void main(String[] args) {
        Point p1 = new Point(3, 7);
        Point p2 = new Point(p1); // separate object
        p2.x = 99;
        System.out.println(p1.x); // 3 — unchanged
    }
}
Best practices
  • Always define at least one constructor explicitly in important classes.
  • Use constructor chaining (this()) to avoid duplicated initialisation logic.
  • Validate inputs inside constructors — throw IllegalArgumentException for invalid values.
  • Keep constructors simple; heavy logic belongs in factory methods.
  • For immutable classes, accept all state via constructor and declare all fields final.
Interview Q&A
Q1. What happens if you define a parameterised constructor but no default?
The compiler will NOT provide a default no-arg constructor. Any call like new MyClass() will cause a compile error. You must explicitly define a no-arg constructor if needed.
Q2. Can a constructor be private?
Yes. A private constructor prevents instantiation from outside the class. This is the foundation of the Singleton design pattern and utility classes with only static methods.
Q3. Why must this() be the first statement?
The JVM must complete object initialisation before any other code runs. Having this() first ensures the delegated constructor sets up the object before any statements in the current constructor execute.
Q4. Can constructors be inherited?
No. Constructors are not inherited. A subclass always needs its own constructor. However, it can call a parent constructor using super() as the first line.
Q5. What is the difference between a constructor and a method?
Constructor: same name as class, no return type, called automatically with new. Method: any name, must declare a return type (or void), called explicitly on an object.

Session 03 — The this Keyword

this is a reference variable that refers to the current object. It is available in every non-static context and resolves naming conflicts, enables chaining, and passes the current object as an argument.

What
Reference to the current object
Why
Resolve name conflicts; chain constructors; pass self
When
In any instance method or constructor
Where
Non-static methods and constructors only
How
this.field, this.method(), this(args)
Interview one-liners

"this refers to the current instance and resolves ambiguity between instance variables and constructor parameters."

Core use

"this() is used for constructor chaining within the same class and must always be the first statement."

Constructor chain

"You can use this to pass the current object as an argument to another method."

Method argument

"this cannot be used inside a static method because static methods have no object context."

Limitation

"return this enables method chaining — the foundation of the Builder design pattern."

Builder pattern
Code examples
Example 1 — Disambiguate field vs parameter
public class Person {
    String name; int age;

    // Without 'this', name = name assigns param to itself (bug!)
    Person(String name, int age) {
        this.name = name; // this.name = instance field
        this.age  = age;
    }
}
Example 2 — Constructor chaining
public class Book {
    String title, author; double price;

    Book(String t, String a, double p) {
        title=t; author=a; price=p;
    }
    Book(String t, String a) { this(t, a, 299.0); } // chains up
    Book(String t)            { this(t, "Unknown"); }
}
Example 3 — Pass current object as argument
class Printer {
    void print(Student s) { System.out.println(s.name); }
}
public class Student {
    String name = "Rahul";
    void send() {
        new Printer().print(this); // passes current object
    }
}
Example 4 — Method chaining with return this
public class Builder {
    String name; int age; String city;

    Builder setName(String n) { name=n; return this; }
    Builder setAge(int a)    { age=a;  return this; }
    Builder setCity(String c) { city=c; return this; }

    public static void main(String[] args) {
        Builder b = new Builder()
            .setName("Meera").setAge(28).setCity("Pune");
        System.out.println(b.name + " | " + b.city);
    }
}
Example 5 — Outer.this in inner class
public class Outer {
    int x = 10;
    class Inner {
        int x = 20;
        void show() {
            int x = 30;
            System.out.println(x);               // 30 local
            System.out.println(this.x);        // 20 Inner field
            System.out.println(Outer.this.x);  // 10 Outer field
        }
    }
}
Best practices
  • Always use this.field in constructors and setters when parameter names match field names.
  • Use return this to enable fluent/builder-style APIs.
  • Never call both this() and super() in the same constructor — only one can be first.
  • Use Outer.this explicitly in inner classes to avoid scope confusion.
  • Avoid circular constructor chaining — it causes StackOverflowError.
Interview Q&A
Q1. What are the five uses of the this keyword?
1) Refer to current instance field. 2) Call current class method. 3) Call current class constructor via this(). 4) Pass current object as method argument. 5) Return current object from a method.
Q2. Can we use this in static methods?
No. Static methods belong to the class, not any object. Since this refers to the current object, and no object may exist when a static method runs, using this there is a compile-time error.
Q3. What is the difference between this and super?
this refers to the current class's instance. super refers to the immediate parent class. this() calls a constructor in the same class; super() calls the parent class constructor.
Q4. What happens if this() is not the first statement?
It causes a compile-time error: "call to this must be the first statement in constructor." The JVM enforces this to guarantee that the delegated constructor completes initialisation before any other code runs.
Q5. How does this enable the Builder design pattern?
Each setter returns this (the same object), allowing method calls to be chained: builder.setName("A").setAge(25).build(). This produces readable object construction without telescoping constructors.

Session 04 — Inheritance

Inheritance lets a subclass acquire the properties and behaviours of a superclass using the extends keyword. It promotes code reuse and establishes an IS-A relationship.

What
Subclass reuses and extends parent class members
Why
Code reuse, IS-A modelling, polymorphism base
When
When classes share common attributes and behaviours
Where
class Dog extends Animal
How
extends keyword (single only for classes)
Interview one-liners

"Java supports single inheritance for classes to avoid the diamond problem; multiple inheritance is achieved through interfaces."

Java design

"Inheritance creates an IS-A relationship — a Dog IS-A Animal, so Dog inherits Animal's properties."

IS-A relation

"A subclass inherits all non-private members of its parent class."

Visibility rule

"Every class in Java implicitly extends java.lang.Object — the root of the entire class hierarchy."

Object class

"Constructors are NOT inherited, but a subclass constructor implicitly calls super() if not explicitly stated."

Constructor rule
Code examples
Example 1 — Basic single inheritance
class Animal {
    String name;
    void eat() { System.out.println(name + " is eating"); }
}

class Dog extends Animal { // inherits name and eat()
    String breed;
    void bark() { System.out.println(name + " says: Woof!"); }
}

public class Main {
    public static void main(String[] args) {
        Dog d = new Dog();
        d.name = "Bruno";    // inherited field
        d.eat();  // inherited method — Bruno is eating
        d.bark(); // own method   — Bruno says: Woof!
    }
}
Example 2 — Multilevel inheritance
class Vehicle    { void run()   { System.out.println("Vehicle runs"); } }
class Car extends Vehicle { void drive() { System.out.println("Car drives");   } }
class SportsCar extends Car {
    void race() { System.out.println("SportsCar races"); }
}
// SportsCar has: run() + drive() + race()
Example 3 — Hierarchical inheritance
class Shape { void draw() { System.out.println("Drawing"); } }
class Circle    extends Shape { void area() { System.out.println("Circle area");    } }
class Rectangle extends Shape { void area() { System.out.println("Rectangle area"); } }
Example 4 — instanceof operator
class Animal {} class Dog extends Animal {}
Dog d = new Dog();
System.out.println(d instanceof Dog);    // true
System.out.println(d instanceof Animal); // true — IS-A Animal
System.out.println(d instanceof Object);  // true — root of all
Example 5 — Why multiple inheritance is not allowed for classes
// class C extends A, B {}  ← COMPILE ERROR (diamond problem)
// Use interfaces instead:
interface Flyable   { default void move() { System.out.println("Flying");   } }
interface Swimmable { default void move() { System.out.println("Swimming"); } }

class Duck implements Flyable, Swimmable {
    public void move() { System.out.println("Duck moves"); } // must override
}
Best practices
  • Favour composition over inheritance for HAS-A relationships.
  • Use inheritance only for true IS-A relationships — not just to reuse code.
  • Mark classes final if they should not be subclassed.
  • Always call super() first in a subclass constructor.
  • Override toString(), equals(), and hashCode() from Object when needed.
Interview Q&A
Q1. Why does Java not support multiple inheritance of classes?
To avoid the "diamond problem." If two parent classes have a method with the same name, the child class would be ambiguous about which version to inherit. Java solves this using interfaces with mandatory override when default methods conflict.
Q2. What is the difference between IS-A and HAS-A?
IS-A is inheritance: Dog IS an Animal. HAS-A is composition: a Car HAS-A Engine as a field. Prefer composition — it is more flexible and easier to test than deep inheritance hierarchies.
Q3. Can we access a parent class's private members from a child?
No. Private members are not inherited. A subclass can access them only through public or protected getters/setters defined in the parent class.
Q4. What is the implicit parent of every Java class?
java.lang.Object. Every class extends Object implicitly if no other parent is declared, giving every object toString(), equals(), hashCode(), getClass(), etc.
Q5. What types of inheritance does Java support?
Single (A→B), Multilevel (A→B→C), and Hierarchical (A has multiple children). Java does NOT support multiple inheritance of classes — but achieves it through interfaces.

Session 05 — The super Keyword

super is a reference to the immediate parent class. It accesses parent fields, calls parent methods, and invokes parent constructors from within a subclass.

What
Reference to immediate superclass
Why
Access overridden members; invoke parent constructor
When
In subclass constructors and overriding methods
Where
Non-static subclass methods and constructors
How
super.field, super.method(), super(args)
Interview one-liners

"super refers to the immediate parent class object and is used to call its constructor, methods, or access its fields."

Definition

"super() must be the first statement in a subclass constructor; the compiler inserts it implicitly if omitted."

Constructor rule

"You use super.methodName() when a subclass overrides a method but still needs to execute the parent's version."

Method use

"super cannot be used in a static context, just like this."

Limitation

"super only refers to the immediate parent — you cannot chain it to reach a grandparent directly."

Scope boundary
Code examples
Example 1 — Calling parent constructor
class Animal {
    String name;
    Animal(String name) { this.name = name; }
}
class Dog extends Animal {
    String breed;
    Dog(String name, String breed) {
        super(name);   // MUST be first — calls Animal(String)
        this.breed = breed;
    }
}
Example 2 — Access parent field hidden by child
class Parent { String msg = "Parent"; }
class Child extends Parent {
    String msg = "Child";
    void show() {
        System.out.println(msg);        // Child
        System.out.println(super.msg); // Parent
    }
}
Example 3 — Calling parent's overridden method
class Animal { void sound() { System.out.println("Animal sound"); } }
class Dog extends Animal {
    @Override
    void sound() {
        super.sound(); // run parent logic first
        System.out.println("Dog: Woof");
    }
}
Example 4 — super in multilevel — only reaches immediate parent
class A { void show() { System.out.println("A"); } }
class B extends A  { void show() { System.out.println("B"); } }
class C extends B  {
    void show() {
        super.show(); // calls B.show() — IMMEDIATE parent only
        System.out.println("C");
    }
} // Output: B then C
Example 5 — Implicit super() added by compiler
class Vehicle { Vehicle() { System.out.println("Vehicle"); } }
class Car extends Vehicle {
    Car() {
        // Compiler inserts: super(); as the first line automatically
        System.out.println("Car");
    }
}
// new Car() → Vehicle then Car
Best practices
  • Always call super(args) explicitly when the parent has no no-arg constructor.
  • Use super.method() in overrides to extend (not replace) parent behaviour.
  • Never call both this() and super() in the same constructor.
  • Document why you call super.method() — signals intentional extension.
  • Remember super only reaches the immediate parent — design deep hierarchies carefully.
Interview Q&A
Q1. When does the compiler automatically insert super()?
When a subclass constructor has no explicit super() or this() as its first statement, the compiler inserts super() automatically. If the parent has no no-arg constructor, this causes a compile error.
Q2. Can super() be used anywhere inside a constructor?
No. super() must always be the first statement in a constructor. The JVM requires the parent to be fully initialised before any subclass logic executes.
Q3. What is the difference between super.method() and a normal method call?
A normal method call uses dynamic dispatch — the overridden version in the actual runtime class is invoked. super.method() bypasses dynamic dispatch and directly calls the parent's version regardless of runtime type.
Q4. How is super different from this?
this = current class instance. super = immediate parent class. this() chains to another constructor in the same class; super() calls the parent class constructor.
Q5. Can you use super in a static method?
No. super is an instance concept. Static methods belong to the class, not any object, so there is no valid super reference. It causes a compile-time error.

Session 06 — Polymorphism

Polymorphism means "many forms." A single method name behaves differently based on the object or arguments. Java has two types: compile-time (overloading) and runtime (overriding).

What
One interface, many implementations
Why
Flexibility, extensibility, loose coupling
When
Overload: same class diff params. Override: child redefines parent
Where
Method signatures or inheritance hierarchy
How
Same name + different signatures OR @Override
Interview one-liners

"Method overloading is compile-time polymorphism — the method to call is decided by the compiler based on the argument list."

Overloading

"Method overriding is runtime polymorphism — the JVM decides which method to call based on the actual runtime object type."

Overriding

"Overloading changes the method signature; overriding keeps the same signature but changes the body."

Key difference

"Static, private, and final methods cannot be overridden — only instance methods can."

Overriding rules

"An overriding method cannot throw a broader checked exception than the parent method."

Exception rule
Code examples
Example 1 — Method overloading
public class Calculator {
    int    add(int a, int b)         { return a+b; }
    double add(double a, double b)   { return a+b; }
    int    add(int a, int b, int c)  { return a+b+c; }
    String add(String a, String b)  { return a+b; }
    // compiler picks the correct version from argument types
}
Example 2 — Method overriding (runtime dispatch)
class Animal { void sound() { System.out.println("Generic sound"); } }
class Dog extends Animal {
    @Override void sound() { System.out.println("Woof"); }
}
class Cat extends Animal {
    @Override void sound() { System.out.println("Meow"); }
}
Animal a = new Dog(); a.sound(); // Woof — JVM decides at runtime
a = new Cat();         a.sound(); // Meow
Example 3 — Upcasting and dynamic dispatch
class Shape     { void draw() { System.out.println("Shape");     } }
class Circle    extends Shape { void draw() { System.out.println("Circle");    } }
class Rectangle extends Shape { void draw() { System.out.println("Rectangle"); } }

static void render(Shape s) { s.draw(); } // runtime decides
render(new Circle());    // Circle
render(new Rectangle()); // Rectangle
Example 4 — Overloading vs Overriding at a glance
// OVERLOADING — compile-time (static binding)
void print(int x)    { } // different param type
void print(String s) { }

// OVERRIDING — runtime (dynamic binding)
class Base    { void show() { System.out.println("Base");    } }
class Derived extends Base {
    @Override void show() { System.out.println("Derived"); }
}
Example 5 — Covariant return type
class Animal { Animal get() { return new Animal(); } }
class Dog extends Animal {
    @Override
    Dog get() { return new Dog(); } // narrower return type — valid!
}
Best practices
  • Always use @Override — catches signature mismatches at compile time.
  • Honour the parent's contract in overriding methods (Liskov Substitution Principle).
  • Keep overloaded methods semantically related — don't overload for unrelated purposes.
  • Prefer polymorphic dispatch over if/instanceof chains.
  • An overriding method's access cannot be more restrictive than the parent's.
Interview Q&A
Q1. Compile-time vs runtime polymorphism?
Compile-time = method overloading — compiler picks correct method from argument types. Runtime = method overriding — JVM picks correct method from actual object type via the virtual method table.
Q2. Can you overload by changing only the return type?
No. Overloading depends solely on the method signature (name + parameter list). Changing only the return type does not create a valid overload — it causes a compile error.
Q3. What is the Liskov Substitution Principle?
A subclass object must be usable wherever the parent class is expected, without breaking the program. When overriding, the child must honour the parent's contract — same preconditions, no broader checked exceptions.
Q4. Why can't we override static methods?
Static methods use compile-time dispatch based on reference type, not runtime type. A subclass can declare a static method with the same name — that is method hiding, not overriding, and polymorphism does not apply.
Q5. What is the virtual method table (vtable)?
The JVM maintains a vtable per class — a lookup of method references. When an overriding method is called on a parent reference, the JVM checks the actual object's vtable at runtime and calls the most specific version. This is how runtime polymorphism works internally.

Session 07 — Abstraction

Abstraction means hiding implementation details and exposing only the essential features. Achieved through abstract classes and interfaces — the user sees "what" an object does, not "how."

What
Hide complexity, expose clean contract
Why
Reduce complexity; allow different implementations
When
Designing APIs, frameworks, plug-in systems
Where
Abstract class (partial) or Interface (full contract)
How
abstract keyword; methods with no body
Interview one-liners

"Abstraction hides implementation and exposes only functionality — like a car's steering wheel without exposing the steering mechanism."

Analogy

"Abstract classes provide partial abstraction; interfaces provide 100% abstraction (before Java 8 default methods)."

Degree

"You cannot instantiate an abstract class — it must be subclassed and all abstract methods implemented."

Rule

"Abstraction = what an object does; encapsulation = how it is done and restricting access."

Vs encapsulation

"A concrete subclass of an abstract class must implement all abstract methods, or itself become abstract."

Chain rule
Code examples
Example 1 — Abstract class with abstract and concrete methods
abstract class Shape {
    abstract double area();    // no body — subclasses MUST implement

    void display() {           // concrete — shared for all shapes
        System.out.println("Area = " + area());
    }
}
class Circle extends Shape {
    double r;
    Circle(double r) { this.r = r; }
    @Override double area() { return Math.PI * r * r; }
}
Example 2 — Cannot instantiate abstract class
abstract class Vehicle { abstract void start(); }
// Vehicle v = new Vehicle();  ← COMPILE ERROR
class Car extends Vehicle {
    void start() { System.out.println("Car starts"); }
}
Vehicle v = new Car(); // VALID — upcasting
Example 3 — Abstract class with constructor
abstract class BankAccount {
    String owner; double balance;
    BankAccount(String o, double b) { owner=o; balance=b; }
    abstract void calculateInterest();
    void show() { System.out.println(owner + ": " + balance); }
}
class Savings extends BankAccount {
    Savings(String o, double b) { super(o, b); }
    void calculateInterest() { System.out.println(balance * 0.04); }
}
Example 4 — Abstraction via interface
interface Printable { void print(); }
class Document implements Printable {
    public void print() { System.out.println("Printing doc"); }
}
// Caller knows only Printable — implementation is hidden
Example 5 — Payment abstraction
abstract class Payment {
    double amount;
    Payment(double a) { amount = a; }
    abstract void pay();
    void receipt() { System.out.println("Paid ₹" + amount); }
}
class UPI  extends Payment {
    UPI(double a) { super(a); }
    void pay() { System.out.println("UPI"); receipt(); }
}
Best practices
  • Use abstract classes when multiple classes share common state (fields) and partial behaviour.
  • Use interfaces when you only need a contract — no shared state needed.
  • Design to abstractions, not concretions (Dependency Inversion Principle).
  • Keep abstract APIs minimal — declare only what callers truly need.
  • Don't make a class abstract just to prevent instantiation — use a private constructor for utility classes instead.
Interview Q&A
Q1. Difference between abstraction and encapsulation?
Abstraction = what an object does (hides complexity by showing only relevant features). Encapsulation = how it does it (wraps data + code together, controls access). Abstraction is design-level; encapsulation is implementation-level.
Q2. Can an abstract class have a constructor?
Yes. Abstract class constructors are called via super() from a subclass constructor. Even though you can't instantiate an abstract class directly, the constructor sets up shared state when a concrete subclass object is created.
Q3. Can an abstract class have all concrete methods?
Yes. A class can be declared abstract with no abstract methods — this just prevents direct instantiation. This is valid but rare; it's mostly a design signal.
Q4. What if a concrete subclass doesn't implement all abstract methods?
Compile error — unless the subclass itself is declared abstract. The responsibility then passes down to the next concrete class in the hierarchy.
Q5. How does abstraction support Open/Closed Principle?
By coding against abstract types, you can add new implementations (open for extension) without modifying existing client code (closed for modification). New payment methods can be added without changing the code that processes payments.

Session 08 — Encapsulation

Encapsulation wraps data (fields) and the code that operates on them into a single class, restricting direct outside access using access modifiers like private.

What
Bundle data + methods; restrict access
Why
Data hiding, validation, security, loose coupling
When
Always — every well-designed class
Where
Field declarations + getter/setter methods
How
Fields private; access via public getters/setters
Interview one-liners

"Encapsulation binds data and the methods that operate on it into a single unit and hides internal state from the outside world."

Definition

"A fully encapsulated class has all fields private and provides public getters and setters — also called a POJO or JavaBean."

JavaBean

"Encapsulation provides data hiding, which prevents unauthorised direct access to an object's internal state."

Data hiding

"You can add validation logic inside setters, which is not possible with public fields."

Setter validation

"Making a class read-only (only getters) or write-only (only setters) is also a form of encapsulation."

Read/write control
Code examples
Example 1 — Basic encapsulation
public class Employee {
    private String name;
    private int    age;
    private double salary;

    public String getName()   { return name; }
    public int    getAge()    { return age;  }
    public double getSalary() { return salary; }

    public void setName(String n) { if(n!=null&&!n.isEmpty()) name=n; }
    public void setAge(int a)    { if(a>0&&a<120) age=a; } // validation!
    public void setSalary(double s){ if(s>=0) salary=s; }
}
Example 2 — Read-only / immutable class
public final class ImmutablePoint {
    private final double x, y;

    public ImmutablePoint(double x, double y) { this.x=x; this.y=y; }
    public double getX() { return x; }
    public double getY() { return y; }
    // No setters — immutable like java.lang.String
}
Example 3 — Access modifier summary
public class Demo {
    private int   priv = 1; // only this class
    int           pkg  = 2; // same package
    protected int prot = 3; // same package + subclasses
    public int    pub  = 4; // everywhere
    //           class  pkg  subclass  world
    // private:     ✓     ✗     ✗        ✗
    // default:     ✓     ✓     ✗        ✗
    // protected:   ✓     ✓     ✓        ✗
    // public:      ✓     ✓     ✓        ✓
}
Example 4 — JavaBean pattern
public class Product {
    private int    id;
    private String name;
    private double price;

    public Product() {} // no-arg constructor required by frameworks

    public int    getId()               { return id;  }
    public void   setId(int id)         { this.id=id; }
    public String getName()             { return name; }
    public void   setName(String name)  { this.name=name; }
    public double getPrice()            { return price; }
    public void   setPrice(double p)   { if(p>=0) price=p; }
}
Example 5 — Bad vs good encapsulation
// BAD — public field, no validation
class BadAccount { public double balance; } // anyone can set -999999

// GOOD — encapsulated with business rules
class GoodAccount {
    private double balance;
    public void deposit(double a)  { if(a>0) balance+=a; }
    public boolean withdraw(double a) {
        if(a>0 && balance>=a) { balance-=a; return true; }
        return false; // rule enforced — cannot go negative
    }
    public double getBalance() { return balance; }
}
Best practices
  • Always declare instance fields private — the starting point of good encapsulation.
  • Add validation in setters — never let objects enter an invalid state.
  • Return copies of mutable objects from getters (defensive copy) to prevent external mutation.
  • Use IDE tools to auto-generate getters/setters, then customise validation.
  • Consider Java records (Java 16+) for simple immutable data carriers.
Interview Q&A
Q1. Difference between data hiding and encapsulation?
Encapsulation is the technique (wrapping fields + methods together + access modifiers). Data hiding is the result — private fields can't be accessed directly from outside. Encapsulation enables data hiding.
Q2. Why use getters/setters instead of public fields?
Getters/setters let you add validation, fire change events, make fields read-only, and change internal representation without breaking callers. Public fields expose internals — any change to the field type or name breaks all calling code.
Q3. What is a JavaBean?
A class with: all private fields, a public no-arg constructor, public getters (getXxx) and setters (setXxx), and Serializable. Frameworks like Spring, Hibernate, and JSP rely on this convention for dependency injection and ORM mapping.
Q4. How does encapsulation support loose coupling?
When external code interacts only through a public API, the internal implementation can change freely without affecting callers. This reduces dependencies between classes — the definition of loose coupling.
Q5. Difference between protected and default access?
Default (package-private) allows access only within the same package. Protected allows access within the same package AND from subclasses in any package. Protected is wider — it's designed specifically to support inheritance across packages.

Session 09 — Abstract Classes (Deep Dive)

Declared with the abstract keyword, an abstract class can mix abstract methods (no body) with concrete methods (with body). It cannot be instantiated but serves as a template for subclasses.

What
Partial class — abstract + concrete members mixed
Why
Force subclass to implement; share common code
When
Related classes share state but differ in some behaviours
Where
Base class in hierarchy (Animal, Shape, Account)
How
abstract class + abstract void method();
Interview one-liners

"An abstract class cannot be instantiated but can have constructors called via super() from a subclass."

Constructor

"Abstract classes support the Template Method pattern — define the algorithm skeleton in base class, let subclasses fill in steps."

Design pattern

"A class can extend only one abstract class but can implement multiple interfaces."

Single inheritance

"Abstract classes can have instance variables, static fields, final methods, and even main() — no restrictions compared to concrete classes."

What's allowed

"If a subclass does not implement all abstract methods, it must itself be declared abstract."

Chain rule
Code examples
Example 1 — Template Method Pattern
abstract class DataProcessor {
    final void process() {  // final: subclass cannot change the flow
        readData();
        processData();
        writeData();
    }
    abstract void readData();
    abstract void processData();
    abstract void writeData();
}
class CSVProcessor extends DataProcessor {
    void readData()    { System.out.println("Read CSV");    }
    void processData() { System.out.println("Process CSV"); }
    void writeData()   { System.out.println("Write CSV");   }
}
Example 2 — Abstract class vs interface choice
// Abstract class: has shared state (fields)
abstract class Animal {
    protected String name; // shared — interfaces can't have this
    Animal(String n) { this.name = n; }
    abstract void makeSound();
    void breathe() { System.out.println(name + " breathes"); }
}
// Interface: pure contract, no state
interface Trainable { void train(); }

class Dog extends Animal implements Trainable {
    Dog(String n) { super(n); }
    public void makeSound() { System.out.println("Woof"); }
    public void train()     { System.out.println(name + " trained"); }
}
Example 3 — Abstract class with final and static
abstract class Logger {
    private static int count = 0;

    final void log(String msg) {
        count++;
        System.out.println("[" + getLevel() + "] " + msg);
    }
    abstract String getLevel();
    static int getCount() { return count; }
}
class InfoLogger  extends Logger { String getLevel() { return "INFO";  } }
class ErrorLogger extends Logger { String getLevel() { return "ERROR"; } }
Example 4 — Partial implementation chain
abstract class A { abstract void m1(); abstract void m2(); }
abstract class B extends A {
    void m1() { System.out.println("B implements m1"); }
    // m2 still abstract — deferred to C
}
class C extends B {
    void m2() { System.out.println("C implements m2"); }
}
Example 5 — Anonymous subclass of abstract class
abstract class App {
    abstract String getAppName();

    public static void main(String[] args) {
        App a = new App() { // anonymous subclass
            public String getAppName() { return "MyApp"; }
        };
        System.out.println(a.getAppName()); // MyApp
    }
}
Best practices
  • Use the Template Method pattern when you have a fixed algorithm with variable steps.
  • Mark template methods final to prevent subclasses from breaking the flow.
  • Prefer abstract classes over repeated code — if two classes share 80% of code, factor into an abstract parent.
  • Make abstract class fields protected so subclasses can use them directly.
  • Document which abstract methods are required and what contract each must fulfil.
Interview Q&A
Q1. Abstract class vs interface differences?
Abstract class: constructors, instance fields, concrete + abstract methods, any access modifier, single inheritance only. Interface: no constructors, no instance fields, all fields are public static final, methods public abstract (default/static since Java 8), multiple implementation. Choose abstract class for shared state + IS-A; interface for capability contracts.
Q2. Can an abstract class implement an interface?
Yes. An abstract class can implement an interface without implementing all its methods — leaving some abstract. The first concrete subclass must then implement all remaining abstract methods.
Q3. What is the Template Method pattern?
An abstract class defines the skeleton of an algorithm in a final method, calling abstract methods for the variable parts. Subclasses override the abstract methods to fill in the steps. The flow is fixed; only implementation details vary.
Q4. Why can't we instantiate an abstract class?
Abstract methods have no body. If instantiation were allowed, calling those methods would have nothing to execute. The JVM prevents it at compile time to guarantee every object has complete behaviour.
Q5. Can an abstract class have a final method?
Yes. A final method in an abstract class prevents subclasses from overriding it — useful in the Template Method pattern to lock down the algorithm's structure while allowing individual steps to be customised.

Session 10 — Interfaces

An interface defines a contract — a set of methods a class must implement. Since Java 8, interfaces can also have default and static methods with bodies. Since Java 9, they can have private methods.

What
Pure contract / capability type
Why
Multiple inheritance, loose coupling, plug-in architecture
When
Define what a type can DO, not what it IS
Where
API design, callbacks, strategy pattern, DI
How
interface keyword; class uses implements
Interview one-liners

"An interface is a blueprint of a class — it specifies what methods must be provided, but not how."

Definition

"A class can implement multiple interfaces, achieving multiple inheritance of type without the diamond problem."

Multiple inheritance

"From Java 8, interfaces can have default methods (with body), enabling backward-compatible API evolution."

Java 8

"All fields in an interface are implicitly public, static, and final — effectively constants."

Interface fields

"A functional interface has exactly one abstract method and can be used as a lambda expression target."

Functional interface
Code examples
Example 1 — Basic interface and implementation
interface Drawable {
    void draw();       // implicitly public abstract
    double area();
}
class Circle implements Drawable {
    double r;
    Circle(double r) { this.r=r; }
    public void draw()   { System.out.println("Drawing circle"); }
    public double area() { return Math.PI*r*r; }
}
Drawable d = new Circle(5); // interface reference
Example 2 — Multiple interface implementation
interface Flyable   { void fly();  }
interface Swimmable { void swim(); }
interface Runnable  { void run();  }

// Duck implements THREE interfaces — multiple inheritance of type
class Duck implements Flyable, Swimmable, Runnable {
    public void fly()  { System.out.println("Duck flies");  }
    public void swim() { System.out.println("Duck swims");  }
    public void run()  { System.out.println("Duck runs");   }
}
Example 3 — Default and static methods (Java 8)
interface Vehicle {
    void start(); // abstract

    default void stop() {            // class can override or inherit
        System.out.println("Vehicle stopped");
    }
    static void fuelInfo() {        // called on interface, not object
        System.out.println("Petrol/Diesel/EV");
    }
}
class Car implements Vehicle {
    public void start() { System.out.println("Car started"); }
}
Car c = new Car();
c.start();           // Car started
c.stop();            // Vehicle stopped (default)
Vehicle.fuelInfo();  // static on interface
Example 4 — Functional interface and lambda
@FunctionalInterface
interface Greeting {
    void greet(String name); // exactly ONE abstract method
}
// Lambda implements the single abstract method
Greeting hello   = name -> System.out.println("Hello, "   + name);
Greeting namaste = name -> System.out.println("Namaste, " + name);
hello.greet("Riya");     // Hello, Riya
namaste.greet("Arjun"); // Namaste, Arjun
Example 5 — Interface extending interface
interface Animal   { void breathe(); }
interface Domestic extends Animal   { void interact(); }
interface Pet      extends Domestic { void play(); }

// Must implement ALL three methods: breathe, interact, play
class GoldenRetriever implements Pet {
    public void breathe()  { System.out.println("Breathes");    }
    public void interact() { System.out.println("Interacts");   }
    public void play()     { System.out.println("Plays fetch"); }
}
Best practices
  • Name interfaces after capabilities: Runnable, Comparable, Serializable — not IAnimal.
  • Use @FunctionalInterface for single-abstract-method interfaces to get compiler enforcement.
  • Prefer interfaces over abstract classes for type definitions — allows clients to also extend other classes.
  • Use default methods sparingly — they add implementation to a contract and can cause surprises.
  • In Spring/DI frameworks, always inject interface types (not concrete classes) for loose coupling.
Interview Q&A
Q1. Abstract class vs interface — final comparison?
Abstract class: single inheritance, constructors, instance fields, any access modifier, concrete + abstract methods. Interface: multiple implementation, no constructors, no instance fields, all fields public static final, methods public abstract (default/static Java 8+, private Java 9+). Rule of thumb: abstract class for IS-A with shared state; interface for CAN-DO contracts.
Q2. What is a marker interface?
A marker interface has no methods or fields — it simply tags a class as having a certain property. Examples: Serializable, Cloneable, Remote. The JVM or a framework checks instanceof MarkerInterface at runtime to apply special treatment.
Q3. What happens when two interfaces have the same default method?
The implementing class gets a compile error unless it overrides the conflicting default method. It can call a specific parent's version using InterfaceName.super.method(). This is Java's controlled solution to the diamond problem for interfaces.
Q4. What is a functional interface and how do lambdas relate?
A functional interface has exactly one abstract method. Lambdas provide a concise way to implement it inline. Java 8's java.util.function package provides built-ins: Predicate<T>, Function<T,R>, Consumer<T>, Supplier<T>.
Q5. Can an interface extend multiple interfaces?
Yes. interface C extends A, B {} is valid. A class implementing C must implement all abstract methods from A, B, and C. This is how Java enables a multiple inheritance hierarchy through interfaces without the diamond problem for classes.
