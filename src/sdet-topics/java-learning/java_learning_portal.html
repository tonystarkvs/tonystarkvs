Session 01 — The static Keyword

The static keyword makes a field or method belong to the class itself, not to any individual object. It's loaded once by the JVM, shared across all instances, and accessible without creating an object.

What
Class-level member (field, method, block, nested class)
Why
Share data/behaviour across all instances; utility methods
When
Counters, constants, helper/util methods, factory methods
Where
Inside a class, before or after constructors
How
Add the static modifier before the type or return type
Interview one-liners

"A static member belongs to the class, not to any particular instance of it."

Memory model

"Static variables are stored in the Method Area (Metaspace) and are initialised when the class is loaded."

JVM internals

"You cannot use the this keyword inside a static method because there is no object context."

Common pitfall

"A static block runs exactly once, before any constructor, when the class is first loaded by the ClassLoader."

Execution order

"Overriding is not possible for static methods; it is called method hiding, not overriding."

Polymorphism boundary
Architecture — Memory Layout
Code examples
Example 1 — Static field as object counter
public class Counter {
    // static field: belongs to class, shared by all objects
    static int count = 0;

    // instance field: each object has its own copy
    String name;

    Counter(String name) {
        this.name = name;
        count++; // increments the shared counter
    }

    public static void main(String[] args) {
        new Counter("Alpha");
        new Counter("Beta");
        new Counter("Gamma");
        // Access via class name (preferred) — no object needed
        System.out.println(Counter.count); // Output: 3
    }
}
Example 2 — Static method (utility / helper)
public class MathUtils {
    // static method — no instance needed to call it
    public static int square(int n) {
        return n * n;
    }

    public static void main(String[] args) {
        // Call directly on class — like Math.sqrt() in Java SDK
        System.out.println(MathUtils.square(5)); // 25
    }
}
Example 3 — Static block (one-time initialisation)
public class Config {
    static String dbUrl;

    // static block runs ONCE when class is loaded — before any constructor
    static {
        System.out.println("Static block executed");
        dbUrl = "jdbc:mysql://localhost:3306/mydb";
    }

    public static void main(String[] args) {
        // Output order: Static block → main() body
        System.out.println(Config.dbUrl);
    }
}
Example 4 — Static constant with final
public class Circle {
    // static + final = compile-time constant — never changes, shared globally
    public static final double PI = 3.14159265;

    public static double area(double radius) {
        return PI * radius * radius; // uses constant directly
    }

    public static void main(String[] args) {
        System.out.println(Circle.area(5)); // 78.53981625
    }
}
Example 5 — Static inner class
public class Outer {
    static int x = 10;

    // static nested class: does NOT hold reference to Outer instance
    static class Inner {
        void show() {
            // can access Outer's static members, but NOT instance members
            System.out.println("Outer.x = " + x);
        }
    }

    public static void main(String[] args) {
        // Create Inner without creating Outer first
        Outer.Inner obj = new Outer.Inner();
        obj.show(); // Outer.x = 10
    }
}
Best practices
  • Always access static members via class name (ClassName.member), not via an object reference — avoids confusion.
  • Use static final for constants and name them in UPPER_SNAKE_CASE.
  • Never store mutable shared state in static fields in multi-threaded apps unless synchronised.
  • Avoid overusing static — it breaks testability (hard to mock) and violates OOP principles.
  • Static utility classes (like Math, Collections) should have a private constructor to prevent instantiation.
Interview Q&A
Q1. Can we override a static method in Java?
No. Static methods are resolved at compile time using the reference type, not the actual object. This is called method hiding. If a subclass declares a static method with the same signature, it hides the parent's method, but this is NOT polymorphism.
Q2. Why can't a static method access instance variables?
A static method belongs to the class and can be called without an object. Since there may be no object in memory when it runs, there is no this reference — and therefore no way to know which object's instance variable to access.
Q3. What is the execution order: static block vs constructor?
Static block runs first — exactly once when the class is loaded by the JVM. Constructor runs every time a new object is created. So for 3 objects, the static block runs 1 time and the constructor runs 3 times.
Q4. Where are static variables stored in JVM memory?
Static variables are stored in the Method Area (called Metaspace from Java 8 onwards). This is separate from the Heap (where objects live) and the Stack (where local variables live).
Q5. Can a static method be synchronised in Java?
Yes. public static synchronized void method() acquires a lock on the Class object (the .class file in memory), not on any instance. This means only one thread can execute that static method at a time, class-wide.

Session 02 — Constructors

A constructor is a special method that is automatically called when an object is created using new. It initialises the object's state. It has the same name as the class and no return type (not even void).

What
Special method for object initialisation
Why
Ensure every object starts in a valid, known state
When
Every time new ClassName() is executed
Where
Inside the class, before or after other methods
How
Same name as class, no return type, parameters optional

Chaining

"Constructors are not inherited by subclasses, but a subclass constructor can call the parent constructor using super()."

Code examples
Example 1 — Default constructor
public class Student {
    String name;
    int age;

    // Default constructor: no parameters
    Student() {
        name = "Unknown";
        age = 0;
        System.out.println("Default constructor called");
    }

    public static void main(String[] args) {
        Student s = new Student(); // triggers constructor
        System.out.println(s.name + ", " + s.age); // Unknown, 0
    }
}
Example 2 — Parameterised constructor
public class Student {
    String name;
    int age;

    // Parameterised: caller provides values at object creation time
    Student(String name, int age) {
        this.name = name; // 'this' disambiguates param vs field
        this.age  = age;
    }

    public static void main(String[] args) {
        Student s1 = new Student("Riya", 21);
        Student s2 = new Student("Arjun", 23);
        System.out.println(s1.name); // Riya
        System.out.println(s2.name); // Arjun
    }
}
Example 3 — Constructor overloading
public class Rectangle {
    int width, height;

    Rectangle() { width = 1; height = 1; }           // default square
    Rectangle(int side) { width = side; height = side; } // square by side
    Rectangle(int w, int h) { width = w; height = h; }  // custom shape

    int area() { return width * height; }

    public static void main(String[] args) {
        System.out.println(new Rectangle().area());       // 1
        System.out.println(new Rectangle(5).area());      // 25
        System.out.println(new Rectangle(3,4).area());    // 12
    }
}
Example 4 — Constructor chaining with this()
public class Employee {
    String name; int id; String dept;

    Employee(String name, int id, String dept) {
        this.name = name; this.id = id; this.dept = dept;
    }

    Employee(String name, int id) {
        this(name, id, "General"); // MUST be first statement; calls above
    }

    Employee(String name) {
        this(name, 0); // chains to 2-param constructor
    }

    public static void main(String[] args) {
        Employee e = new Employee("Priya");
        System.out.println(e.name + " | " + e.id + " | " + e.dept);
        // Priya | 0 | General
    }
}
Example 5 — Copy constructor
public class Point {
    int x, y;

    Point(int x, int y) { this.x = x; this.y = y; }

    // Copy constructor: accepts object of same class
    Point(Point other) {
        this.x = other.x;
        this.y = other.y;
    }

    public static void main(String[] args) {
        Point p1 = new Point(3, 7);
        Point p2 = new Point(p1);  // deep copy — separate object
        p2.x = 99;                 // does NOT change p1.x
        System.out.println(p1.x);  // 3 (unchanged)
        System.out.println(p2.x);  // 99
    }
}
Best practices
  • Always define at least one constructor explicitly — relying on the compiler's default is risky in larger classes.
  • Use constructor chaining (this()) to avoid duplicated initialisation logic.
  • Validate inputs inside constructors (throw IllegalArgumentException for invalid values).
  • Keep constructors simple — heavy logic belongs in factory methods or builders.
  • For immutable classes, accept all state via constructor and declare all fields final.
Interview Q&A
Q1. What happens if you define a parameterised constructor but no default constructor?
The compiler will NOT provide a default constructor. Any call like new MyClass() will cause a compile error. You must explicitly define a no-arg constructor if you need one.
Q2. Can a constructor be private?
Yes. A private constructor prevents instantiation from outside the class. This is used in the Singleton design pattern and in utility classes with only static methods.
Q3. What is constructor chaining and why is this() required to be first?
this() delegates to another constructor in the same class. It must be the first statement because the JVM must complete object initialisation before any other code runs — ensuring the object is in a consistent state from the start.
Q4. Can constructors be inherited?
No. Constructors are not inherited in Java. A subclass always needs its own constructor. However, it can call a parent constructor using super() as the first line of its constructor.
Q5. What is the difference between a constructor and a method?
A constructor has the same name as the class, has no return type, and is called automatically with new. A method can have any name, must declare a return type (or void), and is called explicitly on an object.

Session 03 — The this Keyword

this is a reference variable that refers to the current object — the object whose method or constructor is currently being executed. It is available in every non-static context.

What
Reference to the current object in memory
Why
Resolve name conflicts; chain constructors; pass self as argument
When
In any instance method or constructor
Where
Inside non-static methods and constructors
How
Use this.field, this.method(), or this(args)
Interview one-liners

"this refers to the current instance of the class and resolves ambiguity between instance variables and constructor parameters."

Core use

"this() is used for constructor chaining within the same class and must always be the first statement."

Constructor chain

"You can use this to pass the current object as an argument to another method."

Method argument

"this cannot be used inside a static method because static methods have no object context."

Limitation

"return this enables method chaining — the foundation of the Builder design pattern."

Builder pattern
Code examples
Example 1 — Disambiguate field vs parameter
public class Person {
    String name; int age;

    // Without 'this', 'name = name' would assign param to itself (bug!)
    Person(String name, int age) {
        this.name = name; // 'this.name' = instance field, 'name' = param
        this.age  = age;
    }

    void display() {
        System.out.println(this.name + ", " + this.age);
    }
}
Example 2 — Constructor chaining with this()
public class Book {
    String title; String author; double price;

    Book(String title, String author, double price) {
        this.title  = title;
        this.author = author;
        this.price  = price;
    }

    Book(String title, String author) {
        this(title, author, 299.0); // must be first line — delegates upward
    }

    Book(String title) {
        this(title, "Unknown"); // chain continues
    }
}
Example 3 — Pass current object as argument
class Printer {
    void print(Student s) {
        System.out.println("Printing: " + s.name);
    }
}

public class Student {
    String name = "Rahul";

    void send() {
        Printer p = new Printer();
        p.print(this); // passes the current Student object to Printer
    }

    public static void main(String[] args) {
        new Student().send(); // Printing: Rahul
    }
}
Example 4 — Return current object (method chaining)
public class Builder {
    String name; int age; String city;

    // Each setter returns 'this' — enables chained calls
    Builder setName(String name) { this.name = name; return this; }
    Builder setAge(int age)      { this.age  = age;  return this; }
    Builder setCity(String city) { this.city = city; return this; }

    public static void main(String[] args) {
        // Fluent / Builder pattern — each call returns same object
        Builder b = new Builder()
            .setName("Meera")
            .setAge(28)
            .setCity("Pune");
        System.out.println(b.name + " | " + b.city); // Meera | Pune
    }
}
Example 5 — this with inner class
public class Outer {
    int x = 10;

    class Inner {
        int x = 20;

        void show() {
            int x = 30;
            System.out.println(x);              // 30 — local var
            System.out.println(this.x);       // 20 — Inner field
            System.out.println(Outer.this.x); // 10 — Outer field
        }
    }

    public static void main(String[] args) {
        new Outer().new Inner().show();
    }
}
Best practices
  • Always use this.field in constructors and setters when parameter names match field names.
  • Use return this to enable fluent/builder-style APIs in your classes.
  • Never call this() and super() in the same constructor — only one can be first.
  • Use Outer.this explicitly in inner classes to avoid confusion with three levels of scope.
  • Avoid circular constructor chaining — A() calling this(x) which calls this() — it causes StackOverflowError.
Interview Q&A
Q1. What are the five uses of the this keyword?
1) Refer to current instance field. 2) Call current class method. 3) Call current class constructor via this(). 4) Pass current object as method argument. 5) Return current object from a method.
Q2. Can we use this in static methods?
No. Static methods belong to the class, not to any object. Since this refers to the current object, and no object may exist when a static method runs, using this there is a compile-time error.
Q3. What is the difference between this and super?
this refers to the current class's instance. super refers to the immediate parent class. this() calls a constructor in the same class; super() calls the parent class constructor.
Q4. What happens if this() is not the first statement in a constructor?
It causes a compile-time error: "call to this must be the first statement in constructor." The JVM enforces this to guarantee that the delegated constructor completes initialisation before any other code executes.
Q5. How does this enable the Builder design pattern?
Each setter method returns this (the same object), allowing method calls to be chained: builder.setName("A").setAge(25).build(). This produces readable, concise object construction without telescoping constructors.

Session 04 — Inheritance

Inheritance lets a class (child/subclass) acquire the properties and behaviours of another class (parent/superclass) using the extends keyword. It promotes code reuse and establishes an IS-A relationship.

What
Subclass reuses & extends parent class members
Why
Code reuse, IS-A modelling, polymorphism base
When
When classes share common attributes/behaviours
Where
Class declaration: class Dog extends Animal
How
Use extends (single); implement multiple via implements
Interview one-liners

"Java supports single inheritance for classes to avoid the diamond problem, but multiple inheritance is achieved through interfaces."

Java design choice

"Inheritance creates an IS-A relationship — a Dog IS-A Animal, so Dog can inherit Animal's properties."

IS-A relation

"A subclass inherits all non-private members of its parent class."

Visibility rule

"Every class in Java implicitly extends java.lang.Object — the root of the entire class hierarchy."

Object class

"Constructors are NOT inherited, but a subclass constructor implicitly calls super() if not explicitly stated."

Constructor rule
Inheritance hierarchy
Code examples
Example 1 — Basic single inheritance
class Animal {
    String name;
    void eat() { System.out.println(name + " is eating"); }
}

// Dog inherits all non-private members from Animal
class Dog extends Animal {
    String breed; // Dog's own field

    void bark() { System.out.println(name + " says: Woof!"); }
}

public class Main {
    public static void main(String[] args) {
        Dog d = new Dog();
        d.name = "Bruno";  // inherited from Animal
        d.breed = "Labrador";
        d.eat();    // Bruno is eating — inherited method
        d.bark();   // Bruno says: Woof!
    }
}
Example 2 — Multilevel inheritance
class Vehicle  { void run()   { System.out.println("Vehicle runs"); } }
class Car     extends Vehicle { void drive() { System.out.println("Car drives");   } }
class SportsCar extends Car {
    void race() { System.out.println("SportsCar races"); }
}

public class Test {
    public static void main(String[] args) {
        SportsCar sc = new SportsCar();
        sc.run();   // from Vehicle (grandparent)
        sc.drive(); // from Car (parent)
        sc.race();  // own method
    }
}
Example 3 — Hierarchical inheritance
class Shape { void draw() { System.out.println("Drawing shape"); } }

// Multiple classes extend the same parent — hierarchical
class Circle    extends Shape { void area() { System.out.println("Circle area");    } }
class Rectangle extends Shape { void area() { System.out.println("Rectangle area"); } }
Example 4 — IS-A check with instanceof
class Animal {}
class Dog extends Animal {}

public class Test {
    public static void main(String[] args) {
        Dog d = new Dog();
        System.out.println(d instanceof Dog);    // true
        System.out.println(d instanceof Animal); // true — IS-A Animal
        System.out.println(d instanceof Object);  // true — root of all
    }
}
Example 5 — Why multiple inheritance is not allowed for classes
// This would cause a COMPILE ERROR in Java
// class C extends A, B {}   — diamond problem: which parent's method wins?

// Solution: Use interfaces for multiple inheritance
interface Flyable    { default void move() { System.out.println("Flying"); } }
interface Swimmable  { default void move() { System.out.println("Swimming"); } }

class Duck implements Flyable, Swimmable {
    // MUST override conflicting default to resolve ambiguity
    public void move() { System.out.println("Duck moves"); }
}
Best practices
  • Favour composition over inheritance for HAS-A relationships (e.g. Car HAS-A Engine).
  • Use inheritance only for true IS-A relationships — don't extend just to reuse code.
  • Mark classes final if they shouldn't be subclassed (e.g. String, Integer).
  • Always call super() or super(args) first in a subclass constructor to ensure parent initialisation.
  • Override toString(), equals(), and hashCode() from Object when needed.
Interview Q&A
Q1. Why does Java not support multiple inheritance of classes?
To avoid the "diamond problem" — if two parent classes have a method with the same name, the child class would be ambiguous about which version to inherit. Java uses interfaces (with default methods + mandatory override) to achieve multiple inheritance safely.
Q2. What is the difference between IS-A and HAS-A?
IS-A is inheritance: Dog extends Animal means Dog IS an Animal. HAS-A is composition: a Car HAS-A Engine object as a field. Use inheritance sparingly; favour composition because it's more flexible and easier to test.
Q3. Can we access a parent class's private members from a child class?
No. Private members are not inherited. They remain accessible only within the defining class. A subclass can access them indirectly through public or protected getters/setters defined in the parent.
Q4. What is the implicit parent of every Java class?
java.lang.Object. Every class in Java implicitly extends Object if no other parent is declared. This gives every object methods like toString(), equals(), hashCode(), getClass(), clone(), and wait()/notify().
Q5. What types of inheritance does Java support?
Single (A→B), Multilevel (A→B→C), and Hierarchical (A has multiple children). Java does NOT support multiple inheritance of classes — but achieves it through interfaces.

Session 05 — The super Keyword

super is a reference to the immediate parent class of the current object. It's used to access parent class fields, call parent class methods, and invoke parent class constructors.

What
Reference to the immediate superclass
Why
Access overridden methods/fields; invoke parent constructor
When
In subclass constructors and overriding methods
Where
Inside any non-static subclass method or constructor
How
super.field, super.method(), super(args)
Interview one-liners

"super refers to the immediate parent class object — it's used to call the parent's constructor, methods, or access its fields."

Definition

"super() must be the first statement in a subclass constructor; the compiler inserts it implicitly if omitted."

Constructor rule

"You use super.methodName() when a subclass overrides a method but still needs to call the parent's version."

Method use

"super cannot be used in a static context, just like this."

Limitation

"super only refers to the immediate parent — you cannot chain it to reach a grandparent directly."

Scope boundary
Code examples
Example 1 — Calling parent constructor
class Animal {
    String name;
    Animal(String name) {
        this.name = name;
        System.out.println("Animal constructor: " + name);
    }
}

class Dog extends Animal {
    String breed;
    Dog(String name, String breed) {
        super(name);   // must be first — calls Animal(String)
        this.breed = breed;
        System.out.println("Dog constructor: " + breed);
    }
}
// Output: Animal constructor: Bruno → Dog constructor: Lab
Example 2 — Accessing parent field hidden by child
class Parent { String msg = "Hello from Parent"; }

class Child extends Parent {
    String msg = "Hello from Child"; // shadows parent field

    void show() {
        System.out.println(msg);       // Child field
        System.out.println(super.msg); // Parent field
    }
}
Example 3 — Calling parent's overridden method
class Animal {
    void sound() { System.out.println("Animal makes sound"); }
}

class Dog extends Animal {
    @Override
    void sound() {
        super.sound(); // first run parent logic
        System.out.println("Dog says: Woof"); // then extend it
    }
}
// Animal makes sound → Dog says: Woof
Example 4 — super in multilevel hierarchy
class A { void show() { System.out.println("A"); } }
class B extends A { void show() { System.out.println("B"); } }
class C extends B {
    void show() {
        super.show(); // calls B.show() — IMMEDIATE parent only
        System.out.println("C");
    }
}
// Output: B → C   (cannot directly jump to A from C with super)
Example 5 — Implicit super() added by compiler
class Vehicle {
    Vehicle() { System.out.println("Vehicle constructor"); }
}

class Car extends Vehicle {
    Car() {
        // Java compiler silently inserts: super(); as the first line
        System.out.println("Car constructor");
    }
}
// new Car() → Output: Vehicle constructor → Car constructor
Best practices
  • Always call super(args) explicitly when the parent class does not have a no-arg constructor.
  • Use super.method() in overrides to extend (not replace) parent behaviour.
  • Never call this() and super() in the same constructor — only one first-line call is allowed.
  • Document why you're calling super.method() — it signals intentional extension, not accident.
  • Remember super only reaches the immediate parent — design deep hierarchies carefully.
Interview Q&A
Q1. When does the compiler automatically insert super()?
When a subclass constructor does not have an explicit super() or this() as its first statement, the compiler automatically inserts super() — the no-arg parent constructor. If the parent has no no-arg constructor, this causes a compile error.
Q2. Can super() be used anywhere inside a constructor?
No. super() must always be the first statement in a constructor. The JVM requires the parent to be fully initialised before any subclass logic runs.
Q3. What's the difference between super.method() and a normal method call?
A normal method call uses dynamic dispatch (virtual method table) — the overridden version in the actual runtime class is called. super.method() bypasses dynamic dispatch and directly calls the parent's version, regardless of the actual runtime type.
Q4. How is super different from this?
this refers to the current object (current class scope). super refers to the immediate parent class. this() chains to another constructor in the same class; super() calls the parent class constructor.
Q5. Can you use super in a static method?
No. super is an instance concept. Static methods belong to the class, not any object, so there is no "current object" and therefore no valid super reference. Using it in a static context causes a compile-time error.

Session 06 — Polymorphism (Overloading & Overriding)

Polymorphism means "many forms." A single action can behave differently based on the object. Java has two types: compile-time (method overloading) and runtime (method overriding).

What
One interface, many implementations
Why
Flexibility, extensibility, loose coupling
When
Overload: same class, diff params. Override: child redefines parent
Where
Method signatures (overload) / inheritance hierarchy (override)
How
Same method name, different signatures OR @Override annotation
Interview one-liners

"Method overloading is compile-time polymorphism — the method to call is decided by the compiler based on the argument list."

Overloading

"Method overriding is runtime polymorphism — the JVM decides which method to call based on the actual runtime object type."

Overriding

"Overloading changes the method signature; overriding keeps the exact same signature but changes the body."

Key difference

"Static, private, and final methods cannot be overridden — only instance methods can be."

Overriding rules

"An overriding method cannot throw a broader checked exception than the parent method."

Exception rule
Code examples
Example 1 — Method overloading (compile-time)
public class Calculator {
    // Same method name, different parameter list = overloading
    int    add(int a, int b)            { return a + b; }
    double add(double a, double b)    { return a + b; }
    int    add(int a, int b, int c)    { return a + b + c; }
    String add(String a, String b)    { return a + b; }

    public static void main(String[] args) {
        Calculator c = new Calculator();
        System.out.println(c.add(2, 3));          // 5 (int)
        System.out.println(c.add(2.5, 3.5));      // 6.0 (double)
        System.out.println(c.add("Hi", "!"));    // Hi! (String)
    }
}
Example 2 — Method overriding (runtime)
class Animal {
    void sound() { System.out.println("Generic animal sound"); }
}
class Dog extends Animal {
    @Override                           // annotation enforces intent
    void sound() { System.out.println("Woof"); }
}
class Cat extends Animal {
    @Override
    void sound() { System.out.println("Meow"); }
}

public class Test {
    public static void main(String[] args) {
        Animal a = new Dog(); // reference = Animal, object = Dog
        a.sound();             // "Woof" — runtime decides via vtable
        a = new Cat();
        a.sound();             // "Meow"
    }
}
Example 3 — Upcasting & dynamic dispatch
class Shape     { void draw() { System.out.println("Drawing Shape");     } }
class Circle    extends Shape { void draw() { System.out.println("Drawing Circle");    } }
class Rectangle extends Shape { void draw() { System.out.println("Drawing Rectangle"); } }

public class Test {
    static void render(Shape s) {
        s.draw(); // actual method depends on runtime object type
    }
    public static void main(String[] args) {
        render(new Circle());    // Drawing Circle
        render(new Rectangle()); // Drawing Rectangle
    }
}
Example 4 — Overloading vs Overriding comparison
// OVERLOADING: resolved at compile time (static binding)
class Printer {
    void print(int x)    { System.out.println("int: "    + x); }
    void print(String s) { System.out.println("String: " + s); }
}

// OVERRIDING: resolved at runtime (dynamic binding)
class Base    { void show() { System.out.println("Base");  } }
class Derived extends Base {
    @Override
    void show() { System.out.println("Derived"); }
}
Example 5 — Covariant return type in overriding
class Animal {
    Animal getInstance() { return new Animal(); }
}

class Dog extends Animal {
    // Covariant return: child can narrow the return type
    @Override
    Dog getInstance() { return new Dog(); } // Dog IS-A Animal — valid!

Best practices
  • Always use @Override annotation — it catches typos and signature mismatches at compile time.
  • Don't return null from overriding methods when the contract says non-null.
  • Keep overloaded methods doing semantically similar things — don't overload for unrelated purposes.
  • Prefer overriding to conditionals — replace if (type == "dog") with polymorphic dispatch.
  • An overriding method's access cannot be more restrictive than the parent's (public → cannot become protected).
Interview Q&A
Q1. What is the difference between compile-time and runtime polymorphism?
Compile-time (static) polymorphism = method overloading — the compiler resolves the correct method from the argument types. Runtime (dynamic) polymorphism = method overriding — the JVM resolves the correct method from the actual object type via the virtual method table.
Q2. Can you overload methods by changing only the return type?
No. Overloading is determined solely by the method signature (name + parameter list). Changing only the return type does NOT create a valid overload — it causes a compile error.
Q3. What is the Liskov Substitution Principle and how does it relate to overriding?
LSP states a subclass object must be usable wherever the parent class is expected, without breaking the program. When overriding, the child should honour the contract of the parent method — same preconditions, same or stronger postconditions, no broader exceptions.
Q4. Why can't we override static methods?
Static methods belong to the class, not the object. Overriding requires runtime dispatch on the actual object. Static methods use compile-time dispatch based on the reference type. A subclass can define a static method with the same name, but that is called method hiding, not overriding.
Q5. What is the virtual method table (vtable)?
The JVM maintains a vtable for each class — a lookup table of method references. When an overriding method is called on a parent reference, the JVM checks the actual object's vtable at runtime and calls the most specific version. This is how runtime polymorphism is implemented under the hood.

Session 07 — Abstraction

Abstraction means hiding implementation details and showing only the essential features. In Java it is achieved through abstract classes and interfaces. The user interacts with "what" an object does, not "how" it does it.

What
Hide complex implementation, expose clean contract
Why
Reduce complexity; allow different implementations
When
Design APIs, define contracts, build frameworks
Where
Abstract class (partial) or Interface (full contract)
How
abstract keyword; unimplemented methods
Interview one-liners

"Abstraction hides implementation details and exposes only functionality to the user — like a car's steering wheel without exposing the steering mechanism."

Real-world analogy

"Abstract classes provide partial abstraction; interfaces provide 100% abstraction (before Java 8 default methods)."

Degree of abstraction

"You cannot instantiate an abstract class — it must be subclassed and all abstract methods must be implemented."

Instantiation rule

"Abstraction focuses on what an object does; encapsulation focuses on how it is done and restricts access."

Vs encapsulation

"A concrete class that extends an abstract class must implement all abstract methods, or itself become abstract."

Implementation rule
Code examples
Example 1 — Abstract class with abstract and concrete methods
abstract class Shape {
    // abstract method: no body — subclasses MUST implement
    abstract double area();

    // concrete method: shared implementation for all shapes
    void display() {
        System.out.println("Area = " + area());
    }
}

class Circle extends Shape {
    double radius;
    Circle(double r) { this.radius = r; }

    @Override
    double area() { return Math.PI * radius * radius; }
}

class Square extends Shape {
    double side;
    Square(double s) { this.side = s; }

    @Override
    double area() { return side * side; }
}

public class Test {
    public static void main(String[] args) {
        Shape s1 = new Circle(5);
        Shape s2 = new Square(4);
        s1.display(); // Area = 78.53...
        s2.display(); // Area = 16.0
    }
}
Example 2 — Cannot instantiate abstract class

    abstract void start();
}

class Car extends Vehicle {

// Vehicle ref = new Car();  ← VALID (upcasting)

Best practices
  • Use abstract classes when multiple classes share common state (fields) and partial behaviour.
  • Use interfaces when you only need a contract — no shared state needed.
  • Design to abstractions, not concretions — depend on abstract types, not concrete classes (Dependency Inversion Principle).
  • Keep abstract APIs minimal — only declare what callers truly need.
  • Don't make a class abstract just to prevent instantiation — use a private constructor for utility classes.
Interview Q&A
Q1. What is the difference between abstraction and encapsulation?
Abstraction is about what an object does — hiding complexity by showing only relevant features. Encapsulation is about how it does it — wrapping data and code together and controlling access. Abstraction is design-level; encapsulation is implementation-level.
Q2. Can an abstract class have a constructor?
Yes. An abstract class can have a constructor — it's called by subclass constructors via super(). Even though you can't instantiate an abstract class directly, the constructor sets up shared state when a concrete subclass object is created.
Q3. Can an abstract class have all concrete (non-abstract) methods?
Yes, technically. A class can be declared abstract with no abstract methods — this prevents direct instantiation but still allows subclassing. However, this is rare and mostly a design signal. If there are no abstract methods, consider whether an abstract class is really needed.
Q4. What happens if a concrete subclass does not implement all abstract methods?
It causes a compile error — unless the subclass itself is declared abstract. In that case, the responsibility passes down to the next concrete class in the hierarchy.
Q5. How does abstraction support the Open/Closed Principle?
The Open/Closed Principle says a class should be open for extension but closed for modification. By coding against abstract types (abstract classes or interfaces), you can add new implementations (open for extension) without modifying existing client code (closed for modification).

Session 08 — Encapsulation

Encapsulation is the process of wrapping data (fields) and the code (methods) that operates on them into a single unit (class), and restricting direct access to the data using access modifiers.

What
Bundle data + methods; control access via modifiers
Why
Data hiding, validation, loose coupling, security
When
Always — every well-designed class should be encapsulated
Where
Field declarations + getter/setter methods
How
Make fields private; expose via public getters/setters
Interview one-liners

"Encapsulation binds the data and the methods that operate on it into a single unit and hides internal state from the outside world."

Definition

"A fully encapsulated class has all its fields private and provides public getters and setters — it is also called a POJO or Java Bean."

JavaBean

"Encapsulation provides data hiding, which prevents unauthorised direct access to an object's internal state."

Data hiding

"You can add validation logic inside setters, which is not possible with public fields."

Setter validation

"Making a class read-only (only getters, no setters) or write-only (only setters, no getters) is also a form of encapsulation."

Read/write control
Code examples
Example 1 — Basic encapsulation with getters and setters
public class Employee {
    // private fields — hidden from outside world
    private String name;
    private int age;
    private double salary;

    // public getter — controlled read access
    public String getName()   { return name; }
    public int    getAge()    { return age;  }
    public double getSalary() { return salary; }

    // public setter — controlled write access with validation
    public void setName(String name) {
        if (name != null && !name.isEmpty()) this.name = name;
    }
    public void setAge(int age) {
        if (age > 0 && age < 120) this.age = age; // validation!
    }
    public void setSalary(double salary) {
        if (salary >= 0) this.salary = salary;
    }
}
Example 2 — Read-only class (no setters)
public final class ImmutablePoint {
    private final double x;
    private final double y;

    // All state set at construction — never changes
    public ImmutablePoint(double x, double y) {
        this.x = x; this.y = y;
    }

    public double getX() { return x; }
    public double getY() { return y; }
    // No setters — this object is immutable like java.lang.String
}
Example 3 — Access modifiers summary
package com.example;

public class AccessDemo {
    private int   priv   = 1; // only this class
    int           pkg    = 2; // default: same package
    protected int prot   = 3; // same package + subclasses
    public int    pub    = 4; // everywhere
    //  Visibility:  class  package  subclass  world
    //  private:      ✓      ✗        ✗         ✗
    //  default:      ✓      ✓        ✗         ✗
    //  protected:    ✓      ✓        ✓         ✗
    //  public:       ✓      ✓        ✓         ✓
}
Example 4 — JavaBean (standard encapsulation pattern)
// JavaBean: private fields, public no-arg constructor, getters/setters
public class Product {
    private int    id;
    private String name;
    private double price;

    public Product() {} // no-arg constructor required by frameworks

    public int    getId()              { return id;    }
    public void   setId(int id)        { this.id = id; }
    public String getName()            { return name;  }
    public void   setName(String name) { this.name = name; }
    public double getPrice()           { return price; }
    public void   setPrice(double p)   { if (p >= 0) price = p; }
}
Example 5 — Without encapsulation vs with (contrast)
// BAD — public fields, no validation possible
class BadAccount {
    public double balance;   // anyone can set balance = -999999
}

// GOOD — encapsulated with business rule enforced
class GoodAccount {
    private double balance;

    public void deposit(double amount) {
        if (amount > 0) balance += amount;
    }
    public boolean withdraw(double amount) {
        if (amount > 0 && balance >= amount) {
            balance -= amount; return true;
        }
        return false; // insufficient funds — rule enforced
    }
    public double getBalance() { return balance; }
}
Best practices
  • Always declare instance fields private — this is the starting point of good encapsulation.
  • Add validation in setters — never let objects enter an invalid state.
  • Return copies of mutable objects from getters (defensive copy) to prevent external mutation.
  • Use IDE tools (IntelliJ, Eclipse) to auto-generate getters/setters — then customise as needed.
  • Consider Java records (Java 16+) for simple immutable data carriers — they handle encapsulation automatically.
Interview Q&A
Q1. What is the difference between data hiding and encapsulation?
Data hiding is a consequence of encapsulation. Encapsulation is the technique (wrapping fields + methods together); data hiding is the result (private fields can't be accessed directly from outside). Encapsulation enables data hiding, but also provides other benefits like validation and controlled access.
Q2. Why should we use getters and setters instead of public fields?
Getters/setters let you: add validation, fire events on changes, make fields read-only, change internal representation without breaking callers, and debug/log access. Public fields expose internals — any change to the field type or name breaks all calling code.
Q3. What is a JavaBean?
A JavaBean is a class that follows a specific convention: all fields are private, there is a public no-arg constructor, all fields are accessible via public getters (getXxx) and setters (setXxx), and the class is serializable. Frameworks like Spring, Hibernate, and JSP rely on this convention for dependency injection and ORM mapping.
Q4. How does encapsulation support loose coupling?
When external code interacts only through a public API (getters/setters), the internal implementation can change freely without affecting callers. This reduces dependencies between classes — the definition of loose coupling.
Q5. What is the difference between protected and default access?
Default (package-private) allows access only within the same package. Protected allows access within the same package AND from subclasses in any package. So protected has wider access — it's specifically designed to support inheritance across packages.

Session 09 — Abstract Classes (Deep Dive)

An abstract class is a class declared with the abstract keyword. It can have both abstract methods (no body) and concrete methods (with body). It cannot be instantiated but serves as a template for subclasses.

What
Partial class — mix of abstract + concrete members
Why
Force subclass to implement key behaviour; share common code
When
Related classes share state/logic but differ in some behaviours
Where
Base class in a hierarchy (Animal, Shape, Account)
How
abstract class + abstract void method();
Interview one-liners

"An abstract class cannot be instantiated but can have constructors that are called via super() from a subclass."

Constructor

"Abstract classes support the Template Method pattern — define the skeleton of an algorithm in the base class and let subclasses fill in steps."

Design pattern

"A class can extend only one abstract class but can implement multiple interfaces."

Single inheritance limit

"Abstract classes can have instance variables, static fields, final methods, and even main() — none of these restrictions apply to concrete classes."

What's allowed

"If a subclass does not implement all abstract methods, it must itself be declared abstract."

Implementation chain
Code examples
Example 1 — Template Method Pattern with abstract class
abstract class DataProcessor {
    // Template method: defines the algorithm skeleton
    final void process() {  // final — subclasses cannot override the flow
        readData();   // step 1 — subclass defines HOW
        processData(); // step 2
        writeData();  // step 3
    }

    abstract void readData();
    abstract void processData();
    abstract void writeData();
}

class CSVProcessor extends DataProcessor {
    void readData()    { System.out.println("Reading CSV");    }
    void processData() { System.out.println("Processing CSV"); }
    void writeData()   { System.out.println("Writing CSV");    }
}

public class Test {
    public static void main(String[] args) {
        new CSVProcessor().process();
    }
}
Example 2 — Abstract class vs interface decision
// Use abstract class when: shared state (fields) + partial behaviour
abstract class Animal {
    protected String name; // shared state — interfaces can't have this
    protected int age;

    Animal(String n, int a) { this.name = n; this.age = a; }

    abstract void makeSound();  // each animal differs
    void breathe() { System.out.println(name + " breathes"); } // all share
}

// Use interface when: pure contract, no state needed
interface Trainable {
    void train(); // contract only
}

class Dog extends Animal implements Trainable {
    Dog(String n) { super(n, 3); }
    public void makeSound() { System.out.println("Woof"); }
    public void train()     { System.out.println(name + " trained"); }
}
Example 3 — Abstract class with final, static, and concrete methods
abstract class Logger {
    private static int logCount = 0; // static field allowed

    final void log(String msg) {   // final — cannot override logging flow
        logCount++;
        System.out.println("[" + getLevel() + "] " + msg);
    }

    abstract String getLevel();  // subclass defines log level

    static int getLogCount() { return logCount; } // static method allowed
}

class InfoLogger  extends Logger { String getLevel() { return "INFO";  } }
class ErrorLogger extends Logger { String getLevel() { return "ERROR"; } }
Example 4 — Partial implementation chain
abstract class A {
    abstract void m1();
    abstract void m2();
}

// B is abstract — only implements m1, defers m2 to concrete subclass
abstract class B extends A {
    void m1() { System.out.println("B implements m1"); }
    // m2 still abstract — must be implemented by C
}

class C extends B {
    void m2() { System.out.println("C implements m2"); }
    // m1 is inherited from B — no need to re-implement
}
Example 5 — Abstract class with main() method
abstract class App {
    abstract String getAppName();

    // main() can exist in an abstract class
    public static void main(String[] args) {
        App a = new App() {              // anonymous subclass!
            public String getAppName() { return "MyApp"; }
        };
        System.out.println(a.getAppName()); // MyApp
    }
}
Best practices
  • Use the Template Method pattern when you have an algorithm with fixed steps but variable implementations.
  • Mark template methods final in abstract classes to prevent subclasses from breaking the flow.
  • Prefer abstract classes over repeated code — if two classes share 80% of their code, factor it into an abstract parent.
  • Abstract class fields should be protected (not private) so subclasses can use them without going through getters.
  • Always document which abstract methods are required and what contract each must fulfil.
Interview Q&A
Q1. What is the difference between abstract class and interface?
Abstract class can have: constructors, instance fields, concrete + abstract methods, any access modifier — but only single inheritance. Interface can have: constants, abstract methods, default/static methods (Java 8+), no constructors or instance fields — but supports multiple implementation. Choose abstract class for shared state and IS-A; interface for capability contracts.
Q2. Can an abstract class implement an interface?
Yes. An abstract class can implement an interface without implementing all its methods — it can leave some as abstract. The first concrete subclass in the hierarchy must implement all remaining abstract methods.
Q3. What is the Template Method pattern?
It's a behavioural design pattern where an abstract class defines the skeleton of an algorithm in a final method, calling abstract methods for the variable parts. Subclasses override these abstract methods to fill in the steps. The flow is fixed; only the implementation details vary.
Q4. Why can't we instantiate an abstract class directly?
An abstract class has abstract methods — methods with no body. If instantiation were allowed, calling those methods would result in no implementation to execute. The JVM prevents this at compile time to guarantee that any object you create has complete behaviour.
Q5. Can an abstract class have a final method?
Yes. A final method in an abstract class prevents subclasses from overriding it — useful in the Template Method pattern to lock down the algorithm's structure while allowing individual steps to be customised via abstract methods.

Session 10 — Interfaces

An interface is a pure abstract type that defines a contract — a set of methods a class must implement. Since Java 8, interfaces can also have default and static methods with bodies.

What
Contract of capabilities; 100% abstract type
Why
Multiple inheritance, loose coupling, plug-in architecture
When
Define what a type can DO, not what it IS
Where
API design, callbacks, strategy pattern, DI frameworks
How
interface keyword; class uses implements
Interview one-liners

"An interface is a blueprint of a class — it specifies what methods a class must provide, but not how."

Definition

"A class can implement multiple interfaces, achieving multiple inheritance of type without the diamond problem."

Multiple inheritance

"From Java 8, interfaces can have default methods (with body), allowing backward-compatible API evolution."

Java 8 evolution

"All fields in an interface are implicitly public, static, and final — effectively constants."

Interface fields

"A functional interface has exactly one abstract method and can be used as a lambda expression target."

Functional interface
Interface evolution (Java 7 → 9)

Example 1 — Basic interface and implementation

Best practices
  • Name interfaces after capabilities: Runnable, Comparable, Serializable — not IAnimal.
  • Use @FunctionalInterface annotation when creating single-abstract-method interfaces to make intent clear and get compiler enforcement.
  • Prefer interfaces over abstract classes for type definitions — they allow clients to also extend other classes.
  • Use default methods sparingly — they add implementation to a contract and can cause unexpected behaviour if overused.
  • In Spring/DI frameworks, always inject interface types (not concrete classes) to keep components loosely coupled.
Interview Q&A
Q1. What is the difference between abstract class and interface? (final answer)
Abstract class: single inheritance, can have constructors + instance fields + any access modifier + concrete methods. Interface: multiple implementation, no constructors, no instance fields, all fields are public static final, methods are public abstract (or default/static from Java 8). Use abstract class for IS-A with shared state; interface for CAN-DO contracts and multiple inheritance of type.
Q2. What is a marker interface?
A marker interface has no methods or fields — it simply marks a class as having a certain property. Examples: Serializable, Cloneable, Remote. The JVM or a framework checks instanceof MarkerInterface at runtime to determine special treatment.
Q3. What happens when two interfaces have the same default method?
The implementing class gets a compile error unless it overrides the conflicting default method. This is Java's solution to the diamond problem for interfaces. The class must provide its own implementation, optionally calling one of the parent defaults with InterfaceName.super.method().
Q4. What is a functional interface and how does it relate to lambdas?
A functional interface has exactly one abstract method. Lambda expressions provide a concise way to implement it inline. Java 8's java.util.function package provides many built-in ones: Predicate<T>, Function<T,R>, Consumer<T>, Supplier<T>, BiFunction<T,U,R>.
Q5. Can an interface extend multiple interfaces?
Yes. An interface can extend multiple interfaces: interface C extends A, B {}. A class implementing C must implement all abstract methods from A, B, and C. This is how Java enables a form of multiple inheritance hierarchy through interfaces.
