Session 1 of 10 · Fundamentals
What is OOP? — The Big Picture
Understand why OOP exists, how it maps to the real world, and how it powers test automation
Why OOP?
Simple definition: OOP is a programming style where you model your code like real-world objects — things that have properties (data) and behaviors (actions). Instead of writing a long list of instructions, you group related data and actions together into a "class".
🏗️

Organised

Group related code into classes. No messy global variables.

♻️

Reusable

Write once, inherit everywhere. BasePage → all page classes.

🔒

Secure

Hide internal data. Only expose what others need.

🔧

Maintainable

Change one class without breaking others.

📦

Modular

Each class is a self-contained unit. Easy to test and debug.

🌍

Real-world fit

Browser, Page, Element — all natural OOP objects.

Session 2 of 10 · Core Concepts
Classes & Objects
Blueprint vs instance — the foundation of every Java program
Key concepts
Class = blueprint/template (no memory allocated). Object = instance of a class (memory allocated, has real values). Think: Class = cookie cutter 🍪, Object = the actual cookie.
Code examples
// Class definition — the blueprint
public class LoginPage {

    // Fields (properties)
    String pageTitle = "Login";
    String url       = "https://app.com/login";

    // Method (behaviour)
    public void printInfo() {
        System.out.println("Title: " + pageTitle);
        System.out.println("URL: " + url);
    }
}

// Creating objects (instances)
public class Main {
    public static void main(String[] args) {

        LoginPage page1 = new LoginPage();  // Object 1
        LoginPage page2 = new LoginPage();  // Object 2

        page1.url = "https://staging.app.com";
        page2.url = "https://prod.app.com";

        page1.printInfo();  // prints staging URL
        page2.printInfo();  // prints prod URL
    }
}
// Constructor — called automatically when object is created
public class BrowserConfig {

    String browserName;
    String environment;
    int    timeout;

    // Default constructor (no args)
    public BrowserConfig() {
        browserName = "chrome";
        environment = "staging";
        timeout     = 30;
    }

    // Parameterised constructor
    public BrowserConfig(String browser, String env, int t) {
        this.browserName = browser;
        this.environment = env;
        this.timeout     = t;
    }
}

// Usage
BrowserConfig cfg1 = new BrowserConfig();
BrowserConfig cfg2 = new BrowserConfig("firefox", "prod", 60);
// Selenium real-world: Page as a class, driver as object
public class LoginPage {

    private WebDriver driver;  // driver = object of WebDriver

    // Locators as class-level fields
    private By txtUsername = By.id("username");
    private By txtPassword = By.id("password");
    private By btnLogin    = By.id("loginBtn");

    // Constructor injects the driver
    public LoginPage(WebDriver driver) {
        this.driver = driver;
    }

    // Methods = page actions
    public void enterUsername(String user) {
        driver.findElement(txtUsername).sendKeys(user);
    }

    public void enterPassword(String pass) {
        driver.findElement(txtPassword).sendKeys(pass);
    }

    public void clickLogin() {
        driver.findElement(btnLogin).click();
    }

    // Chain actions in one method
    public void login(String user, String pass) {
        enterUsername(user);
        enterPassword(pass);
        clickLogin();
    }
}

// In test class
WebDriver driver = new ChromeDriver();
LoginPage loginPage = new LoginPage(driver);  // Object!
loginPage.login("admin", "pass123");
Common mistakes & fixes

Watch out for these

Wrong

Declaring WebDriver as static in page classes — causes parallel test failures because all threads share the same driver instance.

Fix

Use instance-level driver: private WebDriver driver; and inject via constructor.

Wrong

Creating a new driver object inside the page class: driver = new ChromeDriver(); inside page constructor.

Fix

Always pass the driver from outside (dependency injection). The page should never own the driver lifecycle.

Q1

Q2

"In POM, every page is a separate class. When the test runs, an object of that page class is created, holding the WebDriver reference."

Q3

"A constructor initialises an object's state. In Selenium page classes, we use a parameterised constructor to inject the WebDriver dependency."

Session 3 of 10 · Encapsulation
Encapsulation — Protect Your Data
private fields + public getters/setters = controlled, validated access
What & why
Encapsulation = wrapping data (fields) and code (methods) together inside a class, and restricting direct outside access using access modifiers. It's like a medicine capsule — ingredients are hidden inside, you only interact with the outer shell.
🔴

private

Accessible only inside the same class. Most restrictive.

🟡

protected

Accessible inside same class + subclasses + same package.

🟢

public

Accessible from anywhere. Least restrictive.

⚪

default

No keyword = accessible within same package only.

Code examples
// Encapsulated class
public class TestUser {

    private String username;  // hidden
    private String password;  // hidden
    private String role;      // hidden

    // Getter — read access
    public String getUsername() { return username; }
    public String getRole()     { return role; }

    // Setter — write access
    public void setUsername(String u) { this.username = u; }
    public void setRole(String r)     { this.role = r; }

    // No getter for password — can't be read!
    public void setPassword(String p)  { this.password = p; }
}

// Usage
TestUser user = new TestUser();
user.setUsername("admin");
user.setPassword("secret");
System.out.println(user.getUsername()); // OK
// System.out.println(user.password); // ERROR — private!
// Validation inside setter — prevents bad data
public class TestConfig {

    private String baseUrl;
    private int    timeout;
    private String browser;

    public void setBaseUrl(String url) {
        if (url == null || !url.startsWith("https://")) {
            throw new IllegalArgumentException(
                "URL must start with https://");
        }
        this.baseUrl = url;
    }

    public void setTimeout(int seconds) {
        this.timeout = (seconds > 0) ? seconds : 30;
    }

    public void setBrowser(String b) {
        String[] valid = {"chrome", "firefox", "edge"};
        for (String v : valid) {
            if (v.equals(b)) { this.browser = b; return; }
        }
        throw new IllegalArgumentException("Unsupported: " + b);
    }

    public String getBaseUrl() { return baseUrl; }
    public int getTimeout()    { return timeout; }
    public String getBrowser()  { return browser; }
}
// Encapsulation in Selenium page class
public class CheckoutPage {

    // Private — no test class can directly access locators
    private WebDriver driver;
    private By cartTotal   = By.id("cart-total");
    private By btnCheckout = By.id("checkout-btn");
    private By orderNum    = By.id("order-number");

    public CheckoutPage(WebDriver d) { this.driver = d; }

    // Public methods — controlled interface for tests
    public String getCartTotal() {
        return driver.findElement(cartTotal).getText();
    }

    public void clickCheckout() {
        driver.findElement(btnCheckout).click();
    }

    public String getOrderNumber() {
        return driver.findElement(orderNum).getText();
    }
}

// Test class ONLY calls public methods
// It has no idea HOW locators work — only WHAT the page can do
CheckoutPage checkout = new CheckoutPage(driver);
checkout.clickCheckout();
String orderId = checkout.getOrderNumber();
Best practices checklist
  • Always declare fields as private in page and config classes
  • Provide getters only for fields that need to be read externally
  • Add validation logic inside setters, not in the test class
  • Never make locators public — they are implementation details
  • Use protected for fields that child page classes need to access
  • Driver should always be private in page classes, injected via constructor

Interview one-liners

Q1

"Encapsulation bundles data and methods together and restricts direct access using private fields with public getters/setters."

Q2

"In Selenium POM, locators are private fields — tests can't directly access them and must go through public methods, keeping internal details hidden."

Q3

"Encapsulation allows validation in setters — for example, ensuring a base URL always starts with https:// before storing it."

Session 4 of 10 · Inheritance
Inheritance — Reuse Without Rewriting
extends keyword · parent/child classes · super() · method overriding
Inheritance hierarchy in Selenium framework
Code examples
// Parent class — shared utilities for ALL pages
public class BasePage {

    protected WebDriver     driver;  // protected = child can use
    protected WebDriverWait wait;

    public BasePage(WebDriver driver) {
        this.driver = driver;
        this.wait   = new WebDriverWait(driver,
                        Duration.ofSeconds(10));
    }

    public void clickElement(By locator) {
        wait.until(ExpectedConditions
            .elementToBeClickable(locator)).click();
    }

    public String getText(By locator) {
        return driver.findElement(locator).getText();
    }

    public void enterText(By locator, String text) {
        driver.findElement(locator).clear();
        driver.findElement(locator).sendKeys(text);
    }

    public boolean isDisplayed(By locator) {
        return driver.findElement(locator).isDisplayed();
    }
}
// Child class — extends BasePage, inherits everything
public class LoginPage extends BasePage {

    private By txtUser  = By.id("username");
    private By txtPass  = By.id("password");
    private By btnLogin = By.id("loginBtn");
    private By errMsg   = By.id("errorMessage");

    // super() calls BasePage constructor
    public LoginPage(WebDriver driver) {
        super(driver);
    }

    // Uses inherited enterText() and clickElement()
    public DashboardPage login(String user, String pass) {
        enterText(txtUser, user);      // inherited!
        enterText(txtPass, pass);      // inherited!
        clickElement(btnLogin);        // inherited!
        return new DashboardPage(driver);
    }

    public String getErrorMessage() {
        return getText(errMsg);         // inherited!
    }

    public boolean isErrorDisplayed() {
        return isDisplayed(errMsg);    // inherited!
    }
}
// Method Overriding — child replaces parent behaviour
public class BasePage {
    public String getPageName() {
        return "Unknown Page";  // default
    }

    public void logNavigation() {
        System.out.println("Navigated to: " + getPageName());
    }
}

public class LoginPage extends BasePage {

    @Override
    public String getPageName() {
        return "Login Page";  // overrides parent
    }
}

public class DashboardPage extends BasePage {

    @Override
    public String getPageName() {
        return "Dashboard Page";  // overrides parent
    }
}

// Using super to call parent method AND add child logic
public class AdminPage extends BasePage {
    @Override
    public String getPageName() {
        return super.getPageName() + " [Admin]";
    }
}

Common mistakes

Wrong

Forgetting super(driver) in child class constructor — causes NullPointerException because parent fields are never initialised.

Fix

Always call super(driver); as the FIRST line of every child class constructor.

Wrong

Using private for driver in BasePage — child classes can't access it and will get compile error.

Fix

Use protected WebDriver driver in BasePage so all child page classes can access it directly.

Interview one-liners

Q1

"Inheritance lets a child class reuse fields and methods of a parent class using the extends keyword, avoiding code duplication."

Q2

"In our framework, all page classes extend BasePage to inherit driver, wait, clickElement(), and getText() — no need to rewrite these 50+ times."

Q3

"Method overriding lets a child class redefine a parent method. We use @Override annotation to catch mistakes at compile time."

Session 5 of 10 · Abstraction
Abstraction — Hide Complexity, Expose Simplicity
abstract classes · abstract methods · hiding HOW, showing WHAT
What & why
Abstraction = hiding implementation details and only exposing what's necessary. Like driving a car — you press the accelerator (interface), you don't need to know how the fuel injection works (implementation).
Code examples
// Abstract class — cannot be instantiated directly
public abstract class BaseTest {

    protected WebDriver driver;

    // Concrete method — same for all test classes
    @BeforeMethod
    public void setUp() {
        driver = DriverFactory.getDriver();
        driver.manage().window().maximize();
        driver.manage().timeouts()
            .implicitlyWait(Duration.ofSeconds(10));
        System.out.println("Browser started: " + getBrowserName());
    }

    @AfterMethod
    public void tearDown() {
        if (driver != null) driver.quit();
    }

    // Abstract method — MUST be implemented by child
    public abstract String getBrowserName();
    public abstract String getTestSuiteName();
}

// Cannot do: new BaseTest() — compile error!
// Concrete child class — must implement ALL abstract methods
public class LoginTest extends BaseTest {

    @Override
    public String getBrowserName() {
        return "Chrome";
    }

    @Override
    public String getTestSuiteName() {
        return "Login Regression Suite";
    }

    @Test
    public void testValidLogin() {
        // setUp() from BaseTest already launched browser!
        LoginPage page = new LoginPage(driver);
        page.login("admin", "pass123");
        Assert.assertTrue(
            driver.getCurrentUrl().contains("/dashboard"));
    }
}

// LoginTest has no idea how driver was set up
// It just uses 'driver' — complexity is abstracted!
// DriverFactory — abstracts browser creation complexity
public class DriverFactory {

    // ThreadLocal for parallel execution safety
    private static ThreadLocal<WebDriver> driver
        = new ThreadLocal<>();

    public static WebDriver getDriver() {
        if (driver.get() == null) {
            String browser = System.getProperty(
                "browser", "chrome");

            switch (browser.toLowerCase()) {
                case "chrome":
                    driver.set(new ChromeDriver()); break;
                case "firefox":
                    driver.set(new FirefoxDriver()); break;
                case "edge":
                    driver.set(new EdgeDriver());   break;
                default:
                    throw new IllegalArgumentException(
                        "Unknown browser: " + browser);
            }
        }
        return driver.get();
    }

    public static void quitDriver() {
        if (driver.get() != null) {
            driver.get().quit();
            driver.remove();
        }
    }
}

// Test class just calls: DriverFactory.getDriver()
// It never knows about ChromeDriver setup internals!
Abstract class vs Interface — quick comparison
📐

Abstract class

Can have both abstract and concrete methods. Has constructors. Use when sharing common state (fields like driver).

📋

Interface

All methods are abstract (Java 7). No instance fields. Use when defining a contract multiple unrelated classes must follow.

Q1

Q2

Q3

"You can't instantiate an abstract class directly — it exists purely as a template for subclasses to build upon."

Session 6 of 10 · Polymorphism
Polymorphism — One Name, Many Forms
method overloading · method overriding · runtime vs compile-time polymorphism
Two types of polymorphism
⚙️

Compile-time (Overloading)

Same method name, different parameters. Decided at compile time. Example: click(By), click(WebElement).

🔄

Runtime (Overriding)

Child class redefines parent method. Decided at runtime. Example: generateReport() in HTML vs Allure reporter.

Code examples
// Method Overloading — same name, different signatures
public class BasePage {

    // Version 1 — click by locator
    public void click(By locator) {
        driver.findElement(locator).click();
    }

    // Version 2 — click by WebElement
    public void click(WebElement element) {
        element.click();
    }

    // Version 3 — click with custom wait
    public void click(By locator, int waitSeconds) {
        WebDriverWait customWait = new WebDriverWait(
            driver, Duration.ofSeconds(waitSeconds));
        customWait.until(ExpectedConditions
            .elementToBeClickable(locator)).click();
    }

    // Version 4 — type text with or without clear
    public void type(By locator, String text) {
        driver.findElement(locator).sendKeys(text);
    }

    public void type(By locator, String text, boolean clear) {
        if (clear) driver.findElement(locator).clear();
        driver.findElement(locator).sendKeys(text);
    }
}

// Java decides WHICH click() to call at COMPILE TIME
// Method Overriding — child redefines parent's method
public class BaseReporter {
    public void logTestResult(String test, String status) {
        System.out.println(test + " -> " + status);
    }
}

public class ExtentReporter extends BaseReporter {
    @Override
    public void logTestResult(String test, String status) {
        // Writes to Extent HTML report
        extentTest.log(Status.PASS, test + ": " + status);
    }
}

public class AllureReporter extends BaseReporter {
    @Override
    public void logTestResult(String test, String status) {
        // Attaches to Allure report
        Allure.step(test + ": " + status);
    }
}

// Polymorphic reference — same code, different behaviour
BaseReporter reporter;

reporter = new ExtentReporter();
reporter.logTestResult("Login", "PASS"); // Extent runs

reporter = new AllureReporter();
reporter.logTestResult("Login", "PASS"); // Allure runs
// Real-world: polymorphic reporter list
public interface Reportable {
    void generateReport(String testName, String status);
}

public class HtmlReporter implements Reportable {
    @Override
    public void generateReport(String name, String status) {
        System.out.println("[HTML] " + name + "=" + status);
    }
}

public class SlackReporter implements Reportable {
    @Override
    public void generateReport(String name, String status) {
        System.out.println("[Slack] " + name + "=" + status);
    }
}

// One loop, multiple behaviours — true polymorphism!
List<Reportable> reporters = Arrays.asList(
    new HtmlReporter(),
    new SlackReporter()
);

for (Reportable r : reporters) {
    r.generateReport("Login Test", "PASSED");
    // Each object runs its OWN generateReport()
}

Session 7 of 10 · Interfaces
Interfaces — Contracts in Code
implements · default methods · multiple interface implementation
What is an interface?
Interface = a contract. It says "any class that implements me MUST provide these methods." It defines WHAT a class must do, not HOW it does it. A class can implement multiple interfaces (Java doesn't allow multiple inheritance of classes).
Code examples
// Interface — defines a contract
public interface Navigable {
    void navigateTo(String url);
    void goBack();
    void refresh();

    // Default method — optional to override (Java 8+)
    default void navigateAndWait(String url, int ms) {
        navigateTo(url);
        try { Thread.sleep(ms); }
        catch (InterruptedException e) {}
    }
}

// Class implementing the interface
public class BrowserActions implements Navigable {

    private WebDriver driver;

    public BrowserActions(WebDriver d) { driver = d; }

    @Override
    public void navigateTo(String url) {
        driver.get(url);
    }

    @Override
    public void goBack() {
        driver.navigate().back();
    }

    @Override
    public void refresh() {
        driver.navigate().refresh();
    }
}
// Multiple interfaces — Java allows this!
public interface Loggable {
    void logInfo(String message);
    void logError(String error);
}

public interface Screenshottable {
    void takeScreenshot(String fileName);
}

public interface Waitable {
    void waitForElement(By locator, int seconds);
    void waitForUrl(String url, int seconds);
}

// One class, three interface contracts
public class BasePage
        implements Loggable, Screenshottable, Waitable {

    protected WebDriver driver;

    @Override
    public void logInfo(String msg) {
        System.out.println("[INFO] " + msg);
    }

    @Override
    public void logError(String err) {
        System.err.println("[ERROR] " + err);
    }

    @Override
    public void takeScreenshot(String fileName) {
        File src = ((TakesScreenshot) driver)
            .getScreenshotAs(OutputType.FILE);
        // copy src to screenshots/fileName.png
    }

    @Override
    public void waitForElement(By locator, int secs) {
        new WebDriverWait(driver, Duration.ofSeconds(secs))
            .until(ExpectedConditions
                .visibilityOfElementLocated(locator));
    }

    @Override
    public void waitForUrl(String url, int secs) {
        new WebDriverWait(driver, Duration.ofSeconds(secs))
            .until(ExpectedConditions.urlContains(url));
    }
}

Page Object Model — OOP in Action
Full POM implementation using all 4 OOP pillars together
POM architecture diagram
Full POM example
// BasePage.java — Abstraction + Encapsulation
public abstract class BasePage {

    protected WebDriver     driver;
    protected WebDriverWait wait;

    public BasePage(WebDriver driver) {
        this.driver = driver;
        this.wait   = new WebDriverWait(
            driver, Duration.ofSeconds(15));
        PageFactory.initElements(driver, this);
    }

    protected void click(By by) {
        wait.until(ExpectedConditions
            .elementToBeClickable(by)).click();
    }

    protected void type(By by, String text) {
        WebElement el = wait.until(ExpectedConditions
            .visibilityOfElementLocated(by));
        el.clear();
        el.sendKeys(text);
    }

    protected String getText(By by) {
        return wait.until(ExpectedConditions
            .visibilityOfElementLocated(by)).getText();
    }

    protected boolean isVisible(By by) {
        try {
            return driver.findElement(by).isDisplayed();
        } catch (NoSuchElementException e) {
            return false;
        }
    }

    // Abstract — each page must say who it is
    public abstract String getPageTitle();
}
// LoginPage.java — Inheritance + Encapsulation
public class LoginPage extends BasePage {

    // Private locators — encapsulation
    private final By USERNAME  = By.id("username");
    private final By PASSWORD  = By.id("password");
    private final By LOGIN_BTN = By.id("loginBtn");
    private final By ERROR_MSG = By.id("errorMsg");

    public LoginPage(WebDriver driver) {
        super(driver);  // calls BasePage constructor
    }

    @Override
    public String getPageTitle() {
        return driver.getTitle();
    }

    // Business-level methods for tests to call
    public DashboardPage loginWithValidCredentials(
            String user, String pass) {
        type(USERNAME, user);        // inherited from BasePage
        type(PASSWORD, pass);
        click(LOGIN_BTN);
        return new DashboardPage(driver);
    }

    public void loginWithInvalidCredentials(
            String user, String pass) {
        type(USERNAME, user);
        type(PASSWORD, pass);
        click(LOGIN_BTN);
        // no return — stays on login page
    }

    public String getErrorMessage() {
        return getText(ERROR_MSG);   // inherited
    }

    public boolean isErrorDisplayed() {
        return isVisible(ERROR_MSG); // inherited
    }
}
// LoginTest.java — Uses all 4 pillars together
public class LoginTest extends BaseTest {  // Inheritance

    private LoginPage loginPage;  // Encapsulation

    @BeforeMethod
    public void initPage() {
        loginPage = new LoginPage(driver);
        driver.get("https://app.com/login");
    }

    @Test
    public void validLoginRedirectsToDashboard() {
        DashboardPage dash =
            loginPage.loginWithValidCredentials(  // Abstraction
                "admin", "pass123");
        Assert.assertTrue(
            dash.isWelcomeMessageDisplayed(),
            "Welcome message not shown after login");
    }

    @Test
    public void invalidLoginShowsError() {
        loginPage.loginWithInvalidCredentials(
            "bad", "wrong");
        Assert.assertTrue(
            loginPage.isErrorDisplayed(),
            "Error message not shown");
        Assert.assertEquals(
            loginPage.getErrorMessage(),
            "Invalid username or password");
    }
}

ConfigReader — encapsulation + singleton
// Singleton + Encapsulation — reads config once
public class ConfigReader {

    private static ConfigReader instance;
    private Properties props;

    private ConfigReader() {
        props = new Properties();
        try (FileInputStream f = new FileInputStream(
                "src/main/resources/config.properties")) {
            props.load(f);
        } catch (IOException e) {
            throw new RuntimeException("Config not found");
        }
    }

    public static ConfigReader getInstance() {
        if (instance == null) instance = new ConfigReader();
        return instance;
    }

    public String getBaseUrl()  { return props.getProperty("base.url"); }
    public String getBrowser()   { return props.getProperty("browser", "chrome"); }
    public int    getTimeout()   { return Integer.parseInt(props.getProperty("timeout", "30")); }
    public String getUsername()  { return props.getProperty("username"); }
}

// Usage anywhere in framework:
ConfigReader config = ConfigReader.getInstance();
driver.get(config.getBaseUrl());
// TestListener — implements TestNG interface (Abstraction)
public class TestListener
        implements ITestListener, ISuiteListener {

    private Reportable reporter;  // Polymorphism!

    public TestListener() {
        String type = ConfigReader.getInstance()
            .getProperty("report.type", "extent");
        reporter = type.equals("allure")
            ? new AllureReporter()
            : new ExtentReporter();
    }

    @Override
    public void onTestSuccess(ITestResult result) {
        reporter.generateReport(
            result.getName(), "PASSED");
    }

    @Override
    public void onTestFailure(ITestResult result) {
        reporter.generateReport(
            result.getName(), "FAILED");
        // Take screenshot on failure
        DriverFactory.getDriver()
            .takeScreenshot(result.getName());
    }
}
Complete best practices checklist
  • All page classes extend BasePage — inherit driver, wait, and utility methods
  • All test classes extend BaseTest — inherit setUp/tearDown lifecycle
  • Locators are private final fields in page classes
  • Driver managed via ThreadLocal in DriverFactory for parallel execution
  • ConfigReader as Singleton — reads config.properties once
  • Use interfaces (Reportable, DataProvider) for pluggable components
  • Tests call business-level methods (loginWithValidCredentials), not Selenium calls
  • @Override annotation on every overridden method for compile-time safety
  • super(driver) as first line in every page class constructor
  • No static WebDriver — breaks parallel execution

Quick-fire concept summary
Class/Object

Blueprint vs instance. LoginPage is a class. new LoginPage(driver) is an object.

Encapsulation

Private fields + public getters/setters. Locators are always private in page classes.

Inheritance

extends keyword. All page classes extend BasePage to reuse driver/wait/utility methods.

Abstraction

abstract classes + methods. BaseTest hides driver setup. Tests only see setUp()/tearDown().

Polymorphism

Overloading = compile-time. Overriding = runtime. WebDriver is an interface — Chrome/Firefox override its methods.

Interface

Contract via implements. Reportable, DataProvider, ITestListener are all interfaces in real frameworks.

Top 9 interview questions answered

Encapsulation, Inheritance & Abstraction

Q1

"What is Encapsulation?" → "Wrapping data and methods in a class and restricting direct access via private fields with public getters/setters. In POM, locators are private — tests can't access them directly."

Q2

"What is Inheritance in Selenium?" → "All our page classes extend BasePage which has driver, wait, and click/type/getText utilities. Child classes inherit these without rewriting."

Q3

"Difference: abstract class vs interface?" → "Abstract class can have concrete methods and state (fields). Interface is a pure contract. Use abstract class for BasePage (has driver field), interface for Reportable (no state needed)."

Polymorphism & Advanced OOP

Q4

"What is method overloading?" → "Same method name, different parameter list. In BasePage we have click(By), click(WebElement), and click(By, int waitSeconds). Resolved at compile time."

Q5

"What is method overriding?" → "Child class redefines parent method. LoginPage overrides getPageName() from BasePage. Must use @Override. Resolved at runtime."

Q6

"How is polymorphism used in frameworks?" → "We declare WebDriver driver = new ChromeDriver(). WebDriver is an interface — ChromeDriver implements it. Same variable works for any browser."

Design & Framework Questions

Q7

"What is POM and which OOP concepts does it use?" → "POM = Page Object Model. Each page = a class (OOP). Locators are private (Encapsulation). Page classes extend BasePage (Inheritance). Driver setup is hidden in BaseTest (Abstraction)."

Q8

"Why use ThreadLocal for WebDriver?" → "For parallel execution. Static WebDriver is shared across threads — causes race conditions. ThreadLocal gives each thread its own driver instance."

Q9

"What is the Singleton pattern?" → "A class that has only ONE instance throughout the application. ConfigReader is a Singleton — properties file is read once, shared everywhere via getInstance()."
