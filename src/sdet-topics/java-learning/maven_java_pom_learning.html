📦 Overview — What / Why / When / Where / How
What
Maven is a build automation & dependency management tool for Java projects
Why
Automates compiling, testing, packaging, and deploying Java code without manual steps
When
Used in every Java/Selenium/TestNG SDET project to manage JARs and run builds
Where
Runs locally via CLI or inside CI/CD pipelines (Jenkins, GitHub Actions, GitLab)
How
Reads pom.xml file, downloads dependencies from Maven Central, executes lifecycle phases
💬 5 Interview One-Liners
Q: What is Maven? → Maven is a build & dependency management tool that automates Java project compilation, testing, and packaging using pom.xml.
Q: What is pom.xml? → Project Object Model — the XML file that describes project config, dependencies, plugins, and build settings.
Q: What is Maven Central? → A public online repository where Maven downloads all dependency JARs automatically.
Q: What is a Maven repository? → A storage location for JARs — can be Local (~/.m2), Central (internet), or Remote (private Nexus/Artifactory).
Q: What is a Maven Lifecycle? → An ordered sequence of build phases like validate → compile → test → package → install → deploy.

# Skip tests when you only want to build # -DskipTests = compile test code but DON'T run mvn package -DskipTests # -Dmaven.test.skip=true = skip compile AND run mvn package -Dmaven.test.skip=true # Difference: # -DskipTests → test code compiled but not run # -Dmaven.test.skip → test code not even compiled # For SDET: use -DskipTests to catch compile errors
# Run specific test class or method # Run specific test class mvn test -Dtest=LoginTest # Run specific test method in a class mvn test -Dtest=LoginTest#testValidLogin # Run tests matching a pattern (wildcard) mvn test -Dtest="Login*" # Run multiple test classes mvn test -Dtest="LoginTest,RegisterTest" # Useful in CI when only certain tests should run
# Clean build — removes old compiled files first # Always use clean + install for fresh builds mvn clean install # clean + package (no install to local repo) mvn clean package # Verbose output for debugging build issues mvn clean install -X # Quiet — only show errors mvn clean install -q # Offline mode (no internet) — use cached JARs mvn clean install -o
# Maven Profiles — run different configs # Activate a profile named "staging" mvn test -Pstaging # In pom.xml — profile definition: <profiles> <profile> <id>staging</id> <properties> <env.url>https://staging.myapp.com</env.url> </properties> </profile> </profiles> # SDET use: switch between dev/staging/prod URLs
💬 5 Interview One-Liners
Q: What is the Maven default lifecycle? → validate → compile → test → package → verify → install → deploy — each phase runs all previous ones.
Q: Difference between install and deploy? → install copies artifact to local ~/.m2 repo; deploy pushes to a remote repo for other teams to use.
Q: What does mvn clean do? → Deletes the /target directory — all previously compiled files — so next build starts fresh.
Q: What is a Maven profile? → A set of config overrides (properties, plugins) activated with -P flag to support different environments.
Q: What is the difference between -DskipTests and -Dmaven.test.skip? → -DskipTests compiles tests but doesn't run them; -Dmaven.test.skip skips both compilation and execution.

<!-- compile scope (DEFAULT) --> <dependency> <groupId>org.seleniumhq.selenium</groupId> <artifactId>selenium-java</artifactId> <version>4.18.1</version> <scope>compile</scope> <!-- Available: compile + test + runtime Included in final JAR/WAR output Example: Selenium (needed to run app) --> </dependency>
<!-- test scope — only for test code --> <dependency> <groupId>org.testng</groupId> <artifactId>testng</artifactId> <version>7.9.0</version> <scope>test</scope> <!-- NOT available in main source code NOT included in final JAR output Example: TestNG, JUnit, Mockito --> </dependency>
<!-- provided scope — server gives it at runtime --> <dependency> <groupId>javax.servlet</groupId> <artifactId>javax.servlet-api</artifactId> <version>4.0.1</version> <scope>provided</scope> <!-- Compile: YES (use API in code) Final JAR: NO (Tomcat provides it) Example: Servlet API, JSF --> </dependency>
<!-- runtime scope — needed only at execution --> <dependency> <groupId>mysql</groupId> <artifactId>mysql-connector-java</artifactId> <version>8.0.33</version> <scope>runtime</scope> <!-- Not needed to compile (use JDBC interface) Needed to execute DB queries at runtime Example: DB drivers (MySQL, PostgreSQL) --> </dependency>
💬 5 Interview One-Liners
Q: What is dependency scope? → Controls at which build phases a dependency is on the classpath and whether it's included in the final artifact.
Q: When to use test scope? → For libraries only used in test code — TestNG, JUnit, Mockito — so they don't bloat the production JAR.
Q: What is provided scope? → Dependency is needed to compile but provided by the runtime container (e.g. Tomcat provides Servlet API).
Q: What is transitive dependency? → When A depends on B, and B depends on C — Maven automatically pulls in C as a transitive dependency.
Q: How to exclude a transitive dependency? → Use the <exclusions> block inside the dependency to explicitly exclude unwanted transitive JARs.

💬 5 Interview One-Liners
Q: What is a Maven plugin? → A plugin adds extra goals/tasks to Maven — e.g., Surefire runs tests, Compiler compiles code, Allure generates reports.
Q: What is maven-surefire-plugin? → The plugin responsible for running unit tests during the test phase — supports TestNG and JUnit.
Q: Difference between Surefire and Failsafe? → Surefire runs unit tests (*Test.java) in the test phase; Failsafe runs integration tests (*IT.java) in verify phase.
Q: What is a plugin goal? → A specific action within a plugin — like surefire:test or compiler:compile — bound to a lifecycle phase.
Q: What is plugin configuration? → The <configuration> block inside a plugin where you pass settings like thread count, suite file path, or Java version.

Q: What is Maven Standard Directory Layout? → src/main/java for production code, src/test/java for test code, target/ for compiled output — a convention all Maven projects follow.
Q: What is the target folder? → Maven's output directory where compiled classes, test reports, and the final JAR/WAR are placed — always in .gitignore.

💬 5 Interview One-Liners
Q: What is dependencyManagement? → A section in parent pom.xml to declare dependency versions once — child modules inherit them without specifying versions.
Q: What is a BOM in Maven? → Bill of Materials — a pom imported with type=pom/scope=import that provides a curated set of compatible dependency versions.
Q: How do you resolve dependency conflicts? → Run mvn dependency:tree to find conflicts, then use <exclusions> or force a version in dependencyManagement.
Q: What is mvn dependency:tree? → Command that prints all direct and transitive dependencies as a tree, showing their versions and relationships.
Q: What does -U flag do? → Forces Maven to check remote repos for updated SNAPSHOT versions instead of using locally cached ones.

Q: What is distributionManagement? → POM section that defines where to deploy artifacts — the remote repo URL for releases and snapshots.

🎯 5 SDET Interview Q&A
Q1: How do you integrate Maven with Jenkins for running Selenium tests? ▼
In Jenkins, I use a Declarative Pipeline with Maven as a configured tool. The pipeline stages include: 1) checkout from Git, 2) mvn clean compile, 3) mvn test -P{environment} to run Selenium tests, and 4) publish JUnit XML reports from target/surefire-reports/ using the junit step. Q2: How do you run only regression tests in CI using Maven? ▼
I use Maven profiles and TestNG groups. In pom.xml, I define a profile named "regression" pointing to a regression-testng.xml suite file. In CI, I run: mvn test -Pregression. Alternatively I use: mvn test -Dgroups=regression to run only TestNG tests tagged with @Test(groups="regression").
Q3: What happens when two dependencies have different versions of the same transitive JAR? ▼
Maven applies "nearest wins" — whichever version is declared closer to the root of the dependency tree wins. To verify: run mvn dependency:tree to see which version is selected. To enforce a specific version: declare it explicitly in <dependencyManagement> which always takes highest priority.
Q4: How do you pass environment-specific URLs to tests from Maven? ▼
Two approaches: 1) Maven profiles — define a profile with <properties><base.url> per environment, activate with -Pstaging. 2) System properties — pass with -Dbase.url=https://staging.app.com at runtime. In Java: System.getProperty("base.url", "http://default") reads the value. I prefer profiles for team-wide config and -D for quick overrides.
Q5: How do you make Maven not fail the build if some tests fail? ▼
Set <testFailureIgnore>true</testFailureIgnore> in the Surefire plugin configuration. This ensures Maven continues after test failures and generates the full report. In CI, I still mark the pipeline as UNSTABLE (not FAILED) using post-build steps that read the XML report — so failures are visible without blocking deployments.
