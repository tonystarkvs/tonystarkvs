What is Maven? Maven is a build automation and project management tool for Java. It manages your project's build, reporting, and documentation using a central file called pom.xml.

Why Maven? Before Maven, developers had to manually download JAR files and manage classpaths. Maven introduced dependency management — you declare what you need, Maven downloads it automatically from a central repository.

When to use? Any Java project (Selenium, TestNG, RestAssured, Spring) needs Maven to manage dependencies, compile code, run tests, and package the application.

Where does it run? On your local machine (CLI or IDE), CI/CD pipelines (Jenkins, GitHub Actions), and build servers.

Interview one-liners
  • Maven is a build lifecycle management tool that uses convention over configuration.
  • Maven uses pom.xml as its central configuration file to define project metadata and dependencies.
  • Maven downloads dependencies automatically from Maven Central Repository or a custom Nexus/Artifactory server.
  • Maven follows a standard directory structure: src/main/java, src/test/java, src/test/resources.
  • Maven has three built-in lifecycles: default, clean, and site — each with ordered phases.
  • The .m2 folder in the home directory is the local Maven repository where downloaded artifacts are cached.

Best practices
  • Always use mvn clean install before pushing code to ensure a fresh build.
  • Never manually place JAR files in the project — always declare them as dependencies in pom.xml.
  • Keep your Maven version consistent across the team using the mvnw wrapper script.
  • Add target/ to .gitignore — it's generated and should never be committed.
  • Use -DskipTests only for quick packaging, never in CI pipelines for test automation projects.
  • Prefer mvn dependency:tree to debug dependency conflicts and transitive dependencies.
Interview Q&A
Q1: What is Maven and why do we use it in test automation? +
Maven is a build and dependency management tool. In test automation, we use it to automatically download Selenium, TestNG, and other libraries declared in pom.xml, compile test classes, and run tests with a single command like mvn test, making the setup consistent across all team members and CI servers.
Q2: What is the difference between mvn install and mvn deploy? +
mvn install builds the project and copies the artifact to your local ~/.m2 repository. mvn deploy does everything install does, plus it uploads the artifact to a remote repository (like Nexus or Artifactory) so other team members can use it as a dependency.
Q3: What is a transitive dependency in Maven? +
A transitive dependency is an indirect dependency. For example, if your project depends on Selenium, and Selenium internally depends on Guava — then Guava is a transitive dependency. Maven automatically resolves and downloads transitive dependencies. You can see the full tree with mvn dependency:tree.
Q4: What is the role of the .m2 folder? +
The ~/.m2/repository folder is Maven's local cache. When Maven downloads a dependency for the first time, it stores it here. On subsequent builds, Maven reads from this local cache instead of downloading again, making builds faster. The settings.xml inside ~/.m2 can configure proxy, mirrors, and credentials.
Q5: What does mvn archetype:generate do? +
It generates a new Maven project from a pre-built template called an archetype. It creates the standard directory structure (src/main/java, src/test/java), an initial pom.xml, and a sample class — saving you from manual setup. It's the fastest way to bootstrap a Maven project.

What is Maven Lifecycle? A lifecycle is an ordered sequence of named steps called phases. When you run a phase, Maven automatically executes all phases that come before it.

Three built-in lifecycles: default (build & test), clean (delete output), site (generate docs/reports). Each lifecycle has its own ordered phases.

Plugins & Goals: Each phase is implemented by a plugin goal. Example: the test phase runs the goal maven-surefire-plugin:test. A goal is the actual unit of work a plugin performs.

Interview one-liners
  • Maven has three built-in lifecycles: default, clean, and site.
  • When you run mvn test, Maven automatically runs validate → compile → test-compile → test phases in order.
  • A lifecycle phase is a named step; a plugin goal is the actual execution unit bound to that phase.
  • The default lifecycle has 23 phases; the most commonly used are compile, test, package, verify, install, deploy.
  • You can bind multiple goals to a single phase and they execute in the order declared in pom.xml.
  • The clean lifecycle has three phases: pre-clean, clean, post-clean — it removes the target/ folder.
  • Running mvn clean test combines two lifecycles: clean lifecycle then default lifecycle up to test phase.

Best practices
  • Always prefer mvn clean test over just mvn test to avoid stale build artifacts.
  • Use mvn help:effective-pom to debug inherited settings from parent pom.
  • Understand that running a later phase automatically triggers all earlier phases — never skip phases manually.
  • In CI pipelines, use mvn clean verify to ensure both unit and integration tests run.
  • Use mvn dependency:analyze to detect unused or undeclared dependencies.
Interview Q&A
Q1: What is the difference between a Maven lifecycle, phase, and goal? +
Lifecycle is the highest level — a sequence of phases (e.g., default lifecycle). A phase is a named step within a lifecycle (e.g., compile, test, package). A goal is the actual executable task performed by a plugin (e.g., surefire:test). Goals are bound to phases. Running mvn test triggers the test phase which executes the surefire:test goal.
Q2: What plugin runs TestNG tests in Maven? +
The maven-surefire-plugin runs TestNG tests. You configure it in pom.xml to point to your testng.xml suite file. The plugin is bound to the test phase by default and generates test reports in target/surefire-reports/.
Q3: If I run mvn package, what phases execute? +
Maven executes all phases up to and including package in order: validate → initialize → generate-sources → process-sources → generate-resources → process-resources → compile → process-classes → generate-test-sources → process-test-sources → generate-test-resources → process-test-resources → test-compile → test → package. The key ones you need to remember are: validate, compile, test, package.
Q4: What is the purpose of mvn verify? +
mvn verify runs all phases up to and including verify. It's used after package to run integration tests and quality checks (code coverage, static analysis) to verify the package is valid. In SDET work, it's commonly used in CI to run the full test suite including Selenium/API tests configured with the Failsafe plugin.
Q5: How is the clean lifecycle different from the default lifecycle? +
The clean lifecycle has only 3 phases (pre-clean, clean, post-clean) and its purpose is to remove the target/ folder (build output). The default lifecycle has 23 phases and handles compilation, testing, packaging, and deployment. They are completely separate lifecycles — you often combine them: mvn clean install runs clean lifecycle then default up to install.

POM = Project Object Model. The pom.xml file is the heart of every Maven project. It defines everything about your project: coordinates (group, artifact, version), dependencies, plugins, properties, and build configuration.

All elements explained: groupId = your organization namespace, artifactId = project name, version = project version, packaging = output type (jar/war/pom), dependencies = libraries you use, build = plugins and configuration, properties = reusable variables.

Effective POM: Maven merges your pom.xml with the Super POM (Maven's default configuration) to create the effective POM. Run mvn help:effective-pom to see the merged result.

Interview one-liners
  • POM stands for Project Object Model — it is the fundamental unit of work in Maven.
  • Every pom.xml must have exactly three required elements: groupId, artifactId, and version — these form the GAV coordinates.
  • The <dependencies> section declares external libraries; Maven downloads them from the configured repository.
  • The <properties> section allows you to define reusable variables like version numbers to avoid duplication.
  • The <build> section configures plugins and specifies the source and output directories.
  • Dependency scope controls when a dependency is available: compile, test, provided, runtime, system.
  • A parent pom.xml inherits settings to child modules — used in multi-module Maven projects.

Code example 4 — complete pom.xml for SDET project
pom.xml — full SDET project setup
<!-- XML declaration: required at top of every XML file -->
<?xml version="1.0" encoding="UTF-8"?>

<!-- <project> = root element of every pom.xml -->
<!-- xmlns = XML namespace to specify Maven POM schema -->
<project xmlns="http://maven.apache.org/POM/4.0.0">

  <!-- modelVersion: always 4.0.0 for Maven 2/3/4 -->
  <modelVersion>4.0.0</modelVersion>

  <!-- GAV = GroupId + ArtifactId + Version (unique identifier) -->
  <groupId>com.sdet.automation</groupId>
  <artifactId>SeleniumTestNGProject</artifactId>
  <version>1.0-SNAPSHOT</version>
  <!-- SNAPSHOT = under development, not a stable release -->
  <packaging>jar</packaging>
  <!-- packaging: jar (default), war, ear, pom -->

  <!-- <properties>: define reusable variables, avoid duplication -->
  <properties>
    <maven.compiler.source>17</maven.compiler.source>
    <maven.compiler.target>17</maven.compiler.target>
    <selenium.version>4.18.1</selenium.version>
    <testng.version>7.9.0</testng.version>
    <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
  </properties>

  <dependencies>
    <!-- Selenium WebDriver dependency -->
    <dependency>
      <groupId>org.seleniumhq.selenium</groupId>
      <artifactId>selenium-java</artifactId>
      <!-- ${} syntax references the property defined above -->
      <version>${selenium.version}</version>
    </dependency>

    <!-- TestNG: scope=test means available only during testing -->
    <dependency>
      <groupId>org.testng</groupId>
      <artifactId>testng</artifactId>
      <version>${testng.version}</version>
      <scope>test</scope>
    </dependency>
  </dependencies>
</project>
Code example 5 — properties and dependency management
pom.xml — advanced dependency control
<!-- <dependencyManagement>: define versions in parent pom -->
<!-- Child modules inherit these versions without specifying them -->
<dependencyManagement>
  <dependencies>
    <dependency>
      <groupId>org.seleniumhq.selenium</groupId>
      <artifactId>selenium-java</artifactId>
      <version>4.18.1</version>
    </dependency>
  </dependencies>
</dependencyManagement>

<!-- <exclusions>: remove a transitive dependency you don't want -->
<dependency>
  <groupId>org.seleniumhq.selenium</groupId>
  <artifactId>selenium-java</artifactId>
  <version>4.18.1</version>
  <exclusions>
    <exclusion>
      <!-- Exclude a conflicting transitive dependency -->
      <groupId>com.google.guava</groupId>
      <artifactId>guava</artifactId>
    </exclusion>
  </exclusions>
</dependency>
Best practices
  • Always define version numbers in <properties> so you can update them in one place.
  • Use scope=test for TestNG, JUnit, Mockito — they should never end up in production JARs.
  • Use <dependencyManagement> in a parent pom to centralize version control for multi-module projects.
  • Run mvn dependency:analyze to find declared but unused, or used but undeclared dependencies.
  • Never hardcode credentials in pom.xml — use Maven settings.xml or CI environment variables.
Interview Q&A
Q1: What are the three required elements in every pom.xml? +
Every pom.xml must have groupId (organization namespace like com.company), artifactId (project/JAR name), and version (release version). These three together form the GAV coordinates that uniquely identify your artifact in any Maven repository.
Q2: What is the difference between <dependencies> and <dependencyManagement>? +
<dependencies> declares actual dependencies that get added to the classpath immediately. <dependencyManagement> only declares version information — the dependency is NOT added to the classpath unless a child pom or the same pom also lists it in <dependencies>. It's used in parent poms to manage versions centrally.
Q3: What does SNAPSHOT mean in Maven version? +
SNAPSHOT (e.g., 1.0-SNAPSHOT) means the artifact is still under active development and is not a stable release. Maven checks the remote repository for newer SNAPSHOT versions on each build (unless configured otherwise). When you release, you remove SNAPSHOT and publish a stable version like 1.0.
Q4: What is the Super POM? +
The Super POM is Maven's built-in default pom.xml that all projects implicitly inherit from. It defines default values for source directories (src/main/java), output directories (target/), default plugin bindings, and the Maven Central repository. You can view the merged result with mvn help:effective-pom.
Q5: How do you resolve a dependency version conflict in Maven? +
Maven uses nearest-definition wins: the dependency closest in the dependency tree wins. To override a transitive dependency version, you can: (1) explicitly declare the desired version in your own <dependencies>, (2) use <exclusions> to remove the unwanted transitive dependency, or (3) use <dependencyManagement> to enforce a specific version. Always verify with mvn dependency:tree.

Common Maven Plugins for SDET: Maven uses plugins to execute every build action. For SDET, the key plugins are maven-surefire-plugin (unit/TestNG tests), maven-failsafe-plugin (integration tests), maven-compiler-plugin (Java version), and maven-resources-plugin (copy test data files).

Surefire vs Failsafe: Surefire runs *Test.java files during the test phase. Failsafe runs *IT.java files during the integration-test phase. In SDET, Selenium end-to-end tests are often configured under Failsafe so they don't block the package phase if they fail.

Plugin configuration: Every plugin is configured inside the <build><plugins> section of pom.xml. You specify the plugin's GAV coordinates, version, and configuration parameters.

Interview one-liners
  • maven-surefire-plugin runs TestNG and JUnit tests during the test phase.
  • maven-compiler-plugin compiles Java source using the JDK version specified in source and target configuration.
  • Surefire generates reports in target/surefire-reports/ as XML and plain text files.
  • maven-failsafe-plugin is designed for integration tests — a test failure does not immediately fail the build, allowing post-integration cleanup.
  • Plugin executions allow you to bind a plugin goal to a specific lifecycle phase with custom configuration.
  • The maven-resources-plugin copies files from src/test/resources to target/test-classes so tests can access them at runtime.
  • You can run a specific plugin goal directly: mvn surefire:test without running earlier lifecycle phases.

Code example 7 — maven-surefire-plugin with TestNG suite
pom.xml — surefire plugin config
<plugin>
  <groupId>org.apache.maven.plugins</groupId>
  <artifactId>maven-surefire-plugin</artifactId>
  <version>3.2.5</version>
  <configuration>
    <!-- suiteXmlFiles: tell Surefire to use testng.xml -->
    <!-- instead of auto-discovering *Test.java files -->
    <suiteXmlFiles>
      <suiteXmlFile>src/test/resources/testng.xml</suiteXmlFile>
    </suiteXmlFiles>

    <!-- reportsDirectory: where XML reports are saved -->
    <reportsDirectory>${project.build.directory}/surefire-reports</reportsDirectory>

    <!-- testFailureIgnore: true = don't fail build on test failure -->
    <!-- Use with caution — only when you still want reporting -->
    <testFailureIgnore>false</testFailureIgnore>

    <!-- parallel: run tests in parallel threads (TestNG feature) -->
    <parallel>methods</parallel>
    <threadCount>4</threadCount>
  </configuration>
</plugin>
Best practices
  • Always specify the plugin version explicitly — never rely on Maven's default plugin versions, which may vary between Maven versions.
  • Use maven-failsafe-plugin for Selenium E2E tests so a failed UI test doesn't prevent packaging.
  • Set maven.compiler.source/target in <properties> rather than in the plugin config — it's cleaner and reusable.
  • Use parallel=methods and threadCount in Surefire only after confirming tests are thread-safe.
  • Check target/surefire-reports/*.xml in CI for detailed failure messages before looking at console output.
Interview Q&A
Q1: What is the difference between Surefire and Failsafe plugins? +
Surefire runs unit tests (files matching *Test.java) during the test phase — a failure immediately fails the build. Failsafe runs integration tests (*IT.java) during the integration-test phase — a failure is recorded but the build continues to the verify phase where results are checked. For Selenium E2E tests, Failsafe is preferred so cleanup code (e.g., stopping a server) still runs.
Q2: How do you configure Surefire to run a specific testng.xml file? +
Inside the Surefire plugin's <configuration>, add a <suiteXmlFiles> element pointing to your testng.xml path. Example: <suiteXmlFile>src/test/resources/testng.xml</suiteXmlFile>. This tells Surefire to use TestNG's XML-based test organization instead of auto-scanning for test classes.
Q3: Where are Surefire test reports saved and how do you read them? +
Surefire saves reports in target/surefire-reports/ as two formats: XML files (TEST-*.xml) that CI tools like Jenkins parse to display test results, and plain text files (*.txt) with console output. To view them as formatted HTML, configure the maven-surefire-report-plugin and run mvn surefire-report:report — the HTML is saved to target/site/.
Q4: What is a plugin execution in Maven? +
A plugin execution is a way to bind a plugin goal to a specific lifecycle phase with custom configuration. Using <executions><execution>, you can run the same plugin multiple times in different phases with different configurations — for example, running surefire in both the test phase and again in verify with different test suite files.
Q5: How do you skip test execution in Maven? +
There are two ways: -DskipTests=true compiles test classes but skips running them (Surefire and Failsafe), -Dmaven.test.skip=true skips compilation AND execution of tests entirely. The first is preferred when you still want to verify tests compile. Neither should be used in regular CI — only for quick local packaging when you've already verified tests pass.

TestNG in pom.xml — multiple ways: You can configure TestNG test execution in pom.xml in several ways: pointing to a single testng.xml, multiple testng.xml files, using includes/excludes patterns, using groups, or using Maven profiles to switch between suites based on environment.

Maven Profiles: A <profile> is a set of configuration values that override or extend the default build. Profiles allow you to run different browsers, different test suites, or different environments with a single command-line flag: -P profileName.

Command line alternatives: Maven allows you to pass system properties (-D) and activate profiles (-P) to control which tests run, which browser to use, and which environment to target — all without changing pom.xml.

Interview one-liners
  • You can pass TestNG XML file location via command line: mvn test -DsuiteFile=regression.xml
  • Maven profiles (-P) allow switching between different browser or environment configurations without code changes.
  • System properties passed with -D are accessible in Java via System.getProperty("key").
  • TestNG groups can be included or excluded in Surefire config using <groups> and <excludedGroups> elements.
  • Multiple testng.xml files can be specified in Surefire's <suiteXmlFiles> — they run sequentially.
  • A profile can be activated automatically by OS, JDK version, property value, or file presence.
  • In CI/CD, mvn clean test -Pbrowser=chrome -Denv=staging is a typical pipeline command in SDET projects.

<profiles>

Code example 10 — reading system properties in Java test
Java — BaseTest.java reading Maven -D properties
import org.openqa.selenium.WebDriver;
import org.openqa.selenium.chrome.ChromeDriver;
import org.openqa.selenium.firefox.FirefoxDriver;
import org.testng.annotations.BeforeClass;

public class BaseTest {
    protected WebDriver driver;

    @BeforeClass
    public void setUp() {
        // System.getProperty reads -Dbrowser=chrome from command line
        // "chrome" = default value if -Dbrowser is not specified
        String browser = System.getProperty("browser", "chrome");

        // System.getProperty reads -Denv=staging from command line
        String env = System.getProperty("env", "dev");

        if (browser.equalsIgnoreCase("chrome")) {
            driver = new ChromeDriver();
        } else if (browser.equalsIgnoreCase("firefox")) {
            driver = new FirefoxDriver();
        }

        // Load base URL based on environment
        String baseUrl = env.equals("staging")
            ? "https://staging.myapp.com"
            : "https://dev.myapp.com";
        driver.get(baseUrl);
    }
}
/* Run with: mvn clean test -Dbrowser=firefox -Denv=staging */
Best practices
  • Use -P profile for structural changes (different plugins, suites) and -D property for values (browser name, URL).
  • Always provide a default value in System.getProperty("key", "defaultValue") to prevent NullPointerExceptions.
  • Name profiles descriptively: smoke, regression, chrome, staging — not p1, p2.
  • In Jenkins pipelines, inject browser and environment as build parameters and pass them as -D flags.
  • Use <activeByDefault>true</activeByDefault> on your most common profile to set sensible defaults.
Interview Q&A
Q1: How do you run only smoke tests using Maven command line? +
Two ways: (1) If you have a separate smoke testng.xml: mvn test -DsuiteFile=smoke.xml. (2) If your tests use TestNG groups (@Test(groups="smoke")): configure Surefire with <groups>smoke</groups> or pass it via command line mvn test -Dgroups=smoke. The second approach is more flexible for filtering without XML changes.
Q2: What is the difference between -D and -P in Maven? +
-D sets a Java system property that your test code reads with System.getProperty(). For example, -Dbrowser=chrome sets the browser. -P activates a named Maven profile defined in pom.xml, which can change plugin configurations, add/remove dependencies, or override properties. You often use both together: mvn test -Pchrome -Denv=staging.
Q3: Can you run multiple browsers in parallel with Maven? +
Yes. You can: (1) Use TestNG's parallel="tests" in testng.xml with separate <test> blocks for each browser. (2) Use Surefire's <parallel>suites</parallel> with multiple testng.xml files — each suite runs in its own thread. (3) Use a Grid or cloud provider (Selenium Grid, LambdaTest, BrowserStack) and pass the browser capability via System.getProperty().
Q4: How do you activate a Maven profile automatically? +
Using the <activation> element inside a profile. Profiles can auto-activate based on: OS (<os><name>Windows</name>), JDK version (<jdk>17</jdk>), a property value (<property><name>env</name><value>prod</value>), or file existence (<file><exists>path</exists>). The most common in SDET is property-based activation.
Q5: What is systemPropertyVariables in Surefire config? +
<systemPropertyVariables> inside the Surefire plugin configuration sets Java system properties specifically for the test JVM process that Surefire forks. Unlike -D on the command line (which sets the property in the Maven JVM), systemPropertyVariables ensures the property is available in the forked test process. This is important because Surefire runs tests in a separate JVM by default.

TestNG XML — the test suite file: A testng.xml file defines which tests to run, in what order, with what parallelism, and with what parameters. Maven's Surefire plugin reads this file when configured with <suiteXmlFiles>.

Key elements: <suite> (top level), <test> (a group of classes), <classes> (specific test classes), <groups> (include/exclude test methods by group), <parameter> (pass data to @Parameters), <listeners> (reporting listeners).

Parallel execution: TestNG supports parallel running at suite, test, class, or method level — configured in the parallel attribute of <suite> or <test>.

Interview one-liners
  • testng.xml is the suite configuration file that organizes which test classes and methods TestNG should run.
  • The parallel attribute can be set to tests, classes, methods, or instances to enable concurrent execution.
  • <parameter name="browser" value="chrome"/> in testng.xml passes data to @Parameters({"browser"}) annotated methods.
  • TestNG listeners (implementing ITestListener) are added in testng.xml to capture events like onTestFailure.
  • The preserve-order="true" attribute ensures tests run in the order listed in testng.xml.
  • You can include or exclude entire packages, classes, or methods using <packages>, <classes>, and <include/exclude>.

testng.xml — 4 different configurations
<!-- CONFIG 1: Basic suite with specific classes -->
<!DOCTYPE suite SYSTEM "https://testng.org/testng-1.0.dtd">
<suite name="SmokeSuite" verbose="1">
  <test name="LoginTests">
    <classes>
      <!-- name = fully qualified class name -->
      <class name="com.sdet.tests.LoginTest"/>
      <class name="com.sdet.tests.HomeTest"/>
    </classes>
  </test>
</suite>

<!-- CONFIG 2: Parallel execution with browser parameter -->
<!-- parallel="tests" = each <test> block runs in its own thread -->
<suite name="CrossBrowser" parallel="tests" thread-count="2">
  <test name="Chrome Tests">
    <!-- parameter: passed to @Parameters({"browser"}) -->
    <parameter name="browser" value="chrome"/>
    <classes><class name="com.sdet.tests.CheckoutTest"/></classes>
  </test>
  <test name="Firefox Tests">
    <parameter name="browser" value="firefox"/>
    <classes><class name="com.sdet.tests.CheckoutTest"/></classes>
  </test>
</suite>

<!-- CONFIG 3: Using groups to include/exclude -->
<suite name="RegressionSuite">
  <test name="Full Regression">
    <groups>
      <run>
        <include name="smoke"/>
        <include name="regression"/>
        <!-- exclude: skip tests tagged @Test(groups="wip") -->
        <exclude name="wip"/>
      </run>
    </groups>
    <packages>
      <package name="com.sdet.tests"/>
    </packages>
  </test>
</suite>
Best practices
  • Keep testng.xml in src/test/resources/ so Maven copies it to target/test-classes/ and Surefire can find it.
  • Use preserve-order="true" in <test> when test execution order matters (e.g., end-to-end flows).
  • Create separate testng.xml files for different scopes: smoke.xml, regression.xml, sanity.xml.
  • When using parallel, use ThreadLocal WebDriver to avoid sharing browser instances across threads.
  • Add the ExtentReports or Allure listener in testng.xml <listeners> to get rich HTML reports automatically.
Interview Q&A
Q1: What is testng.xml and why is it needed in Maven? +
testng.xml is TestNG's suite configuration file. It organizes which test classes, packages, and methods to run, in which order, with what parallelism, and with what parameters. In Maven, when Surefire is configured with <suiteXmlFiles> pointing to testng.xml, it uses TestNG's XML-based test organization instead of its default pattern-matching (which finds *Test.java files). This gives you full control over test execution.
Q2: How do you run tests in parallel using testng.xml? +
Set the parallel attribute on the <suite> or <test> element. Values: methods (each test method runs in its own thread — fastest but tests must be stateless), classes (each class in its own thread), tests (each <test> block in its own thread — safest for browser testing), instances. Set thread-count to control max concurrency. Always use ThreadLocal WebDriver to keep browser instances isolated per thread.
Q3: What is the @Parameters annotation and how does testng.xml feed it? +
@Parameters({"browser", "env"}) annotates a @BeforeClass or @Test method to receive values from testng.xml's <parameter> elements. Example: <parameter name="browser" value="chrome"/> in testng.xml → the method receives "chrome" as the browser argument. This allows data-driven browser or environment configuration without hardcoding in test code.
Q4: How do TestNG groups work? Give an SDET example. +
Groups allow you to tag test methods with labels like @Test(groups={"smoke", "regression"}). In testng.xml, you use <groups><run><include name="smoke"/> to run only smoke-tagged tests. SDET example: smoke group for 5-minute CI sanity check, regression for full nightly suite, wip for excluded work-in-progress tests. Combine with Maven profiles to run different groups in different pipelines.
Q5: Where should testng.xml be placed in a Maven project? +
It should be in src/test/resources/. Maven's maven-resources-plugin automatically copies everything from src/test/resources/ to target/test-classes/ during the build. Surefire then reads it from there. You reference it in pom.xml as src/test/resources/testng.xml. Placing it at the project root also works but is not the Maven convention.

Reporting in Maven: Maven generates test reports automatically via Surefire (XML/text) and optionally via the maven-surefire-report-plugin (HTML). For rich SDET reporting, teams integrate ExtentReports or Allure Framework.

Where reports are saved: Surefire XML/text → target/surefire-reports/. Surefire HTML → target/site/surefire-report.html. Allure results → target/allure-results/ → HTML at target/site/allure-maven-plugin/. ExtentReports → configurable, usually test-output/ExtentReport.html.

Site lifecycle: Running mvn site generates a complete project website with all configured reports in target/site/. Open target/site/index.html in a browser to view it.

Interview one-liners
  • Surefire generates XML reports in target/surefire-reports/ which Jenkins's JUnit plugin parses to display test results.
  • Run mvn surefire-report:report to generate a formatted HTML test report at target/site/surefire-report.html.
  • Allure Framework provides rich visual reports with steps, screenshots, attachments, and trend charts.
  • The maven-site-plugin aggregates all configured reporting plugins into a single project website.
  • ExtentReports is configured as a TestNG listener — not a Maven plugin — and generates its HTML wherever you set the output path.
  • The <reporting> section in pom.xml configures plugins that generate reports when mvn site runs.

Interview Q&A
Q1: Where does Maven Surefire save test reports and in what format? +
Surefire saves two formats in target/surefire-reports/: (1) XML files (TEST-classname.xml) containing pass/fail counts, test names, and failure messages — these are read by Jenkins's JUnit plugin and GitHub Actions; (2) plain text files (classname.txt) with the console output of each test class. To get HTML, run mvn surefire-report:report which reads those XMLs and generates target/site/surefire-report.html.
Q2: What is the difference between Surefire report and Allure report? +
Surefire report is a basic HTML/XML report showing pass/fail counts and stack traces. Allure report is a rich interactive dashboard showing test steps (via @Step), screenshots (via @Attachment), test history trends, categories of failures (broken vs failed), severity, and feature groupings. Allure requires adding the allure-testng dependency and a listener; Surefire report is available out of the box with Maven.
Q3: How does mvn site work and what does it generate? +
mvn site runs the site lifecycle which triggers all plugins configured in the <reporting> section of pom.xml. It generates a complete HTML project website in target/site/ including: project information (from pom.xml metadata), test reports (from surefire-report-plugin), dependency reports, source cross-reference, and any other configured reporting plugins. The entry point is target/site/index.html.
Q4: How do you integrate ExtentReports with Maven TestNG project? +
ExtentReports doesn't use a Maven plugin — it integrates as a TestNG listener. Steps: (1) Add ExtentReports dependency in pom.xml <dependencies>. (2) Create a class implementing ITestListener that initializes ExtentReports and logs pass/fail events. (3) Add this listener class to testng.xml in <listeners><listener classname="..."/>. (4) ExtentReports generates HTML at the path you configure, typically test-output/ExtentReport.html.
Q5: How do you publish test reports in Jenkins for a Maven project? +
Two main options: (1) Jenkins's built-in JUnit plugin — add a post-build step "Publish JUnit test result report" pointing to **/surefire-reports/*.xml — Jenkins displays trend graphs automatically. (2) Allure Jenkins plugin — configure it to read target/allure-results/ after each build and generate the rich Allure report as a build artifact with history. The first is simpler; the second gives much richer reporting with trends, categories, and step-level details.
